Tutorial

Your first API call

By the end of this tutorial we will have authenticated against the Study API, created a user, read it back by composite key, and seen how idempotency saves us from accidental duplicates. End-to-end time: ~5 minutes.

Prerequisites

  • curl in the shell (any version ≥ 7.50).
  • An API key — either the local-dev placeholder or one issued by an operator.
  • A running Study API instance — version v1, default bind 127.0.0.1:8000.

What we will do

  • Authenticate with the X-API-Key header.
  • Send a write with an Idempotency-Key for safe retry.
  • Read the freshly created user by (system_uuid, system_user_id).
  • See the error envelope when a request fails validation.

1 · Set the base URL and API key

Export the local-dev base URL and our API key into the shell. Every endpoint lives under /api/v1.

export BASE_URL="http://127.0.0.1:8000"
      export API_KEY="your-key-here"

For the rest of this page we reuse both variables. The env-vars reference lists every consumer-relevant configuration knob.

2 · Authenticate the request

Every protected endpoint requires the X-API-Key header. Missing or invalid keys come back as 401 with code SECURITY_AUTH_REQUIRED (COMMON_401). See the authenticate how-to for the full token lifecycle and the security model for the rationale.

3 · Create a user (POST)

Required fields: system_user_id, system_uuid, full_name. Optional: username, timezone. The composite key (system_user_id, system_uuid) must be unique — re-using it returns 409 USER_CREATE_ALREADY_EXISTS. 

curl -X POST "$BASE_URL/api/v1/user" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $API_KEY" \
  -H "Idempotency-Key: tutorial-create-user-1" \
  -d '{
    "system_user_id": "42",
    "system_uuid": "b2c3d4e5-0002-4000-8000-000000000002",
    "full_name": "Ivan Petrov",
    "timezone": "Europe/Moscow"
      }'

Successful response (201 Created):

{
  "client_uuid": "1f2c…",
  "system_user_id": "42",
  "system_uuid": "b2c3d4e5-0002-4000-8000-000000000002",
  "full_name": "Ivan Petrov",
  "username": null,
  "timezone": "Europe/Moscow",
  "is_row_invalid": 0,
  "created_at": "2026-05-08T10:15:30.123456+00:00",
  "updated_at": "2026-05-08T10:15:30.123456+00:00"
      }

The server-issued client_uuid is the stable identifier for subsequent reads. The Idempotency-Key we sent makes this call safely retryable: re-running the exact same request returns the same 201 with the original body. See handle idempotency for the contract details.

4 · Read the user back (GET)

Reads use the composite path key (system_uuid, system_user_id) — note the order:

curl "$BASE_URL/api/v1/user/b2c3d4e5-0002-4000-8000-000000000002/42" \
      -H "X-API-Key: $API_KEY"

We get back the same payload as the create response. If the user does not exist the server returns 404 USER_NOT_FOUND with the standard error envelope (handle errors).

5 · See a validation failure

To see the error envelope first-hand, we send a deliberately bad request — e.g. omit full_name:

curl -X POST "$BASE_URL/api/v1/user" \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $API_KEY" \
  -H "Idempotency-Key: tutorial-bad-1" \
      -d '{"system_user_id":"99","system_uuid":"b2c3d4e5-0002-4000-8000-000000000002"}'

Response (422 Unprocessable Entity):

{
  "type": "about:blank",
  "title": "Validation failed",
  "status": 422,
  "code": "USER_CREATE_FULL_NAME_REQUIRED",
  "detail": "Field `full_name` is required.",
  "request_id": "01HE…",
  "errors": [
    {"field": "full_name", "code": "USER_CREATE_FULL_NAME_REQUIRED", "message": "Field is required"}
  ]
      }

Stable codes (e.g. USER_CREATE_FULL_NAME_REQUIRED) are safe to switch on — they never change between releases. The full catalogue is in reference / errors.

What we did

  • Set BASE_URL and API_KEY, authenticated with X-API-Key.
  • Created one user with an Idempotency-Key guard, read it back, observed a validation failure with a stable code.
