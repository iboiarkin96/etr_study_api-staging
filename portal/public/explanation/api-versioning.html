Explanation

API versioning

How the HTTP surface of the ETR Study API evolves over time — what changes you can rely on inside a major version, what counts as breaking, and how to lock your client to a known shape.

The promise

The HTTP surface is versioned via the URL path: every endpoint lives under /api/v1. Within a major version, the contract evolves additively — new fields, new endpoints, new error codes, new optional headers — but does not break what was already published.

Additive vs. breaking

Change kind Allowed in v1? Notes
New endpoint added Allowed Existing endpoints unchanged.
New optional request field Allowed Default behaviour preserved when the field is absent.
New response field Allowed Clients must ignore unknown fields; that is the integrator's contract.
New error code value Allowed Surfaces as the existing HTTP status; integrators that switch on the new code opt in deliberately.
Tighter validation on an existing field Breaking Requires v2. Loosening validation is additive.
Renamed / removed field Breaking Requires v2.
Status-code change for the same condition Breaking Requires v2.
Removed / repurposed error code Breaking Codes are stable for the lifetime of v1; deprecation is announced separately.

Compatibility is enforced by an automated check against the committed openapi-baseline.json: the build fails on any breaking diff.

When a new major version ships

If a breaking change is unavoidable, it lands behind a new prefix (/api/v2) — v1 continues to serve. Both versions are documented in the OpenAPI explorer until the deprecation window for v1 closes. The plan, dates, and migration guide live in a release note; integrators get at least one full release of overlap before retirement.

How to lock to a known shape

  • Pin against a snapshot of openapi-baseline.json in your client repo and regenerate models on a schedule.
  • Surface the API's emitted request_id in your client logs — it is the fastest way to correlate a problem report with a server-side trace.
  • Read the public CHANGELOG before upgrading the pinned schema.
