SLO & error budget

What we measure for the consumer-facing surface, what we commit to, and how breaches are handled.

Reliability commitment

The Study API publishes Service-Level Objectives (SLOs) for the metrics integrators care about: availability and latency. This page is the consumer-facing summary of what we measure, what we commit to, and how breaches are handled.

What we measure

  • Availability SLI. Fraction of /api/v1/* responses with status < 500 over a 28-day rolling window. 4xx client errors do not consume the budget — your bad request is your budget, not ours.
  • Latency SLI. 95th-percentile end-to-end response time per route, measured server-side, over the same window. Excludes external network time.
  • Health probes. /live and /ready — used by load balancers; not part of the consumer SLO.

How a breach is handled

  1. The error budget is the complement of the availability target; if the SLO is 99.5%, the monthly budget is 0.5% of requests.
  2. Burn-rate alerts (fast burn / slow burn) fire when the budget is consumed faster than expected; they go to on-call.
  3. Each consumed-budget incident gets a postmortem.
  4. While the budget is exhausted, feature work pauses in favour of reliability work — a hard policy, not a guideline.

Why 99.5%, not 99.9%

99.5% over 28 days is a 0.5% error budget — about 3.4 hours of allowed disruption per month. A 99.9% target (0.43 hours/month) sounds better but would force two changes the public surface is not yet ready for: a multi-zone hot-standby database, and a release process gated on canary success criteria measured against 4xx-isolated latency. Both have their own roadmap items; until they land, committing to 99.9% would mean either burning the budget on infrastructure work that has not happened or quietly breaching the SLO. We picked the number we can defend with the deployment we actually run.

Latency follows the same logic: the p95 target excludes external network time because that is the integrator's tail to control, not ours. Tightening to p99 would force us to optimise for a long-tail population we cannot reliably identify — gateway scrapers, mobile carriers, misconfigured retry storms — none of which are signal about the service's health.

What integrators should expect

  • Transient 5xx happen — retry with the same Idempotency-Key (see how-to: handle idempotency). The contract guarantees no duplicate side effects.
  • Long-tail latency events (1-in-1000 slow requests) are normal. Use a sane client-side timeout and your own retry policy.
  • Planned maintenance is announced via the public CHANGELOG. There is no separate status page in v1; that lands when traffic warrants it.
