Explanation

Idempotency rationale

Network failures, client timeouts and proxy retries make every POST ambiguous: did the server apply the write or not? The Study API resolves that ambiguity with a header-based Idempotency-Key contract — chosen over the obvious alternatives because it keeps the client API trivial, the server logic explicit, and the failure mode loud.

Why keyed replay, not server-side deduplication

Without a stable idempotency key, a single retry of a successful-but-unacknowledged write creates two rows. A naive deduplication strategy server-side (hash the body, look for a recent match) silently breaks the moment two genuinely-distinct intents share a payload shape — for example, two human "Create" clicks for the same form. The header-based contract makes intent explicit: one key per logical operation. The server cannot guess; the client cannot drift.

What "logical operation" means

One key per intent, not one key per HTTP call. Concretely:

  • If your code retries on a transient failure (timeout, 5xx, connection reset), reuse the key — that is the same intent.
  • If a human user clicks "Create" twice in a UI, that is two intents — mint two keys.
  • If you need to recreate the same row after a soft-delete, that too is a new intent — mint a new key.
  • If a background job processes a queue with at-least-once delivery, the queue message ID is a natural idempotency key — there is exactly one intent per message, even if the worker crashes mid-write.

Trade-offs we accepted

OptionVerdictWhy
Header-based Idempotency-Key (the choice)AdoptedStripe-style contract integrators already know; server logic is one read + one hash compare; failure mode (COMMON_409) is loud and unambiguous.
Server-side body-hash deduplication (no header)RejectedConflates "two retries of one intent" with "two intents that happen to look identical"; silent correctness bug.
Idempotent PUT with client-issued resource IDsPartially usedWorks for resources whose business key is client-issued; does not generalise to flows where the server mints an ID, and forces clients to invent ID schemes.
At-most-once delivery (no retries)RejectedPushes the duplication-prevention problem entirely to the network — incompatible with retry-aware clients and queue workers.

Implications for your client

  • Generate the key before the first attempt and persist it locally with the request payload until you receive a definitive (non-retryable) response.
  • Retry with the same key on transient errors (5xx, network reset). The server-side replay window is 24 hours.
  • Treat 409 IDEMPOTENCY_KEY_REUSED_WITH_DIFFERENT_PAYLOAD as a programmer error — never auto-recover.
  • For queue workers, prefer using the queue message ID as the idempotency key — it gives you "exactly once" semantics on top of "at least once" delivery.
