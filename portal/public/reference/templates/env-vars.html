Env-vars template

Use this shape when documenting the consumer-relevant configuration surface of a service. Distinguish loudly between consumer-relevant (part of the public contract) and operator-only (internal, may change without notice).

1 · Profiles

Brief paragraph naming the profile selector (e.g. APP_ENV) and the file layout under env/. State precedence: file ⟶ parent shell ⟶ explicit override.

2 · Consumer-relevant variables

Single .docs-table with columns Variable, Default, What it does. Defaults that come from the profile read "(profile)"; defaults that come from code read literally.

3 · Where they are loaded

One paragraph describing the loader's precedence order. Refer back to the profile file path.

4 · Internal-only (excluded)

One sentence listing categories that are not on this page (DB DSN, log paths, Prometheus scrape config, etc.). Internal-only env-vars are documented in the SA-internal handbook.

5 · Related

  • Link to the security model for the defaults' rationale.
  • Link to the matching OpenAPI explorer for any error codes these vars can trigger (rate limit, body cap, etc.).

Anti-patterns to reject in review

  • Mixing internal-only vars with consumer vars on the same page — splits the public contract.
  • Rationale paragraphs ("we chose this default because…") — that is Explanation.
  • Step-by-step ("set this, then export that") — that is How-to.
