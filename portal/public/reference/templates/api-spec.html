API spec template

Use this template when documenting a single endpoint by hand (i.e. when Scalar/OpenAPI cannot do the job). It mirrors §1–§6 of the canonical Reference template from diataxis.fr/reference/ and matches the Scalar surface our OpenAPI explorer generates.

1 · Subject

One short paragraph naming the endpoint, its HTTP method, and the resource it acts on. No marketing language, no rationale.

<h2 id="subject">1 · Subject</h2>
      <p><code>POST /api/v1/<resource></code> — creates a new ⟨resource⟩.</p>

2 · Spec (parameters, types, ranges)

Table per category (path · query · headers · body). Use the kit's .docs-table with sticky head; describe types in OpenAPI terms (integer, string(uuid), RFC3339).

3 · Returns

One sub-section per success status. Show the canonical body shape; never describe behaviour ("usually returns") — state it ("returns").

4 · Errors

Table mapping HTTP status × stable code × meaning. Link each row into the error catalogue rather than duplicating the catalogue inline.

5 · Examples

Minimal curl + minimal JSON. No storytelling. Two examples max: happy path + one canonical failure.

6 · Related

  • Other endpoints on the same resource.
  • An Explanation page covering the why behind the design.

Anti-patterns to reject in review

  • Step-by-step «send this, then send that» — that is a How-to, not Reference.
  • Rationale paragraphs ("we chose X because Y") — that is Explanation.
  • Hedging language ("usually", "may", "we recommend") — drop or move to Explanation.
  • Hand-authored copy of fields that already live in openapi-baseline.json — generate instead.
