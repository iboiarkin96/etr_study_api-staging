Error catalogue template

Use this shape when listing the public-facing error codes of a service. Group codes by surface (COMMON_* for transport / auth / idempotency; <DOMAIN>_* for business and validation). The catalogue is a human index over the OpenAPI baseline — never a parallel source of truth.

1 · Envelope

One JSON snippet of the canonical error body. Spell out each field once below the snippet (status, code, detail, request_id, errors[]).

2 · COMMON_* codes

Cross-surface codes: framing, auth, idempotency, infrastructure. Always include status 400/401/409/413/429.

3 · <DOMAIN>_* codes

Per-resource codes split into business (e.g. USER_101) and validation (USER_001–USER_0xx) tables.

4 · Code stability

  • Stable identifier: the code string.
  • Mutable: the detail message.
  • Numeric prefix is organisational, not semantic — switch on the symbolic name.

Anti-patterns to reject in review

  • Codes invented on the page that do not appear in openapi-baseline.json.
  • Hedging language ("usually returns 422") — error catalogues are normative.
  • Inlining the recovery recipe — that belongs in a How-to.
