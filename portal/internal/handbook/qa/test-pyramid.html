QA Explanation

Test pyramid & layer ownership

The pyramid models where each kind of check lives in this codebase. Each layer answers a specific question — fastest at the bottom, broadest at the top. A test belongs to exactly one layer; if you can't decide, push it down.

Shape

  1. Manual
  2. System / E2E
  3. Contract
  4. Integration
  5. Unit
  • Manual / exploratory minutes per session — charter-based
  • System / E2E (manual + API) golden paths only — run on releases
  • Contract (HTTP / OpenAPI) every endpoint — schema + errors
  • Integration (DB, repo) every repository & migration
  • Unit (pure functions) the bulk — fast, isolated, no I/O

Read top-down for "what does the user feel" or bottom-up for "where do we get fast feedback". A test belongs to exactly one layer; when you can't decide, push it down.

Unit

Owner: developer. QA reads but rarely writes — uses unit tests as documentation of intended behavior.

Tests one function or one small class with no I/O. No DB, no HTTP, no filesystem (except a tmp_path), no time, no randomness.

  • What we expect: exhaustive on logic; one test per branch; one test per error path.
  • Where in this repo: tests/core/, tests/domain/.
  • Run: pytest tests/core tests/domain — should be sub-second per module.
  • QA signal: if a unit test is missing for a defect you found, file a follow-up referencing the test that should have caught it.

Integration (DB, repository, infra adapters)

Owner: developer. QA reviews for missing coverage.

Tests a real component with one real adapter — usually the database. We keep the transaction scope tight and roll back per test. We do not mock the database in this layer on principle: mocked tests passed in the past while a migration broke production.

  • Where: tests/repositories/ and migration tests.
  • Test data: created in fixtures; never relies on prior test order.
  • QA signal: if a defect is rooted in SQL behavior or migration order, the regression test belongs here, not on the API layer.

Contract (HTTP / OpenAPI)

Owner: developer + QA. This is the layer QA actively co-owns.

Tests every public endpoint against the documented contract. Verifies status codes, response schema, error envelope, idempotency keys, headers (X-Request-ID), pagination, and known error catalog entries. The OpenAPI baseline is verified as a single test so contract drift fails CI.

  • Where: tests/api/v1/, tests/openapi/.
  • Oracle: the spec page under services/portal/internal/services/api/reference/<resource>/ and the shared docs under _shared/.
  • Per endpoint, expect: one happy-path success, one or more validation-error cases, one auth-failure case, and schema validation against the OpenAPI document.
  • QA signal: any defect with a status-code or schema dimension lands here.

System / end-to-end

Owner: QA, with developer pairing on flow design.

Tests a user-visible flow across multiple endpoints or pages. We keep this layer small — only golden paths and one or two failure modes per flow. Every E2E test you add is a new tax forever; pay it only for high-impact paths.

  • Examples: "create user → create conspectus → mark reviewed → appears in due list".
  • How we run them: manual via the API playbook for now; candidate flows for automation are tracked in backlog.
  • Evidence: request/response captures attached to the run report.

Manual & exploratory

Owner: QA.

Two flavours:

  • Scripted manual — running a checklist line by line. See the documentation visual checklist.
  • Exploratory — chartered, time-boxed sessions to find what scripted tests miss. See exploratory playbook.

We deliberately keep manual scripted runs short — anything that repeats every release should be a candidate for the contract layer.

Anti-patterns to avoid

  • Inverted pyramid (ice-cream cone). Heavy E2E, thin unit. Slow CI, flaky signal. If our pyramid leans this way, push tests down.
  • Mocking the database in integration tests. A historical incident is the reason we don't. Use a real DB or skip the test.
  • Asserting on log strings as a proxy for behavior. Assert on the behavior; logs are diagnostics.
  • Tests that depend on test order. Use independent fixtures.
  • Snapshot tests on rapidly changing UI. They become "press accept" tests. Reserve snapshots for stable contracts.
  • Manual checks for things a contract test would catch. If the rule is "field X must be UUID", that is contract-layer, not a checklist line.

Tradeoffs

The pyramid encodes deliberate costs. Choosing it means accepting these tensions rather than pretending they don't exist:

  • Speed vs realism. Unit tests are sub-second but don't catch wiring bugs. Contract tests catch wiring but take 10–100× longer. We pay the realism cost only at layers above unit.
  • Coverage vs coupling. A test at a higher layer covers more code per assertion but also breaks for more reasons. A broken e2e test rarely points at a single line; a broken unit test usually does. The pyramid optimises for diagnostic resolution by keeping the base wide.
  • Confidence vs cost. Adding an e2e test feels safer but doubles maintenance for every refactor it covers. Pushing the same check down to contract or unit gives you ~80% of the confidence for ~20% of the long-term tax.
  • «No mocks» at integration vs CI speed. A real DB at the integration layer is the project's hard-won lesson, but it does slow CI vs mocked alternatives. We accept the slower run to avoid the «mock passes / migration fails in prod» class of bug.
  • Manual layer as a feature, not a gap. A small dedicated manual layer is cheaper than automating every rarely-changing flow. The discipline is to actively prevent the layer from growing into a regression sink.
