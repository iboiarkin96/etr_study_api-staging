Severity & priority matrix

Two independent dimensions: severity = how bad is it, priority = how soon to fix. A defect can be S1 / P3 (data corruption on a feature nobody uses yet) or S4 / P1 (annoying typo on the home page that the CEO wants gone today). Always set both.

Severity — impact only

LevelDefinitionExamples
S1 — CriticalProduction-down, data loss or corruption, security exposure, or a release blocker that affects all users.Auth bypass; 500 on every request; rows persisted with wrong tenant; secret leaked in response.
S2 — MajorSignificant feature broken or contract violated; workaround exists but is costly or partial.POST accepts invalid data and persists it; pagination loses pages; idempotency replay creates duplicates; endpoint returns 500 instead of 422.
S3 — MinorFunctional defect with a low impact or a clean workaround.Wrong field label; sidebar group does not auto-open; tooltip text outdated; rare flaky behavior.
S4 — TrivialCosmetic / wording issue; no functional impact.Typo; small spacing inconsistency; missing trailing period.

Priority — when to fix

LevelDefinition
P1 — NowStop other work; fix this release. Hotfix-eligible.
P2 — SoonPlan into the current sprint; fix before next release.
P3 — BacklogTracked, scheduled within the next two sprints.
P4 — SomedayTracked, no commitment.

Worked examples

  • POST /user persists a user without an email. Severity S2 (data integrity), Priority P2 (visible to a few in current testing).
  • Auth header parsing throws 500 instead of returning 401. Severity S2, Priority P1 (visible to all integrators on a basic mistake).
  • Sidebar group does not auto-open on a deep-linked page. Severity S3, Priority P3.
  • Typo on the home hero ("Documenttation"). Severity S4, Priority P3.
  • SQL migration drops a non-empty column when re-applied. Severity S1, Priority P1.
