Reference QA Checklist

Regression

A reusable template for any wide change set (refactor, dependency bump, infra swap) where smoke alone is not enough. Instantiate one run per change set. Broader than smoke, shallower than feature-level acceptance.

How to use this template

  1. One run per wide change set (a refactor, a dep bump, an infra swap).
  2. Copy this page to audits/<YYYY-MM-DD>-<change-slug>/regression.html, or paste the tables into the change ticket.
  3. Fill the Run block, then walk every table top-to-bottom and fill Status / Evidence / Notes.
  4. Apply the Pass / fail rules and append any newly-closed defect to Defect-prone areas.

When to run

  • Major refactor merged (touches > 1 module).
  • Dependency upgrade across the stack (FastAPI / Pydantic / SQLAlchemy / Alembic).
  • Container or runtime image change.
  • Database engine, indexing, or migration change.
  • Docs portal frontend change touching shared assets (sidebar, theme, runtime bundle).

Pass / fail rules

  • Pass: all rows pass; new defects (if any) are S3 or below.
  • Fail: any S1/S2 found; or any defect-prone area regresses; or contract drift on the OpenAPI baseline.
  • Add to defect-prone areas any newly closed defect that surfaced through this run.
