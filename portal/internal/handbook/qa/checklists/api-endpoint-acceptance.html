API endpoint acceptance

A reusable template for any new or changed HTTP endpoint. Instantiate one run per endpoint (e.g. POST /v1/conspectus, PATCH /v1/users/{id}) against the operation's spec under internal/services/api/reference/<resource>/. Mark each row Pass / Fail / N/A; attach evidence per the API endpoint testing playbook.

How to use this template

  1. Pick one endpoint (method + path + version) — one endpoint = one run.
  2. Copy this page to audits/<YYYY-MM-DD>-<endpoint-slug>/api-endpoint-acceptance.html, or paste the tables below into the ticket comment.
  3. Fill the Run block, then walk each table top-to-bottom and fill Status / Evidence / Notes.
  4. Apply the Pass / fail rules and paste the verdict into the ticket.

Preconditions

The checklist assumes the spec and the endpoint are both available before any row is scored.

  • A spec page exists for the endpoint and passes the artefact DoD.
  • The API is reachable on a test environment and the build SHA is known.
  • A valid bearer token is available for an account with the required role.
  • An HTTP client (curl, httpie, Postman) is configured and capturing requests.

Evidence required for sign-off

  • Request/response capture for one happy path, one validation error, one auth error.
  • One X-Request-ID traced in the structured log.
  • Screenshot of Swagger UI showing the operation expanded.
  • Result line per row above (Pass / Fail / Needs follow-up), pasted into the ticket.

Pass / fail rules

  • Pass: all rows above pass; evidence attached.
  • Fail: any contract, error envelope, auth, idempotency, or observability row fails.
  • Needs follow-up: non-blocking wording issues with a tracked ticket.
