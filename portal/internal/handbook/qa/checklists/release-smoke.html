Release smoke

The broad shallow check before tagging a release. Use this as a reusable template — instantiate one run per release candidate, against any service (api, portal, …). Deeper walkthrough lives in the release smoke playbook; paste the result into the release ticket.

How to use this template

  1. One run per release candidate (per service if releases are split).
  2. Copy this page to audits/<YYYY-MM-DD>-<service>-<build-sha>/release-smoke.html, or paste the tables into the release ticket.
  3. Fill the Run block, then walk every table top-to-bottom and fill Status / Evidence / Notes.
  4. Apply the Sign-off rules and paste the verdict.

Preconditions

The checklist assumes the release build is deployable and the target environment is prepared.

  • CI is green on the release commit; the image is built and tagged.
  • Migrations (if any) have been applied to the target environment.
  • A release ticket exists with the build SHA and target environment recorded.
  • Auth token, base URL, and a clean test account are available to the runner.

Sign-off

  • Go: all rows pass; defects (if any) below S2.
  • Conditional go: S2 defects with explicit acceptance from the release owner.
  • No-go: any S1 open or any row in Health, API golden paths, or Observability failed.

Paste the sign-off note from the release smoke playbook into the release ticket with evidence.
