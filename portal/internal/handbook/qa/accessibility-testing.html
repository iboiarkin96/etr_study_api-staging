QA How-to

Run an accessibility audit on a page

Target: WCAG 2.1 Level AA. This playbook is how to verify that target on a docs page or portal feature. Line-by-line outcomes live in the accessibility checklist.

Tools

  • Browser DevTools — Lighthouse accessibility audit, color picker for contrast.
  • axe DevTools (extension) — automated rule scan; not a replacement for manual checks.
  • Keyboard only. Disconnect or ignore the mouse for the session.
  • Screen reader. macOS VoiceOver (Cmd+F5) or NVDA on Windows.
  • Zoom. Browser zoom to 200% and 400%; test reflow.

Keyboard reachability

  • Every interactive control reachable by Tab. No control reachable only by mouse hover.
  • Tab order matches reading order; Shift+Tab reverses cleanly.
  • Buttons activate on Space and Enter. Links activate on Enter only.
  • Escape closes any modal, drawer, lightbox, or popover.
  • Inside the search panel, ↑ / ↓ move through results, Enter follows, Esc closes.
  • Skip-link (if present) is the first Tab stop and visibly focuses on activation.

Focus management

  • Focus indicator is visible — at least 3:1 contrast against adjacent colors and at least 2 CSS pixels thick.
  • Modal opens → focus moves into the modal. Modal closes → focus returns to the launcher.
  • No focus trap on regular pages. Focus is trapped in modals and drawers only while open.
  • Programmatic focus moves announce sensibly (e.g. an error summary that receives focus reads its heading).

Contrast

  • Normal body text: contrast ≥ 4.5:1 in both light and dark themes.
  • Large text (18.66px bold or 24px regular): ≥ 3:1.
  • Non-text UI components (form borders, focus rings, icons that convey state): ≥ 3:1.
  • Links are distinguishable from body text without relying on color alone (underline, weight, or persistent indicator).
  • Disabled controls do not need to meet contrast, but should be unambiguously disabled.

Semantics & structure

  • One <h1> per page.
  • Heading levels descend without skipping (h1 → h2 → h3).
  • Landmarks present: <header> / <nav> / <main> / <footer>, each with appropriate aria-label if multiple.
  • Lists use <ul> / <ol>, not divs.
  • Tables have <th scope> for header cells.
  • Form fields have associated <label> elements.

ARIA — only when native semantics fall short

  • Native HTML wins. Don't add role="button" to a <div> when you can just use <button>.
  • State attributes (aria-expanded, aria-current, aria-pressed) update when the state changes.
  • Modals: role="dialog", aria-modal="true", labelled by a heading via aria-labelledby.
  • Live regions (aria-live) used for transient announcements (form errors, search progress); avoid spam.

Screen reader pass

  • Page title (browser tab) communicates page identity and product.
  • Landmarks list shows expected regions (banner, navigation, main, contentinfo).
  • Headings list reads as a useful outline.
  • Read in order — does the page make sense top-to-bottom?
  • Activate every interactive control once with the screen reader running; verify the announced state.

Motion & animation

  • Honour prefers-reduced-motion: reduce: animations stop or shorten significantly.
  • No content flashes more than 3 times per second.
  • Auto-playing motion (carousels, parallax) has a pause control.
  • Hero / WebGL effects respect the reduced-motion media query.

Verify

  • Lighthouse Accessibility score ≥ 90 on the audited page (Chrome DevTools → Lighthouse → Accessibility category).
  • axe DevTools scan reports zero «critical» or «serious» violations; remaining «moderate» / «minor» tracked with owners.
  • Every section above (Keyboard / Focus / Contrast / Semantics / ARIA / Screen reader / Motion) walked manually with a row-by-row PASS / FAIL recorded; no row left as «?».
  • Audited page renders cleanly at 200 % and 400 % browser zoom with no horizontal scroll on content.
  • Findings filed against the team backlog with severity per the team's defect rubric — no defects discovered live and forgotten.
