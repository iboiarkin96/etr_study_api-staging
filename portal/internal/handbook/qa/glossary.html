QA glossary

Definitions used across the QA portal. When two terms are commonly confused, the entry says explicitly which one we use here. Aligned with ISTQB terminology unless otherwise noted.

A–C

Acceptance criteria
Conditions a feature must meet to be accepted by the product owner. Written in the spec or ticket.
Acceptance test
A test confirming acceptance criteria are met. Often runs at the contract or system layer.
Accessibility
Whether the product is usable by people with disabilities. We target WCAG 2.1 AA.
Bug vs Defect
We treat them as synonyms. Defect is preferred in tickets; bug is fine in conversation.
Charter (exploratory)
A one-line mission for an exploratory session: "Explore X with Y, looking for Z, in T minutes".
Checklist
An ordered list of pass/fail steps for a class of testing. Runs as a script.
Contract test
A test that asserts the public surface (HTTP API / OpenAPI document) matches the documented contract.

D–F

Defect lifecycle
The state machine a defect moves through. See defect lifecycle.
Definition of Done
A checklist an artefact must satisfy to be considered complete (applies to API specs, ADRs, runbooks, postmortems). Rationale and conventions: Why a Definition of Done.
Edge case
An input or condition at the boundary of the valid range — empty, max length, negative, off-by-one, last-day-of-month, etc.
End-to-end (E2E)
A test exercising a full user flow across multiple components. Sits at the top of the pyramid.
Entry / exit criteria
Preconditions to start a test phase, and postconditions to declare it done.
Equivalence partition
A set of inputs the system should treat the same way; one example per partition is enough.
Evidence
Screenshot, log line, or capture proving a test step passed or failed. Required for sign-off.
Exploratory testing
Time-boxed, charter-driven testing where design and execution are simultaneous. Not "unscripted" — chartered.
Flaky test
A test that passes and fails on the same code without changes. We treat flakes as defects against the test.

G–L

Golden path
The most common, expected use of a feature. Always covered by automated tests.
Heuristic
A rule-of-thumb used as a test oracle when no formal spec exists. We use HICCUPPS-F.
Idempotency
Repeating a request yields the same result. See idempotency.
Integration test
A test exercising a component with at least one real adapter (DB, file, network).

M–P

Oracle
The source of truth a test compares against. See strategy: oracles.
Priority
How soon to fix. Independent of severity. See matrix.
Pass / fail / needs follow-up
The three outcomes of a checklist row. Needs follow-up means works but with a non-blocking concern.

Q–S

Regression
A previously working behavior that broke. A regression test is the test that would have caught it.
Risk-based testing
Allocating coverage by impact × likelihood, not by feature count.
Sanity testing
A narrow check that a specific area still works after a focused change. Smaller than smoke.
Severity
How bad the impact of a defect is, independent of priority. See matrix.
Smoke testing
A broad shallow check that the build is "live and lit": health, golden paths, key UI.
Sign-off
QA's recorded recommendation on a release: Go / Conditional / No-go.

T–Z

Test case
A documented scenario with steps, inputs, and expected results. See template.
Test data
The fixtures or seed values needed by a test. Belongs in the test, not in shared databases.
Test plan
The "what we will test" document for a feature or release. See template.
Traceability
The link from a requirement → test case → defect. Achieved here by referencing spec anchors.
Triage
Classifying defects after intake — severity, priority, owner.
Unit test
A test for one function/class with no I/O. Bottom of the pyramid.
UAT (User acceptance testing)
Acceptance tests run by or on behalf of end users. Internal sign-off for this project; we do not maintain a separate UAT environment.
