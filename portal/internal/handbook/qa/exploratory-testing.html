QA How-to

Run an exploratory testing session

Time-boxed, charter-driven sessions to find issues that scripted tests miss. Exploratory testing is not "unscripted" or "ad-hoc" — it has a charter, a budget, and a report. It is the highest-yield activity per minute for finding novel defects.

Charter — the one-sentence mission

Pattern: "Explore <target> with <resources>, looking for <risk>, in <duration>."

  • Explore POST /api/v1/conspectuses with realistic and adversarial bodies, looking for validation gaps and 5xx leaks, in 60 minutes.
  • Explore docs portal search with multilingual queries and edge whitespace, looking for ranking and crash bugs, in 30 minutes.
  • Explore the docs sidebar drawer on iOS Safari, looking for focus traps and orientation-change bugs, in 30 minutes.

Heuristics — what to test against when there's no spec

HICCUPPS-F:

  • History — does it work like the prior version?
  • Image — does it look right (style guide, brand)?
  • Comparable products — does our docs portal behave like a known good one?
  • Claims — does it match our docs / spec / changelog?
  • User expectations — would a reasonable user be surprised?
  • Product — internal consistency (e.g. all pages should have a History block).
  • Purpose — does it serve the goal stated in the spec?
  • Statutes — legal / accessibility (WCAG 2.1 AA).
  • Familiarity — does it behave like other parts of this app?

Tour types — pick a lens for the session

  • Money tour. Walk every step that a paying user would walk. (Here: every documented golden path in the API.)
  • Bad-data tour. Push every input field with garbage — empty, oversize, unicode, control characters, scripts.
  • Configuration tour. Vary settings — theme, viewport size, locale, slow network, throttled CPU.
  • Interruption tour. Start something, refresh / navigate away / lose connection mid-flow.
  • History tour. Replay defect-fix history — every regression class we have seen.
  • Land of the lost tour. Click the rarely-clicked: deep nav, footer links, error states, edge routes.

Session shape

  1. Setup (5 min). Read the charter, prepare environment, open notebook.
  2. Setup → execution split. Note the time you start probing in earnest.
  3. Test design + execution (45–90 min). Alternate: think → try → observe → take note. Don't pause to file defects mid-flow — capture and move on.
  4. Wrap-up (10 min). File defects from your notes, write the report.
  5. Cool-down. Sessions are intense; budget 15 minutes of recovery before the next.

One session is one charter. If you find yourself drifting, end the session, write the report, start a new charter for the new lead.

Session report template

- Charter: <text>
- Tester: <name>
- Date: <ISO>
- Duration: <mins>  (setup <mins> / test design <mins> / bug investigation <mins>)
- Build / env: <sha + env>
- Areas covered: <list>
- Areas not covered (out of charter): <list>

- DEF-002 <short title> — link

- <observations that did not yet become defects>

### Follow-up charters proposed
- <new charters surfaced from this session>

Charters to copy — a starter library

  • Bad-data tour against POST /user — empty, oversize, unicode, mixed whitespace, scripted input.
  • Interruption tour on conspectus review — refresh / navigate / disconnect mid-PATCH.
  • Configuration tour on docs portal — light/dark, 360px / 1280px, throttled 3G.
  • Land-of-the-lost tour on rarely-followed sidebar branches — Audit, RFC drafts, runbooks.
  • Search behaviour with non-Latin queries, very long queries, leading/trailing spaces, special chars.
  • Auth boundary cases — expired token, malformed token, missing scheme, both query+header tokens.
  • Pagination edges — limit=0, limit beyond max, malformed cursor, cursor reused across resources.
  • Idempotency edges — same key + same body twice, same key + different body, key after window expires.

Verify

  • Session report exists in the team's report store with all template sections filled (charter, duration breakdown, build/env, areas covered + not covered, findings, notes, follow-up charters).
  • Every finding linked to a filed ticket — no defect left only in session notes.
  • Duration budget respected (session ended at or before the charter's stated minutes); cool-down recorded.
  • «Areas not covered» line is filled — explicitly naming what was left for future charters, not silently skipped.
  • Follow-up charters routed back into the charter backlog so leads surfaced today don't vanish.
