QA How-to

Verify a docs page before merge

How to verify a docs page is correct, navigable, accessible, and consistent with the portal's design system. Line-by-line outcomes are in the documentation visual checklist and the accessibility checklist.

What to verify on any docs page

Walk the page from the top down and check, in priority order:

  1. Correctness — does it tell the truth about the system?
  2. Structure — does it follow the page contract?
  3. Navigation — can a reader move around (sidebar, chips, anchors, drawer)?
  4. Search — is the page indexed and findable?
  5. Themes & responsiveness — does it hold up in light, dark, and on small screens?
  6. Accessibility — keyboard, focus, contrast, semantics.
  7. Assets & links — no 404s, no console errors.

Each section below covers one of these. Use the visual checklist for the line-by-line pass.

Correctness checks

  • Statements about endpoints, schemas, or behavior match the spec page in services/portal/internal/services/api/reference/<resource>/ and the OpenAPI baseline.
  • Code samples compile / run as written. Copy them out and try.
  • Numbers and dates make sense (no "as of 2024" on a 2026 page).
  • Diagrams are PlantUML (per ADR 0020) and render in the lightbox without overflow.

Structural contract

Each page should have:

  • <article class="docs-prose"> wrapping the body, with <header class="page-head"> at the top.
  • <header class="page-head"> with a chip row, h1, and p.lede.
  • <h2 id="…"> for each major block — TOC links target these.
  • Tables wrapped in .docs-table-wrap with .docs-table--sticky-head --zebra modifiers.
  • Code blocks as <pre class="docs-code"><code class="language-…">.
  • <footer class="docs-history"> with at least one row.
  • Right-rail <aside class="docs-toc"> + <button data-component="toc-fab"> for mobile.

For API spec pages, see also the DoD for the full required-section list.

Navigation checks

  • Internal sidebar (data-component="sidebar") shows the active page highlighted; surrounding group is open.
  • Breadcrumbs in the topbar match the page's path in the docs tree.
  • Right-rail "On this page" anchors all scroll to live targets.
  • TOC FAB drawer (mobile / narrow) opens, closes on Escape.
  • External links use rel="noopener" and don't break navigation.

Search

  • Search-index entry for the new page exists (rebuild the index if needed).
  • Searching for the page title returns it within the first 5 results.
  • Search results panel opens, navigates with arrow keys, closes on Escape, opens result on Enter.
  • Empty query state and no-results state both render without console errors.

Theme & responsiveness

  • Page renders correctly in light, dark, and system-default modes.
  • Theme toggle cycles system → light → dark with a tooltip and aria-label that match.
  • No element relies on a hard-coded color; tokens come through. Spot-check critical text contrast.
  • At ≤760px (mobile), 761–1024px (tablet), ≥1025px (desktop): no horizontal scroll, no clipped text, tables scroll their own region.

Accessibility (quick pass)

  • Tab through the page — focus order matches reading order; focus rings are visible.
  • Headings are hierarchical (one h1, h2 below it, h3 inside h2).
  • Images have meaningful alt text or empty alt for decorative.
  • Diagram lightbox is reachable by keyboard and closable with Escape.
  • Use the accessibility checklist for a deeper pass.

Assets & links

  • All hrefs resolve (no 404). Run a link checker if available.
  • Relative paths are correct for the page's depth (one level deeper = ../).
  • JS/CSS asset paths match — nothing in the console.
  • The inline theme-bootstrap script is present and not blocked by CSP.

Verify

  • make docs-check exits 0 with Docs check passed (no drift).
  • Every row of the documentation visual checklist recorded as PASS / FAIL — no «?».
  • Browser DevTools console is empty on the audited page (no asset 404, no JS error, no React-style hydration warning).
  • Page found via the portal search box on the first 5 results for its title.
  • All hrefs validated by a link-checker pass — zero broken external links, zero broken internal anchors.
  • Findings filed against the team backlog with one owner each.
