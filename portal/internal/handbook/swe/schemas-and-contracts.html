SWE Explanation

Schemas as the public contract

Why the shape of data at the edge of the system is a promise to every caller — and why every change to that shape has to be reasoned about as a contract change, not as a code change.

A schema is a boundary, not a data class

Inside the application, a Pydantic model and an ORM entity and a domain dataclass are all just Python objects. From the outside, the request and response schemas are something different in kind: they are the only surface a caller can see. Everything the caller knows about our system — what fields exist, what types they have, what errors mean — comes from those shapes.

That changes how a schema must be reasoned about. A refactor inside the service layer can move things around freely; nobody else is looking. A refactor of a schema is a change to a public interface, and every existing caller is, by silent assumption, holding the previous version. The two changes look identical in the diff. They are not the same kind of change.

Why “additive by default” is the rule

The simplest way to keep a contract trustworthy is to forbid the changes that quietly break it. Three categories of change have very different blast radii.

  • Adding an optional field is safe: an old client that does not know the field ignores it; a new client that knows it can read it. The contract grows, no caller is forced to change.
  • Renaming or removing a field is breaking: any client reading the old name now gets nothing. The change must be coordinated, versioned, or staged through a deprecation window.
  • Reinterpreting a field — same name, new meaning — is the most dangerous: nothing in the schema signals that anything changed, yet callers silently misinterpret responses. This is the failure mode behind the rule that an error code may never be reused for a new meaning.

“Additive by default” is the policy that pushes the safe case to the front. It is not that breaking changes are forbidden — sometimes they are necessary — but that they require a version bump, a migration story, and an ADR. The cost is on the change, not on the callers.

Three kinds of schema, three reasons they exist

The codebase carries three families of schema, and conflating them is a recurring cause of bugs.

  1. Request & response The wire format. Reviewed by every caller; bound to OpenAPI; the most carefully versioned.
  2. Error envelope A second, parallel contract for the failure path. Its code and key values are part of the public interface.
  3. Internal types Domain dataclasses, ORM entities, intermediate transfer objects. Private; free to change without coordination.

The first two carry the contract obligations. The third does not. The discipline is to keep the boundary between them sharp: a service should not return its ORM entity directly through the router, because the moment it does, every database column becomes part of the public schema.

Why errors are a contract, not an afterthought

Clients write code against success responses. They also write code — branchier code — against failures: retries, fallbacks, user-facing messages. An error envelope that changes its code value without notice breaks every one of those branches in a way that is hard to detect, because tests on the client side usually exercise success paths and a small set of expected failures.

This is why the error code and key semantics are immutable within a major version: clients are entitled to assume that code: "duplicate_key" means today what it meant yesterday. New error situations get new codes; existing codes do not silently widen. The error contract governance ADR describes the consequences of getting this wrong.

Idempotency as part of the contract

For write endpoints, the schema is not the whole contract: the rules around retrying a write are too. A client that retries a request after a network blip needs to know whether the second call will create a second resource, return the original result, or fail.

Declaring an idempotency contract in the spec turns this from “depends on the implementation” into a property the client can rely on. The contract has three branches that have to be answered together: same key with the same body returns the original result; same key with a different body returns a conflict; missing key when one is required is a validation error. Specifying only the first branch silently leaves the others undefined, and undefined behaviour is the most reliable bug factory there is.

Tradeoffs

Treating the contract as an append-only artefact buys consumers stability — and costs the maintainer flexibility every time the truth changes.

Win

Existing clients never break silently

Additive-only means a field rename is a new version with a sunset window, not a Tuesday accident. Integrators can pin and forget.

Cost

Mistakes live in the contract for a year

A poorly named field is locked in for the version's life. We pay for haste up front; consumers pay if we get the name wrong.

Win

CI catches drift before review

The baseline diff in make verify turns every accidental contract change into a PR comment, not a production incident.

Cost

Intentional changes need an explicit accept step

Every legitimate contract change requires make openapi-accept-changes and a written rationale. Friction is the point — and the toll.

Common failure modes

  • Returning ORM entities directly. Every column becomes a public field; adding a column is now a contract change.
  • Reusing an error code for a new meaning. Looks like a one-line change; breaks every client that branches on the code.
  • Examples that drift from schemas. The schema says one thing, the OpenAPI example says another; clients believe the example. Examples must be generated from or asserted against the schema.
  • Implicit idempotency. A write that happens to be idempotent in the current implementation, and is documented nowhere. The next refactor breaks every retrying client.
  • Validation rules in two places. The Pydantic model says “1–200 characters”; the service says “1–500”. Clients see whichever rule fires first; the other one becomes a latent bug.
