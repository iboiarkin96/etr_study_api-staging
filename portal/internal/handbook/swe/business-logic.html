SWE Explanation

Where business logic lives

Why the codebase is split into transport, application, and persistence layers — and why each one must remain ignorant of the others.

What we mean by business logic

“Business logic” is the part of the code that would still be true if the HTTP protocol disappeared tomorrow. The rule that a conspectus belongs to exactly one user, that a review cannot be scheduled before it was created, that duplicate composite keys are forbidden — these are statements about the domain, not about any particular framework. They would survive a rewrite from FastAPI to gRPC, from PostgreSQL to a key-value store, from a synchronous request to a background job.

Everything else — JSON parsing, header validation, SQL dialect, ORM session management — is a delivery detail. The codebase is structured around the difference between these two kinds of code.

Layered architecture, in one diagram

The system stacks in four horizontal layers, and each layer talks only to the one immediately below it. Reads down the stack are allowed; reads up the stack are not.

  1. HTTP / transport Routers, request/response shapes, status codes. Bound to the OpenAPI contract.
  2. Schemas & 422 contract Pydantic models; mapping of validation failures into the canonical error envelope.
  3. Application / services Use cases and domain invariants. The only layer that knows what a “conspectus” is.
  4. Persistence Repositories and the ORM session. Storage details stay local.

The system-wide C4 diagrams live in system design; this page is about the rationale behind the split, not its precise file paths.

Why this shape — ports, adapters, and contract-first

Three ideas converge on this layering.

Layered architecture is the oldest of them: each layer talks only to the one below, and inner layers never import outer concerns. The reward is testability — the application layer can be exercised without spinning up HTTP — and replaceability of the outer adapters.

Ports and adapters (Hexagonal) is the same idea stated differently. The application defines ports in terms of its own vocabulary; FastAPI is one adapter that plugs in at the inbound port, SQLAlchemy is one adapter that plugs in at the outbound port. Either could be swapped without rewriting the domain. We do not adopt the strictest hexagonal vocabulary (we use plain Python interfaces rather than abstract Port classes), but the discipline is the same.

Contract-first means the OpenAPI document defines the HTTP surface before code is written. The router cannot quietly change a field name; the contract is the source of truth, the code is bound to it.

Why the direction of dependency is strict

The rule that the service does not depend on the router is not aesthetic. It is the only way three things stay tractable.

  • Tests. A service that imports fastapi.Request can only be tested by spinning up a request. The unit-test base of the test pyramid collapses.
  • Reuse. The same domain rule needs to run from an HTTP handler, a CLI command, and a background worker. If the rule lives in the router, it has to be copied to each caller.
  • Change. A breaking change to a transport-only concern — say, the error envelope — should not cascade into the domain. The boundary is what isolates it.

The symmetric rule — that the repository does not return HTTP-shaped structures — protects the application from the storage choice. A migration from one SQL dialect to another, or from SQL to a different store, must not require touching service code.

Two kinds of validation, two places they live

A request is checked twice, against two different kinds of rule, and the layering reflects that.

Schema-level checks — “the body has a title field of length 1–200” — are properties of the message format. They depend on nothing but the incoming bytes, so they belong at the transport boundary and run before any application code executes. Pydantic does this; FastAPI returns a 422 if it fails.

Domain-level checks — “no other conspectus with this composite key exists” — are properties of the world. They require database lookups, sometimes cross-aggregate reasoning. They cannot live in a schema model and they must run in the service layer, where the domain is.

Confusing the two leads to a recurring failure mode: validation rules that quietly require a database connection, schemas that drift from the OpenAPI contract, error envelopes that differ between Pydantic failures and domain failures. The error contract governance ADR exists to keep both flavours surfacing the same envelope to clients.

Transactions are an application-layer concern

It is tempting to put transaction boundaries in the repository — open a transaction per write, commit on success. This breaks the moment a use case touches two repositories: an order of operations that should atomically succeed or fail together is split across two independent commits, and a crash in the middle leaves the system in a state no rule allows.

The transaction boundary therefore belongs at the use case — the service method — because only the service knows what set of writes constitute one business event. The repository receives the session; it does not own it.

Tradeoffs

The layering is not free. Every win it buys comes with a recurring cost that has to be paid in every change.

Win

The domain is testable without transport

A service test runs in milliseconds without spinning up FastAPI; the wide base of the test pyramid stays cheap.

Cost

More indirection per request

A trivial CRUD operation passes through router → service → repository. Three files to read instead of one.

Win

Storage is swappable

The SQLite → Postgres migration touches the persistence layer only; services and routers do not change.

Cost

Newcomers don't see the win on day one

The reward shows up the first time the layering saves a refactor — month two, not week one. Until then it feels like overhead.

Common failure modes

  • Logic creep into the router. A request handler grows an if branch that checks a domain rule. A year later, the rule lives in five routers and nowhere else, and a background worker silently breaks it.
  • The anaemic service. Routers do all the work and services are pass-through. The layering is decorative; the testability benefit is gone.
  • Repositories that know about HTTP. A query that returns a JSON-ready dict instead of a domain entity ties the persistence layer to the response shape. Changing either now requires changing both.
  • Validation that needs the database. A Pydantic validator that runs a query. Schemas should not require IO; the moment they do, they cannot be reused, cached, or generated from OpenAPI.
  • Cross-layer exceptions. Domain code raising HTTPException directly couples it to the transport. We accept this pragmatically today, but the principled boundary is a domain exception translated by a single handler at the edge.
