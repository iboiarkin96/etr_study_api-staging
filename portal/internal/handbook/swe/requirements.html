SWE Explanation

How requirements become code

Functional intent, non-functional guarantees, and the chain of artefacts that carries a requirement from product statement to implemented behaviour — and why each link in the chain exists.

Why we formalise requirements at all

A requirement is a promise about behaviour. “The system shall allow a user to schedule a review” is a sentence anyone can read, but two engineers will implement two subtly different things from it unless the sentence is sharpened: what does schedule mean for a review that already exists? What if the user is not the owner? What happens on retry?

The point of writing requirements down is not bureaucracy. It is to push these questions to the surface before the implementation is written, so that the answers are decided once, in one place, by someone who has the context to decide. The alternative — discovering each question during coding — produces inconsistent answers across endpoints and a permanent tail of after-the-fact bug reports.

Functional and non-functional, two different shapes

A functional requirement describes what the system does in observable terms: an endpoint exists, a state transition is allowed, an error code is returned. It is the kind of statement that can be turned into an acceptance test directly.

A non-functional requirement (NFR) describes a property of the system rather than a behaviour: latency under load, observability, maintainability, idempotency under retry. NFRs cannot be tested by a single example; they are properties that hold across a population of executions, and they constrain the implementation rather than describing it.

The two shapes do not compete — they intersect. A functional requirement says “the user can post a conspectus”; the NFRs say that any endpoint, including this one, must use the canonical error envelope, must log a structured event with request_id, must keep its 95th-percentile latency under a stated threshold. The NFRs are the rules of the game, not its moves.

The chain from product statement to merged code

A requirement moves through four artefacts on its way to production. Each one exists because the previous one is not enough.

  1. Product statement Prose, in the language of the domain. Establishes intent and constraints, not implementation.
  2. Operation spec A concrete API contract: request shape, response shape, errors, edge cases, NFR thresholds. Reviewed against the DoD.
  3. Tests Acceptance criteria turned into executable assertions before the code exists.
  4. Implementation Code that satisfies the tests. By construction, satisfying the tests satisfies the spec, and the spec satisfies the product statement.

The discipline is not that each artefact exists — anyone can produce four files. It is that each one is derived from the one above it, and that the derivation is checked. A test that does not trace back to an acceptance criterion is a test about something nobody asked for; a piece of implementation that does not satisfy a test is unverified behaviour that may or may not match intent.

Why traceability matters six months later

Six months after an endpoint ships, a bug report arrives. “The system is returning 200 on this case; it should be 409.” The first question is not what the code should do — it is what was originally asked for. Was 200 a mistake, or was the original spec ambiguous? Was the test always wrong, or did the behaviour regress?

Traceability — the ability to walk backwards from a line of code to the requirement that motivated it — is what makes that question answerable. A bug fix that contradicts the spec is a contract change; a bug fix that aligns with the spec is a defect repair. The two have different consequences for callers, and you cannot tell them apart without the chain.

This is why operation specs carry an explicit Traceability section linking to methodology, system design, and ADRs, and why ACs carry IDs that the test plan references. Without those links the chain breaks, and behaviour silently drifts.

NFRs as platform, not per-endpoint work

Non-functional requirements look like a checklist when read out — performance, reliability, observability, idempotency, environments, contracts — and that framing misleads. Treated as a per-endpoint to-do, they get partially satisfied at best: each developer reinvents what “structured logging” means; each endpoint gets a slightly different error envelope; idempotency works in three places out of seven.

The right framing is that NFRs are mostly satisfied by the platform, not by the endpoint. The error envelope is implemented once in a central handler; the idempotency rule is a shared module; the observability conventions are enforced by middleware. The endpoint's responsibility is to not undo these guarantees and to declare the parameters (latency target, idempotency key composition) the platform cannot infer.

This shifts the cost of NFRs from endpoint-by-endpoint discipline to platform investment, and it is the only way NFRs hold up under growth. Every endpoint that opts out of the platform's mechanism — even for a defensible reason — has to be tracked, because it is a permanent source of inconsistency.

Common failure modes

  • “The code is the spec.” When the implementation is the only source of truth, every behaviour is intentional by definition; bugs become contract changes; the system cannot be reasoned about above the line-of-code level.
  • Acceptance criteria without IDs. A test that asserts some behaviour does not tell you which requirement is broken when it fails. Without IDs, traceability is a wish.
  • NFRs as decoration. A section titled “Non-functional requirements” containing the word performance and nothing measurable. The requirement is whatever the implementation happens to do.
  • Spec drift after the fact. Implementation diverges from the spec; nobody updates the spec; the spec gradually becomes folklore. The next reader cannot tell what is current.
  • Requirements added in code review. A reviewer asks for a behaviour that was not in the spec. The behaviour gets added; the spec does not. Future endpoints inherit the same omission.
