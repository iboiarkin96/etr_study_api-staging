🔎 SA reference · Discoverability

Discoverability — how readers find pages

A page that exists but nobody can find is content waste. The portal is nested up to five folders deep — discoverability is a first-class concern, not «and then there's a search bar somewhere». Four mechanisms stack: the sidebar tree, the search index, in-page cross-links, and the entry hubs. Each one is mechanical, owned, and audited.

Rules

01

The discoverability stack — four layers

A reader lands on the portal one of four ways. Each layer optimises for a different intent.

🗂️

Layer 01 · Browse

Sidebar tree

The reader knows roughly where the answer lives — under Services / Foundations / Governance / Team. The sidebar mirrors the filesystem, grouped by Diátaxis quadrant via catalog-info.yaml.

🔎

Layer 02 · Search

Full-text index

The reader knows what the answer is about but not where it lives. Search hits a Pagefind bundle (BM25 with stemming) rebuilt from every page on every make docs-fix / CI run.

🕸️

Layer 03 · Traverse

In-page cross-links

The reader is on a page and wants the related page. Every relevant card ends with an inline See also: <link> line — reader follows the link without re-searching.

🏠

Layer 04 · Land

Entry hubs

The reader is new. Top-level hubs (index.html, onboarding/, role pages) frame the portal so the reader knows what categories exist.

Rules

02

Sidebar contract — nav-tree-internal.json

The sidebar is rendered from a single JSON file at services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json. Every page that ships also ships its sidebar entry in the same PR.

Shape

{
  "sections": [
    {
      "id": "foundations",
      "label": "Foundations",
      "href": "/services/portal/internal/foundations/index.html",
      "children": [
        {
          "id": "foundations-sa",
          "label": "SA reference",
          "href": "/services/portal/internal/foundations/reference/sa/index.html",
          "children": [ /* deep-dives */ ]
        }
      ]
    }
  ]
}

Ordering & nesting rules

  • Section order matches reader-importance. Foundations & Governance precede Services / Team because they're the «how we work» layer that frames everything else.
  • Nest by topic, not by folder depth. The sidebar's job is to group conceptually; the filesystem can be flat or nested, the sidebar decides what reads as one cluster.
  • Emoji prefix on deep-dives. 🧭 🗣️ 🔁 📐 🎯 🗺️ 🧩 🎨 — same icon on the principle card, the deep-dive page, and the sidebar entry. Pattern recognition is the discovery shortcut.
  • Acronym casing is preserved. API, SRE, AsyncAPI, gRPC, QA, SA, RFC, ADR — never lower-cased in labels.
  • Max nesting depth: 3. Deeper than that, the sidebar becomes unreadable. If a topic needs 4 levels, refactor the IA — see IA contract.

Rules

03

Search — Pagefind bundle, rebuilt on every docs-fix

Search hits a Pagefind bundle (Rust-built static-site index, BM25 with stemming) lazily loaded by the topbar. No backend, no per-query LLM call — fast, deterministic, works offline. Authority: ADR 0033 (supersedes the in-house BM25 of ADR 0027).

  • Builder. tools/docs/build_pagefind_index.py (step [10/10] of make docs-fix) mirrors every HTML page under services/portal/ into a staged tree, injects data-pagefind-filter="visibility:internal|public" and data-pagefind-meta="kind:<taxonomy>" onto each <main>, then runs the pinned Pagefind CLI. The bundle lands in services/frontend/portal/pagefind/ (gitignored — built on every CI run by .github/workflows/deploy-prod.yml).
  • Cadence. Rebuilt on every make docs-fix locally, and on every Pages deploy in CI. The visibility invariant is asserted by tools/docs/check_pagefind_visibility.py right after the build — a public URL must carry visibility:public and an internal URL visibility:internal; build fails otherwise.
  • Ranking signals. BM25 with English stemming, owned by Pagefind — we don't tune ranking. What we control: what to index (a Python pre-pass mirror), how to split internal vs public (the filter), and what each result calls itself (the kind meta).
  • Indexed-page count. Surfaced on the portal home as a live counter — readers see at a glance the portal's footprint.

Apply

04

Cross-link discipline — inline See also inside cards

The third discoverability layer (Traverse) only works if every card actually lists its siblings. Standalone «See also» footer sections are forbidden — they fragment the page and never get scanned. Instead, inline a one-line See also: <link> at the bottom of each relevant card (principle card · tradeoff card · deep-dive card).

  • Inline, not standalone. Pattern: <p class="card__see">See also: <a href="...">🧩 Catalog</a></p> inside the card. The reader hovers a topic, sees the next link at the card's foot.
  • One link per card; max two. Hub-style pages may carry up to two — parent and one sibling. Anything more becomes link-soup; let the sidebar carry the breadth.
  • Link by emoji+title. «🧭 Diátaxis» not «click here». The reader scans for the emoji they know.
  • Pick the parent or one sibling. Card content already covers depth; See-also opens the next sideways hop or surfaces context.
  • Bidirectional pairs. If card A links to B, B's matching card links to A. The graph stays connected; one-way links are dead ends. scripts/audit_front_docs_links.py flags missing reverse links.

Rules

05

Entry hubs — how new readers land

A first-time reader doesn't know the IA — the entry hub teaches them the layout in 30 seconds.

  • internal/index.html — top-level portal home. Card grid: Onboarding / Services / Foundations / Governance / Team. One sentence per card.
  • onboarding/index.html — Day 1 / Week 1 / Month 1 walk-through. The «start here» path for a new team member.
  • team/roles/<role>/index.html — per-role hub. The reader picks their role; the hub frames what's relevant for them (e.g. SRE sees runbooks + on-call + SLOs first).
  • SA-reference foundations/reference/sa/index.html — entry for «how do I write a page?» Three sections: For authors · Lifecycle · For readers.

Rationale

06

Tradeoffs of the discoverability stack

Wins

Static, fast, offline-capable

No backend search. The JSON index ships with the portal; search works on a plane. Average query response is < 50ms in the browser, no network round-trip.

Cost

5-level nesting hurts

internal/foundations/reference/sa/principles.html is five folders deep. URL paths look heavy, breadcrumbs are long. We mitigate with the sidebar (collapses nesting visually) and the search (skips the path entirely), but the URLs are still ugly.

Wins

Emoji-as-shortcut works

A reader who has seen 🧭 on the principle card can spot it in the sidebar, the search results, and the breadcrumbs without reading the text. Pattern recognition trumps reading.

Cost

Sidebar maintenance is manual

The nav-tree JSON is hand-edited. Adding a page = adding a sidebar entry in the same PR. If someone forgets, the page is orphaned (still searchable, but invisible in browse). Reviewers catch this; we don't yet auto-generate the tree.

Wins

Bidirectional cross-links survive refactor

When a page moves, its See-also targets still exist (we don't break links). And when a target moves, the back-link gets updated by scripts/audit_front_docs_links.py in the next CI run.

Cost

No personalisation

The hub is the same for every reader. A new SRE sees the same landing as a returning architect. We accept this — personalisation needs auth and a backend; both add complexity for marginal gain on a small portal.
