🗑️ Process · Deprecation

Deprecation — how we retire pages, APIs, entities

Pages, APIs, and entities don't just stop being relevant — they get retired through a five-stage flow with explicit consumer notification, a sunset date, and a migration target. «Deprecated but kept around» is forbidden by Principle 03: a page is either current or replaced, never silently dead.

Rules

01

What gets deprecated — three flavours

The flow is the same; the timeline and the artefacts differ by kind.

Kind When to deprecate Required artefacts
Page The page is superseded by another, or the underlying capability is removed. Deprecation banner · supersession link · merged into main + git rm in the removal PR.
API endpoint / contract v2 ships; v1 is no longer the canonical contract. Deprecation banner · Sunset / Deprecation HTTP headers · migration guide · ADR if the boundary changed.
Catalog entity (Component / Resource / Library) The thing itself is going away — service decommissioned, library archived. spec.lifecycle: deprecated · banner on every entity page · ADR ratifying the decision · migration target entity link.

Apply

02

The five-stage deprecation flow

From decision-to-deprecate to actual removal. Skipping a stage is a CI failure — the spine check rejects any deprecated entity without a sunset date and migration target.

  1. Decide.

    An ADR ratifies the deprecation. Includes context, alternatives considered (e.g. keep it / soft-deprecate / hard-remove), the chosen sunset date, and the migration target. No deprecation without an ADR.

  2. Mark.

    Flip the entity to spec.lifecycle: deprecated (catalog), add the data-status="deprecated" attribute to pages, return Sunset + Deprecation headers on API endpoints. The deprecation banner renders at the top of every affected page automatically.

  3. Notify consumers.

    Cross-reference the deprecation in every consumer page (use spec.consumedBy to enumerate). Open issues in consumer repos. For APIs, post to the engineering channel one cycle ahead of the sunset date.

  4. Wait out the sunset window.

    The window is per-kind (see § 04 below). During this window the entity is read-only — new consumers are forbidden (check_no_new_deprecated_consumers.py), existing consumers must migrate.

  5. Remove.

    On or after the sunset date, the removal PR git rms the entity files, removes its catalog-info.yaml, and leaves a tombstone redirect at the old URL pointing at the migration target. The ADR's status flips from «Accepted» to «Active» — the decision is now executed.

Rules

03

Required artefacts — the deprecation banner

Every deprecated page carries the canonical banner immediately after the hero. It's a UI Kit component, not a per-page invention — copy the snippet, set the dates, ship. 

<aside class="docs-banner docs-banner--deprecated" role="alert">
  <p class="docs-banner__head">
    🗑️ <strong>Deprecated.</strong> Sunset <time datetime="2026-12-31">2026-12-31</time>.
  </p>
  <p class="docs-banner__body">
    Use <a href="path/to/replacement.html">the v2 replacement</a> instead.
    See <a href="../../../governance/adr/00XX-deprecation.html">ADR 00XX</a>
    for context.
  </p>
</aside>
<!-- Banner styles & icon come from sa-reference.css; no per-page CSS needed -->

The banner is rejected at review if any of the three required fields is missing — sunset date, migration target link, ratifying ADR link. validate_docs_design.py checks that docs-banner--deprecated always carries a <time datetime> child.

Rules

04

Timeline norms — sunset windows by kind

The default sunset window depends on the kind of thing being deprecated and how many consumers it has. Shorter windows need an exception (granted in the ratifying ADR).

Kind Default sunset window Why
Internal page (Tutorial / How-to / Explanation)4 weeksPages have no programmatic consumers; readers find the supersession via search and the banner.
Reference page that's cited from code (e.g. error catalog)12 weeksCode references need time to migrate; we wait one release-train cycle plus a buffer.
API endpoint (internal · ≤ 5 consumers)1 quarterSmall consumer base; we coordinate directly in the engineering channel.
API endpoint (external or wide internal · ≥ 6 consumers)2 quartersWide blast radius; consumers need a release of their own.
Catalog entity (Library)1 quarter after the last consumer migratesLibrary can stay published but archived; we remove the catalog entry only after consumers move off.
Catalog entity (Component / Resource)2 quartersOperational dependency — runbooks, on-call rotations, and dashboards all need re-pointing.

Apply

05

Anti-patterns to avoid

  • «Soft-deprecation forever.» A page sits with a banner for two years and nobody removes it. The sunset date isn't a suggestion — when it passes, the removal PR is overdue. CI flags any deprecated entity past its sunset date.
  • Deprecation without an ADR. An engineer adds a banner because «we should probably stop using this». A deprecation that isn't ratified can be silently reverted; an ADR makes the decision durable.
  • Hard-remove without notification. The migration target is named only in the removal PR description. Consumers find out when their CI breaks. Always run § 03 «Notify consumers» before § 05 «Remove».
  • Tombstone-as-content. The removed page is replaced with a 200-word essay about why it's gone. A tombstone is a single line redirect: «Moved to X». The history lives in the ADR.
  • Banner without a target. «This page is deprecated.» But to what? Every banner names the migration target — no exceptions.

Rationale

06

Tradeoffs

Wins

Nothing rots silently

Every removed page has a date, a reason, and a successor. A reader who lands on an old URL gets a one-line redirect; an engineer who reviews the deprecation can audit the decision via the ADR.

Cost

Removing is a 5-PR effort

ADR PR → mark PR → notify PRs (one per consumer) → wait → remove PR. For a tiny page change this is overkill; we accept the friction because the alternative is rot.

Wins

Sunset windows are predictable

Consumers don't have to negotiate per-deprecation. The kind tells you the window; the ADR documents any exception.

Cost

Bookkeeping for small teams

A 3-person team feels the per-PR overhead more than a 30-person team. We mitigate with a «micro-deprecation» exception in the ADR template (≤ 1 consumer, ≤ 1 week sunset).
