📚 Style guide · Document types

Document types — matrix & templates

Each document has a fixed shape so readers know where to look. Seven canonical types — each with a primary purpose and a non-negotiable «must include» list. Picking the right type at write-time avoids the «is this an ADR or an RFC?» argument three pages in.

Rules

01

The seven document types

One card per type. Each says why the type exists and which sections it must carry. The corresponding scaffold lives in templates/ — copy, never start blank.

Decision

ADR

Records one architectural decision with its rationale. Cited by later code & pages.

Must include: Status · Context · Decision · Consequences · Alternatives · History.

Proposal

RFC

Implementation-level proposal that needs review before becoming an ADR.

Must include: Status · Goals & non-goals · Architecture / flow · Operations & troubleshooting.

Operational

Runbook

Incident triage & recovery procedure. One file per alert / failure mode.

Procedural

How-to

Task-oriented instructions. Reader is doing something, knows the context.

Contract

API spec

Engineering narrative for one endpoint or contract. Sits next to the OpenAPI / proto source.

Assessment

Audit report

Assessment of the docs / architecture / IA against a reference standard.

Navigation

Index / hub

Orients the reader inside a section. The «table of contents» page for a folder.

Must include: Overview card · List or table of pages · One-line descriptions in the same grammar.

02

Picking the right type — quick decision tree

When the type isn't obvious, walk this list top to bottom. The first row that matches is the type.

If you're capturing… …and the artefact is… Type
A decision that's been made and you want to cite laterOne page, normativeADR
A proposal that needs discussion before commitmentOne page, status «proposed»RFC
«When this alert fires, do these steps»Ordered triage listRunbook
«How do I do X?»Procedure, assumes contextHow-to
The contract of an endpoint / message / eventReference page next to the spec sourceAPI spec
An assessment scored against a referenceFindings + mitigationsAudit report
A pointer to several other pages in a folderCatalog / indexIndex / hub

If the page doesn't fit any row, you're either writing two pages at once (split) or you're inventing a new type (don't — pick the closest match and adapt the template).

Rules

03

Canonical ADR sections

Seven required sections, in this order. Skipping or reordering them invalidates the document.

  1. Status. Proposed · Accepted · Superseded · Deprecated. Includes the supersession link when applicable.
  2. Context & problem statement. What forced the decision — constraints, deadlines, prior failures.
  3. Decision. One sentence stating what was decided.
  4. Consequences & trade-offs. What this decision costs us — honest list of downsides.
  5. Alternatives considered. ≥ 2 options that were rejected, with the reason each one lost.
  6. References. Related ADRs, RFCs, external links.
  7. History. Status transitions with dates and authors.

Rules

04

Canonical RFC sections

Six required sections. Heavier than an ADR — an RFC is the «design doc» preceding a decision.

  1. Status. Proposed · Accepted · Ratified · Withdrawn.
  2. Goals & non-goals. What this proposal does, and what it explicitly doesn't.
  3. Architecture or data flow. Diagram + prose. PlantUML, per ADR 0020.
  4. Implementation details. The how — files touched, migration steps, dependencies.
  5. Operational guidance & troubleshooting. What ops needs to know when this ships.
  6. Change management. Rollout plan, rollback plan, owner.

Rules

05

Canonical Runbook sections

Six required sections. Written for someone half-awake at 03:00 — keep prose minimal, ordered steps maximal.

  1. Incident definition & trigger conditions. What alert / symptom triggers this runbook.
  2. Immediate safety checks. Before you touch anything — confirm the blast radius.
  3. Triage checklist. Diagnostic questions that narrow down the cause.
  4. Ordered recovery actions. Numbered steps. Each step has a verification.
  5. Post-recovery validation. How you know the system is back.
  6. Escalation & owner handoff. Who to call if the runbook didn't work.
