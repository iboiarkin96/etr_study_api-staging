🪪 Catalog · Entity card

Entity card — canonical visual template per kind

Every services/<name>/index.html opens with the same visual artefact — an entity card. This page is the normative template for that card: what pills sit in the head, what fields land in the <dl>, what dep-lanes look like, per Backstage convention and adapted for our spine. Use the API service card as the canonical exemplar; this page is the spec the exemplar implements.

Rules

01

Anatomy — the universal shape

Every entity card — System, Component, Library, API, Resource — follows the same five-block skeleton. The kind-specific differences are which pills and fields are required; the shape is constant.

stable FastAPI Python 3.11 Backstage descriptor

<name>

One-sentence lede explaining what this entity is and what role it plays in the System.

Owner
Group (preferred) or User
Lifecycle
experimental · stable · production · deprecated
Provides
What this entity makes available to others (APIs, capabilities, resources)
Depends on
(structured .dep-lanes block, see below)
Repo
Code entry point — for Components
Code reference
Auto-generated reference (pdoc / TypeDoc / etc.) — for Components & Libraries
  • 1Pills row. Lifecycle pill first (colour-coded), then up to 3 tech/spec pills (framework, language, descriptor source).
  • 2Title. The entity's metadata.name — kebab-case, unique within the catalog.
  • 3Lede. One sentence: «<Name> is <role> — the only artefact that <single-sentence purpose>.»
  • 4Fields list (<dl>). Required fields first, optional after. Owner is always row 1; Lifecycle is always row 2.
  • 5Dep-lanes. Structured «Depends on» — grouped chips per category (Framework / Persistence / Schema / Observability / Cache).
  • 6External links. Repo · Code reference · Database tables · Public docs — last block in the card.

02

Lifecycle pills — the four states

The first pill in every card is the lifecycle state. Colour is mechanical — readers learn it once and recognise it on every service.

experimental stable production deprecated

Rules

03

By kind — what each card must / may / must-not carry

Five blocks, one per kind. Each lists required (REQ), optional (OPT) and forbidden (NO) fields, plus an inline rendered example so you see what the card looks like fully filled in.

03.1

🏗System

umbrella · landscape

System cards are thin — Systems don't deploy, so they have no runtime pills, no dep-lanes for Framework / Persistence. They carry the System-of-Components map and product-level provides.

FieldStatusWhat it carries
spec.type REQproduct or platform
Owner REQGroup only (Systems don't have a User owner)
Lifecycle REQtypically stable or production
Provides REQProduct-level capability (one sentence)
spec.hasPart REQList of Components inside this System
spec.dependsOn OPTExternal Systems / SaaS at the boundary
Dep-lanes NOSystems don't run code → no framework / persistence pills
Repo NOSystems span repos; cite individual Component repos instead
On-call NOSystems don't ship code → no rotation
03.2

🧩Component

deployable service

The heaviest card — Components carry runtime pills (framework / language), structured dep-lanes, repo path, code reference, and links to live exemplars. Compare with the api card.

FieldStatusWhat it carries
spec.type REQservice · website · library · tool
Owner REQGroup preferred
Lifecycle REQany of the four states
Provides REQWhat capability this Component exposes (one sentence)
spec.providesApis REQList of API entities this Component implements
spec.consumesApis OPTAPI entities this Component calls
spec.dependsOn REQResources & Components this depends on at runtime / build
Dep-lanes REQStructured: Framework · Persistence · Schema · Observability · Cache
Repo REQEntry point (e.g. services/api/app/main.py)
Code reference REQAuto-gen reference (pdoc / TypeDoc / Sphinx) — link, never inline
Database tables OPTLink to field-level reference under reference/data/
Public docs OPTOpenAPI / SDK / external docs URL if any

Live exemplar: services/api/index.html — the canonical Component card, fully filled in.

03.3

📦Library

reusable code · never deployed alone

Libraries are Components with spec.type: library. The card drops on-call / runbooks fields (forbidden) and adds Consumers + Compatibility policy.

FieldStatusWhat it carries
spec.type REQAlways library
Owner REQGroup
Lifecycle REQany state
Provides REQWhat capability this library provides
Consumers REQComputed from inverse of spec.dependsOn; list of Components
Compatibility policy REQSemVer scope (major/minor) + deprecation window
Changelog REQPer-release notes, link to CHANGELOG.md
Repo + Code reference REQSame as Component
On-call · Runbooks · SLO NOLibraries don't deploy → no operational concerns
03.4

🔌API

cross-service contract

API cards point at the schema file as source of truth. The owning Component is named explicitly; consumers are computed.

FieldStatusWhat it carries
spec.type REQopenapi · asyncapi · grpc · graphql
Owner REQGroup of the providing Component
Lifecycle REQany state
spec.definition REQPath or URL to the schema file (openapi.yaml, schema.proto, etc.)
Provided by REQExactly one Component — the producer (inverse of spec.providesApis)
Consumed by REQList of Components — computed from spec.consumesApis
Availability target REQDelegated to provider's SLO — link, not value
Repo / Code reference NOAPIs are schemas, not code — repo is the producer Component's
Migration guide REQOn every breaking change; vN→vN+1 step-by-step for consumers
03.5

💾Resource

infrastructure · DB · queue · bucket

Resource cards add Backup / Recovery / Access fields. They drop Provides/consumesApis (Resources are consumed, not API providers).

FieldStatusWhat it carries
spec.type REQdatabase · queue · bucket · cache
Owner REQGroup (SRE rotation typically)
Lifecycle REQany state
Consumed by REQList of Components that dependsOn this Resource
Schema REQDB / storage schema link (auto-rendered from migrations)
Backup & recovery REQRPO / RTO + last drill date + restore runbook link
Access policy REQRoles, secrets, IAM bindings — audited quarterly
On-call REQWho pages on Resource alerts
Provides / consumesApis NOResources don't provide APIs — they're consumed via libraries
Diátaxis quadrants NOResources serve ops, not application developers

04

Backstage → our card — field mapping

The card visual is ours, but the field semantics are Backstage's. This table lets a reader who knows Backstage read our cards (and vice-versa) without translation.

Backstage YAML fieldOur card rowNotes
metadata.name Card titlekebab-case; unique in catalog
metadata.description Card ledeone sentence, no marketing
spec.type Pill (second slot) + table «spec.type»kind-specific values
spec.lifecycle Pill (first slot)4 states; colour-coded
spec.owner «Owner» fieldGroup preferred
spec.system «Part of» (implicit from breadcrumb)tied to the parent System
spec.providesApis «Provides» listlinks to API cards
spec.consumesApis Dep-lanes · API rowlinks to API cards
spec.dependsOn Dep-lanes (structured)grouped by lane (Framework / Persistence / …)
metadata.tags Pills 3+ (tech keywords)free-form; searchable
metadata.annotations Not shown on card (used by tooling)github.com/project-slug · pagerduty/integration-key
metadata.links External-links block (bottom)Repo · Code reference · Public docs

Apply

05

Anti-patterns to avoid

  • Lifecycle pill in the wrong colour. Reviewers immediately catch a deprecated service rendered with the «stable» green pill. Colour is mechanical — never per-page taste.
  • «Production-ready» / «GA» as the lifecycle. Marketing words. Use one of the four canonical states.
  • Long lede. The lede is one sentence. If it needs three sentences, put them in the body of the page below the card.
  • Dep-lanes as a flat list. Dependencies grouped into one chip row defeat the lane pattern. Group by purpose (Framework / Persistence / Schema / etc.) — reader skims by lane, not by chip.
  • Inline code reference. Code reference is always a link (to pdoc / TypeDoc output). Never paste generated docs into the card.
  • Skipping Owner. «We'll figure it out later» — never. Owner is required at create time. If no Group exists, create the Group first.
  • Component card with a runtime pill but no dep-lanes. If the entity runs, it depends on something — fill the lanes. An empty Depends-on is a smell.
