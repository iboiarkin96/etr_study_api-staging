🎨 Principle 08 · Design

Visual contract — why uniformity makes a portal

A great portal is great because it feels like one product, not 487 documents loosely glued together. That feeling comes from a strict visual contract — same components, same icons, same page shape — and from never starting a page from a blank file. This page covers what the contract is, why it matters, what it covers, how we enforce it, and what it costs.

Stance

01

The claim

One visual language. One component kit. One template per document type. A reader who sees a card on page A and the same card on page B has learned that pattern once; they can now read page C without a manual.

Documentation is read under bad conditions — on-call at 03:00, mid-sprint with five tabs open, in a language that isn't the reader's first. Every cognitive cycle the reader spends figuring out «is this card the same as the other card?» is a cycle they're not spending on the content. The visual contract shifts that cost from the reader (every visit, forever) to the authoring stack (once, at template-time).

Rationale

02

Why a uniform visual contract is necessary

Four reasons — each one is enough on its own; together they're non-negotiable.

🧠

01 · Cognition

Recognition beats recall

A reader who has seen the «tradeoff card» on Diátaxis recognises it instantly on Catalog. They don't have to learn the visual a second time. Recall (looking up «what does a yellow border mean here?») is ~5× slower than recognition; over 487 pages, that compounds.

🤝

02 · Trust

Consistency signals maintenance

When every page has the same hero, the same section kickers, the same footer, the reader infers «this portal is curated». When pages look hand-rolled and each one is unique, the reader infers «this is stale or abandoned» — even if the content is fresh.

⚡

03 · Authoring speed

Templates eliminate the blank-page tax

The hardest part of writing isn't writing — it's deciding the shape. Templates make that decision for you: copy, fill in, ship. We measure: authoring time drops ~40% when an author starts from a template versus a blank file.

🛡️

04 · Reviewability

Visual drift is a review signal

A PR whose page doesn't match the kit is immediately spottable in review — different fonts, missing kicker, no history footer. The reviewer doesn't need to read the page to know it's broken structurally. The kit is the fast-fail gate.

Rules

03

What the contract covers — three layers

From tokens at the bottom (colours, spacing, type) up to templates at the top (whole-document shapes). Every layer is the source of truth — never reinvented per page.

Layer 01 · Tokens

Colours · spacing · typography

The atoms. Defined once in assets_v2/ui-kit/tokens/ and consumed by every component.

  • --accent · --info · --success · --warn · --rose · --teal
  • --space-1…7 spacing scale
  • --fs-100…600 font-size scale
  • --radius-xs/sm/md/lg/full corners
  • Light + dark theme via the same tokens — never hand-rolled hex codes

Layer 02 · Components

Hero · section · card · table · …

The molecules. Every UI shape used on more than one page lives in assets_v2/ui-kit/components/.

  • page-hero / numbered-section — page shell & numbered sections (the same kicker pattern you see on every SA-reference page)
  • docs-table / docs-table--zebra — every comparison table
  • docs-status-timeline — the git-flow timeline on Process
  • docs-concept-layers — the nested-matryoshka pattern
  • footer-history · docs-toc · topbar · breadcrumbs

Layer 03 · Templates

One scaffold per document type

The whole-document shapes. templates/ holds 17 scaffolds — one per type.

  • Decision: adr.html · rfc.html
  • Operational: runbook.html · postmortem.html
  • Reference: api-spec.html · component-spec.html · screen-spec.html · service-descriptor.html
  • Diátaxis: tutorial.html · how-to.html
  • Audit: audit.html · bug-report.html · test-case.html · test-plan.html

Rationale

04

Without the contract vs with it

The same hypothetical page, written by two engineers a month apart. The first reaches for whatever tags & colours they remember; the second copies a template. Compare the result.

Without contract — drift

Hand-rolled page

Author A reaches for whatever shapes feel right today.

  • Custom hex colours: #0a66c2 in light, hardcoded; dark theme broken
  • New heading hierarchy: h2 as «section», then a screenshot, then a new h2 with different style
  • Page-history block missing — reviewer has to file a follow-up PR
  • Different table CSS than every other page
  • Reader: «is this the same portal? did someone get the design wrong?»

With contract — kit + template

Templated page

Author B copies screen-spec.html and fills it in.

  • v3 tokens; --accent ⇒ correct in light + dark automatically
  • Same .page-hero + .numbered-section__head + .numbered-section__kicker as every sibling page
  • Page-history block is in the template — can't forget it
  • docs-table + docs-table--zebra classes — same look as every comparison table
  • Reader: «I know how to read this; let me focus on the content»

Apply

05

How we enforce it

The contract isn't a memo on a wiki — it's a build-time gate plus a small set of habits.

  • Single CSS entry. Every page links one file: assets_v2/runtime/internal/entry.css. That file @imports every kit component. A page that doesn't link it can't render correctly; a CSS rule that isn't in the kit doesn't exist for that page.
  • No inline CSS over ~50 lines. If a page invents enough CSS to need a kit component, the component goes in the kit (see sa-reference.css for example) and the inline block disappears. Reviewers flag any page with a large <style> block.
  • Templates are the only blank page. Authoring a page from scratch is rejected at review — copy from templates/, adapt, ship. The «start from scratch» PR loses 30 minutes of review time on a shape debate that the template would have settled.
  • Token-only colours. No hex codes in page <style> blocks. Reviewers ask «does this work in dark mode?» — if you used color: #1f7a3b, the answer is no; if you used var(--success), the answer is yes-by-construction.
  • CI checks. check-h1-structure · check-asset-refs · validate-docs-a11y · check-css-vars — the contract has automated teeth, not just human reviewers.

Rationale

06

Tradeoffs & honest costs

What the visual contract costs us. The architecture lead will ask «is this just over-engineering for a small team?» — the honest answer below.

Wins

Compounding reading speed

Pattern recognition lets a reader skim a 500-line page in a minute because the shape is familiar. Without the contract, the same skim takes 3-5 minutes — the reader has to parse the shape before reading the content.

Cost

Components have to be built first

Every new pattern is a kit component before it appears on a page. Adding .feature-card took a 1-hour design + a 30-minute extraction PR — friction the first time, free every time after.

Wins

Dark theme is free

Token-only colour rules mean dark theme is one CSS variable swap, applied across the whole portal. Pages with hex codes need per-page dark-theme fixes — multiplied by 487 pages, that's a quarter of work we avoid.

Cost

Less freedom per page

An author who has a «cool idea» for a unique visual treatment can't ship it inline — they have to upgrade the kit first. Some ideas die in friction; that's the cost. The kit-first rule trades creative locality for portal-wide coherence.

Wins

Onboarding is cheaper

A new author who has read one templated page can author the next page from a template without re-learning the system. Senior-author review time drops on each PR.

Cost

The kit needs an owner

The visual contract is only as good as the kit. Someone has to maintain sa-reference.css, retire deprecated patterns, version-bump tokens. For a small team that's a fractional role; on a tiny team it's overhead we accept consciously.
