Authoring Runbook On-call

Add a runbook

A runbook is the page an on-call engineer reads at 3 am. One known failure mode per file. Lives under how-to/incidents/runbooks/.

When to write one

  • You hit a failure mode that took non-trivial detective work to resolve.
  • The same failure has happened more than once.
  • A monitoring alert exists that can fire and you want to attach a response.

The threshold is low — a runbook that exists is worth more than a better one that does not.

Steps

  1. Pick the next number. Look in services/portal/internal/how-to/incidents/runbooks/ for the highest 0XYZ- prefix.
  2. Copy the template. 
    cp services/portal/internal/handbook/templates/runbook.html \
          services/portal/internal/how-to/incidents/runbooks/0XYZ-short-slug.html
  3. Write the symptoms section first. What does the on-call engineer see? Alert name, log pattern, dashboard graph shape. This is the search target.
  4. Write the diagnosis steps. Numbered, copy-paste-able commands. No prose between steps unless absolutely needed.
  5. Write the mitigation. What stops the bleeding right now? Separate from the root-cause fix — the runbook is for stopping pain, not preventing recurrence.
  6. Add escalation. Who to wake up if mitigation does not work in N minutes.
  7. Link the alert. If a monitoring alert exists, add the runbook URL into the alert's annotations.
  8. Update navigation per Update sidebar — runbooks are listed under «Documentation operations → Runbooks».
  9. Run make docs-fix.

Voice for runbooks

Strict imperative. Short sentences. No background. Assume the reader is tired and stressed. «Run X», «Check Y», «If Z, then …». Save the explanation for the postmortem.

Verify

  • New runbook file exists at services/portal/internal/how-to/incidents/runbooks/<NNNN>-short-slug.html with the next free 4-digit ID.
  • All required sections present: Trigger / Fast triage / Most common causes / Recovery steps / Exit criteria / Escalation / Follow-up.
  • Runbook appears in the runbook index with the expected domain pill and sidebar entry.
  • Trigger condition is concrete (command line, alert name, or log substring — not a vague feeling).
  • Recovery commands are copy-paste-runnable, with no placeholders that would 404 or fail silently.
  • make docs-check exits 0; portal search returns the runbook on its title and trigger keyword.
