Build and run the API as a Docker image

Build the API as an OCI image and run it with Docker for packaging checks and demos.

Optional tooling and daily development

Docker is optional for day-to-day work. The usual loop is venv, make run, and tests; see Local development. You do not need an image for every change.

The local loop (code → test → make verify → commit) stays the same. A container image is how teams ship the exact bits that passed CI to a registry, and then run them in a target environment.

Real deployments usually expect a container image. Some build step produces the image and pushes it to a registry (Docker Hub, GitHub Container Registry, a cloud-vendor registry). The target environment then pulls that image and runs it with production configuration and secrets. The repo's Dockerfile is the starting point.

What a real deployment looks like (high level)

A full production setup is unique to each team, but the pattern is almost always the same:

  1. Build and test in CI. The pipeline runs lint and tests, and usually builds the Docker image and tags it (for example :1.4.2 or :<git-sha>).
  2. Push to a registry. docker push uploads the image to a URL your servers can read (authentication required). Nothing is uploaded automatically by git push unless a workflow is wired up.
  3. Roll out to an environment. The platform or the ops process updates running services to the new image tag (for example, on a VM with Docker or on a managed container service).
  4. Configuration and data. Production APP_ENV, secrets, database URL, and scaling are set in the environment, not in the image. SQLite is fine for demos. Multi-instance setups typically use a shared database such as PostgreSQL.
  5. Traffic. DNS points at a load balancer or a reverse proxy, TLS terminates at the edge, and requests reach the service on port 8000 (or behind a reverse proxy).

What you need installed

To build or run the image locally, install Docker Desktop or Docker Engine — Get Docker.

Docker in one minute

Docker builds an image (a filesystem snapshot plus metadata) and runs it as a container (an isolated process on the host). The API image installs dependencies from the same services/api/requirements.txt as make install, copies services/api/app/ and services/api/alembic/, and starts the app through services/api/scripts/container_entrypoint.sh (mounted in the container as /app/scripts/container_entrypoint.sh): it runs alembic upgrade head and then Uvicorn. The same script runs on the host through your .venv; the image does not invoke make, and there is no .venv inside the container.

  • Image name used in this repo: study-app-api:local (tag is arbitrary).
  • Build: docker build -t study-app-api:local .
  • Run (example): mount a volume for SQLite so data survives container removal: 
    docker run --rm -p 8000:8000 \
-e APP_ENV=dev \
-e SQLITE_DB_PATH=/data/study_app.db \
-e API_MOCK_API_KEY=local-dev-key \
-v study-app-data:/data \
          study-app-api:local
    Then open http://127.0.0.1:8000/live. For qa/prod rules, non-default API keys and stricter CORS apply (services/api/app/core/config.py). Pass secrets via -e or your platform's secret mechanism — not committed files.

SQLite and replicas

This API uses SQLite. A single container with a mounted data directory is fine for demos. Do not assume multiple replicas can share one SQLite file. Horizontal scaling moves the data to a shared database such as PostgreSQL, which is a future architectural change.

Steps

  1. Build the image. 
    docker build -t study-app-api:local .
    The build reads services/api/Dockerfile; installs services/api/requirements.txt; copies services/api/app/ + services/api/alembic/; sets the entrypoint to services/api/scripts/container_entrypoint.sh.
  2. Run the container. 
    docker run --rm -p 8000:8000 \
  -e APP_ENV=dev \
  -e SQLITE_DB_PATH=/data/study_app.db \
  -e API_MOCK_API_KEY=local-dev-key \
  -v study-app-data:/data \
          study-app-api:local
    The named volume study-app-data survives container removal; the SQLite file lives inside it. Pass any other env vars (API_RATE_LIMIT_*, CORS settings) via additional -e flags.
  3. Hit the live probe. 
    curl -i http://127.0.0.1:8000/live
    Expect HTTP 200 with JSON body. If the container exited during startup, see Troubleshooting below.
  4. Stop and clean up (when done). 
    docker ps    # find the container id
docker stop <id>
          docker volume rm study-app-data   # only if you want to discard data

Verify

  • docker build exits 0 and prints a final image id; docker images study-app-api lists the new tag.
  • docker run stays in foreground (or daemonised via -d) without immediate exit; docker ps shows status Up.
  • curl -i http://127.0.0.1:8000/live returns HTTP/1.1 200 with application/json.
  • curl -i http://127.0.0.1:8000/ready also returns 200 after Alembic finishes (might take a few seconds on first start).
  • Container logs (docker logs <id>) show the Alembic «upgrade head» pass and the Uvicorn startup line, with no Python traceback.

Troubleshooting

  • «Bind for 0.0.0.0:8000 failed: port is already allocated.» Something else is bound to 8000. Either stop the conflicting process (lsof -i :8000 to find it) or pick a different host port: -p 8001:8000.
  • Container exits immediately with «alembic.util.exc.CommandError: Can't locate revision identified by ...». The mounted volume contains a DB from a different schema version. Either drop the volume (docker volume rm study-app-data) or downgrade Alembic to match the schema.
  • Permission denied writing to /data. The volume was created with different UID/GID. docker volume rm study-app-data then re-run; Docker will recreate the volume with the container user's permissions.
  • Health probes return 503. Check docker logs <id> — usually a missing env var (APP_ENV / API_MOCK_API_KEY) or a config check that didn't pass. Restart with the missing var passed via -e.
