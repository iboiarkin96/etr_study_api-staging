Docs How-to

Add a new API resource page

Wire a new HTTP resource (hub page + per-operation pages + sidebar entry + search index) into the internal portal. Worked example uses Order; the procedure is the same for any resource.

Before you start

  • The resource is already defined in OpenAPI (handlers + schemas exist under services/api/app/); this how-to documents the resource, it does not implement it.
  • make venv && make install ran successfully — you'll run make docs-fix at the end.
  • You've skimmed the page-skeleton reference once so the standard HTML shell (topbar, sidebar mount, docs-prose, history footer) is familiar.
  • Use an existing resource as a template: internal/services/api/reference/user/ is the most stable worked example.

Steps

  1. Create the resource folder. 
    mkdir -p services/portal/internal/services/api/reference/order/operations
    The hub will live at order/index.html; one HTML file per endpoint goes under order/operations/.
  2. Copy and adapt the resource hub. Start from user/index.html as a known-good skeleton: 
    cp services/portal/internal/services/api/reference/user/index.html \
          services/portal/internal/services/api/reference/order/index.html
    Rewrite the lede, the breadcrumb tail, business context, contract summary, and the operations index. Keep the page-skeleton shell intact.
  3. Add one operation page per endpoint. Filename pattern is <method>-<path-template-kebab>.html — derived from the HTTP method and the URL path, NOT from the Python operationId. Examples from user/operations/: 
    post-api-v1-user.html
get-api-v1-user-system_uuid-system_user_id.html
          patch-api-v1-user-system_uuid-system_user_id.html
    For Order that gives post-api-v1-order.html, get-api-v1-order-order_id.html, etc. Copy a sibling operation as a skeleton.
  4. Register in the sidebar. Open services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json and add the resource entry under the API-reference subtree. Mind acronym casing per the sidebar-acronym-casing rule (API stays uppercase). The hub gets a parent children list; each operation file gets a leaf with id, label, and an absolute href.
  5. Link from the API reference hub. Add a card on services/api/catalog/by-quadrant/reference.html alongside the existing resource cards (user · conspectus · error-log).
  6. Run the docs pipeline. 
    make docs-fix
    Stage 5 normalises your new HTML; stage 9 indexes it. See How-to · Run the docs pipeline for the full 9-stage breakdown.

Verify

  • make docs-check exits with Docs check passed (no drift).
  • Open the local portal — the sidebar shows the new resource under API reference, with the operation files listed underneath.
  • Type the resource name into the portal search — the hub and at least one operation page appear in results.
  • The API reference hub renders the new resource card and the «See all» link to order/index.html resolves.
  • Each operation page renders with the v2 topbar, sidebar, prose, and history footer — no console errors.

Troubleshooting

The four mistakes that cost the most time on the first run.

  • Operation file named after the Python operationId. The portal convention is HTTP-method-plus-path-template (e.g. get-api-v1-user-system_uuid-system_user_id.html). operationId changes with refactors; the URL doesn't. Rename and re-register in the sidebar.
  • Sidebar entry doesn't appear after a docs-fix. JSON syntax error in nav-tree-internal.json — a missing comma or a stray trailing one. Validate: python -m json.tool < services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json > /dev/null. Silence means valid.
  • New pages don't show up in search. The search index was rebuilt but the browser cached the old JSON. Hard-reload (Cmd/Ctrl+Shift+R) or open the page in a private window.
  • Acronym broken in the sidebar label. «Api Reference» instead of «API Reference». Edit nav-tree-internal.json and preserve API / AsyncAPI / SRE / gRPC casing per the sidebar-acronym-casing rule; make docs-fix doesn't auto-fix labels.
