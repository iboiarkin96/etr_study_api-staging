Authoring API 21 required sections

Add an API operation spec

One file per METHOD /path. Lives under reference/api/{entity}/operations/. The 21 required sections are defined in RFC 0003 Appendix A.

When to add one

  • The endpoint exists in code (controller + route) and has been merged.
  • The OpenAPI schema has been updated in the same PR or earlier.
  • You have a worked example of the request and response payloads.

If the endpoint is still a sketch, write it as an RFC or a how-to under how-to/api/add-… instead — those are working-out-loud artefacts, not contracts.

Steps

  1. Choose the file name. Pattern: method-path-with-dashes.html. For POST /api/v1/conspectuses/{id}/actions/review → post-api-v1-conspectuses-conspectus_uuid-actions-review.html.
  2. Copy the template. 
    cp services/portal/internal/handbook/templates/api-spec.html \
          services/portal/internal/services/api/reference/{entity}/operations/{filename}.html
  3. Fill identification, summary, request, response, errors, examples first; these are what readers need on the first scan.
  4. Link cross-cutting concerns from services/api/reference/_shared/ — auth, idempotency, error envelope, pagination, versioning, observability, field conventions. Do not restate; link.
  5. Add the page to navigation under the right entity in both nav files (see Update sidebar).
  6. Run make docs-fix. The marker-sync stage will compare your spec against the live OpenAPI schema and fail if they diverge — that is the contract gate.

Per-section checklist

Walk this list before opening the PR. RFC 0003 Appendix A is the source.

  • Method + path identifier (matches OpenAPI operationId)
  • One-line summary; lede paragraph
  • Auth requirements (link to _shared/auth.html)
  • Idempotency contract (link to _shared/idempotency.html)
  • Request schema (path / query / headers / body)
  • Response schema (success status, headers, body)
  • Error envelope and concrete error codes
  • Worked example: request + response, copy-paste ready
  • Validation rules
  • Side effects and observable state changes
  • Rate limits and quotas
  • Observability hooks (metrics / logs / traces emitted)
  • Security considerations
  • Versioning notes (when introduced, deprecation status)
  • Related operations
  • Implementation pointers (controller, service, repository)
  • History

Verify

  • Spec page exists at services/portal/internal/services/api/reference/<resource>/operations/<method>-<path-template-kebab>.html; all 21 sections (per the API spec template) are present and filled in (no placeholder bodies).
  • Operation appears on the resource hub (<resource>/index.html) and in the sidebar with API-uppercase casing.
  • Linked ADR is named in the spec's «Traceability» section and referenced from the «Decision» bullet.
  • OpenAPI baseline accepted via make openapi-accept-changes; make contract-test passes.
  • make docs-check exits 0; spec_lint and spec_consistency pass for the new operation page.
  • Portal search returns the spec on its title + endpoint path queries.
