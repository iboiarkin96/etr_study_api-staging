Authoring ADR Governance

Add an ADR

One decision per record. Open a GitHub Issue first, ratify in discussion, then merge the ADR file with the agreed outcome. Follow ADR 0018 for the full lifecycle.

When to write an ADR

  • You are about to make a choice that future contributors will need to know why we made.
  • The choice is irreversible, expensive to reverse, or affects multiple subsystems.
  • Reasonable engineers would disagree about the right answer.

If the answer is obvious or the change is local, do not write an ADR — just ship it and document the result where it belongs (reference page, how-to, runbook).

Steps

  1. Open a discussion Issue. Title starts with [ADR]. State the question, the candidate answers, and the constraints. Tag stakeholders.
  2. Pick the next ADR number. Look at services/portal/internal/governance/adr/ and use the next free 4-digit number.
  3. Copy the template. 
    cp services/portal/internal/foundations/reference/sa/templates/adr.html \
          services/portal/internal/governance/adr/0XYZ-short-slug.html
    The canonical scaffold lives under foundations/reference/sa/templates/ alongside the other 15+ document scaffolds.
  4. Fill the Context, Decision, Scope, Alternatives, Consequences sections. Most of the value of an ADR is the alternatives — what you considered and rejected, with one-line reasons.
  5. Add the page to navigation. Insert under the right group in both nav files — usually under «Documentation and diagrams», «Security», or whichever cluster the decision belongs to.
  6. Run make docs-fix to format the HTML and refresh the search index.
  7. Open the merge PR. Link the discussion Issue in the description. Once approved, fill the Ratification block at the top of the ADR with the merge PR URL and acceptance date.

Anti-patterns

  • One ADR, many decisions. Split into separate ADRs. Each row in the index should be one yes-or-no question.
  • No alternatives. «We chose X» without the rejected options gives future readers no way to revisit the choice.
  • ADR for an obvious fact. If no one would argue, it is a reference page, not an ADR.
  • Editing a ratified ADR. Once accepted, ADRs are immutable. Supersede with a new ADR; link the new one from the Status of the old.

Verify

  • New ADR file exists at services/portal/internal/governance/adr/0XYZ-short-slug.html, with data-adr-weight set on <main>.
  • ADR appears in the sidebar under the right group; clicking it from any internal page navigates correctly.
  • ADR appears in the ADR registry card grid with the expected title and status badge.
  • Portal search returns the ADR for queries on its title or decision keywords.
  • make docs-check exits 0 (the ADR/RFC cross-reference hook validates anchors and back-links).
  • Ratification block at the top of the ADR is filled in with the merge PR URL and acceptance date once the PR lands.
