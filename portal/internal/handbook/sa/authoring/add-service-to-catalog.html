Add a service to the catalog

You have a new deployable artefact and want it to appear in services/portal/internal/services/. Write one catalog-info.yaml, run make catalog-render, and the generator updates the entity-card HTML, the hub tile, the hero ticker, and the nav-tree subtree for you. This page walks the procedure end to end.

Before you start

  • Read the descriptor schema — every required field is listed there.
  • Pick the canonical service name — lowercase, hyphen-separated, matches the directory. Examples that exist today: api, portal, ui-kit, monitoring.
  • Decide the lifecycle up front: stable (in production with on-call), experimental (shipping but provisional), or deprecated (kept for archaeology).
  • Pick the display-order integer. Existing values: api=10, portal=20, monitoring=30, ui-kit=40. Insert at a multiple of 10 between two neighbours.

Step 1 — Create the six-page spine directory

Every catalog entry exposes the six-page spine at fixed filenames. Create the directory and six files even if some are stubs. 

mkdir -p services/portal/internal/services/<name>
cd services/portal/internal/services/<name>
      touch index.html architecture.html api-reference.html runbooks.html dependencies.html on-call.html

Stubs are fine for day one — the constraint is that the URL exists, not that the page is complete. A one-paragraph "not applicable, here's why" stub satisfies the spine rule.

Step 2 — Author the index.html shell

The page surrounding the entity card is hand-authored (topbar, breadcrumb, sidebar mount, page chrome). The entity card itself is generated. Reserve its slot with the marker comments shown below — the generator rewrites whatever sits between them, and preserves everything outside. 

<main>
<article class="docs-prose">
<header class="page-head">
<h1><name></h1>
<p class="lede"><one or two sentences of context, hand-authored></p>
</header>

<!-- catalog:start id="entity-card" -->
<!-- generated by tools/docs/render_service_descriptors.py — do not hand-edit -->
<!-- catalog:end -->

<!-- … spine cards, hand-authored prose, etc. … -->
</article>
      </main>

Copy the chrome (head, topbar, sidebar mount, footer history, TOC, scripts) from an existing service's index.html — they're identical across the four services and not generator-managed.

Step 3 — Write catalog-info.yaml

Create services/portal/internal/services/<name>/catalog-info.yaml. Copy the skeleton below and fill in every placeholder. Fields and their semantics are documented in the descriptor schema. 

# Backstage-compatible service descriptor.
# Source of truth for the entity card on this directory's index.html and for
# the corresponding tile on services/portal/internal/services/index.html.
# Rendered by tools/docs/render_service_descriptors.py — do not hand-edit the
# generated HTML regions (marked by <!-- catalog:start/end -->).
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
name: <name>
title: <name>
description: <one verb-led sentence — the "Provides" row>
annotations:
backstage.io/source-location: <monorepo path, e.g. app/main.py or services/foo/>
backstage.io/last-reviewed: "YYYY-MM-DD"
tags:
- <stack-tag-1>
- <stack-tag-2>
links:
# All three of these are optional — delete blocks the service doesn't have.
- url: code-catalog/by-quadrant/reference.html
title: pdoc (auto-generated)
type: code
- url: <public surface URL or label>
title: <display text>
type: public
spec:
type: service           # or "library" for design systems / shared code
lifecycle: <stable | experimental | deprecated>
owner: user:<owner-id>
dependsOn:
- component:<other-service-name>   # repeat per upstream service

x-study-app:
tone: <info | success | warning | neutral>
display-order: <integer; pick a multiple of 10 between existing services>
lede: >-
<one or two sentences expanding "description" — entity-card subhead>
tile-lede: >-
<shorter one-clause restatement for the hub tile>
tile-stack: "<Tech1> · <Tech2> · <Tech3>"
header-pills:
- { text: <Stack pill 1>, tone: info }
- { text: <Stack pill 2>, tone: neutral }
dependencies:
- label: <Lane name, e.g. Framework>
chips:
- { name: <tech>, icon: <icon-key>, status: current }
- label: <Lane name, e.g. Persistence>
chips:
- { name: <tech>, icon: <icon-key>, status: current }
      - { name: <tech>, icon: <icon-key>, status: planned, hint: next }

Pick icon keys from the registry in the schema page. Unknown keys fail the generator with a precise error — there is no silent fallback, so add the icon to ICONS in tools/docs/render_service_descriptors.py first if you need a new one.

Step 4 — Run the generator

make catalog-render

The generator writes four artefacts:

  • the entity-card region inside your new <name>/index.html;
  • the catalog-grid region inside services/portal/internal/services/index.html (your tile is inserted at the right display-order position);
  • the catalog-tickers region inside the same hub page (totals + per-lifecycle breakdown update automatically);
  • the services subtree inside services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json (your service appears in the sidebar with its existing spine pages as children).

Re-running make catalog-render produces zero diff if nothing changed. Use this as the green-build signal locally.

Step 5 — Wire cross-service dependencies

For every other service that depends on the new one (or that the new one depends on), update both ends in YAML:

  • In each upstream's catalog-info.yaml, add the new service to its spec.dependsOn list as component:<name>.
  • In each consumer's catalog-info.yaml, add a chip to the relevant x-study-app.dependencies lane referring to the new service.
  • In the new service's own catalog-info.yaml, both fields must already reflect its upstreams (Step 3).

Re-run make catalog-render after each YAML edit so the entity cards on both ends update consistently. The dependency surface is the audit's #1 ask — failing to wire it on both ends turns the catalog into a one-way map.

Verify

  • make catalog-render exits 0 and reports updated artefacts; a second run reports zero diff (idempotent).
  • make catalog-render-check exits 0 (the pre-commit hook runs the same check).
  • All five required top-level keys in catalog-info.yaml: apiVersion, kind, metadata, spec, x-study-app.
  • All required metadata fields filled (name, title, description, both annotations); all required x-study-app fields filled (tone, display-order, lede, tile-lede, tile-stack, dependencies).
  • All six spine files exist (even as one-paragraph stubs).
  • spec.dependsOn wired on both ends for every cross-service relationship.
  • No fields from the omitted list snuck in.
  • Sidebar updated — open any internal portal page; the new service appears under its expected group in the left nav with its six spine pages as children, acronym casing preserved.
  • Hub tile renders — visit services/portal/internal/services/index.html; the new tile is inserted at the expected display-order slot, the hero tickers updated, and the dependency graph wires both ends.
  • Commit both catalog-info.yaml and the four generated HTML/JSON files together; the pre-commit hook refuses a partial commit.

Rollback

If the new service was added by mistake, or its scope changed before merge:

  1. Delete the new services/portal/internal/services/<name>/ directory entirely (the six spine files + catalog-info.yaml).
  2. Revert the spec.dependsOn additions in every upstream / consumer catalog-info.yaml you touched in Step 5.
  3. Revert the x-study-app.dependencies chip additions in the same files.
  4. Re-run make catalog-render — the generator rewrites the entity cards, hub tile, hero ticker, and nav-tree subtree back to their pre-change state.
  5. Confirm with make catalog-render-check + git diff — the diff should now be empty except for the deletion of the new directory.
