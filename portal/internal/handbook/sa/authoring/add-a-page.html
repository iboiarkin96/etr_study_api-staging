Authoring Workflow 5 steps

Add a page

The five steps from I want to write a doc to merged and searchable. This is the master workflow; type-specific recipes (ADR, RFC, API spec, runbook) live as sub-pages and inherit these steps.

Step 1 — Pick the document type

Open Principles → Diátaxis table and identify which quadrant your reader is in. Then pick the document type that lives in that quadrant:

Reader isDocument typeLives underRecipe
Learning the systemTutorial («Your first X»)tutorials/—
Performing a taskHow-tohow-to/—
Recovering from a known failureRunbookhow-to/incidents/runbooks/add-a-runbook
Looking up API contractAPI operation specreference/api/{entity}/operations/add-an-api-spec
Looking up DB schemaDatabase table specreference/data/—
Understanding rationaleExplanationexplanation/—
Recording a decisionADRgovernance/adr/add-an-adr
Proposing a designRFCgovernance/rfc/add-an-rfc
Auditing a systemAuditgovernance/audits/—

If you cannot decide between two types, you are writing two pages. Split the scope.

Step 2 — Copy the template

Open the template catalogue, find your type, and save the file to its canonical location with a real name:

  • File name: lowercase kebab-case. Numeric prefix when ordering matters (0042-…).
  • Operation pages: METHOD-path-with-dashes.html, for example post-api-v1-user.html.
  • Never edit the template in place. Always copy first.

Step 3 — Fill every section

Templates list the sections your reader expects. Walk every heading:

  • If a section truly does not apply, write «N/A — <one-line reason>»; do not delete the heading.
  • Use sentence case in headings. Preserve acronym casing (API, ADR, gRPC).
  • Read the style guide § plain language aloud before you write the lede.
  • Add a history row with today's date (cell schema → SA reference · History format).

Step 3b — Set the metadata tags (mandatory)

Every page declares five attributes on <body>. The catalog (catalog/by-quadrant/, by-service/, by-topic/) is rendered from these tags by tools/docs/build_catalog.py (step [9/10] of make docs-fix) — a missing or wrong tag means the page does not appear in its lens, the lint warns, and the choice rule in ADR 0032 cannot enforce IA.

AttributeClosed set / formatHow to choose
data-page-type tutorial · how-to · reference · explanation · landing · landing-section · blog Diátaxis quadrant. tutorial = sequenced learning. how-to = recipe for a task. reference = canonical facts. explanation = essay on why. landing/landing-section = hub. blog = personal essay (outside Diátaxis).
data-service api · portal · datastore · monitoring · ui-kit · none If the page would die when the service is retired → name the service. Otherwise none (cross-service handbook content).
data-role space-separated, any of sa · swe · qa · sre · architect · manager; empty = cross-role Primary audience. Multiple roles permitted ("architect swe"). Leave empty for truly universal pages (glossary, personas).
data-lifecycle draft · review · published · deprecated Per ADR 0018 lifecycle vocabulary. Default published after merge.
data-updated ISO-8601 date (YYYY-MM-DD) The date of the last meaningful update. Drives catalog/recent.html.

Concrete example for a service-tied runbook authored by SRE:

<body data-page-type="how-to"
      data-service="api"
      data-role="sre"
      data-lifecycle="published"
      data-updated="2026-06-06"
      class="docs-shell"
      data-maintainer-ids="...">

Defaults inferred by scripts/sweep_roles.py from path (e.g. services/api/runbooks/X.html → data-service="api" data-role="sre") — but the authoring page still must set them explicitly; tooling defaults are a safety net, not the source of truth.

One manifest drives the sidebar (see Update sidebar for the full procedure):

Step 5 — Run make docs-fix

From the repo root: 

make docs-fix

The pipeline runs in order: PlantUML render → marker sync → HTML formatter → pdoc → search-index regeneration. If any stage fails, fix the root cause; do not commit a partial result. See the pipeline reference for stage details.

Before opening the PR, run the CI gate locally: 

make docs-check

Verify — pre-merge checklist

Tick every item. Tooling that fails (or is silent when it should not be) means a missing tag, a stale autogen surface, or a wrong relative path. Run them in this order.

  1. Rendering & assets. Page renders at its URL with no broken assets and no console errors.
  2. Breadcrumb + sidebar. Both reach the page; sidebar entry preserves acronym casing (API / SRE / gRPC).
  3. Metadata tags present. <body> declares all five required attributes (Step 3b above): data-page-type · data-service · data-role · data-lifecycle · data-updated.
  4. Catalog regeneration. Step [9/10] of make docs-fix runs tools/docs/build_catalog.py automatically. The new page must appear in:
    • catalog/by-quadrant/<page-type>.html — under its service group.
    • catalog/by-service/<service>.html — under its quadrant group (if data-service != "none").
    • catalog/by-topic/<runbooks|postmortems|tests|on-call>.html — if path matches a topic predicate.
    • catalog/recent.html — by data-updated.
  5. Role pill. The card in any lens shows the correct role pill(s). Empty data-role renders as cross-role; if that is wrong, set the role explicitly.
  6. Pipeline gate. make docs-fix runs cleanly (UML render → marker sync → HTML repair → normalize → catalog render → maintainer fix → pdoc → search-index).
  7. Drift gate. make docs-check green (no drift). If it fails on the first run, run make docs-fix + make docs-check once more to converge — drift is normal after a new page lands.
  8. Search. Portal search returns the new page for at least one obvious query (title, slug, or main keyword) after the index regenerates.
  9. Diagrams. Every diagram is PlantUML-sourced with a committed SVG (no Mermaid; per ADR 0020).
  10. History. Page-history row added (date + author + one-line note); cell schema per SA reference · History format.
  11. PR coupling. The code change that motivated the doc landed in the same PR.
One-liner sanity check — paste into terminal after authoring 
# from repo root
make docs-fix \
  && make docs-check \
        && echo "OK · page is catalog-visible, drift-free, ready to commit"

Verify — legacy notes

  • Page renders at its URL with no broken assets and no console errors.
  • Breadcrumbs and the sidebar both reach the page; the sidebar entry uses the correct acronym casing (API / SRE / gRPC).
  • Portal search returns the new page for at least one obvious query (title, slug, or main keyword).
  • Every diagram is PlantUML-sourced with a committed SVG (no Mermaid; per ADR 0020).
  • Page-history row added (date + author + one-line note).
  • The code change that motivated the doc landed in the same PR.
  • make docs-check exits 0 — no drift in search index, no broken asset refs, no design violations.
