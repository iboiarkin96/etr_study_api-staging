Add an RFC

An RFC is the design document that drives several related ADRs. Use it when one decision is not enough — the whole shape of a subsystem or a process needs to be argued in one place. Follow this when adopting a new framework, redesigning an IA layer, or replacing a significant part of the platform.

ADR or RFC?

  • ADR — one yes-or-no choice. Short. Immutable once accepted.
  • RFC — design proposal that contains several choices and their interactions. Long-form. May spawn multiple ADRs as it is ratified.

If you find yourself writing an ADR that needs sections for «Component A», «Component B», «Migration plan» and «Open questions» — it is an RFC.

Steps

  1. Open a discussion Issue. Title starts with [RFC]. State the problem, the proposed shape, and the open questions. Tag stakeholders by area (architecture, SRE, security).
  2. Pick the next RFC number. Look at services/portal/internal/governance/rfc/.
  3. Copy the template. 
    cp services/portal/internal/handbook/templates/rfc.html \
          services/portal/internal/governance/rfc/000X-short-slug.html
  4. Write in two passes. First pass: full draft with «Open questions» listed honestly. Second pass: after the discussion, close each open question with a paragraph and link to any ADR that ratifies a sub-decision.
  5. Update navigation per Update sidebar.
  6. Run make docs-fix.
  7. Open the merge PR with status Proposed. Switch to Accepted in a follow-up PR after the discussion closes.

Anti-patterns

  • RFC instead of an ADR. If there is only one decision, an ADR is faster and more discoverable.
  • RFC that buries the question. The first paragraph must state what is being decided.
  • RFC without industry references. Cite at least three external sources (papers, vendor docs, mature OSS projects) that informed the design.

Verify

  • RFC file exists at services/portal/internal/governance/rfc/<NNNN>-short-slug.html with data-rfc-weight set on <main>.
  • Status badge shows the current milestone (draft → in review → approved → implemented → superseded) and matches data-rfc-weight.
  • RFC appears in the RFC registry with its number, title, and current status; the sidebar entry resolves.
  • Industry references (at least 3) listed in the Context or Alternatives sections, each as a real external link.
  • Discussion Issue linked from the Status block; merge PR linked once landed.
  • make docs-check exits 0 (the ADR/RFC link-check validates the back-references and status-badge presence).
