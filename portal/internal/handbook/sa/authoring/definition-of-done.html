How-to Authoring 11-item checklist

Definition of Done — self-check

Walk this checklist before you request review on a docs PR. Each item declares which gate enforces it: AUTO means a script in pre-commit or CI catches the violation; REVIEW means a reviewer rejects the PR if the item is missing or unclear. The rationale for each gate lives in Process → § 03 Gates.

When to run this check

  • Right before you flip the PR from draft to «ready for review».
  • Again after responding to review comments, if any item could have regressed.
  • Whenever the page is touched by an automated formatter or migration script — confirm the AUTO items still hold.

The 11-item checklist

  1. REVIEW The behaviour change is implemented and tested in code.
  2. REVIEW The relevant doc page is updated in the same PR — no «docs follow-up» allowed.
  3. AUTO A Page-history row is appended on every page that was edited — today's date, one-sentence description.
  4. AUTO Every diagram has a .puml source with a rendered SVG checked in.
  5. AUTO No links are broken; data-* attribute names match between markup and runtime.
  6. REVIEW Plain-language self-check: short sentences, common words, no unexplained jargon.
  7. AUTO The page has exactly one <h1> and a working in-page TOC.
  8. AUTO For new pages: hero + sections (with kicker) + Page-history footer + #mobile-contract block are present.
  9. REVIEW Any new domain term is defined in the glossary in the same PR.
  10. REVIEW The PR description mentions both the code change and the doc change.
  11. AUTO make docs-check exits 0 locally before the PR is opened — catches drift that pre-commit hooks alone miss (search-index, team pages, catalog).

Verify

  • Every AUTO item is green — make docs-check exits 0.
  • Every REVIEW item is one you can defend in the PR thread without hand-waving.
  • The PR description names both the code change and the doc change in one sentence each.

Troubleshooting

  • «My page is generated — does the History row still apply?» Yes, on the source (the YAML, the OpenAPI spec, the pdoc input). Generated HTML is rebuilt by the script and inherits the source's row.
  • «The page is a one-line typo fix — do I still need a History row?» No. Typo-only and whitespace-only edits are explicitly excluded — see History format → Rules.
  • «make docs-check fails on a file I didn't touch.» An auto-generator (search-index, team pages, catalog) needs to rebuild. Run the relevant generator and commit the regenerated output in the same PR.
