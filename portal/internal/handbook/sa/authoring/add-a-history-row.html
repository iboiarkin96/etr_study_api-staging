Authoring History Per-edit

Add a history row

Every substantive edit to a hand-written page earns one row in the <footer class="docs-history"> block at the bottom of the page. This is the recipe for adding one. The cell schema — date · author · note · maintainer attribute — lives in SA reference · History format; this page is the procedure, not the format.

When to add one

A row goes in for changes that move the reader's understanding of the page. Skip the row entirely for formatting passes that make docs-fix would have written anyway.

Worth a row:

  • New section, removed section, or a section rewritten paragraph-up.
  • Changed contract — schema, vocabulary, anchor IDs, public examples.
  • Migration to a new component, kit version, or template.
  • Rewording that changes what the reader takes away (e.g. softening a normative rule to a recommendation).
  • Splitting or merging a page with another.

Not worth a row:

  • Typo fixes, whitespace, lint cleanups, link rewrites with no semantic change.
  • HTML formatting passes by make docs-fix.
  • Autogen drift fixes (search index, nav tree regen, catalog rebuild) — those are tracked by the autogen step, not the page history.

Steps

  1. Pick today's date. ISO YYYY-MM-DD, the calendar date of the edit. Never backdate; never use the file's mtime.
  2. Find your maintainer UUID. Look under services/portal/internal/team/people/<handle>/. The UUID is the value of data-person-id on that page's <article>.
  3. If this is your first edit on the page, add your UUID to <body data-maintainer-ids="...">, space-separated. The page becomes co-maintained; existing maintainers stay.
  4. Insert a new <li class="docs-history__row"> at the top of <ul class="docs-history__list">. Newest first — never below an existing row.
  5. Fill the three cells per the schema in History format · Fields: docs-history__date · docs-history__author (with author-chip) · docs-history__note.
  6. Write the note as one present-tense sentence describing what changed. Wrap symbols, classes, files, and anchors in <code>. See History format · Examples for the legal / illegal contrast.
  7. Run make docs-fix to refresh the search index and re-format the file, then make docs-check to confirm no autogen drift before you commit.

Anti-patterns

  • Empty filler note. «Updated.» / «Minor changes.» — the row is information for a future reader, not a checkbox for the author.
  • Several unrelated edits in one row. If two paragraphs solve two different problems, they get two rows on two dates.
  • Backdating. The date is when the edit happened, not when the underlying decision was made. A 2024 ADR rewritten today gets a 2026-06-06 row.
  • Reordering or deleting existing rows. History is immutable. Corrections go in a new row that supersedes; the original stays.
  • Row for an autogen pass. make docs-fix regenerating the search index or catalog is not a substantive edit. Commit it without a row.
  • Author chip without a UUID. A row with data-person-id="" renders as a broken chip. If the legacy author is unknown, use «—» as plain text instead of an empty chip.

Verify

  • The new row is the first <li> inside <ul class="docs-history__list"> — older rows below.
  • Date renders as ISO YYYY-MM-DD matching today's calendar date.
  • Author chip hydrates in the browser — avatar + handle visible, no broken-image icon.
  • Note reads as one present-tense sentence describing what changed; <code> spans render in mono.
  • <body data-maintainer-ids> contains your UUID (if this is your first edit on the page).
  • make docs-check exits 0 — no autogen drift, no broken anchors.
