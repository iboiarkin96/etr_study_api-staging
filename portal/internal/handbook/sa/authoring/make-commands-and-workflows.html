Use the Make workflows

Targets grouped by job, diagrams for the composite pipelines, and an if → then table for common tasks. The full target list lives in make help. Governing policy: ADR 0008.

Recommended entry points (start here)

Most days you only need one of these. Each one wraps the same atomic targets (format, lint, tests, OpenAPI, docs); they differ in the order and in how the docs are handled at the end.

CommandUse whenOutcome
make fixYou want quick auto-fixes before committing.format-fix → lint-fix → docs-fix
make verifySingle strict gate for local and CI.deps-audit → lint-check → type-check → openapi-check → contract-test → test → docs-check → docs-a11y-check
make release-checkPre-release or deploy rehearsal.env-check → make verify
make release DEPLOY_CMD='…'Deploy only after the full gate.release-check → your shell command

If → then (scenarios)

If you…Then…
Just cloned the repomake venv && make install && make env-init, edit .env, make migrate, make run
Want auto-fixes before commitmake fix
Want full local gate and regenerated docsmake verify
Changed the API intentionallyReview, make openapi-accept-changes, commit baseline, make verify
Debug one testmake test-one path=tests/…
Run like production Docker entrypoint locallydocker run … study-app-api:local (see Docker guide)
Need metrics dashboards locallydocker compose -f services/monitoring/docker-compose.observability.yml up -d plus make run
Want logs in Kibana / Elasticsearch locallydocker compose -f services/monitoring/docker-compose.logging.yml up -d, run the API with NDJSON to ./logs (default LOG_FORMAT is json), create Kibana data view *study-app-logs* — see Local development.
Prepare a releasemake release-check; deploy with make release DEPLOY_CMD='…'

Verify

  • The target you ran exited 0; the last log line is the target's documented success line (e.g. verify: OK, == DOCS-FIX: SUCCESS ==, release-check: OK).
  • make help still lists every target you used — drift between this page and make help means the page is stale (open an ADR per ADR 0008).
  • git status after the target shows only files the target is documented to touch; anything else means the target reached outside its scope.
  • For composite pipelines (verify, fix, release-check), every constituent atomic target also passed — the composite is no greener than its weakest link.
