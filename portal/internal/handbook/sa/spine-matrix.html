📋 Catalog · Spine matrix

Spine matrix — what each catalog entity must document

For every catalog kind, this page names the canonical pages the entity owes — what the team actually fills in on disk and what CI rejects if missing. Five kinds carry pages (System · Component · Library · API · Resource); the other three (Group · User · Location) are organisational scaffolding and don't have a spine.

Apply

01

How to read this page

Each kind below opens a tone-tinted block with a grid of spine tiles — one tile per canonical page the entity owes. The tiles match the visual language of every services/<name>/index.html page (compare with the API service card). Below the per-kind tiles, the full required-vs-optional-vs-forbidden matrix lives in § 07.

  • ✅Required for lifecycle: production. CI fails the merge if absent.
  • ○Optional. Permitted; «—» on the entity tile if absent.
  • ❌Forbidden. Creating the page is itself a CI failure.
  • —N/A. The concept doesn't apply to this kind.

Apply

03

Adding a new entity end-to-end

  1. Pick the kind. If you can't decide between two, the entity isn't ready to be created — re-read Catalog · § 01 Eight kinds.
  2. Copy the matching template from templates/: Component → service-descriptor.html · API → api-spec.html · Library / Resource / System use component-spec.html or foundation-spec.html as a starting point.
  3. Author every ✅ page in the matrix (§ 04 below) for your kind. For lifecycle: experimental, Overview + Explanation are enough; the rest can come later.
  4. Register the entity in catalog-info.yaml next to the code — see the envelope example in Catalog · § 02 YAML envelope.
  5. The renderer render_service_descriptors.py auto-syncs the sidebar (nav-tree-internal.json) from your YAML on every run.
  6. Push. Pre-commit hooks validate catalog drift; CI gates reject any missing ✅ page or any forbidden ❌ page.

Spine vs policy

The grid above is the spine today. For the policy that governs how the spine itself grows — when a new row joins, when a kind is split, how the spine survives going from ~250 to 1000+ pages — see IA scalability. Without that cross-link the spine reads as a one-off design; with it, the spine reads as a versioned policy. Treat the two pages as a pair: this matrix names the obligations, IA scalability names the rules for changing them.
