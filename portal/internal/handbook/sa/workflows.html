🔁 SA reference · Workflows

Workflows — cross-service flows

Three headline use cases, each rendered as a sequence diagram, a numbered step list, and an error matrix. The diagram source lives next to the page as PlantUML (per ADR 0020 — no Mermaid). What you see in the rail below is the ASCII preview; the SVG rendered from the same source is what ships.

01

Why workflow diagrams sit in the SA spine

A workflow crosses bounded contexts and services. No single service owns it. The SA spine is the place where cross-cutting flows are authored, dated, and reviewed — anywhere else they fragment.

  • Source of truth. Each flow has one PlantUML source file under assets/uml/workflows/; service docs link to it, never duplicate it.
  • Pace of change. Workflow edits trigger ADR review when a step changes ownership or contract — not every diagram tweak.
  • Coverage. Every Tier-1 business use case has exactly one workflow page; smaller flows are inlined where they are used.
  • Format. PlantUML only. Mermaid is rejected at PR (ADR 0020) because GitHub-flavoured Mermaid drifts from local rendering.

Flow 01 — learner completes an exercise

WF-01 · Tier-1

Learner submits an attempt, gets a grade, mastery updates

Triggered by the learner submitting an exercise in the web app. Spans Identity (session check), Learning (content lookup), Assessment (grading), Progress (mastery roll-up), and Observability (event capture). Synchronous on the hot path; async on the mastery roll-up. 

@startuml — assets/uml/workflows/wf-01-learner-attempt.puml
Learner          Web            Identity        Learning        Assessment      Progress
  |                |               |                |                |                |
  |submit attempt  |               |                |                |                |
  |───────────────►|               |                |                |                |
  |                |verify session |                |                |                |
  |                |──────────────►|                |                |                |
  |                |◄──────────────|                |                |                |
  |                |fetch exercise |                |                |                |
  |                |───────────────────────────────►|                |                |
  |                |◄───────────────────────────────|                |                |
  |                |POST grade     |                |                |                |
  |                |───────────────────────────────────────────────►|                |
  |                |               |                |                |grade.published (event)
  |                |               |                |                |───────────────►|
  |                |◄───────────────────────────────────────────────|                |
  |grade visible   |               |                |                |                |
  |◄───────────────|               |                |                |                |
  |                |               |                |                |                |update mastery
  |                |               |                |                |                |(async)
@enduml
  1. Web receives POST /v1/attempts with the learner's payload.
  2. Identity verifies the session JWT — rejected requests return 401 immediately.
  3. Learning fetches the exercise definition (rubric version pinned at this read).
  4. Web forwards the attempt + pinned rubric to Assessment.
  5. Assessment grades synchronously and returns the grade to Web.
  6. Assessment emits assessment.grade.published.v1 on the event bus.
  7. Progress consumes the event asynchronously and updates the mastery materialised view.
  8. Observability captures spans for every hop via the shared OpenTelemetry kernel.

Flow 02 — Prometheus alert → on-call → postmortem

WF-02 · Tier-1

Alert fires, on-call responds, incident closes with a postmortem

The reliability spine. Starts when Alertmanager fires a Prometheus alert with a runbook_url annotation and ends when a published postmortem is linked from the incident record. 

@startuml — assets/uml/workflows/wf-02-incident.puml
Prometheus     Alertmanager  PagerDuty     On-call       Runbook       Postmortem
  |               |              |              |              |              |
  |rule fires     |              |              |              |              |
  |──────────────►|              |              |              |              |
  |               |route + group |              |              |              |
  |               |─────────────►|              |              |              |
  |               |              |page on-call  |              |              |
  |               |              |─────────────►|              |              |
  |               |              |              |open runbook  |              |
  |               |              |              |─────────────►|              |
  |               |              |              |execute steps |              |
  |               |              |              |──────────────|              |
  |               |              |              |              |              |
  |               |              |              |open incident |              |
  |               |              |              |─────────────────────────────►|
  |               |              |              |              |              |draft & review
  |               |              |              |              |              |(48h SLA)
  |               |              |              |              |              |publish
@enduml
  1. Prometheus evaluates a recording or alert rule and fires when the condition holds.
  2. Alertmanager deduplicates, groups, and routes by severity + service.
  3. PagerDuty pages the on-call engineer; the payload carries the runbook_url annotation.
  4. On-call opens the runbook link and follows the diagnostic ladder.
  5. If mitigated within 30 minutes, alert auto-resolves and on-call files a short note.
  6. Otherwise, on-call opens an incident record (with severity, services touched, customer impact).
  7. Postmortem draft is owed within 48h of incident close (sev-1 & sev-2 only).
  8. Postmortem is reviewed by a non-author principal, then published and linked from the incident.

Flow 03 — page draft → published & indexed

WF-03 · Tier-2

A new page goes from draft to indexed inside the portal

Docs-as-code in motion. The same pipeline production code uses — AUTO gates first, REVIEW gates second. Search index regenerates on the merge job, not in CI, so PRs stay fast. 

@startuml — assets/uml/workflows/wf-03-publish.puml
Author          PR             CI (AUTO)      Reviewer       Merge job      Portal
  |                |               |                |                |                |
  |open PR         |               |                |                |                |
  |───────────────►|               |                |                |                |
  |                |trigger checks |                |                |                |
  |                |──────────────►|                |                |                |
  |                |◄──────────────|                |                |                |
  |                |request review |                |                |                |
  |                |───────────────────────────────►|                |                |
  |                |◄───────────────────────────────|                |                |
  |                |merge          |                |                |                |
  |                |───────────────────────────────────────────────►|                |
  |                |               |                |                |regen index    |
  |                |               |                |                |───────────────►|
  |                |               |                |                |               |sidebar refresh
@enduml
  1. Author opens a PR from a feature branch; PR template enforces «what changed · why now · who reviewed».
  2. CI (AUTO) runs check_h1_structure, check_asset_refs, check_adr_rfc_links, validate_docs_html, validate_docs_a11y, check_path_literals.
  3. Failing AUTO gate blocks the PR — no human override. Author fixes and re-pushes.
  4. Reviewer applies the REVIEW gate: persona fit, narrative coherence, factual accuracy, link discipline.
  5. Reviewer requests changes inline or approves; approval requires a non-author reviewer.
  6. Merge job (.github/workflows/deploy-prod.yml) rebuilds the Pagefind bundle at services/frontend/portal/pagefind/ before the Pages artifact is staged.
  7. Portal picks up the new sidebar entry on the next page load (TTL ≈ 60 s on cached nav).

05

Rules for adding a new workflow

  • One source file. Each workflow has exactly one PlantUML file under assets/uml/workflows/wf-NN-slug.puml. Pages render the SVG; never duplicate the diagram in another doc.
  • Tier-1 only. Promote a flow to this page only when it crosses three or more services. Smaller flows live inline on the service's reference page.
  • Three blocks. Every workflow renders as: (a) sequence rail, (b) numbered step list, (c) error matrix or owner table.
  • Dated. Last review date is mandatory; the freshness CI warns at 90 days, fails at 180.
  • ADR-anchored. The «Frozen by» field names the ADRs/RFCs that locked the flow. Steps that change without an ADR update fail review.
