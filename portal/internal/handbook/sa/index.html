How we author this portal

SA handbook — one shared rulebook for every page on this portal. Two questions it answers: how a page should look (Principles · Style guide) and what should be inside it (Substance contract). When you sit down to ship a page, open Templates and copy the closest scaffold — never start from a blank file.

Section lens chips — five chips on every handbook page

Every numbered section on a handbook page (under foundations/reference/sa/) carries one chip telling the reader the section's mode at a glance. Authors pick the chip that fits; reviewers reject any section without one.

ChipThe section is…Typical content
Stance …a position the handbook commits to «We do X, not Y.» One aphorism + a paragraph of stance.
Rules …a normative enumeration Numbered or bulleted rules reviewers enforce. Most common chip.
Rationale …the reasoning behind a rule or trade-off Why we picked X over Y. Trade-off cards, wins-vs-costs.
Apply …actionable guidance or pointers «What to read next», «Where the live data lives», jump-cards.
Sources …provenance Where the claim comes from. Used sparingly, footer-style.

Full rules and the closed vocabulary live in ADR 0031 · D2. Adding a new chip requires a follow-up ADR — reviewers reject unilateral values.

02

Page lifecycle — driven by git

Every page lives in git. To ship one you go through four steps — draft on a branch, open a PR, pass review (automated checks + a human), then merge into main. After the merge the page is live and the search index rebuilds.

  1. Branch
  2. Pull request
  3. Review & audit
  4. Merged
  • 1 · Branch git switch -c docs/<slug> Cut a branch off main. The draft lives there — your local workspace, no one else sees it yet.
  • 2 · Pull request gh pr create Push and open a PR. CI runs make docs-check (links, h1, a11y, acronym casing) before a human looks at it.
  • 3 · Review & audit LLM prompts + human Two gates in one step: an LLM runs canned audit prompts (Diátaxis fit, tone, broken claims) and a human reviewer signs off the substance.
  • 4 · Merged git merge → main Squash-merge into main. The page goes live, search reindexes, the history block records the change.
