02

Audit & governance

Small enough to live on the hub. The audit protocol governs every PR; the LLM prompt below is the canonical prompt for an automated review pass; governance defines how this guide itself evolves.

Audit protocol

  • Run the quality gates. make docs-fix, make docs-check; CI gates link / h1 / a11y / acronym casing.
  • Verify links & navigation on every changed or added page. No 404s, no broken anchors, no orphan pages.
  • Compare section structure to the template for the document type. ADRs have ADR sections; runbooks have runbook sections — see Document types.
  • Check terminology against nearby index & hub pages. One term per concept, repeated everywhere.
  • Confirm the changelog & cross-links were updated when policy or process changed.
  • Record deviations & remediation in the PR checklist or audit notes under governance/audits/.

LLM conformance prompt (copy & paste)

Use this prompt when an LLM reviews a docs PR — covers the same checks a human would do, in the same order.

Validate the changed docs under services/portal/internal/ against the
style guide at foundations/reference/sa/style-guide.html and its three
deep-dives (voice.html · page-skeleton.html · document-types.html).

Return:
  1) Violations grouped by severity (critical / major / minor).
  2) For each violation: file path, violated rule, short evidence, concrete fix.
  3) Missing required sections by document type (ADR / RFC / Runbook / …).
  4) Terminology inconsistencies (same concept, different naming).
  5) Broken or suspicious links.
  6) Final pass / fail verdict and residual risks.

How this guide evolves

  • A style-rule change lands in this guide first, then propagates to related pages in the same PR.
  • When a rule changes architecture or process, the matching ADR or RFC moves in the same PR.
  • On any conflict, this guide overrides per-page wording. Page-level deviations need a written justification in the history row.

03

How to cite external sources

Operationalises Substance Principle 04 — «cite specific pages, not industry generally». A reader who follows a citation expects to land on the same paragraph the author read. «SRE Book» is not a citation; «SRE Workbook ch. 5, p. 88, fig. 5-3» is.

Citation shape

  • Anchored. Author or organisation, work title, edition or year, chapter or section, page or paragraph anchor. The reader must be able to verify the claim by opening one specific page.
  • Plain prose. Cite inside the sentence that uses the source, not as a footnote at the end of the section. «As SRE Workbook ch. 5 argues, multi-window burn-rate alerts trade detection latency for false-positive resistance.»
  • Internal first, external second. If the same fact is on an internal Reference or ADR page, link the internal page; cite the external source as «derived from SRE Book ch. 4». The portal is the source of truth for our system.
  • One citation per claim, not per paragraph. If a paragraph leans on five external pages it is doing too much; split the claim.

Anti-pattern vs canonical

Anti-pattern Why it fails Canonical rewrite
«As described in the SRE Book…» Two books, three editions, dozens of chapters. Reader cannot verify. «As Site Reliability Engineering (Beyer et al., 2016) ch. 4 «Service Level Objectives» argues…»
«Per industry best practice…» No source named. Appeals to an authority that does not exist. «Following the rationale in Accelerate (Forsgren, Humble, Kim, 2018) ch. 2 — software delivery performance is measured by lead time, deployment frequency, MTTR, and change-failure rate.»
«See the OpenAPI specification.» Which version? Which section? OpenAPI 3.1 differs from 3.0 in load-bearing ways. «See OpenAPI 3.1 §4.8.24 «Discriminator Object» — the mapping field is the canonical way to keep polymorphic schemas backwards-compatible.»
«Diátaxis is a framework for documentation.» Diátaxis has a single authoritative URL. Not citing it forces the reader to find it. «Diátaxis (Procida, accessed 2026-05-25) splits documentation into four quadrants — tutorial, how-to, reference, explanation.»

When not to cite

  • Common knowledge. «HTTP 404 means not found» needs no source. Citing it adds noise.
  • Tutorial & How-to bodies. Procedural pages should not interrupt the steps with literature. Move citations to a sibling Explanation or to a See-also link at the end — Tutorials carry the learner forward, not sideways.
  • Internal facts. «Our error budget is 99.9 %» cites the SLO page, not Google SRE. The decision is ours.
