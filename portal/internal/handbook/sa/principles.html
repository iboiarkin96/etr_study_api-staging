Principles — how we author this portal

Good documentation doesn't happen by accident — it rests on a small set of load-bearing principles that the whole team agrees on and applies every time. Below are the eight pillars our team uses to keep this portal in shape: they decide what kind of page you're writing, where it lives, what it looks like, and how it stays visually consistent.

Treat them as the working contract for any new or edited page — read them once, then lean on them instead of taste whenever a decision comes up. These eight cover form; the Substance contract covers the other half — what has to be inside a page.

Rules

TL;DRThe eight principles

Three are about how a page reads. Five are about how the portal is shaped. Apply all eight; none is optional — consistency is what turns a pile of pages into a portal.

🧭 01 · Structure

Diátaxis quadrants

Every page belongs to exactly one of four quadrants — Tutorial, How-to, Reference, Explanation — picked by what the reader is doing, not by topic.

See more: 🧭 Diátaxis

🗣️ 02 · Voice

Plain language

Short sentences, active voice, second person, one name per concept. Sentence case in headings with acronyms preserved (API, gRPC, SRE).

See more: 🗣️ Style guide

🔁 03 · Lifecycle

Docs as Code

Pages live in the repository, ship through the same pipeline, and review in the same PR as the code they describe. CI enforces the contract.

See more: 🔁 Process

📐 04 · Diagrams

PlantUML only

Every diagram is authored as a .puml source and rendered to SVG. No Mermaid, no draw.io, no screenshot-as-diagram. C4 levels and stencils are fixed.

See also: ADR 0020

🎯 05 · Provenance

Internal portal = source of truth

The internal portal is the canonical version of every service's docs. Public-facing pages, when they exist, derive from these — never the reverse.

See also: ADR 0026

🗺️ 06 · IA contract

Flat disk, Diátaxis sidebar

Files sit flat on disk (refactor-safe). The sidebar renders Diátaxis groups from each catalog entity. Two axes, one source.

See more: 🗺️ IA contract

🧩 07 · Catalog

Software Catalog (aka Backstage)

Every thing the portal documents is an Entity with a kind, a catalog-info.yaml descriptor, and explicit relations. We borrow the Backstage taxonomy — we don't invent kinds.

See more: 🧩 Catalog taxonomy

🎨 08 · Design

Visual contract & templates

One visual language across every page — same icons, same components, same hero / section / card patterns. Every doc is templated, never blank. Uniformity is what turns a pile of pages into a portal.

See more: 🎨 Visual contract

Apply

01

The five-question checklist

Walk this list before opening the PR and again during review. Each question maps to one principle — if an answer isn't a confident «yes», stop and fix it (the linked deep-dive page tells you how).

0 / 5 gates passed

Stop rule. If any gate isn't ticked — stop authoring, open the linked deep-dive, fix the source of the doubt. Shipping a «not sure» page guarantees a follow-up PR that nobody will land. State persists in localStorage across visits, so you can step away mid-review.
