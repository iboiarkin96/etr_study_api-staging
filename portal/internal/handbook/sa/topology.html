🗺️ Principle 06 · IA contract

Flat on disk, Diátaxis in the sidebar

The hardest single decision in a docs portal is how to lay files out on disk. Folders by quadrant? By entity? By role? Each option is wrong in a different way. So we don't choose. Disk stays predictable; the Diátaxis grouping lives in the sidebar, driven by per-entity YAML.

Rules

01

What «IA contract» means

The page title uses two words that each do a specific job. Before reading the claim, the rules, or the tradeoffs, pin down what they actually mean — once they're concrete, the rest of the page reads as «what the rule actually says».

  • IA — Information Architecture. The shape of the knowledge: which page lives where, which page groups with which, what the reader sees when they open the sidebar or the file tree. It is not visual design (that's UI); it is not the words on the page (that's content). IA is the skeleton between them — the answer to «where does this piece live and how does it relate to its neighbours?». In a docs portal of ~600 pages, IA is what makes the difference between «I found the runbook in 15 seconds» and «I gave up and Slack'd someone».
  • Contract. A binding rule we defend in code review, not a preference and not a guideline. If a page violates the contract, the PR doesn't merge. The principles handbook (SA · Principles) holds eight such contracts; this one is Principle 06.

Put together:

The IA contract is the rule that fixes how files lay out on disk vs. how they group in the sidebar — and how to change one without breaking the other.

Two axes, one source of truth. Disk shape is stable so links and bookmarks survive any refactor. Sidebar shape is Diátaxis-grouped so a reader picks an intent (learn / do / look up / understand) and gets there without knowing the file system. The rest of the page is what that rule actually says — the claim (§02), the example (§03), the YAML that drives it (§04), and what we give up to get it (§05).

Stance

02

The claim

Disk is flat. Sidebar is grouped. Files sit predictably under services/<name>/. The sidebar renders those same files into Diátaxis groups (📚 🔧 📖 💡) from each service's catalog-info.yaml. Two axes, one source of truth.

Who else does this: Django docs, Microsoft Learn, Cloudflare Workers, Backstage TechDocs (natively, via nav.yml). It's the industry default for a reason — refactor-safe, no broken links, no folder-religion debates.

Rules

04

The single source: nav: in catalog-info.yaml

The full entity envelope lives in Catalog taxonomy. The IA contract concerns one optional block — nav: — that tells the renderer which on-disk page belongs to which Diátaxis quadrant. 

# metadata + spec as in Catalog taxonomy …
nav:
  - group: tutorials
    label: Tutorials
    icon: 📚
    pages: []                       # empty → collapsed in sidebar
  - group: how-to
    label: How-to
    icon: 🔧
    pages: [runbooks, on-call]
  - group: reference
    label: Reference
    icon: 📖
    pages:
      - architecture
      - api-reference
      - dependencies
  - group: explanation
    label: Explanation
    icon: 💡
    pages: []

Moving a page from How-to to Reference is a one-line YAML edit: no git mv, no broken links, no redirect. The rendering script (tools/docs/render_service_descriptors.py) walks every catalog-info.yaml and emits nav-tree-internal.json — which the sidebar component reads at runtime.

Rationale

05

Tradeoffs & limitations

Wins

Refactor-safe by construction

Moving a page between quadrants doesn't move the file — it edits one YAML line. Links don't break. Search-index URLs don't change. Bookmarks survive.

Cost

Disk doesn't tell you the quadrant

ls services/api/ shows a flat list — you have to open catalog-info.yaml to see how those files group. Tooling that walks the filesystem misses the grouping unless it reads the YAML too.

Wins

Two axes don't fight

«By entity» (the directory) and «by quadrant» (the sidebar) coexist. No need to pick one ontology over the other — both render from the same files.

Cost

Custom renderer to maintain

nav: is our extension; Backstage itself doesn't ship a built-in sidebar generator. We own render_service_descriptors.py — when it breaks, no one else has fixed it before us.

Wins

Industry default

Django, Cloudflare, Microsoft Learn, Backstage TechDocs all converge on the same pattern — files-flat, sidebar-grouped. Reviewers from those teams recognise it on sight.

Cost

One page = one quadrant only

A page can't appear in two groups (e.g. both Reference and Explanation). When content legitimately straddles, we split into two linked pages — cleaner long-term but more files.
