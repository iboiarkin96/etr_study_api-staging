👥 SA reference · Ownership

Ownership — RACI for the doc system

Rules without owners rot fast. This page names — explicitly, in a matrix — who is responsible for each load-bearing artefact of the doc system: the UI Kit, the glossary, the principles, the templates, the spine matrix. Every artefact has exactly one Accountable role; otherwise «everyone's responsible» becomes «nobody's responsible».

Rationale

01

Why a RACI for the doc system itself

The portal's content has owners — every page declares its data-maintainer-ids. But the infrastructure of the doc system — the kit components, the term definitions, the principles, the scaffolds — has historically had no named owner, so changes happened ad-hoc and inconsistency leaked.

A RACI matrix solves three concrete problems:

  • Who can change what without an RFC? The Accountable role can change their artefact directly; everyone else needs review.
  • Who gets pinged when something breaks? A broken UI Kit component pages the Visual-system role; a broken glossary pings the SA-lead.
  • What's the escalation if the Accountable disagrees with a contributor? The Consulted role steers; if still deadlocked, an RFC opens.

Rules

03

What each role can change without an RFC

The Accountable role has direct-edit authority on their artefact. Bigger changes — semantic, breaking, or cross-cutting — always go through an RFC.

  • SA lead. Can change wording / examples / additions in Principles, Templates, Glossary, Nav tree without an RFC. Removing a principle, renaming a template, or changing a glossary term's definition needs an RFC.
  • Architect. Can add rows to the Spine matrix, change required-vs-optional flags for new entity kinds. Removing rows or changing existing required flags needs an RFC (it breaks existing entities).
  • SWE lead. Can add components to the UI Kit, add CI gate scripts, tweak the search-index builder. Renaming a kit component or removing one needs an RFC (it breaks downstream pages).
  • SRE. Can edit Runbooks, runbook-section requirements, audit cadence intervals. Removing a runbook needs an RFC.
  • QA. Can edit any glossary entry tied to testing terminology, add QA templates. Changing the test-case template shape needs an RFC.

Apply

04

Escalation paths

Two scenarios where escalation kicks in.

  1. Stale page, owner unresponsive.

    After the two-week re-verify window (freshness §04), the page is escalated to the owner's manager via the role chain above. If still unresolved within another week, the SA lead opens a deprecation ADR.

  2. Contributor disagrees with the Accountable.

    First, the Consulted role(s) for that artefact weigh in (e.g. Architect for a UI Kit dispute). If still deadlocked, an RFC opens. The Accountable does not have unilateral veto — they have direct-edit authority on small changes, but disagreements escalate to consensus, not to one-person decree.

Rationale

05

Tradeoffs of an explicit RACI

Wins

No invisible ownership

Every artefact has exactly one Accountable. There's no «I assumed somebody else owned the glossary». Onboarding is faster — new contributors know who to ask.

Cost

Small teams wear many hats

On a 3-person team the SA lead is also the Architect is also the SWE lead. The matrix still has value — it names which hat applies for which artefact — but the role-vs-person distinction matters less.

Wins

Escalation is mechanical

When something stalls, the next step is named. No improvising «who do we ping?».

Cost

RACI is a living document

If roles re-org, the matrix needs updating. We accept the maintenance cost — but it does mean someone (the SA lead) is Accountable for keeping this page itself fresh.
