🧩 Principle 07 · Catalog

Catalog taxonomy — eight kinds, one descriptor

We borrow the Backstage Software Catalog taxonomy to answer one question: what kind of thing is this page about? Every entity in the portal falls into exactly one of eight kinds, with a fixed YAML envelope and an explicit relations graph. We borrow the model — not the runtime; the tradeoff is honest.

Rules

01

The eight kinds

Five describe what we ship and operate. Three describe the people and pointers around them. We don't invent new kinds — see the tradeoffs section for why.

🧩

Component

Software you can build & ship

Anything with a repo and an artefact. Services, websites, frontends, libraries — distinguished by spec.type. Today: API, Portal, Monitoring, UI Kit.

spec.type: service · website · library · tool

🏗

System

A group of Components that act as one product

Holds the C4 L1 landscape and portal-level ADRs that affect multiple Components. Owns Components but doesn't deploy anything itself. Today: Study App.

🔌

API

A contract between Components

First-class entity, not a sub-page. One Component provides the API; others consume it. The definition is OpenAPI / AsyncAPI / proto — never hand-written prose.

spec.type: openapi · asyncapi · grpc · graphql

💾

Resource

Infrastructure that Components depend on

Databases, queues, object stores, caches. Has operational concerns (runbook, backup, on-call) and a consumer list — but no Diátaxis quadrants.

spec.type: database · queue · bucket · cache

🌐

Domain

A business area that contains Systems

Optional grouping one level above System. Today we have one Domain — the whole product — but we keep the kind so the catalog grows without restructuring.

👥

Group

A team or org unit

The default owner of any entity. backend-team, sre, qa. Groups nest (spec.parent) and have members. Prefer Group as owner — people leave, teams persist.

👤

User

A single person

Login, name, email. Referenced by Groups (membership) and as the maintainer of a page or the on-call for a Component. Almost never used as an entity owner directly.

📍

Location

A pointer to other catalog files

Bootstrap entity that says «also load every catalog-info.yaml under this glob». Lets the catalog grow by adding folders, not by editing a master list.

This page defines what each kind is. For the visual card spec — which fields are required / optional / forbidden per kind — see 🪪 Entity card.

Rules

02

The YAML envelope

One Entity = one thing the portal knows. Every entity has a kind, metadata (name, description, owner, links), a spec (sub-type, lifecycle, relations), and lives in one catalog-info.yaml descriptor next to the thing it describes.

All kinds share the same outer shape — apiVersion, kind, metadata, spec. Only spec differs by kind. A realistic minimal Component:

apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: api                          # unique within the catalog (kebab-case)
  title: API                         # human label for the UI
  description: REST gateway for Study App
  tags: [backend, go, http]          # free-form, searchable
  labels:
    tier: "1"                        # key/value, indexable filter
  annotations:                       # integrations: read by plugins
    github.com/project-slug: org/study-app
    backstage.io/techdocs-ref: dir:.
spec:
  type: service                      # Component sub-type
  lifecycle: production              # experimental | production | deprecated
  owner: group:backend-team          # Group (preferred) or User
  system: study-app                  # partOf — the System above
  providesApis:
    - study-app-public
  dependsOn:
    - resource:postgres-primary
    - component:monitoring

Field roles, in one line each. tags are free-form keywords for search; labels are typed key/value pairs the catalog can filter on; annotations wire the entity to external systems (GitHub, PagerDuty, Grafana, TechDocs); links are «open this in a new tab» for humans.

Rules

03

Relations — turning YAMLs into a graph

Every relation has an inverse, so the portal answers both «what does this depend on?» and «who depends on me?» from the same data.

You write Backstage computes the inverse What it means
spec.owner ownerOf Group or User on the hook for the entity. Lives on every kind.
spec.system hasPart Component or API belongs to a System.
spec.subcomponentOf hasSubcomponent Smaller Component inside a larger one (e.g. a frontend inside a service).
spec.dependsOn dependencyOf Runtime or build dependency on another Component or Resource.
spec.providesApis apiProvidedBy Component publishes this API contract.
spec.consumesApis apiConsumedBy Component calls this API contract.

References use kind:name — group:backend-team, resource:postgres-primary, api:study-app-public. A bare name is allowed when the kind is unambiguous from context (spec.owner: backend-team implies a Group).

Rules

04

Lifecycle states

spec.lifecycle tells the catalog (and CI) how strictly to enforce the spine — which pages each kind must, may, or must not have. Four states; «alpha» / «beta» / custom values aren't allowed. (Backstage's canonical list is three — experimental / production / deprecated — we add stable as a distinct middle state because for us «contract is firm but SLO isn't paged yet» is a common, real state of a service.)

State What it means What CI requires
experimentalEarly; shape can still change; no SLO; no on-call.Only Overview and Explanation.
stable Contract is firm; real consumers; SLO target written but not paged yet.Full spine except on-call & runbooks (warn, not block).
production Real users, real SLOs, paged on failure.Full spine — every ✅ row in spine matrix, including on-call & runbooks.
deprecated Going away; new consumers forbidden.Deprecation banner with sunset date + migration target.

Rationale

05

Tradeoffs & the «Backstage taxonomy without Backstage» question

The single hardest question an architecture lead asks here. We answer it head-on.

Wins

Off-the-shelf mental model

Any engineer who has worked with Backstage already knows the kinds, the YAML shape, and the relations vocabulary. Onboarding cost is near zero for hires from Spotify, Adobe, Splunk, Roblox.

Cost

We use the taxonomy, not the runtime

We don't run a Backstage server — no plugin marketplace, no TechDocs auto-rendering, no scaffolders. The model is just YAML files we read with our own scripts. The benefit is portability (we can adopt Backstage later); the cost is rebuilding plugin-shaped features by hand.

Wins

Closed taxonomy = enforceable spine

Eight kinds is small enough that CI can lint exhaustively. A page that doesn't belong is mechanically rejectable; the reviewer doesn't have to argue taste.

Cost

spec.type is the escape hatch

We claim «no new kinds», but we do invent sub-kinds via spec.type (service, website, library, tool). Be honest: the contract is «no new top-level kinds» — sub-kinds are the pressure-release valve, and they will multiply over time.

Wins

Graph from the same source

Forward and inverse relations are computed from the same field, not maintained in two places. The «who depends on me?» query that breaks in 80% of doc systems is a single map-reduce here.

Cost

One-Domain feels silly today

We have one Domain — the whole product. It's a smell. We keep the kind for forward-compatibility, but a reviewer is right to ask «what triggers a second Domain?» The answer: when two Systems no longer share a single PM / VP / budget line.
