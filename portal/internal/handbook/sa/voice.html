🗣️ Style guide · Voice

Voice — plain language & global style

The portal is read by engineers whose first language is rarely English. Voice rules keep the prose predictable: short sentences, common words, one idea per sentence, one name per concept. Reviewers reject pages that break these rules — including pages written by an LLM that wandered.

Rules

01

Plain language

Stay formal and accurate — but prefer plain words and short sentences over academic phrasing.

  • Sentence length. One main idea per sentence. Split chained clauses into separate sentences.
  • Vocabulary. Pick the common word when both are accurate. «Use», not «utilize». «Show», not «surface».
  • Jargon. Keep necessary technical terms; spell out every acronym the first time it appears on a page.
  • Voice. Active voice for instructions: «Run make verify.» Passive only when the actor doesn't matter.
  • Nouns. Don't stack noun phrases. «The pipeline runs tests» beats «the test-execution phase of the pipeline».
  • Consistency. When one row of a hub page is simplified, every row gets the same treatment.

Rationale

02

Rewrite examples — before & after

Three real drifts caught in PR review and the one-line fix each got.

Before — noun-stacked

«The configuration value resolution preference cascade order rule.»

After — verb sentence

«How the system resolves a config value, in order: env var → file → default.»

Before — passive + jargon

«The deserialization phase is initiated upon receipt of the inbound payload by the orchestrator.»

After — active + plain

«When the orchestrator receives a payload, it parses it.»

Before — marketing tone

«Effortlessly seamless modern API for next-generation observability.»

After — engineering tone

«Push metrics over HTTP; returns 202 on success, 4xx on schema mismatch.»

Rules

03

Global style rules

Cross-cutting rules that apply to every page, regardless of document type.

  • Language. English. Plain engineering prose. No slang, no marketing copy.
  • Tone. Directive. State what is required, what is optional, and what is forbidden.
  • Tense. Present tense for current rules and behaviour. Future tense only for planned work.
  • Heading case. Sentence case. Acronyms keep their canonical casing (API, gRPC, SRE, AsyncAPI).
  • Terminology. One term per concept, repeated everywhere. Pick «OpenAPI contract test» and never write a variant.
  • Links. Name the destination, not the action. Write «ADR 0027», not «Open ADR 0027».
  • Commands and paths. Wrap in <code>. Examples must be safe to copy and paste.
  • Lists. Ordered for procedures; unordered for policies and reference rows.

Rules

04

Naming & linking conventions

  • File names. Lowercase kebab-case. Numeric prefix when ordering matters (ADRs, RFCs, runbooks).
  • Operation page names. HTTP method + path pattern, e.g. post-api-v1-user.html.
  • Link labels. Stay consistent inside one index — use ADR 00XX everywhere; don't mix with «Open ADR 00XX».
  • Index descriptions. One sentence per row, the same grammar across rows.
  • Anchor IDs. Kebab-case nouns; never sentence-shaped (#why-h not #why-do-we-do-this).

Rules

05

Voice by Diátaxis quadrant

The §§ 01–04 rules are universal. The four Diátaxis quadrants then bend those rules in opposite directions — a Tutorial that discloses tradeoffs has lost the learner; an Explanation that hides them has lost the architect. One row per quadrant; if a page reads as the wrong voice, that is a quadrant-fit problem, not a prose problem.

Quadrant Stance Mood & person Tradeoffs & alternatives Failure mode
Tutorial Teach by doing — the reader follows along on a real machine. Imperative second-person («Run make verify»). Present tense throughout. Suppress. A learner cannot evaluate a trade-off they have not yet experienced. Side-roads are forbidden. Drifts into encyclopaedia tone: «There are three ways to install dependencies…». Stops being a tutorial.
How-to Get the reader from A to B — they already know what B is. Imperative second-person. Action verbs in step headings («Create the migration», not «Migration creation»). Mention only when load-bearing. If two methods are common, name both with the choice criterion in one line — do not justify. Becomes Explanation in disguise — three paragraphs of background before the first command. Reader bounces.
Explanation Disclose the model — the reader walks away with «I now understand why». Indicative third-person («The orchestrator deserialises…»). Authorial voice is permitted («we chose X because…»). Mandatory. Every Explanation page carries a § Tradeoffs with at least two named «we gain X, we lose Y, we accept it because Z». See Tradeoff disclosure. Reads like a textbook chapter rewritten in our names — no opinion, no rejected alternatives, no «what we would do differently». Substance fails.
Reference State the facts — the reader looks up one specific thing and leaves. Indicative third-person. Tabular, terse, no narrative connectives between rows. State as § Limits. Reference does not justify; it names the ceiling, the supported versions, the failure modes. Justification belongs in a sibling Explanation page. Drifts into prose between the rows — reads like a long explanation with tables in it. Search loses its grip.

Cross-quadrant rule. The same engineering fact can appear in all four quadrants — and it must sound different in each one. A reader who lands on the Explanation page expects to be argued with; a reader on the Tutorial expects to be carried. If a page is unclear which quadrant it lives in, the voice gives it away.
