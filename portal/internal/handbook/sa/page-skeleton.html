📐 Style guide · Page skeleton

Page skeleton — markup contract, mobile, history

The HTML contract every page in the portal must satisfy. Five required slots in the skeleton, a fixed set of mount points the docs runtime injects into, mobile rules CI enforces, and the Page-history block that ledgers every meaningful edit. If a reviewer can't find one of these, the PR doesn't merge.

Rules

01

Required page skeleton

Every non-redirect page carries these five slots in this exact order. Redirect stubs are the only exception (they ship one «Page moved» card + the canonical link).

  1. Topbar & breadcrumb. The shared <header class="topbar"> with the breadcrumb trail ending at the current page.
  2. Sidebar mount. <aside data-component="sidebar" data-nav-src="…nav-tree-internal.json"> — sidebar JS hydrates it.
  3. Hero / page-head. <header class="page-hero page-head"> with eyebrow, <h1>, optional CTAs.
  4. Body sections. One topic per <section class="numbered-section">. Each carries an .numbered-section__head with kicker + <h2 id>.
  5. Page-history footer. <footer class="docs-history"> with at minimum one row per meaningful edit.

The in-page TOC (<aside class="docs-toc">) and the report-bug FAB are injected by the runtime in older pages or authored explicitly in new pages — both options are valid.

Rules

02

Frontend mount contract

Every hand-written page declares the mounts and data-* attributes below. The list is the contract between the page and the shared docs runtime.

Feature Required markup Notes
Top navigation & quick actions <div id="docs-top-nav"></div> Mount for top nav, quick actions, command palette.
Page-editors widget data-maintainer-ids on <body> Renders an avatar stack. Clicking «Page editors» opens the full maintainer list.
Profile maintained pages <div id="portal-maintained-mount"></div> + data-person-id on <body> Used on team/people/*/index.html. Includes pagination and a skeleton state.
Author chip data-component="author-chip" data-person-id="…" Used inside history rows. Hydrated by sidebar JS.
Sidebar <aside data-component="sidebar" data-nav-src="…"> Reads nav-tree-internal.json and renders the left nav.
Auto in-page TOC — Injected by the runtime if absent. Opt out with <main data-inpage-toc="off">. Desktop-only (min-width: 1025px).
Report-bug FAB — Injected by the runtime. Do not author <button class="docs-report-bug-fab"> by hand.

Rules

03

Authoring for mobile

The portal must remain readable on a phone (≈ 360–440 CSS px) — no shrunk type, no horizontal overflow. CSS handles most of the work. Five rules cover the cases authors still get wrong; reviewers reject pages that break them.

  • Wrap multi-column tables. Any table with three or more columns, or any column with long identifiers / code, lives inside <div class="docs-table-scroll" role="region" aria-label="…">…</div>. The aria-label is required for screen-reader navigation.
  • Don't paste raw long URLs into prose. A URL longer than ~50 characters belongs inside a named anchor: <a href="…">readable label</a>.
  • Block <pre> for code longer than ~20 characters. Inline <code> is for short tokens; long inline code crowds the content column on a phone.
  • Don't add a breakpoint below 760 px without a written reason. Canonical tiers: ≥ 1025 / 761–1024 / ≤ 760. A 640/720/900 px rule is allowed only as a layout reflow inside an existing tier, never as a new device tier.
  • Test on a phone-sized viewport before merge. Run the mobile-contract acceptance smoke on iPhone 17 Pro Max (440 px) and iPhone SE (375 px). Failing any item blocks merge.

Every page declares its mobile contract in a normative #mobile-contract block at the bottom — the spec for what reflows at each tier. See example below ↓

Rules

04

Page-history rules

Every page ends with the history block — the ledger that lets a future reader see when something was last verified, and by whom. The skeleton mounts it as the last block before the in-page TOC; the contract for what goes inside lives on a dedicated Reference page.

  • Required on every non-redirect page. Mount as the last child of <main class="container">, after the <article class="docs-prose"> closes.
  • Cell schema → SA reference · History format. Field vocabulary (docs-history__date · docs-history__author · docs-history__note), the ISO date constraint, the author-chip hydration contract, the reverse-chronological order rule, and the legal / illegal note examples — all canonical there. Don't redefine them here.
  • Procedure → Authoring · Add a history row. When a row is worth adding, the mechanical steps, anti-patterns reviewers reject, the verify checklist.

05

Canonical example — the minimal page

A real, working skeleton you can copy. Everything that's required is present; nothing that isn't is. 

<header class="topbar"> … breadcrumbs … </header>
<aside data-component="sidebar" data-nav-src="…nav-tree-internal.json"></aside>

<main class="container">
  <article class="docs-prose">

    <header class="page-hero page-head">
      <p class="page-hero__eyebrow">Section · Subsection</p>
      <h1 class="page-hero__h1">Page title</h1>
      <div id="docs-top-nav"></div>
      <p class="page-hero__lede">One-sentence promise.</p>
    </header>

    <section class="numbered-section" aria-labelledby="topic-h">
      <div class="numbered-section__head">
        <span class="numbered-section__kicker">01</span>
        <h2 id="topic-h">One topic</h2>
      </div>
      <p class="numbered-section__lede">What this section covers.</p>
      … body …
    </section>

  </article>

  <footer class="docs-history" aria-label="History">
    … one row per meaningful edit …
  </footer>

  <section id="mobile-contract" aria-hidden="true" hidden>
    … normative tier rules …
  </section>
</main>
