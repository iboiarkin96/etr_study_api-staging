🔁 Principle 03 · Lifecycle

Process — how docs ship & stay alive

Pages live in this repository and ship through the same pipeline as the code they describe. This is the procedural deep-dive of Principle 03 — eight operating rules, the ten-step lifecycle, the Definition of Done, the audit cadence, and the anti-patterns that break the loop.

Gate legend

AUTO Mechanical gate. A script catches the violation in pre-commit or CI — before any human looks at the PR.
REVIEW Judgement gate. A reviewer rejects the PR in code review if the item is missing or unclear.

Stance

01

Operating principles

Eight rules. Each rule answers «why does this process look the way it does?»

  1. Single source of truth. One topic = one canonical page. Other pages link to it; they never re-state it. If two pages contradict, fix the canonical one and update the others to point at it.
  2. Co-located with code. Docs live in the same repository, on the same branch, and change in the same PR as the code they describe. There is no separate doc CMS or wiki.
  3. Living, not archived. A page is either current or replaced. No «deprecated but kept around» pages — outdated content is removed in the same change that supersedes it.
  4. Plain language. Pages are read by engineers whose first language is rarely English. Short sentences, common words, one idea per sentence — see 🗣️ Style guide.
  5. Glossary first. No new domain term lands without a definition added to the glossary in the same PR. Term consistency is the foundation of every other doc.
  6. History is required. Every page ends with a History block. Every meaningful edit appends a row — the ledger that lets a future reader see when something was last verified.
  7. Diagrams are PlantUML. Every diagram is authored as a .puml source and rendered to SVG. No Mermaid, no draw.io, no screenshot-as-diagram — see ADR 0020.
  8. Definition of Done includes docs. A code change that touches user-visible behaviour is not done until the corresponding doc is updated in the same PR. See § 03 below.

Apply

02

Lifecycle of a doc change — 10 steps

From «something changed» to «merged into main». Every doc edit walks these ten steps; if you skip one, the gate at the end catches it.

  1. Trigger.

    A change happens in code (new endpoint, behaviour tweak), an audit flags drift between docs and reality, or the team decides to start documenting an area that wasn't covered.

  2. Identify the canonical page.

    Find the existing page that owns the topic; if no canonical page exists, create one from a template.

  3. Cut a feature branch.

    git switch -c docs/<slug>. The draft lives there, no one else sees it yet.

  4. Edit only what changed.

    Preserve the page skeleton (Hero → sections → History). Plain language; reference the style guide when in doubt.

  5. Update the glossary if any new term appears.

    Add a row with a plain-language definition in the same PR. Cross-link from the page.

  6. Append a History row.

    One sentence summarising what changed; today's date in YYYY-MM-DD; link to the maintainer's portal page.

  7. Self-audit against the Definition of Done.

    Walk § 03 below before requesting review. If any item is «not sure», stop and fix that first.

  8. Open a pull request.

    gh pr create. The automated gate set runs in pre-commit and CI — every AUTO check below must pass before a reviewer engages.

  9. Review & audit.

    The manual gate set runs here — an LLM walks canned audit prompts and a human reviewer signs off every REVIEW check below.

  10. Merge into main.

    Squash-merge. The page is live on the portal, the Pages workflow rebuilds the Pagefind bundle (tools/docs/build_pagefind_index.py) before deploy, and the history block records the change.

Rules

03

Gates — automated & manual

Every doc PR walks two gate sets. AUTO gates are mechanical checks — they run in pre-commit and CI, fast, deterministic, no human required. REVIEW gates need judgement — quadrant fit, tone, tradeoff honesty — and live in the PR review thread. Both gate sets must pass before merge.

AUTO Mechanical gates

pre-commit + CI · scripts/check_*.py
  • Page structure. check_h1_structure.py · validate_docs_html.py · ensure_docs_page_history.py · ensure_docs_maintainers.py — exactly one <h1>, breadcrumb chain, history block, maintainer attribution.
  • Links & references. check_asset_refs.py · check_adr_rfc_links.py · check_path_literals.py · audit_front_docs_links.py — every href / src resolves; ADR/RFC anchors are valid; no stringly-typed paths in JS/JSON/CSS.
  • Visual contract. validate_docs_design.py · check_css_vars.py — pages use UI Kit components and v3 tokens; no rogue hex codes, no missing CSS variables.
  • Accessibility. validate_docs_a11y.py — WCAG 2.1 AA: alt text, ARIA, focus order, contrast, reduced-motion handling.
  • Catalog spine. check_spine_completeness.py — every lifecycle: production entity carries every ✅ page from the spine matrix; rejects pages forbidden for the kind (e.g. a runbook on a Library).
  • Discoverability. build_pagefind_index.py · check_pagefind_visibility.py — Pagefind bundle rebuilt on every make docs-fix / Pages deploy; visibility:internal|public filter must classify every URL correctly.
  • Hygiene. format_docs_html.py · link_page_history_authors.py — HTML normalised; author chips wired to portal profiles.

REVIEW Judgement gates

PR review · LLM prompts + human
  • Diátaxis fit. The page satisfies exactly one quadrant — Tutorial · How-to · Reference · Explanation. If it straddles two, the reviewer asks for a split.
  • Voice. The first paragraph reads aloud cleanly — short sentences, active voice, common words. No noun stacks, no marketing language.
  • Glossary discipline. Every new domain term is defined in the glossary in the same PR. Reviewer flags any acronym that lacks a glossary entry.
  • Tradeoffs honesty. For Explanation pages, the «cost» section is as visible as «wins». Pages that argue only the upside are sent back.
  • LLM audit. The canonical audit prompt from Style guide § Audit runs against the changed pages; the LLM's findings land as PR comments.
  • Substance sign-off. A human reviewer confirms the page matches reality — runbooks actually recover the failure mode they claim, API spec matches the code, etc.

Definition of Done

A change is not «done» until every item in the checklist is green. The full 11-item checklist with gate tags lives as the self-check procedure in Authoring → Definition of Done self-check — contributors walk it before requesting review.

05

Audit cadence

Docs drift even behind the PR gates. Four cadences catch drift before it becomes wrong-by-default — every audit lands under governance/audits/.

Cadence What gets checked Output
Per PR AUTO Full mechanical gate set runs in pre-commit + CI (links · h1 · a11y · visual contract · spine).
REVIEW Diátaxis fit · voice · glossary · tradeoff honesty · LLM audit prompt. 		Inline PR comments; «request changes» on any failed gate.
Quarterly · per service REVIEW The listed owner re-runs the acceptance checklist of each owned service spec and verifies the spine matrix still matches reality. Audit PR fixing any divergence; row in governance/audits/maturity/.
Per release AUTO Search index rebuilt + sub-set check.
REVIEW Grep new code for unfamiliar terms; add to glossary or fix the naming. 		Glossary PR in the same release branch.
Semiannual · ADR alternatives pass REVIEW Walk every published ADR; flag any whose «Alternatives considered» block reads as «we decided X» without two named rejected options. Strong exemplars (e.g. ADR 0008, 0020, 0029) are not re-opened. Refactor PRs for each flagged ADR; audit summary under governance/audits/adr-alternatives/.
Annual REVIEW Architecture map cleanup — retire features that no longer exist, reassign ownership, audit broken cross-links. Maturity audit under governance/audits/maturity/.

ADR audit pass — semiannual procedure

ADR quality drifts faster than any other artefact in the portal because the decision is recorded once and never re-read. After 12–18 months a third of the catalogue typically reads as «we decided X» without naming the rejected paths — the ADR is recorded but not auditable. The semiannual pass catches this before it compounds. Owner: SA. Calendar slot: first working week of January and July.

  1. List every ADR. ls services/portal/internal/governance/adr/*.html; the count is the audit cohort. Skip 0000-template.html.
  2. Open each ADR and grade the alternatives block. Three buckets: strong (≥ 2 named rejected with one-line «rejected because…»), thin (1 named or vague phrasing), missing (no alternatives block at all).
  3. File the thin / missing list. One row per ADR in governance/audits/adr-alternatives/<yyyy-mm>.html with current grade and target grade. Strong ADRs are listed too, as the canonical exemplar set.
  4. Reach back to the original author for each flagged ADR — they have the rejected options in memory; do not invent them from outside. If the author has left, ask the current owner of the affected component what alternatives they would have rejected.
  5. Refactor in small batches. Three ADRs per PR, with the audit page updated in the same PR. Each refactor names at least two rejected alternatives with a one-line «rejected because…» each — see Tradeoff disclosure.
  6. Close the audit. When the thin / missing list is empty, publish the audit page under governance/audits/adr-alternatives/ with the final summary row («N audited · M strong · 0 thin · 0 missing») and the date of the next pass.

Anti-pattern. Do not invent rejected alternatives retroactively if no one remembers considering them. A fabricated «rejected because…» is worse than an empty block — it lies to future readers about the decision space. If memory is genuinely lost, record «historical: alternatives not captured at decision time» and move on. The audit cohort for the next pass is the same ADR.

Apply

06

Anti-patterns to avoid

The most common ways the process breaks. Catch them in review, not next quarter's audit.

  • «Docs PR follows.» Code merges with the promise that docs will land later. The follow-up almost never ships. Always bundle.
  • Two pages, same topic. Splitting a topic across two pages because each one is «almost about it» creates contradictions. Merge or pick one canonical home.
  • Stale screenshots / diagrams. A diagram of a screen or architecture that no longer looks like that. Either refresh in the same PR or remove.
  • Empty History. A page with no history block is invisible to future maintainers and quietly rots. Always add the block, even on the first commit.
  • Jargon without a glossary entry. A page introduces «the conspectus orchestrator» with no definition → future readers either guess or ask. Define it.
  • «TBD» / «WIP» in shipped docs. If a section isn't ready, omit it. Placeholder text encodes ambiguity.
  • Inconsistent terminology. Same thing called three different names across three pages. Pick one and unify in a single PR.
  • Hand-edited generated content. If the page is generated from OpenAPI / pdoc, never hand-edit the output — fix the source.

Rationale

07

Tradeoffs of the docs-as-code process

The question an engineering lead will ask: «what does this process cost us in delivery speed?» The honest answer:

Wins

Drift is bounded

A code change without a doc change fails the gate. The window where docs can lie about reality is one PR-review-cycle long, not «however many quarters until someone notices».

Cost

Every PR is slower

A 20-line code change with a doc update is genuinely heavier than a 20-line code change. We accept ~15% PR-cycle overhead; below that the gate isn't worth the friction, above that the docs would lie.

Wins

No separate doc system to learn

Authoring is git + HTML + the templates folder. Anyone who can ship code can ship docs. No CMS login, no separate review tool.

Cost

Non-engineers can't edit easily

A PM or designer who wants to fix a typo has to open a PR. We mitigate with «typo» labels that bypass review, but the floor is still git literacy.
