Substance contract

The content half of the authoring contract — companion to Visual contract. Visual contract enforces how a page looks; substance contract enforces what has to be inside the page for it to read as «mature-team docs», not as «well-formatted notes». Form is the cheap half of the contract; substance is what an architect from Google or Stripe actually reviews when they land on a page. Apply to every Explanation, ADR, RFC, and normative Reference page.

TL;DR

A page is well-formed when the eight form-principles pass. A page is substantive when a senior engineer, after reading it once, can reconstruct the model, see the rejected alternatives, find measured numbers, follow specific citations, and detect production-tested wisdom. Five tests, applied to every Explanation page, every ADR, every RFC, every Reference page that promises invariants.

Rules

The five substance principles

Three are about what the page exposes — model, tradeoffs, measurements. Two are about how the page anchors itself in reality — citations and production wisdom. Apply all five whenever the document is non-procedural (Explanation, ADR, RFC, normative Reference). Tutorials and How-to pages are exempt — their substance lives in the form-principles.

🧠 01 · MODEL

Expose the model

Reader must be able to reconstruct why the system is shaped this way, not just what it does. Good docs describe; world-class docs lay bare the thinking behind the description.

See test
⚖️ 02 · TRADEOFFS

Disclose tradeoffs honestly

Every non-procedural page names what the choice costs. Every ADR lists at least two rejected alternatives with one-line «rejected because…». No silent wins — every decision is a trade.

See test
📏 03 · MEASUREMENTS

Show measurements, not assumptions

SLOs carry numbers. Capacity carries numbers. Latency carries p50/p95/p99. If a claim about behaviour is in the page, the number that backs it (or the test that produced it) is in the page too.

See test
🔖 04 · CITATIONS

Cite specific pages, not industries

Not «SRE Book». «SRE Workbook ch. 5 «Alerting on SLOs», pp. 88-93». Not «Backstage taxonomy». «Backstage Component kind v1.0, fields type and lifecycle». Citations are load-bearing.

See test
🔥 05 · PRODUCTION

Production-tested wisdom

«We saw X in production» weighs more than «X usually happens». SRE Explanation pages, runbooks, and any page about incident response carry at least one «what we actually saw» line per major claim.

See test

Rules

01

Expose the model

A page passes this test when a reader who knows nothing about the system can, after reading once, sketch the components on a whiteboard and explain why each one exists. If they can repeat what the page said but cannot reconstruct the underlying decisions, the page failed.

What «exposing the model» looks like

  • Name the forces. What constraints shaped this part of the system? (latency budget? team size? legacy compatibility? compliance?) Forces appear before solutions.
  • Show the seam. Every non-trivial system has a riskiest seam — the interface where one bad assumption breaks everything. Name it. Explain why it's that seam.
  • Surface the invariants. What stays true no matter what the system does? (e.g. «conspectus.created_at ≤ conspectus.updated_at always»). Invariants are model — without them the page is just description.
  • Distinguish primitive from compound. Which concepts are atomic in our model? Which are derived? Reader needs the type hierarchy to think.

Anti-pattern: «narrate the code»

A page that reads as «module A calls module B, which writes to table C» describes the system but does not model it. The reader learns the call-graph and forgets it. Replace with «the read-path optimises for X at cost of Y — that's why module A bypasses the cache». Now the reader can predict what would change if X mattered less.

How to apply

  • Open any Explanation page you wrote. Cover the diagrams and code blocks. Re-read.
  • If only narration remains — the page is description, not model. Add the «why this shape».

Rules

02

Disclose tradeoffs honestly

Every architectural choice costs something. A page that lists only the benefits reads as marketing. The Architect's Bar test is: an experienced reader should be able to say, within 30 seconds, what this design gives up to be what it is.

The mandatory shape

BlockRequired forMinimum content
§ TradeoffsEvery Explanation pageAt least 2 named tradeoffs, each as «we gain X, we lose Y, we accept it because Z».
§ Alternatives consideredEvery ADR · every RFCAt least 2 rejected options, each with one-line «rejected because…». Empty block ⇒ ADR fails review.
§ LimitsEvery Reference page that promises behaviourWhere the promise breaks: load ceiling, supported versions, known failure modes.
§ Would we do this again?ADRs older than 6 months, on revisitOne-line retroactive «still right» or «here's what we'd change». Honest is more valuable than confident.

See the dedicated reference: Tradeoff disclosure — the section template, the anti-patterns, the canonical examples (ADR 0008, 0020, 0029) and the audit checklist for retroactive passes.

Anti-pattern: «alternatives: none considered»

ADRs that ship with empty or «we chose X because it's right» alternatives blocks rot fastest. Six months later nobody remembers the constraint set, so the decision can't be revisited without re-running the whole analysis. The «rejected because…» one-liner is what makes the ADR auditable, not just recorded.

Rules

03

Show measurements, not assumptions

A claim about behaviour without a number is a guess. Every SLO carries the target and the burn-rate windows. Every capacity claim carries the load that produced it. Every latency claim carries p50, p95, p99 and the percentile horizon.

The measurement contract

  • SLOs — target number (e.g. 99.5%), error budget, multi-window burn-rate table. handbook/sre/slos.html is the canonical example.
  • Capacity — RPS at which saturation begins, headroom buffer, the drill that produced the number. Without a drill, no claim.
  • Latency — p50 / p95 / p99 with the percentile horizon (last 24 h? last 7 d?). A single «p99 < 400 ms» without horizon is meaningless.
  • Throughput / concurrency — sustained vs peak, the duration over which «sustained» applies.
  • Data sizes — current row counts for the tables that gate growth, plus growth rate.

Anti-pattern: aspirational numbers

«We support 10k RPS» without a drill that proved it is worse than no number at all — it's a false signal. Replace with «drill on 2026-04-12 sustained 4k RPS at p95 < 300 ms; we designed for 4× headroom, but the headroom is unmeasured». Honest, useful, auditable.

Rules

04

Cite specific pages, not industries

A page that references «the SRE Book» tells the reader nothing — there are two SRE books, each 500 pages. A page that references «SRE Workbook ch. 5 «Alerting on SLOs», pp. 88-93» is auditable, falsifiable, and earns the reader's trust.

Citation tiers

  • Tier A — pinpoint. Book + chapter + page range, or paper + section, or RFC + section number. Use for every load-bearing claim.
  • Tier B — anchored. Page-level URL to a stable spec / canonical doc (W3C, IETF, vendor docs with a date). Acceptable for procedural detail.
  • Tier C — informational. Blog posts, talks, X / Mastodon threads. Acceptable only as background, never as the basis of a normative claim.

Rules

  • Every ADR carries at least one Tier A or B citation in its «References» block.
  • Every Explanation page that names an industry pattern (Diátaxis, C4, SLO, STRIDE, RACI) anchors it to its canonical source.
  • «Industry standard X» without a citation is a citation gap — flag for refactor.

Example: good vs sloppy

Sloppy

«We use the SRE Book's burn-rate pattern for alerting.»

Good

«We use the multi-window, multi-burn-rate pattern from Site Reliability Workbook (Beyer et al., 2018), chapter 5 «Alerting on SLOs», page 91, table 5-2. The two windows are 1 h (fast burn) and 6 h (slow burn); the burn-rate thresholds are 14.4 and 6 respectively.»

Rules

05

Production-tested wisdom

A page that reads as a re-statement of the SRE Book signals «we read books». A page that names a real incident, a real false-positive alert, a real dashboard signals «we lived through it». The second is what mature-team docs sound like.

Where production wisdom appears

  • SRE Explanation pages (observability · logging · monitoring · slos) — at least one «what we've actually seen» note per major section.
  • Runbooks — at least one «non-obvious symptom» entry, drawn from a real incident.
  • Postmortems — the entire genre is production wisdom. Zero on file = zero signal.
  • ADRs revisited after 6 months — the «would we do this again?» one-liner is the cheapest form of production wisdom you can buy.

Shape of a production-wisdom line

What we saw:    <the surprising observation>
What we learned: <the model update that came from it>
What we changed: <the code / alert / dashboard / runbook that captured it>

Anti-pattern: «textbook in the third person»

A logging page that walks through «logs should be structured, levels should be consistent, correlation IDs should propagate» is correct and useless — every team already read it. Replace with «we lost three hours to an outage because request_id stopped propagating across the worker boundary; we now treat propagation as a unit-tested invariant, see tests/logging/test_correlation.py». Now the reader has a piece of your team's actual experience.

Apply

06

The substance checklist

Run this when opening a PR for any non-procedural page (Explanation, ADR, RFC, normative Reference). Five gates. State persists in localStorage — step away mid-review and pick up where you left off.

0 / 5 gates passed

Stop rule. If any gate fails for a non-procedural page, the page is form-compliant but not substantive. Either fix the gap before merge, or mark the page data-substance-status="draft" so the next audit pass picks it up.
