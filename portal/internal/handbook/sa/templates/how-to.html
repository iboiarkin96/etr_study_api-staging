How-to template

Canonical shape for a how-to page (Diátaxis quadrant: task-oriented). Copy this skeleton, replace the placeholders, and keep the section order intact — every how-to in this portal answers a single, well-scoped «how do I X?» question with the same scaffolding.

When to use this template

  • The reader is not a beginner in the domain — they already know what they're trying to do and need the recipe, not a rehearsal.
  • There is a single, well-named task behind the page: «Add a write endpoint», «Rotate a token», «Run a load test». If you cannot name the task with a verb, pick a different quadrant.
  • The task has a verifiable outcome — a check command, a passing test, a status code, a green CI gate.
  • The task may legitimately have conditionals («if Postgres → A, if MySQL → B»). Unlike tutorials, branches are allowed and labelled.

If any of the above is false — pick a different quadrant: Tutorial (learning, beginner), Reference (lookup, exhaustive spec), or Explanation (understanding, trade-offs).

Hero (page-head)

Every how-to opens with the same hero block. Two pills, a verb-first h1, and a lede that states the goal and the smallest preconditions in one sentence.

<header class="page-head">
  <div class="docs-chip-row">
    <span class="docs-pill docs-pill--warn"><Domain></span>
    <span class="docs-pill docs-pill--neutral">How-to</span>
  </div>
  <h1><Verb> <the thing> (worked example optional)</h1>
  <div id="docs-top-nav"></div>
  <p class="lede">
    <What you'll have at the end>. Preconditions: <smallest precise list>.
    For the «why», see <link to Explanation>. For the full spec, see
    <link to Reference>.
  </p>
</header>

Domain pill colour follows the registry on the hub: warn for API, accent for Service, info for Docs, success for QA, danger for Incidents.

Title formula (verb-first)

Every how-to title starts with an imperative verb naming the task. The optional worked example or scope qualifier goes in parentheses.

  • «Add a write endpoint (worked example: POST /api/v1/contract)»
  • «Run an API load test»
  • «Verify an API endpoint against its contract»
  • «Rotate a service-account token»
  • «Add an Alembic migration»

Anti-titles (do not ship these):

  • «Local development» — noun phrase; rename to «Run the API locally».
  • «Accessibility testing playbook» — wraps the verb in a noun; rename to «Run an accessibility audit on a page».
  • «Docker image and container» — describes a topic, not a task; either split into Explanation + How-to or rename to «Build and run the API as a Docker image».
  • «Documentation pipeline (docs-as-code)» — describes an artefact; that is a Reference or Explanation page.

Lede formula

One sentence. State the deliverable and the smallest precondition. Link out for context, do not embed it.

  • «How to verify any HTTP operation under /api/v1/… against its internal specification. Line-by-line outcomes go in the API endpoint acceptance checklist.»
  • «The minimum bar before a release tag goes out. Smoke is broad and shallow: confirm the build is alive, golden paths still work, the docs portal still loads.»
  • «Register a new deployable artefact in the catalog. Write one catalog-info.yaml, run make catalog-render, the generator updates the entity-card HTML, hub tile, hero ticker, and nav-tree.»

Avoid leding with «In this guide we will…», «Welcome to…», or any context-setting paragraph. The reader already knows what they want; cut to the recipe.

Section skeleton

Every how-to uses the same five section anchors. Drop optional sections by leaving them out; do not reorder.

  1. <section id="before"> — Before you start. Three to six bullets: required tools, required permissions, required state (migrations applied, env vars set, branch checked out). Link the prerequisite tutorial or reference if the reader is missing context.
  2. <section id="steps"> — Steps. Numbered list, ≤10 top-level items. Conditionals labelled inline («If using SQLite: …»). Code snippets in docs-code blocks. Each step is one action with one outcome. Link to Reference / Explanation pages inline within the step text — this is the Diátaxis cross-quadrant hinge.
  3. <section id="verify"> — Verify. The exit check. A command, a status code, a passing test, an inspect-the-dashboard step. Without this section the page is not a how-to.
  4. <section id="troubleshooting"> — Troubleshooting (optional). Two to five common failure modes with one-line fixes. Not a full diagnostic flow — runbook territory.
  5. <section id="rollback"> — Rollback (optional, recommended for destructive tasks). The minimum sequence to undo the change.
  6. <footer class="docs-history"> — page history rows.

Verify block (canonical shape)

The verify block carries the contract: «if these checks pass, the task is done». Keep it copy-paste-runnable.

<h2 id="verify">Verify</h2>
<ul>
  <li><code><check command></code> exits 0.</li>
  <li><code>GET <url></code> returns <status> with <assertion on body>.</li>
  <li><CI gate> passes (<code>make verify</code>).</li>
  <li><Reference page> reflects the change (link).</li>
</ul>

For incident-response how-tos (runbooks), use the runbook skeleton instead: Trigger → Fast triage → Causes → Recovery → Exit criteria → Follow-up. See runbook template.

Voice rules

  • Imperative throughout: «Run», «Add», «Confirm», «Open». Never «давайте», «we will», «you can».
  • Second-person sparingly: «You'll see», «You should observe» — only where imperative is awkward.
  • Conditionals are allowed (unlike tutorials), but always labelled inline: «If SQLite: …», «If you skipped step 2: …».
  • Do not explain the «why» on this page. Link to an Explanation page instead. One-sentence rationale is OK; multi-paragraph context belongs in explanation/.
  • Cite real artefacts: link to actual ADR / RFC / runbook / checklist IDs so the recipe is anchored in production-grade context.
  • Code blocks copy-paste runnable: no $ prompt, no truncation. Real values where possible.

Anti-patterns

  • Multi-task megapage — one page = one task. If the page covers 5 tasks, split into 5 pages. Exception: a labelled collection page («Playbooks for docs-frontend changes») with deep anchors per playbook.
  • Reference disguised as How-to — if the page is an inventory («Commands by theme», «Generated paths table»), it is a Reference. Move it.
  • Explanation embedded inline — if a section explains a concept («Docker in one minute», «HICCUPPS-F heuristics»), hoist it to explanation/ and link out.
  • Template living in how-to/ — templates describe shape (a Reference activity). Templates live in handbook/templates/, never in how-to/.
  • How-to without Verify — without an exit check, the reader cannot confirm success. Always include Verify.
  • How-to duplicating a tutorial — same topic, two pages. Either the tutorial graduates into the how-to (learning loop done, recipe stays) or the how-to is collapsed back into the tutorial.
  • «Playbook:» prefix in the title — drop it. Either the page is a playbook or it is not; the chip row already says How-to.

Cross-links and history

Cross-quadrant links (Reference, Explanation, sibling how-tos) live inline inside Steps and Verify — never as a trailing «See also» / «Related» footer block. Every how-to closes with a history footer; the first history row should reference this template:

<footer class="docs-history" aria-label="History">
  <h2 class="docs-history__title">History</h2>
  <ul class="docs-history__list">
    <li class="docs-history__row">
      <span class="docs-history__date">YYYY-MM-DD</span>
      <a class="docs-history__author"
         data-component="author-chip"
         data-person-id="<maintainer-id>"
         data-variant="sm">@<handle></a>
      <span class="docs-history__note">Created on
        <a href="how-to.html">how-to template v1</a>:
        <short summary>.</span>
    </li>
  </ul>
</footer>
