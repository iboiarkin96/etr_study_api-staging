Service descriptor (YAML schema)

The schema every services/portal/internal/services/<name>/catalog-info.yaml must follow. The YAML file is the single source of truth; the entity-card HTML, the hub tile grid, the hero lifecycle tickers, and the internal nav-tree subtree are generated from it by tools/docs/render_service_descriptors.py. A pragmatic subset of the Backstage software-catalog descriptor with an x-study-app extension block carrying renderer-only fields.

At a glance

  • Source of truth: services/portal/internal/services/<name>/catalog-info.yaml. Never hand-edit the generated HTML regions (marked by <!-- catalog:start --> / <!-- catalog:end -->).
  • Generator: tools/docs/render_service_descriptors.py — invoked via make catalog-render; idempotent.
  • Drift guard: make catalog-render-check (also wired as a pre-commit hook on catalog-info.yaml and the regions it owns) fails non-zero if the on-disk HTML/JSON does not match what the YAML would produce.
  • Two namespaces: canonical Backstage keys (apiVersion, kind, metadata, spec) describe the service the way a future Backstage importer expects; x-study-app holds renderer-only fields a Backstage importer must ignore.
  • Worked example: the full api descriptor is reproduced below. The four real entries — api, portal, monitoring, ui-kit — are the authoritative reference.

Top-level keys

Every catalog-info.yaml must declare all five top-level keys in this order. Missing any of them fails the catalog review.

Key Required? Purpose
apiVersion required Always backstage.io/v1alpha1. Locks the descriptor to a known Backstage schema version.
kind required Always Component for these four services. Other Backstage kinds (API, System, Group) are not modelled today.
metadata required Identity + Backstage annotations + tags + typed links. See metadata block.
spec required Backstage's policy fields: lifecycle, owner, depends-on. See spec block.
x-study-app required Renderer-only extension. Tone, lede, header pills, dependency lanes, hub-tile copy, display order. See x-study-app block.

The metadata block

Field Required? Rendered as Notes
metadata.name required Used by spec.dependsOn values (component:<name>) and as the nav-tree key. Must match the directory name under services/portal/internal/services/. Kebab-case.
metadata.title required Currently mirrors name; reserved for human-readable override. Keep equal to name until we need a different display label.
metadata.description required "Provides" row in the entity card (one verb-led sentence). Answers "why does this service exist" without opening the page.
metadata.annotations.backstage.io/source-location required "Repo" row in the entity card (or "Source" if x-study-app.source-display is set). Monorepo-relative path — e.g. services/api/app/main.py, services/portal/.
metadata.annotations.backstage.io/last-reviewed required <time datetime="YYYY-MM-DD"> in the card footer. Freshness signal — drives the staleness policy. ISO date string in quotes.
metadata.tags optional Not rendered visually today; reserved for hub filtering once we wire a tag chip filter. Visible header pills are separate — see x-study-app.header-pills.
metadata.links[] optional "Code reference", "Database tables", "Public", "Links" rows — selected by type. See typed-link rules. Each entry has url, title, type and optional note/display.

Typed-link rules (metadata.links[].type)

The generator dispatches on type to render one of four rows. Only the first entry of each type is rendered; multiple type: code/data/public entries are not supported.

type Rendered row When to include
code Code reference — anchor to url Service ships a generated reference (pdoc, JSDoc, etc.).
data Database tables — anchor plus optional note Service owns persistent state (per ADR 0029).
public Public — anchor by default, or plain text if display: text Service exposes a public surface (OpenAPI URL, portal URL).
links Links — middot-separated row of anchors Generic catch-all. Use sparingly.

The spec block

Field Required? Allowed values Why
spec.type required service, library (today only these two) Backstage component sub-kind. ui-kit is a library; the rest are services.
spec.lifecycle required stable · experimental · deprecated Drives the entity-card lifecycle pill, the hub tile pill, and the hero ticker grouping.
spec.owner required user:<id> reference into internal/team/people/ Resolves to a person link in the entity card. Unknown ids fall back to the raw string.
spec.dependsOn required List of component:<name> references, or empty list Backstage's machine-readable dependency surface. The visual dependency lanes live in x-study-app.dependencies; this list is for future importers.

The x-study-app extension

Everything below this key is consumed only by render_service_descriptors.py. Backstage-compatible importers will ignore the entire block — that's why it sits outside metadata/spec rather than as annotations.

Field Required? Type Purpose
tone required enum: info · success · warning · neutral Drives the entity-card and dep-lanes tone (border, icon hue).
display-order required integer (ascending) Sort key across the hub tile grid and the nav-tree. Use multiples of 10 to leave room for inserts.
lede required YAML folded scalar (>-) Entity-card subhead. Whitespace is collapsed by the renderer; inline HTML (<code>, etc.) is allowed.
tile-lede required YAML folded scalar (>-) Shorter copy for the hub tile (catalog grid). Usually a one-clause restatement of lede.
tile-stack required middot-separated string Stack signature on the hub tile — e.g. "Python 3.11 · FastAPI · SQLAlchemy".
header-pills[] optional list of { text, tone } Extra pills next to the lifecycle pill in the entity-card header. Tones match the docs-pill palette.
dependencies[] required list of lanes — see dependency-lane schema Powers the "Depends on" row in the entity card. Empty list [] is allowed (renders nothing).
source-display optional string Overrides the "Repo" row label to "Source" (used by ui-kit because its source is a docroot, not a Python module).
repo-label optional string Overrides the literal "Repo" label without switching to source-display semantics.

Dependency-lane schema (x-study-app.dependencies)

Each lane groups chips under a label; each chip is a single technology. The generator renders these into the dep-lanes component.

Field Required? Allowed values Notes
lane.label required string Rendered as dep-lanes__label — e.g. "Framework", "Persistence", "Observability".
lane.chips[] required list of chip objects Order is preserved; render order in the HTML matches YAML order.
chip.name required string The visible technology name.
chip.icon required icon key from the icon registry Unknown keys fail the generator with a precise error — there is no silent fallback.
chip.status optional current (default) · planned Renders into the chip's data-status attribute. Drives the dashed-border styling for planned items.
chip.hint optional string Trailing micro-text — e.g. ":8000/metrics", "next", "shipper".

Icon registry

Chip icons are looked up in the ICONS dict in tools/docs/render_service_descriptors.py. Add new icon keys there as single-line inline SVGs to keep per-chip diffs readable. Today's keys include bolt, database, cylinder, cylinder-planned, refresh, shield, logs, cache, python, plantuml, pdoc, nginx, palette, braces, font, scrape, grid, file, containers.

What the generator owns

Each region is delimited by <!-- catalog:start id="…" --> / <!-- catalog:end -->. Anything between those markers is rewritten on every make catalog-render; anything outside is preserved.

Region id File What it renders
entity-card services/portal/internal/services/<name>/index.html The full <section class="entity-card"> block — header pills, lede, the <dl class="entity-card__fields"> table, the dep-lanes component, the footer.
catalog-grid services/portal/internal/services/index.html The hub tile grid — one svc-tile per service, ordered by display-order.
catalog-tickers services/portal/internal/services/index.html The hero lifecycle tickers grouped by spec.lifecycle.
(nav subtree) services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json The services subtree — one child per service, with spine pages that exist on disk.

Backstage fields we deliberately omit

This is the load-bearing part of the schema: it makes the omissions explicit so a future contributor doesn't quietly add them back.

Omitted field Reason
metadata.namespace Single team, single namespace — no consumer reads it, so it's pure noise.
metadata.system, spec.system, spec.subcomponentOf The catalog is flat: four services with no group-of-services layer. Modelling systems before they exist is premature.
spec.providesApis / spec.consumesApis APIs are catalogued in internal/services/<svc>/api-reference.html and publicly under public/reference/api/. Duplicating into the descriptor adds drift risk without payoff.
metadata.labels (k8s-style key/value pairs) No consumer that reads them. Tags + header-pills cover the only real use case — filtering and stack badges.
Multiple typed links of the same type The generator picks only the first; if you need more, escalate to an RFC rather than silently extending the schema.

Worked example — api/catalog-info.yaml

The full descriptor for the api service. Every required field is filled; three of the four typed-link slots are present (no links entry).

# Backstage-compatible service descriptor.
# Source of truth for the entity card on this directory's index.html and for
# the corresponding tile on services/portal/internal/services/index.html.
# Rendered by tools/docs/render_service_descriptors.py — do not hand-edit the
# generated HTML regions (marked by <!-- catalog:start/end -->).
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
name: api
title: api
description: Stable API contract for ETR study methodology
annotations:
backstage.io/source-location: app/main.py
backstage.io/last-reviewed: "2026-05-17"
tags:
- fastapi
- python-3-11
links:
- url: code-catalog/by-quadrant/reference.html
title: pdoc (auto-generated)
type: code
- url: ../../reference/data/index.html
title: Field-level reference
type: data
note: (lives under <code>reference/data/</code> per Diataxis)
- url: services/portal/public/reference/api/
title: OpenAPI in the public developer portal at <code>services/portal/public/reference/api/</code>
type: public
display: text
spec:
type: service
lifecycle: stable
owner: user:ivan-boyarkin
dependsOn: []

# Renderer-only extensions (consumed by tools/docs/render_service_descriptors.py).
# Sits outside spec/metadata so Backstage importers ignore the block cleanly.
x-study-app:
tone: info
display-order: 10
lede: >-
FastAPI runtime exposing the public <code>/api/v1</code> surface — the only
artefact that an external integrator interacts with at runtime.
tile-lede: >-
FastAPI runtime — the only artefact an external integrator interacts with
at runtime.
tile-stack: "Python 3.11 · FastAPI · SQLAlchemy · SQLite / Postgres"
header-pills:
- { text: FastAPI, tone: info }
- { text: "Python 3.11", tone: neutral }
dependencies:
- label: Framework
chips:
- { name: FastAPI, icon: bolt, status: current }
- label: Persistence
chips:
- { name: SQLAlchemy, icon: database, status: current }
- { name: SQLite, icon: cylinder, status: current }
- { name: Postgres, icon: cylinder-planned, status: planned, hint: next }
- { name: Alembic, icon: refresh, status: current }
- label: Schema
chips:
- { name: Pydantic, icon: shield, status: current }
- label: Observability
chips:
- { name: structlog, icon: logs, status: current }
- label: Cache
chips:
- { name: Redis, icon: cache, status: planned, hint: next }
