Canonical shape for a tutorial page (Diátaxis quadrant: learning-oriented). Copy this skeleton, replace each placeholder, and keep the section order intact — readers tab between tutorials and expect identical scaffolding.

When to use this template

  • The reader is a beginner in this domain or stack and needs to learn the loop on safe, throwaway work.
  • You can produce tangible output in 15–60 minutes — a passing test, a green build, a one-page report, a published page.
  • The reader follows one main path; no decision trees, no "Lane A / Lane B" branches.
  • Real production work uses the same shape; the tutorial is a scaled-down rehearsal, not a toy.

If any of the above is false — pick a different quadrant: How-to playbook (task-oriented, expert reader), Reference (lookup), or Explanation (understanding).

Hero (page-head)

Every tutorial opens with the same hero block. Two pills, an action-verb h1, a three-verb lede, and a four-chip metadata row.

<header class="page-head">
  <p class="home-hero__eyebrow">
    <span class="docs-pill docs-pill--info">Tutorial</span>
    <span class="docs-pill docs-pill--teal"><Role> · Foundations</span>
  </p>
  <h1><Action verb + tangible subject></h1>
  <div id="docs-top-nav"></div>
  <p class="lede">
    <Verb 1> <X>, <verb 2> <Y>, <verb 3> <Z>. The point is to feel
    the <noun> loop — <contrast with real work>, not to <real task>.
    Real <noun> uses the same shape with <difference>.
  </p>
  <div class="docs-chip-row docs-chip-row--mt-3">
    <span class="docs-chip">⚙️ N phases</span>
    <span class="docs-chip">⏱ M–M' min</span>
    <span class="docs-chip">📚 First-time</span>
    <span class="docs-chip">💻 Local stack</span>
  </div>
</header>

Role pill colour follows the role: info for SWE/SA/Architect, teal for QA, danger for SRE, warn for Manager. Last chip is 💻 Local stack for code-driven tutorials, 📝 Paper-only for whiteboard/study tutorials.

Lede formula

Three imperative verbs in a row, then a "the point is to feel … not to …" contrast. Anchors the reader: this is a rehearsal, not a production task.

  • Boot the project, write a tiny read endpoint, document it, and ship a green make verify. The point is to feel the full dev loop — not to learn how to ship production code.
  • Manufacture a tiny failure, page yourself, and walk the runbook end-to-end until exit criteria are met. The point is to feel the drill loop — trigger → triage → recovery — not to handle a real incident.

Section skeleton

Every section uses an id anchor and a docs-step-number badge inside the h2. Numbers run – (Before you start), 0 (Goal), then 1..N for phases.

  1. <section id="prereqs"> — "Before you start", step-number –. Three to six bullets: what must be installed, what state the repo is in, what other docs the reader skimmed.
  2. <section id="goal"> — "Goal", step-number 0. Three to five concrete deliverables: "Pick X", "Run Y", "Write a report at path".
  3. <section id="phase-1">..<section id="phase-N"> — main loop. Each phase ends with a checkpoint card (see below).
  4. The last phase replaces the checkpoint with a "You're done" card containing a final deliverable checklist plus a one-line recap of the loop ("That is the X loop — Y → Z → W").
  5. Optional: a "Want to start over?" ghost-card with a rollback shell snippet (for multi-day tutorials, or where local state needs cleaning).
  6. <aside class="docs-card docs-card--gradient"> — "Next step" pointing to 3–5 deeper links in How-to / Reference / Explanation / Practice.
  7. <footer class="docs-history"> — page history rows.

Checkpoint card

Every phase except the last ends with the same shape. This is where conditionals belong — never in the main step body.

<div class="docs-card docs-card--ghost">
  <h3 class="docs-card__title">✓ Check your progress</h3>
  <p>
    <Verifiable condition> — e.g. "the JSON file exists", "the command
    exited 0", "the page renders with one h1". If <failure mode>, <hint>.
  </p>
</div>

Conditionals are only allowed inside this card (as troubleshooting hints). The main step body is always imperative and unconditional.

"You're done" card

<div class="docs-card docs-card--ghost">
  <h3 class="docs-card__title">✓ You're done</h3>
  <ol>
    <li><Deliverable 1> exists.</li>
    <li><Deliverable 2> passes.</li>
    <li><Deliverable 3> is recorded.</li>
  </ol>
  <p>That is the <X> loop — <recap step 1> → <step 2> → <step 3>. Real
    <X> widens the surface; the shape is the same.</p>
</div>

Voice rules

  • Imperative throughout: "Create", "Run", "Open", "Confirm". Never "you can", "optionally", "you might want to".
  • Second-person sparingly: "You'll see", "You should be able to" — only where imperative is awkward.
  • No decision trees in the main path: tutorials have one path. Variants belong in How-to playbooks.
  • Conditionals only in checkpoints: "If you see X, do Y" — never in the step body.
  • Cite real artefacts: link to actual checklist / ADR / runbook IDs so the reader sees how the rehearsal connects to production work.

Anti-patterns

  • Lane A / Lane B in the main path — pick one main path, demote the other to a ghost-card shortcut.
  • Long lecture before any action — tutorials are doing-first; defer the "why" to Explanation pages or short side-notes.
  • 80-step megapages — split into multiple tutorials if the loop has > 6 phases; readers lose track.
  • "This is a placeholder" — never ship a tutorial that doesn't end with verifiable output.
  • Production-volume task as the rehearsal — the tutorial must be a scaled-down rehearsal, not the real task.

Cross-links and history

Every tutorial closes with a "Next step" card and a history footer. The first history row should reference this template:
