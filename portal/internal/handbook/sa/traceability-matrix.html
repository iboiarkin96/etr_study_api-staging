🧾 SA reference · Traceability matrix

Traceability matrix — requirement → release

A traceability matrix is a single-page audit tool. Each row covers one concrete user-facing feature the team has committed to ship and keep working — things like «a user can sign in», «a search returns within 1 second», «a refund completes within 24 hours». The six columns then name the five artefacts that prove each feature actually works, plus the row's overall health: the team that owns it · the design decision that froze how it works · the code that implements it · the test that catches regressions · the runbook on-call opens when production breaks · the status pill summarising whether the chain is whole today. A blank cell is not a styling gap; it means a piece of the system has no owner, no proof, or no response plan — and the whole purpose of the matrix is to make those blanks impossible to ignore.

Rules

01

What one row tells you

Before reading the whole matrix, read one row. Take a teaching example — REQ-EX-01: «A user can sign in.» Left to right, the row answers six questions; everything in this page (the column definitions, the YAML, the anti-patterns) is just enforcement of these six answers being present and trustworthy.

  1. Who is on the hook? identity-team — paged when login breaks, signs the audit, can say «yes / no» without escalation.
  2. How did we decide to build it? ADR-EX-01 — the «sign-in via OIDC» decision that froze the auth approach. If you want to change how login works, you open that ADR, not this row.
  3. Where does it actually live in code? POST /v1/sessions — the HTTP operation the client calls. (For non-API features this can be an event topic, a CI step, or a process — n/a is legal when there's literally no code surface.)
  4. What proves it still works? identity/sign_in_test.go — the test that fails the build when the login flow regresses.
  5. What happens when it breaks in production? RB-EX-01 — the runbook on-call follows at 3 a.m.
  6. Is the chain whole right now? Status pill. ok = all five upstream answers are filled; gap = at least one is missing.

The full matrix is just «N such rows side by side». A blank cell anywhere along the chain means one of those six questions has no answer — and an unanswered question is backlog, not a documentation gap.

Five rules the matrix enforces:

  • One row per user-facing feature. If a commitment isn't in the matrix, it isn't a commitment — it's a wish. The row is the contract.
  • Empty cell = ticket. A missing ADR, test, or runbook is filed against the owner. The quarterly audit walks the matrix row by row.
  • Read both directions. Down a column: «which requirements depend on ADR-EX-01?» Up a row: «if REQ-EX-01 breaks at 3 a.m., who picks it up and which runbook do they open?»
  • IDs are immutable. An ID once issued never moves and never gets reused — incidents, ADRs, postmortems all keep pointing at it forever.
  • Quarterly review. Broken rows become work items, not commentary. The matrix is a backlog input, not a wall of shame.

Reading the three rows out loud:

  • REQ-EX-01 — ok. All six cells are filled. The chain is closed: owner exists, decision is on record, code is identified, test guards regressions, runbook covers production. This is what every row aspires to.
  • REQ-EX-02 — wip. Runbook cell is yellow (…) — a draft exists but doesn't cover all alert paths yet. The owner has a dated commitment this sprint, so status is wip, not gap.
  • REQ-EX-03 — gap. Runbook cell is red (∅) — nothing on disk, no committed sprint. On-call would scramble if a refund-SLA breach happened tonight. The row is owned and ticketed, but status gap blocks the «ok» pill until the runbook ships.

These three rows are entirely fictional — the EX- prefix in every ID makes that obvious. They exist to show what the symbols mean in context, not to describe the Study App. The current Study App matrix is at the pointer in §06.

Rules

03

What each column means

One sentence per column; rules of legitimate n/a and the most-serious gaps inline.

  • ID

    The handle every other doc points at — REQ-NNN, immutable, never reused. Retired requirements stay in the matrix with status retired.

  • Requirement

    What we committed to ship, in one active-voice sentence — «A user can X», «An admin can Y». Acceptance criteria live in the linked spec page, not the cell.

  • Owner

    The single team on the hook — paged on regressions, signs the audit, can say «yes / no» without escalation. An empty owner is the worst gap a row can carry: no one will notice when the chain breaks. Aligns with Ownership.

  • ADR

    The decision that froze how the requirement is satisfied. If two ADRs apply, the matrix names the freshest and both link from the spec.

  • API operation

    The thing in code that implements the commitment — HTTP operation, event topic, or pipeline step. n/a is legal for process-only rows (e.g., postmortem SLA).

  • Test

    The test that fails when the commitment breaks. End-to-end is preferred; unit is acceptable if the chain is independently covered.

  • Runbook

    The on-call playbook for production regressions. n/a only for build-time concerns (e.g., publish pipeline) — never for user-facing paths.

  • Status

    Overall row health. ok — full chain. wip — in flight this sprint. gap — at least one cell missing & ticketed. retired — kept for history.

Rules

04

How a row stays accurate (don't hand-edit cells)

You never type a cell into the HTML. Each row lives in governance/traceability/matrix.yaml as one stanza like the one below — the HTML matrix is rendered from it. CI then walks every referenced ADR, API operation, test, and runbook and fails the build if any of them aren't actually on disk. That's what makes a ok status a guarantee, not a hope. 

- id:          REQ-EX-01
summary:     "A user can sign in."
owner:       "identity-team"     # single team on the hook
spec:        "governance/requirements/REQ-EX-01.html"
adr:         "governance/adr/EX-01-sign-in-via-oidc.html"
api:
  - operation: "POST /v1/sessions"
    page:      "services/api/reference/operations/post-v1-sessions.html"
test:
  - path:      "tests/identity/sign_in_test.go"
    kind:      "e2e"
runbook:     "foundations/how-to/incidents/runbooks/RB-EX-01-sign-in-failure.html"
status:      "ok"
last_review: "2026-06-06"
# A blank field renders as ∅ in the matrix. CI rejects a row where status
# is "ok" with any blank field — you have to either fill the cell or
# downgrade the status to "wip" / "gap".

Anti-patterns

05

Anti-patterns reviewers reject

  • Hand-edited HTML cells. The table is rendered from the YAML. Manual edits silently desync and the next regen pass wipes them.
  • «Status: ok» with an empty cell. Either fill the cell or downgrade the status to wip / gap. CI fails the build on this mismatch — that's the whole point.
  • Requirement written as a paragraph. One active-voice sentence per row. Multi-sentence commitments get split into separate rows with separate IDs.
  • Reusing a retired ID. IDs are permanent — reusing an ID after retirement silently re-routes every incident report, ADR back-reference, and postmortem that used to point at it.
  • Skipping the ADR. «We just built it» is not legal for a user-facing feature. No ADR ⇒ the row is a gap, even with API + test + runbook all present — because there's no record of why the thing was built the way it is.
  • Inventing an owner. «platform-team» is not an owner if no one on platform-team has been paged for this row in the last quarter. The owner field is operational, not aspirational.
  • Putting live coverage on this page. This page is the contract — what a matrix is, how a row is shaped, how the symbols read. Actual current rows and statuses live in the live-coverage page (§06). Mixing them re-makes the «contract drifts with the data» problem the matrix exists to solve.

Apply

06

Where the live Study App matrix lives

This page is the contract — the shape of a row, the meaning of each column, the YAML stanza, the anti-patterns. It doesn't change every sprint and isn't a report on today's coverage.

The live matrix for the Study App — current rows, current owners, current ∅ / … / status pills, current sev-1 gaps — is rendered as a separate page under governance/traceability/. That page is generated from governance/traceability/matrix.yaml and refreshed every CI build, so it always reflects what is actually on disk; this page stays stable while that one churns.

Split rationale: a reviewer reading «what does the matrix look like?» doesn't want their answer to drift when this quarter's coverage changes; a reviewer reading «is REQ-X covered today?» doesn't want to scroll through the contract first. Two pages, one contract.
