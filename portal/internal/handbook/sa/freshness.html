🕒 Process · Freshness

Freshness — staleness SLA & drift detection

Living docs need a freshness contract — otherwise «living» means «rotting at room temperature». Each document type has an explicit staleness SLA based on its last-verified date; pages that cross the SLA get a banner, ping the owner, and either get re-verified or moved into deprecation. No page silently turns into a lie.

Stance

01

The freshness contract

Every page declares a last-verified date in its <body data-last-verified="YYYY-MM-DD"> attribute. The freshness state is a function of that date and the page's type-specific SLA:

  • FRESH — within the SLA. No action.
  • STALE-SOON — within 30 days of the SLA limit. Banner appears; owner is notified weekly.
  • STALE — past the SLA. Red banner; owner is on the hook to re-verify or deprecate within two weeks.

A re-verification PR bumps data-last-verified to today and adds a row to the history block — one sentence stating «verified, no changes needed» or naming what was updated.

Rules

02

SLA by document type

The SLA reflects how fast the underlying reality changes for that type. A Tutorial about a service can rot in months; a Reference page generated from OpenAPI rots in days; an ADR is timeless unless superseded.

Document type SLA Why this window
ADRNever stalesA ratified decision is a historical record. It can be superseded but it doesn't go stale.
RFC (accepted)Never stalesSame logic as ADR — implementation may diverge but the proposal is what was decided.
Reference (auto-generated · OpenAPI / pdoc)Rebuilt every releaseGenerated artefact — staleness is impossible if the build runs.
Reference (hand-written matrix · spine / RACI)3 monthsTied to one release-train cycle. Owners re-verify when the matrix changes.
Runbook6 monthsThe system the runbook acts on changes — alerts, paths, escalation chains. Twice-a-year audit catches most drift.
How-to9 monthsProcedural pages; reality moves slower than reference but the cost of stale steps is real.
Tutorial12 monthsTutorials are linear and brittle — they break on every UI refresh. Annual re-walk is the floor.
Explanation18 months«Why we do it this way» rots slowest. Even if the implementation changed, the rationale often survives.
Audit reportFrozen on issue dateAn audit is a snapshot — never re-verified. A new audit replaces it.

Rules

03

How drift is detected

Three layers of detection — fast / slow / opportunistic. They cover staleness from different angles.

Layer Cadence What it catches
Mechanical scan Nightly cron check_freshness.py walks every page's data-last-verified and computes the state per type. Outputs a per-owner freshness report under governance/audits/freshness/.
LLM drift scan Weekly Canonical LLM prompt compares each page's claims to the current code surface (file paths cited, function names mentioned, error codes referenced). Pages with high drift score get flagged for re-verification regardless of SLA.
Author flag On any PR If a code PR touches a path that a doc page cites, the PR template prompts the author: «Does docs/page-X still match reality?» Manual flagging surfaces drift before the SLA expires.

Apply

04

Escalation when a page goes stale

The four-step escalation kicks in the moment a page hits STALE. Owner ownership is non-negotiable — every page has exactly one.

  1. Banner appears.

    The page renders a red «Stale — last verified YYYY-MM-DD» banner automatically. The reader knows before reading a sentence that the content may be wrong.

  2. Owner pinged.

    check_freshness.py opens a tracking issue assigned to the page's owner (from data-maintainer-ids). The issue title is «Re-verify or deprecate: page-path».

  3. Two-week window.

    The owner either re-verifies (bumps data-last-verified, adds a history row) or opens a deprecation ADR. The freshness report tracks the window per page.

  4. Auto-escalation.

    If the two-week window passes with no action, the page is escalated to the owner's manager via the ownership RACI, and the page is queued for the next quarterly audit's «to-deprecate» list.

Rationale

05

Tradeoffs

Wins

Readers see the truth

A stale page wears a red banner. The reader knows not to trust it, and the link to the owner is one click away — they can ping for clarification instead of guessing.

Cost

Owner toil scales with page count

500 pages × 3 re-verifications/year = 1,500 re-verifications. That's not free. We mitigate with «verified, no changes needed» one-line re-verifications (1 minute per page), but at scale this is a real time sink.

Wins

SLA matches the type

An ADR doesn't get pinged every quarter; a runbook does. The SLA reflects how fast reality moves for each kind, instead of one-size-fits-all annual audits.

Cost

«Verified» is a weak signal

An owner can bump the date in 10 seconds without actually reading the page. We accept this — the date is a commitment, not a proof; the audit cadence (process.html §05) is what proves substance.
