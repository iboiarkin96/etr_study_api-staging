🌐 SA reference · Domain model

Domain model — bounded contexts & ubiquitous language

The domain model is the line we draw between «business meaning» and «implementation choice». Six bounded contexts carry the system; one published language joins them. Every API operation, every ADR, every entity in the catalog has to map back to a context here — otherwise it lives outside the spine.

01

Bounded contexts

Each context owns one slice of business meaning, one team, and one source of truth. Words mean different things across contexts — that is the point, not a bug.

BC-01 · Learning

Learning

Authoring, structure, and delivery of study material. Owns «course», «module», «lesson», «exercise».

Owner
Content guild
Surface
services/api · POST /v1/courses
Storage
relational + object store

BC-02 · Identity

Identity & access

Who the actor is and what they are allowed to see. Owns «user», «session», «role», «entitlement».

Owner
Platform team
Surface
POST /v1/sessions · OIDC
Storage
relational + KV cache

BC-03 · Progress

Progress

Learner state over time — what was attempted, what passed, what is still pending. Owns «attempt», «mastery», «streak».

Owner
Learning team
Surface
GET /v1/users/{id}/progress
Storage
append-only log + materialised view

BC-04 · Assessment

Assessment

Scoring rubric, grading policy, and certificate issuance. Owns «rubric», «grade», «certificate».

Owner
Learning team
Surface
POST /v1/attempts/{id}/grade
Storage
relational; certificates in object store

BC-05 · Catalog

Catalog (commerce)

What is on offer, at what price, with what entitlements attached. Owns «offering», «price», «promotion».

Owner
Commerce team
Surface
GET /v1/offerings
Storage
relational + cache

BC-06 · Observability

Observability

How the system reports its own state. Owns «metric», «log», «alert», «incident», «runbook».

Owner
SRE
Surface
Prometheus · Grafana · Alertmanager
Storage
time-series + S3 archive

Reading rule. «User» in Identity means «account holder»; «user» in Progress means «the learner whose attempts we count». Same word, two definitions, two contexts — no shared schema.

02

Context map — relationships between contexts

Strategic DDD relationships. Each row says how one context depends on another and what shape that dependency takes. The pill tells you the integration pattern.

Upstream Downstream Pattern Why this pattern
Identity Learning · Progress · Assessment · Catalog Published language One canonical session/JWT shape; every downstream parses it the same way.
Learning Progress Customer / supplier Progress's schema follows Learning's course/module IDs; Learning publishes change ahead.
Assessment Progress Customer / supplier Progress consumes Assessment's grade events; Assessment owns the rubric versioning.
Catalog (commerce) Identity Anti-corruption layer Catalog talks to a 3rd-party billing vendor; the ACL keeps vendor terms out of our domain.
Catalog (commerce) Learning Conformist Catalog adopts Learning's «course» entity verbatim — no remapping, simplest integration.
All BCs Observability Shared kernel Single OpenTelemetry schema; every team owns a slice of it together.
03

Ubiquitous language — the published terms

Words that appear in API contracts, UI strings, and code identifiers must use these spellings exactly. Synonyms are not allowed inside a single context.

course
A bundle of modules sold or assigned as one offering. Owned by Learning.
module
A chapter inside a course. Owns lessons. Owned by Learning.
lesson
One unit of study content — text, video, or exercise. Owned by Learning.
exercise
A graded question or task inside a lesson. Owned by Learning; graded by Assessment.
attempt
One submission of an exercise by a learner. Immutable event in Progress.
mastery
A derived score (0..1) of how well a learner knows a module. Progress.
rubric
Versioned grading rules that turn an attempt into a grade. Assessment.
grade
The output of applying a rubric to an attempt. Assessment.
certificate
Signed credential issued after course completion. Assessment.
offering
A course or bundle of courses listed for sale. Catalog.
entitlement
A purchased right to access an offering. Catalog, consumed by Identity.
session
An authenticated period bound to a user agent. Identity.
role
Coarse-grained authorisation bundle (learner · author · admin). Identity.
incident
An open production failure tracked end-to-end with a postmortem. Observability.
runbook
The on-call playbook keyed to a specific alert. Observability.
04

Cross-context event envelope

When one context emits a fact for others to consume, it uses this envelope. The envelope itself is part of the shared kernel; the data block is whatever the producer publishes. 

event_id:     "01HVXY7M2K1S8KQKZJ4N5R7T9P"   # ULID, source of truth for dedup
event_type:   "assessment.grade.published.v1"
occurred_at:  "2026-05-24T10:14:33Z"
producer:
  bounded_context: "assessment"
  service:         "assessment-api"
  version:         "1.7.2"
subject:
  user_id:    "usr_3K9..."
  attempt_id: "att_8X1..."
data:
  rubric_version: "2026.03"
  grade:          0.82
  passed:         true
05

Cross-context invariants

  • One owner per term. Every term in §03 belongs to exactly one bounded context. Cross-context use happens through the published language, not by sharing schemas.
  • No upstream calls downstream. Identity does not know about Progress; Learning does not know about Catalog. Cycles in the context map are bugs.
  • Events are immutable. A grade event published yesterday is still true today; corrections are new events, not edits.
  • Rubric versioning is visible. Every grade carries the rubric_version that produced it. Re-grading uses a newer rubric and emits a new event.
  • Anti-corruption layers are explicit. Vendor terms (Stripe products, Auth0 connections) never leak past their ACL. Mapping code lives in one file per ACL.
06

Worked example — domain.yaml for one team

A complete one-file projection a team can commit alongside its code. This is the smallest format we have found that captures all five sections above (contexts, relationships, language, events, invariants) without sprawling into folders. 

version: "2026.05"
team:    "learning-platform"
owners:  ["@learning-eng", "@learning-pm"]

contexts:
  - name: "learning"
    purpose: "Author and serve courses, modules, lessons, exercises."
    owns:    [course, module, lesson, exercise]
    data_store: "postgres"
  - name: "assessment"
    purpose: "Apply rubrics to attempts; mint grades and certificates."
    owns:    [rubric, grade, certificate]
    data_store: "postgres"
  - name: "progress"
    purpose: "Append-only learner history; derive mastery."
    owns:    [attempt, mastery]
    data_store: "postgres + materialised views"

relationships:
  - from: progress
    to:   assessment
    pattern: "customer-supplier"
    note: "Progress sends attempts; Assessment returns grades."
  - from: assessment
    to:   learning
    pattern: "conformist"
    note: "Assessment grades inherit learning's exercise shape."

language:
  attempt:   "One submission of an exercise. Immutable in Progress."
  grade:     "Output of applying a rubric to an attempt. Owned by Assessment."
  mastery:   "Derived score (0..1) of how well a learner knows a module."

events:
  - type:     "assessment.grade.published.v1"
    producer: "assessment-api"
    consumers: ["progress-projector", "notification-fanout"]
    retention: "180d"

invariants:
  - "An attempt has exactly one grade."
  - "A certificate exists only if all required grades are 'passed'."
  - "Mastery is recomputed from attempts; it is never written directly."

A team that opens a new repo copies this file, swaps the names, and is 80% of the way to a published domain model. The remaining 20% — diagrams, ADRs, OpenAPI — references back to this YAML by team and version.

07

Tradeoffs we accepted

  • Six contexts, not three. Splitting Identity and Catalog from Learning increases integration cost — accepted to keep team ownership clean.
  • Progress as append-only. Materialised views add latency on schema changes — accepted because audit replay is more valuable than write-path simplicity.
  • Catalog as a Conformist to Learning. Catalog inherits Learning's pace of change — accepted because Catalog is small and the integration is read-mostly.
  • Shared OpenTelemetry kernel. Every team must coordinate on schema edits — accepted because consolidated dashboards beat per-team observability silos.
