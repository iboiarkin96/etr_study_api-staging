Reader personas — who reads this portal

Information architecture is a service to specific readers, not an abstract ideal. Four reader personas drive every IA decision in this portal: who they are, what they came to find, and the exact path the sidebar should give them. If a new page does not serve at least one persona, it does not ship.

01

Why personas, not «just write good docs»

A 500-page portal is no longer a single document — it is a small product. Without a reader model the authoring conversation drifts into «what looks complete» rather than «what gets read».

  • Forcing function for IA. A persona makes «is this discoverable?» a falsifiable question — does the persona reach the page in ≤3 clicks?
  • Scope discipline. A page tuned for the newcomer reads differently than one tuned for an architect. Naming the audience prevents middle-ground prose that helps no one.
  • Review heuristic. Reviewers ask «which persona does this serve, and does it land within their first sidebar pane?» rather than chasing taste.
  • Honest trade-off. If a page serves three personas equally, it is probably a hub — and should be split into a hub + three deep-dives, not one wall of headings.
02

The four personas

One row per persona. Each row gives goal, friction, the three sidebar paths they actually use, and the page that defines «done» for them.

🐣

Newcomer engineer

Day 1 · Week 1 · first PR

«I have a laptop, a git clone, and a vague brief. Where do I start, what runs locally, and which PR will not embarrass me?»

Goal
Get to a green build locally, then ship a docs-only PR through the same pipeline as production code.
Friction
Doesn't yet know the domain vocabulary; needs the glossary inline, not buried two levels deep.
Time budget
~2 hours on Day 1, ~1 day across Week 1.
Done when
First PR merged via the docs-as-code pipeline (Vale + lychee + Spectral pass).

Top sidebar paths

  • onboarding/day-1.html — 30-minute checklist (clone, make verify, open the local portal).
  • onboarding/week-1.html — end-to-end docs PR via the same gate set production code uses.
  • onboarding/glossary.html — every acronym in one page (SA · ADR · RFC · RACI · gRPC · AsyncAPI).
📐

SA peer (Systems Analyst)

requirements · domain · traceability

«I'm reviewing a proposal. Show me the requirement, the domain term it touches, the ADR that froze the decision, and which API operation implements it.»

Goal
Trace a single business requirement end-to-end: requirement → ADR → API operation → test → runbook.
Friction
If the chain is broken anywhere — missing ADR, no test, no runbook — that gap is the actual finding.
Time budget
10-15 minutes per requirement during async review; longer for a new feature kickoff.
Done when
The traceability matrix renders one row per requirement with no empty cells, or the empty cell is filed as work.

Top sidebar paths

  • foundations/reference/sa/traceability-matrix.html — Requirement × ADR × API × Test × Runbook grid.
  • foundations/reference/sa/domain-model.html — bounded contexts + ubiquitous language.
  • foundations/reference/sa/workflows.html — cross-service BPMN flows for the headline use cases.
🛠️

SWE (software engineer)

API contract · runbook · changelog

«I need the exact request shape, the failure modes, and the on-call page if my change breaks production. Anything else is reading for fun.»

Goal
Implement against a contract, then prove the implementation matches it. Then know who to page when it breaks.
Friction
Hates prose between sections of YAML. Wants the snippet first, the explanation second.
Time budget
2-5 minutes per lookup, dozens of times a day.
Done when
API operation page renders the OpenAPI snippet, status codes, and a runbook link without a page reload.

Top sidebar paths

  • services/api/reference/operations/* — one page per HTTP operation.
  • services/*/runbooks/* — on-call playbook keyed to each alert.
  • services/*/changelog.html — Keep-a-Changelog format, latest at top.
🗺️

ARCH (architect / staff+)

C4 · ADR · topology · ownership

«Show me the system in one diagram, the decisions that froze it, who owns each piece, and what we'd have to undo to change it.»

Goal
Reason about the system as a whole — components, contracts, blast radius, ownership boundaries.
Friction
Diagrams without an «as-of» date are noise. Wants every diagram source-of-truth-linked.
Time budget
Short bursts during design review; longer during quarterly architecture audits.
Done when
Can answer «if I remove component X, what breaks?» from the catalog + topology page alone.

Top sidebar paths

  • foundations/reference/sa/topology.html — C4 context + container diagrams.
  • governance/adr/* — every frozen decision, dated, with consequences.
  • foundations/reference/sa/catalog.html + spine-matrix.html — Backstage-style entity catalog.
03

Every page is a four-step journey

Each reader takes the same four-step trip across the portal, regardless of persona. Authoring a page means asking «which step am I writing for?» — pages that try to cover all four become hub pages.

1

Land

Arrive from search, sidebar, or an outbound link. First viewport answers «am I on the right page?».

2

Orient

Breadcrumbs + hero lede establish where in the IA they are and which persona this page targets.

3

Act

The body delivers the artefact the persona came for — snippet, table, diagram, checklist.

4

Leave

Inline «See also:» links and the sidebar push the reader to the obvious next page, not back to a hub.

04

Anti-patterns — pages that fail at least one persona

Patterns reviewers reject. Each maps to a persona the page silently ignored.

The «universal hub»

One page that tries to onboard the newcomer, satisfy the SA peer, and brief the architect. Result: no persona reaches what they need within their time budget.

Fix: split into hub + per-persona deep-dives. Hub is one card per audience, not one section per audience.

The diagram-without-date

A C4 or topology diagram with no «as-of» timestamp. ARCH cannot tell if it reflects reality or last quarter's plan.

Fix: every diagram has a generation date + a link to the YAML/PlantUML it was rendered from.

The contract-as-prose

An API page that describes the request in English instead of YAML. SWE has to mentally re-encode the prose into code.

Fix: snippet first, prose second. Prose explains «why» — never re-explains the schema.

The acronym wall

A page that opens with SA, ADR, RFC, SLO, RACI, gRPC, AsyncAPI without expansion. Newcomer bounces inside 30 seconds.

Fix: spell out every acronym the first time it appears on the page, even if it appears elsewhere on the portal.

The broken trace

A requirement page with no link to an ADR; an ADR with no link to the API operation it freezes. SA peer cannot complete the chain.

Fix: traceability matrix renders the chain — empty cells are bugs, not gaps.

The dead-end leaf

A page that delivers its artefact and then stops. The reader has to go back to the hub to continue.

Fix: last block of every leaf page suggests the next inline (See also: X, Y) — not as a footer block.

05

How authors and reviewers use this page

  • Authors. Before writing, name the persona in your draft's hero eyebrow. If you cannot pick one, the page is probably a hub — split it.
  • Reviewers. Open the PR side-by-side with this page. Ask «which persona does this serve, and does the first viewport prove it?». If no, request a hero rewrite, not body edits.
  • LLM assistants. Prompt with «target persona = newcomer / SA peer / SWE / ARCH». Outputs drift less when the audience is explicit.
  • Audit cycle. Quarterly persona audit picks 8 random pages, scores each against its declared persona's «done when» — failing pages go to the backlog with an owner.
