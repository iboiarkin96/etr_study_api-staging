Threat model — STRIDE per component

The catalogued threats against the three runtime components (integrator, api service, database) and the explicit mitigation status for each. Built on STRIDE: Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege. Read with the architectural narrative open — the three boxes there are the three columns here.

Scope & out-of-scope

The model covers the runtime request path described in architecture narrative § The shape: Integrator (someone else's mobile/web app, talking HTTP) → api service (FastAPI on Python 3.11) → database (SQLite today, Postgres planned). The docs portal is out of scope here — it is a static site with no PII surface; threats against it are misinformation/defacement, not data theft, and they live in the portal's own threat surface.

Status legend. covered — mitigation is in code today and verified. partial — mitigation exists but has known gaps named in this row. open — no mitigation; recorded as a known risk.

Component 1 — Integrator (untrusted client)

The integrator is anyone who holds an X-API-Key. The threat surface here is "the client behaves badly", whether intentionally or by misconfiguration. Per ADR 0005 the principle is: trust the API key for tenancy, never trust the body for identity.

STRIDEThreatMitigationStatus
SpoofingStolen API key replayed by an attacker who is not the integrator.API key sent only over TLS (HTTPS enforced at the edge); rotation procedure documented; key hash, not the key, is logged. No second factor today.partial — no key rotation cadence; no second factor
TamperingBody fields modified in transit.TLS in transit; HMAC body-signing is not enforced today — the API key is the integrity proof.partial — TLS only; no app-level body integrity
RepudiationIntegrator denies having sent a write that succeeded.Every write logged with request_id, API-key hash, body hash, and idempotency key; request_done info line carries the full audit triplet.covered
Information disclosureIntegrator A reads data owned by integrator B by guessing UUIDs.Ownership check at the service layer: cross-tenant access collapses to 404, never 403, so resource existence is not leaked. Per _shared/auth § why-404.covered
Denial of serviceIntegrator floods the API with requests.Per-key sliding-window rate limit (in-process, services/api/app/core/security.py); body-size limit (~32 KB on writes) rejects oversize payloads before the handler runs.partial — single-instance limiter; horizontal scale would need Redis-backed limits
Elevation of privilegeIntegrator escalates from tenant access to admin / cross-tenant access.No admin endpoints exposed today; no cross-tenant API surface exists. The system has no admin role to escalate to.covered by absence

Component 2 — api service (FastAPI)

The api service is the only component that holds business logic. The threat surface here is "an attacker reaches code execution in the service", whether through a vulnerable dependency, a bug in our code, or a logic error in a third party. The mitigations cited reference ADR 0019 (dependency pinning & pip-audit) and ADR 0023 (structured logging).

STRIDEThreatMitigationStatus
SpoofingForged X-Request-Id or middleware header to impersonate a previous request.Server generates request_id when absent; never trusts client-supplied identity headers. X-Request-Id is correlation only, not identity.covered
TamperingSQL injection via path / body / query parameters.All DB calls go through SQLAlchemy parameterised queries; raw SQL is forbidden by lint convention. Pydantic models validate types before the service layer runs.covered
RepudiationServer denies a state change it actually made.Every successful write writes an idempotency_keys row with the response body — replay-traceable. request_done log line includes the status and elapsed time.covered
Information disclosureStack traces or DB strings leaked in error responses.Global exception handler returns 500 with no body details; stack trace goes to request_failed log at exception level only. Per error envelope: forbidden contents include stack traces.covered
Information disclosurePII in logs (full_name, learner-typed paragraph text).full_name / username classified PII medium per field conventions; masked at warn level. dense_paragraph & bullets never logged. Idempotency keys hashed before logging.covered
Denial of serviceVulnerable dependency exploited to crash the worker or exhaust memory.pip-audit runs in CI per ADR 0019; dependencies pinned; SBOM committed. No mitigation against zero-day exploits between scans.partial — pinning + scans, no zero-day defence
Elevation of privilegeDomain exception HTTPException raised by service layer is mis-mapped at the edge, leaking a more powerful response status.Single global error mapper translates domain exceptions to code/key/message envelope. Service layer never sets HTTP status directly.partial — pragmatic HTTPException usage today; principled domain-exception layer is named as future work in business-logic § failure modes

Component 3 — Database (SQLite today, Postgres planned)

The database holds every byte of authoritative state and every authoritative ownership claim. The threat surface here is "an attacker reaches the file or the wire". Today the database is a SQLite file colocated with the service; the planned Postgres swap changes the surface (network protocol, separate auth) but not the catalogued threats.

STRIDEThreatMitigationStatus
SpoofingProcess other than the api service connects to the DB and impersonates it.SQLite today: file-system permissions only — anyone with shell access on the host can read the file. Postgres plan: separate connection role with restricted DDL, password from secrets manager.partial on SQLite (filesystem perms only); covered by the Postgres swap
TamperingDirect DB writes that bypass service-layer validation.SQLite today: any process with the file can write. Postgres plan: revoke direct write on the app role; all writes must go through ORM. Migrations gated by Alembic.open on SQLite (no enforcement); planned with Postgres swap
RepudiationApplication can deny what was committed.WAL log on SQLite; Postgres WAL on the planned swap. idempotency_keys table stores the response served; conspectus_events records every state-changing write with diff. See PATCH conspectus § side-effects for the event-row pattern.covered
Information disclosureBackup files or DB dumps end up in places they shouldn't.No backup automation today; manual dumps are excluded from git via .gitignore. Backup discipline is named as a Postgres-swap prerequisite (encrypted backups, retention policy, restore drills).open — no formal backup policy
Denial of serviceWrite-lock contention from concurrent retries saturates the single-writer engine.SQLite single-writer behaviour documented as a known capacity ceiling in postmortem 2026-04-02; load tests run with client retries disabled to measure raw saturation. Postgres swap removes the single-writer bottleneck.partial — documented limit, no live alerting on the saturation signal
Elevation of privilegeService connection used for DDL beyond its scope.SQLite today: no role separation. Postgres plan: app role gets DML on owned tables, no DDL; migrations run from a separate role with DDL.open on SQLite; planned with Postgres swap
