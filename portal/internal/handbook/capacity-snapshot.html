Capacity snapshot

What the system can actually do, with the numbers that back the claim and the saturation signal that proves it. Every row says where the measurement came from, when it was taken, and what the ceiling is. Honest "not measured" rather than aspirational targets — a capacity number without a saturation signal is a guess.

Rules for this page

  • Every number cites its source. Either a postmortem (real-world measurement), a load-test run (controlled), or "not measured" (honest gap).
  • Every number cites its saturation signal. The signal is what tells you the number is real — without it, the number is folklore.
  • Targets are separate from measurements. Where we know what we want but have not measured it, the cell says target not measured.
  • Updated every quarter. Or after any architectural change (storage swap, new endpoint family, new dependency). Last-updated below.

Storage ceiling

SQLite has a single-writer constraint: at any moment exactly one process holds the write lock, and concurrent writers serialise on it. This is acceptable at the current scale; it is the reason a retry storm crashed the write path during the 2026-04-02 load test and the reason Postgres is named as the planned swap.

BackendWrite ceilingRead ceilingConcurrency model
SQLite (today) ~50 sustained write RPS against a single hot row; higher against distinct rows not measured, expected ~1000+ RPS in WAL mode single writer, many readers
Postgres (planned) row-level locks; ceiling moves to connection pool size and DB CPU same as today, plus replica fan-out when added row-level locks; multi-writer

What changes these numbers

  • Storage swap to Postgres. The single-writer ceiling disappears; the new ceiling is connection-pool × DB CPU. The read-side stays the same. Postmortem 2026-04-02 will become historical.
  • Horizontal scale of the api process. Multi-instance deployment requires moving the rate limiter out of process (currently in services/api/app/core/security.py); see architectural narrative § what we deliberately did not build.
  • A new endpoint family that does heavy reads. Adds a new row here, with its own NFR latency in the spec. Default to "inherited" until measured.
  • Observability pipeline goes live. Today the metrics that would prove these numbers are scraped but not dashboarded. The 2026-04-02 postmortem named this as a gap; until it closes, "we know what to measure" is not the same as "we are measuring".

Forbidden patterns

  • Publishing a capacity number without a saturation signal. The number is unfalsifiable without the signal. Don't.
  • "It scales horizontally" without naming the bottleneck. Every system scales horizontally until the bottleneck reveals itself. Name the bottleneck or write "not measured".
  • Targets dressed up as measurements. If you have not run the load test, the cell says target and links nothing. If you have, it says measured and links to the postmortem or run.
  • Snapshot drift. A capacity page that hasn't been touched in a year is folklore again. Quarterly review or after architectural change, no exceptions.
