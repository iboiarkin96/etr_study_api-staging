SRE Explanation

What logs are for

A log line is an answer to a question someone has not asked yet. Treating logging as engineering — structured, levelled, joined by identifier — is what makes that answer findable. Treating it as printf is what makes it noise.

Logs vs metrics — different jobs

If monitoring tells you that something is wrong, logs tell you what happened to whom. The two answer questions of different shape, and conflating them makes both worse. Counting log lines at query time is slow and expensive; trying to debug one specific failing request by staring at a dashboard is impossible. Each artefact does the job the other cannot.

This page is about why we structure logs the way we do. The binding contract — the field set, retention, the local Elasticsearch + Kibana setup — lives in ADR 0023.

Why structured, not free-form

A free-form log line — “user 42 failed to create a conspectus because the duplicate check rejected the composite key” — reads well to a human and is nearly useless to a machine. Filtering for “all failures by user 42” requires regex; counting “failures by reason” requires text mining; correlating with a metric is impossible because the line has no identifier.

The same event as a structured log line — a JSON object with user_id, action, outcome, reason, request_id — supports all three queries in milliseconds, and stays correlatable with metrics and traces because the identifier is shared. The cost is upfront: choosing a field name and using it consistently. The payback is every future search.

The consistency is the load-bearing property. A field that is called user_id in one module and uid in another is two fields as far as the search engine is concerned. The discipline is to pick a name and reuse it, even when the alternative looks shorter.

Levels as severity, not as verbosity

Log levels exist to answer one question: at what level of trouble should I look at this line? Used well, the level is a signal. Used badly, it is a knob to tune how much output appears.

  1. debug Detail for local development or for chasing a specific bug. Off in production by default; never alerts.
  2. info The normal life of the request — “request started”, “job finished”. Routine, expected, not actionable.
  3. warning The system kept working but something is suspicious. A retry that succeeded after one failure; a deprecated path was hit.
  4. error One operation failed for a clear reason. Always actionable in principle — if it is not, the level is wrong.
  5. critical The service or a major part of it cannot continue. Pages.

The reliable failure mode is level drift: routine events get logged at error because the author wanted them to stand out; a real error gets logged at info because error is now too loud. After a few months of this, the level no longer means anything.

The corollary is the alert hygiene rule: a log line at error that appears every minute and that nobody acts on is not an error. It is noise wearing an error's clothes. Either lower the level, fix the underlying condition, or aggregate the line.

The join key is the whole point

Logs gain most of their value from a single property: every line for one request shares the same request_id. That identifier is what turns a stream of independent events into a story. With it, you can pick one failed line, copy the identifier, filter on it, and read everything that one request did in the order it happened. Without it, you have a haystack and no needle.

Generating the identifier at the edge and propagating it through every log line is therefore not optional decoration; it is the architectural feature that makes the rest of the structure pay off. The same identifier — when it can be propagated — also pivots into metrics labels (subject to cardinality limits) and trace spans, which is why observability treats the three pillars as one system rather than three.

Why redaction is policy, not paranoia

Logs are searchable, exported, and stored. They are inspected during incidents, sometimes by people outside the immediate team, sometimes by automated pipelines. The right mental model is that anything written to a log is broadly visible — to your colleagues today, to backups in six months, to whoever inherits the system in two years.

That changes what belongs in a log. Tokens, passwords, raw bodies that may contain user content, personal data — each is a liability the moment it is captured, and the cost of cleaning it up later is much greater than the cost of not logging it now. A redaction filter at the edge is a safety net, not a substitute: it catches the values that slipped through, but it does not catch the ones the author quietly decided were “probably fine”.

The strategic discipline is to not log the value in the first place. The filter catches accidents; the policy is what prevents them.

The cardinality trap, and where it lives

A log line can carry as many high-cardinality fields as you like: user_id, conspectus_id, even individual error messages. A metric label cannot. This asymmetry is the single most common cause of monitoring outages, because the temptation — “if it works as a log field, it works as a label” — is intuitive and wrong.

The split is what makes the three telemetry pillars complement each other. Metrics are cheap because their dimensions are bounded. Logs are expressive because their dimensions are not. Traces sit between the two: rich per-request structure, cheaper than logs because the keys are not free-form.

When in doubt about where a piece of data belongs, ask how many distinct values it could take. If the answer is “unbounded”, the data goes in logs, possibly traces, never metric labels.

Tradeoffs

Structured logging buys queryability and contract stability; it costs verbosity, storage, and the cognitive load of a schema everyone has to agree on.

Win

Logs are queryable like a table

"All request_done for user X in the last hour" is a single Kibana query, not a grep across 12 files. Investigation collapses from minutes to seconds.

Cost

Schema drift becomes a bug

Renaming a field silently breaks every dashboard built on it. Logs are a contract, just like the API; treat field renames as breaking.

Win

Correlation IDs make multi-step incidents tractable

A single request_id threaded through router → service → repo turns "what happened to that one user?" into a single query that returns 40 lines instead of 40 grep sessions.

Cost

Storage scales with traffic, not with events

Verbose info-level logs at peak generate gigabytes per day. We pay in storage, indexing latency, and the discipline to drop debug-level logs that nobody reads.

Common failure modes

  • Logging the same event twice. Once by the framework, once by the handler. Searches double-count; storage cost doubles; the second line carries no new information.
  • Variable data in msg. The user id ends up in the human-readable summary instead of a field; full-text search is noisy and aggregation is impossible.
  • Free-form keys. A field called uid here and user_id there. The schema is in everyone's head, inconsistently.
  • Missing request_id. You can see that things failed; you cannot reconstruct what one user saw.
  • “We will redact in the pipeline.” The value is logged with the intention of being scrubbed downstream. The scrubber breaks, or misses an edge case, and the value is now indexed.
  • Levels as a volume knob. Bumping things up or down to control how loud the logs are. Once levels stop reflecting severity, the on-call cannot trust them.
