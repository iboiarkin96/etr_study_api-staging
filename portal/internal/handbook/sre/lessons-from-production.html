Lessons from production — what the incidents on file actually taught us

The three postmortems on file describe specific incidents. This page extracts the reusable patterns that hold across all three — the lessons that would have prevented the next incident of the same shape, not just the one that already happened. Treat the postmortems as evidence, this page as the rule.

Why this page exists

A postmortem describes one incident in detail and produces SMART action items that prevent its exact recurrence. That is the load-bearing work; without it the next incident has nothing to point at. But a postmortem corpus is also a corpus — patterns repeat, and the second time a class of incident appears it is too late to call it a surprise. This page is where the cross-cutting patterns live so they can be cited at design time, not at the next on-call shift.

Today's corpus is three incidents: OpenAPI baseline drift, SQLite write-lock burst, docs-fix search-index drift. The patterns below are not pretending to be universal — they are what three incidents make plain. The next two or three will sharpen or contradict them; this page evolves accordingly.

Pattern 1 — Client behaviour is part of the contract

The SQLite incident's root cause was not a server bug. It was a load generator with an aggressive retry-on-5xx policy and no jitter. The server's idempotency layer worked exactly as designed; it just could not keep up with a thundering-herd retry the client had been told to send. The lesson is not "tune the load test" — the lesson is that any API which advertises idempotent retry is making a contract with the client about how retries should be issued, and a contract that names only one side is a contract that breaks on the other side first.

What this means in practice: every write endpoint should document, in §11 (Idempotency & concurrency) of its spec, the expected client retry shape — exponential backoff with jitter, bounded budget, hard cap on attempts. The OpenAPI document can carry this in the operation description. The runbook that handles 5xx bursts should ask "is the client retrying sanely?" before "is the server saturated?". This is uncomfortable because it shifts blame off the server; that is the entire point.

Pattern 2 — "Tests pass" is not the same as "contract safe"

The OpenAPI baseline drift incident's root cause was that make test ran every test green — including the tests the engineer had updated to match the new field name. The local test suite asserted the new shape; the contract test asserted continuity with the previously shipped artefact. Only the second is the contract, and only the second caught the break.

The pattern is broader than OpenAPI. Any artefact whose shape ships to a consumer — the wire schema, the search index, the error catalogue, the OpenAPI baseline, a published metric name — is a contract, and the test suite that asserts the current shape is not the test that asserts continuity. Continuity tests have to compare today's output against yesterday's committed baseline. CI is the cheapest place for this; the developer's terminal is cheaper still (pre-push hook). The 2026-03-14 postmortem closed the gap for OpenAPI by adding a pre-push hook; the 2026-05-07 postmortem closed it for the search index by rewriting the CI check. Other artefact contracts are still implicit; review them.

Pattern 3 — Silent failures need loud signals before they need fixes

The docs-fix search-index drift sat undetected for six days. The signal that would have caught it — a search that returns zero results — existed, but no telemetry counted it. The fix in the action items is twofold: a CI check that re-runs the generator and diffs the output (the loud signal), and zero-result-search telemetry (the louder signal). The CI check landed; the telemetry is open.

The pattern is: a class of failures is invisible to the user (they assume the feature is missing rather than broken). For these, the first action item is never "fix it" — it is "make it visible". A failure that pages someone within an hour gets fixed within a day; a failure that pages no one survives until a customer reports it, which is the worst kind of detection. Every silent-failure-mode in a system deserves an explicit "would we know?" answer; if the answer is "we'd know from a customer", that is an outage class, not an annoyance class.

Pattern 4 — The cheapest place to catch a contract break is the developer's terminal

This is the OpenAPI lesson restated for emphasis: make test took 12 seconds; CI took 4 minutes; the integrator-detection scenario would have taken hours or days. The cost-of-fix curve is well-established, and every step along it moves the cost by a factor. The OpenAPI baseline diff is a 4-second check on the developer's terminal when it lives in a pre-push hook; it is a 4-minute CI loop when it lives in make verify; it is a production rollback when it doesn't exist. Move every check that can live in pre-push to pre-push, even if CI still runs it as the gate. The pre-push hook is the developer-facing version of the same check.

The corollary: --no-verify is one git ergonomic switch with no fine-grained control. A pre-push that runs a fast subset of CI is the right shape; a CI gate that catches what pre-push misses is the safety net. The 2026-05-07 incident showed what happens when there is no safety net under --no-verify.

Pattern 5 — A generated artefact in git is a contract

Both 2026-05-07 and 2026-03-14 turn on the same observation. The OpenAPI baseline, the search index, the rendered SVG diagrams, the regenerated error catalogue — every artefact that the build produces and that the repo commits — is a contract. Consumers (CI checks, downstream pages, integrators' deserialisers) read these as ground truth without re-computing them. They are a contract in exactly the way a wire schema is a contract: the version in the repo is the version the consumer believes.

This has two implications. One, every generated artefact deserves a continuity check: regenerate on a clean tree, diff against the committed version, fail the build if they differ. The 2026-05-07 postmortem rewrote make docs-check exactly along this rule. Two, generated artefacts deserve their own consumer documentation. A search-index consumer should be able to find a page that says "this artefact is regenerated from these pages by these scripts; CI guarantees freshness". Without it, the artefact looks hand-maintained, and hand-maintained artefacts rot.

Tradeoffs

Extracting patterns from a postmortem corpus buys reusable rules at the cost of generalisation that may not hold. Three incidents are a corpus; they are not a sample size.

Win

Patterns can be cited at design time

A future spec author writing the idempotency section can be pointed at "Client behaviour is part of the contract" with one link. That is cheaper than re-deriving the rule from the postmortem during review.

Cost

Three incidents may not be a pattern

The temptation is to over-generalise. A pattern grounded in one incident is a hypothesis; a pattern grounded in three is a hypothesis with thicker evidence. Treat the rules as defaults, not laws.

Win

A corpus has predictive value

The patterns describe what the system's weakest points look like today. The next incident is more likely to match a listed pattern than to be novel; that is what makes prevention cheaper than detection.

Cost

Patterns age

A pattern derived from an incident two years ago against a system that has since been rewritten may no longer apply. This page deserves a re-review after every major architectural change; otherwise it becomes folklore.

Forbidden patterns

  • Reading the postmortem instead of citing it. A postmortem describes the specific incident; this page describes the rule. Both are useful; conflating them turns one-off lessons into vague generalities.
  • Adding a pattern from a single incident. The bar is "this rule, applied at the time, would have prevented at least two postmortems on file (or one + a plausible counterfactual that we can name)". Single-incident rules go in the postmortem's action items, not here.
  • Lessons without a "would have" test. A lesson that does not name what behaviour change it asks for is a thought, not a lesson. Every rule on this page can be applied as a check at design time.
  • Ignoring a contradiction. When a new postmortem produces evidence that contradicts a listed pattern, the contradiction is data. Either narrow the pattern's scope or remove it. Patterns that survive contradiction become dogma.
