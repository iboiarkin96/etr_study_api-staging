SRE Explanation

Debugging as a discipline

Why moving from a symptom to a cause is a methodical process, not an act of inspiration — and why the difference between a trigger, a cause, and a contributing factor is what makes fixes stick.

Debugging is not incident response

Incident response is about stopping pain quickly: roll back, fail over, drain the queue. The goal is restoring the system, and you may achieve it without understanding what went wrong. Debugging is about understanding what went wrong well enough to fix it for good — to write a sentence that explains the failure, satisfies you that the same failure cannot quietly recur, and points to the change that prevents it.

The two activities often run in parallel under incident pressure, and they get confused because both are urgent. They are not the same thing. Incident response that never feeds into debugging produces a system that has been rescued many times and learned from none of them.

The diagnostic loop, and why it works

Effective debugging looks like the scientific method. A loose narrative — “let me try this” — burns time on guesses, because pattern-matching against the noise of a real system is unreliable. The disciplined alternative is to form an explicit hypothesis, predict what data would be consistent with it, look at the data, and update the hypothesis based on what you saw. Each cycle costs minutes; each iteration narrows the search space.

  1. Reproduce A bug you cannot trigger is a bug you cannot study. Reproducing it is the first proof that you understand the conditions.
  2. Hypothesise Write it down: “X is happening because Y.” Writing forces specificity; specificity is testable.
  3. Predict “If Y, then I should also see Z.” The prediction is what keeps the next observation honest.
  4. Test cheaply Read what is already there before adding instrumentation. Cheap tests narrow the field faster than rich ones.
  5. Update Confirmed, refined, or killed — the hypothesis must change shape after each observation. If it does not, you are not learning.

The discipline that distinguishes practitioners from beginners is not where to look first — that is local knowledge — but writing the hypothesis down before looking. Without it, the eye sees what it wanted to see; with it, the eye sees what is there.

Bisection — why halving the search space wins

For most non-trivial bugs, the unknown is where the failure lives: in which time window, which endpoint, which subset of users, which commit. Linear search through the search space — try this, try that — has a worst case proportional to the size of the space. Bisection has a worst case proportional to the logarithm of the size.

The discipline transfers across dimensions. If today is broken and last week was fine, asking about Wednesday rules out half the time window. If the symptom is “the API is slow”, asking whether every endpoint is slow rules out — or rules in — the framework as a culprit. If only some users see the failure, finding the smallest pattern that captures them is bisection along the dimension of the population.

What makes bisection work is the willingness to commit to a midpoint and accept whichever half the evidence eliminates. The temptation to “check both sides to be sure” is the temptation to do linear search. It feels safer; it is slower.

Trigger, cause, contributing factor — three depths of answer

A failure is a chain, not a single event. The diagram below reads left-to-right: a trigger activates a cause; the cause was enabled by one or more contributing factors. The depth at which you stop the investigation determines how thoroughly you prevent recurrence.

  1. Trigger What happened

    The event just before the symptom.

    • A deploy, a config change, a traffic spike.
    • Easy to spot — but fixing only this lets the same failure reappear from a different angle.

  3. Cause Why it broke

    The technical mechanism of the failure.

    • “The new code path read a null field.”
    • Fixable in code — but the system shipped it anyway.
  4. enabled by
  5. Factors Why we allowed it

    Conditions that let the cause ship.

    • Missing test, missing migration, brittle dependency, unclear ownership.
    • Almost always more than one — that is normal.

Stopping at trigger — “the deploy caused it” — is the shallowest answer; the same trigger will arrive again. Stopping at cause is better but rarely sufficient: a fix at this layer assumes the system will catch the same class of bug next time, which it will not unless something else changed. The fixes that durably prevent recurrence are almost always in the contributing factors layer — the missing test, the unclear ownership, the dependency without a fallback.

This is why postmortems insist on asking “why?” multiple times. Each why moves you one layer deeper. The answer at the deepest layer is rarely a person and usually a system property: a feedback loop that is missing, a check that does not exist, a category of change that nobody owns.

A differential diagnosis — what categories of cause hide where

An experienced debugger is partly a list of priors. Certain shapes of symptom strongly suggest certain shapes of cause, and starting the hypothesis loop with the prior in mind shortens the path. The patterns are not rules — they are starting hypotheses, and a good debugger discards them quickly when the evidence does not support them.

Shape of symptomCategories of cause that frequently produce it
Sudden 5xx spike right after a deployA change in the deploy diff; configuration drift between environments; a missed migration.
Slow endpoint, no errorsDatabase query plan change; N+1 access pattern; a dependency whose latency degraded silently.
Intermittent failure, no clear patternResource exhaustion at a percentile (pool, file descriptors); retry storms amplifying a downstream blip; race condition.
Works in one environment, fails in anotherEnvironment-specific configuration; feature flags; data shape differences; migration order.
Background work stuckWorker liveness or oom; queue head-of-line blocked by a poison message; idempotency hazard on retry.
“It started on the 1st of the month”Scheduled job; data growth crossing a threshold; certificate or token rotation.
Dashboards say fine, users say brokenSampling that hides the affected segment; the wrong metric being measured; silent exceptions.

The reason these are priors rather than recipes is that any of them can be wrong. Treating them as recipes — “the deploy diff is always the cause of a deploy-aligned spike” — produces confident wrong answers. The point is to start the loop somewhere informed and update fast.

Tradeoffs

The disciplined loop costs time you don't feel you have. Skipping it costs more — but the cost is invisible until the wrong guess has already wasted an hour.

Win

Written hypotheses prevent confident wrong answers

A written guess is checkable against later evidence; an unwritten one quietly mutates to fit. The 30 seconds of writing pay back the first time you would have anchored.

Cost

Discipline feels slower than instinct on small bugs

For a one-line typo the loop is overkill; experienced engineers will (rightly) skip it. The trap is when you skip it on a real bug because the last three felt like typos.

Win

Bisection beats hunting

Halving the search space twice gets you closer than 60 minutes of "where could it be?". git bisect, binary diff of working vs broken config, halve-the-input — all cheaper than they look.

Cost

Bisection requires a reliable reproduction

When the bug is flaky, halving is statistical, not deterministic. Each bisection step takes longer because you have to retry to be confident in each "good/bad" classification.

Common failure modes

  • Pattern-matching without a written hypothesis. You see something that looks like a thing you have seen before and stop looking. Half the time the resemblance is coincidence.
  • Adding logging before reading existing logging. Instrumentation is expensive to ship and slow to read. Exhaust what is already there first.
  • Stopping at the trigger. A satisfying answer that does not prevent recurrence. The next deploy fires the same chain.
  • Confirmation bias under pressure. Once a hypothesis becomes appealing, contradictory evidence is rationalised. The remedy is to write the prediction before looking.
  • Investigating exhausted. Past a certain point, additional time on the keyboard produces fewer correct conclusions than stepping away. The team's job is to make stepping away socially acceptable.
