Runbook Observability

0003 — Logging failing

No log lines, bad LOG_DIR path, or missing write permission on logs/.

Impact

  • Local developer loop blocked until the failing stage clears.
  • No production user impact unless the failure is in shipped artefacts.

Follow-up

  • Capture the root cause in the related postmortem if SEV-2 or higher.
  • Open backlog items for any preventive change discovered during recovery.

Trigger

  • No new entries in logs/app.log.
  • App crashes on startup with logging-related error.
  • Wrong log path or no write permission.

Fast triage

  1. Check .env: LOG_DIR, LOG_FILE_NAME, LOG_LEVEL.
  2. Check directory and file permissions for logs/.
  3. Start app and send test request: make run, then call /health.

Most common causes

  • Invalid/unexpected LOG_LEVEL value.
  • Non-writable log directory.
  • App started from a different folder; relative paths break.
  • Logging code changed but startup in services/api/app/main.py does not match.

Recovery steps

  1. Reset to known-good defaults in .env: LOG_DIR=logs, LOG_FILE_NAME=app.log, LOG_LEVEL=INFO.
  2. Restart app: stop server and run make run again.
  3. Verify request log line appears after calling /health.
  4. If still failing, inspect services/api/app/core/logging.py path/handler setup.

Exit criteria

  • New requests write lines to logs/app.log.
  • App startup is stable.
  • make test still passes.

Escalation

If 30 minutes of triage didn't restore logging, or the failure mode isn't on the «Most common causes» list:

  1. Capture state — app startup output, tail -200 logs/app.log, the relevant logging config (app/config/*, env vars on LOG_*), and recovery steps tried.
  2. Open a SEV-3 incident in the team backlog and link this runbook from it. Lost-logging on a production-facing build jumps to SEV-2 — every minute without logs is a minute of blind operation.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last-known-good commit hash.
  4. If a rollback is config-only, revert the offending env / config commit before debugging further; logging-loss windows compound incident cost.
  5. If the failure blocks a release tag, escalate to SEV-2 and queue a postmortem (how to write one).

Follow-up

  • If this changes observability policy, add or update an ADR.
  • If it is a new failure pattern, update this runbook.
