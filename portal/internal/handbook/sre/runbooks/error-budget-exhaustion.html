0009 — Error budget exhaustion and SLO alerts

SLO alerts firing — sustained 5xx burn, latency violations, or readiness probe failures. Incident response and feature-freeze rules.

Impact

  • Local developer loop blocked until the failing stage clears.
  • No production user impact unless the failure is in shipped artefacts.

Follow-up

  • Capture the root cause in the related postmortem if SEV-2 or higher.
  • Open backlog items for any preventive change discovered during recovery.

Trigger

  • Prometheus alerts StudyAppErrorBudgetExhausted, StudyAppErrorBudgetCritical, StudyAppFast5xxBurn, StudyAppHighHttp5xxRate, StudyAppLatencySLOViolation, or StudyAppReadyProbeFailing are firing.
  • Grafana shows error budget burn over 1, a high 5xx rate, or readiness at 0.
  • Users see an outage or you breach an external SLA at the same time.

What "error budget exhausted" means

For this API we allow at most 0.1% of requests to return HTTP 5xx in a rolling 30-day window (99.9% success). The rule job:study_app:http_5xx:burn30d compares real 5xx share to that allowance. When burn is above 1, you have used more errors than budget for that window.

Full policy and SLI text: ADR 0011: SLOs, SLAs, error budget, and monitoring alerts.

Fast triage (order)

  1. Readiness / dependencies — If StudyAppReadyProbeFailing fires, use Observability scrape failing: API up, DB reachable, curl -i http://127.0.0.1:8000/ready returns 200.
  2. 5xx and latency — Open Grafana dashboard Study App Observability and Prometheus Alerts / Graph.
  3. Recent change — Look for deploys, config edits, migrations, or traffic spikes in that window.
  4. Rollback — If a new release is likely the cause, roll back to the last good version.
  5. Saturation — If latency SLO fires but 5xx are low, check CPU, memory, pools, and rate limits.

Most common causes

  • Sustained 5xx spike after a recent deploy, config change, or migration.
  • Latency degradation that drives timeouts (saturation: CPU, memory, pool exhaustion).
  • Dependency or readiness failure (DB unreachable, downstream returning 5xx).
  • Traffic spike that exceeds rate-limit capacity for the current window.

Recovery steps

  • Incident — Sustained burn or user impact: name a lead, use one channel, write a short timeline.
  • Feature freeze — Until things are stable, pause non-urgent releases; ship only fixes for the incident.
  • Root cause — After mitigation, record cause and follow-ups (tests, alerts, ADR if needed).
  • External SLA — If you have a published SLA, follow its customer comms and escalation steps.

Escalation

Error-budget exhaustion is inherently SEV-2 or higher — SLOs are violated by the time this runbook fires. If 15 minutes of response hasn't bent the burn-rate curve, or the failure mode looks like a release rather than a one-off spike:

  1. Capture state — Grafana dashboard screenshot, the firing alert list (http://127.0.0.1:9090/alerts), the recent release/config-change timeline (correlate burn start with deploy timestamps), and the response steps tried.
  2. Open a SEV-2 incident in the team backlog (not SEV-3 — error-budget exhaustion has external customer impact by definition) and link this runbook from it. If a published SLA is at risk of breach in the current window, escalate to SEV-1 and trigger customer comms per the SLA contract.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last known-good release tag. Do not wait for response steps to fully complete before paging — SLO breaches have a separate, faster acknowledgement SLA than regular incidents.
  4. If a rollback to the last known-good release tag is single-step (Docker tag swap, env var revert, traffic shift), execute it before debugging further; the burn-rate clock dominates the cost.
  5. Every SEV-2+ error-budget incident queues a postmortem (how to write one) — non-optional, regardless of duration.

Exit criteria

  • Firing alerts are resolved or acknowledged with a documented exception.
  • 5xx rate and latency panels are back within SLO targets on the Grafana dashboard.
  • Error budget burn is going down, or the team agreed a new window rule in writing.
