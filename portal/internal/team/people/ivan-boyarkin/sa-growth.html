Block 2. Theoretical directions (provable-maturity format)

If a technology cannot yet be promoted to production (for example, GraphQL/Kafka/gRPC), the competency is confirmed via 4 mandatory artifacts:

  • Architectural design: when and why the technology is needed, constraints, risks, alternatives.
  • PoC environment: a minimal working implementation in an isolated environment/branch.
  • Test contract: a repeatable check of schemas/compatibility/errors.
  • Operational runbook: startup, monitoring, typical failures, rollback/degrade scenarios.
Direction How to learn it and demonstrate the artifact without production rollout Value to the project Which competencies it covers
GraphQL (gateway/read-model pilot) 1) Design: define GraphQL boundaries (read-only use cases where aggregation matters).
2) PoC: stand up a local gateway over the existing REST.
3) Contract: schema lint + compatibility checks + negative cases.
4) Runbook: N+1, depth limit, timeout, rollback to REST. 		The team gains GraphQL practice without risk to the live environment and learns where it is genuinely useful. 21, 29
Kafka (eventing pilot without prod rollout) 1) Design: event model (topic, key, schema evolution, delivery guarantees).
2) PoC: producer/consumer in the dev environment for a single business event.
3) Contract: schema compatibility tests + idempotent consumer tests.
4) Runbook: lag, retries, DLQ, replay, emergency shutdown. 		Prepares the team for event-driven scenarios when the business prioritizes them. 21, 33, 34, 35
gRPC (internal service API pilot) 1) Design: where gRPC pays off (latency, typed contract, internal RPC).
2) PoC: one internal method + health endpoint.
3) Contract: protobuf breaking checks + error mapping tests.
