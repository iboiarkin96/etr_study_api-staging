ADR · RFC — numbered, durable decisions

Every significant decision is a numbered file with Context, Decision, and Consequences.

TL;DR

ADRs (Architecture Decision Records) and RFCs (Requests for Comments) are how we remember why the project looks the way it does. Each is a short, numbered file with three sections — Context, Decision, Consequences. ADRs record accepted decisions; RFCs propose them. The numbering is permanent; the file is never deleted, only superseded.

What it is

An ADR is a one-page record of an architectural decision. The page has three sections: Context (what we know that forced the choice), Decision (the path we picked), and Consequences (what we now have to live with). The file has a number that never changes.

An RFC is the same idea, but used while a proposal is still under discussion. RFCs invite the team to comment; ADRs record the result.

Both are written in HTML/Markdown, stored in the repo, and indexed by number. The index is generated from the file list.

Why we need it

A year from now, the team will not remember why we picked the inverted index over the cached grep. The ADR remembers. A new engineer reads ADR 0042 once and stops asking the question in chat.

The number gives every decision a stable link. We can say 'this was settled in ADR 0026' and the reader can jump straight to the record. No verbal retellings, no tribal knowledge.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Michael Nygard (originator)

Wrote the original ADR essay in 2011: a one-page record with Context, Decision, and Consequences for every architectural choice. Picked up by ThoughtWorks and the wider community within a year.

What we take: Three sections, one number, never edited after acceptance. Adding more fields kills adoption.

Source: Cognitect blog — Documenting architecture decisions

Spotify

Uses ADRs at squad and tribe level; their MADR-based template adds Date, Status, Deciders, and a Considered Options table on top of the Nygard original.

What we take: The extra metadata (Deciders + Status) makes the record traceable across a multi-team org without breaking the one-page rule.

Source: MADR (Markdown ADR) project

AWS Well-Architected

Recommends ADRs as the canonical artefact for the 'Operational Excellence' pillar; provides their own template in the Well-Architected Tool.

What we take: When a cloud provider names a practice, it shows up in audits — the ADR you wrote last year is the answer to next year's compliance question.

Source: AWS prescriptive guidance — ADRs

ThoughtWorks Tech Radar

ADR has been at 'Adopt' on the radar since 2016; ThoughtWorks teams ship ADRs alongside the code they describe.

What we take: An ADR next to the code (not in a wiki) survives reorgs, repo moves, and team changes.

Source: ThoughtWorks Technology Radar — ADRs

Across companies the rule is the same: one numbered file, three sections, never deleted, only superseded. The numbering is the contract — every link to ADR 0042 still resolves five years later.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Number

Sequential, never reused. 0042, not 'the search rewrite ADR'.

Title

One short sentence describing the decision, not the area.

Status

proposed / accepted / superseded by NNNN / declined.

Date

The day the decision was accepted, not when the file was created.

Context

What the team knew and why a decision was needed. Plain prose.

Decision

The chosen path, in the active voice: 'We will…'

Consequences

What changes because of this — both good and bad. Be honest about the bad.

How it works in this project

ADRs live under services/portal/internal/governance/adr/; RFCs under services/portal/internal/governance/rfc/. Each file is named NNNN-kebab-case-title.html. The index pages (adr/index.html, rfc/index.html) are auto-generated by CI from the directory listing.

We use the MADR 4.0 template, slightly trimmed to fit our HTML skeleton. A pre-commit hook checks that every new ADR has a status, a date, and at least one consequence.

Deep dive — the theory you need

MADR template — what each field captures

MADR (Markdown Architecture Decision Records) is the de-facto template. Its fields, with the question each answers:

FieldQuestion it answersNote
TitleWhat was decided?One sentence, active voice. 'Adopt OpenAPI 3.1 as the contract source', not 'API improvements'.
StatusWhere is this in its life?proposed / accepted / deprecated / superseded by NNNN.
DecidersWho is accountable?Names, not roles. Roles change; names are searchable.
DateWhen did this become real?Acceptance date; not creation date.
Technical storyWhat forced this?Link to the JIRA/issue/RFC that triggered the decision.
Context and Problem StatementWhat did the team know?Plain prose.
Decision DriversWhat did the team care about?Constraints and priorities, ranked.
Considered OptionsWhat else was on the table?Two or more, with one-line pros/cons each.
Decision OutcomeWhat did we pick?The chosen option plus the reasoning.
ConsequencesWhat now changes?Both good and bad. An ADR with no downside is half-written.

The supersession rule

An ADR is never edited after acceptance — typos can be fixed, content cannot. If a later decision replaces it, the old one is marked superseded by NNNN with a forward link; the replacement carries a back-link. The trail is the value: 'why did we change our mind?' is answered by reading two ADRs in order.

Trade-offs — what this practice costs

  • Number-as-commitment. Once ADR 0042 is merged, every link to it is a permanent promise. A team that hesitates to commit will hesitate to write ADRs, and the practice dies. Acceptance threshold must be lower than 'we are 100% sure'.
  • Status discipline. An ADR list that grows but never has anything moved to superseded is a sign the team doesn't revisit decisions. The supersession ritual costs minutes per quarter but pays back for years.
  • Hidden decisions stay hidden. ADRs only capture what the team thought to write down. Major implicit choices (which database, which auth flow) often have no ADR until something breaks. Periodic 'decision archaeology' is the only way to catch them.

Lifecycle

draft → proposed → accepted → superseded → declined

A new ADR starts as draft in a branch. When the PR opens, it becomes proposed. Merge moves it to accepted. If a later decision replaces it, the old ADR is marked superseded by NNNN — the file stays, the consequences are now history. Declined covers proposals that were considered and rejected.

Common pitfalls

PitfallWhy it happensWhat to do instead
Rewriting old ADRsAn old ADR turns out to be wrong; the author edits it.Never edit accepted ADRs. Open a new one that supersedes the old.
Vague titles'API improvements' tells the reader nothing.One sentence, active voice: 'Adopt OpenAPI 3.1 as the contract source'.
Missing consequencesThe author lists only the upsides.An ADR with no downside is half-written. Force at least one negative consequence.
Reused numbersTwo ADRs end up with the same number.CI lints the numbering: gaps are fine, duplicates fail the build.

Worked example

ADR sample

0020 — Adopt PlantUML as the only diagram source

Status. accepted, 2025-11-12.

Context. The project had a mix of Mermaid, ASCII, and inline SVG diagrams. Mermaid renders client-side, depends on a CDN, and the rendered output drifts between Mermaid versions. Accessibility tooling can't reach the SVG generated client-side.

Decision. We will use PlantUML for all diagrams in the docs portal. The source lives next to the page; CI renders it server-side; the resulting SVG is committed.

Consequences. Diagrams are pinned to a renderer version. Pages work without JS. Authors lose Mermaid's inline brevity — accepted. Onboarding requires teaching PlantUML, which is one extra page in the handbook.

Outcome. ADR 0020 is filed at /governance/adr/0020-c4-plantuml-diagram-style-and-conventions.html; CI rejects Mermaid blocks from the merge date forward; a one-line review comment on any Mermaid PR points back at the ADR.
