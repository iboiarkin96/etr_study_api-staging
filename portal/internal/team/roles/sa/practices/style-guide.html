Style guide as code — Vale, alex.js, cspell

One published style canon (Google or Microsoft), enforced by Vale rules that block PRs with style errors.

TL;DR

Without a linter, every author writes in their own voice. The reader sees a patchwork and slowly loses trust. A style guide as code fixes the tone at the PR level — no copy editor, no debates, just a build that fails when the prose drifts.

Looking for the canonical rules? This page is about the SA practice of automating style enforcement. The actual tone-of-voice, sentence, and page-structure rules that linters enforce in this portal live in the Documentation handbook → Style guide.

What it is

A style guide is the rulebook for the tone of voice. We adopt an established one (Google Developer Documentation Style or Microsoft Writing Style Guide) instead of writing our own — both are well-tested and well-supported by tooling.

Vale is an open-source linter for prose. The rules live in .vale/. alex.js flags non-inclusive language. cspell catches typos. CI runs all three on every PR.

Why we need it

Tone is invisible until it's wrong. A page that drifts in voice — formal here, breezy there — slowly erodes the reader's trust. A linter restores consistency without adding a copy-editor to every PR.

The rules also accelerate review. Reviewers spend zero time on 'we don't say it like that' and 100% of their time on content. The author gets feedback the moment they push.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Google

Maintains the public Google Developer Documentation Style Guide; used as the canon by many open-source projects.

What we take: Adopting an existing style guide saves the year you'd spend writing your own. Pick one, then customise sparingly.

Source: Google Developer Documentation Style Guide

Microsoft

Maintains the Microsoft Writing Style Guide with detailed sections on voice, tone, terminology, and inclusive language. Linted across docs.microsoft.com.

What we take: Microsoft's guide is the most opinionated; useful when you want explicit rules rather than principles.

Source: Microsoft Writing Style Guide

GitLab

Uses Vale to lint the entire handbook; rules under .vale/ are reviewed in merge requests like code.

What we take: Style rules in version control are the only style rules that survive turnover.

Source: GitLab — Vale configuration

18F (US Government)

Open-source content guide for federal digital services; explicit, simple, written for plain-language compliance.

What we take: Government content guides are excellent references because plain-language laws force concrete rules.

Source: 18F Content Guide

The common move: adopt an established style canon (Google, Microsoft, 18F), then add 5–10 project-specific rules in Vale. Never write a style guide from scratch.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Rule set

Inclusive language

Spell check

CI gate

Allow-list

How it works in this project

The style canon is Google Developer Documentation Style. Vale rules will live under .vale/styles/; cspell config at cspell.json; alex.js at .alex.yaml. A pre-commit hook runs all three locally so authors see the issues before push.

The rule set starts small: sentence length, voice (active by default), common confusions (then/than, its/it's), product-name capitalisation, banned weasel words. The set grows as the team finds repeat issues in review.

Deep dive — the theory you need

Vale architecture — how the linter actually works

A Vale rule is a YAML file with a fixed structure:

extends: existence       # or 'substitution', 'capitalization', 'occurrence', etc.
message: "Avoid '%s' — use 'X' instead."
level: warning           # warning | error | suggestion
scope: paragraph         # or 'sentence', 'heading', 'raw'
ignorecase: true
tokens:
- basically
        - obviously

The four most-used rule types

  • existence — flags any occurrence of a word. Used for banned/weasel words.
  • substitution — flags a word AND suggests the replacement.
  • occurrence — flags words that occur more than N times (used for 'too many sentences with just').
  • readability — flags sentences over a Flesch-Kincaid grade.

Starting set — five rules that cover most issues

  1. Weasel words (existence): just, simply, basically, obviously, of course.
  2. Banned substitutions: e.g. → for example; i.e. → that is; via → through.
  3. Sentence length (readability): max 25 words.
  4. Heading capitalisation: sentence case in headings.
  5. Inclusive language (existence): blacklist → blocklist; master → main.

Five rules, all as warnings, is the right starting point. Promote each rule to error only after it fires < once per 100 PRs.

Trade-offs — what this practice costs

  • Vale rules block PRs. Authors who hit a rule on every push lose patience. Start lenient (warnings), promote to errors only when a rule fires < once per 100 PRs.
  • Rule maintenance has its own backlog. A bad rule keeps firing; nobody removes it. Rules need a named owner and a quarterly cull, the same as any other linter config.
  • International contributors hit English-only rules. A non-native English author's prose may legitimately need patterns the linter rejects. Per-file allow-lists are honest; sweeping exceptions hide the problem.

Lifecycle

proposed → piloted → accepted → enforced

A new rule is proposed in a PR; piloted as a warning for two weeks; accepted once the team is comfortable; promoted to error and enforced as a PR gate. Starting strict drives authors crazy; starting lenient and tightening over time works.

Common pitfalls

PitfallWhy it happensWhat to do instead
Too many rules at onceEvery PR drowns in warnings; authors learn to ignore them.Start with five rules. Add one per week.
Rules with no exampleA rule fires; the author has no idea what to write instead.Every rule has an example of right and wrong in its YAML.
Rules without ownerA bad rule keeps firing; nobody removes it.Vale config has a per-rule owner; quarterly review.
Allow-list grows unboundedAuthors silence rules instead of fixing prose.CI counts allow-list comments per file; warn at >3.

Worked example

Vale rule

banned weasel words

# .vale/styles/Project/Weasel.yml
extends: existence
message: "Avoid weasel word '%s' — say what you mean."
level: warning
ignorecase: true
tokens:
- basically
- simply
- obviously
- of course
- just
          - merely

The rule fires as a warning on push. Authors fix the prose; reviewers don't have to flag it; the docs read tighter. Promotion to error happens once the rule fires < once per 100 PRs.

Outcome. The rule fires as a warning on push; authors fix prose; reviewers stop flagging weasel words manually; once the rate falls below 1 firing per 100 PRs, the rule is promoted from warning to error.
