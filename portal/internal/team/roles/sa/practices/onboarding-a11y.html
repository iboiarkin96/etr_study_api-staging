Onboarding · accessibility · i18n

README, CONTRIBUTING, devcontainer, WCAG 2.2 AA audit, and the path to a second UI language.

TL;DR

A new contributor should ship their first commit in one working day. Users on a keyboard or screen reader should read every doc page. Translating to a second language should not require a portal rewrite. These three goals share infrastructure and travel together.

What it is

This practice bundles three concerns that touch every contributor:

  • Onboarding. README + CONTRIBUTING + .devcontainer/ + CODE_OF_CONDUCT + good-first-issue labels. The path from git clone to a green PR is measured in hours.
  • Accessibility (a11y). WCAG 2.2 AA on every published page. axe-core in CI. Keyboard navigation and screen-reader sanity on a sample.
  • Internationalisation (i18n). The repo stores translatable strings in a way that lets a second UI language land without a rewrite.

Why we need it

Onboarding pays the team back on every new hire. A clean devcontainer means a junior is productive on day one instead of day three. A good first-issue label gives them a confidence-building first PR instead of two days of pairing.

Accessibility is two things: ethical (some readers cannot use the docs without it) and legal (the European Accessibility Act is enforced from 2025). Building a11y into the CI gate is much cheaper than retrofitting it after a complaint.

i18n is on the list because the lift is small if we plan early. A page that hardcodes English in every string requires a rewrite to translate; the same page with an extraction step is translated in an afternoon.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

GitLab

Public 'Getting Started' page for new team members; devcontainer + run-local script that bring up the whole product in under an hour.

What we take: Public onboarding pages double as recruiting material — candidates read them before they apply.

Source: GitLab — Engineering onboarding

Microsoft (devcontainers)

Originated the Development Container spec adopted across editors (VS Code, JetBrains, GitHub Codespaces).

What we take: A .devcontainer/ in the repo is the single highest-leverage onboarding investment — one-click reproducible environment.

Source: Development Containers spec

Mozilla

Public Accessibility Engineering guidelines; every Firefox release blocks on axe-core regressions.

What we take: Accessibility regressions caught at PR time cost minutes; caught after release they cost user complaints and legal exposure.

Source: Mozilla Accessibility

European Commission (EAA)

European Accessibility Act enforced from June 2025; covers e-commerce, banking, ticketing — and the docs/UI of any product sold in the EU.

What we take: Accessibility is now a compliance requirement, not a nice-to-have. WCAG 2.2 AA is the baseline for any EU-facing product.

Source: European Accessibility Act

The common thread: bundle onboarding + accessibility + i18n because they share the same lift (devcontainers, CI gates, string extraction). Doing them sequentially is more expensive than doing them together.

How it works in this project

Currently the repo has a README, CONTRIBUTING, and partial CODEOWNERS. The devcontainer is missing. Accessibility passes are manual; axe-core is not in CI. i18n strings are inline.

The rollout follows the order above. The devcontainer is the highest-leverage change — once a new contributor can run everything with one command, the rest of the onboarding pages get read more carefully because the reader is already invested.

Deep dive — the theory you need

WCAG levels — what each level requires

Web Content Accessibility Guidelines come in three levels. Each level is a superset of the previous:

LevelPromiseExamples
AMinimum. Without it the page is unusable for some.Alt text on images, keyboard navigation, no auto-playing audio.
AAPractical standard. Most regulations require this.Colour contrast ≥ 4.5:1, focus indicator visible, headings in order.
AAAHighest. Aspirational; some pages can't meet it.Contrast ≥ 7:1, sign language for audio, no abbreviations without expansion.

AA is the practical floor. The European Accessibility Act, US Section 508, and most other regulations point at it.

What axe-core catches and what it doesn't

Automated tools (axe-core, Pa11y, Lighthouse) catch about 30–40% of WCAG issues. The rest need a human:

  • Catches: missing alt text, low colour contrast, missing form labels, duplicate IDs, bad heading order.
  • Misses: alt text that says 'image' (passes the check, fails the user); keyboard traps in interactive widgets; screen-reader narratives that are technically correct but useless.

The right setup: axe-core in CI as the fast gate (catches 40% cheaply); a quarterly manual audit with a screen reader for the rest.

Trade-offs — what this practice costs

  • Devcontainer maintenance. A .devcontainer/ that bit-rots within a quarter is worse than no devcontainer — it tells a new hire the team doesn't care. Treat it as production code: CI builds the image weekly.
  • Accessibility audits cost time. Manual axe-core + keyboard + screen-reader runs are 2–4 hours per release. The CI gate catches 30–40%; the manual quarter is non-optional for AA compliance.
  • i18n pays off only at scale. String extraction adds 5–10% overhead to every new feature. Worth it when you have a second language coming in 12 months; pure overhead if you don't. Don't start i18n infrastructure without a translation commitment.

Lifecycle

unable → installed → green PR → contributor

Day 0: clone + devcontainer up. Day 1: a green PR on a good-first-issue. Within a month: an independent contribution. The onboarding documents shape this journey; a11y and i18n make sure the journey is available to as many people as possible.

Worked example

Good first issue

What the label points at

A good first issue meets four tests:

  1. Scoped. One file or one small folder. No 'understand the whole system' first.
  2. Verifiable. A clear acceptance test. The contributor can know they're done without asking.
  3. Reviewer ready. Someone is committed to reviewing within two working days. SLA in writing.
  4. Not blocked. No 'we should decide X first'. If X isn't decided, this isn't ready.

A label that points at issues failing any of these turns 'good first issue' into 'good way to lose a contributor'.

Outcome. The good-first-issue label is picked by 2–3 new contributors per quarter; first-PR time stays below one working day for that cohort; the reviewer SLA is honored because the label gate ensures someone signed up for the review before the issue went on the list.
