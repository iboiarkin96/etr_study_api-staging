Template registry — one shape per page type

One folder of Jinja templates for the major page types: ADR, RFC, Design Doc, Postmortem, Runbook, API spec, Audit, Practice.

TL;DR

A single registry of templates means every ADR has the same shape, every runbook has the same sections, every audit follows the same structure. Authors copy a skeleton instead of inventing one. Readers can predict where to find what they need on a page they've never seen.

What it is

A template registry is one folder that holds the canonical skeleton for every major page type. Each template is a Jinja file (or HTML with placeholders), with required sections, a short author guide, and a fixed metadata block.

The templates fix the grammar of the page — what sections appear, in what order. The style guide fixes the grammar of sentences — voice, length, terminology. They are different disciplines and have different owners.

Why we need it

Without templates, every author invents the page structure. Thirty ADRs look like thirty different projects; runbooks vary so much that on-call has to re-learn the layout for each alert. With templates, the skeleton is decided once; authors spend their effort on content, not layout.

Templates also make tooling possible. Once the shape is fixed, CI can check that every ADR has Status, Decision, and Consequences; that every runbook has Detect, Diagnose, Mitigate. The check is a one-line guarantee for every reader.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

GitLab

Maintains a registry of handbook page templates: each role, each process, each policy has a template; new pages must use one.

What we take: When the templates are mandatory, the handbook reads like one document instead of 3 000 different ones.

Source: GitLab handbook — templates and patterns

Backstage (Spotify)

Ships 'Software Templates' as a built-in feature: every new service is scaffolded from a template, including its catalog-info.yaml, README, CI config, and docs skeleton.

What we take: When the scaffold is the template, the template stays current because it's used every time.

Source: Backstage — Software Templates

Cloudflare

Runbook templates are part of the docs repo; alerts in PagerDuty link to the matching runbook page; the template enforces Symptom/Detect/Diagnose/Mitigate.

What we take: Required sections per template are the cheapest way to keep ops docs consistent across hundreds of alerts.

Source: Cloudflare Engineering blog

When the template is part of the scaffold (Backstage) or required by CI (GitLab), it stays alive. Optional templates always drift — engineers copy an old page instead.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Template file

One per page type. Jinja, HTML, or Markdown.

Required sections

Headings the renderer enforces. Missing = build fails.

Metadata block

Front-matter or HTML data-* with status, owner, date.

Author guide

One paragraph at the top of each template explaining when to use it.

Showcase

One rendered example in the UI Kit.

How it works in this project

Codified in RFC 0003. Templates live under services/frontend/portal/assets_v2/ui-kit/templates/ with showcase pages at services/portal/ui-kit/pages/templates/. Each template ships with a 'fill me in' marker that authors replace; CI fails when markers are left in published pages.

Currently nine templates cover most of the portal: doc-article, doc-tutorial, doc-howto, doc-reference, doc-explanation, landing-portal, landing-section, ops-cockpit, ops-runbook. ADR, RFC, Audit, and Practice templates are planned.

Deep dive — the theory you need

Required vs optional sections — how to decide

The first design choice for any template: which sections are required (the build fails without them) and which are optional?

Rule of thumb:

  • Required if a reader cannot use the page without it. Runbook without Mitigate is useless at 3 AM.
  • Required if a tool depends on it. Service catalog entry without an owner cannot route on-call.
  • Optional if some pages have it and some don't. ADRs sometimes have a Migration Plan; not always.

The 'template registry' folder

Putting all templates in one folder (_templates/) does two things: it makes the catalogue browsable, and it makes the templates themselves reviewable. A PR that adds a required section to a template touches one file; CI re-runs the validator on every page that uses the template.

One folder, named owners per template, version tag in the front-matter — that's the minimum for the registry to survive past launch.

Trade-offs — what this practice costs

  • Required sections feel pedantic. An ADR for a tiny choice still demands Context, Decision, Consequences. Authors complain; reviewers must hold the line. The discipline is the practice.
  • Templates calcify. A template written in 2024 may not fit a 2026 page type. Quarterly review of every template — keep it under 30 minutes; merge or retire any template no page has used in 90 days.
  • Template-creep. Once 'every page has a template', there is pressure to add a template for every nuance. Cap the registry at ~10 templates; if you need an 11th, ask whether two of the existing 10 should merge.

Lifecycle

template proposed → accepted → used → deprecated

A new template is proposed in a Design Doc that argues for its scope. Once accepted, the template lives in the registry and shows up in the UI Kit showcase. When a page type is no longer needed, the template is marked deprecated; new pages use the replacement.

Common pitfalls

PitfallWhy it happensWhat to do instead
Template-creepFive templates for slightly-different ADR types.Fewer is better. Differences should be sections, not separate templates.
Required sections driftSome ADRs have Consequences; some have Outcomes; some have neither.Templates declare data-required='Detect,Diagnose,Mitigate'; scripts/check_required_sections.py runs per PR and fails on any missing required heading.
Showcase is staleShowcase page doesn't match the actual template.Showcase pages declare data-template='ops-runbook'; CI regenerates them from the Jinja source on every merge — never hand-edited.
Author guide is missingTemplates exist; nobody knows which to pick.One-paragraph 'when to use' at the top of every template; link the four most common from CONTRIBUTING.

Worked example

Template skeleton

runbook.html (excerpt)

<!-- ops-runbook.html — template for SRE runbooks.
Use when: an alert exists; the on-call should follow this page at 3 AM. -->
<article data-template="ops-runbook">
<header data-section="hero">
<span class="pill pill--danger">Runbook</span>
<h1>{{ runbook.title }}</h1>
<p>Owner: {{ runbook.owner }} · MTTR target: {{ runbook.mttr }}</p>
</header>
<section id="detect" required><h2>Detect</h2>…</section>
<section id="diagnose" required><h2>Diagnose</h2>…</section>
<section id="mitigate" required><h2>Mitigate</h2>…</section>
<section id="post" required><h2>Post-incident</h2>…</section>
          </article>

Required sections are marked with required. CI fails any runbook that ships without one of them. The template's first line tells the author when to use it.
