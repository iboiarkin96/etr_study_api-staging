Glossary — one dictionary for the whole project

One curated YAML file of terms with canonical definitions; the HTML page and inline tooltips are generated from it.

TL;DR

When the same word means three different things to three engineers, every review turns into a debate about definitions. A glossary ends those debates. Each term has one canonical meaning, a list of synonyms to avoid, related terms, and a one-sentence example.

What it is

A glossary is a curated YAML file with one entry per term. Each entry has a canonical definition (one sentence), common synonyms (so authors don't accidentally introduce a second name for the same idea), related terms (so a reader can wander), and a one-sentence example. CI renders an HTML page from the YAML and the inline tooltips on every doc page.

The glossary is the ubiquitous language of the project, applied to the documentation.

Why we need it

Without a shared dictionary, the same concept slowly grows three different names. A new engineer reads contract in one doc, API in another, and endpoint in a third — all meaning the same thing. Mental models fragment; PR comments turn into definition fights.

A glossary makes the language one. It also disciplines requirements: when you have to pick the canonical term, you find out that your idea is fuzzier than you thought.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

BBC

Publishes a public 'Domain Dictionary' for engineering: each term has one definition, an owner, and a link to where the term is used in the codebase.

What we take: Public glossaries force precision — you cannot write a fuzzy definition that everyone can read.

Source: BBC — A11y glossary (engineering)

Mailchimp

The content style guide has a 'Terminology' section: canonical names for product features, with synonyms to avoid. Writers see warnings in their editor when they pick the wrong word.

What we take: Pair the glossary with a linter — definitions that aren't enforced drift within a quarter.

Source: Mailchimp Content Style Guide — Terminology

Microsoft Writing Style Guide

Adopts an A–Z 'word list' as the canonical vocabulary across all docs; every doc is linted against the list.

What we take: A flat A–Z list scales better than a tree of categories — authors don't have to guess where the term lives.

Source: Microsoft Style Guide — A–Z word list

The shared trick is to pair the glossary with a linter or editor warning. A glossary nobody enforces is a graveyard within six months.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Term

The canonical name. Lower-case, no plural, no punctuation.

Definition

One sentence. No jargon. No circular references.

Synonyms

Names we use to call this term in the wild. Authors get a warning if they use one.

Related

Other terms a reader might wander to next.

Example

One short usage sentence.

Owner

Who keeps the entry up to date.

Status

proposed / accepted / deprecated / removed.

How it works in this project

The glossary will live at services/portal/internal/_shared/glossary.yaml. A CI script (scripts/build_glossary.py) renders it into _shared/glossary.html and into a JSON file consumed by docs-glossary-tooltip.js on every page.

A Vale rule flags new prose that uses a synonym instead of the canonical term, and another rule flags terms that are not in the glossary at all. Both are warnings, not errors, so an author can introduce a new term and add it to the glossary in the same PR.

Deep dive — the theory you need

Ubiquitous language — the DDD root of the idea

The glossary practice is a small, concrete version of ubiquitous language from Eric Evans' Domain-Driven Design. The idea: across the team — domain expert, engineer, tester, doc author — every concept has exactly one name, and that name is used everywhere, from conversations to code to docs.

When the language fragments, three small failures compound:

  • Code reviews argue about naming because reviewers default to different words.
  • Tickets get assigned to the wrong team because they mention the wrong word.
  • New engineers learn three names for the same thing in their first week and waste days on the confusion.

The two-line glossary entry that works

Long definitions don't get read. The format that survives:

term: contract
definition: The written agreement between two parties about how they communicate.
For HTTP services this is an OpenAPI document; for async, an AsyncAPI document.
synonyms: [API spec, interface]
owner: ivan-boyarkin

One sentence for the definition; one clarifying example; a list of synonyms so the linter can warn authors who used a non-canonical word.

Trade-offs — what this practice costs

  • Onboarding friction. A new author who used 'API' for what the team calls 'contract' gets a Vale warning on their first PR. That's a real cost — onboarding glossary takes a week. Soft-fail for the first 30 days of a new contributor.
  • Definition by committee. A glossary owned by 'the team' drifts: three reviewers, three small edits, meaning slowly shifts. The named-owner rule is the only durable fix.
  • Dead terms. Glossaries grow without pruning. A quarterly review that removes terms no doc has cited in 90 days is unglamorous and essential.

Lifecycle

proposed → accepted → deprecated → removed

A term is proposed when a PR adds it; it becomes accepted once the owner approves. Terms we stop using move to deprecated with a pointer to the replacement, and are removed only after no doc still uses them.

Common pitfalls

PitfallWhy it happensWhat to do instead
Two entries for the same ideaAuthors don't search before adding.CI fuzzy-matches new entries; flags duplicates.
Definitions written by committeeThree reviewers, three small edits, meaning drifts.owner: is a required field in glossary.yaml; CI fails on a missing or unknown owner. Final wording is the owner's call.
Glossary becomes a graveyardTerms added but never removed.Quarterly review: prune what no doc has used in 90 days.
Recursive definitionsEndpoint: a URL that exposes an API. API: a set of endpoints.Each definition must stand alone, without reading related entries.

Worked example

YAML entry

contract — one canonical record

- term: contract
          definition: |
          The written agreement between two parties about how they communicate.
          For HTTP services this is an OpenAPI document; for async, an AsyncAPI document.
          synonyms: [API spec, interface, agreement]
          related: [openapi, asyncapi, schema]
          example: |
          The contract for the sessions endpoint lives in openapi.yaml.
          owner: ivan-boyarkin
          status: accepted

The renderer turns this into a page entry with the definition, synonym chips, related-term links, and a code-fenced example block.

Outcome. CI renders glossary.html with the contract entry; tooltips on every internal page show the one-sentence definition on hover; PRs that introduce 'API spec' as a synonym get a Vale warning pointing at the canonical term.
