C4 + PlantUML — diagrams as code

Architecture drawn at three levels — Context, Container, Component — with PlantUML as the only diagram source.

TL;DR

C4 is the layered way of drawing systems: zoom out for the big picture, zoom in for the parts. PlantUML is how we draw it — text in the repo, SVG in CI, no client-side rendering. The two together give us diagrams that stay in sync with reality because they live next to the code.

What it is

C4 is a system-architecture notation invented by Simon Brown. It says: don't draw one big diagram. Draw three smaller ones, each at a different zoom level.

  • Context. What is around the system — the users, the external services, the data sources.
  • Container. What the system is made of — services, databases, queues, front-ends.
  • Component. What is inside a single container — the modules, classes, or libraries.

PlantUML is the format we write those diagrams in: a small text DSL that CI renders into SVG. The source is in the repo, the rendered SVG is committed, and the same diagram looks the same on every machine.

Why we need it

A picture saves a new engineer a day of code reading. But only if the picture is current. A diagram in Figma or a Mermaid block that depends on a CDN slowly drifts away from the code — and a stale diagram is worse than no diagram, because the reader trusts it.

Storing the source in the repo and rendering server-side fixes both problems. Reviewers see the diff between the old and new diagram in the same PR; the rendered SVG is accessible (screen-readers, no-JS); the renderer version is pinned.

ING Bank

Standardised on C4 across hundreds of internal services; every service has Context and Container diagrams in its repo, rendered from PlantUML in CI.

What we take: C4 only works at scale when the source lives next to the code — diagrams in Confluence drift within months.

GitHub Engineering

Uses PlantUML for sequence and state diagrams in engineering RFCs; the diagrams are rendered server-side and committed as SVG.

What we take: Server-side rendering means the diagram works on every page that links to it, including the wiki, the PR view, and the editor preview.

Source: GitHub Engineering blog

The repeated lesson: pictures live longer when they're text in the repo, not exports from a design tool. The renderer is replaceable; the source is not.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Level 1: Context

One page. Shows the system as a single box, the users, and the external systems it talks to.

Level 2: Container

One page per system. Shows the containers (services, databases, queues) inside the box.

Level 3: Component

One page per container. Shows the components (modules, libraries) inside it.

Source

A .puml file next to the page that displays the SVG.

Renderer

PlantUML invoked in CI; version pinned in the build container.

Output

Committed .svg file. The HTML page <img>s it.

How it works in this project

The cross-service map at services/portal/internal/explanation/architecture-narrative.html follows the C4 layout. Each PlantUML source lives under services/portal/internal/handbook/uml/ next to a render target. CI runs tools/docs/regenerate_docs.py (step 1 of make docs-fix), which dispatches to PlantUML via Kroki and writes the SVG into handbook/uml/rendered/. The SVG is committed so the page works without JS and without a network call.

The style is locked in ADR 0020: same colour palette, same arrows, same legend on every diagram. Authors don't pick their own style; the team picks one.

Deep dive — the theory you need

The three C4 levels — when to draw which

C4 has four levels in the original model, but Level 4 (Code) is rarely useful because the IDE already shows the code. The three that matter:

LevelAudienceShowsCap on boxes
1. Context Anyone (PMs, executives, new hires) The system as one box + users + external systems it talks to ~10
2. Container Engineers joining the team What's inside the system: services, databases, queues, front-ends ~15
3. Component Engineers inside one container Modules, libraries, classes inside one service ~12 per container

The 'one diagram per question' rule

Each diagram answers one question. If you cannot state the question in one sentence, you have two diagrams pretending to be one.

  • Context diagram: 'What does the system talk to?'
  • Container diagram for service X: 'What does service X contain?'
  • Component diagram for module Y inside X: 'How is Y structured?'

If you find yourself drawing 25 boxes on one canvas, you are either at the wrong level (zoom out) or trying to answer two questions at once (split into two diagrams).

Why PlantUML, not Mermaid

Both are text DSLs that render diagrams. The difference is where they render:

  • PlantUML renders server-side in CI; the result is a static SVG that ships with the docs.
  • Mermaid renders client-side in the browser; the result depends on the user's JavaScript runtime and the Mermaid version loaded from the CDN.

Server-side rendering gives three things Mermaid cannot: accessibility (screen readers reach the SVG), version pinning (the same source always renders the same picture), and offline viewing. The project's ADR 0020 locks PlantUML for these reasons.

Trade-offs — what this practice costs

  • CI minutes. Rendering 30+ diagrams in CI costs ~30–60 seconds per build. For a docs portal with frequent merges, this can be 10% of build time. Cache renders by source hash to keep it manageable.
  • Learning curve. Engineers know how to draw on a whiteboard. PlantUML's syntax takes a week to feel natural; in the meantime they avoid drawing at all. Adoption requires a short workshop and a 'paired diagram' for the first three PRs.
  • Free-form sketches are gone. Some early-stage design drawings deserve to be quick whiteboard photos, not committed PlantUML. Allow a scratch/ folder for these — never confuse them with the canonical diagrams.

Lifecycle

draft .puml → CI renders SVG → SVG committed → page embeds SVG → review in PR

Author edits the .puml source. The PR triggers CI, which renders the SVG and pushes it back into the branch. Reviewers see the SVG diff inline; nothing needs to be regenerated locally. When the PR merges, the committed SVG ships with the docs.

Common pitfalls

PitfallWhy it happensWhat to do instead
One mega-diagram Author tries to fit context, containers, and components into one picture. Force the three-level split. If a diagram has more than 12 boxes, it is at the wrong level.
Mermaid sneaks back in Mermaid is convenient inline; an author drops one in. CI lints for ```mermaid; rejects the PR.
Diagram drifts from code A service is renamed; the diagram still shows the old name. CODEOWNERS routes diagram files to the same owner as the service; the rename PR has to touch both.
Style is per-author Every diagram looks different. Shared !include for colours, arrows, legend. Authors include it; do not redefine.

Worked example

PlantUML source

Context diagram for the portal

@startuml portal-context
!include <C4/C4_Context>
!include common-style.iuml
Person(reader, "Reader", "Engineer, auditor, or AI assistant")
System(portal, "Docs portal", "HTML docs served from the repo")
System_Ext(github, "GitHub", "Source of truth — PRs, CI, hosting")
System_Ext(grafana, "Grafana", "Dashboards referenced from runbooks")
Rel(reader, portal, "Reads docs, opens issues")
Rel(portal, github, "Fetched from", "git pull on build")
Rel(portal, grafana, "Embeds links to")
        @enduml

CI renders this into portal-context.svg; the architecture page embeds the SVG. Reviewers see a clean diff: + Rel(portal, grafana, ...).

Outcome. One PR changes both the diagram source and the rendered SVG; reviewers see exactly what the architecture map will show after merge, without having to render anything locally.
