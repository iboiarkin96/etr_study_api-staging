10 Group B · Architecture & modeling

DDD bounded contexts — one model per context

A map of bounded contexts: each service belongs to one context; relationships between contexts are typed.

TL;DR

Without an explicit map, the concept of 'User' silently grows three different definitions across three services. With a context map, every service belongs to one bounded context, and the relationships between contexts are named: Customer-Supplier, Conformist, Anti-corruption Layer, Shared Kernel.

What it is

A bounded context is a part of the domain where one specific model applies. Inside that context, words have one meaning. Across contexts, the same word may mean different things — and that's fine, as long as the boundary is explicit.

A context map draws the contexts as boxes and the relationships between them as typed edges. Five edge types cover most cases: Customer-Supplier (one team's API drives the other), Conformist (you adopt the upstream model as-is), Anti-corruption Layer (a translation layer protects your model), Shared Kernel (two teams co-own a small model), Partnership (teams plan releases together).

Why we need it

When microservice boundaries don't align with domain boundaries, the same concept appears in multiple services with slightly different shapes. Every cross-service change becomes risky; integration code grows; new engineers spend their first week learning whose 'User' is whose.

A context map names the alignment explicitly. Once a team can say 'we are a Customer of the catalog context and we use an Anti-corruption Layer', they have a vocabulary to talk about the boundary instead of arguing about the shape of one shared object.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Plaid

Restructured engineering around bounded contexts after a re-platform; each context owns its data model and exposes a typed API to the others.

What we take: Context boundaries are also team boundaries. When the context map is unclear, the org chart is unclear too.

Source: Plaid Engineering — Plaid's API platform

Yelp

Public talks on splitting the monolith into bounded contexts; uses an Anti-Corruption Layer between every pair of contexts that share concepts.

What we take: Default to Anti-Corruption Layer between any two contexts. Conformist relationships only when both teams plan releases together.

Source: Yelp Engineering — Bounded contexts

GitHub

Used DDD bounded contexts to plan the GitHub Issues + GitHub Projects + GitHub Discussions split; each is its own context with its own model.

What we take: Three products that look related are easier to grow when they are explicitly different contexts than when they share one model.

Source: GitHub Engineering blog

The repeated pattern: name the contexts, draw the relationships, and default to Anti-Corruption Layer between any two. Most refactor pain comes from sharing models across what should have been separate contexts.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Bounded context

A part of the domain with its own model and its own ubiquitous language.

Service

One service belongs to one context. Multiple services can share a context.

Customer-Supplier

Upstream team commits to a contract for the downstream team.

Conformist

Downstream adopts the upstream model with no translation.

Anti-corruption Layer (ACL)

Translation layer at the boundary; protects the downstream model.

Shared Kernel

A small model jointly owned by two teams.

Partnership

Two teams plan releases together; tight coupling, acknowledged.

How it works in this project

The context map lives at internal/analysis/context-map.html. It is one diagram (PlantUML, of course) plus a paragraph per edge describing the relationship and the responsible owner from each side.

Every new service or service-split goes through a Design Doc that names the bounded context and the relationship type. Splitting a service that changes the relationship requires an ADR; the context map updates in the same PR.

Deep dive — the theory you need

The five relationship types — when to use which

A context map shows boxes (contexts) and edges (relationships). The edge type is the important bit:

TypeWho has powerWhen it fits
Shared Kernel Two teams co-own a small shared model. Two teams with permanent overlap; small scope; high trust.
Customer-Supplier Downstream depends on upstream; upstream commits to a contract. Stable platform team upstream; product teams downstream.
Conformist Downstream adopts upstream's model as-is; no translation. Upstream is a third party (Stripe, Auth0); fighting them is futile.
Anti-Corruption Layer Downstream translates upstream's model into its own. Upstream model leaks concepts you don't want; default choice.
Partnership Two teams plan releases together; tight coupling, acknowledged. Two teams shipping one product feature together.

The default that prevents the most pain

If you cannot defend a choice in writing, default to Anti-Corruption Layer. It costs more upfront (a translation layer) but protects you from the most common failure: the upstream changes its model and the change ripples through three downstream services.

Conformist is the riskiest default — it locks you into upstream's evolution speed.

Trade-offs — what this practice costs

  • Vocabulary overhead. 'Anti-Corruption Layer' is jargon to a team that hasn't read Evans. Teams reject what they can't pronounce. Start with the simpler 'translation layer'; introduce ACL once the idea is familiar.
  • Over-modelling small systems. A 3-service system does not need a context map; the boundaries are obvious. Introduce DDD vocabulary at the team-of-teams level (~20+ services).
  • Event storming is missing here. The dominant discovery technique for bounded contexts is event storming — a workshop where domain events are written on stickies until clusters emerge. Worth adding to the team's playbook; mentioned by Vaughn Vernon as the cheapest way to find context boundaries.

Lifecycle

context proposed → context accepted → relationships drawn → service assigned → relationship change → ADR

A new context starts in a Design Doc. Once accepted, it appears on the context map with its relationships to neighbours. Adding or splitting a service happens inside an existing context unless the Design Doc argues for a new one. Changing the relationship type (e.g. from Conformist to ACL) is a structural change and gets its own ADR.

Common pitfalls

PitfallWhy it happensWhat to do instead
One context for the whole system Every service is inside the same context. Concrete check: if User.id in service A is a UUID and in service B is an email, you have two contexts. Admit it before the next migration.
Conformist by accident Downstream copies the upstream model without thinking. Default to Anti-Corruption Layer for any cross-team relationship; Conformist requires an ADR co-signed by both team leads.
Shared Kernel that grows A small joint model slowly becomes the whole domain. Cap Shared Kernel at ~5 entities; once it grows past 10, promote it to its own bounded context with a Customer-Supplier relationship.
Context map drifts Map shows three contexts; the code shows seven services in unclear places. Service catalog YAML includes the bounded context; CI fails if a service has no context.

Worked example

Context-map fragment

Sessions and decks

Sessions context. Owns the model of a study sitting: session, review, session_tag.

Decks context. Owns the model of a deck of cards: deck, card, card_variant.

Relationship. Sessions → Decks: Customer-Supplier with an Anti-corruption Layer. The Sessions service consumes the Decks API; an ACL inside Sessions translates Deck.card_count into the local SessionDeckSnapshot. The ACL means a future refactor of the Decks API does not ripple through Sessions.

Drawn on the context map as: [Sessions] <--ACL-- [Decks]

Outcome. The Sessions service owns SessionDeckSnapshot internally; a refactor of the Decks API stops at the ACL boundary and never ripples into Sessions or its consumers.
