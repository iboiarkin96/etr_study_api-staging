SSOT — one source of truth per fact

Every fact in the project has exactly one authoritative location. Everything else is derived from it or links to it.

TL;DR

The Single Source of Truth principle: pick one file to own each fact, then make every other appearance of that fact either generated from it or a link back to it. Drift is the silent killer of documentation; SSOT is how we prevent drift without manual diligence.

What it is

SSOT (Single Source of Truth) is the rule that every piece of data, configuration, decision, or fact in the project has exactly one authoritative location. Every other appearance — another file, another tool, another team's wiki — is either generated from the source or links to it.

The source can be edited; the derivatives cannot. If two places allow editing, they will eventually disagree, and the loudest copy wins — usually the wrong one.

Why we need it

When the same fact lives in two places, no rule tells the reader which is correct. The two copies drift, the engineer ships against the stale spec, and the on-call follows a runbook pointing at a dashboard that no longer exists.

SSOT prevents that quietly. The cost it pays is regeneration: a single edit may need a CI step to rebuild the derivatives. We accept that cost because silent drift is more expensive than explicit regeneration.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Spotify

Built Backstage on the idea of one machine-readable catalog-info.yaml per service; the catalog UI, on-call routing, dashboards, and dependency graphs all read from the same file.

What we take: A YAML file is a better source of truth than an HTML page, because tools can read it too.

Source: Backstage architecture

GitLab

The GitLab handbook (3000+ pages) is the explicit single source of truth for company process; if it isn't in the handbook, it isn't policy.

What we take: Saying out loud 'this is the SSOT' changes behaviour — it ends 'I heard in standup that…' as an answer.

Source: GitLab handbook — Handbook First

Stripe

API reference, SDK clients, and the public docs are all generated from the same OpenAPI document; engineers cannot edit the rendered HTML.

What we take: Removing the ability to edit derivatives is the cheapest way to enforce SSOT.

Source: Stripe API reference (generated)

Shopify (Sorbet typing)

Sorbet treats Ruby type signatures as the SSOT; runtime checks, IDE autocomplete, and CI all derive from the same .rbi files.

What we take: One file feeds three consumers (runtime, editor, CI). Drift becomes impossible.

Source: Sorbet — Typing Ruby at scale

The shared rule across all four: pick the source, make every other appearance a derivative, and remove the ability to edit the derivative. The discipline lives in the tool, not in the team's memory.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

The source

The one file that allows edits. Owner: one named person.

Derivatives

Anything generated from the source. Never edited in place.

References

Links back to the source. Renderable, but never copy the content.

The build step

The script (CI job) that regenerates derivatives. If it fails, the build fails.

The owner

One person who answers questions about this fact and reviews changes.

How it works in this project

Examples currently in use:

  • API contract. The source is openapi.yaml. SDK clients, mock servers, and the public reference page are generated from it.
  • Database schema. The source is the migration files. ORM models, ER diagrams, and table-doc pages derive from them.
  • Service catalog. The source will be catalog-info.yaml (Backstage format). The HTML catalog pages will be generated from it.
  • Design tokens. The source is tokens.json. CSS custom properties and Figma styles are derived.
  • Architecture decisions. The source is the ADR file in governance/adr/. Indexes and cross-links are generated.

Deep dive — the theory you need

Derivative types — and how to pick the right one

Not every derivative is the same. Three patterns cover most cases:

TypeHow it worksWhen to useTrade-off
GeneratedCI rebuilds the file on every commit to the source.Content changes when the source changes; reader expects 'always current'.Adds a CI step; build failure if regeneration fails.
ReferencedDerivative shows a link to the source; never copies the content.Content rarely changes; reader is willing to click through.Reader has to follow a link; not great for inline reading.
CachedDerivative is a snapshot, refreshed manually on a schedule.Source changes faster than readers need it (analytics dashboards).Staleness must be visible (timestamp); easy to forget.

The drift detector

SSOT works only if you can detect drift before users do. A small CI check covers most of it:

# pseudo-code: fail the build if any derivative is older than its source
for derivative in find_marked('# AUTO-GENERATED FROM'):
src = parse_banner(derivative)
if mtime(src) > mtime(derivative):
        fail(f'{derivative} is stale — re-run {src.regenerate_cmd}')

The banner pattern # AUTO-GENERATED FROM … at the top of every derivative is the cheapest way to make this work — the check finds the relationship without any registry.

Trade-offs — what this practice costs

  • Build-step latency. Every derivative pays a CI cost on every edit to its source. For docs this is fine (seconds); for API SDKs it can mean a 5-minute delay before a fix is testable. Budget the build time when you pick what to derive.
  • Generator failure modes. If render_service_descriptors.py silently produces an empty card region when the YAML is malformed, the team will trust the empty doc and ship the bug. Loud failures (non-zero exit, banner on the output) are non-negotiable.
  • Migration cost. Moving from 'docs in Confluence' to 'docs derived from code' is a quarter of full-time work. Budget it explicitly; do not promise it as a side-effect of another project.

Lifecycle

one source → build → derivatives → references

A change starts at the source. CI rebuilds the derivatives. Everywhere else, references are still pointing at the same authoritative file — they don't need to be updated because they never copied the content in the first place.

Common pitfalls

PitfallWhy it happensWhat to do instead
Hand-maintained nav treeSidebar JSON duplicates the filesystem.Generate the nav from a directory walk or a YAML manifest.
Two specs for the same APIOne is in the repo, one in a wiki page.Pick the file as the source; wiki page header gets a <meta http-equiv="refresh"> to the repo source on the agreed sunset date.
Inline copy of an enumAn enum is defined in the code AND restated in three docs.scripts/extract_enum.py reads the enum file and emits a Markdown table; pages include the table via {% include %} — never copy values inline.
Hidden derivativeA 'doc' page is actually edited and silently used as a source by automation.Every generated file carries a top-of-file banner (<!-- AUTO-GENERATED FROM x.yaml — run scripts/build.py -->); CI flags files modified without a matching source change.

Worked example

From source to UI

Service descriptor YAML → entity card + hub tile + sidebar

Source. services/portal/internal/services/api/catalog-info.yaml — Backstage-shaped descriptor with hand-authored fields (description, tags, links, x-study-app tone/lede/deps).

Build step. tools/docs/render_service_descriptors.py (step [6/10] of make docs-fix) reads every services/portal/internal/services/*/catalog-info.yaml, validates against the schema, and rewrites three derivative surfaces inside <!-- catalog:start --> / <!-- catalog:end --> marker blocks.

Derivatives. The entity card on the per-service index.html; the tile in the catalog hub services/index.html; the services subtree in nav-tree-internal.json. All three regenerate from one YAML.

References. Any prose mentioning the API service links to the entity card; never restates owner / tier / dependencies inline.

If a developer hand-edits the rendered HTML inside a marker block, make catalog-render-check in pre-commit fails. They cannot silently fork the descriptor.
