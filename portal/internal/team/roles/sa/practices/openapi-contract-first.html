OpenAPI contract-first — the spec is the agreement

The OpenAPI 3.1 document is written before the implementation; SDK, mocks, and tests are generated from it.

TL;DR

Write the OpenAPI spec first. From the spec, generate the SDK clients, the mock server (so the front-end can start), the reference docs, and the contract tests. The code implements the spec, not the other way around. CI fails when they drift.

What it is

Contract-first means the OpenAPI document is written, reviewed, and accepted before the endpoint is implemented. The implementation must satisfy the contract; the contract is the agreement between every party that touches the API — server, client, integrator, documentation, QA.

Once the spec is accepted, several artefacts derive from it automatically: the SDK client (one per language), a mock server (for front-end work that doesn't want to wait for the backend), the public reference HTML, and the contract tests (Schemathesis or similar).

Why we need it

When the implementation comes first and the spec is written later, the spec is documentation of what the code already does. It never catches design mistakes; it never gives the front-end a head start; and it always drifts because no one notices when the code changes and the spec doesn't.

Contract-first flips the order. The discussion happens on the spec PR, before any code is written. Once accepted, the spec gates every implementation: a build that fails contract tests fails the merge.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Stripe

API design at Stripe is contract-first: the spec is reviewed by an API review board, then implemented; SDKs, docs, and CLI are all generated from the spec.

What we take: An API review board sounds heavy, but it is the cheapest way to keep 200+ engineers shipping a consistent API surface.

Source: Stripe — Designing robust and predictable APIs

Slack

Adopted OpenAPI for the Web API in 2018; published the spec publicly so third-party developers could generate their own clients.

What we take: A public spec is also a contract with integrators — once you ship it, you own its evolution.

Source: Slack — OpenAPI spec for the Web API

GitHub

Maintains both a REST OpenAPI and a GraphQL schema as canonical sources; the docs site, Octokit SDKs, and the CLI all generate from them.

What we take: Two contracts (REST + GraphQL) are workable when both are generated from a single source-of-truth process, not maintained by hand.

Source: GitHub REST OpenAPI description

Microsoft Graph

Uses OpenAPI as the source for Graph Explorer, SDKs, and reference docs across 100+ teams; PR-blocking linting via Spectral.

What we take: Spectral rules let an API platform team enforce consistency without becoming a bottleneck on every change.

Source: Microsoft Graph — design guidelines

The shared playbook: write the spec first, generate the SDK + docs + tests from it, lint with Spectral, and treat any drift between code and spec as a build failure.

Reference HTML

Generated as the public API reference.

Contract tests

Generated; run in CI against the deployed service.

How it works in this project

The spec lives at services/api/openapi.yaml. CI runs Spectral against it (style rules in .spectral.yaml), generates the SDK with openapi-generator, and runs Schemathesis against the test deployment. The reference HTML at services/portal/public/reference/api/openapi-explorer.html embeds the rendered spec.

Two ADRs anchor the practice: ADR 0025 (external/internal API doc split) and ADR 0007 (governance and usability standard).

Deep dive — the theory you need

API versioning — three strategies and when each fits

Once you have a contract, you have to think about how it changes. Three strategies, with examples:

StrategyHowUsed byWhen it fits
URL versioning/v1/sessions, /v2/sessionsGitHub, TwilioPublic APIs with clear major versions and few breaking changes per year.
Header versioningStripe-Version: 2024-04-10StripeMany small breaking changes per year; clients pin to a date.
No versioning (evolve)Only additive changes; never break.GraphQL, Slack Web APIInternal APIs; teams accept the discipline of never breaking.

What counts as a breaking change

OpenAPI tools (openapi-diff) classify changes into three buckets:

  • Compatible — adding an optional field; adding an endpoint; adding an enum value to a response.
  • Breaking — removing a field; renaming; adding a required request field; making an optional field required; changing a type.
  • Behavioural — same shape, different semantics. Not caught by tools; only caught by tests and humans.

The CI rule: a breaking change is allowed only with an explicit version bump. Behavioural changes need an ADR.

Trade-offs — what this practice costs

  • Slower at first. Writing the spec before code feels slower for the engineer who has the implementation in their head. The win comes from the integrator and the front-end team who unblock 1–2 days earlier — counted across the org, contract-first is faster.
  • Spec choices lock you in. Once SDK clients are generated from the spec, changing a field name is a breaking change for every downstream. Naming review at spec time is expensive but cheaper than the rename.
  • Hot-fix friction. When prod is on fire, the rule 'spec first' slows the fix. A documented escape hatch ('emergency fix can skip spec; backfill within 24 h') is honest and prevents under-the-table workarounds.

Lifecycle

spec proposed → spec accepted → mock served → implementation → contract tests pass → SDK published

A new endpoint starts as a PR on the spec. Once accepted, the mock is available within minutes; the front-end can begin. The implementation follows; contract tests gate the merge. The SDK is published on release.

Common pitfalls

PitfallWhy it happensWhat to do instead
Spec drifts from codeEngineer changes the response shape; forgets to update the spec.Contract tests in CI; mismatch fails the build.
Spec has no lintField names, error envelopes, and security defaults vary per endpoint.Spectral ruleset; PR is blocked on lint.
Spec is hand-renderedAuthors edit the HTML reference directly.Spec-to-HTML rendering runs in CI on every commit (in this repo: make docs-fix via tools/docs/sync_docs.py); rendered HTML carries an AUTO-GENERATED banner so authors can't accidentally edit it.
Breaking change goes unnoticedA removed field ships without anyone realising.Run openapi-diff on every PR; fail the build on a breaking change without a major version bump.

Worked example

OpenAPI fragment

POST /api/v1/sessions

paths:
          /api/v1/sessions:
          post:
          operationId: createSession
          summary: Open a study session
          requestBody:
          required: true
          content:
          application/json:
          schema: { $ref: '#/components/schemas/SessionCreate' }
          responses:
          '201':
          description: Session created
          content:
          application/json:
          schema: { $ref: '#/components/schemas/Session' }
          '400':
          $ref: '#/components/responses/SessionAlreadyOpen'
          '422':
          $ref: '#/components/responses/InvalidDeck'

The PR to add this endpoint is reviewed by backend, front-end, and a tech writer. Once it lands, the mock serves a fake Session the next day; the front-end starts; backend implements; Schemathesis verifies on the test environment.

Outcome. The spec is reviewed and accepted; the mock server is live within the day; the front-end begins on the spec the same week; backend implements; Schemathesis verifies the contract against the test environment before merge; the SDK is published on the release tag.
