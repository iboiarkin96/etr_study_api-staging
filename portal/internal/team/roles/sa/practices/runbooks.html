Runbooks — one per alert

Every alert points at a runbook with symptom, diagnosis, actions, escalation, and dashboard links.

TL;DR

At 3 AM, the on-call should not have to think. A runbook gives them the symptom, the first diagnostic command, the most likely actions, and the escalation path. Every alert points at one runbook; every runbook follows the same template.

What it is

A runbook is a one-page guide for one specific alert. It says: when this alert fires, here is what is happening; here is how to confirm; here is what to do; here is who to call if you can't fix it.

The template is fixed — Symptom, Detect, Diagnose, Mitigate, Post-incident — so the on-call doesn't have to learn a new layout for every alert. The runbook is linked from the alert itself; one click from the page on the phone.

Why we need it

On-call without runbooks is a memory test under stress. The engineer remembers half of what they need, makes a guess on the rest, and the postmortem ends with 'next time we should have written it down'. Runbooks are the writing-it-down.

The other benefit is the rotation. With runbooks, on-call scales beyond the engineers who built the system. Without them, every new on-call has to spend a quarter shadowing before they can handle anything alone.

The repeated pattern: one alert, one runbook, standard template, link in the alert itself. The runbook is in version control; CI fails if an alert has no runbook URL.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Symptom

What the on-call sees. Alert name; pages affected; user-visible behaviour.

Detect

How to confirm the alert is real. One or two commands or dashboard links.

Diagnose

Probable causes in order of likelihood. Each with a quick check.

Mitigate

Actions to stop the bleeding. Reversible first.

Escalate

Who to page if the runbook ends with the problem still open.

Post-incident

Link to the postmortem template; what to capture before memory fades.

How it works in this project

Runbooks live at internal/sre/runbooks/. Each one is named NNNN-short-title.html. Every alert in alertmanager.yaml has a runbook_url annotation pointing at the matching page.

The template is in the UI Kit (ops-runbook). CI fails any alert in the config without a runbook_url; CI also fails any runbook missing one of the required sections.

Deep dive — the theory you need

The Detect-Diagnose-Mitigate-Post structure — why it works

The four-section structure is not arbitrary. It maps to the four cognitive states the on-call passes through:

StateQuestionSection
Confused (just paged)'Is this real?'Detect — one command to confirm
Investigating'What's happening?'Diagnose — checks in order of likelihood
Acting'How do I fix it?'Mitigate — reversible action first
Recovering'Is it over?'Post — exit criteria + postmortem trigger

The 'reversible first' rule

Mitigate actions are listed in order of reversibility: the most-reversible action first (restart a pod), the least-reversible last (drop a table). The reason: at 3 AM, you make mistakes; reversible mistakes are recoverable, irreversible ones are not.

What goes in the 'Detect' section

One command, copy-paste-able, that confirms or denies the alert in < 30 seconds. Not 'check the dashboard' — the dashboard link plus the exact panel. Not 'check the logs' — the kubectl command with the exact filter.

If Detect takes more than 30 seconds, the alert is too vague; rewrite the alert before the runbook.

Trade-offs — what this practice costs

  • Runbook drift. A service rename two months ago breaks half the runbook references. CODEOWNERS the runbook to the same person as the service config; the rename PR must touch both.
  • One alert, one runbook — strictly. Tempting to write a 'general 5xx' runbook for everything. Tempting and wrong; on-call needs a runbook that matches the alert's exact symptom, not the family.
  • Maintenance scales with alert count. 50 alerts = 50 runbooks. Aggressive alert reduction (the right move for any healthy on-call) shrinks the runbook backlog by the same factor.

Lifecycle

alert fires → on-call opens runbook → detects → diagnoses → mitigates → postmortem

The alert page carries the runbook link. The on-call clicks it from the phone. They confirm the symptom, walk the diagnose section, run the mitigate action. If the runbook ends with the issue still open, they escalate. After the incident, the postmortem updates the runbook.

Common pitfalls

PitfallWhy it happensWhat to do instead
Runbook is a storyLong prose; the on-call reads paragraphs at 3 AM.Use bullet points, commands in code fences, dashboard URLs that work without modification.
Generic runbook for everythingOne 'troubleshoot the API' page covers ten alerts.Every alert in alertmanager.yaml has a runbook_url annotation; CI fails the build on missing or shared URLs.
Runbook driftsService was renamed; runbook still references the old name.CODEOWNERS routes the runbook to the same owner as the service; the rename PR has to touch both.
No exit criteriaRunbook never says when the incident is resolved.Last bullet of every Mitigate section: 'Resolved when X is back to Y for 5 minutes'.

Worked example

Runbook fragment

SEV-2 — Sessions service 5xx burst

Symptom. 5xx rate on POST /api/v1/sessions > 5% for 5 minutes. Burn rate on sessions/availability SLO > 14x.

Detect.

# latest errors from the sessions pod
          kubectl logs deploy/sessions -n prod --since=5m | grep ERROR | head -20

Diagnose, in order of likelihood:

  1. DB connection pool exhausted. db_pool_in_use / db_pool_size > 0.95. Mitigate: bump pool size (deploy patch).
  2. Decks API timing out. http_request_duration_seconds{service="decks"} p99 > 2s. Mitigate: open the Decks runbook.
  3. Recent deploy. Check sessions deploys in the last 30 min. Mitigate: rollback.

Escalate to @backend-leads if not resolved within 20 min.

Resolved when 5xx rate < 1% for 5 minutes and SLO burn rate < 1x.

Outcome. On-call follows the runbook from the alert page; MTTR on this alert class drops from ~30 min to ~10 min; the post-incident step updates the runbook with the latest evidence (new diagnostic command, refreshed escalation path).
