Systems analysis practices at project

These 25 practices are the working set we want in place so the project stays easy to maintain over time, easy for new contributors to join, and easy to keep current as the system grows. They cover everything from Design Docs and ADRs to SLOs, runbooks, and blameless postmortems. The main radar gives the panorama of all 25 axes; below, you'll find a pentagon radar per group and a card for each practice with what it is, why we need it, and how we plan to roll it out.

Group A · 5 practices

Knowledge management

How we keep decisions, facts, and shared terms inside the project. When this group is weak, every new engineer asks the same questions for days, every old decision gets re-argued from scratch, and the team's knowledge lives only in chat messages. Strong knowledge management is the project's long-term memory — it turns the team's accumulated thinking into a resource the next contributor can read.

01

Design Docs — discuss before you code

2.0→8.0

What it is. Before any non-trivial change, the author writes a short 2–10 page Design Doc. The doc covers the context and the problem, two or three options with trade-offs, the path we chose, success metrics, risks, and open questions. ADRs record a decision once it is accepted; Design Docs propose it before that point.

Why it matters. When we discuss a design in writing first, the team catches the wrong path early — before we spend a week coding it. A reader who comes back a year later can see the alternatives we rejected and the reason we chose what we chose, instead of repeating the same conversation again.

How we will roll it out. Add a design-docs/ folder under services/portal/internal/. Add a Jinja template with the required sections. Make a Design Doc mandatory for changes that touch more than one service or that introduce a new dependency. Link every Design Doc to the resulting ADR.

02

ADR · RFC — numbered decisions

7.4→8.5

What it is. Every significant decision is a numbered file with three short sections: Context, Decision, and Consequences. ADRs hold accepted decisions; RFCs hold proposals still under discussion. The lifecycle is draft → ongoing → done / declined / superseded.

Why it matters. Once a decision is written down, it stops being verbal folklore. A new engineer reads ADR 0007 and understands why the project rejected library X — without asking in chat or rediscovering it through a bug. The numbering also gives every link a stable target.

How we will roll it out. Keep the current numbering. Use MADR 4.0 as the canonical template. Add a pre-commit check that every new ADR has a status, a date, and at least one consequence. Generate a navigable ADR index page in CI.

03

SSOT — one source of truth per entity

5.0→8.0

What it is. For every fact in the project (API contract, DB schema, service catalog, design tokens, glossary terms), there is exactly one authoritative file. Everything else is either generated from it by tooling or referenced through a link.

Why it matters. Two copies of the same fact drift apart over time. When that happens, half of our docs become wrong and we cannot tell which copy is correct. One owner per fact keeps the project honest and the docs trustworthy.

How we will roll it out. Write a short SSOT list on the principle page: what facts exist and which file owns each one. Add a CI check that flags duplicate definitions of the same constant. Replace any "copy this list into your doc" pattern with an include or a generated table.

04

Glossary — unified domain language

2.5→7.0

What it is. One dictionary of the project's terms. Each entry has the canonical definition, common synonyms, related terms, and a short example. The source is YAML; the HTML page and the inline tooltips in prose are generated from it.

Why it matters. When "contract", "endpoint", and "API" mean three different things to three engineers, every code review and every requirements doc ends in a debate. A shared dictionary removes that friction, makes onboarding faster, and disciplines requirements before they reach implementation.

How we will roll it out. Start a glossary.yaml with the top 30 terms we use most often. Render the HTML glossary in CI. Add a Vale rule that flags new prose using a term that is not yet in the glossary.

05

Tech Radar — technology portfolio

1.0→7.0

What it is. A simple ADOPT / TRIAL / ASSESS / HOLD chart of the technologies the project uses or considers. The source is tech-radar.json; the visual page is rendered by the Zalando or ThoughtWorks renderer. Reviewed once a quarter.

Why it matters. When somebody picks a new library on Monday, the rest of the team needs to know on Tuesday whether the project already evaluated it. The radar shows that at a glance and stops the project from quietly accumulating ten experimental libraries that nobody owns.

How we will roll it out. Add tech-radar.json with the libraries we already use. Mark obvious HOLDs (anything we have explicitly rejected). Reserve a 30-minute slot at the start of every quarter to review and update the radar.

Group B · 5 practices

Architecture & modeling

How we describe what we built, at the right level of detail. Diagrams, schemas, contracts, and domain boundaries are the artefacts an engineer needs to read before changing the code with confidence. When these are missing or out of date, every non-trivial change becomes a careful reading of the source — which is slow, error-prone, and the main reason refactors get postponed.

Group avg score: AS-IS 0.0→0.0 / 10
06

C4 + PlantUML — diagrams as code

5.6→8.0

What it is. Architecture is drawn at three C4 levels. Context describes what is around the system, Container describes what the system is made of, and Component describes what is inside a single service. The source is PlantUML or Structurizr DSL stored in the repo; CI renders the diagrams as part of the build.

Why it matters. A picture saves a new engineer a day of code reading — but only if the picture is current. Storing diagrams as code keeps them in the same PR as the change, so they stay in sync with reality instead of slowly turning into fiction.

How we will roll it out. Pick one service and draw all three C4 levels for it as a reference. Add a PlantUML rendering step to the docs portal CI. Add an ADR that fixes the C4 style and the file layout (one file per level, named conventionally).

07

ER · data domain model

5.2→8.0

What it is. The database is documented one table at a time. Each table has its purpose, owner, PII tags, retention rule, indexes, foreign keys, and enum values. The ER diagram is regenerated from the metadata after every migration.

Why it matters. Compliance and security questions — "is the email field PII?", "what is the retention for sessions?" — get a single, machine-checked answer instead of a long thread on Slack. The schema becomes self-describing, which matters every time the team adds a new analyst or auditor.

How we will roll it out. Adopt a metadata format (dbt-style column properties). Backfill the top 10 tables. Add a CI job that fails the build if a new table is added without metadata.

08

API contract-first — OpenAPI as the source

6.4→8.5

What it is. The OpenAPI 3.1 spec is written before the implementation. The server, SDKs, mock servers, and contract tests are generated from the spec. CI fails the build when the running code drifts away from the spec.

Why it matters. When the spec is the agreement, integrators and the team speak the same language from day one. Without a contract, every team writes its own client by reading the source, and every minor change turns into a coordination problem with five Slack threads.

How we will roll it out. Keep the existing OpenAPI file and lint it with Spectral. Add a contract test stage to CI that hits the running service. Generate the SDK and the public reference docs from the spec on every release.

09

AsyncAPI · event schemas

1.0→7.0

What it is. For asynchronous messages (Kafka, RabbitMQ, EventBridge, SNS), the project ships an AsyncAPI specification. Producers and consumers respect the contract; a schema registry checks compatibility before a new version goes live.

Why it matters. Event-driven services break each other quietly. There is no 4xx and no alert — just a stream of corrupted data that surfaces a week later. AsyncAPI makes events as contractual as REST and gives the schema registry a place to enforce that contract.

How we will roll it out. Start an AsyncAPI file for the most important event we publish. Add a schema check to the publisher's CI. Document the migration path for breaking changes and make a small example of a backwards-compatible event evolution.

10

DDD · bounded contexts

2.0→7.0

What it is. A context map of the project. Each service belongs to one bounded context; contexts are connected by typed relationships (Customer-Supplier, Conformist, Anti-corruption Layer, Shared Kernel).

Why it matters. When microservice boundaries align with domain model boundaries, "User" means one thing inside each service and the integration code stays small. Without a context map, the same concept silently grows three different definitions, and every cross-service change becomes risky.

How we will roll it out. Draw the first context map with the services we already have. Mark explicit translation layers between contexts. Use the map as a required input in every new Design Doc that adds or splits a service.

Group C · 5 practices

Information delivery

How a reader finds, understands, and updates what was written. Page structure, templates, style, lifecycle, and accessibility together make up the user experience of the documentation itself — for engineers, auditors, and AI assistants. If the docs are hard to read, people stop reading them, and a portal nobody reads is no better than no portal at all.

Group avg score: AS-IS 0.0→0.0 / 10
11

Diátaxis — four quadrants, no mixing

6.4→8.0

What it is. Every page belongs to one mode: tutorial (teach a beginner), how-to (solve a task), reference (precise data), or explanation (build understanding). The four modes have different goals, so the project avoids "everything-in-one" pages.

Why it matters. The reader can predict what is on a page before opening it. Tutorials teach but are not reference; reference is precise but does not teach. Mixing modes is the most common cause of pages people skim once and forget.

How we will roll it out. Tag every existing page with its mode. Move any mismatched section to the right page. Add one template per mode and a short tag at the top of the page so readers and authors can both see the mode at a glance.

12

Docs as code — PR · CODEOWNERS · CI

6.5→8.5

What it is. Documentation lives in the repo, is reviewed as part of a PR, is versioned with the code, and is validated by CI (link checker, style linter, accessibility check). A CODEOWNERS file routes every docs folder to a maintainer. There is no parallel Confluence acting as the source of truth.

Why it matters. When docs live next to the code, the same PR that changes the API also updates the doc. We cannot forget to update them because the build will fail. A parallel CMS guarantees drift; one repo guarantees one truth.

How we will roll it out. Keep the current setup. Add a stricter link checker and a public-vs-internal coverage check. Make sure every top-level docs folder has an entry in CODEOWNERS, including the audit folders.

principle ADR 0001 Write the Docs
13

Template registry — Design Doc · ADR · Postmortem · Runbook

5.4→8.0

What it is. One folder of templates for the major page types: Design Doc, ADR, RFC, Postmortem, Runbook, API spec, Screen spec, Audit, Principle. Each template is a Jinja file with required fields and a short author guide.

Why it matters. A registry fixes the grammar of the page. The 30+ ADRs we already have should look like one project, not 30 different ones. Templates also make the next page easier to start — the author copies a known skeleton instead of inventing one from scratch.

How we will roll it out. Collect every existing template under one folder. Add the missing ones (Postmortem, Runbook). Run a quick audit that every page on the portal is based on a registered template.

14

Style guide as code — Vale + alex.js

1.5→7.5

What it is. The project adopts a single style guide (Google Developer Documentation Style or Microsoft Writing Style) as the reference. Vale rules live under .vale/; alex.js checks for inclusive language; cspell catches typos. CI blocks a PR with style errors.

Why it matters. Without a linter, every author writes in their own voice. The reader sees a patchwork and slowly loses trust in the docs. A style linter keeps the tone of voice consistent without needing a copy-editor on every PR.

How we will roll it out. Pick the Google style as the canon. Add the Vale config to the repo. Start with a small rule set (sentence length, voice, common confusions) and grow it gradually — too many rules at once and authors will fight the linter instead of the prose.

15

Onboarding · accessibility · i18n

3.5→7.0

What it is. The repo root has a README, a CONTRIBUTING file, a .devcontainer/ for one-click local setup, a CODE_OF_CONDUCT, and good-first-issue labels. The portal passes an accessibility audit (WCAG 2.2 AA, axe-core, keyboard navigation, screen reader). The build also leaves room for a second UI language.

Why it matters. A new contributor should ship their first commit in one working day. People who rely on a keyboard or a screen reader should be able to read and contribute too. After the EAA 2025 deadline, accessibility is also a legal requirement in the EU, not a nice-to-have.

How we will roll it out. Refresh the README and CONTRIBUTING. Add a .devcontainer/. Run axe-core in CI on every doc page and fail the build on critical issues. Pick a primary second language and prepare the string extraction so future translation is straightforward.

Group D · 5 practices

Reliability & operations

How we know the service is healthy, and what we do when something breaks. Without this group, the project stays at prototype level: we cannot tell whether it is reliable enough, every incident becomes an improvisation, and the on-call engineer carries the load alone. This is the weakest group in the project today and the place with the highest leverage for the next quarter.

Group avg score: AS-IS 0.0→0.0 / 10
16

SLO · SLI · error budgets

0.5→7.0

What it is. Every user-facing service has a Service Level Objective (e.g. 99.9% availability), a small set of Service Level Indicators that measure the SLO, and an error budget. When the budget is spent, releases slow down until reliability recovers.

Why it matters. SLOs give the team a number to argue with. Without them, "is the service good enough" becomes a feeling. With them, product and ops share one frame, and the team can prioritise stability work when the budget is running low instead of debating it.

How we will roll it out. Pick one critical endpoint and define availability and latency SLOs for it. Add Prometheus rules for the matching SLIs. Add a dashboard panel that shows the error-budget burn rate; review it in the weekly engineering sync.

17

Runbooks — one per alert

1.5→7.5

What it is. Every alert has a runbook. The runbook follows a fixed template: symptom, diagnosis, actions, escalation, and links to dashboards. The runbook is tied to the alert ID and is updated after every incident.

Why it matters. The on-call engineer at 3 AM should not need to wake the architect. Clear runbooks cut MTTR, reduce on-call stress, and let a new engineer join the rotation without years of background knowledge.

How we will roll it out. Inventory the alerts that fire today. Write a runbook for the top five most frequent ones. Add a CI check that every alert in the alert config has a matching runbook file with the required sections filled in.

18

Blameless postmortems

1.0→7.5

What it is. Every P1 or P2 incident gets a blameless postmortem. The doc has a timeline, the customer impact, the root cause, contributing factors, action items with an owner and a due date, and a short "what we learned" section. Reviewed in a regular forum once a quarter.

Why it matters. A blameless review focuses on the system, not the person who pressed the button. The team learns from the failure and turns it into a lasting fix; without that habit, the same incident repeats every few months and the lesson is lost.

How we will roll it out. Add a postmortem template under sre/postmortems/. Run the first postmortem on the most recent SEV-2. Track action items in the same tool as regular tasks so they do not get forgotten between reviews.

19

Risk register — risk log

0.5→7.0

What it is. A risk log — technical, business, security, and compliance risks — with severity (probability × impact), a mitigation plan, an owner, a status, and a review cadence. Reviewed at the start of every quarter.

Why it matters. Risks are easier to manage when they are written down. A register gives the team and the manager one place to scan for what could go wrong, and forces the project to think about strategic threats — not only the ticket on top of the backlog.

How we will roll it out. Open the register with five risks the team already talks about informally. Use a simple table format that fits on one page. Tie each risk to one owner and a date for the next review.

20

Capacity planning · FinOps

0.5→6.5

What it is. A short capacity plan per service: expected QPS, storage growth, and a cost projection for the next 6–12 months. A FinOps note covers total cost of ownership, the monthly budget, and the known optimisations.

Why it matters. Without capacity planning, infrastructure cost or scale can surprise the team at the wrong moment. A back-of-envelope plan is enough to spot the curves going the wrong way long before they hit a wall.

How we will roll it out. Pick one service and write its plan in a Markdown table. Add a quarterly review of the table to the engineering calendar. Link the plan to the billing dashboard so the projection and the reality stay close.

21

SECURITY · threat modeling

1.0→7.5

What it is. At the repo root, a SECURITY.md (responsible disclosure policy) and a security.txt (RFC 9116). Critical services have a one-page STRIDE or LINDDUN threat model. The vulnerability disclosure SLA is written down.

Why it matters. A researcher who finds a hole should know exactly where to write — instead of posting on social media. A team that thinks about threats as a process catches issues early. These artefacts are also a building block for SOC 2 and ISO 27001 audits later.

How we will roll it out. Add the two top-level files. Write a one-page threat model for the auth flow. Document the disclosure timeline (acknowledge within 48 h, fix within 30 days for non-critical issues).

22

Service catalog — Backstage YAML

3.0→7.5

What it is. Every service ships a catalog-info.yaml in the Backstage format: kind, metadata, owner, type, lifecycle, dependsOn, providesApis, consumesApis. The HTML catalog pages are generated from the YAML.

Why it matters. A machine-readable catalog lets internal tools route alerts to the right owner, build a dependency graph automatically, and answer "who owns service X" in a single call. Without it, every internal tool reinvents its own HTML parser.

How we will roll it out. Add a YAML file to each service. Generate the catalog HTML in CI. Connect on-call routing to the YAML so the source of truth is the same as the catalog page (no manual sync to keep correct).

23

AI-augmented docs — llms.txt

1.8→7.5

What it is. The docs are written so they can be read by people and by AI tools (Claude, ChatGPT, Cursor). A top-level llms.txt marks the canonical pages. A Markdown source is provided next to the HTML because LLMs read MD an order of magnitude better than rendered pages. An "Ask the docs" widget gives RAG answers on the most-used pages.

Why it matters. AI assistants give precise help when they can find clean source. Without llms.txt and an MD copy, the assistant guesses from an HTML soup and confidently returns wrong answers — and wrong answers erode trust in the docs faster than missing answers.

How we will roll it out. Add llms.txt with the list of canonical pages. Generate an index.md alongside index.html for every section. Pilot one Ask-AI widget on a high-traffic page and measure click-through and accuracy.

24

Stakeholder map · personas · JTBD

1.0→7.0

What it is. A list of stakeholders (role, goal, influence, what information they need) and a small set of user personas with JTBD statements, frustrations, and expectations. A Power-Interest grid placed on one page. Refreshed every quarter.

Why it matters. A product designed for "everyone" is designed for no one. Personas keep the docs and the UI honest about who the actual reader is. The stakeholder map keeps communications targeted instead of broadcast to the whole company every time.

How we will roll it out. Draft three personas from the most active user types. Map five external stakeholders on a Power-Interest grid. Cite the personas explicitly in the next two Design Docs so they become a working tool, not a one-time slide.

25

RACI · release · review SLA

2.0→7.5

What it is. A RACI matrix for the key processes (release, security review, on-call rotation). A release process that uses semver, a generated changelog, a canary rollout, and a rollback playbook. A code review SLA — for example, "first reply within two working days" — and a clear escalation path when the SLA is missed.

Why it matters. Predictable processes scale with the team. Without RACI, decisions stall while everyone waits for someone else; without a release process, every release is improvisation; without a review SLA, PRs sit for weeks. None of this is exciting, all of it pays back over years.

How we will roll it out. Publish a one-page RACI for the release flow. Add CHANGELOG.md and switch the version scheme to semver. Set the review SLA in CONTRIBUTING and add a bot that pings reviewers when the SLA is at risk.
