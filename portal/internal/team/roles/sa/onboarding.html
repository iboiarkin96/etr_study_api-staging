SA onboarding — 60-day journey

Your first 60 days as a systems analyst on Study App, mapped to concrete reads and concrete deliverables. Each step has a time horizon, a short description, the canonical pages to read, and one outcome — the thing you will be able to do when you have finished the step.

Work the steps in order. The first three are compulsory; the rest are strongly encouraged. By the end you will have authored a feature spec (with ADR + contract) independently — that is the bar.

The journey · six milestones

  1. 01
    Day 1 · access

    Get into the project

    Clone the monorepo, install the toolchain, get the right accounts (GitLab, Confluence, docs CI), and confirm you can build the docs locally and reach the staging environment. Nothing exciting — but the next five steps depend on this one working.

    • Local development
    • Docs pipeline
    • System design overview

    Outcome. You can build the docs locally, open staging, and locate the canonical service catalog.

  2. 02
    Days 2–3 · foundations

    Read the why before the what

    Understand how we structure docs, where ADRs live, and which contract is single source of truth. Diataxis is the four-quadrant layout you will use for every page; docs-as-code is the workflow; the single authoring model tells you which file is canonical.

    • Diataxis
    • Docs-as-code
    • Single authoring model
    • Single source of truth

    Outcome. You can explain in five minutes which doc layer answers which question and where each artefact lives.

  3. 03
    Week 1 · the way around

    Learn how specs flow

    Walk the service catalog with a senior SA and read three recent ADRs from start to finish. Understand DDD bounded contexts on this project — they decide where new contracts land. Open three recent feature specs and trace each from problem framing to merged contract.

    • Decision records (ADRs)
    • DDD bounded contexts
    • Service catalog

    Outcome. Given any service, you can name its bounded context, its owning team, and the last three ADRs that shaped it.

  4. 04
    Week 2 · first ticket

    Write your first ADR

    Pick one small open decision from the radar (your buddy will point you at one). Write it up as an ADR: context, options, decision, consequences. Run it through the design-doc template, get one peer review, merge.

    • Design docs
    • C4 model + PlantUML
    • Style guide

    Outcome. You have authored and merged one ADR (with a C4 diagram where applicable) on a real decision.

  5. 05
    Weeks 3–4 · first deliverable

    Own one full contract update

    Take a real upcoming feature and steer the contract update end-to-end: write the OpenAPI/AsyncAPI delta, refresh the ER model, update the glossary, and link the spec to a fresh ADR. Drive the contract review with the dev team.

    • OpenAPI contract-first
    • AsyncAPI
    • ER model
    • Glossary discipline

    Outcome. You have shipped one contract update with a passing contract test, an updated ER model, and a glossary entry per new term.

  6. 06
    Day 60 · sign-off

    Own your first feature spec

    You are now the named SA on a shipping feature. Write the spec: problem framing, bounded context, contracts, ADRs, risks, NFRs. Drive the spec review with PM, dev, QA, and SRE. The buddy stays available but does not author. When the spec is signed off — that is the end of onboarding.

    • RACI · release
    • Risk register
    • Security threat modeling

    Outcome. You have authored and signed off one feature spec (with ADR + contract) independently. You are now a fully ramped SA on Study App.
