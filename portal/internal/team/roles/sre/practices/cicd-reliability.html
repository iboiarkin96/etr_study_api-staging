CI/CD Reliability — the pipeline is a production system

A set of standards and practices that treat the CI/CD pipeline as a reliability-critical system — with flaky test elimination, duration budgets, build infrastructure monitoring, and on-call coverage for pipeline failures.

TL;DR

CI/CD reliability means the pipeline is fast, deterministic, and treated as a production system when it breaks. Four standards: flaky test rate below 0.5% (tests that fail intermittently without code changes are quarantined and fixed within one sprint); pipeline duration budget (PR build ≤ 10 min, deploy pipeline ≤ 20 min — breaches are treated as regressions); build infrastructure monitoring (pipeline failure rate, queue depth, and duration tracked with the same dashboards as user-facing services); and pipeline on-call (a broken main branch is a P1 incident with a named responder). A slow or unreliable pipeline is a reliability risk — it delays rollbacks, produces deployment anxiety, and erodes the engineering team's trust in automation.

What it is

CI/CD reliability is the practice of treating the build and deploy pipeline as a production system — one that has SLOs, is monitored, has an on-call owner, and whose regressions are fixed with the same urgency as production bugs. Four core standards:

  • Flaky test elimination: A flaky test is one that fails intermittently without any code change. Flaky tests erode trust in the pipeline (engineers start ignoring red builds) and slow down the development cycle. Target: flaky test rate below 0.5% of test runs. Any test flagged as flaky is quarantined (skipped in CI, tracked in a quarantine backlog) and fixed within one sprint. A quarantined test is not a resolved test — it must be fixed or deleted.
  • Duration budgets: PR pipeline (lint + unit + integration + security scan + build) must complete in under 10 minutes. Deploy pipeline (from merge to production canary) must complete in under 20 minutes. Pipeline duration is tracked over time — a regression (e.g., a new test that adds 4 minutes to the build) is treated as a bug in the same sprint it appears.
  • Build infrastructure monitoring: Pipeline failure rate (failures not caused by code changes), queue depth (number of builds waiting for runners), and pipeline duration P90 are tracked on the same dashboards as user-facing services. Alerts fire when failure rate exceeds 5% or P90 duration exceeds budget.
  • Pipeline on-call: A broken main branch (build that fails in main after a merge) is a P1 incident. There is a named on-call owner for the pipeline. The break is acknowledged within 30 minutes and resolved or reverted within 2 hours. A main branch that stays broken is a deployment blocker for every team.

Why we need it

The CI/CD pipeline is on the critical path of every deployment. A slow or unreliable pipeline delays rollbacks — the most time-sensitive operation in incident response. If rolling back a broken deployment takes 20 minutes because the pipeline is slow, those 20 minutes are added to every incident's MTTR. A 10-minute pipeline budget for a rollback saves 10 minutes from every P0 and P1 incident.

Flaky tests are particularly destructive because their cost is invisible. Each flaky test failure causes an engineer to re-trigger the build — typically 5–10 minutes of waste per occurrence. At 20 flaky test failures per day across the team, this is 100–200 minutes of lost productivity per day. The DORA State of DevOps Report consistently finds that high-performing engineering teams have near-zero flaky test rates — flaky tests are a lagging indicator of test quality problems that compound over time.

How others do it

Google

Google's internal CI system (TAP — Test Automation Platform) runs billions of test cases per day. Their key practice: "test hermiticity" — every test must produce the same result in any environment, at any time, without network access or external dependencies. Any test that fails this criterion is automatically quarantined and reported to the test owner. Google also enforces a "build breakage is a P0" policy: if a change breaks the build, the author has 15 minutes to revert before the CI system does it automatically.

What we take: Automatic revert on build break. 15 minutes is a generous window — the point is that a broken build is never left overnight. The social norm that "it's OK to leave the build broken" is the root cause of most main branch reliability problems.

Source: Google Engineering — Test Reliability

Shopify

Shopify runs one of the largest Ruby monorepos in the world. Their CI challenge: a test suite that takes hours if run naively. Their solution: "test impact analysis" — each PR only runs tests that are affected by the changed files, not the entire suite. This reduced their median PR build time from 40 minutes to 8 minutes. They also publish a "test health score" per service — a weekly metric combining flaky test rate, build duration, and first-pass success rate — which is reviewed in each team's weekly engineering meeting.

What we take: Test impact analysis is the highest-leverage optimisation for large codebases. Running 100% of tests on every PR is wasteful once the suite exceeds a few minutes. Impact analysis is not a shortcut — it's the correct engineering decision.

Source: Shopify Engineering Blog

Anatomy

Flaky test quarantine process

A test is flagged as flaky if it fails in CI without a corresponding code change. The CI system automatically opens a ticket, adds the @quarantine annotation to the test, and skips it in subsequent runs. The test owner has one sprint to fix or delete the test. Tests in quarantine for more than 2 sprints are deleted — a deleted test is better than a permanently skipped test.

Pipeline duration tracking

Every pipeline run records per-stage duration to a time-series store. Weekly report: P50 and P90 duration per stage, trend vs. last 4 weeks. Any stage that regresses more than 20% in a week is a bug assigned to the team that merged the change causing the regression.

Build infrastructure SLO

Pipeline availability SLO: 99.5% of builds must start within 2 minutes of trigger (queue depth SLO). Pipeline success SLO: infrastructure-caused failures (runner crashes, network timeouts unrelated to code) below 1% of builds. Both tracked monthly and reviewed in the SRE team's monthly SLO review.

Main branch protection

Main branch requires passing CI before merge (branch protection rule). Post-merge build result is monitored — if main breaks after merge, the CI system triggers an automatic alert to the pipeline on-call and the author of the last merge. Auto-revert policy: if the break is not acknowledged within 30 minutes, the last merge is reverted automatically.

How it works in this project

Study App CI runs on GitHub Actions. Every PR triggers a workflow whose required checks are: unit tests, integration tests (against ephemeral Postgres + Redis services in the runner), lint (ruff + eslint + stylelint), security scan (Trivy for the image, Bandit/Semgrep for source), and image build (multi-stage Dockerfile, push to GHCR with cosign signature). The set is encoded as required status checks in branch protection on main — a PR cannot merge without all five green.

Deploys are GitOps via Argo CD. The CI job that produces the signed image updates the image tag in infra/argocd/; Argo CD detects the change, runs its sync policy (canary first, then full rollout if health checks pass), and reports back. The DORA dashboard at internal/sre/dora.html tracks deployment frequency, lead time for changes, change failure rate, and time to restore — refreshed weekly from GitHub Actions and Argo CD events. Flaky tests are quarantined via the @pytest.mark.quarantine marker (and .skipFlaky in the JS suites); CI excludes them from the required-check set but the quarantine backlog is reviewed at every sprint planning, and a test cannot sit in quarantine past two sprints without an owner decision to fix or delete.

Deep dive — the theory you need

DORA metrics — measuring delivery as a system

The DORA research programme, summarised in Forsgren et al.'s Accelerate, established four metrics that together describe an engineering organisation's delivery performance. They are outcome metrics, not activity metrics, and they correlate with both organisational performance and engineer well-being.

Deployment frequency

How often the team deploys to production. Elite teams deploy multiple times per day; high performers between once per day and once per week; medium performers between once per week and once per month; low performers less than once per month. The interesting fact is not the number itself but what it implies — high frequency is only possible if the change size is small, which is only possible if the pipeline is fast and the rollback is cheap. Frequency is therefore a proxy for batch size and pipeline confidence.

Lead time for changes

Time from code committed to code running in production. Elite: under one hour. High: one day to one week. Medium: one week to one month. Low: more than one month. Lead time is the most direct measure of pipeline throughput; it includes review, build, test, and deploy stages, so a regression in any one of them shows up here first.

Mean time to restore (MTTR)

Time from incident detected to incident resolved. Elite: under one hour. High: under one day. Medium: one day to one week. Low: more than one week. MTTR is a function of two things — how fast the team can identify the bad change, and how fast the pipeline can ship a fix or roll back. A slow pipeline degrades MTTR linearly, which is why CI/CD reliability is a reliability practice and not a developer-productivity practice.

Change failure rate

Percentage of deployments that cause a production incident (rollback, hotfix, or degraded service). Elite and high: 0–15%. Medium: 16–30%. Low: 46–60%. The single most consistent finding from the DORA dataset: high-performing teams do not have a lower change failure rate by being more careful — they have it by deploying smaller changes more often, which makes each change easier to verify and cheaper to roll back.

Pipeline reliability as a first-class incident class

A broken pipeline is an incident, not a chore. When main is red, every team is blocked from shipping; every minute the pipeline stays broken multiplies across the organisation. The principled response is to give the pipeline an on-call rotation, a P1 severity for main breakage, and the same post-mortem treatment as a customer-facing outage. The reverse — treating pipeline breakage as "someone will get to it" — is how a team ends up with a multi-day red main branch and a deploy log that says nothing has shipped this week.

Test-pyramid economics

Mike Cohn's test pyramid (unit > integration > end-to-end) is an economic argument, not an aesthetic one. A unit test runs in milliseconds and isolates one component; an integration test runs in seconds and exercises a contract; an end-to-end test runs in minutes and exercises the full system. The pyramid says the count of tests should follow that cost curve — many unit tests, fewer integration tests, very few end-to-end tests. Inverting the pyramid (lots of slow e2e tests, few unit tests) produces a long pipeline that nobody trusts and that everyone re-triggers when it fails. The flake rate of e2e tests is structurally higher because the surface area is larger; the test pyramid is the structural answer to flake.

Supply-chain integrity — SLSA and provenance

The build pipeline is itself a target. SolarWinds (2020) and Codecov (2021) showed that an attacker who compromises the build system can ship malware to every downstream consumer with the legitimate signing key of the upstream project. The SLSA framework (Supply-chain Levels for Software Artifacts) defines four levels of build integrity: SLSA 1 (build script documented), SLSA 2 (hosted build service with provenance), SLSA 3 (hardened build platform, non-falsifiable provenance), SLSA 4 (hermetic, reproducible builds, two-party review). The provenance attestation records what was built, from which source, by which builder, with which dependencies — and is signed so a verifier downstream can prove the artifact is legitimate.

Trade-offs — what this practice costs

  • Up-front pipeline engineering. Reaching a 10-minute PR pipeline requires test parallelisation, build caching, and selective test execution — none of which are free. Teams that skip this work end up with 40-minute pipelines and the cultural problems that follow.
  • Discipline at PR time. Required checks, signed artifacts, and SLSA provenance add friction to "just push a hotfix" workflows. The friction is the point; without it the supply-chain protections do not exist when they are needed.
  • Cultural reversal on flake. Engineers learn to re-trigger CI rather than investigate flakes — it is locally rational and globally destructive. The forcing function is the quarantine + delete-after-two-sprints rule: flake becomes visible work that must be closed.

Lifecycle

PR opened → checks run → review → merge → build → deploy gate → deployed

A PR is opened against main. GitHub Actions runs the required checks in parallel — lint, unit, integration, security scan, image build — and reports each as a status. Human review proceeds in parallel with CI; once both pass, the PR is merged. The post-merge build runs once more on the merged commit, produces a signed image with SLSA provenance, and updates the image tag in the Argo CD repo. Argo CD picks up the change, applies it to the canary slice first, watches health and SLO probes for the soak window, and either promotes to the full rollout or rolls back automatically. The DORA timer for that change starts at "code committed" and stops at "deployed and healthy" — the difference is lead time, recorded for the weekly review.

Common pitfalls

PitfallWhy it happensWhat to do instead
Flaky tests tolerated because "they usually pass" The cost of flakiness is invisible — no single failure is expensive enough to fix immediately. Track aggregate flakiness cost: flaky failures per week × average re-trigger time = wasted engineer hours per week. When the number is visible, the fix priority becomes clear. A single test that fails once per day wastes 30+ engineer-minutes per week.
Build duration grows without notice Each new test adds a few seconds — no single addition is a regression. Track duration as a time series and alert on trend, not just threshold. A trend of +5 seconds per week over 12 weeks is a 60-second regression — invisible week-to-week, significant over a quarter.
Pipeline breaks treated as "someone else's problem" No named pipeline on-call; everyone assumes someone else will fix it. Assign pipeline on-call as a rotating weekly responsibility. The on-call engineer's job during a main branch break is identical to incident response: acknowledge, assess, fix or revert. The broken main is a P1 until it's green.
Unsigned artifacts shipped to production Signing adds CI complexity; provenance feels like compliance theatre until the first supply-chain incident. Sign every image with cosign at build time and verify the signature at the Argo CD admission step. The signature is part of the artifact; an unsigned image is not deployable. SLSA provenance is the minimum bar, not an optional extra.
End-to-end tests dominate the suite E2E tests feel "more realistic"; they catch bugs unit tests miss. Invert the instinct: build the unit-test base wide and the e2e tip narrow. E2E tests are expensive and flaky by surface area; use a handful as smoke tests, not a hundred as regression coverage. Push real coverage down to integration and unit layers where the economics work.

Worked example

GitHub Actions workflow

Study App PR pipeline — matrix tests, security scan, signed image, deploy gate

name: ci
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

permissions:
  contents:   read
  id-token:   write   # OIDC for cosign keyless signing
  packages:   write   # push to GHCR

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        suite: [unit, integration]
        python: ["3.12"]
    services:
      postgres:
        image: postgres:16
        env: { POSTGRES_PASSWORD: ci }
        ports: ["5432:5432"]
        options: --health-cmd pg_isready --health-interval 5s
      redis:
        image: redis:7
        ports: ["6379:6379"]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: ${{ matrix.python }}, cache: pip }
      - run: pip install -r requirements-dev.txt
      - run: pytest -m "${{ matrix.suite }} and not quarantine" --junitxml=junit.xml
      - if: always()
        uses: actions/upload-artifact@v4
        with: { name: junit-${{ matrix.suite }}, path: junit.xml }

  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Trivy filesystem scan
        uses: aquasecurity/trivy-action@v0
        with: { scan-type: fs, severity: HIGH,CRITICAL, exit-code: '1' }
      - name: Semgrep SAST
        run: semgrep ci --config auto --error

  build-and-sign:
    needs: [test, security-scan]
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: sigstore/cosign-installer@v3
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - id: build
        uses: docker/build-push-action@v6
        with:
          push:      true
          tags:      ghcr.io/${{ github.repository }}:${{ github.sha }}
          provenance: true       # SLSA build provenance
          sbom:       true       # SPDX SBOM attached
      - name: Sign image (keyless, OIDC)
        run: cosign sign --yes ghcr.io/${{ github.repository }}@${{ steps.build.outputs.digest }}

  deploy-gate:
    needs: build-and-sign
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { repository: study-app/infra, token: ${{ secrets.INFRA_PAT }} }
      - name: Verify cosign signature
        run: cosign verify --certificate-identity-regexp '.*study-app.*' \
             ghcr.io/study-app/api@${{ needs.build-and-sign.outputs.digest }}
      - name: Bump image tag for Argo CD
        run: |
          yq -i '.spec.template.spec.containers[0].image =
                 "ghcr.io/study-app/api@${{ needs.build-and-sign.outputs.digest }}"' \
                 argocd/apps/api/deployment.yaml
          git commit -am "deploy(api): ${{ github.sha }}" && git push

What this workflow does. The PR run does test + security-scan in parallel — both required for merge. The post-merge run additionally builds the image, attaches SLSA provenance and an SBOM, signs the digest with cosign (keyless, via GitHub's OIDC token), and updates the Argo CD repo to point production at the new digest. Argo CD takes it from there: canary slice, soak, promote or roll back. The deploy gate verifies the signature before bumping the tag — an unsigned or untrusted image cannot reach production even if the build job was somehow tricked into pushing one. The matrix split keeps unit and integration parallel so the PR pipeline stays inside the 10-minute budget.
