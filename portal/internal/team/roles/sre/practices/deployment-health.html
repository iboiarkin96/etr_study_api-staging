Deployment Health Gates — confirm the deployment is healthy before calling it done

Automated checks that run immediately after every deployment and block promotion or trigger rollback if any signal degrades — converting the post-deploy observation window from a manual watch into a structured, reproducible gate.

TL;DR

A deployment health gate is an automated check that runs for a defined observation window after every production deployment and compares post-deploy metrics against pre-deploy baseline. Three gate signals: error rate (must stay within 1.5× baseline for 10 minutes post-deploy), P99 latency (within 1.5× baseline), and success rate of the service's critical-path synthetic probe (must remain above 99%). If any signal degrades, the deployment pipeline pauses and the on-call is paged. If degradation persists past a 5-minute grace period, rollback is triggered automatically. The gate eliminates the "we deployed, it looks fine, we went home, something broke at 2 am" failure mode.

What it is

Deployment health gates are automated checks that run during a defined observation window after every production deployment and compare post-deploy metrics against a pre-deploy baseline. A gate has three components:

  • Gate signals: The specific metrics checked — typically error rate, P99 latency, and the success rate of the service's critical-path synthetic probe. Each signal has a pass threshold (e.g., error rate within 1.5× baseline) evaluated across the observation window.
  • Observation window: The time period after deployment during which signals are observed. 10 minutes for canary deployments; 30 minutes for ring 1 promotions. The window must be long enough to accumulate statistically significant traffic, but short enough to limit blast radius if something is wrong.
  • Failure response: If a gate signal degrades, the deployment pipeline pauses and the on-call is paged. A 5-minute grace period allows for transient noise. If degradation persists past the grace period, rollback is triggered automatically without human intervention.

Gates are defined per service as code — a YAML configuration specifying the signals, thresholds, and observation window. The gate runs as a pipeline stage, not a human check.

Why we need it

Most post-deploy outages happen in the hour after a deployment, not during it. The deployment itself goes cleanly, the first manual check looks fine, the team moves on — and then something fails at low traffic that wasn't visible in the observation minute the deployer spent looking at dashboards before going to lunch. Health gates close this gap by extending the observation window programmatically and defining explicit pass/fail criteria.

Health gates also provide the evidence for rollback decisions. Without a gate, the rollback decision requires an engineer to assess degradation in real time, under pressure, with partial information. With a gate, the rollback decision is made by the same criteria that would pass a production review — documented, reproducible, and fast.

How others do it

Netflix

Netflix's deployment system (Spinnaker) includes a "canary analysis" stage that compares the canary version's metrics against the baseline using a statistical comparison — not a simple threshold, but a Mann-Whitney U test across the distribution of metric values during the observation window. This detects regressions in the tail of the distribution that a simple average or P99 comparison would miss. The canary analysis score (0–100) is the gate criterion, not the raw metric values.

What we take: Statistical comparison is more rigorous than threshold comparison for detecting distribution shifts. A simple "error rate within 1.5× baseline" misses cases where the mean is fine but the tail has a new spike. Statistical gates are the correct approach for high-volume services where the distribution matters more than the average.

Source: Netflix Tech Blog — Kayenta

Weaveworks / Flagger

Flagger (open source, developed by Weaveworks) implements progressive delivery health gates as a Kubernetes controller. The controller watches a deployment, routes traffic through canary/stable splits, queries Prometheus or Datadog for gate signals, and advances or rolls back automatically based on configurable thresholds. The gate configuration lives in a CRD (Custom Resource Definition) in the same repository as the service — gates are reviewed in the same PR as the deployment change.

What we take: Gate configuration belongs in the service repository, not in a separate deployment system. When gates are code-reviewed alongside the deployment change, engineers are more likely to think about failure modes before deploying than after.

Source: Flagger — Progressive Delivery Operator

Anatomy

Gate signal selection

Three mandatory signals: error rate (4xx/5xx / total requests), P99 latency, and synthetic probe success rate. Optional: SLO burn rate for the window. Avoid business metrics (conversion rate, revenue) as gate signals — they're too noisy for a 10-minute window and confound A/B effects with reliability regressions.

Baseline calculation

Baseline = same signals averaged over the 24 hours before the deployment, excluding the previous deployment window. Day-of-week and hour-of-day adjustment: if deploying on Monday morning, the baseline is the previous Monday morning, not the Friday evening. Static thresholds (not compared to baseline) produce too many false positives on seasonal traffic patterns.

Grace period design

A 5-minute grace period after gate degradation absorbs transient metric spikes (alerting noise, burst traffic). During the grace period, the pipeline pauses and the on-call is notified but rollback has not triggered. The engineer can manually abort the grace period and trigger immediate rollback if the degradation is clearly severe. Never extend the grace period beyond 10 minutes — it defeats the purpose of the gate.

Gate-as-code requirement

Every Tier 0 and Tier 1 service must have a health gate configuration file (deployment-gate.yaml) in the repository root. CI validates that the file exists and parses correctly before the PR can merge. Services without a gate file cannot deploy to production.

How it works in this project

Study App runs every production deployment through a layered health gate built on Kubernetes readiness and liveness probes, post-deploy smoke tests driven by synthetic checks, and a 10-minute metrics window analysed by Argo Rollouts. Readiness probes hold traffic off a new pod until it returns 200 on /healthz/ready; liveness probes restart pods that wedge after they've been serving. Once pods are ready, a synthetic-check job hits the service's critical-path endpoints (login, fetch lesson, submit answer) and only then opens the traffic split.

During the 10-minute observation window, Argo Rollouts runs an AnalysisTemplate that queries Prometheus for error rate and P99 latency against the pre-deploy baseline. If either signal exceeds 1.5× baseline for more than the 5-minute grace period, the rollout is automatically rolled back — traffic shifted to the previous stable ReplicaSet, the new pods drained, and the on-call paged with the analysis report attached. The gate runs as a pipeline stage in .github/workflows/deploy.yaml, with the gate config living next to each service at services/<name>/deployment-gate.yaml.

Deep dive — the theory you need

Probe design — what each probe is actually for

Kubernetes exposes three probes — readiness, liveness, startup — and they answer three different questions. The most common production bug is treating them as interchangeable; they are not.

Readiness ≠ liveness

The readiness probe answers "should this pod receive traffic right now?" — a No removes the pod from the Service endpoints but does not restart it. The liveness probe answers "is this pod wedged?" — a No restarts the container. Use readiness for transient back-pressure (database connection draining, warm-up incomplete) and liveness only for genuine deadlocks. A common failure mode: pointing liveness at the same /health endpoint that depends on the database — when the database hiccups, every pod restarts simultaneously, turning a degradation into an outage.

The startup-probe gotcha

Slow-starting applications (JVMs, anything loading a model) need the startup probe — it suspends readiness and liveness until startup completes, giving a long initial budget without making the steady-state liveness check generous. Skip the startup probe and you have a choice between liveness restarting the pod mid-startup or making every steady-state liveness check tolerate 60 seconds, which means a wedged pod takes 60 seconds to be restarted. The startup probe is not optional for slow boots.

What "healthy" means at the deploy boundary

Immediately after a deploy, the service is in an unusual state: caches are cold, connection pools are empty, traffic is ramping. "Healthy" cannot mean "indistinguishable from steady state" — it has to mean "no new errors, and the ramp is converging on baseline." That is why the gate compares against a pre-deploy baseline, not a static threshold: a cold-cache P99 of 200ms is fine if the steady-state P99 is 150ms (1.33× baseline, within 1.5×), but a static "P99 must be under 180ms" gate would fail every deploy.

Comparison-window statistics

The two ways to compare post-deploy against baseline are threshold ("post-deploy mean ≤ 1.5 × baseline mean") and distributional (Mann-Whitney U or Kolmogorov-Smirnov on the full sample). Thresholds are cheap and the right choice for low-volume services where the sample is too small for distributional tests to have power. Distributional comparison is correct for high-volume services where the mean stays flat but the tail develops a new lobe (the classic "1% of users now see 5s latency" regression that never moves the average). The Study App default is threshold for now; the migration to Kayenta-style distributional analysis is on the roadmap once traffic justifies the complexity.

Blue-green pre-warm latency

A naive blue-green cutover moves 100% of traffic at once and the new fleet's caches are stone cold — P99 spikes for the first minute, the gate fails, rollback fires, the deploy never lands. Two mitigations: pre-warm the green fleet by replaying a synthetic traffic profile against it before the cut, and extend the gate's grace period for blue-green specifically (15 min instead of 5) to let caches fill. Canary deployments dodge this because traffic ramps gradually; pure blue-green needs the explicit pre-warm.

Trade-offs — what this practice costs

  • Deploy latency. A 10-minute observation window means every deploy carries a 10-minute tail before it is "done". Teams used to instant deploys feel this — the answer is to remember that the alternative is a 2 a.m. page, not a faster deploy.
  • Baseline maintenance. A baseline derived from "the previous 24 hours" needs hour-of-day and day-of-week adjustment, otherwise Monday-morning deploys fail against Sunday-evening baselines. The adjustment logic is non-trivial and has to be tested like production code.
  • Override discipline. The first time a gate blocks an urgent fix, somebody will ask for a manual override. Granting it routinely turns the gate into theatre; refusing it costs time. The right rule: overrides require a written reason and trigger an automatic post-deploy retro entry, so the cost of override surfaces visibly.

Lifecycle

artifact released → pre-deploy checks → deploy to canary → readiness ok → smoke green → metrics window passed → fully promoted

A deploy starts as a tagged artifact in the registry. Pre-deploy checks (image signature, vulnerability scan, gate-file presence) gate the pipeline before any pod moves. The canary ReplicaSet comes up next; Kubernetes holds it out of traffic until the readiness probe goes green. Synthetic smoke tests hit the critical-path endpoints — if any fail, the rollout halts before user traffic is shifted. Once smoke is green, traffic ramps and the 10-minute metrics window opens; Argo Rollouts polls the analysis template on each step. Window passed → step weight increases until the canary is fully promoted; window failed past grace → automatic rollback to the previous stable ReplicaSet, on-call paged with the analysis output. The "fully promoted" state is the only one that retires the previous ReplicaSet — every earlier state keeps it warm for an instant cut-back.

Common pitfalls

PitfallWhy it happensWhat to do instead
Gates that never trigger because thresholds are too loose Engineers set 10× baseline thresholds to avoid false positives after being paged once for a transient spike. Track gate trigger rate. A gate that has never triggered in 6 months either means the service is flawless or the thresholds are wrong. Both are worth investigating. Review gate thresholds quarterly against actual post-deploy degradation events.
Gate signals chosen that aren't sensitive to the deployment "We track CPU utilisation because it's easy to instrument." Gate signals must be causally linked to the deployment. CPU is a resource metric — it changes with load, not just with code changes. Error rate and latency are output metrics that directly reflect user-visible quality. Choose user-visible output signals.
Manual override used routinely to bypass gate failures "We know this is a known metric spike — just override the gate." If a gate failure is routinely bypassed, the gate is wrong. Fix the gate (adjust the threshold or exclude the known spike from the signal) rather than building a culture of overrides. Every manual override is a data point that the gate needs tuning.

Worked example

Deployment spec

lessons-api — Kubernetes Deployment + Argo Rollouts analysis template

# services/lessons-api/k8s/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: lessons-api
spec:
  replicas: 6
  template:
    spec:
      containers:
        - name: lessons-api
          image: registry.studyapp/lessons-api:1.42.0
          ports:
            - containerPort: 8080
          startupProbe:
            httpGet: { path: /healthz/startup, port: 8080 }
            periodSeconds: 5
            failureThreshold: 24          # 2 min budget for cold start
          readinessProbe:
            httpGet: { path: /healthz/ready, port: 8080 }
            periodSeconds: 5
            failureThreshold: 2
          livenessProbe:
            httpGet: { path: /healthz/live, port: 8080 }
            periodSeconds: 10
            failureThreshold: 3
          lifecycle:
            postStart:
              exec:
                command: ["/bin/sh", "-c", "/opt/smoke-test.sh"]
---
# services/lessons-api/rollout/analysis-template.yaml
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: lessons-api-gate
spec:
  args:
    - name: service
      value: lessons-api
  metrics:
    - name: error-rate
      interval: 60s
      count: 10                          # 10-minute window
      successCondition: result[0] <= 1.5 * result[1]
      failureLimit: 5                    # 5-minute grace
      provider:
        prometheus:
          address: http://prometheus.monitoring:9090
          query: |
            (
              sum(rate(http_requests_total{service="{{args.service}},status=~"5.."}[1m]))
              / sum(rate(http_requests_total{service="{{args.service}}}[1m]))
            )
    - name: p99-latency
      interval: 60s
      count: 10
      successCondition: result[0] <= 1.5 * result[1]
      failureLimit: 5
      provider:
        prometheus:
          address: http://prometheus.monitoring:9090
          query: |
            histogram_quantile(0.99,
              sum by (le) (rate(http_request_duration_seconds_bucket{service="{{args.service}}"}[1m]))
            )

What this spec does. Three probes give Kubernetes the right answer to three different questions: startupProbe grants a 2-minute cold-start budget, readinessProbe holds traffic off a pod that is not ready right now, and livenessProbe restarts a wedged container. The postStart hook runs the per-pod smoke test before the pod claims readiness. Argo Rollouts queries the analysis template every 60 seconds for 10 polls — error rate and P99 latency both compared to the pre-deploy baseline at 1.5×. A failureLimit: 5 on each metric is the 5-minute grace; once exceeded, the rollout is automatically rolled back and the on-call paged with the analysis run attached. The gate, the deploy, and the smoke test all ship in the same PR — code review covers the failure modes before they reach production.

Quality bar — what good looks like

A checklist of what a healthy implementation has. ✓ done ◐ partial ○ todo

  • ○

    Every Tier 0/1 service has a deployment-gate.yaml

    CI rejects PRs from services missing the gate config; the file is reviewed in the same PR as the deployment change.

  • ○

    Three probes configured correctly

    Startup, readiness, and liveness all present; liveness does not depend on downstream services; startup probe present on any service that takes >10 s to boot.

  • ○

    Synthetic smoke tests run post-deploy

    Critical-path endpoints hit before traffic is shifted; failures halt the rollout before user impact.

  • ○

    10-minute observation window with baseline comparison

    Error rate and P99 latency compared to pre-deploy baseline at 1.5×, evaluated continuously across the window.

  • ○

    5-minute grace period before auto-rollback

    Transient spikes absorbed; sustained degradation triggers automatic rollback without human intervention.

  • ○

    Gate trigger rate measured and reviewed

    A gate that has never triggered in 6 months is either too loose or guarding a flawless service — investigate which.

  • ○

    Overrides require written reason + retro entry

    Manual override is allowed but never free; every override creates a post-deploy retro item so tuning cost stays visible.
