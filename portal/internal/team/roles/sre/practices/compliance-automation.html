Compliance as Code — policies enforced by machines, not spreadsheets

Security and compliance policies expressed as machine-readable rules that run automatically in CI and at deploy time — replacing point-in-time audit checklists with continuous, automated verification that drifts are detected immediately rather than annually.

TL;DR

Compliance as code means expressing security and compliance requirements as automated policies that run in CI (preventing non-compliant changes from merging) and as continuous checks in production (detecting drift from the compliant baseline). Three tools: OPA/Gatekeeper for Kubernetes admission policies (reject pods without resource limits, reject images from unauthorised registries, reject privileged containers); Terraform Sentinel or Checkov for infrastructure-as-code policy (no public S3 buckets, encryption at rest required, no security groups open to 0.0.0.0/0); and cloud security posture management (CSPM) for continuous production compliance checking (AWS Config, GCP Security Command Center). The shift from annual audit compliance to continuous compliance means that non-compliant configurations are detected and remediated in minutes or hours, not discovered in a quarterly audit 90 days after the non-compliant change was made.

What it is

Compliance as code is the practice of expressing security and regulatory compliance requirements as machine-readable policy rules that enforce themselves automatically, rather than as checklists verified by humans during periodic audits. Three enforcement layers:

  • Infrastructure-as-code policy (IaC scanning): Tools like Checkov, tfsec, or Terraform Sentinel scan Terraform, CloudFormation, and Kubernetes YAML for policy violations before the code merges. Common policies: S3 buckets must not be public, security groups must not allow ingress from 0.0.0.0/0 on port 22, encryption at rest must be enabled for all storage resources, all cloud resources must have specific tags. The CI gate blocks the PR if any policy fails.
  • Kubernetes admission control: OPA Gatekeeper or Kyverno enforce policies at pod admission time — before a workload can run. Common policies: all containers must specify resource limits (CPU and memory), all containers must run as non-root, privileged containers are forbidden, images must come from the internal registry only. The admission controller is the last enforcement point before a workload runs — it blocks non-compliant workloads even if they passed the CI gate.
  • Cloud Security Posture Management (CSPM): Continuous scanning of production cloud environments against a compliance baseline (CIS Benchmarks, SOC 2 controls, PCI-DSS requirements). AWS Config, GCP Security Command Center, or a third-party CSPM tool detects configuration drift in real time and creates tickets for any non-compliant finding. The scan runs every 15 minutes — drift is detected and ticketed before the audit cycle, not during it.

Why we need it

Annual compliance audits find violations that have existed for months. A misconfigured S3 bucket discovered in a quarterly audit may have been open for 90 days. Continuous automated compliance checking finds the same misconfiguration within 15 minutes. The risk window is reduced by 99.8% without any additional engineer time — the policy runs automatically.

Compliance as code also makes compliance a development practice, not a compliance team burden. When policies are enforced in CI, developers receive immediate, specific feedback ("this S3 bucket is public — add acl = 'private'") rather than a spreadsheet finding six weeks after the code was merged. Feedback at the PR stage is faster, cheaper, and more actionable than feedback in an audit report.

How others do it

Netflix

Netflix's "Security Monkey" (now "Repokid" and Cloud Custodian internally) was an early implementation of continuous cloud compliance checking — a system that continuously evaluated AWS resource configurations against a policy baseline and automatically remediated simple findings (adding missing tags, removing overly permissive IAM policies). Netflix's key insight: compliance policies that require human action to remediate are always delayed; policies that trigger automatic remediation for low-risk findings converge to compliance faster. Human review is reserved for high-risk findings that require context.

What we take: Distinguish between findings that warrant automatic remediation (add a missing tag, remove a public ACL that should be private) and findings that warrant human review (an IAM role with admin access). Automatic remediation for safe findings removes the queue pressure that causes human-reviewed findings to go stale.

Source: Netflix Tech Blog

Bridgecrew / Prisma Cloud

Bridgecrew (acquired by Prisma Cloud) pioneered "shift-left compliance" — running IaC policy checks in the IDE and pre-commit hooks, not just in CI. Their insight: the earlier in the development cycle a compliance violation is caught, the cheaper it is to fix. An IDE warning at authoring time costs 30 seconds to fix; a CI failure costs 5 minutes; a production finding costs hours. The tool's VS Code extension provides inline policy annotations as the engineer writes Terraform, before any commit. The same policy ruleset runs at all three enforcement points (IDE, CI, production), so there are no surprises when the policy runs in CI.

What we take: Provide IDE-level compliance feedback, not just CI-level. Engineers who get policy feedback while writing code develop compliance intuition over time — they stop writing non-compliant configurations because they see the violation immediately. IDE feedback is the most efficient compliance education mechanism available.

Source: Prisma Cloud Code Security

Anatomy

Policy-as-code in CI

Checkov (for Terraform, Kubernetes YAML, CloudFormation) runs as a required CI stage. Custom policies are written in Python (Checkov) or Rego (OPA) and code-reviewed by the security team. All policies are versioned in the policy repository. Updating a policy requires a PR review by one security team member and one senior SRE. The policy repository is the source of truth for all compliance requirements.

Kubernetes admission controller

OPA Gatekeeper is deployed as a ValidatingWebhookConfiguration in the cluster. Constraint templates are defined in Rego and applied as Constraint objects (Kubernetes CRDs). Constraints are deployed to staging first and validated against existing workloads before applying to production. A constraint that would break existing production workloads is detected in staging.

CSPM integration

AWS Config rules (or GCP Security Command Center) check production resources continuously. Findings above medium severity create a Jira ticket automatically. Finding SLAs mirror vulnerability SLAs: Critical = 24h, High = 7 days, Medium = 30 days. The CSPM dashboard shows compliance score per service, trending over time. Engineering leadership reviews the score monthly.

Policy exception process

When a policy violation cannot be remediated within the SLA (e.g., a vendor component requires privileged mode), a formal exception is requested: business justification, compensating controls, re-evaluation date. Exceptions are approved by the security lead and tracked in the CSPM tool as suppressed findings with an expiry date. Suppressed findings without an expiry date are not permitted.

Common pitfalls

PitfallWhy it happensWhat to do instead
Too many policies, all at the same severity — alert fatigueEvery compliance requirement is treated as equally urgent.Tier policies by risk: Critical (blocks deploy — public database, admin IAM to everyone), High (7-day remediation — unencrypted storage), Medium (30-day — missing tags). Informational findings go to a dashboard only, not the ticket backlog. A policy system that creates 200 tickets per week will be ignored.
Policies applied to production without staging validation"The policy is correct — we can apply it to production directly."Every new policy must run in staging against existing workloads before applying to production. A policy that blocks an existing workload in production causes an outage. Staging validation is the mandatory step — no policy goes to production without it.
Compliance exceptions that never expire"We'll review it next quarter."All exceptions have an explicit expiry date set at creation. The CSPM tool re-alerts on expired suppressions. A compliance exception with no expiry date is a compliance violation with better documentation — it doesn't reduce the actual risk.
