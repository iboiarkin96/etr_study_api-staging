Secret Management — no credential lives in code, config, or environment variable

A centralised secret store with access controls, automatic rotation, and audit logging — eliminating hardcoded credentials, long-lived tokens, and unrotated secrets as a reliability and security risk class.

TL;DR

Secret management means no credential lives in code, environment variables, or config files — all secrets are stored in a centralised secret manager (HashiCorp Vault, AWS Secrets Manager, or equivalent), injected into services at runtime, and rotated automatically on a schedule. Three mandatory practices: no secret in git (enforced by pre-commit hooks and CI scanning that fails on any detected credential pattern); automatic rotation (database passwords rotate every 90 days, API keys on every deployment for services that support it); and access logging (every secret access is logged with service identity, timestamp, and credential identifier — enabling detection of leaked credentials through access anomalies). A leaked credential is a P0 incident; the playbook for credential compromise requires rotation within 30 minutes.

What it is

Secret management is the set of practices that ensure credentials, API keys, certificates, and other sensitive values are stored, accessed, and rotated in a controlled, auditable way. Four pillars:

  • Centralised secret store: All secrets live in a dedicated secret management system (HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager). Access is controlled by service identity (Kubernetes service account, IAM role) — not by who the requesting engineer is. Each service can only access the secrets it is explicitly authorised to read.
  • No secrets in git: A mandatory pre-commit hook and CI scanner check every commit and PR for credential patterns (AWS access keys, private keys, JWT secrets, connection strings with passwords). The CI gate fails on any detected credential. Developers who commit a secret are required to rotate it immediately, even if it was committed to a private repository.
  • Runtime injection: Secrets are injected into services at startup by a sidecar or init container that fetches the secret from the store and passes it to the application as an in-memory value. Secrets are never written to disk, never stored in environment variables of persistent processes, and never included in container images.
  • Automatic rotation: Database passwords rotate on a 90-day schedule. API keys with rotation support rotate on every deployment. TLS certificates rotate before expiry with a 30-day lead time. Rotation is automated — the secret store generates the new credential, updates the service, and invalidates the old credential without human intervention.

Why we need it

Credentials in git are the single most common cause of cloud infrastructure compromise in startups. GitHub's own research shows that hundreds of thousands of secrets are committed to public repositories every week, and the median time from credential commit to first exploitation is under 4 minutes for AWS access keys. Even for private repositories, a leaked credential in git history is permanent — git log retains it even after the file is deleted.

Unrotated credentials compound the risk. A credential that is compromised and never rotated gives the attacker indefinite access. Automatic rotation means the window of opportunity for a compromised credential is bounded by the rotation schedule — 90 days maximum for a database password, 24 hours for a per-deployment API key.

How others do it

HashiCorp

HashiCorp's Vault introduced "dynamic secrets" — credentials that are generated on demand for a specific service request and automatically expire after a configurable TTL (time-to-live). A database dynamic secret issues a unique database user with read/write permissions, valid for 1 hour. After 1 hour, the credential is revoked by Vault automatically. The service requests a new credential for the next connection. There is never a long-lived database password — each connection uses a credential that expires within the session. Compromise of one credential expires in an hour with no action required.

What we take: Dynamic secrets are the correct long-term model for any credential system that supports it. For databases that support it (PostgreSQL, MySQL, MongoDB), dynamic secrets eliminate the rotation problem entirely — there is nothing to rotate because credentials are ephemeral by design.

Source: HashiCorp Vault — Database Secrets

Stripe

Stripe's secret management practice includes a "secret hygiene score" per service — a weekly metric combining: no unrotated secrets older than policy limits, no secrets with overly broad access scopes, no secrets accessed from unexpected services. The score is visible to each team and rolled up to leadership monthly. The score creates a clear, non-attributable signal for teams that are drifting on secret hygiene without requiring a security audit.

What we take: Measure secret hygiene as a service metric, not just as a compliance checkbox. A per-service score that trends over time is a leading indicator of risk accumulation before an incident demonstrates it.

Anatomy

Pre-commit and CI secret scanning

Two layers: a pre-commit hook (gitleaks, detect-secrets) that blocks the commit before it leaves the developer's machine; and a CI gate that scans every PR diff and fails on any detected credential pattern. The CI gate catches cases where the pre-commit hook is not installed or was bypassed. Both layers are required — neither is sufficient alone.

Service identity and least-privilege access

Each service has a dedicated service account (Kubernetes SA, AWS IAM role) that is authorised to access only the specific secrets that service needs. The authorisation is code-reviewed and audited quarterly. A service that can read all secrets has no useful access control — least privilege means the blast radius of a compromised service is bounded to its own secrets.

Credential compromise playbook

If a secret is detected in git or an unexpected access is detected in the audit log: revoke the credential immediately (rotate, don't just delete from git); audit the access log for usage since the credential was created; assess whether any systems were accessed with the credential; file a security incident within 30 minutes. The rotation target: 30 minutes from detection to new credential active and old credential revoked.

Rotation testing

Rotation is tested quarterly in staging: trigger a rotation of a database credential, verify the service connects using the new credential, verify the old credential is rejected. Rotation that has never been tested will fail in production — usually at the moment the old credential expires and the new one hasn't propagated. Test before the expiry date.

Common pitfalls

PitfallWhy it happensWhat to do instead
Secrets in environment variables of Kubernetes pods"It's not in the code — it's in the environment variable."Kubernetes environment variables sourced from Secrets objects are stored in etcd in base64-encoded form (not encrypted by default). They are visible to anyone with kubectl access to the namespace. Use a CSI secrets driver (Vault Agent Injector, Secrets Store CSI Driver) to inject secrets from the secret store at runtime, not at deploy time via environment variables.
Rotation never tested until it fails in production"Rotation is automated — it doesn't need testing."Automated rotation can fail: the rotation credential itself might expire, the service might not handle credential refresh mid-connection, the new credential might not propagate before the old one expires. Test rotation quarterly in staging with a forced rotation. The first time rotation is tested is not production.
Single shared credential used by multiple services"It's easier to share one API key."Shared credentials cannot be rotated for one service independently. If one service is compromised, the credential must be rotated across all services simultaneously — which becomes a coordinated incident rather than a bounded rotation. One service, one credential. Always.
