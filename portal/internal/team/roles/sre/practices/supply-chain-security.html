Supply Chain Security — you are responsible for what you deploy, not just what you write

A set of controls that verify the integrity and provenance of every artefact in the deployment pipeline — from source code to container image — preventing tampered dependencies and compromised build pipelines from reaching production.

TL;DR

Supply chain security uses the SLSA (Supply Chain Levels for Software Artifacts) framework to harden the path from source code to production artefact. Four practices: dependency pinning (all dependencies are pinned to an exact hash, not a version range — a version range allows a compromised package to be silently pulled); SBOM generation (a Software Bill of Materials is attached to every container image, listing all dependencies and their hashes — enabling rapid impact assessment for new CVEs); artefact signing (every container image and release artefact is signed with a private key; the signature is verified before deployment — a tampered image will fail verification); and CI/CD hardening (least-privilege CI runners, no persistent credentials in CI, build reproducibility — two builds from the same commit produce byte-identical artefacts). The SolarWinds attack (2020) and XZ Utils backdoor (2024) demonstrate that supply chain attacks are the highest-leverage attack vector for sophisticated adversaries: compromise one build tool and affect thousands of deployments.

What it is

Supply chain security is the set of controls that protect the integrity of the path from source code to production deployment. The SLSA (Supply-chain Levels for Software Artifacts) framework defines four levels of supply chain hardening; SLSA 2 (build provenance) is the minimum for Tier 0 and Tier 1 services. Four concrete practices:

  • Dependency pinning: All dependencies — direct and transitive — are pinned to an exact cryptographic hash (SHA256), not a version range. npm ci (not npm install), pip install --require-hashes, go.sum. A version range like ^1.2.3 can resolve to a different package on each build — an attacker who compromises a package registry can push a malicious version that satisfies the range. Pinning to a hash makes this attack impossible.
  • SBOM generation: A Software Bill of Materials (in SPDX or CycloneDX format) is generated and attached to every container image and release artefact at build time. The SBOM lists all components and their hashes. When a new CVE is published, the SBOM enables immediate impact assessment: "which of our services includes this dependency?" — answerable in seconds, not hours of manual dependency tree inspection.
  • Artefact signing: Every container image is signed using Sigstore/Cosign with a private key managed in the secret store. The signature is verified by the deployment system before any image is scheduled to run. A tampered image (one that wasn't produced by the canonical CI pipeline) will not have a valid signature and will be rejected.
  • CI/CD hardening: CI runners run with least-privilege IAM roles (write access only to the specific registry they push to, no production access). No persistent credentials in CI — OIDC token exchange with short-lived credentials per pipeline run. Build environments are ephemeral (no shared state between builds).

Why we need it

The SolarWinds attack (2020) inserted malicious code into the build process, affecting 18,000 organisations that deployed the compromised software. The XZ Utils backdoor (2024) was a two-year social engineering campaign that nearly compromised the SSH authentication layer of most Linux distributions. Both attacks targeted the supply chain specifically because it is the single point where one compromise affects every downstream deployment.

Supply chain attacks are increasing in frequency because attackers have learned that compromising a widely-used dependency is more efficient than compromising individual targets. A service with 50 direct dependencies and 500 transitive dependencies has 550 potential supply chain attack surfaces — and most organisations have visibility into zero of them.

How others do it

Google (SLSA framework)

Google developed the SLSA framework (Supply-chain Levels for Software Artifacts) and published it as an open standard. SLSA defines four levels: L1 (provenance exists — a record of what produced the artefact), L2 (hosted build platform — the provenance is signed by the build service), L3 (hardened build platform — the build environment is isolated and reproducible), L4 (two-party review for all changes). Google requires SLSA L3 for all internal services that produce artefacts deployed to production. The framework is notable for being incremental — SLSA L1 is achievable in an afternoon; SLSA L4 is a multi-quarter effort.

What we take: Target SLSA L2 as the minimum for all services — signed provenance from a hosted build system. SLSA L1 and L2 are achievable with existing CI/CD tooling (GitHub Actions, GitLab CI) without custom infrastructure. L3 hardened builds are the right target for Tier 0 services once L2 is in place.

Source: SLSA Framework — slsa.dev

Sigstore / Linux Foundation

Sigstore is a Linux Foundation project that provides free artefact signing infrastructure — a public certificate authority and transparency log (Rekor) that makes artefact signatures publicly auditable. Sigstore's Cosign tool is the most widely adopted container signing solution. The transparency log means that even if a signing key is compromised, the compromise is detectable because all signatures (including malicious ones made with the stolen key) are recorded in an append-only public log. Kubernetes itself uses Sigstore to sign all official releases.

What we take: Use Cosign + Rekor for container image signing. The public transparency log provides tamper detection even in key compromise scenarios — a property that private signing systems do not provide. Cost: zero (Sigstore is free infrastructure).

Source: Sigstore — sigstore.dev

Anatomy

Dependency pinning enforcement

CI fails if a dependency file contains version ranges instead of hash pins. For npm: package-lock.json must be committed and npm ci used (not npm install). For Python: requirements.txt with --hash flags. For Go: go.sum committed and verified. The CI gate that checks this runs before any build step.

SBOM generation and storage

SBOM is generated at image build time using Syft or Trivy. The SBOM (in SPDX JSON format) is attached to the container image as an OCI manifest annotation and stored in the artefact registry alongside the image. When a new CVE is published, a service scans all SBOMs in the registry and reports affected images within 1 hour — without needing to rebuild or redeploy anything.

Signature verification at deploy time

Kubernetes admission controller (Kyverno or OPA Gatekeeper) verifies the Cosign signature of every image before a pod is scheduled. Images without a valid signature from the canonical signing key are rejected at the admission stage — they never reach a node. This makes the signing requirement enforcement automatic and impossible to bypass with kubectl.

CI/CD hardening checklist

Quarterly CI audit: no persistent credentials in CI environment variables (use OIDC token exchange); no CI runner with production access beyond the specific registry it publishes to; no build steps that pull from external URLs at build time (all external dependencies pre-fetched and pinned); build logs retained for 90 days; all CI configuration reviewed in code review (no manual CI configuration through UI). CI configuration is code.

Common pitfalls

PitfallWhy it happensWhat to do instead
Version pinning without hash pinning"We pin to an exact version like 1.2.3 — that's safe."Version pinning without hash pinning is not supply chain protection. A malicious publisher can replace a version on a registry with different content (this has happened). Only hash pinning guarantees that you get the same bytes every time. Pin to the SHA256 hash of the expected download, verified by the package manager's lockfile mechanism.
SBOM generated but never queried"We generate the SBOM for compliance — we don't use it operationally."The SBOM is only valuable if it's queried when a new CVE is published. Set up an automated query: on every NVD CVE publication, scan all SBOMs in the registry for the affected component and create tickets for affected services. Without the query, the SBOM is documentation, not a security control.
CI runners with production write access"It's easier if the CI runner can deploy directly."CI runners should only write to the artefact registry. Deployment is a separate step, triggered by a deployment system with its own access controls. A compromised CI runner with production access can deploy arbitrary code to production without going through any signing or verification step.
