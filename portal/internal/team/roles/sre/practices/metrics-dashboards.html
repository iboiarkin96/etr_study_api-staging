Metrics & dashboards — four golden signals, versioned as code

Every service exposes the four golden signals. Every dashboard is stored in git, reviewed before merge, and linked from the runbook. A service without a canonical dashboard is a service without observability.

TL;DR

The four golden signals (latency, traffic, errors, saturation) are the minimum viable metric set for any user-facing service. Every service exposes them via a standard Prometheus client library scraped through /metrics. Dashboards are versioned as Grafana JSON in infra/grafana/dashboards/, reviewed by at least one other engineer before merge, and linked from the service runbook. Every service ships with two canonical dashboards — a RED panel (rate, errors, duration) and a service-overview that drills down — and the SLO dashboards are generated automatically from infra/slo/. A dashboard built during an incident is too late; a dashboard reviewed before production is what enables fast diagnosis when it matters.

What it is

Metrics & dashboards is the discipline of deciding which numerical signals a service emits, how those signals are stored and queried, and how they are arranged into a small number of canonical dashboards that the team actually opens during an incident. The work is upstream of alerting and downstream of instrumentation: the metric has to exist before a dashboard can show it, and the dashboard has to exist before an on-call engineer can use it. A dashboard invented during an incident is built by the wrong person under the wrong conditions and almost always retired the week after.

Google SRE identified the four golden signals — latency, traffic, errors, saturation — as the minimum viable metric set for any user-facing service. Latency is measured at multiple quantiles (p50, p95, p99) and split between success and error paths so a failing endpoint cannot hide inside an aggregate. Traffic is the demand the system is taking. Errors are the rate of failed requests, both explicit (5xx) and implicit (200 with an error payload). Saturation is how full the most constrained resource is. If all four look normal, the service is almost certainly healthy; if any one is degraded, that is where investigation starts. Dashboards-as-code means every Grafana panel lives as JSON in git, is reviewed before merge, and is deployed by pipeline — not poked into the Grafana UI by hand.

Why we need it

Ad-hoc dashboards built during an incident are too late. They are constructed under stress by an engineer who is simultaneously diagnosing the problem, and they are never maintained afterwards. The same shop will have forty half-broken dashboards from forty old incidents and no canonical view of any service. Dashboards-as-code enforce a different discipline: the dashboard is reviewed before the service reaches production, by engineers who are not in crisis mode, and it is reproducible across environments. When the page fires at 3 a.m., the on-call opens the runbook, clicks through to the canonical dashboard, and reads the four signals — not a search bar.

The simplicity matters as much as the discipline. Forty-panel dashboards are noise; on-call attention is the scarcest resource in the system, and a layout that takes five minutes to read costs five minutes of an incident. The canonical dashboard for a service has eight to twelve panels — the golden signals plus SLO burn rate plus two or three service-specific signals — and anything more detailed lives in linked sub-dashboards. The point of a dashboard is not to show everything; it is to answer the one question the on-call needs answered in twenty seconds.

How others do it

Short tour of teams whose monitoring practice is published and copied. Each card is one organisation, what they do, and what we take from it.

Google SRE

Google codified the four golden signals in the SRE Book and built their internal monitoring infrastructure around them. Every Google service has a canonical monitoring dashboard that shows the four signals plus SLO burn rate. The dashboard is linked from the service's on-call runbook and is the first artifact an on-call engineer opens when paged — the routing is part of the rule, not part of the operator's memory.

What we take: Link the canonical dashboard from the runbook and from every alert annotation. When the page fires, the dashboard is one click away. The runbook and the dashboard are the same artifact in two views.

Source: Google SRE Book — Monitoring distributed systems

Weaveworks (RED method)

Tom Wilkie at Weaveworks distilled the golden signals for request-driven services into RED: rate, errors, duration. Three panels per service, identical layout across the fleet, so an on-call engineer who has read one RED dashboard can read every RED dashboard. The pattern is now the de-facto template for microservice observability and is built into the Grafana service-graph view by default.

What we take: Every service ships with a RED dashboard generated from a shared template. Operators never have to learn a new layout per service — the three panels are always in the same positions, with the same units, on the same time range.

Source: Tom Wilkie — The RED method

Brendan Gregg (USE method)

For resources rather than requests, Brendan Gregg's USE method (utilization, saturation, errors) is the canonical analysis. For every resource — CPU, memory, disk, network — measure the three. USE is how a host or queue is diagnosed; RED is how a service is diagnosed. The two methods cover the system end-to-end without overlap.

What we take: RED dashboards for services, USE dashboards for the infrastructure they depend on. The on-call starts with the service RED to see whether users are affected, and drills into the USE board for the suspected resource only when the symptom points there.

Source: Brendan Gregg — The USE method

Grafana Labs

Grafana Labs ships mixins: packages that bundle Grafana dashboards, Prometheus alerts, and recording rules for a specific service or library as code. Teams import the mixin for PostgreSQL, Redis, or NGINX and get production-quality dashboards built by the people who maintain those systems, without rebuilding from scratch.

What we take: Use the published mixins for infrastructure services. Spending three days hand-rolling a Postgres dashboard when postgres-mixin already exists is wasted engineering. Our own service dashboards are generated from an internal mixin template so they stay consistent.

Source: Grafana dashboard documentation

Across every team whose monitoring practice is public, the playbook converges: a tiny canonical metric set (golden signals / RED / USE), shared dashboard templates so layouts are identical across the fleet, versioned-in-git Grafana JSON, and every dashboard linked from the runbook of the service it monitors.

Anatomy

The parts a complete metrics-and-dashboards practice has, with one sentence on each. None are optional.

Shared instrumentation library

One Prometheus client per language, pre-configured to expose the four golden signals with consistent metric names and label schemas. Services import it; they do not register counters by hand.

Canonical /metrics endpoint

Every service exposes /metrics on the admin port. Prometheus scrapes on a fixed schedule. The endpoint contract is enforced by a smoke test in CI — a service that does not expose it does not deploy.

Dashboard-as-code

Grafana JSON stored in infra/grafana/dashboards/, reviewed by at least one other engineer before merge, and deployed via a CI pipeline that pushes to Grafana via the API. Direct UI edits on canonical dashboards are blocked by Grafana RBAC.

RED + service-overview pair

Every service has two dashboards: a RED panel (rate, errors, duration) for the at-a-glance view, and a service-overview that drills down into dependencies, queue depths, and saturation signals.

Generated SLO dashboards

SLO dashboards are auto-generated from infra/slo/ at deploy time so the SLO definition and its dashboard never drift. No hand-edited burn-rate panels.

Runbook link

Every canonical dashboard is linked from the service's runbook; every alert annotation includes the dashboard URL. There is never a step "find the right dashboard" inside an incident.

Quarterly dashboard review

Stale panels are removed, new signals added, thresholds recalibrated. A dashboard with 40% stale panels erodes trust in the panels that are still accurate.

How it works in this project

Study App services expose metrics on a Prometheus-scraped /metrics endpoint published by the shared instrumentation library. Prometheus runs in the platform namespace and scrapes every service on a 15-second cadence; the labels (service, environment, region, version) are stamped on every series so cross-service aggregation works without surprises. Recording rules pre-compute the heavy expressions — RED rates, p95/p99 quantiles, SLO error fractions — so dashboard load times stay below a second even at incident-shift query volume.

Dashboards live in infra/grafana/dashboards/, one folder per service, one JSON file per board. Every service ships with two canonical dashboards: a RED dashboard (rate, errors, duration) generated from the shared template and a service-overview that drills down into dependencies, saturation, and queue depths. The SLO dashboards are auto-generated from the SLO definitions in infra/slo/ — see SLO & error budget for how those are written — so the SLO and its dashboard cannot drift. Direct edits to canonical dashboards via the Grafana UI are blocked by RBAC; the canonical folder is read-only and every change goes through PR review.

Deep dive — the theory you need

RED vs USE — different methods for different objects

The two canonical analyses solve different problems and apply to different things. RED — rate, errors, duration — describes a request-driven service: how many requests is it handling, how many fail, and how long do they take. USE — utilization, saturation, errors — describes a resource: how busy is it, how queued is it, and how often does it fault. For an HTTP service the right view is RED; for the CPU, disk, or queue it runs on, the right view is USE. The mistake is mixing them: a service "RED" board cluttered with CPU graphs becomes hard to read, and a host "USE" board with request rates becomes hard to maintain.

When each applies

RED applies whenever the unit of work is a request — HTTP handler, gRPC method, message consumer, background job. USE applies whenever the unit of analysis is a resource that can be saturated — CPU core, memory pool, disk volume, connection pool, thread pool, Kafka partition. In practice every service has a RED dashboard for itself and links to USE dashboards for the resources it depends on. The on-call starts on RED to see whether users are affected; if errors or duration are bad, they jump to the USE board for the resource the runbook flags first.

Histograms vs summaries — and the quantile-aggregation trap

Prometheus offers two ways to record a latency distribution: histograms (buckets stored on each instance, quantiles computed at query time) and summaries (quantiles pre-computed on the instance). The temptation is summaries — they are cheaper to query — but they have a fatal flaw: summary quantiles cannot be aggregated. The 99th percentile across three instances is not the average or max of each instance's 99th percentile; it is a property of the combined distribution. Summaries throw that distribution away. Histograms keep the buckets, so histogram_quantile over a sum across instances returns a correct global quantile. Rule of thumb: anything you need to roll up across instances must be a histogram, full stop. Summaries are acceptable only for per-instance debugging where global aggregation is never needed.

Cardinality budgets — why high-cardinality labels are death

Every unique combination of label values on a Prometheus metric creates a new time series. A counter labelled with (service, endpoint, status) on a service with 20 endpoints and 5 status codes is 100 series — manageable. Add user_id and on a million users it is a hundred million series, and the TSDB falls over. The same trap waits in request_id, trace_id, raw URL paths with embedded IDs, and any label sourced from user input. The rule is: label values must be drawn from a small, bounded set known at deploy time. High-cardinality identifiers belong in traces and logs, not in metrics. Honeycomb's writing on the cardinality wall is the best public reference for what happens when this rule is broken at scale.

Dashboard design — overview → drilldown layering

A canonical dashboard has at most twelve panels, arranged top-to-bottom in priority order: golden signals first (the answer to "is anything wrong"), SLO burn rate second (the answer to "how bad"), service-specific signals third (the answer to "where to look next"). Detail panels — per-endpoint breakdowns, dependency latencies, queue depths — live in a linked drilldown dashboard, not on the main board. The overview answers questions in twenty seconds; the drilldown answers them in five minutes. Mixing the two means neither works during an incident.

Trade-offs — what this practice costs

  • Up-front template investment. Writing the shared instrumentation library and the RED dashboard template takes a sprint before any service benefits. Teams that skip it end up with 30 hand-rolled dashboards in 30 different layouts and a steep learning curve per service.
  • Discipline at PR time. Dashboards-as-code means every dashboard change is a PR. The first month is friction; after that it is the only way anyone wants to work because rollbacks are git revert and not "remember the panel from Tuesday".
  • Cardinality discipline. Engineers reach for "just add a label" reflexively. A linting step in CI that flags new labels against a known schema is the cheapest way to keep the TSDB alive.

Lifecycle

instrument in code → scrape via /metrics → recording rules → panel authored → dashboard versioned in repo → linked from runbook → reviewed quarterly · retuned · retired

A new metric starts in application code via the shared instrumentation library — never with a hand-rolled counter. Prometheus picks it up automatically on the next scrape because the service exposes /metrics as a contract. If the metric is queried often enough to be expensive, it gets promoted to a recording rule so the dashboard does not recompute it on every load. A panel is authored against the rule, the dashboard JSON is committed to infra/grafana/dashboards/, a PR is opened, and at least one other engineer reviews it. Once merged, the CI pipeline pushes the dashboard to Grafana via the API and the service runbook is updated with the link. Every quarter the dashboard is reviewed against the panels people actually use — stale ones are deleted, drift is corrected, and the layout is brought back in line with the canonical template.

Common pitfalls

PitfallWhy it happensWhat to do instead
Forty-panel dashboards no one reads "More data is better." The author adds a panel for every signal they can think of and never deletes any. The canonical dashboard has eight to twelve panels: golden signals + SLO burn rate + two or three service-specific signals. Additional detail lives in a linked drilldown, not on the main board. The overview answers the on-call's question in twenty seconds; anything that costs more than that gets moved.
Dashboards built in the UI, never in code Grafana's UI is easy; git is friction. The first dashboard ships from a browser tab and the pattern sticks. Lock down direct UI edits on canonical dashboards via Grafana RBAC. Exploration uses a sandbox folder; sandbox dashboards are never linked from runbooks or alerts. All canonical dashboards live in infra/grafana/dashboards/ and ship via PR.
High-cardinality labels (user_id, request_id) "It would be useful to slice by user." Each new label value creates a new series; the TSDB silently bloats until it falls over. Label values must be from a small, bounded set known at deploy time. CI lint blocks any new label not on the approved schema. User-level identifiers belong in traces and logs, not metrics.
Summary quantiles aggregated across instances Summaries look cheaper than histograms; the bug is invisible until p99 across the fleet stops matching reality. Use histograms for anything that has to roll up across instances. histogram_quantile over a summed bucket vector returns a correct global quantile; summaries cannot do that and there is no workaround.
Per-team metric namespaces with no shared schema Each team picks their own label names; cross-service queries become impossible. Enforce a canonical label schema: service, environment, region, version. Without it the SLO calculations break the first time you try to aggregate two services that named the same dimension differently.
SLO dashboards diverge from SLO definitions The SLO is updated in infra/slo/ but the dashboard panel was hand-built and nobody touches it. Generate SLO dashboards from infra/slo/ at deploy time. The definition is the single source of truth; the dashboard is its rendering. Hand-edited burn-rate panels are deleted on sight.

Worked example

Grafana panel JSON

Lessons API — RED panel (rate, errors, p95 / p99 latency)

{
  "title": "Lessons API — RED",
  "uid": "lessons-api-red",
  "tags": ["service:lessons-api", "method:red", "owner:sre-platform"],
  "panels": [
    {
      "id": 1,
      "title": "Request rate",
      "type": "timeseries",
      "targets": [{
        "expr": "sum by (status) (rate(http_requests_total{service=\"lessons-api\"}[1m]))",
        "legendFormat": "{{status}}"
      }],
      "fieldConfig": { "defaults": { "unit": "reqps" } }
    },
    {
      "id": 2,
      "title": "Error rate",
      "type": "timeseries",
      "targets": [{
        "expr": "sum(rate(http_requests_total{service=\"lessons-api\",status=~\"5..\"}[1m])) / sum(rate(http_requests_total{service=\"lessons-api\"}[1m]))",
        "legendFormat": "5xx ratio"
      }],
      "fieldConfig": { "defaults": { "unit": "percentunit", "min": 0, "max": 0.05 } }
    },
    {
      "id": 3,
      "title": "Duration — p95 / p99",
      "type": "timeseries",
      "targets": [
        {
          "expr": "histogram_quantile(0.95, sum by (le) (rate(http_request_duration_seconds_bucket{service=\"lessons-api\"}[5m])))",
          "legendFormat": "p95"
        },
        {
          "expr": "histogram_quantile(0.99, sum by (le) (rate(http_request_duration_seconds_bucket{service=\"lessons-api\"}[5m])))",
          "legendFormat": "p99"
        }
      ],
      "fieldConfig": { "defaults": { "unit": "s" } }
    }
  ],
  "links": [
    { "title": "Runbook", "url": "https://internal.studyapp/runbooks/lessons-api" },
    { "title": "SLO board", "url": "https://grafana.studyapp/d/lessons-api-slo" }
  ]
}

What this dashboard does. Three panels, top-to-bottom: rate split by status (so the on-call sees green vs red traffic at a glance), error ratio with a 5% display ceiling (anything above is clearly off the chart), and p95/p99 duration computed by histogram_quantile over a summed bucket vector so the quantiles aggregate correctly across instances. The dashboard is generated from the shared RED template — every service in the fleet looks identical, in the same panel order, so an on-call who has read one has read them all. Runbook and SLO board are linked from the dashboard header; the alert annotation for any page on this service points directly here.
