Structured logging — machine-readable events, not prose

Every log line is a JSON event with a locked schema and a trace_id — so an operator pivots from a metric spike to the exact request's logs in one query, not an afternoon of grep.

TL;DR

Structured logging means every log line is a JSON object with a canonical schema: ts, level, service, trace_id, request_id, user_id, message, plus domain fields. No printf strings, no free-form prose. The shared library injects trace context automatically; a CI lint rejects any log call that bypasses it. PII is stripped in the pipeline before the event lands in the index. The pay-off is the pivot — metric anomaly → log query by trace_id → distributed trace — measured in seconds, not hours.

What it is

Structured logging is the discipline of emitting every log line as a machine-parseable event — a JSON object with a fixed schema — rather than a human-readable sentence with values interpolated into it. The unit of work is no longer "a line of text" but "an event with fields", and the consumer is no longer a human running grep but a query engine running a filter over an index. The shift sounds cosmetic until the first incident; then it is the difference between answering "show me every error in the request whose trace ID is abc123" in 400 milliseconds versus in 40 minutes.

A log event that ships to production satisfies five mandatory properties: it is valid JSON on a single line; it carries the canonical fields (ts, level, service, trace_id, request_id, user_id, message); it uses field names locked in the schema repository and types that have not changed since the field was introduced; it has been emitted through the shared logging library, not constructed by hand; and it has passed the PII filter before reaching the index. Anything missing one of the five does not ship — CI in the shared library repo enforces this.

Why we need it

Unstructured logs are an asset that ages into a liability. The first week of a service's life, a free-form printf reads beautifully; the first incident a year later, the same line is impossible to query because nobody remembers which field is the user ID and which is the IP. The aggregator cannot help: it sees a string. Engineers fall back to grep | awk | sort | uniq against gigabytes of text, and the time-to-diagnosis becomes a function of typing speed.

Structured logs invert the economics. The aggregator parses once at ingest time and indexes by field; subsequent queries are sub-second. The trace_id field is the join key — the same ID the distributed trace carries, the same ID the request travels with from the edge to the database — so an operator who sees a metric spike at 14:02 can pull every log line from every service that participated in the spike's offending requests with one query. That pivot is the single biggest reduction in mean-time-to-diagnosis available; it costs nothing at runtime and pays back the migration sprint inside one incident.

How others do it

Short tour of teams whose logging practice is published. Each card is one organisation, what they do, and what we take from it.

Datadog

Datadog's platform is built around unified log / metric / trace correlation through a shared trace_id. Their explicit recommendation: use one canonical logging library per language (pino for Node, zap or zerolog for Go, structlog for Python) configured once at startup to read the OpenTelemetry context and inject trace IDs automatically. Individual call sites pass no context; the library does it.

What we take: Automatic trace-context injection via library middleware is the only scalable answer. Manual context plumbing produces inconsistent coverage — some lines have trace IDs, some do not — and the resulting log search is useless on exactly the line you need.

Source: Datadog — Log Collection & Management

Grafana Loki

Loki's best-practice guide is unusual: it tells teams to keep cardinality on labels low and push the high-cardinality fields (user IDs, request IDs, trace IDs) into the JSON body, where LogQL parses them at query time. The trade-off is deliberate — labels are the index, the body is the payload; mixing the two produces an index explosion that kills query latency.

What we take: Keep the index shape boring (service, env, level) and shove everything queryable into the JSON body. Loki parses fields at query time fast enough that there is no reason to promote them to labels — and every promoted label is one more dimension the index has to remember forever.

Source: Grafana Loki — Labels best practices

Charity Majors (Honeycomb)

Majors's blog reframes structured logging as the wide-event story: one log line per request, with as many fields as the request had interesting facts about it. Levels and severity are debatable, schemas drift, but the discipline of "every interesting attribute lives on the event as a typed field" is the part that converts logs into a queryable record of what the system actually did.

What we take: Add fields liberally; prefer one wide event per request over a stream of narrow events. The aggregator handles wide rows fine; an operator filtering by user_id AND error_code AND route would rather have one event with all three than three events that have to be joined.

Source: charity.wtf — observability writing

OpenTelemetry

The OpenTelemetry Logs specification standardises the data model: severity, body, attributes, trace context, resource attributes. It does not prescribe a wire format but it does fix the field semantics — and the SDKs for every supported language produce log records that already speak it, so a service emitting via the OTel logger arrives at any compliant backend ready to be queried by the same field names everywhere.

What we take: Borrow the OTel field semantics even if the backend is Loki rather than an OTel-native vendor. trace_id, span_id, severity_number, service.name mean the same thing across the industry; using anything else trades portability for nothing.

Source: OpenTelemetry — Logs specification

Across every team whose logging practice is public, the playbook converges: a single library per language, automatic trace-context injection, OTel-compatible field names, low-cardinality index labels, wide events with all the facts on them, and PII scrubbed at the collector before anything hits the index.

Anatomy

The parts a complete structured-logging practice has, with one sentence on each. None are optional.

Canonical schema

A locked field list — ts, level, service, trace_id, request_id, user_id, message — with stable types. The schema lives in a repo; renames go through a versioned migration, not a Slack thread.

Shared library per language

zap (Go), pino (Node), structlog (Python). Pre-configured to emit JSON to stdout with the canonical fields. Services call the library; no service constructs log lines by hand.

Automatic trace-context injection

The library reads the OpenTelemetry context from the current goroutine or async-local store and stamps trace_id + span_id on every event. Zero per-call plumbing; zero forgotten lines.

CI lint enforcement

A linter that fails the build if a log call bypasses the canonical library or references a field not in the schema. Catches drift before it reaches production and produces an un-queryable event.

Pipeline: collector + PII filter

Vector tails stdout, applies redaction rules (email, phone, card number patterns) before forwarding to Loki. PII never touches the index; the contract holds even if a developer accidentally logs a raw request body.

Level discipline

DEBUG sampled or off in production, INFO for state transitions, WARN for recoverable failures, ERROR for failures requiring attention, FATAL for process-killing conditions. Levels are a routing primitive, not a developer mood.

Retention policy

Routing rules at the collector: INFO retained 14 days, ERROR retained 90 days, DEBUG dropped in production. Retention is configured centrally, not negotiated per service.

How it works in this project

Study App emits structured logs from every service through one canonical library per language — zap in Go services, pino in Node services — configured at startup to write JSON to stdout with the canonical schema. The schema is locked in infra/logging/schema.json and a CI lint in the shared-library repo rejects pull requests that bypass the library or reference an unknown field. Every event carries ts, level, service, trace_id, request_id, user_id, and message; domain fields (route, error code, latency in ms) ride alongside.

Containers write to stdout; Vector tails the container logs, applies a PII filter (regex-based redaction for email, phone, card numbers, and a small allow-list of fields that may carry user-readable identifiers) and ships the events to Loki. Loki indexes a small, low-cardinality label set (service, env, level) and parses the JSON body at query time — so a typical incident query ({service="lessons-api",level="error"} |= "trace_id=abc123" in LogQL) runs in milliseconds without forcing high-cardinality fields onto the index. Trace IDs propagate through HTTP and gRPC headers using OpenTelemetry instrumentation; the same ID appears in the metric exemplars in metrics & dashboards and on the distributed trace in distributed tracing, so the three views are joinable on the field every event already carries.

Deep dive — the theory you need

Schema discipline — field names locked, types stable

A schema is a contract between the producer (every service emitting logs) and the consumer (every dashboard, every alert query, every on-call runbook). Once a field name and type ship to production they are immutable in practice: rename user_id to userId and every dashboard built on the old name breaks silently — the query keeps running, it just returns nothing. The discipline is to treat the schema like an API. New fields can be added freely; existing fields cannot change name or type. When the field semantics genuinely need to change, the schema is versioned (every event carries a schema_version), the new field is added in parallel, dashboards migrate, and the old field is retired on a schedule. It is more ceremony than developers like; it is the only thing that keeps a year-old dashboard functional.

Log levels and their meanings

Levels are not severity opinions; they are a routing key. The conventional ladder is DEBUG (verbose, useful only inside a development environment — sampled or dropped in production), INFO (a significant state transition: request handled, job started, connection opened), WARN (a recoverable degradation: retry succeeded, fallback used, deprecated path hit), ERROR (a failure that needs attention but did not kill the process), and FATAL (a failure that kills the process; the next line in the log is typically silence). Discipline at this layer is what makes the rest of the practice viable: an INFO that should be DEBUG floods the index with noise, an ERROR that should be WARN pages the on-call for a self-healing retry. Code review enforces the ladder; the linter cannot, because the right level is contextual.

Sampling vs full ingest — the economics

At low traffic, ingesting every log line is cheap and correct. At high traffic the bill becomes the dominant cost of the observability stack — a single chatty INFO line in a 5k-RPS path is gigabytes per day in storage. Sampling trades fidelity for cost: keep every ERROR and WARN, keep a fraction of INFO (1-in-10 or 1-in-100 depending on the route), drop DEBUG. The catch is that sampled INFO loses the property that made it useful — you can no longer answer "show me every line in this user's request" because the lines for that user might be the ones that got dropped. The mitigation is tail-based sampling: hold all the events for a trace in a buffer until the trace ends; if any event in the trace is ERROR, keep the whole trace; otherwise sample by ratio. It is more expensive at the collector but it preserves the join key, which is the entire point.

Correlation IDs — threading request, trace, log

Three identifiers tie the observability stack together. The request_id is the application-level handle on a single inbound request — generated at the edge, propagated to every downstream call, surfaced to the user as part of the error UI so they can quote it in a support ticket. The trace_id is the distributed-tracing handle — usually a 128-bit W3C traceparent value, generated by the OpenTelemetry SDK at the first instrumented hop, propagated via the traceparent header through every service. The span_id identifies the specific operation within the trace. All three appear on every log line. The trace_id is the most valuable because it is the field the distributed trace itself indexes on; the request_id is the most valuable because it is the field a user can type into a search bar. They are not redundant — they are different join keys for different audiences.

Trade-offs — what this practice costs

  • Up-front library work. Picking the canonical library, wrapping it in a thin internal façade, and migrating one service per language to it takes a sprint. Teams that skip this step end up with three logging libraries in the Go services alone and the schema drifts within months.
  • Storage and bandwidth. JSON is bulkier than free-form text by 2-3x, and the index is bulkier still. The cost is real but bounded; it is dwarfed by the cost of an hour-long incident where the on-call cannot find the failing request.
  • Schema-change ceremony. Renaming a field is no longer a one-PR change; it becomes an additive deprecation. Developers will resent this until the first time a year-old dashboard saves an incident — then the ceremony pays for itself in a single afternoon.

Lifecycle

event → log emitter (zap / pino) → stdout buffer → Vector ships + scrubs PII → parse → Loki index → query / alert

An event begins as a function call in service code — log.Error("payment failed", "order_id", id, "err", err). The shared library reads the OpenTelemetry context out of the goroutine-local store, stamps trace_id and span_id onto the record, and writes a single line of JSON to stdout. The container's stdout buffer holds the line for milliseconds. Vector, running as a node-local daemon, tails the buffer, applies the PII filter, and forwards the event to Loki over the push API. Loki parses the JSON body once at ingest, attaches the low-cardinality labels, and writes the record to the chunk store. From here the event is queryable: dashboards filter on it, alert rules count it, the on-call pivots from a trace to it. The full path runs in well under a second under normal load — the budget on the alert path lives in the rule evaluation, not the log pipeline.

Common pitfalls

PitfallWhy it happensWhat to do instead
trace_id missing from most lines SWEs pass context manually on the lines they remember and forget on the lines they don't. Library middleware injects the OpenTelemetry context automatically; a CI lint rejects log calls that bypass the library. Manual propagation always degrades into partial coverage, and partial coverage is the case that ruins the incident query.
INFO spam buries the ERROR signal "Log everything at info, it's useful for debugging." Enforce the level ladder in code review. DEBUG is verbose dev output (off or sampled in production), INFO is significant state transitions, WARN is recoverable degradation, ERROR is attention-required failure, FATAL kills the process. Promoting noise to INFO burns storage and hides the lines that mattered.
Schema drift between services One service uses user_id, another userId, a third uid — nobody noticed. Lock the schema in a repo; CI lint references it. New fields can be added freely; renames are versioned migrations with parallel emission. The aggregator cannot fix a field name it does not know about.
PII written to the index SWE logs the raw request body for "easier debugging"; the request body contains an email or a card number. PII filter at the collector (Vector) with regex-based redaction. The contract holds even if a developer makes a mistake; the index never sees the raw value. Audit the filter rules quarterly against the production stream.
High-cardinality fields promoted to labels "Querying by user_id is slow, let's index it." On Loki this explodes the label cardinality. Keep labels boring (service, env, level); keep high-cardinality fields in the JSON body. LogQL parses the body fast enough that there is no reason to promote anything — and every promotion is one more dimension the index has to remember.
Hand-rolled JSON via fmt.Sprintf Library was unavailable on a hot path, somebody built the JSON by hand "for now". The library is the contract. Hand-rolled JSON drifts within weeks: missing fields, wrong type for the timestamp, no trace context. CI lint forbids string interpolation that looks like JSON in a log call.

Worked example

Log event

ERROR — lessons-api payment retry exhausted, captured during the SLO burn that paged on-call

{
  "ts":            "2026-05-24T14:02:17.413Z",
  "level":         "error",
  "service":       "lessons-api",
  "env":           "prod",
  "schema_version": 2,
  "trace_id":      "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id":       "00f067aa0ba902b7",
  "request_id":    "req_01HZX8K3M2V7N4Q5",
  "user_id":       "usr_94217",
  "route":         "POST /v1/lessons/:id/purchase",
  "error_code":    "PAYMENT_PROVIDER_5XX",
  "retry_count":   3,
  "latency_ms":    1842,
  "upstream":      "stripe",
  "message":       "payment retry exhausted",
  "stack": [
    "lessons-api/internal/payment/charge.go:142 charge.attempt",
    "lessons-api/internal/payment/charge.go:88  charge.run",
    "lessons-api/internal/http/lessons.go:213   handler.purchase"
  ]
}

What this event does. It is the line the on-call lands on when the SLO burn-rate alert from alerting design fires. The alert query in Loki is {service="lessons-api",level="error"} | json | error_code="PAYMENT_PROVIDER_5XX"; the count of matching events in the last minute is what the alert rule thresholds on. Once the alert pages, the operator copies the trace_id from any matching event and pulls the full distributed trace — every span the request touched, the latency on each, the upstream that failed. The stack field gives the entry point inside the service without forcing a separate APM tool. None of this is possible if the same outage produced free-form text lines reading "payment failed for user 94217 after 3 retries" — the join key is gone and the query becomes a substring scan.
