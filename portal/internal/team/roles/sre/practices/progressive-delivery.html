Progressive delivery — ship at any speed, limit blast radius

A deployment strategy that exposes new code to increasingly larger user segments with automated SLO gate checks between rings — converting big-bang deployment risk into a 1% blast radius, detectable in minutes, reversible in seconds.

TL;DR

Progressive delivery promotes a deployment through rings — canary (1%), ring 1 (5%), ring 2 (25%), GA (100%) — with automated SLO gate checks between each promotion. If error rate, latency, or SLO burn rate degrades during a ring, the pipeline pauses automatically and rolls back without human intervention. Feature flags decouple deploy from release for high-risk changes. This model lets teams deploy multiple times per day safely: the blast radius of any failure is limited to the current ring's traffic share until the gate confirms safety.

What it is

Progressive delivery is a deployment strategy that gradually exposes new code to larger user segments with automated quality gates between each step. It has three main components:

  • Canary deployment: New code is deployed to a small percentage of traffic (1–5%). SLO metrics are observed for a set window (typically 15–30 minutes). If metrics are normal, the deployment is promoted to the next ring.
  • Ring model: Traffic is divided into rings. Each ring promotion requires passing automated SLO gate checks — error rate within 2× baseline, P99 latency within 1.5× baseline, SLO burn rate normal. Failed gates halt the promotion automatically and trigger rollback.
  • Feature flags: For high-risk changes, the code is deployed to 100% of users but the feature is gated behind a flag. The flag is enabled progressively — 1%, 10%, 50%, 100% — giving the team control over the exposure without requiring a separate deployment per ring.

Why we need it

Big-bang deployments move the blast radius from "zero users affected" to "all users affected" in a single step. If the deployment has a bug that affects 5% of requests, all users are exposed immediately. With a canary deployment, that bug is discovered when only 1% of traffic is affected — and it's caught before the remaining 99% ever see it.

Progressive delivery is the engineering practice that makes continuous deployment safe at scale. Spotify, Netflix, and Google deploy hundreds of times per day because each deployment is a canary, not a big bang. Frequent deployment and high reliability are not in tension — progressive delivery makes them complementary.

How others do it

Short tour of teams whose progressive-delivery practice is published and copied. Each card is one organisation, what they do, and what we take from it.

Facebook (Meta)

Facebook uses a 5-ring model for their web tier: internal employees, 1% of users, 10% of users, 50% of users, 100%. Every promotion is gated by automated analysis comparing the new version's error rate, latency P99, and user-visible crash rate against the previous version. The entire ring model is automated — no human approves ring promotions unless an engineer explicitly halts the pipeline.

What we take: Automate ring promotions. Human-approved promotions become bottlenecks and eventually get rubber-stamped. Automated promotions based on SLO gates are faster, more consistent, and catch regressions that humans would approve on a Friday afternoon because they just want to finish the deploy.

Source: Meta Engineering Blog

LaunchDarkly

LaunchDarkly (a feature flag platform) defines "feature flags as a safety valve" — the flag's kill switch can disable a feature for all users in under 200ms, faster than a rollback. Their recommended pattern: deploy code to 100%, enable feature flag at 1%, observe for 24h, ramp to 10%, 50%, 100% over 3 days. The deployment is complete on day 1; the feature release takes 3 days.

What we take: Decouple deployment (code goes to production) from release (feature is enabled for users). This separation gives the team independent control over two previously coupled risks. Deploy risks and release risks are different and should be managed separately.

Source: LaunchDarkly Blog

Anatomy

The parts a complete progressive-delivery practice has, with one sentence on each. None are optional.

Ring definition

4 rings: canary (1%), ring 1 (5%), ring 2 (25%), GA (100%). Ring selection criteria can be random traffic sampling, geographic region, or account age. Document the selection criteria before the first deployment — changing it after the canary starts invalidates the comparison.

Automated SLO gates

Three gate checks per ring: error rate ≤ 2× baseline, P99 ≤ 1.5× baseline, SLO burn rate ≤ 1.5× normal. Observation window: 15 minutes for canary, 30 minutes for ring 1, 60 minutes for ring 2. All three must pass within the window or the pipeline halts and rolls back automatically.

Feature flag infrastructure

A centralised flag service (LaunchDarkly, or a home-built variant backed by a fast key-value store) with a kill switch latency under 200ms. Flags are client-evaluated, not server-evaluated — the flag state travels with the request, enabling per-request A/B comparison in the same deployment.

Service tier requirement

Progressive delivery is required for all Tier 0 and Tier 1 services. Tier 2 services may use big-bang deployments if their SLO allows. The tier system drives the requirement — not every service needs a canary, but every user-facing service does.

How it works in this project

Study App runs on Kubernetes and uses Argo Rollouts to drive weighted canary traffic between two ReplicaSets of the same Deployment. A merged PR builds a new image, Argo Rollouts promotes it through the ramp 1% → 5% → 25% → 100% over 30 minutes, and each step pauses for an analysis run before promoting. The analysis template hooks Prometheus: it queries the SLO burn rate and the user-visible error rate over a 5-minute window centred on the canary pods and compares both against the stable baseline. If either metric trips its threshold the rollout aborts and Argo restores the previous version automatically — no human in the loop, no Slack ping waiting on an ACK.

The analysis template is one file per service under infra/rollouts/; the SLO metric it queries is the same one defined in infra/slo/ and consumed by paging rules — see SLO & error budget. Feature flags layer on top of this for the high-risk subset: code ships to 100% of pods on day one through the canary pipeline, the feature flag is enabled at 1% on day two, then ramped on a separate clock. The deployment pipeline does not block on the feature ramp; they are two independent control surfaces.

Deep dive — the theory you need

Canary vs blue/green vs feature flags — the gradient, not the menu

These three strategies are often presented as alternatives. They are not — they sit on a gradient of "what is being progressively exposed", and a mature pipeline uses all three for different risk shapes. Blue/green flips 100% of traffic from one stack to another atomically; it is the safest rollback strategy (instant flip back) but the worst detection strategy (100% of users are exposed at the moment of the flip). Canary inverts that trade-off: detection is excellent because only 1% of users carry the risk, but rollback is slower because the traffic shift has to ramp back down. Feature flags decouple the two — code deploys via canary (good detection), the feature itself enables behind a flag with a sub-second kill switch (best rollback). The three are not competitors; they are layers, each adding a different safety property.

Analysis windows and statistical significance with small canary cohorts

A 1% canary on a service that handles 1000 requests per second sees 600 000 requests per 10-minute window — comfortable sample size for detecting an error-rate shift of 0.5 percentage points at p<0.05. A 1% canary on a service that handles 10 requests per second sees 6 000 requests in the same window, and the statistical power collapses; you cannot tell a real regression from noise. The practical rule is: the canary window must produce enough samples to detect the effect size you care about. Low-traffic services either ramp slower (let the canary soak longer so the sample accumulates), accept larger effect-size thresholds, or skip the 1% step and start at 5%. There is no virtue in a 1% canary that cannot distinguish a 10% error spike from baseline jitter.

Soak time vs ramp speed — the real trade-off

Two failure modes drive the soak/ramp tension. Fast-onset bugs (segfault on a specific request shape, immediate latency spike) show up within minutes — short soaks catch them, and long soaks are wasted clock time. Slow-onset bugs (memory leak, slow disk fill, cache poisoning) only manifest after hours — short soaks let them through, and the bug appears at 25% or 100% where the blast radius is no longer "small". The mature answer is asymmetric: short soaks at 1% and 5% (catch the fast bugs cheaply), longer soaks at 25% (catch the slow bugs before going to GA). Total ramp time of 30–60 minutes for ordinary changes; multi-hour ramps for changes touching stateful systems, cache layers, or memory allocators.

Feature-flag separation vs deployment-flag confusion

Teams routinely conflate two different uses of flags. A deployment flag is a temporary scaffold that gates new code until it has soaked — it should be deleted within a sprint of GA. A feature flag is a long-lived control surface that exists because the business wants to enable, disable, or A/B-test a feature on a different timeline than its code ships. Mixing them produces the worst of both: a "feature" flag that is really a deployment flag accumulates in code forever; a "deployment" flag that is really a feature flag gets removed before the business is ready to disable it. The discipline is to label every flag at creation — kind: deployment or kind: feature — with a TTL on deployment flags and an owner on feature flags. The two kinds live in the same flag service but follow different lifecycle rules.

Trade-offs — what this practice costs

  • Infrastructure cost. Running two ReplicaSets simultaneously (stable + canary) costs roughly +10% capacity during a rollout — the canary pods exist on top of the stable fleet, not instead of it. Worth it, but it is a real line item; small services on tight budgets feel it.
  • Pipeline complexity. A canary pipeline has more failure modes than a recreate-style deploy: analysis runs can be flaky, Prometheus can be slow to return data, the analysis template itself can have bugs that abort healthy deployments. The pipeline becomes a system that needs its own SLOs.
  • Cultural patience. "Deploy in 30 minutes" is slower than "deploy in 3 minutes". Teams used to fast recreate deploys feel the slowdown for a few weeks and then forget about it — but the first weeks are noisy, and leadership has to defend the discipline against "just push it, the test passed" pressure.

Lifecycle

PR merged → canary deployed 1% → SLO gate check → promoted 5% → promoted 25% → 100% or auto-rollback → post-deploy verify

The moment the PR merges, the CI pipeline builds an image and Argo Rollouts kicks off the canary. The first promotion lands at 1% of traffic; the analysis template runs against Prometheus and either advances to 5% or aborts. The same loop repeats at 5% and 25%. If any gate trips, the rollout reverses without human intervention — the stable ReplicaSet scales back up, the canary scales to zero, and a Slack notification names the metric that tripped. On success at 100% the post-deploy verification runs end-to-end smoke tests and confirms the SLO dashboard is green; only then is the rollout marked complete and the deployment flag (if any) cleared for cleanup.

Common pitfalls

PitfallWhy it happensWhat to do instead
Canary that's not comparable to the baseline The 1% canary gets different traffic (e.g., only mobile users) than the 99% baseline. Ensure traffic is randomly sampled for the canary, not biased by any known dimension. Biased sampling invalidates the comparison — the canary might look healthier because it's serving a less demanding traffic slice.
Feature flags that are never cleaned up Flags are easy to add, inconvenient to remove. Set a TTL on every feature flag at creation time. After GA (100% enabled), the flag should be removed from the codebase within one sprint. Flags that stay in code permanently accumulate — old flags make code harder to read and test, and eventually a flag left on is misconfigured and breaks something.
Human ring approvals that are always rubber-stamped "The canary looks fine" without looking at the data. Automate the approval. If the metrics are within bounds, promote automatically. If humans always approve, the gate adds latency without adding safety. Either trust the metrics or improve the metrics — don't add a human approval step that provides neither speed nor quality.
Canary window too short for the traffic volume Default 5-minute soak copied from a high-RPS service onto a low-RPS one. Size the analysis window to the request volume needed for statistical confidence in the effect size you care about. Low-RPS services need longer windows or larger ring percentages; a 1% canary that sees 30 requests cannot tell signal from noise.
Schema or stateful changes shipped through the canary lane Treating database migrations and code changes the same way. Two-phase the stateful change: ship the additive schema migration first (backwards-compatible), let it soak, then ship the code that depends on it through the canary. Rolling back a canary that has already migrated the schema is hard; never put both in the same deploy.
No baseline before the canary starts The analysis compares canary metrics to a snapshot taken at deploy time, which may already be drifting. Capture the baseline over a rolling window (e.g. previous hour) and recompute it during the canary, not once at the start. Static baselines miss diurnal patterns and slow drift; rolling baselines compare apples to apples.

Worked example

Argo Rollouts manifest

lessons-api — weighted canary with Prometheus SLO analysis

apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: lessons-api
  namespace: study-app
spec:
  replicas: 20
  strategy:
    canary:
      canaryService: lessons-api-canary
      stableService: lessons-api-stable
      trafficRouting:
        istio:
          virtualService: { name: lessons-api }
      steps:
        - setWeight: 1
        - pause: { duration: 5m }
        - analysis:
            templates: [{ templateName: slo-burn-and-errors }]
            args:
              - { name: service, value: lessons-api }
              - { name: canary-hash, valueFrom: { podTemplateHashValue: Latest } }
        - setWeight: 5
        - pause: { duration: 10m }
        - analysis:
            templates: [{ templateName: slo-burn-and-errors }]
            args: [{ name: service, value: lessons-api }]
        - setWeight: 25
        - pause: { duration: 15m }
        - analysis:
            templates: [{ templateName: slo-burn-and-errors }]
            args: [{ name: service, value: lessons-api }]
---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata:
  name: slo-burn-and-errors
spec:
  args:
    - name: service
  metrics:
    - name: slo-burn-rate
      interval:        1m
      successCondition: result[0] < 1.5
      failureLimit:    0
      provider:
        prometheus:
          address: http://prometheus.monitoring:9090
          query: |
            job:slo_error_rate:rate5m{service="{{args.service}}"}
              / on() group_left() job:slo_error_rate:steady_state{service="{{args.service}}"}
    - name: canary-error-rate
      interval:        1m
      successCondition: result[0] < 0.02
      failureLimit:    0
      provider:
        prometheus:
          address: http://prometheus.monitoring:9090
          query: |
            sum(rate(http_requests_total{service="{{args.service}}",version="canary",code=~"5.."}[5m]))
              / sum(rate(http_requests_total{service="{{args.service}}",version="canary"}[5m]))

What this manifest does. Argo Rollouts promotes the lessons-api deployment through 1% → 5% → 25% → 100% with a pause and analysis run between each weight. The analysis template runs two Prometheus queries every minute: the SLO burn rate (must stay under 1.5× steady-state) and the canary-only error rate (must stay under 2%). A single failed sample (failureLimit: 0) aborts the rollout and Argo restores the stable ReplicaSet. If both metrics stay green for the pause window, the rollout advances; if either trips, the canary is gone before the on-call's phone vibrates.

Quality bar — what good looks like

A checklist of what a healthy implementation has. ✓ done ◐ partial ○ todo

  • ○

    All Tier 0/1 services on weighted canary

    Argo Rollouts (or equivalent) drives every user-facing deployment; recreate strategy is forbidden for Tier 0/1.

  • ○

    Ramp 1% → 5% → 25% → 100% with pauses

    Total ramp time 30 minutes for ordinary changes; longer soaks for stateful or memory-sensitive services.

  • ○

    Automated analysis at every step

    SLO burn rate and canary-only error rate queried from Prometheus on a 1-minute interval; failure aborts the rollout.

  • ○

    Auto-rollback on gate failure

    No human approval required to roll back; Argo restores the stable ReplicaSet within seconds of the failed sample.

  • ○

    Feature flag service with <200ms kill switch

    Decoupled from deployment; flag state travels with the request; deployment-kind and feature-kind flags labelled at creation.

  • ○

    Flag TTL enforced

    Deployment flags removed within one sprint of GA; feature flags carry an owner and a review-by date.

  • ○

    Stateful changes two-phased

    Schema migrations ship as backwards-compatible additive steps first; code that depends on them ships through the canary in a second deploy.
