Rollback playbook — the fastest path from broken to healthy

A per-service, pre-written procedure for reverting a production deployment to the previous known-good state — tested before an incident, executable in under 5 minutes, available to any on-call engineer without tribal knowledge.

TL;DR

A rollback playbook is a pre-written, tested procedure for reverting a production deployment to the last known-good state. It documents the rollback mechanism (git revert + redeploy, image tag rollback, Kubernetes rollout undo), the database migration strategy (backward-compatible schema changes only; migration rollback procedure if schema change was part of the deployment), the feature flag kill switch (for changes deployed with flags), and the verification checklist (which metrics must return to baseline before the rollback is considered complete). The playbook is tested quarterly in game days. Target: rollback to known-good within 5 minutes of decision to rollback, for any on-call engineer who may not have been the deployer.

What it is

A rollback playbook is a pre-written, tested procedure for reverting a production deployment to the previous known-good state. Each service has its own playbook, covering four areas:

  • Rollback mechanism: The specific command or UI action to revert to the previous version. For Kubernetes services: kubectl rollout undo deployment/[service-name]. For services using image tag promotion: deploy --tag [previous-image-tag]. For feature-flagged changes: the flag kill switch command. The exact command is documented — not "run the rollback script" but the actual command with all parameters.
  • Database migration strategy: If the deployment included a schema change, the rollback procedure for the schema. The standard is backward-compatible migrations only — new columns are nullable by default, old columns are not dropped until the next sprint. If a breaking schema change was deployed and can't be reverted safely, the playbook documents the escalation path.
  • Rollback decision criteria: When to rollback vs. fix-forward. The decision rule: if the estimated MTTR of a fix-forward exceeds 10 minutes, rollback. The deployer makes this decision within 2 minutes of detecting degradation — not after trying to diagnose the root cause.
  • Verification checklist: The specific signals that must return to baseline before the rollback is declared complete: error rate within normal range, P99 latency within 1.2× baseline, synthetic probe green for 5 consecutive minutes.

Why we need it

The most common reason rollbacks take too long during an incident is the time spent deciding whether to rollback and then finding the correct command. Engineers who have never executed a rollback for a specific service often don't know what "rollback" means in that context — is it a kubectl command? A CI pipeline rerun? A database migration? Without a playbook, the first 10 minutes of a deployment incident are spent on discovery that should have been done before the deployment.

Rollback playbooks also change the engineering culture around rollbacks. When rollback is well-documented, tested, and takes 5 minutes, teams are more willing to make the rollback call early — before the impact compounds. When rollback is unclear or risky, teams delay the decision and attempt fix-forwards that make the incident longer. The playbook makes the safe choice the easy choice.

How others do it

Etsy

Etsy's engineering culture famously includes a "deploy and rollback button" — a physical button in the engineering office (and later a Slack command) that rolls back the most recent deployment. Their philosophy: rollback should be so easy that it's the default response to any post-deploy anomaly. The decision overhead ("should we rollback?") is the primary bottleneck, not the execution. By making rollback a single-action operation, they moved the cultural default from "fix forward" to "rollback first, investigate later."

What we take: The deployment system should expose a one-click rollback that requires no knowledge of the deployment mechanism. The on-call engineer presses a button; the system handles the mechanics. The playbook explains what the button does, not how to do it manually.

Source: Etsy Code as Craft

Amazon

Amazon's deployment practice for AWS services includes a "two-pizza team owns deployment and rollback" principle — the team that deploys owns the rollback. There is no separate "ops team" that handles rollbacks. Every team tests their rollback procedure monthly in a game day exercise: deploy a known-bad version, observe the health gates trigger, execute the rollback, and time the procedure from alert to verification. Teams that can't complete rollback in under 5 minutes are not allowed to deploy to production the following week until the procedure is fixed.

What we take: Rollback must be a team practice, not individual knowledge. Monthly rollback drills are the only way to ensure the procedure works and the whole team can execute it — not just the senior engineer who wrote it.

Source: Amazon Builders' Library

Anatomy

Rollback command documentation

The exact command or action to revert the service, per deployment mechanism. Format: the command itself, pre-filled with the service name and any required parameters. Not a description of the command — the command itself. The on-call engineer should be able to copy and paste without editing. Multiple mechanisms documented if the service has different rollback paths for code vs. config vs. schema.

Feature flag kill switch

For any deployment that uses feature flags to gate the change, the kill switch command is documented in the playbook: the flag service URL, the flag name, and the command to disable it. Kill switch latency target: under 200ms from command to all instances receiving the update. Faster than a rollback for high-risk feature releases.

Database migration safety

Schema changes must be backward-compatible. The standard: new columns are nullable and have a default; old columns are retained for at least one release cycle after being removed from code; index changes are additive (never drop an index in the same deployment that adds a new one). If a schema change cannot be made backward-compatible, the playbook escalates to the database team before deployment, not after rollback fails.

Quarterly rollback drill

Once per quarter, each team runs a rollback drill: deploy a known-bad canary to staging, let the health gate fire, and execute the playbook. Time from gate failure to verification complete. If any step fails or the total time exceeds 5 minutes, the playbook is updated before the next production deployment. Drills are not optional for Tier 0 and Tier 1 services.

How it works in this project

Study App's rollback path is documented in internal/sre/rollback.md — one source of truth, linked from every service runbook so the on-call never improvises the command. The standard mechanism is a single-command Argo CD rollback to the previous image: argocd app rollback <service> <rev> reverts the workload to the last green deployment without touching the cluster manifests, while kubectl rollout undo stays available as a break-glass for clusters where Argo is itself the suspect.

Database migrations follow the expand/contract pattern (also called parallel-change): every schema change ships in two phases — an additive "expand" step that is forward- and backward-compatible with both code versions, then a later "contract" step that removes the old shape only after the new code has fully bedded in. The binary can therefore roll forward or back across one release without the schema needing to move with it. The contract phase is never bundled with new code; it is a separate, low-risk deployment after the previous release has soaked. Target rollback time is ≤ 5 minutes from decision to SLO-recovered, measured from the timestamp on the rollback command to the first five-minute window of green synthetic probes.

Deep dive — the theory you need

Forward-compatible migrations — the contract that makes rollback possible

A rollback that takes the schema with it is not a rollback — it is a second deployment under incident pressure, with all the risk that implies. The discipline that makes rollback safe is that the schema only ever moves forward, and only in shapes the previous code version can still read and write. The textbook formulation is the expand/contract pattern (Pramod Sadalage and Martin Fowler's evolutionary database design): a schema change is decomposed into an expand step that adds the new shape alongside the old, a transitional period during which both shapes are valid, and a contract step that removes the old shape only after every code version that reads it is out of production.

Double-write — the bridge across the transition

Where data shape is changing (renamed column, new table, denormalised cache), the transitional code double-writes: every mutation lands in both the old shape and the new shape until the backfill is complete and the new code is reading from the new shape. The old shape stays writable so that a rollback to the previous binary continues to work without data loss. Double-writing is removed only as part of the contract phase, in its own deployment, once the new shape has been the source of truth for at least one release cycle.

Data-shape compatibility across N±1 versions

The invariant is that the schema at any point in time is readable and writable by the previous release (N−1), the current release (N), and the next release (N+1). That gives every deployment exactly one safe rollback target and one safe roll-forward target. The N±1 window is the smallest one that keeps shipping practical — wider windows (N±2) double the bookkeeping for marginal extra safety; narrower windows (N only) break the moment two services deploy out of order. Every PR that touches the schema is reviewed against the N±1 rule explicitly.

When rollback is impossible — one-way data migrations

Some migrations cannot be rolled back. Destructive operations — dropping a column, rewriting data in place, converting a denormalised cache into the source of truth — leave the previous binary unable to function. The honest answer is to flag these migrations before they ship: the playbook calls out "one-way" status, a forward-fix plan is written in advance, and the deployment carries a heightened canary period and a separate go/no-go gate. Pretending a one-way change is rollback-safe is the failure mode that turns a 5-minute incident into a 5-hour one.

Rollback vs roll-forward — the decision rule

Rollback is preferred when the estimated MTTR of a fix-forward exceeds 10 minutes, when the change is rollback-safe (no one-way migration, no destructive data step), and when the symptom is clearly correlated with the deployment. Roll-forward is preferred when the rollback target is itself known-bad (the previous release also had a bug), when the change cannot be rolled back without data loss, or when a one-line fix is already staged and tested. The decision is made within two minutes of detecting degradation, by the deployer or on-call, and recorded in the incident channel.

Trade-offs — what this practice costs

  • Two-step schema changes. Expand/contract doubles the number of deployments for any schema change. The win is the rollback safety net; the cost is calendar time and reviewer attention on every migration.
  • Double-write code in the transitional period. The application carries logic that writes to both shapes until the contract phase. It is dead code the moment the migration finishes, and forgetting to delete it leaves the application slower and the schema more confusing for the next migration.
  • Drill cost. Quarterly rollback drills consume an engineer-day per service. The drill cost is the only thing that proves the playbook still works — services that skip drills discover the playbook is stale during the first real incident.

Lifecycle

regression detected → rollback triggered → deploy freeze → revert → verify SLO recovery → comms posted → postmortem

A regression surfaces — a health gate trips, an alert pages, a synthetic probe goes red. Within two minutes the deployer or on-call calls the rollback; further deploys to the affected service are frozen from the pipeline so a concurrent merge cannot land mid-revert. The Argo CD rollback runs against the previous green revision; the verify step waits for error rate, P99 latency, and synthetic probes to sit inside baseline for five consecutive minutes before declaring recovery. Comms go out on the status page and the incident channel — even a 5-minute event gets an entry — and a postmortem is scheduled within 24 hours regardless of duration. The postmortem is the only output that prevents the same rollback from recurring.

Common pitfalls

PitfallWhy it happensWhat to do instead
Rollback playbook not tested before the deployment it's meant to protect "We'll test it in the next drill." The rollback playbook for a new deployment is tested in staging before the deployment goes to production. A playbook that has never been executed is a theory, not a procedure. The acceptance criterion for any deployment is a successful rollback test in staging.
Fix-forward attempted for too long before rollback decision "We're so close to a fix — just 10 more minutes." Set a hard time limit: if the service has been degraded for 10 minutes and a fix-forward is not in production, rollback. The limit is decided in advance and documented in the playbook — not negotiated during the incident. A 10-minute limit prevents the "almost there" cognitive trap from extending a P0 to 45 minutes.
Rollback fails because of a schema change that can't be reverted A breaking schema change was deployed in the same deployment as the code change. Separate schema changes from code changes. Schema migrations deploy first, with a validation period. Code changes that depend on the schema deploy in the next release. If they must ship together, the schema change must be backward-compatible with the previous code version.

Worked example

Rollback transcript

Lessons API — bad image v1.42.0 rolled back to v1.41.3 in 4 m 12 s

# 14:02:11  Synthetic probe + SLO burn alert fire on lessons-api after v1.42.0 rollout
# 14:02:48  On-call decides rollback (fix-forward MTTR estimate > 10 min)
# 14:03:02  Freeze new deploys to lessons-api
$ argocd app set lessons-api --sync-policy none
$ argocd app actions run lessons-api freeze --kind Rollout

# 14:03:30  Confirm previous green revision
$ argocd app history lessons-api | head -5
ID  DATE                      REVISION   SOURCE
42  2026-05-24T13:58:11Z      v1.42.0    git@studyapp:lessons-api.git
41  2026-05-24T11:14:02Z      v1.41.3    git@studyapp:lessons-api.git  (last green)

# 14:03:55  Revert to previous image via Argo CD
$ argocd app rollback lessons-api 41
rollback to revision 41 (v1.41.3) initiated...

# 14:04:20  Break-glass equivalent if Argo CD itself were the suspect:
# $ kubectl -n lessons rollout undo deployment/lessons-api

# 14:04:48  Verify SLO recovery
$ kubectl -n lessons get pods -l app=lessons-api
NAME                            READY   STATUS    RESTARTS   AGE
lessons-api-5f8c7d9b4-7q2xk     1/1     Running   0          22s
lessons-api-5f8c7d9b4-9k4mp     1/1     Running   0          25s

$ promtool query instant http://prometheus 'rate(http_requests_total{service="lessons-api",code=~"5.."}[1m])'
0.0007  # baseline range 0.0005–0.0010

$ promtool query instant http://prometheus 'histogram_quantile(0.99, rate(http_request_duration_seconds_bucket{service="lessons-api"}[5m]))'
0.183   # baseline ~0.170, within 1.2× threshold

# 14:06:23  Synthetic probe green for 5 consecutive minutes — rollback declared complete

# 14:06:30  Comms posted (status page + #incidents)
#status-page: "lessons-api degraded 14:02–14:06 UTC; reverted to v1.41.3; investigating root cause"
#incidents:   "INC-2871 resolved at 14:06 UTC. Rollback took 4 m 12 s. Postmortem scheduled
                tomorrow 10:00 UTC. v1.42.0 quarantined pending fix."

What this transcript shows. The decision happens fast (37 seconds from alert to rollback call), the freeze closes the pipeline before the revert lands, the rollback itself is a single Argo CD command against an explicit previous revision, and verification waits for both the error-rate query and the latency quantile to sit inside baseline before declaring recovery. Comms go out the moment recovery is declared — not at the end of the postmortem — so users and downstream services see a closed status page within minutes. Total measured rollback time: 4 minutes 12 seconds, inside the 5-minute target.
