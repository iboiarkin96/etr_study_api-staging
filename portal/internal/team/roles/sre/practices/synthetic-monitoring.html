Synthetic monitoring — detect outages before users do

Scheduled scripts that drive the golden user journeys from multiple regions, every minute, regardless of real traffic. The first signal of an outage should be the probe failing, not a user complaint on social media.

TL;DR

Synthetic monitoring runs scheduled k6 or browser scripts that drive the three most critical user journeys (signup, lesson-render, payment) every 60 seconds from three geographic regions. Unlike passive monitoring — which only fires when real users hit errors — synthetics detect outages in low-traffic regions, CDN/DNS/TLS failures, and full-journey regressions before users feel them. Failures wire into PagerDuty at P2 and contribute to the journey-level SLO as synthetic SLIs. Alert only on two consecutive failures from the same region; never page on a single probe blip.

What it is

Synthetic monitoring is the discipline of running scripted, scheduled test scenarios against production from multiple geographic locations, treating their pass/fail rate as a first-class availability signal. Where passive monitoring observes whatever traffic real users generate, synthetics generate traffic the team chooses — the journeys that matter most — so the absence of real users does not also mean the absence of a signal. A synthetic SLI is a deliberate sample, not a hope that the right users will hit the right endpoint at the right time.

A rule that ships to production must satisfy five properties: it drives a complete user journey end-to-end and not a single endpoint; it runs on a schedule measured in minutes, not hours; it executes from multiple regions that mirror the real user distribution; it emits both a success bit and per-step latency as metrics; and it routes to a paging tier with a 2-of-N consecutive-failure guard. Anything missing one of the five is a smoke test, not a synthetic monitor — useful in CI but not load-bearing in production.

Why we need it

Passive monitoring has a structural limit: it needs real user traffic to detect failures. In a low-traffic region — APAC at 3 a.m., a new market with a hundred users — a complete outage may never generate enough error volume to breach a threshold alert. The first signal arrives as a customer tweet, hours after the incident started, and the post-mortem records the same finding every time: we had no monitoring that did not depend on a user being online. Synthetics close that gap by generating the traffic themselves.

Synthetics also catch a class of failures passive monitoring is blind to by construction. A misconfigured CDN, a DNS record pointed at the wrong origin, a TLS certificate that expired at midnight — these stop users from reaching the service at all, so they produce zero events in the application monitoring stack. A synthetic probe trying to complete the full journey sees the failure immediately. That single capability is why every published SRE practice keeps synthetics in the stack even after a mature SLO programme is in place: they cover the journeys, the regions, and the network layers that passive metrics structurally cannot.

How others do it

Short tour of teams whose synthetic-monitoring practice is published and copied. Each card is one organisation, what they do, and what we take from it.

Cloudflare

Cloudflare runs synthetic probes from their global PoP network — 300+ locations — and decomposes every probe into per-phase metrics: DNS resolution, TCP connect, TLS handshake, time-to-first-byte, full page load. When a probe fails, the phase breakdown tells the on-call exactly where in the network stack the failure happened, with no further investigation needed. The fanout also means geographic degradation is visible at a granularity a three-region setup cannot match.

What we take: Break every synthetic check into phases and emit each as its own metric. The phase view is what turns a "the journey failed" page into a "DNS resolution timed out in APAC" runbook entry — without it the on-call still has to rediscover where in the stack the failure lives.

Source: Cloudflare DEX Monitoring

New Relic

New Relic's synthetic monitoring runs two layers side by side: lightweight API probes that fire every minute as the fast-feedback alerting tier, and full scripted-browser journeys that run every five minutes as the SLI input. The browser tests render JavaScript and interact with the DOM, so they catch frontend regressions the API probe never sees; the API probes catch backend outages before the slower browser test completes its cycle. Alerts require two-of-N consecutive failures — never one.

What we take: Run synthetics in two layers, not one. API probes for fast paging and uptime; full browser journeys for SLO and regression coverage. The two-of-N consecutive guard removes probe-level flakiness without adding latency to genuine outages.

Source: New Relic Synthetic Monitoring

Datadog

Datadog Synthetics binds every synthetic test to an SLO definition so test failures consume the same error budget that real-user errors do. Test runs are SLI events: a failing probe in APAC at 3 a.m. burns budget exactly as a real user's failed request would. The platform exposes the failure as a journey-level SLI rather than as a separate "synthetic uptime" metric — there is one availability number per journey, computed from real plus synthetic events.

What we take: Synthetic results feed the journey SLO, not a separate vanity number. One availability metric per journey, real and synthetic events combined, with the synthetic contribution dominating the low-traffic windows where real users are absent.

Source: Datadog Synthetic Monitoring

Grafana Labs

Grafana Synthetic Monitoring (the hosted service backing k6 probes) treats the probe schedule, region set, and assertion list as code in a Git-backed configuration repo. Adding a probe is a pull request, not a UI click; the review covers the journey choice, the region selection, and the alert threshold together. Probes that fire and produce no action are flagged in the monthly synthetic-quality review and either retuned or retired.

What we take: Probes live in a Git repo next to the alert rules. The journey, schedule, regions, assertions, and paging policy are all under the same review gate — so the practice scales the same way alert rules do.

Source: Grafana Synthetic Monitoring

Across every team whose synthetic practice is public, the playbook converges: probe the golden journeys end-to-end, run from multiple regions every minute, decompose into per-phase metrics, feed the journey SLO directly, and require two-of-N consecutive failures before paging.

Anatomy

The parts a complete synthetic-monitoring practice has, with one sentence on each. None are optional.

Golden journey selection

Three to five journeys that together cover >80% of the revenue path — for Study App: signup, lesson-render, payment. Endpoints are not journeys; the journey is the user's intent, scripted end-to-end.

Multi-region probe network

At least three regions chosen to mirror the real user distribution. If 30% of users are in APAC, one probe is in APAC. A failure invisible from the EU probe is still a real outage for those users.

Probe schedule

Every 60 seconds for API probes, every 5 minutes for browser journeys. Slower schedules mean the outage runs longer before the probe trips; faster schedules add cost without adding signal.

Two-of-N consecutive guard

Alert only after two consecutive failures from the same region. Single failures are almost always probe-level network noise; treating them as pages destroys on-call signal-to-noise immediately.

Per-phase metrics

Decompose every probe into DNS, TCP connect, TLS, time-to-first-byte, full journey latency. Each phase is its own metric so the on-call can read the runbook entry instead of starting an investigation from zero.

Synthetic SLI contribution

Test results feed the journey-level SLO directly. Every successful probe is a good event; every failure burns budget. There is one availability number per journey, not a separate synthetic vanity metric.

Probes-as-code

Journey, schedule, regions, assertions, and routing live in a Git repo reviewed the same way alert rules are. Adding a probe is a PR; CI checks the journey owner, the routing labels, and the runbook URL.

Dedicated test credentials

Synthetic accounts have their own emails, tokens, and rotation schedule. Never share with developer logins — a password change must not silently break every probe at once.

How it works in this project

Study App will run its synthetic probes through k6 Cloud (HTTP and scripted-browser) with Grafana Synthetic Monitoring as the hosted probe network — both are k6-native, so the journey scripts are the same artefact regardless of where they execute. Probes live at infra/synthetics/ with one script per golden journey; the schedule, regions, and assertion thresholds are declared alongside in journey.yaml and reviewed in the same PR. CI checks that every probe has an owning team, a runbook URL, a routing label, and a review_by date no further than 90 days out.

The first generation of probes covers the three revenue-critical journeys: signup (anonymous → account created → first lesson visible), lesson-render (sign-in → list courses → open lesson → assert content within p95 budget), and payment (sign-in → checkout → confirmation). Each runs every 60 seconds from three regions (us-east, eu-west, ap-south) to match the user distribution. Failures route to PagerDuty at P2 with a two-consecutive-failures-from-the-same-region guard; per-phase metrics land in Prometheus and feed the journey SLO defined in infra/slo/ — see SLO & error budget for how the SLI events are combined.

Deep dive — the theory you need

Why synthetics exist — the gap they close

Every monitoring stack has a coverage matrix indexed by two axes: the signal source (real-user traffic vs scripted traffic) and the fault domain (application code, network path, third-party dependency, geographic region). Passive monitoring covers most of the application-code column well — if a real user is generating traffic. It covers the network-path column poorly, because most CDN/DNS/TLS failures prevent the user from reaching the application monitoring at all. And it covers low-traffic geographic regions essentially not at all, because there is no traffic for the metrics to be computed from. Synthetics fill those cells: scripted traffic, every region, every minute, full network path included.

Golden journey selection — the 80/20 rule applied honestly

The mistake every team makes on the first iteration is treating "synthetic monitoring" as "monitor every endpoint we have". The right unit is the journey: the user's intent end-to-end. For Study App, three journeys cover >80% of the revenue path — signup, lesson-render, payment. Adding a fourth produces marginal coverage at a real maintenance cost (every probe is one more script to keep in sync with the UI). The honest test of a candidate journey is to ask: if this journey is broken for an hour, does the business lose money? If yes, it earns a probe; if no, the existing application-level metrics are enough.

Check types — HTTP probe, scripted browser, API contract

Three layers of synthetic check exist, each with a different cost and coverage profile. The cheapest is the HTTP probe: a k6 script that hits an endpoint and asserts status, latency, and body shape. It catches backend outages in seconds and costs cents per region per day. Above it sits the scripted browser probe: a full headless browser that renders JavaScript, interacts with the DOM, and follows redirects — the only check that catches frontend regressions, broken CDN paths, and CSP misconfigurations. It is 10–30× the cost of an HTTP probe. The third layer is the API contract probe: a synthetic that exercises the public API surface against the published OpenAPI/AsyncAPI contract, catching contract drift the application monitoring would not detect.

Multi-region geography — why three is the minimum

A single-region probe network has a single point of failure: the probe's own region. If the AWS region hosting the probe degrades, every probe fails and the on-call gets paged for an incident that does not exist. Two regions is better but still fragile — one region down looks like a service outage. Three regions provides quorum: if two of three agree the journey is failing, it is failing. The region choice is not symmetric — pick regions that mirror the real user distribution (one in each of the top-three user geographies) so a regional degradation that hurts users is also visible to a probe nearby.

Blind spots — what synthetics structurally cannot see

Synthetics are not a replacement for real-user monitoring. They cannot see what real users see, by construction: their device mix is uniform (always the same headless Chrome), their network conditions are clean (datacentre uplink, not a flaky 4G connection), their account is the same dedicated test user every time. A regression that only manifests on a specific device class, or on a slow network, or on an account in a specific state, is invisible to a synthetic and visible only in real-user metrics (RUM). The right mental model is complement, not replace: synthetics cover availability and journey health; RUM covers experience quality and long-tail device diagnostics.

Trade-offs — what this practice costs

  • Maintenance burden. Every UI change to the golden journey is a synthetic-script change. Without a "synthetic update required" entry in the feature DoD, synthetics drift out of date within two sprints and start firing for the wrong reasons.
  • Test-account hygiene. Synthetic accounts need their own credentials, their own rotation schedule, and their own access policy. Sharing them with developer logins guarantees a password change will silently break every probe at once.
  • Probe cost. Browser-based probes from three regions at five-minute intervals cost real money on hosted platforms (k6 Cloud, Grafana SM, Datadog). The cost is bounded — but it is not zero, and the practice does not scale to one probe per endpoint without a budget conversation.
  • False reassurance. A green probe is not a green service; it is one journey passing for one synthetic user. Teams that conflate the two end up paged less but surprised more. The probe is the floor of availability monitoring, not the ceiling.

Lifecycle

journey selected → script + assertions written → test creds provisioned → PR + CI checks → soak in staging → prod, multi-region → quarterly review · retune · retire

A new probe starts as a journey statement, not a script — "the user signs up, opens a lesson, and sees content within 3 s" rather than "GET /api/lesson". The team agrees the journey earns a probe (revenue-critical, owner identified, runbook drafted) and only then does the k6 script get written, with per-phase assertions and an explicit latency budget. Dedicated synthetic credentials are provisioned with their own rotation policy. CI validates the runbook URL, the owner labels, the routing tier, and the review_by date. The probe soaks in staging for one full business cycle — long enough to see weekend traffic patterns — and only then ships to production across three regions. After ninety days, or sooner if the probe fires noisily without surfacing real issues, it is reviewed: retuned or retired. Retired probes are deleted, not commented out.

Common pitfalls

PitfallWhy it happensWhat to do instead
Probing every endpoint instead of golden journeys "Synthetic monitoring" is misread as "monitor everything"; endpoints are easier to enumerate than journeys. Pick three to five revenue-critical journeys and probe them end-to-end. Endpoint health is what application metrics are for; the synthetic is the user's intent, not the API surface. A team probing every endpoint ends up maintaining fifty scripts and trusting none of them.
Alerting on single failures "One failure means it is down" — and a script that hit a one-off network blip looks like an outage. Require two consecutive failures from the same region before paging. Probe-level network noise is constant; pages on single failures train the on-call to swipe-dismiss within a week and the real outage gets the same dismissal.
Shared developer credentials in the probe "We have a test account, let's just use it" — the same one the developers log in with. Use dedicated synthetic accounts with synthetic-specific email addresses and tokens, on a separate rotation schedule. A developer password reset must not silently break every probe at once. Treat the synthetic credential as a service account, not a shared login.
Single-region probe network "One probe is enough to see if it's up" — cheaper, and looks like quorum. Run from at least three regions chosen to mirror the user distribution. One region looks like quorum until the region itself degrades and every probe fails for a non-incident. Two regions cannot tell a regional issue from a service outage. Three gives quorum.
Probe drift — UI changed, script did not The login flow gets a new step; the synthetic still expects the old form. Nobody updates the probe with the feature PR. Make synthetic-script update part of the DoD for any user-facing change to a golden journey. Assign each probe a named owner. A probe that always fails because the UI moved is worse than no probe — it teaches the on-call the alert is meaningless.
Synthetic as a separate vanity metric The probe reports its own uptime number; the service reports a different SLO; the two never reconcile. Feed synthetic results into the journey-level SLO directly. One availability number per journey, real and synthetic events combined. The synthetic dominates the low-traffic windows; real users dominate the busy ones. A separate "synthetic uptime" dashboard is a smell.

Worked example

k6 probe

SYN-002 — Lesson-render golden journey (sign-in → list courses → open lesson → assert content within p95 budget)

import http from 'k6/http';
import { check, group, sleep } from 'k6';
import { Trend, Rate } from 'k6/metrics';

export const options = {
  scenarios: {
    lesson_render: {
      executor:    'constant-vus',
      vus:         1,
      duration:    '30s',
      exec:        'lessonRender',
    },
  },
  thresholds: {
    'checks{journey:lesson_render}':       ['rate>0.99'],
    'http_req_duration{step:open_lesson}': ['p(95)<1500'],
  },
  tags: { journey: 'lesson_render', owner: 'sre-platform', tier: 'P2' },
};

const lessonStepLatency = new Trend('synthetic_step_ms', true);
const lessonStepSuccess = new Rate('synthetic_step_ok');

export function lessonRender() {
  const base   = __ENV.BASE_URL || 'https://app.studyapp/';
  const email  = __ENV.SYN_EMAIL;
  const token  = __ENV.SYN_TOKEN;

  group('sign_in', () => {
    const r = http.post(`${base}api/auth/login`,
      JSON.stringify({ email, token }),
      { headers: { 'content-type': 'application/json' }, tags: { step: 'sign_in' } });
    lessonStepLatency.add(r.timings.duration, { step: 'sign_in' });
    lessonStepSuccess.add(check(r, { 'signed in':  res => res.status === 200 }),
      { step: 'sign_in' });
  });

  group('list_courses', () => {
    const r = http.get(`${base}api/courses`, { tags: { step: 'list_courses' } });
    lessonStepLatency.add(r.timings.duration, { step: 'list_courses' });
    lessonStepSuccess.add(check(r, {
      'courses 200':       res => res.status === 200,
      'courses non-empty': res => (res.json('items') || []).length > 0,
    }), { step: 'list_courses' });
  });

  group('open_lesson', () => {
    const r = http.get(`${base}api/lessons/canary-001`, { tags: { step: 'open_lesson' } });
    lessonStepLatency.add(r.timings.duration, { step: 'open_lesson' });
    lessonStepSuccess.add(check(r, {
      'lesson 200':       res => res.status === 200,
      'lesson under 1.5s': res => res.timings.duration < 1500,
      'lesson has body':  res => (res.json('content') || '').length > 0,
    }), { step: 'open_lesson' });
  });

  sleep(1);
}

What this probe does. Every 60 seconds, from each of us-east, eu-west, and ap-south, k6 runs the three-step journey and emits per-step latency and success metrics tagged with the step name. The thresholds block fails the run if the per-step success rate drops below 99% or the open-lesson p95 exceeds 1.5 s. Alertmanager pages PagerDuty at P2 only after two consecutive failed runs from the same region — a single regional blip clears itself before the page fires. If the probe had been a single GET /healthz instead, the lesson-render outage on 2026-04-12 — which left every health endpoint green while the lesson body returned empty — would have been first reported by a customer in APAC, not by the on-call.
