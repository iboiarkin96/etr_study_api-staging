QA onboarding — 60-day journey

Your first 60 days as a QA on Study App, mapped to concrete reads and concrete deliverables. Each milestone has a time horizon, a short description, the canonical pages to read, and one outcome — the thing you will be able to do when you have finished the step.

This page picks up where the Tester onboarding tutorial ends. The tutorial gives you the QA loop in a single week; this ramp scales that loop into ownership — by Day 60 you will have signed off a release independently. That is the bar.

The journey · six milestones

  1. 01
    Week 1 · the loop

    Run the QA loop end-to-end once

    Your entire Week 1 deliverable is the Tester onboarding tutorial. It walks you, hands-on, through access → local setup → foundations skim → first checklist → first defect filed → first verification. Use the two reference links below if you get stuck on access or environments; everything else is covered inside the tutorial itself.

    • Tester onboarding tutorial — primary deliverable for this week
    • Local development — fallback if setup hits friction
    • Test environments — fallback for staging access

    Outcome. You have completed the tutorial: you ran the full QA loop once — file, verify, sign — on real artefacts.

  2. 02
    Week 2 · foundations, with intent

    Re-read the why, now that you've seen the loop

    You skimmed these in the tutorial. Now read them properly, with the loop in mind: what we test, what we don't, and why. The pyramid tells you which layer owns which kind of test; the QA process tells you how a feature flows from spec to sign-off; the glossary disambiguates words you will hear in every meeting. Write down two questions per page and bring them to your buddy.

    • Test strategy
    • Test pyramid
    • QA process & SDLC
    • Glossary

    Outcome. You can explain in five minutes what the project tests, what it does not, who owns each layer, and why the pyramid is shaped that way for this codebase.

  3. 03
    Weeks 2–3 · the way around

    Learn how defects flow

    Sit next to a senior tester for two pair-sessions: one release smoke, one regression run. Read the lifecycle and the severity matrix carefully — they decide how every defect you file is triaged. Open three random tickets from last sprint and trace each from filing to closure.

    • Defect lifecycle
    • Severity & priority matrix
    • Practices radar (browse Group D · operations)

    Outcome. Given any open defect, you can name its current state, its severity, and the next person on the hook.

  4. 04
    Weeks 3–4 · first owned ticket

    Own one test case from spec to closure

    In the tutorial you filed a defect. Now own a whole test case: pick one low-risk item from the backlog (your buddy will point you at one), design it from the spec using the template, execute it against staging, attach evidence, sign it off, and if it produced a defect — drive the defect through its full lifecycle until it is verified. The deliverable is closure, not the filing.

    • Test case template
    • Bug report template
    • Exploratory charter (optional, if the case is open-ended)

    Outcome. One backlog item you designed, executed, and drove to closure — including any defect it produced, signed off as verified in your name.

  5. 05
    Weeks 4–6 · first playbook

    Own one full playbook + checklist run

    Pick a real feature shipping this sprint and run a complete playbook + checklist against it — either the API endpoint playbook (with the API endpoint acceptance checklist) or the documentation testing playbook (with checklists 0001 + 0005, for visual + a11y). Attach evidence, post the result in the ticket.

    • API endpoint playbook
    • Documentation testing playbook
    • Accessibility playbook

    Outcome. You have run one playbook + checklist end-to-end on a real shipping feature, with evidence on the ticket.

  6. 06
    Day 60 · sign-off

    Own your first release smoke

    You are now the named QA on a release cut. Run the release smoke playbook, work the checklist line by line, file any new defects, and post the sign-off report in the release ticket. The buddy stays available but does not click. When you are done — that is the end of onboarding.

    • Release smoke playbook
    • Release smoke checklist
    • Regression checklist

    Outcome. You have signed off a release independently. You are now a fully ramped QA on Study App.
