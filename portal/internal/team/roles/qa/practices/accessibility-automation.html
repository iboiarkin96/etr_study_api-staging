Accessibility test automation — a11y in CI

Runs axe-core on every CI build to catch WCAG 2.1 AA violations before they reach users; ~30–40% of failures caught automatically, the rest by guided manual review.

TL;DR

Accessibility test automation combines axe-core (Deque, OSS rule engine) with keyboard + screen-reader checks against WCAG 2.1 Level AA. axe-core runs in CI and catches ~30–40% of WCAG failures automatically: missing alt text, duplicate IDs, low colour contrast, missing form labels. Critical violations fail the build; serious raise warnings. The remaining 60–70% require manual verification — keyboard-only traversal, screen-reader smoke test, content-flow review. Per-page Lighthouse-CI a11y score acts as the SLA-style gate.

What it is

Accessibility test automation enforces WCAG 2.1 Level AA on every CI build through three complementary layers:

  • axe-core in CI — Deque's OSS rule engine; runs as a Jest/Playwright/Cypress plugin against rendered DOM. Catches ~30–40% of WCAG failures: missing alt text, low contrast, missing form labels, duplicate IDs, missing landmarks, invalid ARIA.
  • Manual keyboard + screen-reader pass — pre-release per critical user flow. Tab-only traversal; NVDA / VoiceOver / TalkBack smoke; content-flow review. Catches the 60–70% axe can't see (focus order, label intelligibility, reading order).
  • Lighthouse-CI per-page a11y score — required gate; target ≥ 95 on key routes; regression blocks merge.

The frame is WCAG's POUR principles: Perceivable (alt text, contrast, captions), Operable (keyboard, time limits, focus), Understandable (clear language, labels, error messages), Robust (valid HTML, ARIA, screen-reader compatibility).

Why we need it

A11y bugs ship silently. The dev with a working mouse + 20/20 vision doesn't notice the missing label; the QA tester running by mouse + manual exploration misses the screen-reader user's experience entirely. Without automation, the first time anyone hears about the broken form is a customer-support ticket from a user with assistive technology — and by then the feature has shipped and the cost to fix is high.

Automation closes the volume gap. axe-core runs on every PR; the most common violations (missing alt, low contrast, broken labels) are caught at diff time. Manual verification covers what automation can't — but only for the cases that survived the automated gate. The combination is what makes WCAG 2.1 AA achievable at startup speed rather than as a "we'll do an audit before launch" disaster.

How others do it

Deque — axe-core ecosystem

axe-core (Deque, OSS) is the de-facto a11y rule engine. ~80 rules covering WCAG 2.1 AA + best practices. Wrappers for Jest, Playwright, Cypress, Selenium, Puppeteer; axe DevTools for browsers. Deque also publishes the "axe Auditor" tooling for SaaS-level a11y monitoring.

What we take: Adopt axe-core early. The Jest / Playwright integration is two lines of code; the rule set is battle-tested and zero-config-good.

Source: github.com/dequelabs/axe-core

W3C WAI — WCAG 2.1 / 2.2

W3C's Web Accessibility Initiative maintains WCAG. WCAG 2.1 AA (2018) is the legal baseline in most jurisdictions; WCAG 2.2 (2023) adds focus-visible, target size, and consistent-help criteria. POUR is the conceptual frame; SC-by-SC (success criterion) is the auditable detail.

What we take: WCAG 2.1 AA is the floor for any public-facing product. WCAG 2.2 adds incremental criteria; adopt as you can. POUR organises the team's mental model.

Source: w3.org/WAI/WCAG

Microsoft / GOV.UK — design-systems with a11y baked in

Microsoft's Fluent and GOV.UK's Design System ship components pre-audited for WCAG 2.1 AA. Adopting their components inherits the a11y work. GOV.UK in particular publishes accessibility statements + audit reports openly; their bar for compliance is industry-leading.

What we take: Build on accessible primitives (Radix, Headless UI, Adobe React-Aria, GOV.UK Frontend). Component-level a11y catches what page-level testing misses.

Source: design-system.service.gov.uk

Google Lighthouse + Chrome DevTools

Lighthouse's a11y category (powered by axe-core under the hood) gives a per-page score 0–100 that maps to detected WCAG failures + best-practice issues. Lighthouse-CI wires it into CI; Chrome DevTools' Issues panel surfaces a11y problems in real time during development.

What we take: Lighthouse-CI per-route a11y budget ≥ 95. Same tool engineers use locally + the CI gate.

Source: github.com/GoogleChrome/lighthouse-ci

Anatomy of the a11y pipeline

axe-core in unit / e2e tests

@axe-core/playwright or jest-axe per critical page / component. Critical violations fail; serious warn.

Per-route Lighthouse-CI a11y budget

Target ≥ 95 on key routes. Regression blocks merge. Per-route, not global.

Accessible component library

Radix / Headless UI / React-Aria / GOV.UK Frontend. Reuse a11y-audited primitives; don't reinvent.

Keyboard-only smoke test

Pre-release per critical flow. Tab through; activate every interactive element with keyboard alone. Manual; not automatable beyond focus order.

Screen-reader spot-checks

NVDA / VoiceOver / TalkBack on key user journeys. Catches label intelligibility, reading order, ARIA correctness.

Accessibility statement page

Public statement listing conformance level + known issues + reporting channel. Legal in EU + UK; ethical everywhere else.

How to run it

  1. Adopt WCAG 2.1 AA as the standard. Documented as the team's a11y baseline.
  2. Install axe-core in test runner. Two lines for jest-axe / Playwright. Run on every PR.
  3. Set Lighthouse-CI a11y budget. ≥ 95 per key route. Required CI check.
  4. Adopt an accessible component library. Radix / React-Aria / Headless UI. Don't roll your own.
  5. Keyboard-only smoke per critical flow. Manual; pre-release.
  6. Screen-reader spot-check pre-release. NVDA / VoiceOver on key journeys.
  7. Publish an accessibility statement. Public; conformance + known issues + report channel.
  8. Quarterly a11y review. Top-3 reported issues; coverage gaps; bump WCAG 2.2 criteria as adopted.

Deep dive — what makes a11y stick

Automation covers ~30–40%, manual is non-negotiable for the rest. axe-core catches missing alt, contrast, label, ID, landmark — the heuristically detectable failures. It can't detect "the alt text is wrong", "the focus order is illogical", "the heading hierarchy doesn't reflect content". Those require human judgement. Treat axe as necessary; manual as also necessary; neither alone is sufficient.

Per-route Lighthouse budget beats a global one. A global "site is 92" tells you nothing actionable. A per-route budget ("checkout ≥ 95, homepage ≥ 98, blog ≥ 90") gives reviewer-actionable feedback at PR time. Set the budget per route; track each separately.

Accessible primitives are the highest-leverage investment. Building on Radix / React-Aria / Headless UI / GOV.UK Frontend means focus management, ARIA, keyboard handlers are correct by default. Rolling your own button / modal / menu produces a11y bugs every time. The component-library choice is more impactful than any amount of testing.

Keyboard-only is the cheapest manual check. Five minutes with the mouse hidden surfaces 80% of operability bugs. Make it part of every critical-flow review. Engineers who do keyboard-only QA fix focus / tab-order issues at design time.

Screen-reader smoke at least monthly. NVDA (Windows, free) + VoiceOver (Mac, free) cover most users. Spend 30 min per critical flow. The first time a team does this they usually find label intelligibility issues that automation never could.

Accessibility statement is the public-facing artefact. EU + UK regulations require it; ethically every product should publish one. List conformance level, known issues, and a contact channel. Honest statements build trust; absent statements signal indifference.

WCAG 2.2 adds focus-visible + target-size + consistent-help. Adopt as you can; 2.2 is a refinement, not a replacement. Focus-visible is the highest-impact addition for keyboard users.

Lifecycle of an a11y check

Authored

Component / page built on an accessible primitive; axe-core test added alongside.

Gated

axe + Lighthouse-CI a11y budget run on every PR. Critical violations fail; serious warn.

Manually reviewed

Keyboard-only smoke + screen-reader spot-check pre-release for critical flows.

Released

Accessibility statement updated if conformance level changed or known issues added.

Monitored

User-reported issues triaged; root cause in axe rule gap, primitive bug, or new-feature regression.

Audited

Quarterly review: rule coverage, top-3 reports, WCAG 2.2 criteria adopted.

Worked example — Study App a11y pipeline

Playwright + axe wired to fail builds on critical violations; Lighthouse-CI per-route budgets; manual checklist for release sign-off.

# tests/a11y/lesson-player.spec.ts
import { test, expect } from "@playwright/test";
import AxeBuilder from "@axe-core/playwright";

test("lesson-player home is WCAG 2.1 AA-clean", async ({ page }) => {
  await page.goto("/lessons/abc");
  const results = await new AxeBuilder({ page })
    .withTags(["wcag2a", "wcag2aa", "wcag21aa", "best-practice"])
    .analyze();
  const critical = results.violations.filter(v => v.impact === "critical");
  expect(critical).toEqual([]);                  // fail build on critical
  for (const v of results.violations) console.warn("a11y:", v.id, v.help);
});
