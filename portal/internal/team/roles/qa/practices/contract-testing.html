Contract Testing — prevent API drift between services

Verifies that consumer expectations and provider capabilities stay aligned without requiring both services to be deployed simultaneously.

TL;DR

Contract testing pins the shape and semantics of an API between a consumer and a provider so changes on either side are caught at PR time, not at integration time. Two flavours dominate: consumer-driven contracts (Pact — the consumer records expectations as a pact, the provider verifies it can satisfy them), and spec-driven contracts (Schemathesis / Dredd — the OpenAPI/AsyncAPI spec is the contract, tests generate from it). Contract tests run without both services deployed — fast, deterministic, focused. Without them, API drift is invisible until the next deploy. Stripe, Atlassian, IBM, and the Pactflow community all wire contract tests as required CI gates; Study App treats the OpenAPI/AsyncAPI spec as the canonical contract and runs Schemathesis on every PR.

What it is

Contract testing verifies the agreement between a service and its consumer(s) without exercising the rest of the stack. The contract describes the request shape (method, path, headers, body schema), the response shape (status codes, headers, body schema), and the semantics (error codes, idempotency, pagination, ordering). The test exercises the contract against the implementation and fails when the implementation no longer satisfies the contract — regardless of whether the implementation has bugs unrelated to the contract.

Two flavours dominate. Consumer-driven contracts (Pact, Spring Cloud Contract) treat the consumer as authoritative: the consumer's test records the actual requests/responses it sends and expects; this pact file is published to a broker; the provider's CI pulls all consumer pacts and verifies it can satisfy them. Spec-driven contracts (Schemathesis, Dredd, Prism) treat the OpenAPI/AsyncAPI spec as authoritative: the test generates inputs from the spec, hits the provider, and verifies the responses conform. Each has a place; many teams use both.

Why we need it

Without contract tests, API drift is invisible. The provider team renames quiz_id to quizId; the consumer's parser silently returns null; the bug is found by a user. With contract tests, the rename fails the provider's CI before the PR can merge — the contract clause "quiz_id in response body" no longer holds. Integration tests don't cover this: they exercise the whole stack but only the happy paths the test author thought of. Contract tests cover every clause in the contract, generated mechanically, every PR.

How others do it

Pact / Pactflow (Beth Skurrie et al., 2013–)

Pact originated at ThoughtWorks/realestate.com.au and is now the de-facto consumer-driven contract framework. The pact file (JSON) records the consumer's expectations; the Pact Broker stores pacts, tracks versions, and runs the can-i-deploy check that tells deployment pipelines whether a provider version satisfies all current consumer pacts. Pactflow (commercial offering) adds bi-directional contract testing and the "provider verification" workflow. Pact is implemented in Ruby, JS, Java, .NET, Python, Go, PHP, Scala, Rust, Swift, and more — making it the polyglot standard.

What we take: consumer-driven contracts are the right shape for service-to-service APIs where consumers have narrow expectations; the pact broker is the missing piece teams forget.

Source: Pact Documentation

Schemathesis

Schemathesis is a property-based testing tool that consumes an OpenAPI or GraphQL spec and generates thousands of test cases to verify the running API conforms. It exercises every endpoint, every schema, every error path — including ones the spec author didn't think to test. Used by HubSpot, Microsoft, the Python community itself, and many open-source projects. Schemathesis catches the classes of bug humans miss: schema drift, error responses that don't match the documented shape, type-coercion mismatches.

What we take: spec-driven generative testing is the highest-leverage contract testing for a single-team API — one tool, one config, every endpoint covered.

Source: Schemathesis Docs

Stripe · SDK conformance as contract

Stripe publishes SDKs in 7+ languages, all generated from a single OpenAPI-like spec. Every API change is gated by running the full multi-language SDK test suite as a contract test — the spec change must produce passing SDKs in every language before it can ship. Stripe also tests its SDK's compatibility by replaying past API requests captured from real customer integrations. The result: Stripe has not made a breaking API change in 10+ years, while still shipping new versions weekly.

What we take: contract tests are the substrate for "no breaking changes" as a culture — without them, the promise is impossible to keep.

Source: Stripe Engineering — API Versioning

Atlassian · provider verification at scale

Atlassian (Jira, Confluence, Bitbucket) uses Pact across 100+ services. Their internal Pact broker tracks pact versions per consumer per provider; the can-i-deploy matrix is a daily artefact for the platform team. Atlassian published "Spec-First API Development" guidance describing the migration from integration tests to consumer-driven contracts — a 60% reduction in cross-team integration defects with no increase in test maintenance.

What we take: for multi-team microservice fleets, the pact broker and the can-i-deploy check are the operational tools that make contract testing work — the test framework alone is not enough.

Source: Atlassian — Spec-First API Development

Anatomy of a contract test

The contract artefact

A pact JSON, an OpenAPI YAML, an AsyncAPI YAML — a machine-readable document under version control; the source of truth.

The consumer side

Pact: the consumer test records expectations and produces the pact file. Schemathesis: the consumer reads the spec; no separate consumer artefact.

The provider verification

Provider CI loads the contract, replays/generates requests against the implementation under test, asserts the response satisfies the contract.

Provider states / fixtures

A way to seed the provider's state before each test ("there is a user with id 7", "the cart is empty"). Without states, the verification is flaky.

The broker / registry

A Pact Broker (for Pact) or a spec registry (Stoplight, Backstage) that stores contract versions, tracks publish/verify status, and enables can-i-deploy.

CI integration

Consumer publishes pact on PR; provider verifies all consumer pacts on PR; the verification job is a required status check. No green = no merge.

How to set up contract testing

  1. Pick the flavour. Spec-first single-team API → Schemathesis. Multi-team service-to-service → Pact. Both for high-leverage services (Stripe's pattern).
  2. Make the spec authoritative. The OpenAPI/AsyncAPI document lives in the provider repo, is reviewed like code, and is the input to both server and SDK generation.
  3. Wire the consumer side. Pact: add the pact library, write tests that exercise the consumer's API client, publish the pact JSON to the broker. Schemathesis: no consumer side — the spec is the contract.
  4. Wire the provider verification. Provider CI job runs the contract test on every PR; pulls consumer pacts from the broker (Pact) or runs schemathesis run http://localhost/openapi.yaml. Failures show the offending clause.
  5. Add provider states. For each pact, define the fixture: "given a user with id 7 exists, when GET /users/7, then 200 with body matching schema." Without states, the test cannot exercise non-empty scenarios.
  6. Make it a required gate. Contract test failure blocks merge. This is the lever that prevents drift; without it, the contract test is decorative.
  7. Add can-i-deploy. Deployment pipeline asks the broker: "can version X of the provider be deployed without breaking any current consumer?" Pactflow and the open-source Pact Broker both expose this endpoint.

Deep dive

The conceptual core of contract testing is "test the seam, not the system." Integration tests run the whole stack and ask "does it work end-to-end?" — slow, flaky, expensive. Unit tests run one component and mock its peers — fast, but mocks can drift from reality (the classic "mocks lie" failure mode). Contract tests sit between: they exercise the real component on one side and assert against the agreement (the contract) on the other side. The agreement is independently versioned and shared. When the consumer's mock drifts from the provider's reality, the contract test catches it because the contract is the same artefact on both sides.

Consumer-driven versus spec-driven is the central design choice. Consumer-driven contracts (Pact) optimise for accuracy: the consumer records exactly what it uses, so the provider only has to guarantee that subset. The drawback is operational complexity — every consumer pact must be tracked, versioned, and verified. Spec-driven contracts (Schemathesis) optimise for simplicity: one spec, one tool, every endpoint covered. The drawback is over-testing — the test exercises endpoints no consumer uses, and the spec may document behaviour no one depends on. Pact is right for federated environments with many consumer teams; Schemathesis is right for centralised teams that publish a single API.

Bi-directional contract testing is the 2022+ evolution. Pactflow's bi-directional model lets the provider publish its OpenAPI spec and the consumer publish a Pact-style expectations file; the broker computes compatibility — does the consumer's expected response shape match the provider's documented response shape? Bi-directional is faster than classic Pact (no provider verification run needed) and works for providers that already maintain OpenAPI. It is the right model when the provider team already invests in spec-first API design.

Provider states are the operational subtlety that catches teams off-guard. A pact says "GET /users/7 returns 200 with body matching schema." The provider verification needs a user with id 7 to exist. The pact framework requires the provider to declare a state handler for each named state: "given a user with id 7 exists" → seed the database. Without state handlers, the verification is non-deterministic. Teams that skip states end up with intermittently-failing contract tests and abandon the practice — the state declaration is mandatory, not optional.

Contract evolution is the long-term test. APIs change; consumers move at different speeds. The pattern is additive evolution: new fields are added (consumers ignore), old fields are deprecated but not removed, breaking changes are versioned. The contract test catches accidental breaking changes — removing a field, changing a type, narrowing an enum. Stripe's 10-year no-breaking-changes record is enforced by a CI gate that reads the diff against the published spec and fails if the diff includes a breaking change pattern; the team consciously chose the slower path of additive evolution and contract tests enforce it.

Schemathesis's property-based engine deserves a paragraph. It reads the spec, generates inputs that conform (and inputs that intentionally don't), hits the endpoint, and asserts the response matches the documented shape. It also checks invariants: 4xx responses must have an error body matching the documented error schema; 5xx responses are always failures (the server should not crash); idempotent operations should be idempotent under repeated requests; documented headers must be present. It catches the boring-but-important class of bugs — "the 422 response has error as a string but the spec says it's an object" — that humans don't write tests for.

The cost model is the closing argument. A contract test suite for a 30-endpoint API: ~2 days to set up Schemathesis end-to-end, ~30 minutes of CI time monthly to maintain, ~5 minutes per PR to run. The benefit: every API drift caught at PR time, every breaking change blocked at PR time, every consumer protected from silent failures. The cost of one production drift defect (one engineer-day to triage, one rollback, one apology to a customer) typically exceeds the annual cost of the contract test suite. The decision to invest is almost always positive once the team has more than one API consumer or more than one team.

Lifecycle

1

Spec authoring

Designer + provider engineer write the OpenAPI/AsyncAPI document or the consumer's pact expectations; reviewed like code.

2

Publish to broker / registry

Pact: consumer publishes pact on PR. Spec-driven: spec lands in repo; downstream consumers pin a version.

3

Provider verification

Provider CI runs the contract test on every PR; failure blocks merge with the offending clause named.

4

Can-i-deploy

Deployment pipeline queries the broker: provider version X is compatible with all current consumer pacts? Green → deploy; red → block.

5

Evolve additively

New fields added (consumers ignore); old fields deprecated with sunset date; breaking changes versioned and contract tests gate accidental breakage.

6

Retire

When a consumer drops a dependency, its pact is removed from the broker; provider stops verifying it on the next release.

Common pitfalls

PitfallWhy it happensWhat to do instead
No provider states Pacts assume "a user with id 7 exists" but the provider has no fixture for that state; verification is flaky. Declare a state handler for every named state in every pact; verification must seed the provider before each replay.
Pacts not in the broker Consumer pacts live in the consumer's CI artefacts; provider CI doesn't see them; drift goes undetected. Run a Pact Broker (open source or Pactflow); consumer CI publishes on PR; provider CI pulls and verifies.
Contract tests not required The test runs but doesn't block merge; over time, failures accumulate and are ignored. Mark contract verification as a required status check in branch protection; no green = no merge.
Over-testing in Pact Consumer test records every possible response variant; the pact becomes a re-implementation of the integration test. Record only what the consumer actually uses; trust Schemathesis for exhaustive endpoint coverage.
Spec out of sync with implementation OpenAPI is hand-maintained, implementation drifts; contract tests pass against the spec, real consumers fail in production. Generate the implementation (server stubs, validators) from the spec, or run Schemathesis on the live implementation to detect drift.

Worked example — Study App quiz API

The Study App backend exposes POST /api/quizzes/{id}/attempts for the frontend to submit a quiz attempt. The OpenAPI spec defines the request body ({answers: [{questionId, choiceId}]}), the success response (201 with {attemptId, score, passed}), and the error responses (422 for invalid answer shape with {error: {code, message, fields[]}}, 404 for unknown quiz, 401 for unauthenticated).

Schemathesis gate. CI runs schemathesis run --checks all http://test-instance/openapi.yaml on every PR that touches the backend. The tool generates ~500 test cases per endpoint: valid payloads (must return 201), payloads missing required fields (must return 422 with the documented error shape), payloads with extra fields (must be ignored per spec), payloads with type mismatches (string for integer — must return 422), unauthenticated requests (must return 401). The job runs in 90 seconds and produces an actionable failure: "endpoint POST /api/quizzes/{id}/attempts returned 500 instead of 422 for input {answers: null}; spec says null answers must return 422 with code ANSWERS_REQUIRED."

Consumer pact. The frontend's quizClient.ts has a Pact-style test that records the exact requests it makes (with the exact headers, payload structure, and expected response fields). The pact JSON is published to the Pact Broker on every frontend PR. The backend CI pulls this pact and verifies the backend can satisfy it; if the backend ever removes passed from the response, the pact verification fails before the backend PR can merge.

Outcome. In week 11, a backend refactor renamed questionId to question_id in the request body. Schemathesis caught it (input matching the spec — using questionId — returned 422 instead of 201, the documented behaviour for valid input). The PR was blocked before merge. Without contract testing, the rename would have reached staging and the frontend's quiz submission would have silently failed.

Cost. Setup: 2 days for Schemathesis + 3 days to add the consumer pact for quiz attempts. Maintenance: ~30 minutes of CI time monthly (cost of running on every PR). Catch rate: 4 API drift defects in 11 weeks, average severity P2, average cost-if-missed estimated at 1 engineer-day each.

Quality bar

Senior bar

Every API has an OpenAPI/AsyncAPI spec under version control as the authoritative contract. Schemathesis or equivalent runs on every PR as a required gate; bi-directional or consumer-driven Pact gates the cross-service boundaries. Provider states are declared and seeded; can-i-deploy gates deployment. Breaking-change detector runs against the spec diff and fails if a breaking change is detected without an explicit version bump. Stripe-style "no breaking changes" is a stated cultural commitment.

Mid bar

OpenAPI spec authored for every public endpoint; Schemathesis runs on every PR; failures block merge. Consumer pacts exist for the critical cross-team boundaries (3–5 services).

Junior bar

Can write a single Schemathesis test against a documented endpoint, run it locally, and explain how a contract test differs from an integration test (no real consumer needed, contract is the assertion).
