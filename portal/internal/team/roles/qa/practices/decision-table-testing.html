Decision Table Testing — model business rule combinations

Maps every combination of business conditions to its expected actions — making combinatorial logic visible, testable, and complete.

TL;DR

A decision table arranges conditions (rows) against rule columns; each column is one combination of condition values and the expected actions for that combination. For N binary conditions there are 2N columns; the discipline is to enumerate them, identify infeasible or impossible combinations explicitly, and produce one test case per remaining column. Decision tables are the canonical technique for business rules with three or more interacting conditions — eligibility, pricing, access control, escalation logic — where intuition systematically misses combinations. ISTQB lists decision tables among the four core black-box techniques; Boris Beizer codified them in Software Testing Techniques; modern decision-management standards (DMN, OMG) inherit the same model. For Study App they govern quiz scoring rules, promotion eligibility, and access tiers.

What it is

A decision table is a tabular representation of a set of business rules, with conditions as rows and rule columns as combinations of condition values. Each column also lists the actions that fire when the condition combination holds. The technique was introduced as a formal software-engineering tool in the 1960s, codified by Boris Beizer in Software Testing Techniques (1990), and is one of the four core ISTQB Foundation Level black-box techniques alongside Equivalence Partitioning, Boundary Value Analysis, and State Transition Testing.

For N binary conditions, the full ("extended-entry") decision table has 2N columns; for limited-entry tables, conditions may have more than two values (low/medium/high), producing more columns. The discipline is to enumerate every combination, identify infeasible ones (mark "—" or remove with documented rationale), and produce one test case per remaining rule. Modern variants — Decision Model and Notation (DMN, OMG 2015), Camunda's decision tables — preserve the same structure.

Why we need it

Combinatorial business logic is the most common source of subtle defects. Three flags with two values each is 8 combinations; intuition typically covers 3 to 5 of them; the missing 3 are where defects hide. Without a decision table, testers select combinations that feel representative and systematically miss the "minority" combinations (premium user with stale activity, free user with high recent score). Decision tables make the combination space visible — so the missing rules are obvious and the test plan is provably complete.

How others do it

Beizer · canonical reference

Boris Beizer's Software Testing Techniques (1990) is the canonical reference: chapter 10 covers decision tables, complete with rule consolidation (combining rules with identical actions), infeasibility marking, and the "cause-effect graphing" precursor that decision tables are derived from. Beizer's rule of thumb — "if you have more than three independent conditions affecting an output, draw the decision table" — has been cited in every subsequent test-design textbook.

What we take: the three-condition threshold — anything with three or more interacting flags warrants a decision table; below that, ad-hoc cases are fine.

Source: Beizer — Software Testing Techniques

ISTQB Foundation Level

ISTQB FL syllabus defines decision-table testing as one of four core black-box techniques and distinguishes coverage levels: minimum coverage (one test per rule column), collapsed coverage (rules with identical actions consolidated via don't-care markers), and extended coverage (every rule explicitly, no collapsing). ISTQB also defines coverage as a metric: (rules tested ÷ feasible rules) × 100%. The technique is examined on every FL sitting and is a prerequisite for ISTQB Advanced Test Analyst.

What we take: report decision-table coverage as a first-class metric next to line and branch coverage; one test per feasible rule is the minimum bar.

Source: ISTQB — Foundation Level Syllabus

DMN (OMG) & Camunda

Decision Model and Notation (DMN, OMG 2015) standardises decision tables as machine-executable artefacts: business rules are authored in a spreadsheet-like format, then executed by an engine. Camunda's open-source DMN engine, Drools' decision tables, and IBM ODM all use the DMN syntax. Modern teams treat the DMN table as both the spec and the implementation — there is no separate "test case derivation" step because the engine evaluates the table directly. Testing becomes verifying the table is correct (which is decision-table testing applied to itself).

What we take: for stable, business-owned rules, an executable DMN table beats hand-written code; the test cases become rows of expected inputs/outputs the business owner can review.

Source: OMG — DMN Specification

Atlassian · feature gating

Atlassian's feature-gating system (LaunchDarkly + internal tooling) is governed by decision tables: each feature has a table of conditions (plan tier, region, A/B group, beta opt-in, customer attributes) and the action is "feature shown / feature hidden." QA teams write decision-table tests against the gating service, exercising every rule column. Atlassian published the approach in their How We Test series, noting that decision-table-based gating tests catch ~30% of feature-rollout bugs that A/B-only testing would miss.

What we take: feature flags + decision tables = a natural pair; every flag-gated feature should have a decision table covering all condition combinations.

Source: Atlassian Engineering Blog

Anatomy of a decision table

Condition rows

Each condition is a question with a discrete value set (Y/N for binary, or low/med/high). One row per condition; the order is the author's choice but the same for every column.

Rule columns

One column per unique combination of condition values. For N binary conditions: 2N columns. Numbered R1, R2, … for citation in defect reports and test cases.

Action rows

Each action is an outcome (grant badge, send email, return error code). One row per action; the cell is filled when the action fires for that rule.

Feasibility markers

Some combinations are impossible (e.g. "not premium AND premium-only feature on") — mark "—" with a one-line rationale; never silently drop.

Consolidation (optional)

Rules with identical actions can be collapsed via "don't-care" markers (-); reduces test count but loses some clarity. Keep both forms during review.

Coverage report

Tested rule columns ÷ feasible rule columns × 100%. Reported per feature, per release. Gaps are explicit risk-acceptances, not silent omissions.

How to build a decision table

  1. Identify the decision under test. "When does the user see the Gold badge?" or "What scoring tier does a quiz attempt fall into?" — one decision per table, named clearly.
  2. List the conditions. Read the spec; enumerate every condition that influences the decision. If the spec is silent on a condition, file a spec defect before continuing.
  3. Enumerate all combinations. For N binary conditions, write out all 2N columns. A simple algorithm: condition 1 alternates Y/N every column; condition 2 alternates YY/NN; condition 3 alternates YYYY/NNNN.
  4. Fill in the actions. For each column, ask the spec: "given this combination, what happens?" Mark every action that fires. If the spec doesn't say, file a spec defect; never guess.
  5. Mark infeasible combinations. Use "—" and a short rationale ("free tier never has premium-only feature on"). Infeasible rules don't generate test cases but stay in the table for completeness.
  6. Optionally consolidate. Group columns with identical actions if there is a clear pattern; mark consolidated cells with "-". Keep the full form for the review.
  7. Generate one test case per feasible column. Each test exercises the system with the inputs that satisfy the condition combination and asserts every action in the column's action cells.

Deep dive

The conceptual claim of decision-table testing is that combinatorial coverage is the only honest coverage for combinatorial logic. Line coverage runs every line; branch coverage runs every branch; neither guarantees that combination "premium + incomplete course + high score" was tested with the right inputs. Only decision-table testing — one test per rule column — guarantees combinatorial coverage. The price is N tests instead of one; the value is that the test set demonstrably covers the rule space.

Extended-entry versus limited-entry tables is a structural choice. Extended-entry tables use Y/N (or T/F) for each condition; they are the cleanest form and the easiest to consolidate. Limited-entry tables allow multi-valued conditions (low/medium/high) directly in cells; they read more naturally for non-binary conditions but balloon in size (3N for 3-valued conditions). The pragmatic approach: use extended-entry for the analysis, expand to limited-entry only when the condition is inherently non-binary (e.g., user tier ∈ {free, plus, gold}).

Cause-effect graphing is the upstream technique. Before drawing the decision table, draw the dependency graph: each cause (condition) connects to each effect (action) via and/or/not gates. The graph reveals which conditions are independent and which interact; it also identifies infeasible combinations mechanically (a cause that contradicts another cause cannot both be true). For complex tables (5+ conditions), the cause-effect graph is the preferred starting point; the decision table is then derived from the graph.

Combination explosion is the universal challenge. Ten binary conditions = 1,024 columns. The mitigation is hierarchical decomposition: identify orthogonal sub-decisions, draw a decision table for each, and compose them. A 10-condition table is usually really three 4-condition sub-decisions; each has 16 columns, total 48 test cases — instead of 1,024. The skill is recognising the decomposition; the discipline is documenting it (so the reader can verify the sub-tables together cover the original space).

Infeasibility annotations are the single most-missed step. Real-world rule tables have many infeasible combinations (premium-only feature off for premium user, etc.). If they are silently dropped, the table appears incomplete; if they are silently included, the test suite has impossible tests. The standard practice: every infeasible combination keeps its column with "—" and a one-line rationale; the rationale is reviewed during the table walkthrough. Infeasibility is a property of the system under test, not the table; documenting it forces clarity.

DMN (Decision Model and Notation) takes decision tables from a test-design technique to an executable artefact. In DMN, the decision table is the implementation; the engine evaluates it directly. The table is authored by business analysts and reviewed by engineers. Testing DMN becomes table verification — does each rule produce the documented action? Camunda, Drools, and IBM ODM all support DMN; Study App's notification escalation rules and badge eligibility could be modelled in DMN if the rule set stabilises (currently they are in code).

Decision tables fit specific shapes of logic poorly. Continuous-valued inputs (price, time, distance) need equivalence partitioning + boundary value analysis, not decision tables — the table would need infinite columns. State-dependent behaviour (login → idle → expired) needs state transition testing, not decision tables — the table doesn't model time. Cause-effect logic with strict ordering needs sequence diagrams. The technique is right when the decision is a function of present-tense conditions, not history or continuous values; it is wrong when sequence, state, or magnitude matters more than combination.

Lifecycle

1

Identify candidate decision

Any feature with three or more interacting conditions is a candidate; the QA radar flags them during design review.

2

Draft table from spec

Conditions, rule columns, actions; gaps in the spec become spec defects before tests are written.

3

Review with product/engineering

Walk through every column; confirm infeasibility annotations; close ambiguities before turning columns into tests.

4

Generate test cases

One test per feasible rule, named by rule ID (R1–R8); inputs satisfy the column, assertions cover every action cell.

5

Execute & report coverage

Run the suite; report rule coverage; gaps are tied to documented risk-acceptance decisions.

6

Maintain on rule change

When a condition or action is added, the table is regenerated and the test set is updated — never patched ad-hoc.

Common pitfalls

PitfallWhy it happensWhat to do instead
Skipping the table for "obvious" 3-condition logic "Three flags is easy, I'll just test it" — and the tester silently covers 5 of 8 combinations. The three-condition threshold is the rule; draw the table and tick off every column, no matter how confident.
Silently dropping infeasible combinations The tester realises a combination can't happen and just removes the column; the next reviewer can't tell if it was omitted or analysed. Keep every column; mark infeasibility with "—" and a one-line rationale; review during the walkthrough.
Tables with continuous-valued conditions Tester models "amount < $100" as a Y/N condition; the table can't distinguish $99 from $50. Use equivalence partitioning + BVA for continuous values; decision tables are for discrete-valued conditions.
One test case covering multiple rules The tester writes one test that "covers" rules R3 and R5 by setting inputs that satisfy both — impossible by construction, since the rules differ in at least one condition. One test per rule column; if it looks like two rules collapse, consolidate the table explicitly with don't-care markers — don't fudge the test.
Table not regenerated on rule change Product adds a fourth condition; the table is patched ad-hoc with two extra columns; the other 6 are inconsistent. Regenerate the table from scratch every time a condition or action is added; preserve old IDs only if they still map to a unique combination.

Worked example — Study App Gold badge eligibility

The Gold badge is granted under these rules (spec §3.2): condition C1 = user is premium subscriber, C2 = course is ≥ 80% complete, C3 = last quiz attempt scored > 90%. Actions A1 = grant Gold badge, A2 = offer 20% renewal discount.

Decision table (3 conditions → 8 rules):

  • R1 (Y/Y/Y): A1 + A2 — premium, completed, high-scoring; the canonical Gold case.
  • R2 (Y/Y/N): — A2 only; the renewal discount fires on engagement, even without high score.
  • R3 (Y/N/Y): — A2 only; partial progress but high score still earns the offer.
  • R4 (Y/N/N): no action — premium but no engagement and no recent score.
  • R5 (N/Y/Y): — A2 only; free user with high engagement is offered an upgrade.
  • R6 (N/Y/N): no action — free + completion + low score doesn't trigger an offer.
  • R7 (N/N/Y): no action — free user with only a single good score.
  • R8 (N/N/N): no action — none of the conditions hold.

All 8 rules are feasible. One test case per rule: 8 tests in tests/badges/test_gold_eligibility.py, each parameterised with (premium, completion%, last_score) and asserting (badge_granted, discount_offered). The suite runs in ~120 ms as part of the unit-test gate.

Outcome: in week 9, a refactor of the eligibility service accidentally inverted condition C3 (changed > to >=, making 90% qualify). The decision-table test for R3 (Y/N/Y with score = 91%) still passed because Y was correct; the test for R5 with score = 90% (added as a BVA-style boundary) failed, catching the inversion. Without the decision table, the test suite would have covered only "premium + completed + high score" and the regression would have shipped.

Quality bar

Senior bar

Every feature with three or more interacting conditions has a decision table authored during design review. Tables are version-controlled alongside the spec; infeasibility annotations include rationale; consolidation is done deliberately, not implicitly. Decision-table coverage is reported as a first-class metric (≥ 95% on high-risk rules). For stable, business-owned rules, DMN-style executable tables are considered.

Mid bar

Decision tables drawn for combinatorial business rules; one test case per feasible rule; infeasibility marked. Coverage reported per release.

Junior bar

Can build a 3-condition decision table from a written spec, identify infeasible combinations, and produce one parameterised test per rule.
