Test Plan Authoring — a living document that drives test execution

A concise, structured test plan written before sprint execution — making the QA approach explicit, reviewable, and reproducible.

TL;DR

A test plan documents six things in 1–2 pages: scope, out-of-scope, approach (which techniques per area), environments, entry/exit criteria, risk. It is written before execution and updated when reality diverges. IEEE 829 (now ISO/IEC/IEEE 29119-3) is the canonical reference; James Bach's lightweight test-strategy templates and the Modern Testing Principles community evolved it for Agile/Lean. The plan is not waterfall — it's writing down what you plan to do before doing it. For Study App every release has a 1-page test plan in the release PR.

What it is

A test plan is a short, structured document recording the QA team's intent for a release, feature, or sprint. The canonical structure (IEEE 829, now ISO/IEC/IEEE 29119-3): scope, out-of-scope, approach, environments, entry/exit criteria, risk. The Agile/Lean evolution (James Bach's heuristic strategy, Modern Testing Principles) compresses this to 1–2 pages and treats the plan as living — updated when reality diverges, not frozen.

The plan is the team's hypothesis about what will need testing and how. It is reviewed at sprint start (PM + dev + QA), executed during the sprint, and retrospected at sprint end.

Why we need it

Without a test plan, the QA approach is implicit and varies by tester. Coverage is anecdotal; entry/exit criteria are debated each release; out-of-scope items become "you said you'd test that" arguments. A 1-page plan written before execution prevents all of these by making the approach explicit. The cost is ~30 minutes of writing; the saving is hours of avoided ambiguity.

How others do it

IEEE 829 / ISO 29119

IEEE 829 (Standard for Software Test Documentation, 1983–2008) is the canonical reference, superseded by ISO/IEC/IEEE 29119-3 (2013). The standard defines the Master Test Plan structure: introduction, items, features tested + not tested, approach, pass/fail criteria, suspension/resumption, deliverables, tasks, environment, responsibilities, schedule, risks, approvals. Heavy for modern Agile teams but provides the shared vocabulary.

What we take: the section list is the right inventory; the formality is optional — use the 6-section condensed form.

Source: ISO/IEC/IEEE 29119-3

James Bach · heuristic test strategy

Bach's "Satisfice Heuristic Test Strategy Model" is a one-page mind-map of test-strategy considerations: Project Environment, Product Elements (SFDPOT), Quality Criteria, Test Techniques, Perceived Quality. Designed for rapid test planning in the heat of a project. The claim: a 30-minute heuristic session beats a 50-page IEEE 829 plan no one reads.

What we take: the heuristic prompts (SFDPOT, CRUSSPIC STMPL) are excellent inputs even when authoring a more structured plan.

Source: Satisfice — Heuristic Test Strategy

Atlassian · sprint test plans

Atlassian's "Sprint Test Plan" template in the Team Playbook is the Agile-evolved version: 1-page document attached to the sprint plan with sections for scope, risk, exploratory charters, regression strategy, environment dependencies, entry/exit. Authored at sprint start in 30 minutes by QA + tech lead + PM; reviewed at sprint review. The lightweight form is now de-facto Agile/DevOps standard.

What we take: 1-page sprint test plan as a default; 5–10 pages only for high-risk releases.

Source: Atlassian — Team Playbook

Modern Testing Principles

Alan Page and Brent Jensen's Modern Testing Principles (2015) reframe test planning: the plan documents intent and learning, not a contract. Principle 5: "the customer is the only one capable of judging quality." Test plans should serve learning, not gatekeeping. The plan is a hypothesis that's updated with new data, not a contract that's enforced.

What we take: the plan is a living hypothesis — write it, review it, update it; don't freeze it as a contract.

Source: Modern Testing Principles

Anatomy of a test plan

Scope

Specific features, endpoints, user flows in scope. Explicit list, not "everything that changed."

Out-of-scope

What we will not test and why. Prevents "but you said you'd test that" arguments.

Approach per area

Which techniques apply where: BVA on numeric, contract on APIs, exploratory on UX, chaos on disruption.

Environments & data

Which tiers, what data is required, how data is seeded and reset.

Entry/Exit criteria

Prerequisites before testing starts; objective criteria to declare "QA done."

Risks & mitigations

Top 3–5 risks from RBT analysis with named mitigations.

How to author a test plan

  1. Start from the spec. Read the feature spec / release notes; list what changed.
  2. Define scope explicitly. Features in scope; user flows in scope; non-functional concerns (perf, a11y, security) in scope.
  3. Declare out-of-scope. What we won't test and why; prevents scope creep arguments.
  4. Pick techniques per area. BVA for numeric, contract for APIs, scenario for journeys, exploratory for UX, chaos for disruption. Use the RBT score to weight effort.
  5. Document environments and data. Which tier, which fixtures, who resets, who has access.
  6. Set entry and exit criteria. Entry: build green, smoke passing, dependencies deployed. Exit: bug bar met, coverage met, sign-off issued.
  7. List top risks. Pull from the RBT register; name 3–5 highest-impact risks and their mitigations.
  8. Review with the team. 15-minute review at sprint start (PM + dev + QA); update on the spot.

Deep dive

The conceptual claim of test planning is that explicit intent beats implicit intent. A plan written in 30 minutes forces decisions that would otherwise happen ad-hoc during execution — what's in scope, what techniques apply, what counts as done. Those decisions are reviewable: the PM can object to scope, the dev can object to technique, the QA can defend the bar. Without the plan, the decisions are still made but invisibly. The plan trades 30 minutes of writing for hours of avoided ambiguity downstream.

The IEEE 829 / ISO 29119 standards are the canonical vocabulary even when teams don't use them directly. The section list (scope, approach, environments, entry/exit, risk) is the right inventory. The 50-page formality is optional and usually counter-productive in Agile contexts — but the structure is sound. The pragmatic recipe: use the section list, compress to 1–2 pages, treat the document as living.

The "approach per area" section is the most-valuable and most-skipped. A plan saying "we'll test the new badge engine" is decoration; a plan saying "badge engine: BVA on score-thresholds, decision-table on 8 eligibility rules, scenario tests on award flow, chaos on concurrent award race" is operational. The technique-per-area mapping is where the plan does its real work — forces the QA to think about which technique catches which defect class.

Entry and exit criteria distinguish a plan from a wish-list. Entry: what must be true before testing starts? (Build green, dependencies deployed, test data seeded, environment healthy.) Exit: what must be true to declare QA complete? (Bug bar — P0=0, P1=0, P2≤3; risk coverage ≥ 80%; release sign-off; documented exceptions.) Without entry criteria, testing starts on broken builds; without exit criteria, testing never ends — or ends arbitrarily based on schedule.

Out-of-scope is the most-defensive section. Every retro has at least one "I assumed you'd test that" conversation; out-of-scope sections preempt them. List things that might reasonably be assumed in scope but aren't (legacy features not touched, third-party integrations relied on, browsers older than X). Each item has a one-line rationale or mitigation. The section is a contract; sticking to it requires discipline but pays off in clarity.

Living test plans avoid the waterfall trap. The historical objection was they froze the approach early and didn't adapt. The modern answer: write the plan as a hypothesis, update it when reality diverges, retrospect on the gap at sprint end. A plan that's never updated is suspicious — either reality matched perfectly (unlikely) or the team didn't update it. The retro question "where did the plan diverge?" produces learning for the next plan.

The 1-page limit is operational. A 1-page plan gets read; a 10-page plan does not. Page-count is the strongest forcing function for clarity. Atlassian, Stripe, Spotify all converged on 1–2 page plans for sprint-level work; longer plans are reserved for new products, major releases, or regulatory contexts.

Lifecycle

1

Draft

QA lead drafts the plan in 30 minutes at sprint start from the spec + RBT analysis.

2

Review

15-min review with PM + tech lead; updates incorporated; agreed at the top of the sprint.

3

Publish

Plan committed to the release PR or sprint ticket; visible to the whole team.

4

Execute

Testing follows the plan; deviations noted as reality diverges.

5

Update

Mid-sprint updates when scope changes; "out-of-scope adds" recorded explicitly.

6

Retrospect

Sprint retro reviews plan vs. reality; gaps inform next sprint's plan template.

Common pitfalls

PitfallWhy it happensWhat to do instead
50-page plan no one reads IEEE 829 templated verbatim; team treats it as ritual. 1–2 page condensed form; readable in 5 minutes.
Plan frozen at sprint start Treated as a contract; not updated when scope changes mid-sprint. Plan is a living hypothesis; updates land as comments or revisions.
Scope without out-of-scope Scope listed but out-of-scope omitted; "but you said" arguments at the retro. Explicit out-of-scope section with rationale per item.
Approach as generic boilerplate "We'll use a mix of unit, integration, and E2E tests" — true of any plan, decision-free. Approach-per-area: specific decisions, not generics.
No entry/exit criteria Testing starts on broken builds, ends arbitrarily when schedule demands. Explicit entry (build, env, data) and exit (bug bar, coverage, sign-off) criteria.

Worked example — Study App release 2.1 test plan

Release 2.1 (week 22) — 1-page test plan committed to release/2.1/test-plan.md:

  • Scope: badge engine, locale routing refactor, lesson rendering pipeline, accessibility settings page, quiz autosave.
  • Out-of-scope: payments (no changes), v1 quiz endpoints (retired in 2.0), browsers older than Chrome 110 / Safari 16.
  • Approach per area: badge engine — decision-table tests (8 rules) + property-based on score boundaries; locale routing — contract tests + 2 exploratory charters; lesson pipeline — scenario tests + visual regression; a11y settings — axe scan + WCAG 2.1 AA manual review; quiz autosave — chaos test (network drop) + 3 exploratory charters covering Resume tour.
  • Environments: standard staging for functional; prod-like rehearsal day before deploy; LaunchDarkly flag gating in production for canary.
  • Entry criteria: all PRs merged, smoke + contract gates green, staging healthy ≥ 99%, test data refreshed.
  • Exit criteria: bug bar P0=0, P1=0, P2 ≤ 3 with mitigations; risk coverage 100%; release sign-off issued; canary 10% traffic green for 4 hours.
  • Top risks: R1 badge engine incorrect awards (16) — deep + exploratory + property-based; R2 quiz autosave data loss (20) — chaos + Resume-tour exploratory; R3 locale routing wrong locale (12) — standard + 2 exploratory.

Plan reviewed at sprint start (15 min); committed to repo; updated once mid-sprint (added "iPad Safari 17" to scope after support data showed 8% of mobile traffic). Sprint retro reviewed plan vs. reality: 1 divergence (locale routing exploratory found more issues, ran 4 charters instead of 2); informed next sprint's plan to budget more exploratory time on refactor work.

Quality bar

Senior bar

1–2 page test plan committed to every release PR; reviewed at sprint start by QA + eng + PM; technique-per-area explicit; scope + out-of-scope + entry/exit + top risks all named. Plan updated when reality diverges; retro reviews divergence. Linked to RBT and quality dashboard.

Mid bar

Test plan exists per release with scope, approach, entry/exit; reviewed with the team; in writing.

Junior bar

Can author a 1-page test plan from a feature spec using the 6-section template; can defend the technique selection per area.
