Release Sign-off — structured GO/NO-GO decision at every release

Replaces the informal "it seems ready" release decision with a documented checklist and an explicit QA sign-off — tracked, auditable, and tied to evidence.

TL;DR

Release sign-off is the structured GO/NO-GO decision at every release: QA runs a documented checklist (CI gates green, bug-bar met, regression suite passed, contract tests clean, accessibility scan clean, performance budget held, security scan clean), gathers the evidence (links to dashboards, test reports, scan outputs), and issues an explicit GO or NO-GO recorded in the release ticket. The decision is auditable — not a verbal nod. A NO-GO blocks the release; a GO is the QA commitment that the release meets the quality bar. Microsoft's ship-room tradition, Google's launch checklists, Atlassian's release-readiness rubric, and Spotify's release-pre-flight all instantiate the same idea. For Study App the sign-off is a markdown template filed on every release PR; the decision and its evidence become institutional memory.

What it is

Release sign-off is a structured decision activity: before each release ships, QA runs a documented checklist of evidence gates, gathers the proof, and issues an explicit GO or NO-GO recorded on the release ticket or PR. The checklist is short (typically 7–10 items), each item is binary (green or red), and each green item links to its source of truth (CI run, Jira filter, scan report). The signatory is named; the time is recorded; the rationale is logged for NO-GO and for conditional-GO decisions.

The practice has roots in the Microsoft ship-room (Maguire, Debugging the Development Process; Zachary, Showstopper!) and the aerospace/medical-device release-control tradition (Stage 4 verification, formal release records). Modern continuous-delivery teams trim the ritual but keep the substance: every release ships with a documented decision, not a verbal nod.

Why we need it

Without structured sign-off, releases ship based on the loudest voice in the room — usually schedule pressure. With it, releases ship based on evidence; if the evidence is missing, the release is blocked. The discipline is uncomfortable when a deadline looms but it is exactly the point: the sign-off forces the trade-off ("ship now with this risk, accepted by this signatory") into the open rather than letting it slide. The audit trail also protects the team: post-incident the question "was the bug known when we shipped?" has a documented answer.

How others do it

Microsoft · ship room

Microsoft's product groups (Office, Windows, Visual Studio) run formal "ship rooms" before every release — a meeting where engineering, QA, PM, and support review a release checklist and produce a GO or NO-GO. Documented in Debugging the Development Process (Maguire, 1994) and Showstopper! (Zachary, 1994). The modern Azure DevOps platform automates the checklist into release-readiness dashboards; the human decision (signed off by the release manager) remains explicit.

What we take: the human decision is the artefact — automated gates don't replace it; they feed it.

Source: Azure DevOps Release Pipelines

Google · Launch Checklists

Google's Launch Coordination Engineering (LCE) is documented in the SRE Book. For every new service or major launch, LCE issues a customised checklist (security review, capacity plan, rollback plan, monitoring coverage, oncall coverage, SLO definition) and the launch is gated by an LCE engineer's sign-off. LCE publishes "Launch Checklist" templates for engineering teams; the practice scaled Google's launch process from ad-hoc to industrial.

What we take: for high-stakes launches (new services, major features), supplement the standard sign-off with a launch-specific checklist tailored to the change.

Source: Google SRE Book — Reliable Product Launches

Atlassian · release-readiness rubric

Atlassian's release engineering practice (documented in their Team Playbook) uses a release-readiness rubric per service: each item (CI green, bug bar, contract tests, security scan, capacity plan) scores binary. A composite score gates the release; below threshold blocks the deploy. Atlassian's rubric is integrated with their Bitbucket pipelines and Jira release tickets; the QA + engineering + product trio signs off jointly.

What we take: joint sign-off (QA + engineering + product) distributes accountability and prevents QA from being the sole "blocker" voice.

Source: Atlassian — Team Playbook

FDA & ISO 13485 · medical-device release records

FDA-regulated medical-device software (per IEC 62304 and ISO 13485) requires a formal Design History File entry for every release: verification evidence, validation evidence, known anomalies, sign-off authority. The records are auditable and form part of regulatory submission. While most consumer software doesn't need this rigour, the discipline (named signatories, documented evidence, archived decisions) is the gold standard the rest of the industry borrows from for high-stakes releases.

What we take: archive the sign-off artefacts (don't just record the decision); 6 months later when an incident happens, the artefacts are the audit trail.

Source: ISO 13485 — Medical Devices

Anatomy of a sign-off

Evidence gates

7–10 binary items: CI green, bug bar, regression pass, contract pass, axe scan, perf budget, security scan, capacity, rollback plan, monitoring.

Source-of-truth links

Each gate links to its evidence (CI build URL, Jira filter, axe report, Lighthouse URL, Snyk dashboard); checklists without links are unverifiable.

Bug bar

Explicit severity threshold (e.g. P0 = 0, P1 = 0, P2 ≤ 3); a defined exception process for over-threshold cases.

Signatory

QA lead is the default signatory; joint sign-off with engineering tech lead + PM for major releases. Named, not "the team."

Decision

GO / NO-GO / conditional-GO; recorded with rationale on the release ticket; conditional-GO carries explicit risk-acceptance.

Archive

Sign-off record retained per release (with linked evidence); auditable 6+ months later for incident postmortems.

How to run sign-off

  1. Publish the checklist template. A markdown file in the QA repo; 7–10 items; each with criteria and the source-of-truth link pattern.
  2. Define the bug bar. Severity thresholds (P0=0, P1=0, P2 ≤ N); document the exception process for over-threshold cases.
  3. Wire the evidence pipeline. Most gates automate (CI status badge, Jira filter URL, scan dashboard); the manual ones (capacity check, rollback plan) are explicit.
  4. Run the checklist before release. QA gathers evidence, checks each gate, fills the template on the release PR.
  5. Issue the decision. GO, NO-GO, or conditional-GO with explicit risk-acceptance rationale; signatory named and timestamped.
  6. Archive the record. Sign-off file committed to the release tag; retained per repo retention policy (Study App: 1 year minimum).
  7. Retrospect on escapes. Every escape's RCA names whether the gate that should have caught it was on the checklist; gaps tune the checklist.

Deep dive

The conceptual claim of structured sign-off is that release decisions deserve the same rigour as commit decisions. A commit goes through code review, automated tests, and explicit author + reviewer signatures; a release that touches a million users should not ship with less rigour. The structured sign-off applies code-review-style hygiene at the release boundary: documented criteria, attached evidence, named signatories, archived record. Continuous-delivery teams sometimes treat this as bureaucratic overhead; the right framing is that automation makes the substance cheaper, not less necessary. A 5-minute sign-off review is faster than a 5-hour post-incident "what happened" investigation.

The bug bar is the most-debated component. "P0 = 0, P1 = 0" is the strict bar; "P0 = 0, P1 ≤ 1 with mitigation plan" is the pragmatic bar. The choice depends on the product's risk profile and the deploy cadence — for a weekly release, the strict bar is reasonable because the next release is days away; for a monthly release, conditional-GO with a documented mitigation is more workable. The discipline is to define the bar before the release pressure hits; defining it under pressure produces ratcheting concessions.

Conditional-GO is the legitimate middle ground. It says: "we know about issue X, the mitigation is Y, the residual risk is documented and accepted by signatory Z." This is explicit risk-acceptance; it is qualitatively different from "we forgot to check" or "we pretended it was fine." When used responsibly, conditional-GO unblocks releases without compromising the discipline. When used recklessly, it becomes routine and the practice degrades. The protection is the named signatory: every conditional-GO names who accepted the risk, and that person owns the consequence if the risk materialises.

The evidence-link discipline distinguishes structured sign-off from theatre. A checklist that says "CI green ✓" with no link is a claim, not evidence; a checklist that says "CI green ✓ (link to build 4892, run 2026-05-25 14:32 UTC)" is auditable. Six months later when an incident triggers a postmortem, only the linked evidence matters. The standard practice: every checkbox links to a URL that proves the check passed. URLs that become broken (CI logs aged out, Jira filter changed) are flagged in the quarterly retention review.

Joint sign-off (QA + engineering + product) distributes the decision and prevents the "QA is the gatekeeper" antipattern. When only QA signs, QA carries the entire blame for any escape; when the trio signs, the decision is shared. The trio's perspectives are also complementary: QA owns the test evidence, engineering owns the operational readiness (capacity, monitoring, rollback), product owns the business risk-acceptance. Atlassian's release-readiness rubric and Google's LCE both use the trio model. The signatures are recorded individually so accountability is clear; the decision is collective.

Archive discipline is the operational equivalent of a flight-data recorder. Sign-off records committed to the release tag (or to a release-records directory) become the audit trail. Six months later, when an incident triggers a postmortem, the team can answer "did we know about this risk at release time?" The answer might be uncomfortable ("yes, we accepted it as conditional-GO") but it is honest. Without the archive, the question is unanswerable and the postmortem becomes speculation. FDA and ISO 13485 require archived release records for medical devices; the rest of the industry should borrow the discipline even when not regulated.

For continuous-delivery teams (multiple releases per day), the sign-off shrinks to an automated checklist with no human meeting — every gate is automated, the decision is "all gates pass = automatic GO," and the record is filed on the release commit. The structure remains: documented gates, linked evidence, archived record. The human is involved only when a gate fails (NO-GO requires explicit override). This is the "structured sign-off in a CI/CD world" form; the substance survives even as the form compresses.

Common pitfalls

PitfallWhy it happensWhat to do instead
Checklist without evidence links QA ticks "CI green ✓" with no link; impossible to audit, easy to fake. Every checkbox links to a URL that proves the check passed; missing links = unchecked.
Verbal sign-off "QA said it's good in standup" — no record, no audit trail. Sign-off is a written artefact committed to the release PR or tag; verbal sign-offs don't count.
Conditional-GO becomes routine Every release ships with 3 "small" exceptions; the bar erodes over months. Track conditional-GO rate; if > 20% of releases need it, the bar is wrong (too strict or too lax) and needs review.
QA as sole signatory QA owns the release decision alone; if it goes wrong, blame falls on QA; if it goes right, no shared learning. Joint sign-off with engineering and product for major releases; distributes accountability.
No archive Sign-offs filed but never retained; six months later the audit trail is gone. Commit sign-off records to the repo; retention policy ≥ 1 year.
Schedule pressure overrides the checklist "Just ship it, we'll fix tomorrow" — the checklist is bypassed under deadline. Bypasses require a named signatory and an audit-logged justification; bypass rate is tracked.

Worked example — Study App release 2.1 sign-off

Release 2.1 (week 22) included the new badge engine, locale routing refactor, lesson-rendering pipeline, accessibility settings, and quiz autosave (the same release used in the RBT example). QA lead ran the sign-off on Friday week 22 before the Monday deploy.

Sign-off checklist (committed to release/2.1/signoff.md):

  • CI gates green ✓ — build #4892, 2026-05-22 14:32 UTC
  • Bug bar ✓ — P0: 0, P1: 0, P2: 2 (both with mitigation plans, < bar ≤ 3) — Jira filter URL
  • Regression smoke + standard ✓ — 530/530 passing — CI test report
  • Contract tests (Schemathesis) ✓ — 0 failures across all endpoints — Schemathesis report
  • Accessibility (axe-core) ✓ — 0 violations on Home, Search, Lesson, Quiz — axe report
  • Performance budget (Lighthouse-CI) ✓ — LCP 1.8 s (budget 2.5), CLS 0.05 (budget 0.1) — Lighthouse-CI URL
  • Security (Snyk) ✓ — 0 critical/high CVEs — Snyk dashboard
  • Capacity + rollback plan ✓ — capacity confirmed by SRE; rollback playbook linked — runbook
  • Monitoring ✓ — new alerts wired for badge engine and autosave — Grafana dashboard

Decision: GO, signed Ivan B. (QA lead) + Marina K. (eng tech lead) + Sergey D. (PM), 2026-05-22 16:45 UTC.

Outcome: release shipped Monday; 1 escape (R2-related autosave corner case) caught in week 24 retro. The escape RCA referenced the sign-off; the missing gate was identified as "chaos-test on intermittent connectivity" which was not on the checklist. Checklist updated for release 2.2 to include the missing gate. Sign-off record retained in release/2.1/signoff.md in the repo.

Quality bar

Senior bar

Every release ships with a documented sign-off committed to the repo; 7–10 gates each with linked evidence; named signatories from QA + engineering + product for major releases. Bug bar explicit; conditional-GO rate < 20% and tracked. Sign-off records retained ≥ 1 year; referenced in incident postmortems. Bypass requires named signatory + audit-logged justification.

Mid bar

Sign-off checklist exists; QA fills it per release with evidence links; GO/NO-GO recorded on the release ticket; signatory named.

Junior bar

Can run a pre-prepared release-sign-off checklist, gather evidence links per gate, and produce a draft sign-off recommendation for the QA lead to approve.
