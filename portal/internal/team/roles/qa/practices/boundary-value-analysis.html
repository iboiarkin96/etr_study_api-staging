Boundary Value Analysis — test the edges

Systematically tests the values at and immediately outside the edges of valid input ranges — because defects cluster at boundaries, not in the middle.

TL;DR

Boundary Value Analysis (BVA), described by Glenford Myers in The Art of Software Testing (1979), selects test values at and immediately around the edges of every bounded input domain — typically min−1, min, min+1, max−1, max, max+1. The rationale is empirical: off-by-one errors, integer-overflow bugs, and validation-by-comparison defects concentrate at the partition edges. BVA pairs with equivalence partitioning (EP picks the partitions, BVA picks the values inside and around them) and is examined on every ISTQB Foundation Level sitting. For Study App it makes boundary coverage mandatory and explicit on every bounded field — quiz score 0/1/99/100, page size 1/100, password length 8/72 — rather than relying on intuition.

What it is

Boundary Value Analysis is a black-box test design technique that selects test values at and immediately around the edges of valid input partitions. Glenford Myers introduced it in The Art of Software Testing (1979) alongside equivalence partitioning, and ISTQB has codified it as one of the four core black-box techniques (BVA, EP, decision tables, state transition). For any input with a defined valid range [min, max], BVA selects: min−1 (invalid, should reject), min (valid, should accept), min+1 (valid, should accept), max−1 (valid, should accept), max (valid, should accept), max+1 (invalid, should reject). The two-value variant uses only min and max; the three-value variant adds min±1 and max±1.

BVA applies to any bounded domain — not just numeric: string lengths, list sizes, date ranges, price ranges, array indices, pagination limits, file sizes, timeout durations. It is used in combination with equivalence partitioning: EP identifies the partitions, BVA selects the values inside and around the partition boundaries.

Why we need it

Off-by-one errors are among the most common defect classes in software. A loop that runs one iteration too many, a validation that accepts 256 characters when the database column is 255, a date range that excludes the last day, an integer that overflows at 2,147,483,648 — all are boundary defects that random or middle-of-range testing misses entirely. Without BVA, testers pick values that "feel right" (50 of 1–100) but systematically avoid the edges where defects actually live. BVA makes boundary testing mandatory and systematic, so the defect class cannot be skipped by accident.

How others do it

Myers / Beizer · classical foundations

Glenford Myers introduced BVA in The Art of Software Testing (1979) and Boris Beizer expanded it in Software Testing Techniques (1990) with the empirical observation that boundary errors account for a disproportionate share of all defects in numeric, string, and array code. Beizer codified the "boundary inside, boundary outside, typical" rule that became the industry default: for each partition, test one value inside the boundary, one outside, and one typical interior value. This is the foundation of every modern BVA technique.

What we take: BVA and EP are inseparable — EP identifies partitions, BVA selects the boundary values; neither alone is sufficient.

Source: Myers, The Art of Software Testing

ISTQB Foundation Level

ISTQB's FL syllabus defines BVA as one of four mandatory black-box techniques and distinguishes two- and three-value BVA. Two-value BVA selects min and max from each partition (minimum coverage); three-value BVA also selects min−1, min+1, max−1, max+1 (full boundary coverage, required for ISTQB Advanced). Boundary coverage is calculated as (tested boundary values ÷ identified boundary values) × 100% and is included in coverage reports as a first-class metric. BVA appears on every FL exam sitting — it is the most frequently examined technique.

What we take: use three-value BVA for high-risk fields (full six-point set); two-value BVA for low-risk fields (just min and max); record boundary coverage as a metric.

Source: ISTQB — Foundation Level Syllabus

Property-based testing (Hypothesis / fast-check / QuickCheck)

Modern property-based testing libraries treat BVA as a first-class concern: Python's Hypothesis automatically biases its generators toward boundary values (0, 1, −1, MAX_INT, MIN_INT, empty string, empty list) before sampling the interior. JavaScript's fast-check and Haskell's QuickCheck do the same. The library's "shrinker" reduces failing test cases toward the simplest boundary that still triggers the defect. Hypothesis's integers(min_value=1, max_value=255) tests min−1, min, min+1, max−1, max, max+1 automatically before any random values.

What we take: for high-risk fields, complement manual BVA with property-based testing — the library automates the six-point set and finds boundaries human analysis missed.

Source: Hypothesis — Strategies

Google · fuzzing as automated BVA

Google's Software Engineering at Google (Winters, Manshreck, Wright, 2020) documents that the OSS-Fuzz programme is, in effect, BVA at scale: the fuzzer is particularly effective at finding boundary values that human testers miss — integer-overflow boundaries (2^31, 2^32, 2^63), string-length boundaries (4096, 65535), and Unicode-codepoint boundaries (U+007F, U+FFFF). OSS-Fuzz has found tens of thousands of boundary defects across open-source projects. Google's internal test-review process flags suites that lack boundary coverage as incomplete.

What we take: for parsers, serialisers, and numeric pipelines, complement manual BVA with fuzz testing — it finds platform-level boundaries (overflow, encoding) that domain analysis cannot anticipate.

Source: Software Engineering at Google — Testing Overview

Anatomy of a BVA test set

Input-domain inventory

List every bounded input for the test area: numeric ranges, string lengths, list sizes, date/time ranges, enum values, pagination limits, file sizes.

Partition map (from EP)

For each input, the valid and invalid equivalence partitions and the explicit boundary expressions (min, max, exclusive vs. inclusive).

Six-point boundary set

For each bounded input: min−1, min, min+1 (lower) and max−1, max, max+1 (upper). Two-value variant drops min±1 and max±1 for low-risk fields.

Expected-outcome table

For each boundary value: the system response — accept, reject with a named error code, return a boundary-specific result (e.g. empty page, last record).

Risk-tiered coverage rule

High-risk fields (security, billing, identity) use three-value BVA; medium-risk use two-value BVA; low-risk fields use representative-value testing only.

Boundary-coverage report

Tested boundary values ÷ identified boundary values, reported per release. Gaps are explicit (not silently skipped) and tied to risk-acceptance decisions.

How to run BVA

  1. Inventory bounded inputs. For the feature under test, list every input with a defined range — from API parameters and form fields to internal counters, page-size limits, and timeout durations. Don't skip non-obvious ones (string lengths, list sizes, date ranges).
  2. Read the spec for exact boundaries. "Username 3–20 chars" means min=3, max=20. Confirm inclusive vs. exclusive (is 20 valid or rejected?). If the spec is silent, file a defect against the spec before writing tests.
  3. Apply EP first. Split each input into valid and invalid equivalence partitions. BVA without EP misses non-numeric boundaries (e.g. role-enum transitions).
  4. Generate the six-point set. For each partition boundary: min−1, min, min+1, max−1, max, max+1. Drop duplicates if min+1 = max−1 (range of width ≤ 2).
  5. Pair each value with an expected outcome. Accept / reject + error code / boundary-specific result. Write the expected outcome before executing — never "we'll see what it does."
  6. Execute and record results. Per-value pass/fail with the observed response. Boundary defects often manifest as silent acceptance of invalid input or silent rejection of valid input — both must be checked explicitly.
  7. Report boundary coverage. Tested / identified boundary values, per release. Gaps are visible in the dashboard with a documented risk-acceptance rationale.

Deep dive

The empirical foundation of BVA is the observation that bugs cluster at boundaries. The mechanism is straightforward: at the interior of a range, all inputs are handled by the same code path, so a single test exercises the whole partition. At the boundary, the code must decide — and decision points are where humans make mistakes. The classic error is the off-by-one in a comparison: if (len > 255) versus if (len >= 255) — the difference is invisible at len = 100, glaring at len = 255. Three-value BVA catches both forms of the bug because it tests both sides of the boundary.

Two-value versus three-value BVA is the most common confusion. Two-value BVA tests only min and max — the boundary values themselves. Three-value BVA also tests min−1 and max+1 (one step outside) and min+1 and max−1 (one step inside). The ISTQB Foundation Level certification covers two-value; the Advanced Level covers three-value. For Study App, three-value is the default for high-risk fields (password length, quiz score, payment amount, page size) because each adds only four extra test cases and catches the entire boundary defect class. Two-value is the fallback for low-risk fields (sort key, display-name preference) where the boundary defect cost is bounded.

Non-numeric boundaries are the most-missed application. String length: empty string, one character, max−1, max, max+1. List size: empty list, one element, max−1, max, max+1. Date range: one day before start, start date, one day after start, one day before end, end date, one day after end. Time: midnight (00:00:00), one second before midnight (23:59:59), the leap second (when applicable). Floating point: the smallest positive value, the largest finite value, NaN, infinity. Every bounded domain has boundaries; BVA forces them into the test set.

Platform-level boundaries are invisible without BVA: integer overflow at 2^31−1 (signed 32-bit), 2^63−1 (signed 64-bit); IEEE 754 floating-point precision loss at 2^53 (the largest integer representable exactly as a double); Unicode codepoint boundaries at U+007F (ASCII), U+FFFF (Basic Multilingual Plane), U+10FFFF (Unicode maximum); HTTP header size limits (typically 8KB), URL length limits (browsers vary, 2KB practical). For parsers and serialisers, fuzz testing is the practical way to cover these; for application code, the BVA test set should include any platform boundary the field could reach.

BVA versus property-based testing is complementary, not competitive. BVA is deterministic — you pick the six points, run them, get pass/fail. Property-based testing is generative — you describe the property ("for all valid inputs, the response is 200") and the library samples inputs, biased toward boundaries. BVA is auditable (you can show the test set in a review); property-based testing is exploratory (it finds boundaries you forgot). The pragmatic recipe: BVA for the named boundaries you must cover (spec compliance, regression catch); property-based for the boundaries you might have missed (defect-finding).

Boundary coverage as a metric distinguishes mature QA orgs from intuition-driven ones. The metric is simple: (tested boundary values ÷ identified boundary values) × 100%. Identified boundaries come from the EP/BVA analysis; tested boundaries come from the test execution log. A field with min=3, max=20 has six identified boundaries; if the test set covers four of them, coverage is 67%. Boundary coverage joins line coverage, branch coverage, and mutation coverage as a first-class quality metric — and it is the one most directly tied to defect-discovery capability.

Common anti-patterns are equally important. "We tested 100, it works" — the interior is the wrong test value; switch to 0/1/255/256. "The spec says 1–255, we trust the validation" — BVA exists because validation often disagrees with the spec at the edges. "Adding boundary tests slows the suite" — six values per field is small; the actual cost is in test-data setup, which is the same regardless. "Property-based testing replaces BVA" — it complements but does not replace, because the spec lists boundaries that the property does not name. The practice is to do both, deliberately.

Lifecycle

1

Spec review

Confirm each bounded input has explicit min/max and inclusive/exclusive semantics; file a spec defect if silent.

2

EP + BVA design

Partition the input domain, then enumerate the six-point boundary set for each partition.

3

Expected-outcome pairing

Every boundary value gets a named expected response (accept / reject with code / boundary-specific result).

4

Execute

Run the BVA test set as part of regression; record per-value pass/fail with the observed response and any defect IDs.

5

Coverage report

Tested boundaries ÷ identified boundaries, per release; gaps tied to a documented risk-acceptance decision.

6

Maintain on spec change

When a range changes (max 255 → 1000), the BVA set is regenerated and the old values are dropped — boundary tests are not frozen.

Common pitfalls

PitfallWhy it happensWhat to do instead
Testing only the valid range (min and max) Testers cover min and max but skip min−1 and max+1, missing the invalid-input rejection defects — the most common BVA failure mode. Always include min−1 and max+1. Invalid boundaries verify the rejection path; they are as important as valid ones.
BVA without defined expected outcomes The test set lists boundary values but says "see what happens" — when a test fails, no one can tell whether the system or the test is wrong. Pair every value with a named outcome before executing: HTTP 200 with body shape, HTTP 422 with error code FIELD_TOO_LONG, etc.
Skipping non-numeric boundaries Testers apply BVA to numeric fields but forget string length, list size, date range, file size, timeout — all bounded domains. The inventory step is non-negotiable: every bounded input gets a BVA set, not just numeric ones.
Off-by-one in the BVA values themselves "max+1" is generated by hand; for max = 255, the tester writes 257 or 254 by mistake — the boundary defect hides in the test data. Generate the six-point set in code (a helper function or parameterised test) so the values are derived from min/max constants, not typed.
Ignoring platform boundaries The field accepts integers, the spec says 1–1,000,000 — but the implementation uses int32 and overflows at 2^31; the spec-level BVA never tests it. For numeric fields, add platform boundaries (INT_MAX, INT_MIN, MAX_SAFE_INTEGER) to the BVA set when the field could plausibly reach them; for parsers, add fuzz testing.

Worked example — Study App quiz-score field

The quiz.score field accepts integer values in the range 0–100 (inclusive). The spec also states: scores below 0 must return HTTP 422 with code SCORE_BELOW_MIN; scores above 100 must return HTTP 422 with code SCORE_ABOVE_MAX; non-integer values must return HTTP 400 with code SCORE_INVALID_TYPE.

EP partitions: invalid-low (score < 0), valid (0 ≤ score ≤ 100), invalid-high (score > 100), invalid-type (non-integer). BVA six-point set for the numeric boundaries:

  • score=-1 → HTTP 422, code SCORE_BELOW_MIN (min−1, invalid-low)
  • score=0 → HTTP 200, score stored as 0 (min, valid edge)
  • score=1 → HTTP 200, score stored as 1 (min+1, valid interior edge)
  • score=99 → HTTP 200, score stored as 99 (max−1, valid interior edge)
  • score=100 → HTTP 200, score stored as 100 (max, valid edge)
  • score=101 → HTTP 422, code SCORE_ABOVE_MAX (max+1, invalid-high)

Platform boundaries (high-risk field, payment-adjacent): score=2147483647 (INT_MAX) → HTTP 422; score=2147483648 (INT_MAX+1, overflow check) → HTTP 400 (parse failure, not overflow). Type boundaries: score="50" (string) → HTTP 400 SCORE_INVALID_TYPE; score=50.5 (float) → HTTP 400 SCORE_INVALID_TYPE; score=null → HTTP 400 SCORE_REQUIRED.

Boundary coverage: 6 numeric boundaries identified, 6 tested = 100%. Platform: 2 identified, 2 tested = 100%. Type: 3 identified, 3 tested = 100%. Total: 11/11 = 100% boundary coverage for the quiz-score field. The full set runs in ~80 ms as part of the API contract suite.

Outcome on a real defect: in week 7 of Study App, a refactor changed the validation to if (score > 100) (strict) without adjusting the lower bound, accidentally rejecting score=100. The score=100 BVA test caught it on the next CI run; without BVA the test set would have used score=50 and the defect would have reached staging.

Quality bar

Senior bar

Every bounded input has an explicit EP + BVA analysis with named partitions, six-point boundary set, and per-value expected outcomes. Three-value BVA on every high-risk field. Boundary coverage reported as a first-class metric (≥ 95% on high-risk fields, ≥ 80% overall). Property-based testing complements BVA on identity, billing, and parser code. Platform boundaries (overflow, encoding) included where the field could reach them. Spec-silent boundaries are filed as spec defects before tests are written.

Mid bar

Three-value BVA on high-risk fields; two-value BVA on medium-risk fields; expected outcomes documented for each value. Boundary coverage reported per release. Non-numeric boundaries (string length, list size, date range) included in the inventory.

Junior bar

For each new bounded field, can produce the six-point BVA set (min−1, min, min+1, max−1, max, max+1), pair each value with an expected outcome, and run the set as part of the regression suite.
