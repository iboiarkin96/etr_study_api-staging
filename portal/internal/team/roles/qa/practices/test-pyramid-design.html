Test Pyramid Design — shape the automation portfolio

Defines the right ratio of unit, integration, and end-to-end tests — so CI stays fast and feedback loops stay tight.

TL;DR

The test pyramid (Mike Cohn, Succeeding with Agile, 2009) prescribes a ratio of test layers: many fast unit tests at the base, fewer integration tests in the middle, a thin layer of slow E2E at the top. Each layer trades execution speed for confidence breadth. Inverted pyramids — too many E2E — produce 40-minute suites that flake constantly. Martin Fowler's "Practical Test Pyramid," Google's testing series, and Kent Dodds' "Testing Trophy" all explore variations. For Study App the working ratio is ~70% unit / 20% integration+contract / 10% E2E+scenario, codified as ADR and reviewed quarterly.

What it is

The test pyramid is a heuristic for the ratio of automated tests by type. Mike Cohn introduced it in Succeeding with Agile (2009) with three layers: unit (base, fast, many), service/integration (middle), and UI/E2E (top, slow, few). Cohn's claim: pyramids work, inverted pyramids fail. The pyramid is a design budget — the team decides the ratio explicitly and the suite is curated to stay within it.

Modern variants refine the layers: Martin Fowler's "Practical Test Pyramid" (2018) renames middle to "service tests" and emphasises contract testing; Kent Dodds' "Testing Trophy" (2018) adds static analysis as a base for typed languages; "testing diamond" inverts for microservices where unit tests have less leverage. The shape varies by context; the principle (deliberate ratio, more fast than slow) is constant.

Why we need it

Without explicit pyramid design, test suites drift to whatever's easy to write that quarter — often E2E. Inverted pyramids produce 40-minute CI runs, flaky tests no one trusts, and triage nightmares. The pyramid is the discipline that keeps the suite fast and feedback tight. The cost of writing the ratio down (one ADR) is negligible; the cost of an inverted pyramid is felt every PR for years.

How others do it

Mike Cohn · the original pyramid

Cohn introduced the pyramid in Succeeding with Agile (2009) with the canonical three layers: unit (fast, many), service/API (medium, fewer), UI (slow, fewest). Each layer up is more expensive to write, slower to run, and more brittle. Therefore most tests should be at the base. The shape is a budget — measure the actual distribution, compare to target, intervene if inverted.

What we take: the pyramid is a budget, not a description; measure and compare quarterly.

Source: Cohn — The Forgotten Layer of the Test Automation Pyramid

Martin Fowler · Practical Test Pyramid

Martin Fowler & Ham Vocke's "The Practical Test Pyramid" (2018) refines the model: rename "service tests" → integration + contract; emphasise "test the boundary, not the system"; introduce specific anti-patterns (ice-cream cone — UI-heavy; cupcake — flat layers). The most-cited modern reference and the source of "test the right thing at the right layer."

What we take: contract tests at the integration layer replace many slow E2E tests with fast targeted tests for the same defect class.

Source: Fowler — Practical Test Pyramid

Google · per-size SLOs

Google's Software Engineering at Google chapter 11 codifies the pyramid for monorepo-scale codebases. The ratio: ~80% small (unit, fast, hermetic), 15% medium (single process boundary), 5% large (E2E, slow). Each size has SLOs on speed, flakiness, and ownership. The discipline scales to ~150M tests/day across Google's monorepo.

What we take: per-layer SLOs (unit ≤ 100ms, integration ≤ 1s, E2E ≤ 30s) make the pyramid enforceable, not aspirational.

Source: Software Engineering at Google — Testing Overview

Kent Dodds · Testing Trophy

Kent Dodds' "Testing Trophy" (2018) refines the pyramid for modern TypeScript/React apps: static analysis at the base, then unit, then integration (the largest layer — DOM-rendering tests), then E2E. The claim: integration tests with React Testing Library + jsdom catch the most defects per dollar in modern frontend; pure unit tests have less leverage when most value is in component composition.

What we take: the shape varies by domain — frontend often has a thicker integration layer; backend keeps a wider unit base.

Source: Dodds — The Testing Trophy

Anatomy of a pyramid

Unit layer (~70%)

Pure functions, classes, components in isolation; no I/O; sub-100ms each; hermetic.

Integration + contract (~20%)

Component boundaries — DB queries, API contracts, message-bus; real services via Testcontainers; sub-second each.

E2E + scenario (~10%)

Full-stack user journeys; Playwright/Cypress; expensive, flaky, but irreplaceable for the integration of components.

Static layer (optional)

Typing, linting, dependency analysis; runs at edit time; catches a defect class no test can.

Per-layer SLOs

Speed and flakiness budgets per layer; tests violating SLO refactored or demoted.

Quarterly review

Measure actual distribution; compare to target; intervene if inverted.

How to design and maintain a pyramid

  1. Pick the target ratio. Default: 70% unit / 20% integration+contract / 10% E2E for full-stack. Adjust for context.
  2. Codify in an ADR. The ratio + rationale + per-layer SLOs in docs/adr/test-pyramid.md; reviewed quarterly.
  3. Set per-layer SLOs. Unit ≤ 100ms each, ≤ 0.1% flake; integration ≤ 1s, ≤ 0.5% flake; E2E ≤ 30s, ≤ 1% flake.
  4. Wire test-tier tags. Decorators / pytest markers / Jest projects route each test to the right CI job and SLO budget.
  5. Measure the distribution. Quarterly: count tests per layer; compute actual ratio; compare to target.
  6. Intervene if inverted. Over-E2E: identify lowest-value E2E; replace with integration or unit equivalents.
  7. Review and tune the target. Quarterly review of ratio target; adjust as the product matures.

Deep dive

The pyramid is fundamentally an economic argument. Each layer has a different cost profile. Unit tests cost ~100ms each, ~10 minutes each to write; integration tests cost ~1s and ~30 minutes; E2E tests cost ~30s and ~2 hours. Unit tests also fail with much higher precision — when a unit test fails, you know which function broke; when an E2E fails, you have a stack trace through ten components. The pyramid encodes these economics.

The "test the boundary, not the system" principle from Fowler distinguishes the integration layer's purpose. Integration tests cover specific seams — the database boundary, the API boundary, the message-bus boundary. At the seam, real components are used; outside the seam, the rest of the system is mocked or absent. This makes integration tests fast and targeted, instead of slow and broad.

Anti-patterns are easier to see than the right shape. The "ice cream cone" — many E2E, few unit — produces the worst possible outcome: slow CI, flaky failures, unclear root causes, and most defects still slip past. The "cupcake" — equal layers — produces a suite that's neither fast nor confident. The right shape is named "pyramid" for a reason.

Contract tests are the most underused layer. Many teams have unit + E2E and skip integration entirely; E2E does work contract tests would do faster. Schemathesis, Pact, Spring Cloud Contract generate or validate against an OpenAPI/AsyncAPI spec at the API boundary. A contract test takes ~100ms and catches the entire class of "API drift" defects. Adding the contract layer is usually the highest-leverage pyramid intervention.

The static-analysis layer (Dodds' addition) is the cheapest of all. TypeScript, Pyright/mypy, Sorbet eliminate entire defect classes at edit time. Lint rules catch style and bug-prone patterns. Sub-second per file; the value is preventing whole bug categories. Treat static analysis as the base of the pyramid for typed languages.

Per-layer SLOs make the pyramid enforceable. Without SLOs, the pyramid is aspirational. With SLOs, tests that violate are flagged for refactoring or demoted to a slower tier. The discipline: a unit test taking 500ms isn't a unit test, it's an integration test mislabelled.

The pyramid shape varies by domain. Microservices often have inverted-ish shapes because most logic lives in component boundaries — "diamond"-shaped distributions where integration dominates. Frontend-heavy apps often look like Dodds' trophy — DOM integration tests catch most defects. ML systems often have a separate evaluation layer. The principle is universal; the exact ratio is contextual.

Common pitfalls

PitfallWhy it happensWhat to do instead
Ice-cream cone (E2E-heavy) E2E feels concrete; team writes them first; suite slows to 40 minutes; flakiness compounds. Replace lowest-value E2E with integration or unit equivalents; cap E2E count.
Missing integration layer Team has unit + E2E; integration layer skipped; E2E does work integration should do. Add contract tests (Schemathesis, Pact) and DB-boundary tests with Testcontainers.
Mislabelled tests "Unit test" that hits the DB and takes 2 seconds; layer boundaries blur. Per-layer SLOs enforce: tests violating SLO are renamed/demoted.
Pyramid frozen at one ratio Target set once 3 years ago; product changed but ratio didn't. Quarterly review of target ratio; adjust as product context shifts.
No static-analysis layer Typed language but no type-checker in CI; entire defect class catchable for free is unaddressed. TypeScript / mypy / Pyright / Sorbet in CI as a required gate.

Worked example — Study App pyramid distribution

Study App's test pyramid is codified in docs/adr/0025-test-pyramid.md with target ratio 70% unit / 20% integration+contract / 10% E2E+scenario; per-layer SLOs documented.

Current distribution (week 24):

  • Unit (~70% target): 504 tests, 70.5% of suite, average run 65ms, flake rate 0.1%. Within SLO. Tools: pytest (backend), vitest (frontend).
  • Integration + contract (~20% target): 144 tests, 20.1% of suite, average run 750ms, flake rate 0.3%. Within SLO. Tools: Testcontainers (Postgres), Schemathesis (OpenAPI contracts), Pact (frontend → backend).
  • E2E + scenario (~10% target): 72 tests, 9.4% of suite, average run 18s, flake rate 0.4%. Slightly under target (intentional — quality over quantity). Tools: Playwright across 6 personas × top journeys.
  • Static (base layer): TypeScript strict + mypy strict + ESLint + Ruff, runs in < 30s on PR. Treated as the base layer; failures block merge.

Total suite p95 wall time: 14 minutes on PR. SLO breaches in the last quarter: 3 unit tests violated the 100ms threshold (all refactored within 2 weeks). Distribution health check is part of the quarterly QA retro.

Historical context: pyramid started inverted (week 1 was ~40% E2E because the team had a manual-E2E mindset before automation). Migration ran weeks 4–12: replaced ~30 low-value E2E with integration equivalents, added Schemathesis contract layer, added unit tests for refactored modules. Suite p95 dropped from 32 minutes to 14 minutes; flake rate dropped from 8% to 0.3%. The pyramid intervention paid back in week 8.

Quality bar

Senior bar

Target ratio codified in ADR; per-layer SLOs; tests tagged and routed. Distribution dashboard reviewed quarterly; ratio within ±5% of target. Static-analysis base layer wired. Inverted-pyramid signals trigger remediation. Shape adjusts to domain context.

Mid bar

Target ratio chosen; tests tagged by layer; per-layer SLOs in place; quarterly distribution review.

Junior bar

Can identify the layer of a given test (unit / integration / E2E), can write a unit test in the project's framework, and can explain why E2E-heavy suites are anti-pattern.
