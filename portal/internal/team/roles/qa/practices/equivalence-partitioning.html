Equivalence Partitioning — reduce redundant tests

Divides the input space into classes where one representative value per class gives the same defect coverage as testing every value in the class.

TL;DR

Equivalence Partitioning (EP), introduced by Glenford Myers in The Art of Software Testing (1979), divides every input into classes (partitions) within which all values are expected to be handled identically by the system. Testing one representative per partition gives the same defect-finding power as testing every value, at a fraction of the cost. EP works for any input dimension — numeric ranges, string formats, enum values, list sizes, file types — and is the canonical partner of Boundary Value Analysis: EP picks the partitions, BVA picks the boundary values inside them. Invalid partitions matter as much as valid ones; missing them is the most common EP failure mode. ISTQB ranks EP among the four core black-box techniques; for Study App it governs the test design of every API input, form field, and configuration value.

What it is

Equivalence Partitioning is a black-box test design technique that divides the input space into equivalence classes (partitions) where every value in a class is expected to produce the same behaviour, and selects one representative value per class as the test input. Glenford Myers introduced EP in The Art of Software Testing (1979) alongside BVA; ISTQB lists EP among the four core black-box techniques (with BVA, decision tables, and state transition). The technique applies to any input dimension: numeric ranges (positive / zero / negative; below threshold / above threshold), string formats (valid email / invalid email / empty string), enum values (each enum case), list sizes (empty / singleton / many), file types (allowed / disallowed / corrupt).

Partitions are valid (the system should accept and process them) or invalid (the system should reject them with a specific error). Both matter; missing invalid partitions is the most common EP failure mode. The minimum test set is one representative per partition; in combination with BVA, the set expands to representatives plus boundary values. EP coverage is measured as (tested partitions ÷ identified partitions) × 100%.

Why we need it

Without EP, test cases are picked by intuition or convenience and the input space is unevenly covered. Three tests in the same partition add zero coverage; one missing partition leaves a defect class undetected. EP forces explicit partition identification, so the test set is provably representative and the cost of testing is bounded (one test per partition, not one per value). Combined with BVA, EP makes the test design auditable: every partition is named, every representative is justified, gaps are explicit.

How others do it

Myers · canonical foundation

Glenford Myers introduced EP in The Art of Software Testing (1979) as the cornerstone of black-box test design. Myers framed EP via two rules: (1) for each input, identify the equivalence classes it imposes on the input domain; (2) cover each class at least once with one representative value. Boris Beizer's Software Testing Techniques (1990) extended EP to handle multi-dimensional inputs and the interaction between partitions of different inputs.

What we take: EP is the prerequisite for BVA — never apply BVA without first identifying the partitions; "min, max" is meaningless without knowing the partition it bounds.

Source: Myers — The Art of Software Testing

ISTQB Foundation Level

ISTQB's FL syllabus defines EP as one of four mandatory black-box techniques, distinguishes valid and invalid partitions explicitly, and defines EP coverage as a first-class metric. ISTQB also formalises the EP-BVA combination: BVA targets the boundary values within and around partitions identified by EP. The technique is examined on every FL sitting and is a prerequisite for ISTQB Advanced Test Analyst.

What we take: EP coverage (% of partitions tested) is the natural complement to line and branch coverage — report it.

Source: ISTQB — Foundation Level Syllabus

Property-based testing (Hypothesis / fast-check)

Property-based testing libraries (Python's Hypothesis, JavaScript's fast-check, Haskell's QuickCheck) automate EP-style sampling: each input strategy generates values from each defined partition, weighted by the partition's importance. The library biases toward partition representatives and boundary values, then samples the interior. Hypothesis's strategy combinators (one_of, integers(min, max), sampled_from) are EP in code; the library handles the bookkeeping and shrinks failing cases to a minimal representative.

What we take: property-based tests are automated EP; for high-risk inputs, write a Hypothesis/fast-check strategy that enumerates partitions explicitly rather than hand-writing five tests.

Source: Hypothesis Documentation

Stripe · input-domain mapping for SDKs

Stripe's SDK test infrastructure treats every API parameter as a partition map: valid currencies, invalid currencies, common amounts, edge amounts (zero, negative, max), null, undefined, the wrong type. Each partition has a documented expected outcome (which HTTP status, which error code). The SDK regression suite cycles through every partition for every parameter on every release across all 7+ SDK languages — millions of test cases generated mechanically from the partition map.

What we take: document the partition map for every API parameter once, and use it to generate tests in every SDK/client — the partition map outlives the language-specific test code.

Source: Stripe Engineering Blog

Anatomy of an EP analysis

Input inventory

Every input to the feature: API parameters, form fields, header values, configuration flags, environmental factors (locale, timezone).

Per-input partitions

For each input, the valid and invalid equivalence classes. A numeric range typically has 1 valid + 2 invalid; an enum has N valid + 1 invalid (off-enum); a string has formats.

Representative values

One per partition. Pick "middle" values (not near boundaries — BVA handles those). Document the rep next to the partition definition.

Expected outcomes

For each representative: what the system should return — accept, reject with code, fall back to default. Documented before execution.

Cross-input combination strategy

When multiple inputs each have N partitions, the full Cartesian product explodes. Strategies: one-factor-at-a-time, pairwise (combinatorial), or all-pairs (orthogonal arrays).

Coverage report

Tested partitions ÷ identified partitions, per release. Gaps are explicit risk-acceptance decisions, not silent omissions.

How to apply EP

  1. Inventory the inputs. For the feature under test, list every input the system accepts; include hidden ones (headers, timezone, locale) that affect behaviour.
  2. Read the spec for behaviour distinctions. What does the system do differently for input X vs. input Y? Each distinct behaviour defines a partition boundary.
  3. Define partitions. For each input, list valid partitions (system accepts and behaves distinctly) and invalid partitions (system rejects with a specific error). Number them P1, P2, … for citation.
  4. Pick representatives. One value per partition, away from the boundaries (BVA covers those). Document the representative on the partition entry.
  5. Pair with expected outcomes. Each representative gets a named expected response — HTTP status, error code, fallback behaviour. Documented before execution.
  6. Decide cross-input strategy. For multi-input features: one-factor-at-a-time for low-risk, pairwise for medium-risk, full Cartesian for the security/financial-risk inputs.
  7. Generate and execute tests. One parameterised test case per representative; layer BVA on top for boundary coverage; report EP coverage per release.

Deep dive

The conceptual claim of EP is that values within a partition are "equivalent" for defect-finding purposes — if the system behaves correctly for value V in partition P, it will behave correctly for every other value in P. This claim is heuristic, not mathematical: in practice some partitions hide internal sub-classes (e.g., a string-validation routine might handle ASCII and Unicode strings differently). The mitigation is granular partitioning: when a partition spans diverse cases (length, encoding, locale), split it into sub-partitions. The rule of thumb: if you can imagine a defect that affects one half of the partition but not the other, the partition is too coarse.

Valid and invalid partitions both matter, and invalid partitions are the most-skipped. A numeric field with range 1–100 has at least three partitions: invalid-low (< 1), valid (1–100), invalid-high (> 100). Many test plans cover only the valid partition, missing the rejection behaviour. The discipline is to enumerate invalid partitions as deliberately as valid ones — invalid behaviour is part of the spec, not a "if there's time" extra. Type-invalid partitions (string instead of integer, null instead of object) are especially important for API testing and are easy to forget.

Multi-dimensional EP and the combinatorial explosion is the central scaling challenge. Three inputs with 4 partitions each have 43 = 64 combinations; testing them all is rare. The standard mitigation is pairwise (2-way) combinatorial testing: ensure every pair of partition values across any two inputs is covered at least once. Pairwise typically reduces 64 combinations to 12–16 while catching ~80% of defects that full Cartesian would catch (per Kuhn's empirical research). Tools: PICT (Microsoft), ACTS (NIST), allpairspy. For security or financial code, use full Cartesian; for everything else, pairwise.

Equivalence partitioning and BVA are inseparable in practice. EP identifies the partitions; BVA tests the boundaries between them. A test plan with EP but no BVA has gaps at the partition edges (where defects cluster); a test plan with BVA but no EP has incorrect or incomplete boundaries (you don't know where the edges are). The standard recipe: do EP first (identify partitions), then BVA (six-point set per boundary), then optionally property-based testing (sample inside partitions for invariants). The three techniques are layers, not alternatives.

EP and decision-table testing differ subtly. EP operates on a single input dimension at a time: "for the age field, the partitions are minor, adult, senior." Decision-table testing operates across multiple boolean conditions: "for the eligibility decision, the combinations of (premium, complete, high-score) are 8 rules." EP is for input-domain partitioning; decision tables are for rule combinations. They are complementary: EP defines each input's partitions, decision tables combine partitions across inputs into testable rules.

Property-based testing automates EP. A Hypothesis strategy like one_of(integers(max_value=-1), integers(min_value=0, max_value=100), integers(min_value=101)) defines three partitions; the library samples from each. The benefit: thousands of values per partition explored automatically, partition definitions live in code (next to the test), and the library shrinks failing cases to the minimal representative. The cost: writing the strategy is more work than picking five hand-chosen values. Use property-based testing for inputs where the partition map is stable and defect cost is high (parsers, validators, authentication); use hand-written EP for one-off feature tests.

EP coverage as a metric distinguishes mature QA orgs from intuition-driven ones. The metric: (tested partitions ÷ identified partitions) × 100%, reported per feature, per release. Identified partitions come from the EP analysis; tested partitions come from the execution log. A field with 5 partitions where the test set covers 4 has 80% EP coverage. Combined with BVA coverage, branch coverage, and mutation coverage, EP coverage gives a complete picture of test-design adequacy — and it is the one most directly tied to "did we think about every input class?"

Common pitfalls

PitfallWhy it happensWhat to do instead
Skipping invalid partitions Tester defines valid partitions but forgets invalid-low, invalid-high, invalid-type — the rejection paths go untested. For every input, partition entry must include at least one invalid partition; the rejection behaviour is part of the spec.
Over-coarse partitions "Valid email" is one partition — but the system handles ASCII and Unicode emails differently; the rep covers only ASCII. If you can imagine a defect affecting one half of a partition but not the other, split the partition.
EP without BVA Tester picks "middle" values for each partition but no boundary values; defects at partition edges go undetected. EP identifies partitions; BVA tests their boundaries — always do both together.
Cartesian explosion accepted as-is Five inputs with four partitions each = 1024 tests; the team writes 20 and hopes. Use pairwise combinatorial testing (PICT, allpairspy) to get 80% of the coverage at ~10% of the cost.
Hidden inputs missed Inventory covers the visible API parameters but misses headers, locale, timezone, feature flags; partitions there go untested. The inventory step explicitly enumerates environmental factors; for web APIs, that includes Accept-Language, Accept, Authorization, feature flags, and tenant context.

Worked example — Study App quiz score input

The Study App API accepts POST /quizzes/{id}/attempts with body {score: integer, time_spent_seconds: integer, locale: string}. Spec: score must be 0–100 (integer); 0–49 = fail, 50–79 = pass, 80–100 = distinction. time_spent_seconds must be 1–3600. locale must be one of {en, de, ru, es}.

EP analysis:

  • score — 5 partitions: P1 invalid-low (score < 0, rep: −5), P2 fail (0 ≤ score < 50, rep: 30), P3 pass (50 ≤ score < 80, rep: 65), P4 distinction (80 ≤ score ≤ 100, rep: 92), P5 invalid-high or wrong-type (score > 100 or string, rep: 150 / "abc").
  • time_spent_seconds — 3 partitions: T1 invalid-low (< 1, rep: 0), T2 valid (1–3600, rep: 1200), T3 invalid-high (> 3600, rep: 5000).
  • locale — 5 partitions: L1–L4 valid (en, de, ru, es), L5 invalid (e.g. "zz" or empty).

One-factor-at-a-time test count: 5 + 3 + 5 = 13 tests. Pairwise (using allpairspy): 18 tests cover every pair of partitions across the three inputs. Full Cartesian: 5 × 3 × 5 = 75 tests. For Study App (medium-risk), pairwise is the right level; the suite runs in ~150 ms as part of the API contract tests.

BVA layer adds boundary values per partition: score 0, 1, 49, 50, 79, 80, 100 (~7 extra); time 1, 3600 (2 extra). Total test set ~27 cases; EP+BVA coverage 100%.

Outcome: in week 8, a refactor accidentally accepted locale = "EN" (uppercase) as valid (the validation was case-insensitive by mistake). The L5 invalid representative was "EN" (intentionally chosen as an "almost-valid" rep to test case-sensitivity); the test failed, catching the regression before merge. Without EP, the test set would have covered only the canonical "en" and the case-insensitivity bug would have shipped, breaking the locale routing in the CDN layer.

Quality bar

Senior bar

EP analysis authored for every feature; partitions documented next to the spec; representatives and expected outcomes named in code. EP coverage reported per release (≥ 95% on high-risk inputs). Pairwise combinatorial testing used for multi-input features; full Cartesian only for security/finance code. Property-based testing complements EP on stable, high-risk inputs. Hidden inputs (headers, locale, flags) included in the inventory.

Mid bar

EP applied to every API endpoint and form input; valid and invalid partitions both covered; representatives documented. EP coverage reported per release.

Junior bar

For a given input with a documented spec, can identify the valid and invalid partitions, pick representatives, and write one parameterised test per partition.
