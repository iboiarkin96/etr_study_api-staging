API Acceptance Testing — per-endpoint sign-off

A structured checklist pass that verifies every HTTP endpoint against its specification across six dimensions before it is marked QA-verified.

TL;DR

API acceptance testing is the QA sign-off gate for every HTTP endpoint, verifying six dimensions: contract conformance, happy-path responses, error codes (4xx/5xx), authentication gates, idempotency, observability. Driven by the OpenAPI spec (SWE Practice 12) — Schemathesis runs the contract fuzz; hand-written tests cover business edge cases; the API endpoint acceptance checklist is the acceptance criterion. Without it, endpoints reach production missing auth edges, with wrong error shapes, or dropped correlation IDs.

What it is

API acceptance testing is a per-endpoint sign-off that verifies six dimensions before QA marks the endpoint verified:

  • Contract conformance — response shape matches the OpenAPI schema. Schemathesis / Dredd fuzz the endpoint with valid + invalid inputs; assertion: every response validates against the declared schema.
  • Happy-path responses — 2xx shape correct, field semantics right, pagination cursors stable, timestamps in ISO-8601. Hand-written; covers business meaning the schema can't.
  • Error codes — 400 / 401 / 403 / 404 / 409 / 422 / 429 / 500 returned correctly; bodies follow RFC 7807 problem-details shape; error codes stable.
  • Authentication / authorisation — 401 without token, 403 with wrong scope, tenant isolation verified (user A can't read user B's data).
  • Idempotency — write endpoints accept Idempotency-Key header; same key returns same response; retry-safe for partial failures.
  • Observability — X-Request-ID in request flows to logs + traces; trace_id attribute on every log line; metric labels include endpoint + tenant.

Driven by the OpenAPI spec (SWE Practice 12 — contract-first). Schemathesis automates the contract dimension; the other five mix automation + hand-written tests + checklist review.

Why we need it

Endpoints reach production with subtle bugs in the dimensions developers aren't optimising for. The happy path works; everything else might not. Auth bypass for tenant B; 500 instead of 422 on validation failure; missing trace_id so incident MTTR balloons; idempotency-key not enforced so a retried checkout charges twice.

The six-dimension checklist forces every endpoint through the same systematic review. Schemathesis catches the contract dimension automatically; the checklist forces the human to look at the other five. Without it, each endpoint's edge-case coverage depends on the engineer's mood that week. With it, the bar is uniform across the API surface.

How others do it

Stripe — API testing at scale

Stripe's engineering blog documents the per-endpoint acceptance discipline: every endpoint has happy + error + auth + idempotency tests; the public Idempotency-Key header pattern Stripe popularised is now the industry default for safe retries on payments-like APIs.

What we take: Adopt Stripe's Idempotency-Key contract for every write endpoint. Same key → same response, even after partial failures. RFC 9457 / IETF draft codifies the pattern.

Source: stripe.com/blog/idempotency

Schemathesis — property-based fuzz

Schemathesis (Dmitry Dygalo, OSS) fuzz-tests a running service against its OpenAPI spec. Generates valid and invalid inputs; asserts every response validates against the declared schema; reports contract drift automatically. The canonical tool for the contract dimension in 2026.

What we take: Run Schemathesis in CI against staging for every PR. Drift between handler and spec caught before merge.

Source: schemathesis.readthedocs.io

Zalando — RFC 7807 problem-details

Zalando's API guidelines mandate RFC 7807 problem-details for every 4xx/5xx body. Stable shape, machine-readable; clients parse error codes reliably across services. Now the de-facto industry default for new HTTP APIs.

What we take: Adopt RFC 7807 error shape. Acceptance test asserts every 4xx/5xx body conforms; clients build on the shape, not on prose messages.

Source: Zalando RESTful guidelines

Postman / Bruno — collection runners

Postman collections (and the OSS Bruno alternative) host hand-written acceptance scenarios per endpoint. Newman runs them in CI; collections double as living documentation. Used by Twilio, GitHub, SendGrid for public-API acceptance + customer-facing examples.

What we take: Maintain a Postman / Bruno collection alongside the OpenAPI spec. Hand-written happy + error + auth scenarios; Newman in CI; collection also serves as docs.

Source: postman.com

Anatomy of API acceptance

OpenAPI spec as source

Contract-first (SWE Practice 12). Acceptance tests derive from the spec; drift fails the build.

Schemathesis contract fuzz

Property-based generation of valid + invalid inputs; assertion on response-schema match. Runs in CI against staging.

Six-dimension checklist (0002)

Contract, happy, errors, auth, idempotency, observability. QA reviewer signs off per endpoint.

RFC 7807 error contract

Every 4xx/5xx body conforms. Acceptance test asserts shape; clients build on machine-readable codes.

Idempotency-Key tests

Write endpoints: same key → same response. Retry-safe on partial failure. Test simulates network drop after server write.

Observability assertions

X-Request-ID echoes back; flows to logs + traces. Per Practice 16 (SWE) on structured logging.

How to run it

  1. Start from the OpenAPI spec. Contract-first (SWE Practice 12). Spec is the source.
  2. Wire Schemathesis in CI. Fuzz against staging on every PR.
  3. Author hand-written tests per dimension. Happy / errors / auth / idempotency / observability. Postman or in-language tests.
  4. Run the checklist per endpoint. Sign-off gate; QA reviewer signs the row before "verified".
  5. Adopt RFC 7807 + Idempotency-Key contracts. Standard shapes; reusable test helpers.
  6. Wire observability assertions. Tests check X-Request-ID + trace propagation.
  7. Quarterly review. Endpoints retroactively re-checked; checklist refined.

Deep dive — what makes the gate work

Schemathesis automates the contract dimension fully. Property-based input generation hits edge cases hand-written tests miss (empty strings, max-length, unicode, deeply-nested null). The schema is the source of truth; the running service is verified against it. Without this layer, schema drift is invisible until customers hit it.

RFC 7807 is the load-bearing default for errors. A consistent error shape lets clients build retry / display / triage logic against machine-readable fields, not prose. Hand-rolled error formats fragment the surface; clients special-case per endpoint. Adopt 7807 once; the entire client ecosystem benefits.

Idempotency-Key is non-negotiable for write endpoints. Network retries are inevitable. Without idempotency, a retried POST charges the user twice. Stripe popularised the header; the IETF draft has crystallised it. Acceptance test simulates partial failure (server write commits, network drops before response) and verifies the retry returns the same result.

Tenant-isolation auth tests are the highest-ROI security check. User A authenticating and reading user B's resource is the most common multi-tenant bug. Acceptance test mints A's token; calls the endpoint for B's resource; asserts 403. Cheap to write; catches a real class of incidents.

Observability assertions catch silent breakage. Missing trace_id in logs doesn't break a feature; it just makes incidents 10× slower to debug. Acceptance test asserts X-Request-ID echoes back and shows up in the log line for the request. Cheap; catches regression.

API endpoint acceptance checklist makes the bar uniform. Without it, each engineer's coverage depends on memory + mood. With it, the same six questions get asked per endpoint. The questions look obvious until you find the endpoint that missed three of them.

Pair acceptance with contract tests in SWE radar. SWE Practice 12 (contract-first) makes acceptance much cheaper — the schema already exists; the SDK is generated; the mock is available. QA acceptance is the final human review on top.

Lifecycle of acceptance

Schema landed

OpenAPI changes merged. Contract-first PR (SWE Practice 12). Generated SDK + mock available.

Schemathesis green

Contract fuzz passes against staging. No schema drift; no unexpected response shapes.

Hand-written tests pass

Happy / errors / auth / idempotency / observability scenarios green. Postman / in-language.

Checklist signed

QA reviewer walks the API endpoint acceptance checklist per endpoint; signs the row.

Verified

Endpoint marked QA-verified in the API catalogue; release sign-off can include it.

Re-checked on change

Schema change triggers re-acceptance. Quarterly retroactive sweep catches drift.

Worked example — Schemathesis + acceptance suite

Pipeline for the lesson-player API: contract fuzz via Schemathesis + hand-written Postman acceptance scenarios + the API endpoint acceptance checklist sign-off.

# .github/workflows/api-acceptance.yml (excerpt)
jobs:
  contract:
    steps:
      - run: schemathesis run \
          https://staging/openapi.yaml \
          --base-url https://staging \
          --checks all \
          --hypothesis-deadline=500
  acceptance:
    steps:
      - run: newman run postman/lesson-player.json \
          --env-var BASE=https://staging \
          --reporters cli,junit

# postman/lesson-player.json (excerpts — six dimensions)
- name: "happy · GET /lessons/{id}/state · 200"
  tests: |
    pm.test("schema match", () => pm.response.to.have.jsonSchema(state200));

- name: "errors · GET /lessons/{id}/state · 404"
  tests: |
    pm.test("RFC 7807", () => {
      const body = pm.response.json();
      pm.expect(body).to.have.property("type");
      pm.expect(body).to.have.property("title");
      pm.expect(body.status).to.eql(404);
    });

- name: "auth · GET /lessons/{id}/state · 401 / 403"
  tests: |
    pm.test("401 without token", () => pm.response.to.have.status(401));
    pm.test("403 cross-tenant", () => ...);

- name: "idempotency · POST /attempts · key replay"
  tests: |
    pm.test("same key → same response", () => ...);

- name: "observability · X-Request-ID echo"
  tests: |
    pm.test("X-Request-ID echoed", () => {
      pm.expect(pm.response.headers.get("X-Request-ID"))
        .to.eql(pm.environment.get("req-id"));
    });

Contract fuzz + hand-written six-dimension scenarios in one CI workflow. Newman exports JUnit for the release dashboard; the API endpoint acceptance checklist is signed by QA reviewer per endpoint before "verified".
