CI Quality Gates — automated pass/fail checks on every PR

Blocks merging when quality criteria are not met — making quality checks automatic and consistent rather than optional acts of discipline.

TL;DR

CI quality gates are automated pass/fail checks that run on every pull request and block merging when the criteria are not met: tests pass, contract specs lint clean, accessibility audit passes, coverage threshold met, visual regression approved, performance budget held, and security scan green. Each gate is fast (< 10 minutes for the merge-blocking set), reproducible, and produces an actionable failure message — not just a red build. Without automated gates, quality checks become optional under deadline pressure and the pipeline becomes decorative. Google, GitHub, Stripe, and Spotify all wire dozens of gates into branch-protection rules; for Study App, the merge-blocking set is unit + contract + lint + axe + coverage + visual + security, all running on the same Buildkite/GitHub Actions pipeline.

What it is

A CI quality gate is an automated check, wired into continuous integration, that produces a binary pass/fail result and is configured as a required status check in branch-protection rules. When a pull request is opened or updated, every gate runs (typically in parallel); the merge button is disabled until every required gate is green. Gates are distinct from "checks that exist in CI but aren't required" — the latter are reports, not gates. Gates encode the team's quality bar in executable form: if it isn't a gate, it isn't enforced.

The standard merge-blocking set for a modern web service: unit tests (100% pass), contract tests against OpenAPI/AsyncAPI (Spectral or Schemathesis), static analysis (ESLint, Ruff, TypeScript), accessibility (axe-core on built pages or storybook), code coverage threshold (line or branch ≥ a per-repo number), visual regression (Chromatic or Percy approval), security scanning (Snyk, Dependabot, Trivy), and performance budget (Lighthouse-CI or k6 smoke). Each gate has a single source of truth (a config file, a threshold) and an owner.

Why we need it

Quality checks that depend on developer discipline are skipped under deadline pressure — every team that has shipped under pressure has experienced this. Gates make discipline structural: the merge button is the only path to production, and the merge button is closed until quality criteria pass. Without gates the CI pipeline is decorative; with gates the pipeline is a control surface. The cost of a gate (10 minutes of CI time, one config commit) is dwarfed by the cost of a single defect reaching production that a gate would have caught.

How others do it

Google · presubmit checks

Google's monorepo enforces "presubmit checks" before any code can be submitted to the main branch: unit tests for affected targets, lint, format, build, and code review. The presubmit system runs on Bazel and is fully hermetic — each gate is reproducible and isolated. Software Engineering at Google documents the cultural rule: "if it isn't a presubmit, it isn't enforced." Google explicitly distinguishes presubmits (block submission) from periodic builds (alert but don't block) — the same distinction every modern CI setup must make.

What we take: the binary distinction between "blocks merge" and "reports only" is the single most important quality-gate decision; ambiguity kills enforcement.

Source: Software Engineering at Google — Continuous Integration

GitHub · branch protection & required status checks

GitHub's branch-protection rules let repository admins designate any GitHub Actions workflow (or external check via the Checks API) as "required." The merge button is disabled until every required check is green; admins can override with an explicit bypass that is audit-logged. GitHub publishes the API spec for the Checks API so external tools (Sentry, Codecov, Chromatic) can register as required gates. The required-status-check model is now the industry default.

What we take: wire every gate through the platform's required-status-check mechanism (GitHub Actions, GitLab Pipelines, Buildkite); never rely on "developer remembered to run it."

Source: GitHub — About protected branches

Stripe · keep main green

Stripe's engineering blog documents their "keep main green" policy: main is always deployable, every PR runs the full gate set, and a failed gate on main triggers a revert (not a forward-fix). Stripe reports that aggressive gate enforcement on main reduced their MTTR from hours to minutes and made every commit a deployable artefact. The team explicitly chose gate latency (CI cycle time) as a first-class metric and invested heavily in parallelisation and test sharding to keep the gate set under 15 minutes.

What we take: gate latency matters; aim for the merge-blocking set under 10 minutes so developers don't bypass gates "just this once."

Source: Stripe Engineering Blog

Spotify · Backstage tech-health checks

Spotify's Backstage includes a "soundcheck" or tech-health system: every service has a score computed from CI quality gates plus operational signals (test coverage, vulnerability count, on-call coverage, runbook quality). The score is visible service-wide and influences platform investment. Gates are not just per-PR — they are also organisation-level inventories: which services are below the bar, who owns them, what is the remediation plan.

What we take: expose gate pass-rates at the service level (not just per PR) so leadership can see "service X has 4 gates failing on main, owner Y" and prioritise.

Source: Backstage — Tech Health

Anatomy of a quality gate

Trigger

The event that starts the gate — pull-request opened/updated, push to main, scheduled run. Gates that don't run on PRs aren't merge gates.

Check command

A single command that exits 0 (pass) or non-zero (fail) — no human interpretation. The command is version-controlled in .github/workflows/ or equivalent.

Threshold

For metric-based gates (coverage, performance), the threshold is a number in a config file (codecov.yml, lighthouserc.json) — never hard-coded in scripts.

Required-status registration

The gate is named in branch-protection rules; the merge button is disabled until the named check is green. Without this, the gate is a report.

Actionable failure message

When the gate fails, the message tells the developer exactly what to do — which test, which line, which threshold, which fix. "Red build" is not enough.

Latency budget

Each gate has an owner-set time budget (e.g. unit ≤ 5 min, contract ≤ 3 min, axe ≤ 4 min); regressions trigger investigation.

How to add a new gate

  1. Decide what failure mode the gate prevents. "Blocks PRs that drop coverage below 80%" — concrete, observable. Vague reasons ("improve quality") signal a gate that won't survive review.
  2. Pick the check command. One command, exit 0 or non-zero, reproducible locally. pytest --cov --cov-fail-under=80 is a good command; "run the test suite and look at the output" is not.
  3. Set the threshold in a config file. codecov.yml, lighthouserc.json, .spectral.yml — never inline in the CI script. Thresholds change; PRs to thresholds get reviewed like code.
  4. Wire it into CI. .github/workflows/ci.yml with a named job. The job runs in parallel with other gates; failures produce a clear annotation.
  5. Mark it required in branch protection. The status name must match the job; without this, the gate is a report. Branch-protection changes are reviewed by the platform owner.
  6. Write the actionable error. The job's failure output explains the cause and the fix — "Coverage 79.2% < 80% threshold; new untested code in quiz/scoring.py:34-58; add tests in tests/test_scoring.py." Not just exit 1.
  7. Measure and tune latency. Track gate wall time over the next 2 weeks; if it climbs above the budget, shard the tests or move the gate to a nightly run.

Deep dive

The fundamental design choice is the binary required-vs-reporting distinction. A required gate blocks the merge button; a reporting gate posts a status without blocking. Teams new to gates often hesitate to mark anything required — "what if it's flaky?" — and end up with a CI pipeline full of yellow dots that no one reads. The cure is to fix flakiness first, then mark the gate required, in that order. A flaky gate is worse than no gate (it teaches developers to ignore CI). The required/reporting line is a constant negotiation as gates are added, stabilised, and promoted.

Gate latency is the second-most-important design choice. If the merge-blocking set takes 45 minutes, developers context-switch, lose state, and start opening "small" PRs to game the suite. If it takes 8 minutes, they wait, read the result, and iterate. Stripe, Google, and Shopify all publish gate-latency as a first-class engineering metric. The standard levers are parallelisation (run all gates concurrently, not sequentially), sharding (split a 20-minute test suite into four 5-minute shards), test impact analysis (only run tests that exercise changed code), and infrastructure (more CI runners is cheaper than the eng time saved). Aim for < 10 minutes on the merge-blocking set, < 30 minutes on the full set including reporting checks.

Coverage as a gate has a specific failure mode: enforcing a global threshold leads developers to write tests-for-coverage rather than tests-for-confidence. The mitigation is patch coverage (coverage on the lines changed in this PR, not the whole repo) plus a strict global threshold (don't let it slip). Codecov and Coveralls both support patch coverage. The right threshold is "don't drop" (no regression) plus "patch ≥ 90%" (new code is well-tested), not "global ≥ 80% forever" which rewards stagnation.

Accessibility as a gate is a specific success story: axe-core on built pages or Storybook catches the WCAG 2.1 AA basics (colour contrast, label associations, ARIA mis-use, missing alt text) at near-zero false positive rate. Wire axe into Playwright or Cypress and run it on every PR. The output is a list of specific violations with line numbers and remediation links. The British government's GOV.UK Design System enforces axe gates and reports a 70% reduction in accessibility defects reaching production. For Study App the gate is "axe finds zero serious or critical violations on the home, search, lesson, and quiz pages."

Contract gates are the highest-leverage QA gate: every PR that changes an API spec runs Spectral (lint) and Schemathesis or Pact (contract tests), and every PR that touches API code runs the same spec checks against the live test instance. The gate prevents breaking changes from reaching consumers and prevents spec/implementation drift. Zalando publishes their REST API guidelines as a Spectral ruleset; Stripe runs their entire SDK suite as a gate against the API. The pattern: spec is source of truth, gate enforces conformance, breakage is caught at PR time not at integration time.

Security gates have a known false-positive problem. Snyk, Dependabot, Trivy, and similar tools report vulnerabilities including transitive ones that may not be exploitable in your code path. The mitigation is severity-based gating (block on critical/high, alert on medium/low) plus a documented suppression workflow (suppress with a justification, audited monthly). Without suppression, the gate either becomes noise or gets disabled. With suppression, the gate stays useful — a critical/high vulnerability blocks the merge until it is patched or suppressed with a reviewed justification.

Bypass mechanisms are non-negotiable. Every gate occasionally has a legitimate reason to be bypassed (broken external dependency, false positive that takes a day to suppress, urgent rollback). The bypass must be audit-logged, named, and reviewed. GitHub supports admin override with a recorded decision; GitLab supports "force merge" with logging. The cultural rule: bypass is permitted, repeated bypass without a fix is escalated. Without a bypass mechanism developers wedge themselves into broken states; without audit they bypass routinely. Both are bad.

Lifecycle

1

Propose

Engineer or QA proposes a new gate with the failure mode it prevents, the check command, and the latency budget.

2

Pilot (reporting only)

Wire the gate as a reporting check for 1–2 weeks; observe false-positive rate and latency before promoting.

3

Promote to required

Once stable (FP rate < 1%, latency within budget), add to branch-protection rules as a required check.

4

Operate

Track pass-rate and latency weekly; investigate flaky gates within 24 hours of detection (flaky gates erode trust fast).

5

Tune

Adjust thresholds via versioned PRs; new code rarely fits a threshold designed for legacy code.

6

Retire

When a gate's failure mode is no longer real (technology replaced, risk eliminated), remove the gate; dead gates erode signal.

Common pitfalls

PitfallWhy it happensWhat to do instead
Gates exist but aren't required Team adds CI jobs but doesn't add them to branch protection — they appear as yellow dots and are routinely ignored. Every gate is either required (blocks merge) or explicitly classified as reporting. No third category.
Flaky gates erode trust A gate that fails 10% of the time for unrelated reasons trains developers to re-run CI rather than read the failure. Treat flakiness as a P1 bug; fix within 24 hours or demote the gate to reporting until fixed. Track per-gate flakiness as a metric.
Slow gates encourage bypass Merge-blocking set takes 45 minutes; developers force-merge or batch unrelated changes to amortise the wait. Set a wall-time budget (< 10 minutes) and treat overruns as a regression; parallelise, shard, or move to nightly.
Unactionable failure messages Gate prints "FAILED" with no context; developer has to dig through 1,000 lines of CI log to find the actual cause. Every gate emits a summary block: which check failed, where, the threshold, the fix. Use GitHub Actions annotations or equivalent.
No bypass mechanism A broken external dependency blocks every PR; team disables the gate globally and forgets to re-enable. Admin-bypass with audit log; bypass triggers a P2 follow-up to fix the gate; never disable a gate as the first response.

Worked example — Study App merge-blocking gate set

The Study App monorepo runs on GitHub Actions. Branch protection on main requires the following named status checks (all must be green to merge):

  • unit / backend — pytest, 100% pass, target < 4 min wall time.
  • unit / frontend — vitest, 100% pass, target < 3 min.
  • contract / spectral — lint all OpenAPI/AsyncAPI specs against Study App ruleset (Zalando-derived), zero errors.
  • contract / schemathesis — generative API tests against the spec running on the test instance, zero failures.
  • lint — eslint (frontend) + ruff (backend) + tsc (frontend) + mypy (backend), zero errors.
  • axe-a11y — Playwright runs axe-core on Home, Search, Lesson, Quiz pages, zero serious or critical violations.
  • coverage — Codecov patch ≥ 90%, global no-regression vs. main.
  • visual — Chromatic, designer approval required for diffs > 0.1% threshold on production-tier components.
  • security — Snyk on npm + pip dependencies, no critical/high CVEs unsuppressed.

Reporting (non-blocking) checks: perf / lighthouse-ci (alerts at threshold), mutation / mutmut (weekly trend), e2e / playwright-smoke (alerts on red main). All gates run in parallel; total wall-clock latency target is 8 minutes (current p50: 7.2 min, p95: 9.8 min).

Bypass policy: only the platform-team admin can bypass a required gate, the bypass is recorded in GitHub's audit log with a justification, and every bypass triggers an automatic P2 ticket to fix the underlying gate. In the last quarter, 4 bypasses occurred (3 for flaky axe-a11y after a Playwright upgrade, 1 for an emergency revert), all with documented follow-ups closed within 5 days.

Outcome: in week 14, a refactor accidentally dropped tracks/route.ts from the build and broke the lesson page. The axe-a11y gate failed because Playwright couldn't load the page; the visual gate failed because the rendered output diverged. The PR was blocked before merge. Without the gates, the defect would have reached the next staging deploy and been found by manual exploratory testing 4 hours later.

Quality bar

Senior bar

All merge-blocking gates are required via branch protection; the required set is wired into CODEOWNERS and audited monthly. Gate latency < 10 min p95; per-gate flakiness < 0.5%; bypass policy is documented, enforced, and audit-logged. New gates go through pilot→required lifecycle with measured FP rate. Coverage, contract, accessibility, visual, security, and performance gates all live; per-service gate-pass dashboards are visible to the org.

Mid bar

Unit + lint + contract + axe-a11y + coverage are required; visual and security run as reporting. Latency < 15 min. Bypass requires justification but no audit log. Flakiness tracked manually.

Junior bar

Unit tests and lint are required; failures block merge. Can add a new test gate via a CI workflow PR, point it at a clear pass/fail command, and explain what failure mode it catches.
