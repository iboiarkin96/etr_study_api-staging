Twelve-Factor App — portable, scalable, ops-friendly

Twelve rules from Heroku that make a service behave the same on a laptop, in CI, and on autoscaling production.

TL;DR

The Twelve-Factor App methodology, published by Adam Wiggins at Heroku in 2011, defines twelve rules for cloud-native services: one codebase, declared dependencies, config in env vars, stateless processes, logs as event streams, dev/prod parity, and more. Every modern PaaS and orchestrator (K8s, Cloud Run, Fly.io, Render, Heroku) assumes them; violations show up as ops pain. Retrofitting costs weeks per service — apply from day one.

What it is

The Twelve-Factor App defines twelve rules for cloud-native services, grouped:

  • Codebase + deps — I. one codebase, II. declared deps, V. build/release/run.
  • Config + backing services — III. env-var config, IV. URL-backed services, X. dev/prod parity.
  • Processes — VI. stateless, VII. port binding, VIII. concurrency via process model, IX. disposable.
  • Logs + admin — XI. logs as streams (stdout), XII. admin as one-off process.

The rules predate Kubernetes but every modern orchestrator + PaaS assumes them. A conformant service works on Heroku, Cloud Run, Fly.io, Render, Fargate, K8s and any future cousin without porting. A non-conformant one needs custom ops work per platform.

Why we need it

Violations show up as ops pain. Config baked into the image — can't change region without rebuild. State on local disk — autoscaling drops sessions. Logs to a file — central log backend doesn't see them. Each violation costs weeks to remediate per service when the platform changes; that change is inevitable as the team grows.

Following the rules from day one costs almost nothing per rule. Retrofitting costs weeks. The economic case is overwhelming for new services.

How others do it

Heroku / Adam Wiggins (2011)

Wiggins captured patterns Heroku had observed across thousands of customer apps. 12factor.net is short, mostly unchanged since 2011, and still load-bearing for every modern PaaS.

What we take: Read 12factor.net. The rules are battle-tested by 15 years of cloud-native deployment.

Source: 12factor.net

CNCF / Kubernetes — assumed default

Kubernetes pods + ConfigMaps + Secrets + the API-driven workload model assume twelve-factor by default. CNCF's Cloud Native Definition explicitly cites it as foundational.

What we take: On K8s, twelve-factor isn't optional. Violations work against the grain.

Source: CNCF Cloud Native Definition

Kevin Hoffman — Beyond 12-Factor (2016)

Hoffman's O'Reilly book extends the model with API-first, telemetry, and security/authentication factors. Reflects post-2011 learnings; doesn't replace the original.

What we take: Layer API-first (Practice 12), telemetry (Practice 17), explicit security on top. The base 12 are necessary; not sufficient for 2026.

Source: Pivotal — Beyond 12-Factor

Fly.io / Cloud Run / Render — modern PaaS

Modern PaaS layers all assume twelve-factor: push a conformant image; platform handles scaling, secrets, logs, deploys. Non-conformant images cost custom integration.

What we take: Follow the rules upstream so the PaaS can do its job. Saves weeks of integration per service.

Source: fly.io/docs

Anatomy by factor

I–II. One codebase, declared deps

One repo per service; deps in package.json / pyproject / go.mod. No system-libs creep.

III–IV. Env-var config + URL backing services

OS env vars; DB / cache / queue as URLs (DATABASE_URL). Same code; different URL = different env.

V. Build / release / run

Three stages. Build = compile; release = build + config; run = execute. Release immutable + numbered.

VI–IX. Stateless, port, concurrency, disposable

No local state; bind to env-PORT; scale via process model; SIGTERM-clean fast shutdown.

X. Dev/prod parity

Same DB engine + version + broker dev → prod. SQLite/Postgres mix is the antipattern.

XI–XII. Logs to stdout, admin one-off

App writes log lines to stdout; platform routes. Admin (migrations) run as one-off process types of the same image.

How to run it

  1. Start from a twelve-factor template. Conformant by default.
  2. Config from env only. No baked-in YAMLs with hostnames.
  3. Backing services as URLs. DATABASE_URL, REDIS_URL, KAFKA_URL.
  4. Stateless. Sessions in Redis / DB; uploads in S3.
  5. Logs to stdout. Practice 16.
  6. Build / release / run separation. Container build → release with config → run.
  7. Disposable processes. SIGTERM clean; sub-30-s startup.
  8. Quarterly twelve-factor audit. Scorecard per service.

Deep dive — what each pillar buys

Codebase + deps = reproducible builds. One repo + declared deps means the build is the same on the laptop, in CI, at deploy. System-lib creep is the antipattern that breaks deployments six months later.

Config + backing services = portability. Same code; different DATABASE_URL = different env. Portable across PaaS, K8s, laptop with no code changes.

Stateless = horizontal scale. No local state lets the platform kill / restart / replicate / scale-out without losing user data.

Disposable = autoscaling-ready. Fast startup + clean SIGTERM make autoscaling responsive + safe. Slow-startup services delay scale-out; unclean shutdowns drop in-flight requests.

Logs to stdout = central aggregation works. Platform routes stdout to the log backend. App-written log files break central aggregation; rotate-and-disk-write breaks autoscaling.

Admin one-off = no scattered scripts. Migrations / data fixes run as one-off process types of the same image. No "SSH into prod and run a script". Same image; different entrypoint; full audit trail.

Layer +3 (Hoffman) for 2026. API-first, telemetry, explicit auth/security. Original 12 still necessary; the +3 are non-negotiable in 2026.

Lifecycle of conformance

Bootstrapped

Service starts from a twelve-factor template; all 12 conformant.

Drift detected

PR adds local-disk write / baked config; quarterly audit flags.

Triaged

Owner classifies: real exception (with rationale + sunset) vs accident.

Fixed

Refactor PR brings back conformance.

Excepted

Justified breach with expiry date; auto-fail on expiry.

Audited annually

Twelve-factor scorecard per service; drift tracked.

Quality bar

  • All 12 factors conformant. Quarterly scorecard per service.
  • Config from env only.
  • Backing services as URLs.
  • Stateless processes. Verified by chaos / scale tests.
  • Sub-30-s startup + clean SIGTERM.
  • Logs to stdout. Structured (Practice 16).
  • Dev/prod parity. Same engine + version.
  • Beyond-12 also conformant. API first + telemetry + auth.
