Continuous refactoring — Boy Scout rule, no rewrite weeks

Refactoring is a daily activity, not a quarterly project; every PR leaves the code slightly better than it found it.

TL;DR

Continuous refactoring folds small, behaviour-preserving improvements into every PR — drawing from Martin Fowler's Refactoring (1999, 2nd ed. 2018) catalogue and Robert C. Martin's Boy Scout Rule: leave the code cleaner than you found it. The alternative is the great rewrite: a six-month engineering sink, a parallel codebase, no new features, an exhausted team. Continuous refactoring keeps maintenance cost approximately constant rather than letting it grow exponentially until a rewrite feels inevitable. Tests are the safety net; the IDE's safe operations are the tool; the PR template is the social glue.

What it is

Continuous refactoring is the discipline of folding small, behaviour-preserving code improvements into the normal flow of feature work. There is no "refactoring sprint" because there is no refactoring debt large enough to require one. Every PR makes the code marginally better — a clearer name here, an extracted function there, a deleted dead branch.

Two named ideas underpin it:

  • The refactoring catalogue. Fowler's Refactoring (1999; 2nd ed. 2018) is the source-of-truth list of behaviour-preserving moves: Rename Variable, Extract Function, Inline Variable, Replace Conditional with Polymorphism, Move Function. Each move is named, has defined mechanics, and is small enough to apply safely.
  • The Boy Scout Rule. Robert C. Martin's formulation: "Always leave the campground cleaner than you found it." Every PR includes at least one improvement to the file(s) it touches.

Continuous refactoring requires a strong test bar: refactoring without tests is just rewriting and hoping. Paired with TDD (practice 02), the green bar after every cycle is the safety net that makes the small moves safe.

Why we need it

The alternative is what Joel Spolsky called "the single worst strategic mistake any software company can make" — the great rewrite. A team that defers code improvement eventually reaches a point where the codebase is so brittle that no feature can be added without a parallel rewrite. The rewrite takes six months, ships no new value, exhausts the team, and produces a system with a different set of bugs.

Continuous refactoring routes around that cliff. Each PR pays a small voluntary cost; the cumulative effect is a codebase whose maintenance cost is approximately constant rather than exponentially growing. The team ships the same kind of change in week 100 as in week 1, without slowing down.

Good looks like: every PR contains at least one named refactoring move; cyclomatic complexity per file stays under a defined threshold; the team has not had a "refactoring sprint" or "tech-debt week" in two years because work never accumulates that high.

How others do it

Martin Fowler — the catalogue

Fowler's Refactoring (1999, 2nd ed. 2018) is the canonical reference. Refactoring is "a disciplined technique for restructuring existing code, altering its internal structure without changing its external behaviour". The catalogue lists dozens of named moves with their mechanics. The 2nd edition updates examples to JavaScript and prunes moves that modern IDEs make trivial. refactoring.com keeps the catalogue current.

What we take: Adopt the named-move vocabulary. "Extract Function" and "Inline Variable" become shared team language; the names are precise and the mechanics are documented.

Source: martinfowler.com — Refactoring (2nd ed.)

Robert C. Martin — Boy Scout Rule

Uncle Bob's Boy Scout Rule — "leave the campground cleaner than you found it" — appears across Clean Code and his blog. His "Cycles of TDD" post ties the rule to the refactor step: every red-green-refactor cycle is a Boy Scout opportunity. The cumulative effect over a year is a codebase touched many times by small improvements.

What we take: Make the Boy Scout Rule a PR-template line: "What did this PR leave better than it found it?" One trivial answer per PR shifts the team norm.

Source: cleancoder — Cycles of TDD

Shopify — refactor at monolith scale

Shopify has written extensively on managing one of the largest production Rails monoliths. Their approach combines static-analysis tooling (Sorbet for types, Packwerk for package boundaries) with cultural norms: every PR is expected to touch only what it needs and leave it in better shape. The multi-year Sorbet type-annotation campaign shows how incremental refactoring scales.

What we take: Tooling helps but does not replace culture. The daily PR-level discipline is what keeps a large codebase healthy; static analysis catches the regressions in between.

Source: shopify.engineering

Kent Beck — TCR & "make the change easy"

Kent Beck's TCR (test && commit || revert) workflow and his famous "make the change easy, then make the easy change" maxim formalise the discipline: before adding a feature, refactor the area so the feature becomes trivial. The refactor commit lands first; the feature commit lands second; both are safe because tests gate every step.

What we take: Before tackling a hard feature, refactor first to make the feature easy. Two small commits beat one large painful one. The refactor commit reads as "no behaviour change"; the feature commit reads as the feature.

Source: Kent Beck — Tidy First? newsletter

Anatomy of the practice

"Refactors included" PR section

PR template has a required field: list named refactorings included. One line per move. Even "Rename Variable: customerId → customerExternalId" counts. Visible expectation produces the behaviour.

Named-move vocabulary

Team uses the Fowler catalogue names: Extract Function, Inline Variable, Replace Conditional with Polymorphism, Move Field. Naming makes the change reviewable as a refactoring (zero behaviour change) rather than mixed with feature logic.

Refactor-only PRs for big moves

Large refactorings live in standalone PRs, not mixed with feature changes. Pure-refactor PRs are easy to review (large diff but unchanged behaviour; tests pass) and easy to roll back.

IDE-safe refactoring tools

Use IDE safe-rename, extract-function, inline-variable rather than hand-edits. The IDE updates every caller atomically; hand-edits miss one and break the build. JetBrains / VS Code with language servers / equivalent.

Test bar as the safety net

A refactor is only safe if tests cover the behaviour. Minimum bar: at least one characterisation test exercising the area's public contract. Without that test, the first move is to write it.

Visible code-health trend

Weekly chart: cyclomatic complexity per file (p90), files-without-tests count, comment-rot score. Trend matters more than absolute number. Flat or improving = continuous refactoring is working.

How to run it

  1. Add "refactors included" to the PR template. One required line per PR; even a single rename counts.
  2. Teach the named-move vocabulary. 30-min brown-bag introducing Fowler's top-10 moves. Pair with cheat sheet linked from the PR template.
  3. Wire code-health dashboard. Cyclomatic complexity p90 per file; files-without-tests; weekly trend in the engineering channel.
  4. Adopt "tidy first". Before a hard feature, land the refactor commit first (clearly labeled, zero behaviour change), then the feature commit.
  5. Pair refactor with TDD. Red → Green → Refactor every cycle. The third step is the practice's primary mechanism, not optional cleanup.
  6. Promote large moves to refactor-only PRs. When the refactor would obscure the feature in a review, split. Pure refactor PR ships first.
  7. Quarterly: review the top-3 most-complex files. Schedule a "tidy" PR each. Not a rewrite week; just three targeted PRs across the quarter.

Deep dive — what keeps the curve flat

The Boy Scout Rule is the load-bearing habit. Continuous refactoring is not a technique; it is a daily reflex. The PR template line ("what did this leave better?") is the social mechanism that turns the reflex into a team norm. Once 80% of PRs answer the line, the codebase trajectory inverts — the team becomes the kind of team that doesn't need rewrite weeks.

Test coverage is the safety net. Refactor without tests is just hopeful rewriting. The minimum bar before any refactor in a legacy area is a characterisation test that pins current behaviour. Adding the test is itself a Boy Scout move — it leaves the area more refactorable for the next engineer.

Tidy first is cheaper than tidy after. Kent Beck's "make the change easy, then make the easy change" inverts the usual order. Refactor first (commit A, zero behaviour change, tests green); add the feature second (commit B, the feature is now small and obvious). Two clean commits beat one tangled PR — and the reviewer can verify each independently.

IDE moves over hand-edits. The Fowler catalogue is a set of mechanically-safe moves; modern IDEs implement them as one-command operations that update every caller atomically. Hand-rolling a rename misses one caller and breaks main; the IDE rename is one command and one git diff. Use the tools; don't be a hero.

Refactor-only PRs are easier to review. A 600-line diff that's purely renames + extractions is faster to review than a 200-line mixed diff: the reviewer skims for "behaviour change?" answers "none", reads the test output, approves. Mixed PRs force the reviewer to mentally separate refactor from feature — slow and error-prone.

Watch the trend, not the number. Cyclomatic complexity p90 = 11 today vs 12 last quarter is a win regardless of the absolute number. A flat-or-improving curve over a year is the leading indicator of "continuous refactoring is actually happening". A growing curve, even slowly, is the leading indicator of an eventual rewrite.

Don't refactor under deadline pressure unless tests catch you. The temptation to "quickly clean this up" on the way to a deadline is when bugs ship. If tests aren't green, write a test first or skip the refactor — never both at once, never under pressure. The rule survives precisely because it has a stop condition.

Lifecycle of a refactor

Spotted

Engineer notices a smell while in the file for feature work: unclear name, long method, duplicate branch. Logs the move in mind.

Tested

If coverage is missing, write a characterisation test that pins current behaviour. Green bar before touching anything.

Applied

Run the named move via the IDE (rename / extract / inline). Tests stay green. Commit with the move name in the message.

Reviewed

PR template lists the move. Reviewer verifies behaviour unchanged (tests + diff scan). Approve fast — refactor-only PRs should land in hours, not days.

Trended

Code-health dashboard reflects the win. Quarterly: top-3 most-complex files reviewed and tidied.

Promoted to refactor-only PR

If the move grows beyond a few hundred lines or touches many files, split into a standalone refactor PR; feature follows in a separate PR.

Worked example — refactor-then-feature commits

A two-commit sequence from the lesson-player: a "tidy first" refactor lands before the feature commit. Each is reviewable independently; tests gate both.

# Commit A · refactor only · zero behaviour change

refactor(lesson-player): extract scoreAttempt body into AttemptScorer

- Extract Function: scoreAttempt -> AttemptScorer.score (Fowler)
- Move Field: difficultyWeight from Lesson into AttemptScorer
- Rename Variable: maxScore -> ceilingScore for clarity
- All existing tests green
- New characterisation test added for prior implicit
  null-on-missing-answer behaviour (pinned)

Refactors:
  - Extract Function (lesson_player/service.py:142)
  - Move Field      (lesson_player/models.py:88)
  - Rename Variable (lesson_player/scorer.py:31)

# Commit B · feature · uses the refactor

feat(lesson-player): partial-credit scoring for multi-step questions

- AttemptScorer now scores each step independently
- Uses the new AttemptScorer.score boundary from Commit A
- Feature-flagged behind player.partial_credit (default off)
- 3 new tests covering partial / full / zero credit paths

Refactors:
  - Rename Variable: stepResult -> partial in test fixtures
  - Boy-Scout: removed dead branch in legacy total-score path

Commit A: pure refactor, no behaviour change, tests still green, includes a characterisation test for the implicit prior behaviour. Commit B: the feature, made small because A pre-extracted the boundary. Reviewer can verify each independently; bisect can pin a regression to the right commit.

Quality bar

  • Refactor line on every PR. Required template field; even one trivial entry counts.
  • Named-move vocabulary used. Extract Function, Inline Variable, Move Field — names appear in commit messages and PR descriptions.
  • Refactor-only PRs for large moves. Pure refactor ships ahead of features touching the same area.
  • Characterisation tests before untested refactors. Minimum bar; no refactor without coverage.
  • IDE safe operations only. Rename / extract / inline via the IDE; hand-edits forbidden for these moves.
  • Code-health dashboard live. Complexity p90 per file + untested-file count, weekly trend in the engineering channel.
  • No "tech-debt sprints" needed. If the team needs one, the practice has lapsed — restore it before scheduling one.
