CI & green-trunk discipline — every commit deployable

Every engineer integrates with main multiple times a day, the full test suite runs on every change, and a red main is the team's top priority until it's green again.

TL;DR

Continuous integration, as Kent Beck described it in XP (1999), is not "we run CI on PRs". It is a team discipline: every engineer pushes to a shared trunk many times a day, every push runs the full build + tests, and if trunk is red, fixing it preempts feature work. The DORA Accelerate research identifies trunk-based development and a green main as two of the strongest predictors of elite delivery. Sub-10-minute CI on every PR, a required green-main gate, a merge queue, and a paging policy on prolonged red trunk are the minimum machinery.

What it is

Continuous integration is a working agreement plus machinery around a shared trunk. Engineers integrate small changes to main multiple times a day. On every push, one short pipeline runs: format check, lint, type-check, unit tests, integration tests, container build. The pipeline is the merge gate; nothing reaches trunk without it going green.

Green-trunk discipline is the social contract layered on top. When trunk goes red — a flake, a bad merge, an environmental failure — fixing it is the highest-priority task for the team that broke it. No new features merge on top. A red main older than 15 minutes pages an on-call dev, the same way a red production graph pages an SRE.

"Every commit deployable" means main is never knowingly in a state that could not safely ship. Half-built features hide behind feature flags (see practice 07), not behind long-running branches. Trunk-based development with short-lived (< 24 h) feature branches is the working pattern; long-running feature branches are explicitly avoided.

Why we need it

A long-running red main is a multiplicative tax on the team. Every engineer who tries to rebase, merge, or deploy on top of a broken trunk loses minutes to confusion ("is this my failure?"), and the first response is to ignore CI signals — which is the failure mode that turns elite teams into medium-performing ones.

The DORA Accelerate research, repeated yearly since 2014, shows that trunk-based development with green-main discipline correlates with elite outcomes on all four DORA metrics: deploy frequency, lead time, change-failure rate, time to restore. The mechanism is simple — small, frequent integrations expose conflicts at the cheapest possible moment, before they compound. CI is the only sustainable way to ship fast and ship safely.

How others do it

Google — TAP + Bazel

Google runs a single monorepo where every change is tested against the entire affected dependency graph by TAP (Test Automation Platform). Bazel's hermetic, incremental builds make a full-graph rerun feasible on every submission. Trunk is green by construction — a CL that would break the build is not even allowed to land.

What we take: Enforce "every commit deployable" upstream of merge, not policed afterwards. The build system, not human judgement, is the gate. Investing in fast, deterministic builds pays back as the team scales.

Source: Martin Fowler — Continuous Integration

GitHub — per-PR CI + merge queue

GitHub runs the full test matrix on every PR via GitHub Actions, with required status checks blocking merge until green. Their engineering blog describes a strict policy of small PRs (target < 400 lines), short-lived branches (hours, not days), and a merge queue that re-runs CI against the latest main before the actual merge — eliminating semantic conflicts.

What we take: A merge queue closes the "passed CI three hours ago, main has since moved" gap. It is the simplest fix for the otherwise-common "PR was green, merge broke main" failure mode.

Source: github.blog — engineering posts

DORA — Accelerate research

The DORA "State of DevOps" reports (now under Google Cloud) consistently find that elite performers do trunk-based development with short branches and have a CI pipeline that runs in < 10 minutes. Low performers either don't have CI or integrate only nightly. The change-failure-rate gap between elite and low performers is 5× or more.

What we take: CI duration is a leading indicator. When CI passes the 10-minute mark, engineers context-switch away while waiting and then back to a stale problem; the cost compounds. Keep the pipeline sub-10-minute even if it requires test sharding or selective execution.

Source: dora.dev/research

Spotify — Backstage + scorecards

Spotify operationalises CI hygiene through Backstage scorecards: each service is graded on CI duration, flake rate, branch lifetime, deploy frequency. Teams below threshold get visibility (and gentle pressure) from the platform team. The scorecard turns CI hygiene from a per-team discretionary thing into a measurable, comparable, improvable signal across the org.

What we take: Publish CI scorecards (or an equivalent dashboard) at the service level: pipeline duration, flake rate, branch lifetime, time-to-green. Comparison makes drift visible early.

Source: Backstage — Spotify open-source platform

Anatomy of green-trunk CI

Sub-10-minute pipeline

Wall-clock from push to mergeable signal < 10 min p95. Parallelise test shards; cache dependencies; skip unaffected paths via path filters or test-impact analysis.

Required green-main check

Branch protection rule: merge blocked unless all required checks pass. No "merge anyway" button; admin bypass logged and reviewed.

Merge queue

PRs re-run against the latest main before the actual merge. Eliminates the "passed CI hours ago, semantic conflict landed" class of break. GitHub merge queue or equivalent.

Red-main paging

If main is red > 15 minutes, the #ci-alerts channel pages whoever pushed the breaking change. Fix or revert; no third option. Revert is the default if fix isn't ready in 30 minutes.

Flake budget

Track flake rate per test. Any test flaking > 1% over a week is quarantined and owned. Quarantine is a JIRA with an owner and due date, not a silent retry.

DORA telemetry

Publish deploy frequency, lead time, change-failure rate, time-to-restore monthly. Trend, not point value, is what matters. CI scorecard visible to all engineers.

How to run it

  1. Branch protection on main. Required status checks: lint, type, unit, integration, build. No force pushes; no admin merges except for emergencies (logged).
  2. One pipeline, < 10 min wall-clock. Parallel shards; dependency cache; path filters so a one-file change doesn't trigger the universe.
  3. Short-lived branches (< 24 h). Long-running features behind feature flags. PR target ≤ 400 lines changed.
  4. Enable the merge queue. PRs re-run against latest main before merging. Closes the semantic-conflict window.
  5. Red-trunk paging. > 15 min red → page the breaker. Revert is the default response if fix isn't immediate.
  6. Quarantine flakes weekly. > 1% flake rate → JIRA + owner + due date. No silent retries.
  7. Publish DORA scorecard. Monthly. Deploy freq, lead time, change-failure rate, time-to-restore per service.
  8. Review CI hygiene quarterly. Top-3 slowest tests, top-3 flakiest tests, longest-lived branches — each gets an owner and a fix.

Deep dive — what makes the discipline hold

Speed is the load-bearing axis. The DORA finding isn't subtle: pipelines > 10 minutes correlate with worse change-failure rates. The mechanism is human — engineers context-switch while CI runs, lose state, and come back to a stale problem; small fixes pile up; the team learns to batch changes and integrate less often. Treat CI duration as a service-level objective with the same rigour as a production SLO.

The merge queue closes a real gap. A PR that passed CI at 10:00 against main@SHA1, then was merged at 13:00 after main moved to SHA9, may semantically conflict in ways the diff doesn't show. The merge queue re-runs CI against the merged state before completing the merge. GitHub merge queue, Bors, Mergify — all implement the same idea. Without it, "main is green" is only true on average.

Flakes are not noise — they are broken tests. Auto-retry on flake teaches the team to ignore failures. Quarantine after > 1% flake rate; create a JIRA with an owner and due date; remove from required checks until fixed. The team that protects its flake budget keeps CI signal high; the team that doesn't ends up with red main being indistinguishable from green.

Revert is the default, not the exception. When main goes red, the right answer is "revert in five minutes; investigate in twenty". A fix that takes longer than 30 minutes is almost always a worse choice than revert + redo. Codify revert-first in the runbook; it removes the social cost of reverting someone's change.

Branch lifetime is the leading indicator. Long-lived branches accumulate semantic conflicts that no CI can catch until merge. < 24 hour branch lifetime is the threshold; week-long branches are a warning sign; multi-week branches are an antipattern. Pair with feature flags to hide unfinished work in main.

Investment in the build system pays compounding interest. Bazel-style hermetic builds, dependency caching, test-impact analysis — each shaves minutes off CI and multiplies across every engineer's day. A 5-minute saving across 30 engineers running CI 5× a day is 750 engineer-minutes daily. Build infra is the highest-ROI platform investment most teams underfund.

Scorecards turn culture into data. A scorecard with pipeline duration, flake rate, branch lifetime, deploy frequency per service makes drift visible early. Backstage's templates are the easy path; a simple Grafana board works for smaller orgs. Comparison creates gentle, sustained pressure to improve.

Lifecycle of a change through CI

Local commit

Engineer runs format + lint + relevant tests locally via pre-commit hooks. Push when local checks pass.

PR opened

CI runs the full pipeline. PR cannot merge until all required checks are green. Reviewer assigned via CODEOWNERS.

In merge queue

PR enters the queue. Re-runs CI against latest main. Semantic conflicts caught at this step, not after merge.

Merged

Squash to main. Deploy pipeline triggers. Trunk remains deployable by invariant.

Red trunk (if it happens)

Page within 15 min. Revert is the default. Fix-and-roll-forward only if < 30 min ETA. Postmortem if > 1 h.

Deployed

Container image promoted through environments. Telemetry updates DORA dashboard. Lead time tracked end-to-end.

Quality bar

  • Pipeline duration p95 < 10 min. Sub-budget per stage; sharding + cache + path filters in place.
  • Branch protection on main. Required checks; no admin bypass without an audit trail.
  • Merge queue enabled. Re-runs CI against latest main before merging.
  • Branch lifetime < 24 h median. Feature flags for in-progress work.
  • Flake rate < 1%. Quarantine workflow with owner + due date; no silent retries.
  • Red main > 15 min pages. Revert-first runbook; postmortem for > 1 h reds.
  • DORA scorecard monthly. Deploy freq, lead time, change-failure rate, TTR per service.
  • Quarterly CI review. Top-3 slowest + flakiest tests get an owner each.
