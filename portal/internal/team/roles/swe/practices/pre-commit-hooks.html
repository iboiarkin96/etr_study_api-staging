Pre-commit hooks — fast feedback before push

A bundle of fast, deterministic checks runs on staged files at commit time. The same hooks re-run in CI, but most issues never reach the PR.

TL;DR

Pre-commit hooks shift fast feedback to the cheapest moment: before the commit is even written. The pre-commit framework (pre-commit.com, Anthony Sottile, 2014) is the de-facto standard; Husky and lefthook are JS/Node equivalents. A typical bundle: formatter, linter, type-check on staged files, secret scan (gitleaks), conventional-commits validator. Same hook bundle runs in CI as the source of truth — laptops and CI never disagree.

What it is

A pre-commit hook is a script git runs before completing a commit; if it exits non-zero, the commit is aborted. The pre-commit framework (pre-commit.com) makes hooks reproducible: a .pre-commit-config.yaml file declares which hooks to run on which file types; pre-commit install wires them into .git/hooks. Hooks run on staged files only, so commits stay fast.

A typical bundle:

  • Format — Prettier (JS/TS/CSS/MD), Black (Python), gofmt, rustfmt. Auto-fix + re-stage so engineers never argue about style in review.
  • Lint + type-check on changed files — eslint, ruff, mypy, clippy. Staged subset only; full check stays in CI.
  • Secret scan — gitleaks / detect-secrets. Blocks AWS keys, GitHub tokens, generic API keys at the gate.
  • Commit-message lint — commitlint enforces Conventional Commits (Practice 09).

The hook bundle is also wired into CI as a required check, so engineers who bypass with --no-verify get caught the next day. The local hook is for fast feedback; CI is the source of truth.

Why we need it

Review time is expensive; CI time is slow. The cheapest place to catch a missing semicolon or an inconsistent indent is before the commit is even written. Pre-commit hooks turn "format / lint / type-check / secret-scan" into a 3–5 second local step that prevents the most common review noise: indent comments, missing imports, accidental console.logs, leaked secrets.

Beyond noise reduction, the same bundle in CI ensures laptop and CI never disagree. If the hook passes locally but CI fails, the team has a configuration bug — and fixes it. If the engineer bypasses with --no-verify, CI catches it. The dual gate is what makes the discipline scale to engineers who don't read the README.

How others do it

pre-commit.com — Anthony Sottile, 2014

The pre-commit framework is the de-facto cross-language standard. Hooks declared in YAML, installed with one command, run on staged files only. Massive ecosystem of community-maintained hook repos (1 000+) covering every major language and tool.

What we take: Adopt pre-commit for polyglot repos. JS-only repos can use Husky / lefthook, but the cross-language ergonomics of pre-commit win at scale.

Source: pre-commit.com

Husky / lefthook — JS-native

Husky (Tysoo, JS) and lefthook (Evil Martians, Go) are the JS-native hook managers. Tighter integration with package.json + npx. lefthook ships parallel execution out of the box, which matters at large bundle sizes.

What we take: JS-only repos: Husky or lefthook. Mixed-language monorepos: pre-commit framework. Don't mix two managers; pick one and standardise.

Source: typicode.github.io/husky

gitleaks — secret scanning at the gate

gitleaks (Zach Rice, OSS) detects committed secrets via regex + entropy heuristics. Runs as a pre-commit hook and in CI. Catches AWS keys, GitHub PATs, generic API tokens before they leave the laptop. Used by hundreds of orgs including major OSS projects.

What we take: Secret scan in the bundle. The 200 ms cost prevents a class of incidents that take days to remediate (rotate every key the leaked credential could have touched).

Source: github.com/gitleaks/gitleaks

GitHub — Husky + commitlint as docs default

GitHub's own engineering blog and many OSS templates ship with Husky + commitlint + prettier as defaults. The pattern is mainstream enough that "modern repo" implies "has a hook bundle". Conventional Commits compliance and Prettier formatting are now expected, not optional.

What we take: Ship the bundle as part of the repo bootstrap. New repos get .pre-commit-config.yaml + Husky + commitlint by template, not by individual engineers remembering to add it.

Source: github.blog

Anatomy of a hook bundle

.pre-commit-config.yaml

Hook bundle declared in YAML. Versioned with the repo. One source of truth for both laptop and CI.

Format → lint → type → secret → commit-msg

Standard order. Format first (auto-fix); lint + type-check next; secret-scan; commit-message lint last.

Staged-file scope

Hooks run on --staged changes only. A 10 000-file repo doesn't slow each commit. Full checks live in CI.

Total budget < 5 s

Faster hooks = engineers don't bypass. > 10 s and the --no-verify habit forms. Profile and optimise the slowest hook.

Same bundle in CI

pre-commit run --all-files in CI as a required check. Laptop bypass caught the next day; canonical answer matches local.

Onboarding: one command

README: "run pre-commit install after clone". New hires get the bundle automatically; devcontainer does it as part of post-create.

How to run it

  1. Pick a manager. pre-commit.com (polyglot); Husky / lefthook (JS).
  2. Write .pre-commit-config.yaml. Format + lint + type + secret + commit-msg.
  3. Wire pre-commit install to devcontainer post-create. New hires get it without thinking.
  4. Add CI step. pre-commit run --all-files as a required check. Canonical gate.
  5. Profile and stay under 5 s. Slow hook = bypass habit. Optimise or move to CI-only.
  6. Document the bypass rule. --no-verify is allowed for emergencies; the next CI run catches it.
  7. Quarterly hook audit. Add new useful hooks; retire ones that fire only on false-positives.

Deep dive — what makes hooks actually catch things

Speed is non-negotiable. Engineers bypass slow hooks. A > 10 s pre-commit creates the --no-verify habit; once a team has that habit, hooks stop catching anything. Profile your bundle; the slowest hook usually has a faster variant (e.g. ruff replaces flake8 + isort + pyupgrade at 10× speed).

Staged files only — full check stays in CI. A "lint everything" hook on a 10 k-file repo takes 30 s per commit. pre-commit defaults to staged files; configure custom hooks the same way. Full-repo lint runs in CI on PR, not on every commit.

Format first, auto-fix, re-stage. Pretty-printers shouldn't fail; they should fix. The hook auto-fixes the staged file and re-stages it. Engineers don't argue about style; review never has indent comments.

Secret scan is the highest-ROI hook. Cost: 200 ms. Saving: a credential-leak incident takes 24–48 hours to remediate (rotate every key the leaked credential could have touched). gitleaks in the bundle prevents the class entirely.

Same bundle in CI catches bypasses. Local hook is fast feedback; CI is the canonical check. --no-verify is allowed in emergencies; the next CI run flags the missed check. The dual gate scales to engineers who don't read the README.

Hook config is part of the bootstrap. New repos get .pre-commit-config.yaml from the template. Devcontainer post-create runs pre-commit install. New engineers don't choose whether to opt in; the env opts them in.

Retire useless hooks aggressively. A hook that never fires (or fires only on false positives) is noise. Quarterly audit removes dead hooks. A small useful bundle beats a large noisy one.

Lifecycle of a hook

Proposed

Engineer suggests a new hook in a PR. Speed measured; useful-fire-rate justified.

Soaked locally

Hook added in warn-only mode for 1–2 weeks. Engineers see fires; team confirms it catches real issues without noise.

Enforced

Hook becomes blocking. Same hook added to CI as required check.

Bypassed (rare)

--no-verify for an emergency. CI re-fails the next push; engineer fixes; postmortem if recurring.

Audited

Quarterly: useful-fire-rate per hook. Hooks with 0 useful fires in a quarter get reviewed for removal.

Retired

Hook removed from config. CI step deleted. Replacement noted in the PR.

Worked example — Study App pre-commit config

The repo-root .pre-commit-config.yaml plus the matching CI step. Total local time < 4 s on a typical PR.

# .pre-commit-config.yaml
repos:
  # format
  - repo: https://github.com/pre-commit/mirrors-prettier
    rev: v3.3.3
    hooks:
      - id: prettier
        types_or: [javascript, typescript, json, yaml, markdown, css]
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.6.4
    hooks:
      - id: ruff-format
      - id: ruff
        args: [--fix, --exit-non-zero-on-fix]

  # type
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.11.2
    hooks: [ { id: mypy, additional_dependencies: [types-requests] } ]
  - repo: local
    hooks:
      - id: tsc
        name: tsc --noEmit (incremental)
        language: system
        entry: npx tsc --noEmit
        pass_filenames: false

  # secrets
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.4
    hooks: [ { id: gitleaks } ]

  # commit-msg
  - repo: https://github.com/alessandrojcm/commitlint-pre-commit-hook
    rev: v9.16.0
    hooks: [ { id: commitlint, stages: [commit-msg], additional_dependencies: ['@commitlint/config-conventional'] } ]

# .github/workflows/lint.yml (CI mirror)
jobs:
  pre-commit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
      - uses: pre-commit/action@v3.0.1
        with: { extra_args: --all-files }

# devcontainer.json post-create
"postCreateCommand": "pip install pre-commit && pre-commit install --install-hooks"

Format (Prettier + ruff-format), lint (ruff), type (mypy + tsc), secrets (gitleaks), commit-msg (commitlint). Same config drives CI via pre-commit/action. Devcontainer post-create installs hooks automatically — new hires never miss it.

Quality bar

  • Hook bundle in .pre-commit-config.yaml. Versioned with the repo.
  • Total local time < 5 s on staged subset. Profiled.
  • Format auto-fix + re-stage. Engineer never sees prettier complaints; review never has indent comments.
  • Secret scan in bundle. gitleaks / detect-secrets.
  • commitlint on commit-msg. Conventional Commits enforced.
  • Same bundle in CI. pre-commit run --all-files required check.
  • Devcontainer installs hooks. Post-create runs pre-commit install.
  • Quarterly hook audit. Useless hooks retired; useful ones added.
