Feature flags — decouple deploy from release

The deploy installs the code; the release flips the flag. Two different decisions, two different rollback paths, two different signals.

TL;DR

A feature flag is a runtime switch that gates a code path. Popularised at Facebook (Gatekeeper) and Flickr in the 2000s; commercialised by LaunchDarkly from 2014. Pete Hodgson's 2017 essay on martinfowler.com is the canonical reference. With flags, "deploy" stops meaning "release to users". Code ships dark, then is enabled for an internal cohort, then a percentage, then everyone — each step independently reversible by a config flip, not a redeploy. Large orgs run with hundreds of active flags plus an explicit lifecycle so flags don't accumulate forever.

What it is

A feature flag (or feature toggle) is a runtime switch that gates a code path. Three common flavours:

  • Release flags — gate a new feature; lifecycle is days to a couple of weeks; removed after launch.
  • Experiment flags — random assignment for A/B testing; lifecycle is the experiment duration; removed when the experiment concludes.
  • Ops flags — kill-switches and circuit breakers; live indefinitely as operational levers.

Pete Hodgson's canonical taxonomy adds permission flags (entitlements / per-plan features) as a fourth category that stays long-lived by design. The patterns are similar; the lifecycles differ. Release flags are the most common and the most often badly maintained — they ship and then never get cleaned up.

Why we need it

Without flags, "deploy" = "release". The team holds the deploy until the feature is perfect; the release is one big-bang event; rollback means revert + redeploy + wait. With flags, "deploy" = "code in production, off" and "release" = "flip the flag, gradually". The two decisions are independent; either can roll back without the other. Code can ship mid-feature behind a flag without anyone noticing.

Flags also enable canary releases (5% → 25% → 50% → 100%), tenant-specific rollouts (enterprise tenants first), and experimentation (A/B for KPIs). The same primitive covers all three. The cost is real — every flag is forking code — so flag lifecycle matters as much as flag creation.

How others do it

Facebook / Flickr — pattern originators

Facebook's "Gatekeeper" (2009-ish) and Flickr's flag-driven deploys popularised the pattern. Both shipped multiple times a day on top of a permanent feature-flag infrastructure; new features rode behind flags by default; the deploy was a non-event.

What we take: Treat "ship behind a flag" as the default for any non-trivial feature. The cost is one wrap; the optionality is enormous.

Source: Pete Hodgson — Feature Toggles (canonical)

LaunchDarkly — vendor that defined SaaS

LaunchDarkly (2014) built the canonical commercial feature-flag platform: SDKs in every language, percentage rollouts, targeting rules, experiment integration, audit log. Most large engineering orgs that don't roll their own use LaunchDarkly or one of its competitors (Statsig, Flagsmith, ConfigCat, Unleash).

What we take: Buy, don't build, for non-trivial scale. Self-hosted (Unleash) for cost-sensitive teams; SaaS (LaunchDarkly, Statsig) for the polished feature set.

Source: launchdarkly.com

GitLab — flag lifecycle policy

GitLab's public handbook documents a strict feature-flag lifecycle: every flag has a default expiry (typically 2 months for release flags), an owner, and a "remove" issue created at the same time as the "add" issue. Stale flags page the owner; the cleanup is enforced, not aspirational.

What we take: Each new flag ships with a "remove" issue + expiry date. Without the expiry, flags accumulate; with it, the backlog stays finite.

Source: about.gitlab.com/handbook

Statsig / Optimizely — experimentation

Statsig (founded by ex-Facebook engineers) and Optimizely combine feature flags with experiment analytics: random assignment, exposure logging, KPI lift computation. The same flag can be a release toggle today and an experiment tomorrow with a config change.

What we take: If you run A/B tests, pick a vendor that unifies flags + experiments. Two-vendor setups create alignment problems; one vendor with both gives you the same audience model end-to-end.

Source: statsig.com

Anatomy of a feature-flag system

SDK per language

Backend services, frontend bundle, mobile apps. Streaming or polling; sub-second propagation.

Targeting rules

Percentage, tenant, geo, user attribute, flag-dependency. Rules versioned + auditable.

Flag registry with owner + expiry

Every flag has a named owner, an expiry date, a "remove" issue. Stale flags surface in dashboards + page owners.

Audit log

Every flip (who, when, what value, what rule) recorded. Postmortems can answer "did anything change at 14:02?" precisely.

Kill-switch hierarchy

High-level ops flags ("disable payments") + per-feature release flags. Ops flags live forever and have explicit owners on the SRE team.

Cleanup PR template

"Remove flag X" PRs follow a template: delete config, delete code path, run lint check for orphan refs, link to the original "add" issue.

How to run it

  1. Pick a platform. LaunchDarkly / Statsig (SaaS); Unleash / OpenFeature + own backend (OSS).
  2. Install SDKs. Every service + frontend + mobile. Streaming preferred; sub-second propagation.
  3. Add a flag registry. Owner, expiry, "remove" issue, type (release / experiment / ops / permission).
  4. Ship behind flag by default. Any non-trivial feature starts with a flag, off, in main. Code can ship daily.
  5. Ramp progressively. Internal → 1% → 5% → 25% → 100%. SLO + KPI watch at each step.
  6. Soak at 100%. 1–2 weeks with the flag still in code; rollback path preserved.
  7. Clean up. Delete flag + delete code path. PR template enforces both. Expiry triggers a page if missed.
  8. Quarterly flag audit. Stale flags listed; owners contacted; orphans removed.

Deep dive — what makes flags scale

Cleanup is the load-bearing discipline. Every new flag is forking code. Without cleanup, after a year the codebase has hundreds of forks, all of which must be considered when reasoning about behaviour. The single most-cited failure mode in feature-flag adoption is "ramp to 100%, never delete". Solve it with: a "remove" issue created at the same time as "add", expiry dates on every flag, paging on stale flags.

Buy, don't build, at scale. A simple in-house flag system is two days of work; a system with audit logs, streaming SDKs, percentage rollouts, targeting rules, experiment analytics, dependency rules, and PII-safe targeting is a small product. Use LaunchDarkly / Statsig / Unleash and spend the saved months on actual features.

Ship dark by default. Any non-trivial feature begins with a flag, off, in main. Code lands daily; the team isn't blocked on "feature complete"; rollback is a flip, not a redeploy. The pattern eliminates the "long-lived feature branch" antipattern simultaneously.

Audit log is the time machine. Postmortems need "what changed at 14:02?". Without an audit log, deployment + flag + config + secret changes all blur together. With an audit log, the team finds the flip in seconds and the root cause in minutes.

Ops flags are forever. Distinguish release flags (lifecycle: weeks) from ops flags (lifecycle: indefinite). "Disable payments" should be an ops flag owned by SRE; "show new checkout layout" should be a release flag owned by the feature team. Mixing the two leads to release flags accumulating + ops flags being deleted by mistake.

Ramp by impact, not by minutes. The right ramp is "wait until SLOs + KPIs have one full hour of normal traffic at this percentage", not "ramp every 15 minutes on a timer". Some features need a full day at 5%; some need an hour. Codify the watch criteria, not the schedule.

Soak before cleanup. A flag at 100% for one day might still need to roll back. 1–2 weeks at 100% (with the rollback path intact) is the canonical soak. After soak, delete with confidence.

Lifecycle of a release flag

Dark

Code merged behind flag, off everywhere. CI green; tests cover both branches.

Internal

Flag on for engineering + dogfood tenant. Dogfood usage surfaces obvious bugs without customer impact.

Canary

5% → 25% → 50%. SLO + KPI watch at each step. Pause if either regresses.

100%

All users. Soak 1–2 weeks; rollback path intact.

Cleaned up

Flag deleted from config; code path removed; cleanup PR closes the original "add" issue.

Rolled back (if needed)

SLO/KPI regression at any step → flip flag off. Investigate; fix; resume the ramp.

Worked example — lesson-player partial-credit flag

A LaunchDarkly-style flag registered in YAML at flag-creation time. Expiry, owner, and remove-issue all set on day one.

# flags/registry.yaml
- key:         player.partial_credit
  type:        release
  owner:       "@lessons"
  description: "Partial-credit scoring for multi-step questions"
  default:     false
  added:       2026-05-08
  expires:     2026-08-08             # auto-page owner on stale flag
  remove_pr:   https://gh.com/studyapp/repo/issues/4521
  tests:       both-branches          # CI matrix asserts both paths
  rollout:
    - { stage: dark,     pct: 0,   tenant_rule: null }
    - { stage: internal, pct: 100, tenant_rule: "tenant ∈ {dogfood}" }
    - { stage: canary,   pct: 5,   slo_watch: ["latency.p99", "error_rate"] }
    - { stage: canary,   pct: 25,  slo_watch: ["latency.p99", "error_rate"] }
    - { stage: canary,   pct: 50,  slo_watch: ["latency.p99", "error_rate"] }
    - { stage: full,     pct: 100, soak_days: 14 }
    - { stage: cleanup,  due:      2026-08-08 }

# code (Python)
if flags.is_on("player.partial_credit", user_id=user.id, tenant=user.tenant_id):
    score = AttemptScorer.partial(answers)
else:
    score = AttemptScorer.binary(answers)

# audit log entry (auto-generated)
2026-05-23T14:02:11Z  player.partial_credit
  who:   @oleg
  from:  off
  to:    rollout(canary, pct=5)
  rule:  +slo_watch=[latency.p99, error_rate]

Flag declared in YAML alongside its code change. Expiry date, owner, remove issue, and CI-tested both-branches policy all set at creation. The audit log records every flip in machine-readable form for postmortems.

Quality bar

  • Flags ship dark by default. Non-trivial features start with a flag, off, in main.
  • Every flag has owner + expiry + remove-issue. Tracked in a registry.
  • Audit log records every flip. Who, when, value, rule.
  • Both branches tested. CI matrix flag-on / flag-off.
  • Ramp by signal, not by timer. SLO + KPI watch with explicit go/no-go.
  • Cleanup PR after soak. Code path + config both removed.
  • Quarterly stale-flag audit. Owners paged; orphans deleted.
  • Release vs ops flags distinct. Different lifecycles; different owners.
