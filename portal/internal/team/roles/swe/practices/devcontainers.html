Reproducible dev env — devcontainers / Nix / Compose

Clone the repo, run one command, and get the same toolchain, database, and runtime version every engineer and every CI runner uses.

TL;DR

A reproducible dev environment is declared in code — .devcontainer/devcontainer.json (OSS Dev Containers spec at containers.dev, originated in VS Code Remote-Containers, now used by GitHub Codespaces), flake.nix (Nix, Eelco Dolstra 2006), or docker compose — and is version-controlled with the application. One command rebuilds the same OS, language toolchain, database, and tool versions on every laptop and on the CI runner. "Works on my machine" stops being a debugging answer.

What it is

Three layered approaches solve the same problem — declarative, version-controlled, one-command dev environments:

  • SWE Containers — .devcontainer/devcontainer.json (containers.dev) declares a base image, "features" (pre-baked language toolchains), extensions, and post-create commands. VS Code Remote-Containers, GitHub Codespaces, JetBrains Gateway, and other tools all read the same spec.
  • Nix flakes — flake.nix + flake.lock describe the toolchain natively, with hash-pinned dependencies and hermetic builds. No daemon, no container; everything is installed in /nix/store. Steeper learning curve; the gold standard for reproducibility.
  • docker compose — compose.yaml orchestrates the data-plane (Postgres, Redis, Kafka, MinIO) the application needs. Runs alongside the devcontainer or as a sibling environment.

The three layers compose: Compose for the data services, a devcontainer for the toolchain + IDE, optionally Nix inside the container for hermetic toolchain pinning. The end-state property is "clone, run one command, work" — and "the CI uses the same image as your laptop".

Why we need it

"Works on my machine" is a debugging answer that costs hours each time it appears. The new hire spends three days getting Postgres on the right minor version, the right Node, the right Python, the right linker; the senior engineer rebuilds their toolchain when a system upgrade breaks GCC; CI fails on a version drift that nobody noticed because production was on the old version.

A declarative dev environment removes the entire class. The toolchain is in code, version-controlled with the app. A new hire's first command is the same as everyone else's. CI uses the same image. Production drift becomes visible because dev and CI both follow the same declared versions. The team gets back hours per week and minutes per build.

How others do it

Microsoft / VS Code — Dev Containers spec

VS Code Remote-Containers (2019) introduced the dev-container format; in 2022 Microsoft moved it to an open spec at containers.dev with a public Features registry. GitHub Codespaces (paid, 2021) is the cloud incarnation: open a repo on github.dev and the devcontainer spins up in seconds. Most major IDEs read the spec.

What we take: Devcontainers are the lowest-friction default. The spec is open, the tooling is mature, the same file works locally and in Codespaces. Start here unless you have specific reasons not to.

Source: containers.dev — Dev Containers spec

Nix / NixOS — hermetic reproducibility

Nix (Eelco Dolstra's 2006 PhD work) and its flakes feature offer hash-pinned, hermetic dev environments. nix develop drops you into a shell with the exact toolchain declared in flake.nix. NumtideHQ, Determinate Systems, and others ship Nix-based DX platforms. Steeper learning curve; unmatched reproducibility.

What we take: For teams that need bit-for-bit reproducibility (compilers, scientific computing, financial systems), Nix is the gold standard. For most app teams, devcontainers cover 95% of the value with much less learning cost.

Source: nixos.org

Stripe — engineering "one-command setup"

Stripe's internal "stripe-dev" tool (described in conference talks) gives new hires a single command that bootstraps the entire local environment in minutes: language versions, services, secrets, sample data. The platform team owns the tool as a product; DX is treated as a competitive advantage in hiring.

What we take: Wrap the underlying tools (compose, devcontainer, Nix) in a single top-level command. ./scripts/dev up hides the complexity. New hires shouldn't need to learn three tools to get coffee on day one.

Source: Stripe Engineering — DX investment

GitHub Codespaces — cloud dev envs

Codespaces (GitHub, 2021) runs the devcontainer in a managed cloud VM, accessible via browser or local VS Code. Teams that use Codespaces report new-hire ramp dropping from days to minutes. The same devcontainer.json works locally for offline work; Codespaces is the cloud option for laptops without the horsepower (or for hot-desking).

What we take: Pick a tool stack that works locally and in the cloud. Devcontainer-first achieves this naturally; pure-Nix or pure-compose setups need extra work to run in Codespaces or similar.

Source: github.com/features/codespaces

Anatomy of a reproducible setup

.devcontainer/devcontainer.json

Base image, features, extensions, post-create commands. Lives in the repo root; works in VS Code, JetBrains Gateway, Codespaces.

compose.yaml for data services

Postgres, Redis, Kafka, MinIO, MailHog. Versioned with the app. Volumes for persistence; healthchecks for readiness.

Single bootstrap command

./scripts/dev up or make dev wraps devcontainer + compose. New hire's day-one command is the same as everyone else's.

Pinned toolchain versions

Node, Python, Go, Java versions pinned in devcontainer.json features or in a .tool-versions file. Drift surfaced at startup, not at runtime.

CI uses the same image

CI workflow runs in the devcontainer image (GitHub Actions uses: devcontainers/ci). What works in dev works in CI; debugging the gap stops being a daily activity.

Seed data + secrets baseline

A scripts/seed step populates the local DB with realistic-but-fake data. Secrets baseline (1Password / age-encrypted) populates a .env with safe local defaults.

How to run it

  1. Pick the stack. Devcontainer-first is the default. Add Compose for data services. Reach for Nix only when bit-for-bit reproducibility matters.
  2. Write devcontainer.json. Base image + features (language toolchains). Extensions (linters, formatters). Post-create command runs scripts/setup.
  3. Write compose.yaml. Data services with healthchecks; volumes for persistence; one named network.
  4. Add the bootstrap script. ./scripts/dev up orchestrates devcontainer + compose + seed.
  5. Pin toolchain versions explicitly. Features or .tool-versions; drift caught at container build.
  6. Wire CI to use the same image. devcontainers/ci action; same toolchain in dev and CI.
  7. Document one command in the README. "Clone, run ./scripts/dev up, open in VS Code." Anything more is a smell.
  8. Test the new-hire path quarterly. A senior engineer wipes their setup and follows the README. Time-to-first-PR measured; regressions get fixed.

Deep dive — what makes the env actually reproducible

Declarative beats imperative every time. A README that says "install Node 20, install Postgres 16, run brew install..." is documentation; engineers will skip half of it and version-drift the rest. A devcontainer.json + compose.yaml is executable; the only way to use it is correctly. Choose the declarative tools; treat README setup steps as a smell.

Devcontainer-first as the default. Most teams don't need bit-for-bit Nix reproducibility; they need "Node 20, Python 3.12, Postgres 16, three Slack-bot tokens" repeatedly. Devcontainers cover that with the lowest learning cost and the widest tool support. Adopt Nix only when you've measured the value of hermetic builds for your specific work.

One bootstrap command. Wrap the underlying tools in one top-level command. New hires shouldn't need to know whether they're using devcontainer + compose or just compose — they run ./scripts/dev up and they're in. The wrapper script is the API of the dev experience.

Same image in CI. The biggest DX win is when dev and CI use the exact same container. Bugs that reproduce on one but not the other disappear. The GitHub Actions devcontainers/ci action makes this a one-line addition; do it on day one.

Pin everything visible. Node, Python, Go versions; Postgres minor version; tool versions. Pinned versions surface drift at container build, not at runtime. The pinning file is part of the codebase — bump it as a PR with the migration steps in the description.

Test the new-hire path quarterly. Setups rot silently as dependencies move, registries deprecate, base images change. A senior engineer wipes their machine and runs through the README once a quarter; missing steps get a same-day fix. Time-to-first-PR is the metric.

Codespaces or equivalent as the cloud fallback. Some laptops can't run the full stack; some engineers travel and code on Chromebooks. Codespaces lets the same devcontainer run in the cloud. Plan for it from day one — devcontainer-first stacks get this for free.

Lifecycle of the dev environment

Bootstrapped

New hire (or new laptop) runs ./scripts/dev up. Devcontainer builds; Compose starts; seed runs. First PR opened within a day.

Daily use

Devcontainer rebuilds automatically when devcontainer.json changes. Compose services restart as needed. Same tooling as CI.

Drift detected

A new feature added at the wrong toolchain version → CI fails on the version constraint. Fix lands as a PR bumping pin + service version + migration notes.

Upgraded

Toolchain bumps land as PRs with clear migration steps. Devcontainer rebuilds; CI passes; all engineers move forward in sync.

Tested quarterly

Senior engineer wipes setup; runs README from scratch. Time-to-first-PR measured. Regressions get same-day fixes.

Cloud-available

Codespaces / JetBrains Space / equivalent reads the same devcontainer.json. Travel laptops get cloud env in seconds.

Worked example — Study App devcontainer + compose

Top-level devcontainer.json, compose.yaml, and bootstrap script. ./scripts/dev up is the day-one command for every engineer.

# .devcontainer/devcontainer.json
{
  "name": "study-app",
  "image": "mcr.microsoft.com/devcontainers/base:ubuntu-22.04",
  "features": {
    "ghcr.io/devcontainers/features/python:1":  { "version": "3.12" },
    "ghcr.io/devcontainers/features/node:1":    { "version": "20" },
    "ghcr.io/devcontainers/features/go:1":      { "version": "1.22" },
    "ghcr.io/devcontainers/features/docker-in-docker:2": {}
  },
  "forwardPorts": [3000, 5432, 6379, 9092],
  "postCreateCommand": "./scripts/setup",
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-python.python", "esbenp.prettier-vscode",
        "dbaeumer.vscode-eslint", "golang.go"
      ]
    }
  }
}

# compose.yaml (data services)
services:
  postgres: { image: postgres:16.4, ports: ["5432:5432"], volumes: ["pg:/var/lib/postgresql/data"] }
  redis:    { image: redis:7.4-alpine, ports: ["6379:6379"] }
  kafka:    { image: confluentinc/cp-kafka:7.7.0, ports: ["9092:9092"] }
  mailhog:  { image: mailhog/mailhog:v1.0.1, ports: ["1025:1025", "8025:8025"] }
volumes: { pg: {} }

# scripts/dev (the day-one command)
#!/usr/bin/env bash
set -euo pipefail
case "${1:-up}" in
  up)    docker compose up -d && code . ;;
  down)  docker compose down ;;
  reset) docker compose down -v && ./scripts/dev up && ./scripts/seed ;;
esac

# .github/workflows/ci.yml (uses the SAME image as dev)
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: devcontainers/ci@v0.3
        with:
          imageName: ghcr.io/studyapp/devcontainer
          runCmd: ./scripts/test

Four artefacts, all in-repo: devcontainer.json (toolchain), compose.yaml (data services), scripts/dev (the wrapper command), and a CI workflow that uses the exact devcontainer image. New hire's day-one path: clone, ./scripts/dev up, open in VS Code.

Quality bar

  • Devcontainer + Compose in-repo. Single bootstrap command (./scripts/dev up).
  • Toolchain versions pinned. Features / .tool-versions / compose tags exact.
  • CI uses the devcontainer image. devcontainers/ci action or equivalent.
  • Seed data on bootstrap. scripts/seed populates a realistic local DB.
  • Rebuild < 90 s. Prebuilt base + features + cached layers.
  • README is one command. Anything more is a smell; documentation steps are anti-patterns.
  • Quarterly fresh-clone test. Senior engineer wipes setup; time-to-first-PR measured; regressions fixed same-day.
  • Cloud fallback works. Codespaces / JetBrains Gateway open the same devcontainer in seconds.
