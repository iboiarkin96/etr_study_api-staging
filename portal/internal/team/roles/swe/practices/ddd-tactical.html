DDD tactical patterns — aggregates, value objects, services

Concentrate business invariants in aggregate roots; make identifierless things immutable value objects; keep persistence behind repositories.

TL;DR

Tactical DDD is the building-block vocabulary from Eric Evans's Domain-Driven Design (2003) and Vaughn Vernon's Implementing DDD (2013). Every state-changing operation goes through an Aggregate root that owns a small consistency boundary and its invariants. Identifierless concepts (Money, Address, EmailAddress) become immutable Value Objects. Cross-aggregate operations become Domain Services. Persistence sits behind Repositories. Done well, one consistency unit lives in one place; done poorly, business logic is sprinkled across controllers, ORMs, and helpers and every change touches all three.

What it is

Tactical DDD is the per-class vocabulary that operationalises strategic DDD (bounded contexts, ubiquitous language). The four core blocks:

  • Aggregate — a cluster of objects treated as one unit for consistency. The root is the only entry point; it enforces invariants; one transaction touches one aggregate. Small (1–3 entities) is the modern guidance.
  • Value object — an immutable type with no identity. Equality is by value, not by reference. Examples: Money, EmailAddress, DateRange. Side-effect-free; cheap to share; impossible to mutate.
  • Domain service — a stateless operation that doesn't belong to any single aggregate (e.g. transferFunds(from, to, amount)). Named in the ubiquitous language. Last resort, not first — most logic belongs on an aggregate.
  • Repository — a port that hides persistence. Domain code does orderRepo.save(order), not db.exec(SQL). The adapter (Postgres, EventStore, in-memory test fake) implements the port.

Domain events (LessonCompleted, AttemptScored) coordinate across aggregates without breaking the one-aggregate-per-transaction rule — write the event in the same transaction as the state change, publish it asynchronously to interested consumers (see Practice 02 on event-driven architecture).

Why we need it

Without aggregates, business invariants leak: controllers validate one rule, ORM signals fire another, a service helper runs a third, and the same invariant ends up checked three different ways in three different places. The first time the rule changes, two of the three places get updated; production accumulates inconsistent state silently.

Tactical DDD concentrates each invariant in one place — the aggregate root — and uses immutability (value objects) and isolation (repositories) to prevent the leak. The result is code that reads like the domain ("Lesson.recordAttempt(attempt)", "Money.add(other)") and a test suite that exercises rules where they live. Engineers spend less time spelunking and more time shipping.

How others do it

Eric Evans — original DDD

Evans's 2003 Domain-Driven Design introduced aggregates, value objects, domain services, and repositories as the tactical vocabulary. The Blue Book is dense but remains the canonical reference; Domain-Driven Design Distilled (Vernon, 2016) is the readable summary.

What we take: The vocabulary itself is the asset. "Where does this rule live?" stops being a debate when the team shares aggregate / value object / domain service / repository as agreed concepts.

Source: domainlanguage.com — Eric Evans

Vaughn Vernon — "small aggregates" rule

Vernon's 2013 Implementing DDD and his "Effective Aggregate Design" paper introduced the small-aggregates rule: aggregates should be small (1–3 entities), reference each other by ID, and never span more than one transaction. The rule is the most influential refinement of Evans's original.

What we take: Aggregates stay small. References across aggregate boundaries are IDs, not object pointers. One transaction = one aggregate; cross-aggregate consistency goes through domain events.

Source: Vernon — Effective Aggregate Design

Martin Fowler — Repository & Value Object catalogue

Fowler's Patterns of Enterprise Application Architecture (2002) and the live entries at martinfowler.com document Repository, Value Object, and the broader catalogue with implementation notes that complement Evans's higher-level treatment. The Repository pattern as commonly understood owes more to Fowler's PEAA than to the Blue Book.

What we take: Value objects with custom equality; repositories as collection interfaces returning domain objects. Pair Evans's strategic framing with Fowler's tactical implementation.

Source: martinfowler.com — PEAA catalogue

"Cosmic Python" — DDD in Python

Percival & Gregory's Architecture Patterns with Python (2020, often called Cosmic Python) is the most accessible modern tactical-DDD textbook for dynamic-language teams. It walks through aggregates, value objects, repositories, and the unit-of-work pattern with runnable examples. The same authors maintain cosmicpython.com with example repos.

What we take: Tactical DDD doesn't require Java/C#. Python (or TypeScript) with frozen dataclasses, repository ports, and explicit aggregates works as well — the patterns are language-independent.

Source: cosmicpython.com

Anatomy of a tactical-DDD module

Aggregate root class

Public mutators (record, complete, cancel). All invariants checked inside. No public field setters — state changes only through named methods that mean something in the domain.

Value-object types

Immutable (frozen dataclasses / records / readonly types). Equality by value. Self-validating in constructor (no invalid Money / EmailAddress can exist).

Repository port

Domain-typed interface (OrderRepository.save(Order), byId(OrderId)). No SQL, no ORM types in the signature. Adapter implements it.

Domain service (sparingly)

Stateless functions for cross-aggregate operations. Named in domain vocabulary (transferFunds, not OrderProcessor). Reach for it only when the operation doesn't fit on a single aggregate.

Domain events

Past-tense facts emitted by aggregates (LessonCompleted, AttemptScored). Written in the same transaction as state; published asynchronously to other aggregates / contexts.

Application service / use case

Thin orchestrator: load aggregate by id → call domain method → save → publish events. Doesn't hold business rules; just wires the use case.

How to run it

  1. Identify aggregates. Start from invariants: "what must always be true together?" That set is one aggregate; the entity that enforces it is the root.
  2. Make value objects immutable. Frozen dataclasses, records, readonly types. Self-validate in the constructor.
  3. Define repository ports. Interface in the domain layer; adapter in the infrastructure layer. Domain types only on the port surface.
  4. Resist domain services. Default to methods on the aggregate. Reach for a service only when the operation genuinely doesn't fit on one root.
  5. Reference across aggregates by ID. Order holds CustomerId, not Customer. Cross-aggregate consistency goes through domain events.
  6. Emit domain events. Past-tense facts on every significant state change. Written in the same transaction; published asynchronously.
  7. Keep aggregates small. 1–3 entities. If it grows, look for a hidden boundary; split.

Deep dive — what the patterns actually buy you

Aggregates concentrate invariants. The single most expensive mistake in non-DDD code is "the same rule checked three different ways in three different places". When the rule changes, two of the three get updated; production accumulates inconsistent state. Aggregates fix this by having one and only one place where the rule lives — the root's method. Controllers, services, and adapters call the method; they don't re-implement the check.

Small aggregates beat big ones. Vernon's rule is non-negotiable: 1–3 entities per aggregate, references across boundaries by ID, one transaction per aggregate. A "Customer" aggregate that owns all their Orders, Invoices, and Subscriptions is a 1990s mistake; it creates concurrency conflicts, transaction-size pain, and tight coupling. Each of those should be its own aggregate, linked by ID, consistent eventually through events.

Value objects eliminate validation noise. If EmailAddress is a value object that self-validates, no other code can produce an invalid email. The 80% of validation logic that lives in controllers and helpers vanishes; the controller takes an EmailAddress from the request mapper, and either it parses or the request is rejected at the boundary. The domain code never sees an invalid value.

Repositories hide the database, not the data model. The repository port surfaces domain types (Order, LessonId); the adapter maps to whatever storage you use (Postgres rows, EventStore streams, in-memory dictionaries for tests). The point isn't abstraction-for-its-own-sake; it's that the domain layer compiles without a database driver on the classpath. Tests run in milliseconds with in-memory fakes.

Domain services are a last resort. Most "service" code is actually a missing method on an aggregate. "Transfer funds from A to B" looks like a service until you realise it's a method on a Transfer aggregate that emits FundsMoved events. Default to methods; reach for services only for truly stateless cross-aggregate operations.

Domain events are the cross-aggregate glue. One transaction = one aggregate, but real systems have cross-aggregate effects. Domain events bridge the gap: the aggregate emits a past-tense fact in the same transaction as its state change; downstream aggregates react asynchronously. The outbox pattern (Practice 02) keeps the write atomic.

Tactical without strategic is brittle. Aggregates and value objects inside the wrong bounded context produce code that's correct-looking but solves the wrong problem. Pair tactical patterns with bounded-context discipline; identify the contexts first, then apply tactical patterns inside each.

Lifecycle of an aggregate change

Loaded

Use case calls repo.byId(id). Repository rehydrates the aggregate from storage. Domain types returned; adapter types hidden.

Invariant-checked

Use case calls a named method on the root (order.cancel()). The method checks invariants; rejects if violated; mutates state if valid.

Event-emitted

Aggregate appends a domain event (OrderCancelled) to its uncommitted-events list. Past-tense fact; immutable.

Persisted

Use case calls repo.save(order). Repository writes both the new state and the new events in one transaction (outbox).

Published

Outbox relay publishes events asynchronously. Other aggregates / contexts react via their own subscribers.

Reconstituted on next load

Future byId(id) calls return the aggregate in its new state. History (events) optionally available for audit or replay.

Worked example — Lesson aggregate (Python)

A trimmed Lesson aggregate plus the supporting value objects and repository port. Invariants live on the root; value objects self-validate; cross-aggregate effects emit domain events.

# domain/lesson.py
from dataclasses import dataclass, field
from typing import Protocol

# --- Value objects (immutable, identity-less) -----------------------
@dataclass(frozen=True)
class LessonId:
    value: str
    def __post_init__(self):
        if not self.value.startswith("L-"):
            raise ValueError("LessonId must start with L-")

@dataclass(frozen=True)
class Score:
    raw: float
    ceiling: float = 1.0
    def __post_init__(self):
        if not 0 <= self.raw <= self.ceiling:
            raise ValueError("score out of bounds")
    @property
    def ratio(self) -> float:
        return self.raw / self.ceiling

# --- Domain event ---------------------------------------------------
@dataclass(frozen=True)
class LessonCompleted:
    lesson_id: LessonId
    user_id: str
    score: Score
    occurred_at: str  # ISO timestamp

# --- Aggregate root (small: 1 entity + value objects) --------------
@dataclass
class Lesson:
    id: LessonId
    user_id: str
    is_completed: bool = False
    score: Score | None = None
    _events: list = field(default_factory=list)

    def record_attempt(self, score: Score, at: str) -> None:
        if self.is_completed:
            raise ValueError("lesson already completed")
        self.is_completed = True
        self.score = score
        self._events.append(LessonCompleted(self.id, self.user_id, score, at))

    def pull_events(self) -> list:
        events, self._events = self._events, []
        return events

# --- Repository port (domain types only) ---------------------------
class LessonRepository(Protocol):
    def by_id(self, id: LessonId) -> Lesson | None: ...
    def save(self, lesson: Lesson) -> None: ...  # writes state + events atomically

# --- Application service (thin orchestrator) -----------------------
def complete_lesson(repo: LessonRepository, publisher, lesson_id: str,
                    raw_score: float, at: str) -> None:
    lesson = repo.by_id(LessonId(lesson_id))
    if lesson is None:
        raise LookupError("unknown lesson")
    lesson.record_attempt(Score(raw_score), at)
    repo.save(lesson)
    for event in lesson.pull_events():
        publisher.publish(event)

One small aggregate, two value objects with constructor validation, one domain event, a repository port speaking domain types only, and a thin application service that orchestrates the use case. Tests run with an in-memory LessonRepository fake; the database driver is never imported in the domain module.

Quality bar

  • Aggregates small (1–3 entities). Cross-aggregate references by ID only.
  • No public setters on aggregates. Only named domain methods that check invariants.
  • Value objects immutable + self-validating. Frozen dataclasses / records / readonly types.
  • Repository ports speak domain types. No ORM / DB types in port signatures; domain compiles without a DB driver.
  • Domain services are rare. Default to methods on aggregates; services are a last resort.
  • One transaction = one aggregate. Cross-aggregate effects via domain events + outbox.
  • Domain events past-tense + emitted in-transaction. Published asynchronously to other contexts.
