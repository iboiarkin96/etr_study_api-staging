Performance budgets — measured, not assumed

Lighthouse-CI on the changed page, bundlesize on the changed package, k6 on the changed endpoint — merge stays grey until the numbers are inside the line in perf-budget.json.

TL;DR

Tim Kadlec popularised performance budgets in Setting a Performance Budget (2013); web.dev's "Performance Budgets 101" is the official guidance. Pick a small set of metrics (LCP, API p99, bundle size); set a numeric target; measure on every PR; fail the build above the target. Lighthouse-CI for the web, k6 / Artillery for APIs, bundlesize / size-limit for JS bundles. Budgets must be enforced on real-user metrics, not on dev hardware — otherwise you ship a slow site to a fast laptop.

What it is

A performance budget is a numeric constraint on a measurable perf metric, enforced in CI on every change. Three common kinds:

  • Frontend metrics — Largest Contentful Paint (LCP), Interaction to Next Paint (INP), Cumulative Layout Shift (CLS), JS bundle size. Measured by Lighthouse-CI in CI; calibrated against CrUX (Chrome's real-user dataset) or your own RUM.
  • Backend metrics — endpoint p95 / p99 latency, throughput at target load, allocation count per request. Measured by k6 / Artillery / wrk in CI against a staging environment; calibrated against production traces (Practice 17).
  • Asset metrics — JS bundle size per chunk, image size budgets, font-file count. Measured by bundlesize / size-limit in CI on every PR that touches frontend.

The budgets live in a versioned file (perf-budget.json, lighthouse-budgets.json, k6 threshold config). PR changes that breach the budget fail the build; the engineer either fixes the perf regression or files an exception with an expiry date.

Why we need it

Without budgets, perf regresses one PR at a time. Each PR is "only 50 ms"; over a year the home page is 4 seconds slower than it started. The team notices when a customer complains; by then the regression is hundreds of merged changes deep and impossible to bisect cheaply.

Budgets invert the curve: every PR that breaches the budget is the one PR that has to defend the slowdown. The conversation happens at the cheapest moment — at the diff. The team learns to think about perf as a first-class constraint rather than something to clean up later. CrUX / RUM data calibrates the budget against reality so the team isn't chasing a number nobody on real hardware experiences.

How others do it

Tim Kadlec — pattern originator

Kadlec's 2013 Setting a Performance Budget framed the concept and made it actionable. Two key ideas: budget per metric (not "make it fast"), and budget enforced in CI (not aspirational). His blog (timkadlec.com) and the Smashing-Magazine essay are the canonical references.

What we take: Budget per metric with numeric target. "Be fast" is not a budget; "LCP ≤ 2.5 s on mobile" is.

Source: timkadlec.com

web.dev / Google — Core Web Vitals

Google's Core Web Vitals (LCP, INP, CLS) and the web.dev guidance set the modern frontend baseline. Lighthouse, Lighthouse-CI, PageSpeed Insights, and CrUX all use the same metric set. The thresholds (LCP ≤ 2.5 s good, ≤ 4 s needs-improvement) are now the de-facto industry standard.

What we take: Use Core Web Vitals as the frontend metric set. Industry-standard, RUM-comparable, and aligned with what your customers experience.

Source: web.dev — Performance Budgets 101

k6 / Grafana — backend budgets as code

k6 (now Grafana k6) ships "thresholds" — assertions like p(95) < 250 in the test script. The k6 run fails if the assertion isn't met. Same syntax for HTTP, gRPC, WebSocket, browser scripts. The thresholds make k6 the de-facto backend perf-budget tool.

What we take: Write backend perf assertions in k6 thresholds. The test is the budget; merge fails if thresholds fail. No separate "perf review" step needed.

Source: k6 thresholds docs

Shopify — RUM-calibrated budgets

Shopify's engineering blog describes their approach: budgets in CI are calibrated against real-user metrics from their RUM pipeline. The CI gate uses a generous-enough number that real-user p75 stays inside Core Web Vitals "good"; the RUM dashboard tracks the trend; the budget tightens as the team improves.

What we take: Calibrate CI budgets against RUM, not against dev hardware. Otherwise you ship a slow site to a fast laptop.

Source: shopify.engineering

Anatomy of a perf-budget pipeline

Versioned budget file

perf-budget.json / lighthouse-budgets.json / k6 threshold block. Numeric targets per metric per route or endpoint.

Per-PR measurement

Lighthouse-CI on changed pages; k6 on changed endpoints; bundlesize on changed packages. Skip-path filters keep the gate fast.

Required CI check

Branch protection: perf-budget check is required. Merge blocked on regression. Exception workflow with expiry date.

Real-user calibration

RUM (CrUX / Datadog / Honeycomb) feeds a monthly dashboard. Budget tightens as p75 improves; loosens if reality forces it.

Mobile-class CI hardware

Lighthouse-CI configured with "Slow 4G" + 4× CPU throttling to mirror real users. Without throttling, the gate passes on dev hardware while real users feel slow.

Monthly perf review

30 min: top-5 regressions, top-5 wins, budget vs RUM. Engineers see where they stand; budget gets tuned.

How to run it

  1. Pick the metric set. Frontend: LCP, INP, CLS + bundle. Backend: p95 / p99 + throughput + error rate. Asset: bundle size + image weight.
  2. Set numeric targets. Per route / endpoint. Start with web.dev "good" thresholds; tighten as the team improves.
  3. Wire CI tools. Lighthouse-CI (frontend); k6 thresholds (backend); bundlesize / size-limit (JS).
  4. Throttle to mobile-class hardware. Slow 4G + 4× CPU on Lighthouse-CI; "mobile" preset.
  5. Make checks required. Branch protection on main; merge blocked above budget.
  6. Stand up RUM. CrUX / Datadog Web Vitals / SpeedCurve. Monthly dashboard.
  7. Monthly perf review. 30 min: top-5 regressions + wins; calibrate budget against RUM p75.
  8. Exception workflow. If a budget breach is justified, add an exception with expiry date; CI re-fails on expiry.

Deep dive — what makes budgets bite

Budgets must be enforced in CI, not aspirational. A budget in a doc is a wish. A required CI check that blocks merge is a budget. The transition from "we should be fast" to "this PR can't merge because LCP regressed 200 ms" is what changes team behaviour.

Throttle CI hardware to mobile. The single most common mistake is running Lighthouse-CI on the default GitHub Actions runner (a beefy desktop) and getting "everything is fine". Configure Slow 4G + 4× CPU throttling so the CI number resembles the median user's experience. Otherwise the budget is fiction.

RUM calibrates the CI budget. The CI number is synthetic; RUM is reality. The two diverge: a perfect CI score with real-user p75 = 5 s means the budget is wrong. Monthly review aligns CI with RUM; the budget tightens when RUM improves and loosens when reality says it must.

Bundle size is the cheapest battle. Frontend perf regressions usually start as bundle growth. A budget of "main bundle ≤ 200 kB gzipped" + bundlesize in CI catches "engineer imported lodash for one function" the same day, when fixing takes 5 minutes.

Backend p99 over p95. p95 hides the slow tail that hurts real users. Budget the p99 (or p99.9) — the tail is what produces the customer-complaints latency.

Per-route budgets, not global. A homepage budget of 2 s LCP is reasonable; a deep "lesson player" budget of 2 s isn't because the content is heavier. Set budgets per route / endpoint, not globally. Lighthouse-CI supports this natively.

Exceptions need expiry dates. "Breach allowed for this PR" turns into "breach allowed forever" without an expiry. Every exception has a date; CI fails again on the date; the team either fixes or renews with justification.

Lifecycle of a perf budget

Drafted

Initial targets in perf-budget.json. Start with web.dev "good" thresholds + 25% slack while team calibrates.

Enforced

Required CI check. Merge blocked on breach. Exception workflow defined.

Calibrated

RUM dashboard live. Monthly review aligns CI budget with reality.

Tightened

As team improves, budget tightens. Pressure stays constant — slack is reclaimed for new optimisation work.

Excepted

Justified breach with expiry date. CI re-fails on expiry; team fixes or renews with new reason.

Retired

Route / endpoint removed; budget entry removed in the same PR. Stale entries flagged at monthly review.

Worked example — Study App perf budgets

Per-route Lighthouse-CI budget + k6 backend thresholds + bundlesize config. All three enforced as required CI checks.

# perf/lighthouse-budgets.json (per-route)
[
  { "path": "/",                 "resourceSizes": [{ "resourceType": "script", "budget": 180 }],
                                  "timings": [{ "metric": "interactive", "budget": 3000 },
                                              { "metric": "largest-contentful-paint", "budget": 2500 }] },
  { "path": "/lessons/*",        "timings": [{ "metric": "largest-contentful-paint", "budget": 3500 }] },
  { "path": "/billing/*",        "timings": [{ "metric": "largest-contentful-paint", "budget": 3000 }] }
]

# perf/lighthouse-ci.yml
ci:
  collect:
    settings:
      preset: mobile             # 1.6 Mbps · 4x CPU throttle
      formFactor: mobile
    numberOfRuns: 5              # median to reduce flakiness
  assert:
    budgetsFile: perf/lighthouse-budgets.json

# perf/k6/lesson-state.js (backend)
import http from 'k6/http';
export const options = {
  thresholds: {
    'http_req_duration{endpoint:state}': ['p(99)<200', 'p(99.9)<500'],
    'http_req_failed{endpoint:state}':   ['rate<0.001'],
  },
  scenarios: {
    sustained: { executor: 'constant-arrival-rate', rate: 100, duration: '5m' },
  },
};
export default function () {
  http.get('https://staging/lessons/abc/state', { tags: { endpoint: 'state' } });
}

# package.json (bundlesize, frontend)
"bundlesize": [
  { "path": "dist/main.*.js",      "maxSize": "200 kB", "compression": "gzip" },
  { "path": "dist/vendor.*.js",    "maxSize": "350 kB", "compression": "gzip" },
  { "path": "dist/runtime.*.js",   "maxSize": "20 kB",  "compression": "gzip" }
]

# .github/workflows/perf.yml (excerpt)
jobs:
  perf:
    steps:
      - uses: treosh/lighthouse-ci-action@v11
        with: { configPath: perf/lighthouse-ci.yml }
      - uses: grafana/k6-action@v0.3.1
        with: { filename: perf/k6/lesson-state.js }
      - run: npx bundlesize

Three tools, three budget files, one perf workflow. Per-route LCP budgets for the frontend; p99 + p99.9 thresholds in k6 for the backend; gzipped-bundle limits per chunk. All three are required checks; merge is blocked on breach.

Quality bar

  • Budget in CI as a required check. Merge blocked on breach.
  • Per-route / per-endpoint targets. No global "fast" goal.
  • Mobile-throttled Lighthouse. Slow 4G + 4× CPU; "mobile" preset.
  • p99 / p99.9 backend. Tail is what customers feel.
  • RUM calibration. Monthly review aligns CI budget with reality.
  • Exception workflow with expiry. No permanent breaches.
  • Bundle size budgeted per chunk. Gzipped; required check.
  • Monthly perf review. 30 min; top-5 regressions + wins.
