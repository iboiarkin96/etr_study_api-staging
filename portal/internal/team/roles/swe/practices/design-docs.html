Design docs & RFCs — written before code

Move the architecture conversation upstream — from code review to a one-to-four-page document reviewed before a line is written.

TL;DR

A design doc — sometimes called an RFC — is a 1–4 page document written before non-trivial code: problem, proposed solution, alternatives considered, trade-offs, open questions. Google has used design docs as the primary design artefact for two decades; Malte Ubl's 2020 essay at industrialempathy.com is the canonical public account. HashiCorp, Rust, Ember, Python (PEPs) all run public RFC processes on the same principle. A code review is a poor place to debate architecture — by then the engineer has invested days and is committed. A design doc moves the debate to the cheapest moment to change direction: before code exists.

What it is

A design doc is a 1–4 page document written before non-trivial code. The canonical sections:

  • Context / problem — what currently exists, what is forcing the change, the constraint being addressed.
  • Proposed solution — the chosen approach in enough detail to evaluate.
  • Alternatives considered — 2–3 other approaches with pros/cons. Non-negotiable — a single-option doc is a proposal in disguise.
  • Open questions — what the author specifically wants feedback on.
  • Success metrics — measurable outcomes that say it worked.
  • Risks & rollout — what could go wrong; how the change is deployed.
  • FAQ — anticipated objections answered up front.

Reviewers comment async over 3–7 days; a Final Comment Period (FCP) gives stragglers a last week. The decider accepts/revises/postpones with written rationale. Accepted docs link to an ADR; implementation PRs cite the doc. The architect-level practice (Practice 15 in the Architect radar) is the same machinery applied with a wider blast radius — see that page for the cross-team scope.

Why we need it

The cheapest moment to discover a wrong design is when it is written down. A 90-minute reading by three senior engineers catches assumptions that would otherwise take 3 weeks of implementation to surface. Writing forces precision — ideas that sound coherent in a verbal pitch reveal their gaps on paper.

Design docs also create institutional memory. A reader returning to the doc a year later sees what alternatives were considered, what the team chose, why. Without the doc, the same conversation is re-litigated every time a new engineer joins. Google's Design Docs at Google post explicitly identifies this as the most consistently valuable technical-writing investment at scale.

How others do it

Google — design docs everywhere

Google has used design docs as the primary engineering artefact for ~20 years. Malte Ubl's 2020 essay at industrialempathy.com is the canonical public account. Any engineer can write one; any engineer can comment; L6+ engineers routinely review across teams as mentoring + cross-pollination.

What we take: Design docs aren't only for senior engineers writing major proposals — they are for any engineer working on a non-trivial change. The act of writing scales thinking before implementation commits.

Source: Industrial Empathy — Design Docs at Google

Amazon — 6-page memo

Amazon's 6-page memo culture (Bezos) replaces PowerPoint with narrative documents. Full paragraphs, explicit FAQ section, silent reading at the top of major meetings. Closely related to design docs; has shaped how many ex-Amazon engineers run teams elsewhere.

What we take: The FAQ section is high-value. Anticipate reviewer objections and answer them in the doc. Saves review iterations.

Source: Bezos 2016 letter to shareholders

Rust / Ember / Python — public RFCs

Rust (github.com/rust-lang/rfcs), Ember (rfcs.emberjs.com), and Python (PEPs) all run formal RFC processes on the same principle: a structured document with required sections, async commentary in PR form, Final Comment Period, named decider, durable record.

What we take: The FCP mechanic catches stakeholders who missed active discussion. After substantive comments stop, give 7 days warning; final objections during FCP only.

Source: github.com/rust-lang/rfcs

Shopify — docs in the repo

Shopify keeps design docs in the same repo as the code, alongside ADRs and PlantUML diagrams. The doc PR is the first PR of the work; implementation PRs cite the doc by path. Comments live in the PR; archived with commit history. Weekly "design office hours" for early-stage proposals.

What we take: Keep design docs in-repo as Markdown. The PR review surface is excellent for async comments and the doc travels with the code. Office hours for early-stage ideas before formal doc.

Source: Shopify Engineering

Anatomy of a design doc

Header with status + decider

Title, authors, named reviewers, status (draft / in-review / FCP / accepted / postponed / superseded), decider, comment-close date, FCP-end date. Header is the load-bearing metadata.

Context that stands alone

A reader new to the area should understand the problem without consulting external sources. 2–4 paragraphs. Cites prior ADRs / docs where relevant.

Options comparison

2–3 real options with pros, cons, effort. Single-option docs are converted to ADRs. The comparison is what makes the recommendation defensible.

Success metrics

Measurable outcomes. "p99 ≤ 200 ms within 30 days at 5 k RPM" is valid; "performance will improve" isn't.

FAQ + open questions

FAQ answers predictable objections up front (Amazon-style). Open Questions list what the author wants feedback on.

In-repo + linked from ADR

Doc lives in docs/design/<slug>.md. Accepted docs link to an ADR; ADR cites the doc. Implementation PRs cite both.

How to run it

  1. Decide it needs a doc. Triggers: cross-service change, new dependency, data-model change, NFR shift, anything you cannot finish in a week.
  2. Draft in 1–2 days. Canonical template. Cap at 4 pages. Two real options minimum. FAQ pre-answers predictable questions.
  3. Open the PR + name reviewers. 3–5 named, spanning relevant domains (tech lead, SRE, security, data). Tag-based, not "@engineering".
  4. Async review window (3–7 days). Comments in-thread; author responds in writing; revisions edit the doc.
  5. Final Comment Period (7 days). Re-broadcast; no major design changes during FCP — clarifications and resolution only.
  6. Decider accepts/revises/postpones. Written rationale. ADR authored on accept. Doc archived with status.
  7. Implementation cites the doc. First implementation PR references both the doc and the ADR. Traceable design → decision → code.

Deep dive — what makes the doc do its job

Write first, code second. The doc is the cheapest place to discover a wrong path. An hour of writing reveals gaps that a week of coding would only hint at. Teams that write after implementation produce documentation, not design — the artefact has the same shape but does none of the work.

Two real options or none. A doc with one real option and a straw-man "do nothing" is a proposal in disguise. If only one option exists, write a short ADR and skip the design doc. Reserve docs for situations where genuine choice exists; that's where the format earns its cost.

Async-first, sync only on stall. Async lets reviewers process at their own pace and forces written resolution. Sync meetings are for the rare moments when async has stalled after 3 unresolved threads or 7 days. Default to async; written trail is the long-term value.

FAQ is free review iterations. Every predictable objection you answer in the FAQ is one review iteration you don't pay for. Borrow Amazon's discipline: write the FAQ before opening for review. First version answers the five most predictable questions; revisions add what reviewers actually raise.

Link to the ADR — both ways. Design doc = analysis; ADR = decision. Without the bidirectional link, future readers find the ADR and have no idea what alternatives were considered, or find the doc and have no idea what was decided.

Cap at 4 pages. Length is a smell. A 30-page doc has scope problems — it's solving multiple problems at once and should be split. Linked appendices for detail are fine; the body fits in one reading.

FCP catches PTO stakeholders. Active comments slow after 5–10 days. Without an explicit Final Comment Period, the doc closes while the senior engineer on PTO never sees the active phase. 7-day FCP pulls them back for a last review.

Lifecycle of a design doc

Draft

Author writes the first version. Not yet circulated. Used for office-hours or early review with the tech lead.

In review

Distributed to named reviewers. Async comments for 3–7 days. Author responds in writing; FAQ updated as questions surface.

FCP

Final Comment Period (7 days). Re-broadcast; no major design changes; only clarifications and resolution.

Accepted

Decider signs off. ADR authored. Doc archived as "accepted". Implementation can start.

Postponed / withdrawn

Decider declines with reason, or author withdraws. Doc kept with status; postponed docs state what would change the answer.

Superseded

Later doc replaces this one. Original kept with "superseded by" link; both ADRs cross-reference.

Worked example — DD-014 lesson-player offline mode

Header + options + FAQ for a real design doc. Full body in docs/design/2026-05-offline-mode.md.

# DD-014 · Lesson-player offline mode

status:        IN REVIEW
authors:       @oleg, @anna
reviewers:     @ivan (architect) · @maria (security) · @sasha (mobile)
decider:       @ivan
created:       2026-05-19
comment close: 2026-05-26
fcp end:       2026-06-02

## Context
Mobile users report dropped lessons on metro commutes — Wi-Fi gaps
cause the player to lose state mid-question. iOS App Store review #4
cited offline as a missing-feature concern.

## Options considered
| Option                             | Pros                    | Cons                       | Effort |
|------------------------------------|-------------------------|----------------------------|--------|
| A. Service-worker cache (PWA)      | reuses PWA shell        | iOS Safari quirks; 50 MB   | 3 wks  |
| B. Native shell + SQLite           | full control            | new pipeline; team gap     | 10 wks |
| C. Hybrid: SW + IndexedDB          | stages naturally        | dual code paths until A2   | 6 wks  |

Recommendation: Option C.

## Success metrics
- offline completion ratio > 95% on metro fixture (today: 38%)
- lesson-resume success ratio > 99% (today: 76%)
- p99 cold-start latency unchanged
- App-Store rating recovers to 4.6+ within 90 days of ship

## FAQ
Q: Why not retry on reconnect?
A: We retry today. Failure mode is in-question state loss, not net loss.

Q: Why not native shell now?
A: Team has no iOS-native depth. Option C buys 3 months to hire.

## Open questions
1. Cache user-generated answers offline? (privacy review needed)
2. Eviction policy at 50 MB cap?
3. Expose offline-mode as a setting or auto-enable?

Two pages, two real options + recommendation, measurable success criteria, FAQ pre-answering predictable questions, named decider. When accepted, ADR-0098 ("Adopt hybrid offline-mode") will cite this doc.

Quality bar

  • Written before code. First implementation PR cites the doc PR. Retroactive docs don't count.
  • Two real options minimum. Single-option docs converted to ADRs.
  • Cap at 4 pages. Detail in linked appendices.
  • Named reviewers + decider. 3–5 explicit reviewers; one named decider.
  • FAQ before review opens. ≥ 5 anticipated questions answered.
  • Success metrics measurable. Quantitative with baselines.
  • FCP explicit. 7-day Final Comment Period before close.
  • Linked to ADR. Acceptance requires ADR + bidirectional link.
