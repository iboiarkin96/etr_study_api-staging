Type safety as default — strict mode, no escape hatches

Strict typing on by default; any / # type: ignore banned except with a written reason and an expiry date.

TL;DR

Type safety has moved from "expert option" to "default" across modern stacks: TypeScript strict mode, Python's mypy/pyright with strict flags, Rust by construction, Kotlin null-safety. Microsoft's 2017 study found a 15% reduction in defects per file after adding TypeScript types. The discipline: strict mode in CI, escape hatches (any / type: ignore) require a comment + expiry date, generated types from external schemas (OpenAPI, GraphQL, Protobuf — Practice 12) replace hand-written ones.

What it is

Type safety as a default means three things working together:

  • Strict mode on day one — TypeScript "strict": true + noUncheckedIndexedAccess; Python mypy --strict or pyright --strict; Rust by construction; Kotlin kotlinOptions.allWarningsAsErrors.
  • Escape hatches require justification — any / as unknown / # type: ignore / @ts-expect-error get a comment with reason + expiry date. Without expiry, escape hatches multiply forever.
  • Generated types from external schemas — OpenAPI / GraphQL / Protobuf → typed client SDK. Practice 12 (API contract-first) is where these come from.

The 2017 Microsoft study (Gao, Bird, Barr) found a ~15% reduction in defects per file after adding TypeScript types to JavaScript. The modern practice extends this: not just "we use types" but "strict everything, escape hatches tracked, generated where possible".

Why we need it

Type bugs at runtime are the most common class of production incidents: undefined is not a function, null-pointer surprises, wrong-shape API responses. A strict type checker catches them at compile time, for free. Without strict mode, types are advisory; with it, the compiler is a fourth reviewer that never sleeps.

Strict typing also unlocks refactoring. A rename of a field surfaces every caller automatically. Without types, the same rename is grep + pray. Teams that adopt strict typing report higher refactoring rates and lower regression bugs; teams that adopt "loose" typing report the type checker passes but the bugs ship anyway because escape hatches let through what should have been errors.

How others do it

TypeScript / Microsoft (2012, strict 2017)

TypeScript shipped strict mode in 2017; Microsoft's research (Gao et al.) measured ~15% fewer bugs per file after the JS → TS migration. "strict": true turns on all strict-family flags; noUncheckedIndexedAccess is the additional opt-in modern teams enable.

What we take: "strict": true + noUncheckedIndexedAccess on day one. Anything less is loose; the bugs that were supposed to be caught aren't.

Source: typescriptlang.org — strict

Python — mypy / pyright (Dropbox, Microsoft)

Python's optional typing (PEP 484, 2014) matured into strict tooling: Dropbox-built mypy and Microsoft's pyright both support strict modes. Modern Python projects (FastAPI, Pydantic, Polars) ship strict-typed. Pyright is faster and stricter than mypy by default; most new teams pick it.

What we take: pyright --strict or mypy --strict in CI as a required check. Type stubs for untyped deps; # type: ignore with reason + expiry only.

Source: pyright

Rust — type safety by construction

Rust's type system + borrow checker make whole classes of bug uncompileable: null deref, use-after-free, data races. The compiler is the strictest reviewer on the team. Production Rust at Discord, Cloudflare, Dropbox, AWS.

What we take: For new system-level code where safety matters (auth, payments, parsers), Rust is the default in 2026. The type system pays back the learning curve within a quarter.

Source: rust-lang.org

Pydantic / Zod — runtime type guarding

Static typing checks at compile time; Pydantic (Python) and Zod (TS) check at runtime — validate inputs at the boundary against the same types the rest of the code uses. Catches what static typing can't: wrong-shape inputs from JSON, env vars, user form data.

What we take: Static + runtime types together. Static checks internal calls; runtime validators (Pydantic / Zod / valibot) check boundary inputs. Both share the same type model.

Source: pydantic.dev

How to run it

  1. Enable strict mode. Day one for new services; staged for legacy.
  2. Add noUncheckedIndexedAccess (TS). Catches the array-of-undefined class of bug.
  3. Required CI check. tsc / pyright / mypy as a required status; merge blocked on error.
  4. Lint rule for bare ignores. Reject any / # type: ignore without a reason + expiry.
  5. Generate types from schemas. Practice 12. OpenAPI / GraphQL / Protobuf → SDK.
  6. Runtime validators at boundaries. Pydantic / Zod / valibot.
  7. Quarterly escape-hatch audit. Expired ignores fail CI; owner fixes or renews.
  8. Stricter over time. Add new strict flags as the codebase matures.

Deep dive — what makes strict typing pay off

Strict from day one beats "we'll strict later". Adding strict mode to a 50 k-LOC TS codebase surfaces thousands of errors; the team fixes the obvious ones and adds any to the rest. Six months later there are 500 anys in the code and strict mode is decorative. Start strict on day one; pay the small cost continuously.

noUncheckedIndexedAccess is the modern bump. TS's "strict": true doesn't cover array[0] returning T | undefined as it should. noUncheckedIndexedAccess does. Modern TS teams enable it; older codebases discover thousands of latent bugs the first time they try.

Escape hatches need reasons and expiries. Bare any / # type: ignore multiply forever. // @ts-expect-error TODO(2026-08-15) library v3 lacks types has a reason and a date. Lint rule rejects bare hatches; CI fails on expired ones.

Generated types beat hand-rolled. OpenAPI / GraphQL / Protobuf → typed SDK. The SDK can't drift from the API by construction. Hand-rolled DTOs are the antipattern; Practice 12 covers the full pipeline.

Pair static with runtime validators. Static catches internal calls; runtime catches boundary inputs (JSON, env, form). Pydantic / Zod share the same type model as the static checker. Both layers together cover what either alone misses.

Untyped deps need typed wrappers. Some libs have no types. Don't any them — wrap in a thin typed adapter. The hatch lives in one file, not scattered across callers; the type contract you wrote is what the rest of the code sees.

Pyright over mypy for new Python. Pyright is faster and stricter by default. Same type-model (PEP 484); different runtime. For new projects, pick pyright; for existing mypy installs, migrate when convenient.

Lifecycle of a type

Defined

From schema (preferred), domain model, or boundary validator. Single source.

Checked

tsc / pyright / mypy / rustc passes in CI as required check.

Validated at boundary

Pydantic / Zod parses external input into the type. Bad shape rejected at the door.

Refactored confidently

Rename / extract / move guided by the type checker; every caller surfaced.

Escape-hatched (rare)

Reason + expiry written. CI fails on expiry; owner fixes or renews.

Retired

Type removed when its concept is gone. Compiler ensures no callers remain.

Worked example — strict TS + pyright + Zod

Three configs for the Study App stack. TS strict + noUncheckedIndexedAccess; pyright strict; Zod at HTTP boundary.

# tsconfig.json (frontend + Node)
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "exactOptionalPropertyTypes": true,
    "useUnknownInCatchVariables": true,
    "verbatimModuleSyntax": true
  }
}

# pyproject.toml (Python services)
[tool.pyright]
typeCheckingMode = "strict"
reportMissingTypeStubs = true
strictListInference = true
strictDictionaryInference = true

[tool.ruff.lint]
extend-select = ["ANN", "TCH"]   # require type annotations

# zod boundary validator (TS)
import { z } from "zod";
const Attempt = z.object({
  questionId: z.string().uuid(),
  answer:     z.string().max(4000),
});
type Attempt = z.infer<typeof Attempt>;

app.post("/attempts", (req, res) => {
  const parsed = Attempt.safeParse(req.body);
  if (!parsed.success) return res.status(400).json(parsed.error);
  // parsed.data is fully typed as Attempt
  return scoreAttempt(parsed.data);
});

# CI workflow (excerpt) — required checks
- run: npx tsc --noEmit
- run: pyright --warnings
- run: cargo check --all-targets

Strict everywhere; runtime validation at the HTTP boundary; CI checks block merge. noUncheckedIndexedAccess + exactOptionalPropertyTypes are the modern TS bumps; pyright strict for Python.

Quality bar

  • Strict mode in config. TS strict + noUncheckedIndexedAccess; pyright/mypy strict.
  • Type check in CI as required. Merge blocked on error.
  • Lint rule for bare ignores. Reason + expiry required.
  • Types generated from schemas. OpenAPI / GraphQL / Protobuf → SDK (Practice 12).
  • Runtime validators at boundaries. Pydantic / Zod / valibot; shared type model.
  • Typed wrappers for untyped deps. Hatch in one file, not scattered.
  • Quarterly escape-hatch audit. Expired ignores fail CI; renewed with reason.
  • Stricter over time. New strict flags added as codebase matures.
