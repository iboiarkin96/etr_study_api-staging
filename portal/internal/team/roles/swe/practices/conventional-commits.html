Conventional Commits — machine-readable history

A simple grammar over commit subjects turns the git log into the source of truth for changelogs, SemVer bumps, and release notes — automated and impossible to forget.

TL;DR

Conventional Commits 1.0 (conventionalcommits.org), derived from Angular's commit conventions, defines a small grammar: type(scope): subject with a fixed type allow-list (feat / fix / docs / refactor / chore / test / build / ci / perf / style) and an explicit ! or BREAKING CHANGE: footer for breaking changes. Tools like semantic-release, release-please, and changesets read the log, decide the next SemVer bump (major / minor / patch), and generate the CHANGELOG automatically. The git log itself becomes the release ledger.

What it is

Conventional Commits is a specification (1.0.0) for commit-message structure. First line: type(scope): subject. Type is one of a fixed allow-list (feat, fix, docs, refactor, chore, test, build, ci, perf, style); scope is an optional one-word identifier of the component; subject is a short imperative-mood description.

Breaking changes are signalled with an exclamation mark after the type — feat(auth)!: drop password login — or a BREAKING CHANGE: footer in the body. That single bit drives SemVer: feat = minor, fix = patch, any ! = major.

The grammar is enforced at commit time (commitlint as a pre-commit hook) and at PR time (PR-title linting, because squash-merge uses the PR title). Release tools — semantic-release, release-please, python-semantic-release, changesets — read the log between two tags and produce the next version, a CHANGELOG entry per category, and the git tag itself.

Why we need it

Without a commit grammar, "what changed in 2.3.0?" requires re-reading every diff. Changelog generation is manual — tedious, skipped, or generated by ChatGPT after the fact, none of which produce a release ledger anyone trusts. Version bumps are guesses: someone judges, often wrongly, whether a release is "feature enough" for a minor or "just bug fixes" for a patch.

Structured commits flip the process. The release tool reads the log and produces a deterministic answer: this set demands a major bump because there is one !; CHANGELOG entries are these, grouped by type; next tag is v2.4.0. Release becomes a build artefact, not a meeting. Downstream consumers can subscribe to specific change types ("tell me on any feat(api)") and build automation on top.

How others do it

Angular / Nx — origin

Google's Angular team published commit guidelines around 2014; they became the seed for Conventional Commits 1.0 in 2018. Nx (monorepo build) and Angular CLI enforce the grammar in CI. The release pipeline generates the changelog and bumps SemVer from the log; no human edits a CHANGELOG.md.

What we take: The spec works at large monorepo scale. Angular has tens of thousands of conformant commits; discipline is sustained by tooling (commitlint + CI), not by human reviewers reminding each other.

Source: conventionalcommits.org

semantic-release — JS ecosystem default

semantic-release (Stephan Bönnemann) is the most popular automated-release tool in JS. Given a Conventional Commits log it computes the next version, writes CHANGELOG, tags, publishes to npm, drafts the GitHub release, and notifies Slack. The release process is one CI job; no human is in the critical path.

What we take: The full pipeline can be derived from the log. Once you trust the grammar, every step downstream is automation. Hours per release saved; wrong SemVer level structurally prevented.

Source: semantic-release.gitbook.io

Google release-please — PR-driven release

release-please (Google) takes a slightly different approach: instead of releasing on every push, a "release PR" is auto-maintained with the next version bump and proposed CHANGELOG. Engineers see the upcoming release inline and merge the PR when ready. Same commit grammar; gentler trigger.

What we take: For teams not ready for continuous release, release-please's "PR for the next release" is a gentler adoption path. Same grammar; human merge replaces auto-tag. Changelog automation without committing to every-push releases on day one.

Source: github.com/googleapis/release-please

changesets — monorepo per-package versioning

Atlassian's changesets is the JS-monorepo standard for per-package versioning. Engineers add a "changeset" file to each PR declaring the affected packages and bump type. At release time, all pending changesets are aggregated into per-package version bumps + per-package CHANGELOG. Used by Vercel, Vite, Astro, and many other modern monorepos.

What we take: For monorepos shipping many packages, changesets adds explicit per-PR bump intent on top of Conventional Commits. Avoids the "every commit affects every package" failure mode of naive log scanning.

Source: github.com/changesets/changesets

Anatomy of a conventional commit

Type (allow-list of ~10)

Exactly one of feat / fix / docs / refactor / chore / test / build / ci / perf / style. No new types without an RFC.

Scope (optional, allow-listed)

Documented scope vocabulary — 10–30 entries, one per major module. Maintained in commitlint.config; CI rejects unknown scopes.

Subject — imperative, ≤ 72 chars

"add SAML support" not "added SAML support". Lower-case start. No trailing period. Hard line limit so it fits in git log --oneline.

Breaking marker

! after the type — feat(api)!: rename endpoint — or a BREAKING CHANGE: footer with a one-paragraph migration note. Either drives a major SemVer bump.

Body + footers

Optional but encouraged for non-trivial changes. Body explains the why. Footers carry metadata: Refs: #1234, Co-Authored-By:, BREAKING CHANGE:.

commitlint at commit + CI

Hook checks the message locally; CI checks the PR title (squash-merge uses the PR title as the commit message). Both gates use the same config.

How to run it

  1. Adopt the spec. Link to conventionalcommits.org from CONTRIBUTING.md.
  2. Install commitlint. Pre-commit hook locally; PR-title CI check on the server.
  3. Define the scope allow-list. 10–30 entries in commitlint.config. Adding a scope requires a small PR.
  4. Configure squash-merge to use PR title. Set as default; commit messages on the branch become noise that gets squashed.
  5. Wire release automation. semantic-release (every-push) or release-please (release-PR style); changesets for JS monorepos.
  6. Train on the breaking marker. Review checklist: "does this change any public API or wire format?" If yes, ! or footer is required.
  7. Quarterly review. Scope allow-list cleanup; commit-message lint stats; CHANGELOG quality spot-check.

Deep dive — what makes the automation trustworthy

The grammar is intentionally small. Conventional Commits ships with ~10 types and a single breaking marker. It is the smallest grammar that still enables full automation. Larger grammars (custom types per team, sub-scopes, severity levels) collapse under their own weight; engineers stop following them; the log becomes garbage. Resist the urge to extend; the small spec is the feature.

PR-title lint is the load-bearing check. Most teams squash-merge; the PR title becomes the commit message. A pre-commit hook that lints individual branch commits is helpful for the author, but the canonical message is the squash result. Wire commitlint to the PR title via a CI check; reject the PR until the title parses.

Scope allow-list keeps the log groupable. Free-text scopes explode: auth, authn, authentication, login, users all appear for the same module. The "log grouped by scope" CHANGELOG view becomes useless. Pin scopes to an allow-list in commitlint config; new scope = small PR with rationale.

Breaking marker discipline lives in PR review. The most expensive error is a breaking change miscategorised as a fix. SemVer bumps a patch level; downstream consumers update without reading the changelog; integrations break Monday morning. Review checklist asks explicitly about public API/wire format; pair with contract tests (Practice 12) so the breaking change can't sneak through silently.

release-please first, semantic-release later. Continuous release on every push is the mature end state. Most teams start gentler: a release-PR sits open with the proposed next version and CHANGELOG; humans merge it on a cadence. Same grammar, gentler trigger. Once the team trusts the automation for a quarter, flip the switch.

The body is for the why, not the what. A subject line of "fix(checkout): correct EU VAT calc" tells the future reader what; the body is where you explain why the prior calc was wrong, what regulatory note you cited, what the test now asserts. The squash-merge default usually drops the body — keep important context in the PR description instead, and link to it from the commit when needed.

Don't bypass under deadline. The temptation under deadline is "merge with bad message, fix later". Later never comes; the log accumulates entropy; the next changelog regenerates with junk. Make the lint a required check; admin bypass logged and reviewed at the next retro.

Lifecycle of a release

Authored

Engineer writes commits on a branch. Pre-commit hook lints each message.

PR opened

PR title lint runs as a required CI check. Title must parse; scope must be allow-listed; breaking marker checked in review.

Squashed + merged

Squash uses the PR title as commit message. Trunk grows by one conformant commit. release-please / changesets update the pending-release PR.

Release pending

Aggregated release PR or in-pipeline pending version. Engineers review the proposed bump + changelog inline.

Released

Tag created; CHANGELOG.md updated; package published; GitHub release drafted; #releases Slack pinged. No human edits the changelog.

Communicated

Downstream consumers subscribed to specific change types get filtered notifications. Major bumps trigger explicit migration outreach.

Worked example — Study App commitlint + release-please

The commitlint config + release-please workflow used by Study App. Scope allow-list at the top; PR-title lint as a required check; release-please opens an aggregated release PR on every merge.

# commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'docs', 'refactor', 'chore',
      'test', 'build', 'ci', 'perf', 'style'
    ]],
    'scope-enum': [2, 'always', [
      'auth', 'lesson-player', 'lesson-author', 'mastery',
      'identity', 'notifier', 'billing', 'docs-portal',
      'sre', 'web-platform', 'api-guild'
    ]],
    'subject-case': [2, 'always', 'lower-case'],
    'header-max-length': [2, 'always', 72],
  },
};

# .github/workflows/pr-title-lint.yml
name: pr-title-lint
on: { pull_request: { types: [opened, edited, synchronize] } }
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: amannn/action-semantic-pull-request@v5
        env: { GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
        with:
          types: |
            feat
            fix
            docs
            refactor
            chore
            test
            build
            ci
            perf
            style
          requireScope: true
          scopes: |
            auth
            lesson-player
            lesson-author
            mastery
            identity
            notifier
            billing
            docs-portal
            sre
            web-platform
            api-guild

# .github/workflows/release-please.yml (excerpt)
name: release-please
on: { push: { branches: [main] } }
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: google-github-actions/release-please-action@v4
        with:
          release-type: node
          package-name: studyapp

Three files: commitlint rules, PR-title lint workflow, release-please workflow. Scope allow-list maintained in two places (commitlint + PR-title lint); a small refactor could share it via a generated file. Quarterly review checks that the scope list still matches the service tree.

Quality bar

  • commitlint pre-commit hook installed. Lints messages locally before push.
  • PR-title lint as required CI check. Squash-merge uses PR title as the commit message; lint guards it.
  • Scope allow-list in config. Unknown scopes rejected; new scope requires a small PR with rationale.
  • Breaking marker enforced. Review checklist asks explicitly; contract tests catch silent miscategorisation.
  • Release automation in place. release-please or semantic-release; no human writes CHANGELOG.md.
  • Grammar not extended. ~10 types only; custom types rejected; small spec is the feature.
  • Quarterly review. Scope cleanup, lint stats, CHANGELOG quality spot-check.
