Progressive delivery — canary, blue-green, ramped

Every deploy reaches a 1% slice first, gets compared to the baseline on automated health metrics, and only proceeds if those gates stay green.

TL;DR

"Progressive delivery" was coined by James Governor (RedMonk, 2018) as the umbrella for canary / blue-green / ramped rollout plus the automation that makes them safe on every deploy. New version takes 1% of traffic; automated health gates compare error rate, p99 latency, and a business KPI between new and old; rollout pauses or rolls back automatically on regression. Argo Rollouts and Flagger are the OSS standard; Netflix's Spinnaker the historical reference. The principle: bugs at 1% scale cost 1/100th the customer impact of bugs at 100%.

What it is

Progressive delivery is an umbrella over three rollout patterns plus the automation that makes them safe:

  • Canary — new version takes a small slice of traffic (1% → 5% → 25% → 100%); automated gates compare it to the baseline; rollback is the default response to regression.
  • Blue-green — full new environment alongside old; traffic switches all-at-once; rollback is "switch back". Higher cost (2× infra during cutover) but instant rollback.
  • Ramped / linear — gradual traffic shift without distinct stages (1% per minute). Smoother than canary; harder to gate.

The automation makes the patterns viable in 2026: Argo Rollouts (Kubernetes, CNCF), Flagger (CNCF, mesh-integrated), Spinnaker (Netflix), AWS App Mesh / Linkerd / Istio, plus managed PaaS layers (Fargate, Cloud Run, Vercel). The deploy describes a strategy ("canary 1-5-25-100, gate on error_rate + p99") and the platform executes it — no humans clicking buttons at 3 am.

Why we need it

"All-at-once" deploys expose 100% of users to any regression. A bad deploy reaches 100% of users at the same moment; rollback is a redeploy (minutes); revenue impact is the full window × full audience. Progressive delivery slices the impact: a bug detected at 1% affects 1% of users and triggers automatic rollback; the same bug at 100% costs 100× more.

The automation also removes humans from the critical path. Engineers don't watch dashboards at 3 am hoping nothing breaks; the platform watches the gates and rolls back. The deploy becomes a non-event; teams ship more often because each deploy is cheaper.

How others do it

Netflix — Spinnaker + Kayenta

Netflix built Spinnaker (open-sourced 2015) as their multi-cloud delivery platform. Kayenta (the canary-analysis service) compares baseline vs canary on dozens of metrics with Mann-Whitney U tests and produces a pass/fail. Netflix runs thousands of canaries a day; the math is the source of truth, not human judgement.

What we take: Automated canary analysis is the load-bearing piece. Without it, "canary" means "deploy slowly and hope someone notices"; with it, the platform decides.

Source: spinnaker.io

Argo Rollouts — CNCF Kubernetes-native

Argo Rollouts (CNCF Graduated) is the modern Kubernetes-native canary controller. Declarative Rollout resource defines strategy + analysis steps; integrates with Prometheus / Datadog / NewRelic for the metrics; pause / abort / promote actions exposed as CRDs. Used by Intuit, Adobe, BlackRock, and many others.

What we take: Argo Rollouts is the right default for Kubernetes teams in 2026. Declarative; CRD-driven; integrates with whatever metrics backend you already have.

Source: argoproj.github.io/rollouts

Flagger — Flux's progressive delivery

Flagger (Flux Project, CNCF) sits on top of a service mesh (Istio, Linkerd, App Mesh, NGINX) and orchestrates canaries / blue-green / A-B. Strong story for service-mesh users who don't want to adopt Argo. Mesh-level traffic split = no application code change.

What we take: If you're on a service mesh, Flagger is the natural fit. Mesh-level routing makes the canary application-agnostic.

Source: flagger.app

Vercel / Cloudflare — preview deploys + ramped

Vercel and Cloudflare Pages ship PR previews + production ramping out of the box. Every PR gets its own URL (a kind of zero-traffic canary); production rollouts can be ramped via "Skew Protection" and gradual traffic split. Same patterns; managed UI; no Kubernetes required.

What we take: If your stack is on a managed PaaS, use its built-in progressive delivery. Vercel / Cloudflare / Fly / Cloud Run all support canary or ramped; using them is cheaper than rolling your own.

Source: vercel.com/docs/deployments

Anatomy of a progressive-delivery pipeline

Rollout strategy as code

Argo Rollouts CRD / Flagger Canary CRD / Vercel config. Stages, percentages, soak times declared in YAML. Versioned with the app.

Automated analysis

Baseline vs canary comparison on error rate, p99 latency, throughput, business KPI. Statistical test (Mann-Whitney U) or threshold check. Pass → next stage; fail → rollback.

Soak windows

10–60 min per stage. Long enough to see real user impact; short enough that the rollout completes in a working day.

Automatic rollback

Gate failure triggers traffic shift back to baseline. No human in the loop for the first action; humans investigate after.

Mesh or LB traffic split

Service mesh (Istio / Linkerd) or load balancer (ALB / NGINX / Envoy) handles the percentage split. Application code unchanged.

SLO + flag integration

Canary gates align with SLOs (Practice 18). Feature flags (Practice 07) handle the per-user variant inside the new version.

How to run it

  1. Pick the controller. Argo Rollouts (K8s) / Flagger (mesh) / Spinnaker (multi-cloud) / built-in PaaS.
  2. Define rollout strategy as code. Stages, percentages, soak windows. 1 → 5 → 25 → 100 is the canonical start.
  3. Wire analysis to metrics. Error rate + p99 latency + throughput + one business KPI. Threshold or Mann-Whitney.
  4. Configure auto-rollback. Gate failure → traffic shift back. No human in the loop.
  5. Set traffic split at mesh / LB. Istio / Linkerd / ALB / NGINX. Application code unchanged.
  6. Align gates with SLOs. Same metrics as SLO (Practice 18); same thresholds.
  7. Pair with feature flags. Hard rollouts of new versions ride canary; per-user feature variation rides flags.
  8. Postmortem rollbacks. Each auto-rollback gets a written cause; fix lands before the retry deploy.

Deep dive — what makes canary actually safe

Automated analysis beats human watching. A human watching a dashboard at 3 am misses the regression and ships it to 100%. Mann-Whitney U or threshold-based analysis is unblinking; if error rate at the canary differs significantly from baseline, rollback fires. Spinnaker Kayenta and Argo Rollouts AnalysisTemplate both implement this; configure it.

Soak windows are calibrated, not arbitrary. Too short = errors don't surface (e.g. a memory leak shows at 30 min, not 5). Too long = rollout takes a day; engineers skip it. 10–60 min per stage is the canonical range; calibrate to your error budgets.

Auto-rollback is the load-bearing feature. "We'll roll back if we see a problem" is human; "the controller rolls back at gate failure" is automation. The two have radically different MTTR. Engineers should arrive at a rollback that already happened, not be the trigger.

Mesh / LB traffic split keeps app code clean. The application doesn't know which version it is. The mesh (Istio / Linkerd) or LB (Envoy / ALB) routes the percentage. Application code unchanged means every service can be canaried with the same machinery.

Pair with feature flags. Canary = "is this version safe?"; feature flag = "should this user see the new feature?". They answer different questions. Use canary for binary deploys; flags for fine-grained variants inside a deployed version. Practice 07 covers flags.

Blue-green is fast rollback at 2× infra cost. Blue-green keeps the old environment fully running until the cutover. Rollback is "switch back" — seconds, not minutes. Cost: 2× infra during cutover. Use for the highest-risk deploys (payments, auth, root-page).

PaaS-native is the cheapest start. If you're on Vercel / Cloudflare Pages / Cloud Run, their built-in ramped rollout is one config line. No Argo / Flagger / Spinnaker needed. Reach for the heavier tools when you outgrow the PaaS.

Lifecycle of a progressive rollout

Triggered

CI merges to main → image built → rollout CRD updated. Controller starts the canary.

Canary at 1%

Mesh shifts 1% of traffic. Analysis runs against baseline. Soak window starts.

Analysed

Mann-Whitney / threshold check on error / p99 / KPI. Pass → advance; fail → rollback.

Advanced

5% → 25% → 100%, same gates at each step. Soak between stages.

Stable at 100%

Old version standby for a soak window (1–2 h). Then retired; new version is the only one.

Rolled back (if needed)

Auto-rollback at gate failure. Engineers arrive to an already-rolled-back state. Postmortem; fix; retry.

Worked example — Argo Rollouts canary

An Argo Rollouts Rollout + AnalysisTemplate for the lesson-player service. 1 → 5 → 25 → 100 with Mann-Whitney analysis against Prometheus error rate and p99 latency.

# k8s/lesson-player/rollout.yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata: { name: lesson-player }
spec:
  replicas: 12
  strategy:
    canary:
      canaryService: lesson-player-canary
      stableService: lesson-player-stable
      trafficRouting:
        istio:
          virtualService: { name: lesson-player, routes: [primary] }
      steps:
        - setWeight: 1
        - pause: { duration: 10m }
        - analysis: { templates: [{ templateName: lesson-player-health }] }
        - setWeight: 5
        - pause: { duration: 20m }
        - analysis: { templates: [{ templateName: lesson-player-health }] }
        - setWeight: 25
        - pause: { duration: 30m }
        - analysis: { templates: [{ templateName: lesson-player-health }] }
        - setWeight: 100
  selector: { matchLabels: { app: lesson-player } }
  template:
    metadata: { labels: { app: lesson-player } }
    spec: { containers: [{ name: lesson-player, image: studyapp/lesson-player:GIT_SHA }] }

---
apiVersion: argoproj.io/v1alpha1
kind: AnalysisTemplate
metadata: { name: lesson-player-health }
spec:
  metrics:
    - name: error-rate
      successCondition: result < 0.005             # < 0.5% error rate
      failureLimit: 1
      provider:
        prometheus:
          address: http://prometheus.monitoring:9090
          query: |
            sum(rate(http_requests_total{service="lesson-player",
              version="canary",status_code=~"5.."}[5m]))
            /
            sum(rate(http_requests_total{service="lesson-player",
              version="canary"}[5m]))
    - name: p99-latency
      successCondition: result < 0.250              # < 250 ms p99
      failureLimit: 1
      provider:
        prometheus:
          address: http://prometheus.monitoring:9090
          query: |
            histogram_quantile(0.99, sum(rate(
              http_request_duration_seconds_bucket{
                service="lesson-player",version="canary"
              }[5m]
            )) by (le))

One Rollout CRD + one AnalysisTemplate; Argo Rollouts handles the rest. Failure of either metric for one analysis run aborts the rollout and shifts traffic back. Engineers arrive to a rolled-back state; postmortem; fix; redeploy.

Quality bar

  • Rollout strategy as code. Argo CRD / Flagger CRD / equivalent. Versioned with the app.
  • Automated analysis on every stage. Error rate + p99 + KPI; pass/fail by controller, not human.
  • Soak windows 10–60 min per stage. Calibrated to slowest-to-surface failure mode.
  • Auto-rollback on gate failure. No human in the critical path.
  • Mesh / LB traffic split. Application code unchanged.
  • Postmortem on every rollback. Fix lands before retry.
  • Aligned with SLOs. Same metrics, same thresholds.
  • Paired with feature flags. Canary for version; flags for variant.
