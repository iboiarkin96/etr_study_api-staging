Tech-debt register — explicit, ranked, quarterly

Every known piece of debt is written down, scored on impact × effort, and gets a fixed slice of next quarter's capacity.

TL;DR

Tech debt is Cunningham's 1992 metaphor; Fowler's quadrant (deliberate/inadvertent × prudent/reckless) is the standard classification. A tech-debt register is the team's single tracked list of every known piece of debt, scored on interest × payoff, owned by a named engineer. Reviewed monthly with leadership; 15–20% of every sprint's capacity protected for payoff. Debt that is not written down doesn't exist for planning and the team keeps adding to it. Pair with Architect Practice 21 (Tech-debt management) for the strategic framing.

What it is

A tech-debt register is the team's single tracked list of debt items. Each item has five attributes (mirrors Architect Practice 21):

  • Description — specific enough that a new engineer understands it. Tagged with Fowler's quadrant (deliberate/inadvertent × prudent/reckless).
  • Interest — engineer-days per sprint of slowdown. Estimated, not measured exactly; preferences without interest are not debt.
  • Payoff — engineer-days to fix, with a 30% buffer for unforeseen complications.
  • Owner — named engineer, not "TBD". Without an owner the item goes stale.
  • Priority — derived from interest ÷ payoff. High-interest, low-payoff items pay back fastest.

Stored as a GitHub Issues project, Linear, or Jira board. 15–20% of sprint capacity protected for payoff. Monthly leadership review surfaces top-5 by interest; quarterly cleanup retires stale items. The Architect-level page (Practice 21) covers the strategic framing; this page is the developer's workflow.

Why we need it

Debt that isn't written down doesn't exist for planning. Every engineer has a mental list and complains about a different subset; the team agrees nothing is the worst. Meanwhile interest accrues invisibly: 30% of every feature's time spent working around a broken auth abstraction; nobody sees the total cost because it's amortised across all work.

The register makes interest visible. "the auth wrapper costs us 3 days/sprint across the team — pays back the 20-day fix in two months" becomes a business case. Leadership can prioritise debt against features in a transparent forum instead of in a hallway conversation. Debt that gets ignored is debt nobody put on the register — fix that first.

How others do it

Ward Cunningham — the metaphor (1992)

Cunningham coined "technical debt" at OOPSLA 1992 to explain to non-engineers why quick-and-dirty code accrues a long-term cost. Later clarified: the metaphor applies specifically to deliberate debt; accidents are something else (Fowler's quadrant covers the rest).

What we take: Tag every register entry with Fowler quadrant — deliberate vs inadvertent × prudent vs reckless. Different flavours need different responses (training vs process change vs scheduled payoff).

Source: c2.com — Ward explains the metaphor

Martin Fowler — debt quadrant

Fowler's 2009 post on TechnicalDebtQuadrant classifies debt: deliberate vs inadvertent × prudent vs reckless. Reckless inadvertent → training; reckless deliberate → process change; prudent → scheduled payoff. The classification drives the response.

What we take: Use the quadrant as a diagnostic tool, not just a label. Reckless inadvertent = skills gap; reckless deliberate = planning failure; prudent = normal engineering tracked in the register.

Source: Fowler — Technical Debt Quadrant

ThoughtWorks — 15–20% sprint allocation

ThoughtWorks consulting practice documents 15–20% sprint capacity as the canonical protected allocation for debt payoff. Less and debt grows faster than it's paid; more and feature velocity suffers. Calibrated by team experience over decades of client engagements.

What we take: Protect 15–20% of every sprint for debt payoff. Continuous small payoff, not periodic dedicated sprints. Dedicated sprints are easy to cancel; continuous allocation is not.

Source: ThoughtWorks — Managing technical debt

Linear / GitHub Projects — tooling

Linear (and GitHub Projects, Jira) host the register as a labelled board with custom fields for interest + payoff + owner + Fowler tag. Filters and sorts give the team a live view; integrations let CI report on aged items.

What we take: Tool choice is less important than schema discipline. Whatever tool the team uses, enforce the five columns (description, interest, payoff, owner, priority) consistently.

Source: linear.app

Anatomy of the register

Single shared location

GitHub Issues project / Linear / Jira board. One location; linked from team README; not 5 spreadsheets.

Measurable interest

Eng-days / sprint of slowdown. Estimated by team. Items without it are rejected at intake — preferences, not debt.

Fowler quadrant tag

deliberate/inadvertent × prudent/reckless. Drives response (training / process / scheduled payoff).

Named owner

Engineer / team lead. No "TBD". Owner is who reports on it at monthly review.

15–20% capacity protected

Continuous small payoff every sprint. Feature overruns cut scope, not debt time.

Monthly review + quarterly cleanup

30-min monthly review with leadership; top-5 by interest. Quarterly cleanup retires irrelevant items.

How to run it

  1. Pick the location. Linear / GitHub Projects / Jira with custom fields. Link from team README.
  2. Seed from pain. Each senior engineer contributes 5–10 items that cost them most. 60-min consolidation session.
  3. Estimate interest + payoff. Per item. Rough is fine; discipline of estimating beats precision.
  4. Compute priority. Interest ÷ payoff. P1: ratio ≥ 0.3 (pays back < 4 mo). P2: 0.1–0.3. P3: < 0.1.
  5. Protect 15–20%. Continuous; not dedicated sprints. Feature overruns scope-cut, not debt-cut.
  6. Monthly leadership review. 30 min. Top-5 by interest, closed items, added items, ratio trend.
  7. Quarterly cleanup. Retire irrelevant items; aim for 30–50 items per team.

Deep dive — making debt visible

Interest is the load-bearing field. Items without measurable interest are preferences. Reject at intake. The interest field converts "we should clean this up" into "this costs 3 eng-days/sprint, pays back the 20-day fix in 2 months". Leadership argues with preferences; can't argue with payback periods.

Ratio over taste. Priority = interest ÷ payoff. Brutal: a beloved refactor with 0.5 days/sprint interest and 60-day payoff loses to an ugly 2-day fix with 2 days/sprint interest. Trust the ratio; track the wins; the math earns trust.

Fowler quadrant drives the response. Reckless inadvertent ("what's layering?") = skills gap → mentor / training. Reckless deliberate ("no time for design") = planning failure → process change. Prudent debt = scheduled payoff (the register). Different responses for different causes.

Protect 15–20% from the inside. Dedicated debt sprints get cancelled under pressure. Continuous small payoff every sprint is invisible to cancel. Feature overruns scope-cut, not debt-cut.

Owners + monthly review keep it alive. Unowned items go stale. Without monthly review, the register sits unused. Both rituals are short (30 min) and high-leverage.

Quarterly cleanup keeps it finite. Items added, never closed = backlog of 200 items, nobody reviews it. Aim for 30–50 items per team. Stale entries retired; not all debt deserves the register.

Lifecycle of a debt item

Proposed

Engineer adds with description + Fowler tag. Interest + payoff drafted.

Triaged

Tech lead validates: measurable interest? Real debt or preference? Accept / merge / reject.

Prioritised

Ratio computed; P1/P2/P3 slotted; owner confirmed; ranked within tier.

In progress

Pulled into a sprint within the 15–20% allocation. Owner driving.

Paid off

Fix landed; interest no longer accruing. Closed with a short note. Featured in next monthly review.

Retired

System removed / replaced; no payoff needed. Closed with explanation. Quarterly cleanup catches stragglers.

Worked example — Study App debt register entry

A representative register entry in Linear (or equivalent). Five fields + Fowler tag + a comment trail.

D-12 · lesson-state cache invalidated on every write
─────────────────────────────────────────────────────
description:  Cache invalidated on every Lesson write. Full DB hit on
              every lesson load; adds ~120 ms p99 on /lesson/{id}/state.
flavour:      prudent · inadvertent       (Fowler quadrant)
interest:     3.0 eng-days / sprint       (whole team feels it)
payoff:       18 eng-days (incl. tests + canary rollout)
ratio:        0.17  → P1
owner:        @oleg
status:       in progress · target 2026-06-12
linked PRs:   #4521, #4522

comments:
  2026-05-12 @anna   - confirmed via SLO burn-rate spike on 2026-05-08
  2026-05-15 @oleg   - design doc DD-018; cache key strategy decided
  2026-05-22 @oleg   - first PR landed; 40% improvement on canary

One row per debt item; five core fields + Fowler tag + a comment trail. Linked PRs let leadership see progress without a status meeting; the ratio + Fowler tag make priority + diagnosis explicit.

Quality bar

  • Single shared register. Linear / Issues / Jira. Linked from README.
  • Measurable interest per entry. Preferences without cost rejected.
  • Named owner per entry. No "TBD".
  • Fowler quadrant tag. Drives response (training / process / payoff).
  • Priority = interest ÷ payoff. Ratio over taste.
  • 15–20% sprint capacity protected. Continuous; not dedicated sprints.
  • Monthly leadership review. 30 min; top-5 by interest; closed + added items.
  • Quarterly cleanup. 30–50 items per team; stale ones retired.
