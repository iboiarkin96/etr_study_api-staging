Structured logging — JSON, context, severity

log.info("checkout.submitted", order_id=42, amount_cents=1999) — every line a queryable event, joined to traces and metrics by correlation IDs the logger fills in automatically.

TL;DR

Structured logging = every log line is a JSON object with fixed schema (timestamp, level, service, trace_id, message + domain fields). Modern libraries — structlog (Python), pino (Node), zerolog (Go), slog (Go stdlib since 1.21) — make the call-site cost essentially zero. The job: pick a verb-shaped message key, attach the IDs the rest of the system joins on, stop writing prose into log messages. Payoff: "find the bug" becomes a SQL-like query instead of multi-tab grep through gigabytes.

What it is

Structured logging emits each log line as a JSON object with a fixed schema:

  • Standard fields — timestamp, level, service, trace_id, span_id, message (verb-shaped key: order.completed, not "order completed!").
  • Context fields — user_id, tenant_id, request_id, feature flags. Attached via middleware once per request.
  • Domain fields — per call-site values (order_id, amount_cents, retry_count). Discrete keys, not interpolated prose.
  • Severity / level — debug / info / warn / error. ERROR pages on-call; INFO is analytics.

Modern libraries — structlog (Python), pino (Node), zerolog (Go), slog (Go stdlib 1.21+), serilog (.NET), Logstash JSON encoder (Java) — make the call-site cost equal to a free-text log. The discipline is naming + attaching context, not the encoding.

Why we need it

Free-text logs are grep contests at incident time. The engineer writes regex against gigabytes; per-engineer styles ("order 42 done" vs "completed order id=42") make queries unreliable; no field types means amount comparisons are string ops. Worst: there's no trace_id, so logs and traces live in separate silos.

Structured logs invert it. "find all order.completed where amount > 10 000 and tenant = enterprise in the last hour" is a one-line query in any backend (Loki, Elasticsearch, Datadog, Honeycomb). The trace_id field links every log line to its distributed trace. The incident-time MTTR drops from hours of spelunking to minutes of querying.

How others do it

structlog / pino / zerolog — modern libraries

Hynek Schlawack's structlog (Python), pino (Node, JS), zerolog (Go), slog (Go stdlib 1.21+) all share the same shape: bound contextvars + keyword arguments + JSON renderer. Call-site cost is identical to print; output is queryable JSON.

What we take: Adopt a modern structured-logging library per language. Don't roll your own; the ecosystem is mature.

Source: structlog.org

Grafana Loki — query-by-label

Loki indexes only labels (small set of low-cardinality fields); content is searched via LogQL. Combined with Grafana, the same UI shows logs + metrics + traces; click a trace span to see its logs; click a log to see its trace. The promise of "three pillars" delivered.

What we take: Pair structured logs with a backend that joins to traces. Loki + Grafana is the OSS path; Datadog / Honeycomb the SaaS path.

Source: grafana.com/oss/loki

OpenTelemetry Logs

OpenTelemetry's Logs signal (GA in 2024) standardises the wire format and auto-injects trace_id/span_id. Auto-instrumentation libraries make existing loggers OTel-aware without code changes. Vendor-neutral; portable; the modern default for new services.

What we take: Pipe logs through the OTel Collector. trace_id correlation comes free; backend choice is a Collector config.

Source: opentelemetry.io/docs/concepts/signals/logs

Stripe / Honeycomb — "wide events"

Stripe's engineering blog and Honeycomb's observability framing both push "wide events": one log per business action with many high-cardinality fields (user_id, build_sha, feature flag). Rather than many narrow logs, emit one wide event at the boundary. Same cost; richer analysis.

What we take: One wide event per business action beats many narrow logs. Attach high-cardinality fields generously; cardinality is the analysis surface.

Source: honeycomb.io — wide events

Anatomy of structured logging

JSON encoder by default

No free text in prod. Library: structlog / pino / zerolog / slog / serilog. Pretty-print only in dev.

Verb-shaped message keys

order.completed, auth.failed, db.query.slow. Stable across versions; queryable.

Context middleware

One bind per request: user_id, tenant_id, trace_id, build_sha. Every log in the request inherits them automatically.

Trace correlation

trace_id + span_id on every log line. Click log → see trace; click trace → see logs.

PII deny-list

Server-side scrubber drops password / token / credit-card fields before storage. Defence-in-depth; client-side allow-list at the SDK.

Sampling for high-volume INFO

100% errors; 100% slow requests; 1–10% of normal INFO. Saves cost without losing signal.

How to run it

  1. Adopt a structured-logging library per language. structlog / pino / slog / serilog.
  2. Configure JSON encoder for prod. Pretty-print for dev only.
  3. Add context middleware. Per-request: user_id, tenant_id, trace_id, build_sha.
  4. Wire trace correlation. OTel logging instrumentation or manual.
  5. Replace free-text logs. log.info("order.completed", order_id=42, amount_cents=1999) shape.
  6. Configure PII scrubber. Deny-list at the SDK + server-side as defence-in-depth.
  7. Sample high-volume INFO. 100% errors; 1–10% normal.
  8. Cross-link in observability UI. Click log → trace → metrics navigation.

Deep dive — what makes logs queryable

Verb-shaped keys + discrete fields. "order completed for user 42 at $19.99" is unstructured. "order.completed" + {order_id, user_id, amount_cents} is structured. Same information; vastly different queryability. Adopt the verb-shape convention and stick to it across services.

Context middleware is the unlock. Manually attaching user_id + tenant_id + trace_id to every log call is tedious; engineers stop doing it. Bind context once per request in middleware; every log in the request inherits automatically. Now every log has the join keys without per-line effort.

trace_id is the load-bearing field. Without it, logs and traces live in silos. With it, one click moves between them. The middleware that adds trace_id to every log call (OTel logging instrumentation handles this) is the highest-leverage line of code in the logger config.

Wide events beat narrow logs. Honeycomb's "one rich event per business action" framing is more useful than "ten skinny logs per request". Attach high-cardinality fields generously (user, tenant, flag, build); the analysis surface scales with cardinality, not with log volume.

Sample, don't drop. 100% errors + 100% slow + 1–10% normal saves 90%+ of cost while preserving signal. Don't sample errors; don't sample your slow tail; do sample steady-state INFO.

PII is a day-one concern. A free-text log of "user submitted password=hunter2" is a privacy incident. Structured logs make scrubbing tractable: deny-list field names at the SDK + server-side. Verify with a test capture before prod traffic.

Pretty-print only in dev. Engineers see colored, pretty logs locally; prod ships JSON. Same logger, different renderer. Don't ship pretty-printed logs to a backend — they're not queryable.

Lifecycle of a log line

Emitted

Service calls log.info("verb.event", k=v). Logger renders JSON; context middleware adds trace_id + user_id + tenant.

Scrubbed

PII deny-list strips sensitive fields client-side; server-side scrubber runs as defence in depth.

Sampled

100% errors / slow; 1–10% INFO. Sample decision at the OTel Collector or logging library.

Exported

OTLP / Fluent Bit / Vector ships the JSON to Loki / ES / Datadog / Honeycomb.

Queried

Engineer runs LogQL / SQL-like query. Filters on fields, joins on trace_id. MTTR drops.

Aged out

Hot 7 d → warm 30 d → cold S3 90 d → deleted. Retention tier per business need.

Worked example — structlog setup

Python structlog config + FastAPI middleware that attaches user / tenant / trace_id on every log line. Verb-shape lint rule catches f-strings.

# app/logging.py
import logging, structlog, os
from structlog.contextvars import bind_contextvars, clear_contextvars

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.processors.dict_tracebacks,
        # OTel adds trace_id/span_id via OTel logging instrumentation
        structlog.processors.JSONRenderer() if os.getenv("ENV") != "dev"
            else structlog.dev.ConsoleRenderer(),
    ],
    wrapper_class=structlog.make_filtering_bound_logger(logging.INFO),
)
log = structlog.get_logger()

# app/middleware.py
async def context_mw(request, call_next):
    clear_contextvars()
    if hasattr(request.state, "user"):
        bind_contextvars(
            user_id=request.state.user.id,
            tenant=request.state.user.tenant_id,
            build_sha=os.getenv("GIT_SHA"),
        )
    return await call_next(request)

# usage at call-sites — verb-shape + discrete fields
log.info("lesson.completed", lesson_id=lesson.id, score=score.raw, ms=elapsed_ms)
log.warning("attempt.duplicate", attempt_id=a.id, reason="seen-already")
log.error("payment.charge.failed", order_id=o.id, code=err.code, retry=3)

# .ruff.toml (lint rule excerpt)
[lint]
extend-select = ["G"]   # flake8-logging-format; bans f-strings in log calls

Quality bar

  • Structured library per language. structlog / pino / zerolog / slog / serilog. JSON in prod.
  • Verb-shaped message keys. order.completed, not free text.
  • Context middleware. Per-request user / tenant / trace_id / build_sha.
  • trace_id on every log. OTel logging instrumentation or equivalent.
  • PII deny-list verified. Test capture confirms scrubbing.
  • Sampling configured. 100% errors / slow + 1–10% INFO.
  • Log → trace navigation works. One click from Grafana / Loki / Honeycomb.
  • Lint rule against f-string-in-log. Ruff / eslint / golangci-lint.
