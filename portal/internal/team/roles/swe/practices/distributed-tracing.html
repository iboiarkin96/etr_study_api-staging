Distributed tracing — OpenTelemetry spans on every call

Wrap the interesting business operation in a span and add attributes for the inputs that matter — get an end-to-end waterfall showing exactly which call burned the 800 ms.

TL;DR

OpenTelemetry (CNCF, 2019 merger of OpenTracing + OpenCensus) is the universal wire format for distributed traces. Ancestor: Google's Dapper paper (2010), turned into product by Honeycomb. Auto-instrumentation gives HTTP/DB spans for free; manual spans on the interesting business operations are what turn "the request was slow" into "the third call to pricing.compute_quote took 820 ms because the FX cache missed". W3C trace-context propagates the trace ID across services; the backend (Honeycomb, Jaeger, Tempo, Datadog) renders the waterfall.

What it is

Distributed tracing models a request as a tree of spans. Each span has a start, an end, a name, and a bag of attributes; spans are linked by parent-child relationships. The whole tree is a trace, identified by a 16-byte trace ID that propagates across services via the W3C traceparent header (or equivalent for Kafka, gRPC, queues).

OpenTelemetry (OTel) is the universally adopted spec + SDK suite:

  • SDK per language (Go, Java, Python, Node, .NET, Ruby, Rust, …) creates spans and exports them via the OTLP protocol.
  • Auto-instrumentation packages (HTTP servers, DB clients, HTTP clients) emit spans for the obvious things without code changes.
  • Manual spans wrap business operations — the parts the engineer cares about — and attach domain-specific attributes (user ID, tenant, feature flag).
  • OTel Collector sits between the apps and the backend: batches, samples, enriches, and routes to one or more backends (Honeycomb, Jaeger, Tempo, Datadog, New Relic).

The point of tracing is not "more data". The point is to answer "what happened in this request?" in seconds — by jumping from a slow trace ID in the logs, into the waterfall, into the span that holds the slow operation.

Why we need it

Logs tell you what one service did; metrics tell you the aggregate; only traces tell you the full path of one request across N services. Without traces, a "checkout is slow" report becomes hours of cross-service log spelunking — which service was called when, with what arguments, and which one took the time. With traces, the engineer pastes the trace ID into the backend UI and sees the waterfall: 820 ms in pricing.compute_quote, with attributes showing the cache miss.

Traces also enable analytics: percentile latency per endpoint per tenant per feature flag, all from the same data. Honeycomb's "BubbleUp" and similar tools make outlier analysis trivial when the spans carry the right attributes. The investment in manual spans (one decorator per business function) pays back the first time a senior engineer doesn't need to spelunk logs to find a latency root cause.

How others do it

Google — Dapper paper (2010)

Google's Dapper paper (Sigelman et al., 2010) introduced distributed tracing at planetary scale. Key contributions: sampled tracing (don't trace everything), parent-child span model, low-overhead in-process tracking. Every modern tracing system is a descendant of Dapper.

What we take: Sampling is non-negotiable at scale. 100% sampling at 10 k RPS produces TB/day of trace data nobody queries. Default to 1% with tail-based sampling that keeps all errors.

Source: research.google — Dapper paper

OpenTelemetry — CNCF standard

OpenTelemetry (2019 merger of OpenTracing + OpenCensus) is the CNCF-incubating, now universally adopted standard. SDKs in 11+ languages; auto-instrumentation for major frameworks; OTLP wire protocol; vendor-neutral. Every major backend (Honeycomb, Datadog, NR, Lightstep, Tempo) reads OTLP natively.

What we take: Adopt OpenTelemetry. Avoid vendor-specific SDKs that lock you in. OTel + a vendor backend gives you portability if you change vendors later.

Source: opentelemetry.io

Honeycomb — high-cardinality first

Honeycomb (Charity Majors et al.) pioneered the "observability" reframing of tracing: store wide events with high-cardinality attributes (user ID, build SHA, feature flag) and analyse them interactively. BubbleUp surfaces "what makes the slow requests different?" automatically.

What we take: Spans need attributes, not just durations. Add user/tenant/flag/build-SHA on every span; the analysis becomes possible at all. Sparse attributes = the trace tells you it was slow; rich attributes = the trace tells you why.

Source: honeycomb.io

Grafana Tempo — OSS at scale

Grafana Tempo (Cortex's tracing sibling) is the open-source backend that scales tracing horizontally on cheap object storage. Pairs with Grafana + Loki + Mimir for the full open-source observability stack. Used by teams that want OTel without per-trace pricing.

What we take: Open-source is viable at scale. Tempo on S3 plus Grafana's TraceQL gives you Honeycomb-style analysis without the per-trace bill. Pick by team size + budget, not by hype.

Source: grafana.com/oss/tempo

Anatomy of distributed tracing

OpenTelemetry SDK + auto-instrumentation

Per-language SDK installed on every service. Auto-instrumentation packages emit HTTP/DB/queue spans without code changes.

Manual spans on business ops

Decorator / context manager around the interesting business function. Attributes: input IDs, business outcome, feature flag. 5–10 manual spans per service is the right size; 100+ is noise.

W3C trace-context propagation

traceparent header on every outbound HTTP / gRPC call. traceparent in Kafka headers / SQS message attributes for async hops.

OTel Collector sidecar / gateway

Receives OTLP from apps; samples (head-based or tail-based); enriches with infra metadata (pod, region, image SHA); exports to one or more backends.

Tail-based sampling

Sample 1–5% of normal traffic but keep 100% of errors + 100% of slow traces. Configured in the Collector; saves cost without hiding anomalies.

Trace ID in logs

Every log line carries the current trace ID. Click a log entry; jump to the trace; debug in seconds. Without this link, traces and logs live in separate silos.

How to run it

  1. Install the OTel SDK + auto-instrumentation. One per service language. Free HTTP/DB/queue spans on day one.
  2. Stand up the OTel Collector. One per cluster as a daemonset / sidecar pattern. OTLP receiver; appropriate exporter for your backend.
  3. Pick a backend. Honeycomb / Tempo + Grafana / Datadog / Jaeger. The OTel layer abstracts the choice; pick by team needs.
  4. Add 5–10 manual spans per service. Wrap business-critical functions. Add attributes: user_id, tenant_id, feature flags, business outcome.
  5. Propagate W3C trace-context. Auto-instrumentation handles HTTP; add explicit propagation for Kafka headers, SQS message attributes, cron jobs.
  6. Configure tail-based sampling. Keep 100% errors, 100% slow (> p99), 1–5% baseline. In the Collector, not the SDK.
  7. Inject trace ID into logs. Logging library or middleware adds trace_id to every log line. Backend supports log→trace navigation.
  8. Wire dashboards. p95 / p99 latency by endpoint + tenant. Error-rate heatmap. Saved queries for common investigations.

Deep dive — what makes traces actually useful

Manual spans on business ops are the load-bearing investment. Auto-instrumentation gives you "HTTP GET /api/checkout 1.2 s" — useful but not enough. Manual spans give you "compute_quote 820 ms, fetch_inventory 250 ms, charge_card 130 ms" — useful enough to debug. Five to ten manual spans per service, on the operations that matter to the business. Skipping manual spans is the most common reason teams adopt OTel and then say it didn't help.

Attributes are the investigation surface. A span with just duration tells you it was slow. A span with user_id, tenant_id, feature_flag, build_sha, region tells you which slow: a single tenant, a specific flag, a specific build, a region. Honeycomb's BubbleUp depends on attributes; so does any percentile-by-attribute analysis. Add the attributes; pay the small cardinality cost; reap the analysis.

Trace ID in logs is the killer feature. "Click a log line, see the trace" collapses MTTR for cross-service investigations. Without it, traces and logs live in silos and engineers context-switch between two backends. With it, one click moves between dimensions. The logging middleware change is small; the operational payoff is large.

Sampling is non-negotiable at scale. 100% sampling at 10 k RPS produces TB/day of trace data nobody queries. Default to tail-based sampling: 1–5% baseline + 100% errors + 100% slow. Tail-based runs in the Collector; the SDK emits everything; the Collector decides what to keep. The biggest cost wins come from getting this right.

OTel Collector enriches centrally. Adding pod name, region, image SHA, deploy SHA at the Collector layer means every span gets them without per-service work. Engineers add business attributes; the infra layer adds infra attributes; analysis joins them automatically.

Don't trace through hot loops. A span inside a per-iteration loop creates thousands of spans per request — both expensive and unreadable. Span the loop as a whole; record per-iteration data as attributes or events on that span. Auto-instrumentation usually gets this right; manual spans need care.

Pair tracing with SLOs. "p99 of compute_quote > 500 ms" is a meaningful SLO; the trace shows you which calls violated it. Without tracing, the SLO is a number; with tracing, it's a number with explainable root causes. SLO + trace + log links are the modern observability stack.

Lifecycle of a trace

Started

Request hits the edge service. Auto-instrumentation creates the root span with the W3C trace ID. Logging middleware adds trace_id to every log line.

Propagated

Every outbound HTTP / gRPC / Kafka call carries traceparent. Receiving services continue the trace as a child span.

Annotated

Manual spans wrap business ops. Attributes: inputs, outcome, user/tenant/flag. Auto-instrumentation continues to emit HTTP/DB spans alongside.

Exported

Spans flow to the OTel Collector via OTLP. Collector enriches with infra metadata; samples by tail-based policy; routes to backend.

Stored

Backend stores spans by trace ID. Indexed by service, endpoint, attributes for query. Retention by sampling tier (errors longer than baseline).

Queried

Engineer pastes trace ID from logs; waterfall renders. Or: BubbleUp / TraceQL surfaces what makes outliers different. MTTR drops from hours to minutes.

Worked example — Python manual span

A decorator wraps the business operation, attaches domain attributes, and lets auto-instrumentation handle the HTTP/DB layer below. One change per business function turns "checkout slow" into "compute_quote 820 ms with cache_miss=true".

# pricing/service.py
from opentelemetry import trace
from opentelemetry.trace import Status, StatusCode

tracer = trace.get_tracer("pricing")

def trace_business(name: str):
    """Decorator: wrap a business op in a span + attach kwargs as attributes."""
    def deco(fn):
        def wrapped(*args, **kwargs):
            with tracer.start_as_current_span(name) as span:
                for k, v in kwargs.items():
                    if isinstance(v, (str, int, float, bool)):
                        span.set_attribute(f"app.{k}", v)
                try:
                    return fn(*args, **kwargs)
                except Exception as e:
                    span.record_exception(e)
                    span.set_status(Status(StatusCode.ERROR, str(e)))
                    raise
        return wrapped
    return deco

@trace_business("pricing.compute_quote")
def compute_quote(*, user_id: str, tenant_id: str, items: list, currency: str):
    span = trace.get_current_span()
    fx = fx_cache.get(currency)              # auto-instrumented Redis span here
    span.set_attribute("app.fx_cache_hit", fx is not None)
    if fx is None:
        fx = fx_service.fetch(currency)      # auto-instrumented HTTP span here
        fx_cache.set(currency, fx)
    return sum(line.amount * fx for line in items)

# logging setup (excerpt) — trace_id appears on every log line
import logging
from opentelemetry.instrumentation.logging import LoggingInstrumentor
LoggingInstrumentor().instrument(set_logging_format=True)
# format includes %(otelTraceID)s · click in log UI → jump to trace

One decorator + one logging line. Every call to compute_quote now appears as a named span with user/tenant/currency/cache-hit attributes. The Redis lookup and HTTP fetch beneath it are auto-instrumented children. Trace ID flows into logs; one click from log line to waterfall.

Quality bar

  • OTel SDK + auto-instrumentation in every service. No vendor SDK lock-in.
  • 5–10 manual spans per service on business ops. Decorator with domain attributes.
  • W3C trace-context propagated. HTTP, gRPC, Kafka headers, SQS attributes.
  • Trace ID in every log line. Logging middleware; log→trace deep-link works.
  • OTel Collector with tail-based sampling. 1–5% baseline + 100% errors + 100% slow.
  • Infra attributes added at the Collector. pod, region, image SHA, deploy SHA.
  • p95 / p99 dashboards per endpoint + tenant. Saved queries for common investigations.
  • SLOs paired with traces. SLO violation → trace showing the violating call.
