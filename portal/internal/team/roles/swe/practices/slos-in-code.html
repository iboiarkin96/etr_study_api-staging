SLOs in code — error budgets that gate releases

SLI as a Prometheus expression, committed next to the service. The burn-rate decides whether the next release ships — instead of negotiating reliability vs features in a meeting.

TL;DR

SLOs originate in Google's SRE Book (2016) and the SRE Workbook chapters on Implementing and Alerting on SLOs. The developer view is "SLOs in code": SLI wired up in the service, SLO target + burn-rate rules live in the repo (Prometheus rules + Pyrra / Sloth), release decision deterministic — budget healthy means ship, budget burning means stop and fix. Pair with the architect-level SLO/SLA design (Architect Practice 17) for the strategic framing; this page is about the developer's implementation.

What it is

"SLOs in code" means the entire SLO machinery lives in the service repository as versioned config:

  • SLI — a Prometheus / OpenMetrics ratio expression: good events / total events.
  • SLO — target + window: "99.9% over 30 days" in a YAML file. Pyrra / Sloth / OpenSLO read it and emit the burn-rate alert rules automatically.
  • Burn-rate alerts — multi-window per the SRE Workbook: 2% / 1 h pages; 5% / 6 h pages; 10% / 3 d tickets.
  • Release gate — CI/CD step queries current burn-rate before promoting a deploy. Budget healthy → ship; burning → block.

The architect-level practice (Architect Practice 17 — SLO & SLA Design) defines strategy: which user journeys, what targets, error-budget policy. This page is the implementation — how the developer wires it into the service.

Why we need it

Without SLOs in code, "is the service reliable enough to ship?" is answered in a meeting. The meeting drifts; reliability work loses; features ship and the next outage forces an unplanned reliability sprint. With SLOs in code, the burn-rate answers the question. Budget healthy = ship; budget burning = stop.

SLOs in code also force the SLI to be measurable in production. Vague targets ("the system should be fast") don't survive contact with a Prometheus expression; concrete targets ("p99 ≤ 200 ms at 5 k RPM over 30 days") do. Writing the SLI in code is the act of forcing the team to agree on what "healthy" means.

How others do it

Google SRE — canonical reference

SRE Book (2016) and SRE Workbook (2018) define SLI / SLO / error budget. The Workbook chapter on Alerting on SLOs introduced multi-window multi-burn-rate alerting (2% / 1 h + 5% / 6 h paging, 10% / 3 d ticket). The recipe is the industry default; every modern SLO tool implements it.

What we take: Use the SRE Workbook burn-rate recipe; don't invent your own. Multi-window multi-burn-rate is the load-bearing alert; static error-rate alerts are the past.

Source: SRE Workbook — Alerting on SLOs

Sloth / Pyrra — SLO generators

Sloth (Slok, OSS) and Pyrra (Polar Signals, OSS) read a YAML SLO spec and emit Prometheus recording rules + burn-rate alert rules. Team writes the SLO once; tool generates the dozens of rule expressions to implement it.

What we take: Don't hand-write burn-rate expressions. Sloth or Pyrra generates them from YAML.

Source: sloth.dev

OpenSLO — vendor-neutral spec

OpenSLO is the open spec for SLO definitions: a YAML format Sloth, Pyrra, and Nobl9 all read. Vendor-neutral; portable; reviewed by CNCF. The right choice for teams that want SLOs portable across observability vendors.

What we take: Write the SLO spec in OpenSLO. Tool choice (Sloth, Pyrra, Nobl9) becomes a vendor decision, not a re-implementation.

Source: openslo.com

Honeycomb / Nobl9 — managed SLO

Honeycomb's SLO module and Nobl9's standalone platform offer managed SLO computation, burn-rate alerting, and dashboards. Either is a viable alternative to running Sloth/Pyrra in-house, particularly for teams that aren't on Prometheus.

What we take: If you're not on Prometheus, use a managed SLO platform. OpenSLO keeps you portable across them.

Source: nobl9.com

Anatomy of SLOs in code

SLI in service code

Prometheus metrics emitted by the service (request_total + status_code). SLI is the ratio. Co-located with the service.

SLO YAML

Per-service slo.yaml with target + window + SLI reference. OpenSLO-formatted. Reviewed in PRs.

Generated alert rules

Sloth / Pyrra generates Prometheus recording rules + multi-window burn-rate alerts. Engineers don't hand-write these.

Burn-rate routing

Fast burn (2% / 1 h) pages on-call; slow burn (10% / 3 d) creates a ticket. Routing in Alertmanager / PagerDuty.

Release gate

CI/CD step queries current burn-rate. If budget below threshold, deploy paused or warning emitted.

Monthly review

30 min: SLO compliance, burn trend, top causes. Tightens / loosens targets based on reality.

How to run it

  1. Pick the user journey. Critical user-facing flow per service.
  2. Emit the SLI metrics. Service exposes http_requests_total + status_code + endpoint.
  3. Write the SLO YAML. OpenSLO format. Target, window, SLI query. PR review.
  4. Generate rules with Sloth / Pyrra. CI step regenerates Prometheus rules on every change.
  5. Wire burn-rate alerts. Multi-window: 2% / 1 h + 5% / 6 h page; 10% / 3 d ticket.
  6. Build the release gate. CI/CD step queries burn-rate via Prometheus; blocks deploy if burning.
  7. Monthly review. SLO compliance, burn-rate trend, target calibration.

Deep dive — what makes "in code" matter

YAML in the repo is what makes SLOs versioned + reviewed. A target in a wiki page drifts; a target in slo.yaml is reviewed in a PR like any other code. Changes have history; rollback is one commit; engineers don't argue about "what did we say last quarter?".

Generate, don't hand-write, alert rules. Multi-window burn-rate involves a dozen recording rules + alert expressions per SLO. Hand-writing them produces typos and inconsistencies; Sloth / Pyrra generate them from the YAML spec.

Burn-rate replaces static error-rate alerts. "Error rate > 1% for 5 min" fires on noise; burn-rate fires on actual budget consumption. Migrate; delete old static alerts; on-call quality jumps.

Release gate makes the decision deterministic. Without it, "do we ship?" is a judgement call. With it, the CI/CD step queries burn-rate and either passes or blocks.

OpenSLO keeps you portable. Vendor-neutral spec; same YAML reads in Sloth, Pyrra, Nobl9. Tool choice becomes a vendor decision, not a re-implementation.

Monthly review tightens calibration. Targets set on day one are guesses. After 90 days of real burn data, refine: tighten where reality beats target; loosen where target is unrealistic. Don't churn weekly; recalibrate quarterly.

Pair with progressive delivery + flags. SLO is the safety net; canary (Practice 10) is the rollout mechanism; flags (Practice 07) are the per-user variant. Three practices, one delivery story.

Lifecycle of an SLO

Drafted

SLI + target + window in slo.yaml. PR review with SRE / architect.

Generated

Sloth / Pyrra emits recording rules + burn-rate alerts. Loaded by Prometheus.

Live

Dashboard live; alerts armed; burn-rate visible. Team monitors weekly.

Burning

Fast / slow burn alert fires. On-call investigates; reliability work prioritised.

Recalibrated

Monthly review tightens or loosens target. PR + commit; version-controlled.

Retired

Service decommissioned. SLO YAML deleted; rules cleaned up; alerts removed.

Worked example — lesson-player Sloth YAML

Sloth SLO YAML for lesson-player + the generated multi-window burn-rate alerts + release gate. Lives at services/lesson-player/slo.yaml; CI regenerates Prometheus rules on every change.

# services/lesson-player/slo.yaml (Sloth · OpenSLO-compatible)
version: prometheus/v1
service: lesson-player
labels: { team: lessons, tier: critical }
slos:
  - name: lesson-state-availability
    objective: 99.9
    description: "GET /lessons/{id}/state succeeds 99.9% of the time"
    sli:
      events:
        error_query: |
          sum(rate(http_requests_total{
            service="lesson-player",
            endpoint="/lessons/_/state",
            status_code=~"5.."
          }[{{.window}}]))
        total_query: |
          sum(rate(http_requests_total{
            service="lesson-player",
            endpoint="/lessons/_/state"
          }[{{.window}}]))
    alerting:
      name: lesson-state-burning
      page_alert:   { labels: { severity: pager  } }
      ticket_alert: { labels: { severity: ticket } }

  - name: lesson-state-latency
    objective: 99.0
    sli:
      events:
        error_query: |
          sum(rate(http_request_duration_seconds_bucket{
            service="lesson-player",endpoint="/lessons/_/state",le="0.25"
          }[{{.window}}]))
        total_query: |
          sum(rate(http_request_duration_seconds_count{
            service="lesson-player",endpoint="/lessons/_/state"
          }[{{.window}}]))

# CI step (.github/workflows/slo.yml)
- name: regenerate Prometheus rules
  run: sloth generate -i services/lesson-player/slo.yaml -o services/lesson-player/slo.rules.yaml
- name: drift check
  run: git diff --exit-code services/lesson-player/slo.rules.yaml

# release gate (.github/workflows/deploy.yml — excerpt)
- name: SLO release gate
  run: |
    BURN=$(curl -s "$PROM/api/v1/query?query=slo:sli_error:ratio_rate1h{slo='lesson-state-availability'}" \
      | jq -r '.data.result[0].value[1]')
    if (( $(echo "$BURN > 0.001" | bc -l) )); then
      echo "fast burn $BURN; blocking deploy"; exit 1
    fi

Two SLOs (availability + latency) declared in YAML; Sloth generates the Prometheus rules + burn-rate alerts; CI fails if the generated file drifts from source; the release gate queries Prometheus and blocks deploy if budget is burning.

Quality bar

  • SLO YAML in the repo. OpenSLO / Sloth / Pyrra format. Reviewed in PRs.
  • Generated rules, not hand-written. Sloth / Pyrra in CI; git diff ensures regeneration.
  • Multi-window burn-rate alerts. Fast burn pages; slow burn tickets; static alerts removed.
  • SLI measurable in production. Prometheus query returns data; histogram buckets exist.
  • Release gate queries burn-rate. Deploy blocked when budget is burning.
  • Monthly SLO review. Compliance, burn trend, target calibration.
  • Paired with progressive delivery + flags. Three practices, one delivery story.
