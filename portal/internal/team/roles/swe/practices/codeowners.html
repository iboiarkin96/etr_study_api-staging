CODEOWNERS & inner-source — every file has an owner

A path-to-team map enforced by the forge; an inner-source contract that says "anyone can propose, the owning team approves".

TL;DR

A CODEOWNERS file — native to GitHub, GitLab, and Gerrit — maps every path in the repo to a team. The forge auto-requests that team's review on any PR touching their files; merge is blocked until they approve. Combined with an inner-source contribution model (Tim O'Reilly's 2000 framing, formalised by InnerSource Commons), this lets a 1 000-engineer org share a monorepo without political traffic jams. "Nobody owns this code" stops being how production fires start.

What it is

The CODEOWNERS file (default location .github/CODEOWNERS on GitHub; CODEOWNERS at repo root on GitLab) is a plain-text map of glob patterns to teams or individuals:

/services/api/**         @org/backend-platform
/services/frontend/**    @org/web-platform
/services/portal/**      @org/docs-platform @org/web-platform
/infra/**                @org/sre
      *                        @org/staff-engineers

When a PR is opened, the forge computes the intersection of changed files and ownership patterns and auto-requests review from each matching team. Branch protection requires approval from at least one member of each requested team before merge.

Inner-source is the cultural layer: anyone in the org can submit a PR to any repo. The owning team holds the merge button — they review, they approve, they merge — but they do not gate who is allowed to contribute. Patterned on open-source: open contribution, owned governance. InnerSource Commons (founded ~2015 by PayPal, Bosch, and others) codifies the patterns.

Why we need it

"Nobody owns this code" is how production fires start. Without explicit ownership, refactors stall in review purgatory because nobody is sure whose approval counts, security patches go unmerged for weeks, and the most-touched files quietly become orphaned. Page-out at 3 am with a broken service and no clear owner: the on-call SRE wakes the wrong person, escalates twice, loses 40 minutes finding the team that wrote the code.

CODEOWNERS makes ownership routable and pageable. Inner-source makes that ownership open without making it ungoverned — a frontend engineer can propose a fix to an API file (lowering cross-team friction) while the API team still controls what lands (preserving conceptual integrity). The combination scales monorepos to thousands of engineers without devolving into either chaos (no ownership) or gatekeeping (only the team can touch their code).

How others do it

GitHub — feature at scale

CODEOWNERS is native to GitHub since 2017. GitHub's own monorepo for github.com (multi-million-line Rails app) is governed by CODEOWNERS with hundreds of patterns and dozens of teams. Branch protection makes CODEOWNERS approval a hard merge gate; the same UI shows the on-call engineer who to ping for any file.

What we take: Adopt CODEOWNERS early — even with a coarse-grained map. The cost grows with the repo; a 20-line file added at 50 k LOC is easy; the same file at 500 k LOC requires hours of archaeology.

Source: docs.github.com — About code owners

InnerSource Commons — governance pattern

InnerSource Commons is a non-profit founded by PayPal, Bosch, SAP and others. It maintains pattern descriptions for "Trusted Committer", "Service vs Library", "Contracted Contributor" — the operational details of inner-source at scale. Together with Spotify's Squad/Tribe model, it gives the cultural template for inner-source-at-scale.

What we take: Inner-source needs both the artefact (CODEOWNERS) and the social contract (a contribution guide describing how outside-team PRs are received). Without the social side, CODEOWNERS becomes a gatekeeping tool.

Source: innersourcecommons.org

Atlassian / GitLab / Gerrit — universal pattern

Bitbucket implements an equivalent "Code owners" feature; GitLab has the same primitive with slightly different syntax (sections, optional vs required reviewers); Gerrit's OWNERS file predates them all, inherited from Chromium. The pattern is universal across forges.

What we take: Don't let "we're on GitLab not GitHub" be an excuse. Syntax differs by ten minutes of documentation; the discipline is identical.

Source: atlassian.com/git/tutorials

Backstage — ownership-as-data

Spotify's Backstage extends ownership beyond CODEOWNERS by modelling services, components, and APIs as Catalog entities with explicit owners. A service Catalog entry names its owner team, on-call rotation, runbook URL, and dependencies. The on-call engineer at 3 am finds the runbook + owner via a single page, not by spelunking through Git history.

What we take: CODEOWNERS gives review-time ownership; a service catalogue (Backstage or equivalent) gives runtime ownership. Pair them — engineers grep CODEOWNERS at PR time, on-call greps the catalogue at 3 am.

Source: Backstage Software Catalog

Anatomy of working code-ownership

CODEOWNERS at repo root

Path patterns from most-specific to most-general. A catch-all * rule at the end guarantees no file is unowned; "files without owners" count in CI must be zero.

Branch protection requires owner approval

Merge blocked until at least one CODEOWNER from each affected pattern approves. Not advisory — enforced by the forge, not by humans.

Teams, not individuals

Patterns map to @org/team-name. Individual usernames create single points of failure ("Alice is on vacation, every PR blocks"); teams distribute.

Inner-source contribution guide

CONTRIBUTING.md explains the cross-team workflow: "anyone can PR; owning team has 48 h to triage; if no response, escalate to the team's manager in #area-help". Sets expectations on response time.

Quarterly ownership review

CODEOWNERS reviewed every quarter alongside org changes: teams renamed, people moved, files reorganised. Stale entries are the #1 source of "PR sits unreviewed for two weeks".

"Orphan files" CI check

Script walks the file tree, computes paths with no CODEOWNERS match, fails CI if the count is non-zero. Run weekly on main; alert the staff engineer if a new orphan appears.

How to run it

  1. Land an initial CODEOWNERS. Even coarse-grained (one team per top-level folder). Catch-all to staff-engineers at the bottom.
  2. Enable forge enforcement. Branch protection: "Require review from Code Owners" on main.
  3. Write CONTRIBUTING.md. 48 h triage SLA per team; escalation path when SLA missed; expectations for outside-team contributions.
  4. Add the orphan-files check. Weekly CI on main; non-zero count fails and alerts.
  5. Pair with the service catalogue. Backstage entries name owner team, on-call rota, runbook — runtime ownership matches CODEOWNERS review-time ownership.
  6. Refine glob patterns over time. Quarterly review: new directories get owners; deprecated teams removed; line distribution audited (no team owns > 60% of PRs).
  7. Track triage SLA. Time-to-first-review per team. Misses get a retro item, not a witch-hunt.

Deep dive — what makes ownership routable

Teams not individuals. The single biggest failure mode is putting @alice in CODEOWNERS instead of @org/payments. Alice goes on PTO; every PR blocks; the team learns to route around CODEOWNERS by self-approving in emergencies. Teams of ≥ 3 distribute the load and survive vacations. Individual handles are technical debt the moment they're added.

Most-specific-wins, with a catch-all bottom line. Glob patterns are evaluated top to bottom on GitHub (last match wins), most-specific-wins on GitLab. Either way, lead with the narrowest patterns and end with a * catch-all so no file is unowned. The catch-all should map to a small staff-engineer team that triages and routes orphan PRs — not a 200-person platform team that becomes the universal bottleneck.

Orphan-files CI is the load-bearing automation. A CODEOWNERS file goes stale silently: directories move, new top-level folders appear, the catch-all gets removed in a refactor. Run a weekly CI job that walks the tree and asserts every path matches at least one rule. Failures alert the staff engineer; the count of orphans is itself a leading indicator of map quality.

Inner-source needs CONTRIBUTING.md. Without a contribution guide, outside-team PRs are received unpredictably — friendly on Tuesday, ignored on Wednesday. The guide sets expectations: how to find the right team, what the triage SLA is, what happens if the SLA is missed, where to ask. Tone matters: invitational, not gatekeeping. "Welcome — here's how this works" beats "you need our approval".

Triage SLA is what makes inner-source work. A team can be the perfect owner on paper and still kill inner-source by taking three weeks to look at outside PRs. 48 h triage (acknowledge + assign) is the industry norm; missing it pages the team manager. Outside contributors who get fast triage keep contributing; those who don't disappear and quietly fork the file.

Pair CODEOWNERS with a service catalogue. CODEOWNERS answers "who reviews this file?". The on-call SRE at 3 am needs "who owns this service, where's the runbook, who's on call right now". Backstage (or an equivalent catalogue) carries that runtime ownership. Both should agree; mismatches get caught at quarterly review.

Quarterly review is not optional. Org reshuffles every few months. Teams get renamed, people move, services get re-homed. A CODEOWNERS file untouched for 12 months almost certainly references deprecated teams. Calendar the quarterly review; pair it with the architecture retrospective.

Lifecycle of an ownership entry

Drafted

New directory or service appears. Owning team proposes a CODEOWNERS entry in the same PR that adds the code.

Active

Forge auto-requests reviews from this team on matching PRs. Triage SLA tracked; the team's catalog entry agrees with CODEOWNERS.

Re-routed

Team split or merged. CODEOWNERS PR updates the pattern to the new team(s). Old team kept as a comment trail for a quarter.

Stale (flagged)

Quarterly review flags entries pointing at teams with < 2 active members, or paths that no longer exist. Action item to fix in two weeks.

Orphaned (CI fail)

Weekly CI finds files with no matching rule. Staff engineer triages; either a new entry is added or the files are moved under an existing owner.

Retired

Code removed. Entry deleted in the same PR. Catalog entry archived. Git history preserves who used to own it.

Worked example — Study App CODEOWNERS

Study App's CODEOWNERS file plus the orphan-files CI job. Coarse-grained at the top, fine-grained where teams overlap, catch-all to staff-engineers at the bottom.

# .github/CODEOWNERS
# Pattern order: most-specific first; catch-all last.
# Teams only — never individual handles.

# Services
/services/lesson-player/**     @studyapp/lessons
/services/lesson-author/**     @studyapp/lessons
/services/mastery/**           @studyapp/lessons
/services/identity/**          @studyapp/identity
/services/notifier/**          @studyapp/notifier
/services/payments/**          @studyapp/billing      @studyapp/security

# Frontend
/services/frontend/portal/**   @studyapp/docs-platform
/services/frontend/**          @studyapp/web-platform

# Infra + ops
/infra/**                      @studyapp/sre
/.github/**                    @studyapp/sre
/.github/CODEOWNERS            @studyapp/staff
/runbooks/**                   @studyapp/sre

# Schemas + contracts (sensitive, need extra eyes)
/services/**/openapi.yaml      @studyapp/api-guild   @studyapp/staff
/services/**/schema.graphql    @studyapp/api-guild   @studyapp/staff
/services/**/*.proto           @studyapp/api-guild   @studyapp/staff

# Catch-all
*                              @studyapp/staff

# .github/workflows/orphan-files.yml (excerpt)
name: orphan-files
on: { schedule: [{ cron: "0 9 * * 1" }] }   # Monday 09:00
jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: enumerate orphans
        run: |
          python tools/codeowners_check.py --fail-on-orphans
      - if: failure()
        uses: slack-actions/notify@v1
        with: { channel: "#staff-eng", text: "Orphan files in main" }

Five team patterns, two cross-team patterns (payments has security in the loop, schemas have the API guild), catch-all to staff. The weekly orphan check keeps the map honest as the tree evolves.

Quality bar

  • CODEOWNERS at repo root. Most-specific patterns first; catch-all at the bottom; teams only — no individuals.
  • Branch protection requires owner approval. Not advisory; enforced by the forge.
  • Orphan-files CI check runs weekly. Zero orphans on main; failures alert the staff engineer.
  • CONTRIBUTING.md with triage SLA. 48 h ack target; explicit escalation path when missed.
  • Service catalogue agrees with CODEOWNERS. Quarterly cross-check; mismatches treated as bugs.
  • Quarterly ownership review. Calendared with the architecture retro; stale teams removed; new directories assigned.
  • Triage SLA tracked. Time-to-first-review per team visible on a scorecard; misses get retros not witch-hunts.
