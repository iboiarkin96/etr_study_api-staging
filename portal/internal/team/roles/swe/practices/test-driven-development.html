Test-driven development — red, green, refactor

Write the failing test first; let the test drive the smallest production change that makes it pass; refactor under the safety of the green bar.

TL;DR

TDD, formalised by Kent Beck in Test-Driven Development: By Example (2002), is a three-step micro-cycle: red (failing test) → green (smallest code that passes) → refactor (with green safety net). Treats the test as the executable specification and production code as a consequence. Output isn't "more tests" — it's code with a small, composable public surface that survives change because every line was demanded by a test.

What it is

TDD is a three-step micro-cycle, repeated in minutes:

  • Red — write a failing test that describes the smallest meaningful behaviour the system doesn't yet have. The test must fail for the right reason — running the test before writing it is the diagnostic.
  • Green — write the smallest production code that makes the test pass. Beck explicitly allows "fake it 'til you make it" (return a constant, generalise later) — the green bar is the goal, not elegance.
  • Refactor — clean up under the safety of green. Named Fowler moves (Extract Function, Inline Variable, Move Field — see Practice 05). Tests stay green throughout.

The cycle takes minutes, not hours. Done well, the team has dozens of red-green-refactor loops per day. The output is not "more tests" — it's a public surface shaped by the way the tests had to call it, and a refactoring story that always has a green bar to fall back on.

Why we need it

Tests-after development produces tests for code that already exists in the shape the engineer chose. The tests confirm what the code does, not what it should do. They're often coupled to internal structure (because the structure was decided first), so refactoring breaks them and the team learns to refactor less.

Tests-first inverts the dependency: the test states the goal, the production code emerges from the test's call site. The public API ends up small and composable because the test forced it to be callable. The refactor step is safe because the green bar exists before refactoring starts. Teams that adopt TDD report higher refactoring rates, fewer regression bugs, and code that's smaller than it would have been written speculatively.

How others do it

Kent Beck — TDD by Example (2002)

Beck's Test-Driven Development: By Example is the canonical text. Two worked examples (Money + xUnit) walk through the rhythm. Key contribution: "fake it 'til you make it" — green by any means; generalise later. Beck's recent Tidy First? newsletter and TCR (test && commit || revert) are modern refinements.

What we take: Read the book. The rhythm — minutes per cycle, dozens per day — is not negotiable; it is the practice. Anything longer isn't TDD, it's "tests after with a delay".

Source: Beck — TDD by Example

Robert C. Martin — Three Laws of TDD

Uncle Bob's "Three Laws of TDD": (1) you may not write production code unless it is to make a failing test pass; (2) you may not write more of a test than is sufficient to fail; (3) you may not write more production code than is sufficient to pass the test. Aggressive framing; clarifies the discipline.

What we take: The Three Laws are the strict default. Loosen them deliberately for spike work; never accidentally.

Source: cleancoder — Cycles of TDD

London-school TDD — mockist style

Steve Freeman + Nat Pryce's Growing Object-Oriented Software, Guided by Tests (2009) introduced the "London school" — outside-in TDD with extensive mocking of collaborators. Distinct from Beck's "Detroit school" (classicist, real objects, fewer mocks). Modern teams typically mix both.

What we take: Know both schools. Classicist for value objects + pure functions; mockist for adapters at system boundaries. Don't dogmatise either.

Source: Freeman & Pryce — GOOS book

testcontainers + property-based testing

Modern TDD pairs with testcontainers (real DBs in tests) and property-based testing (Hypothesis, fast-check, jqwik). Generative tests find edge cases the engineer wouldn't write; testcontainers eliminate the mock-vs-real-DB drift.

What we take: Layer property-based + testcontainers on top of unit TDD. Same red-green-refactor cycle; richer assertions; fewer mocks at the boundaries.

Source: testcontainers.com

Anatomy of a TDD session

Sub-minute red

Test written + run in < 60 s. Fails for the right reason (not a syntax error).

Sub-5-minute green

Smallest code that passes. "Fake it" allowed; generalise later. Beck-style triangulation.

Sub-3-minute refactor

Named Fowler moves under the green bar. Tests stay green every step.

Fast test suite

Unit tests < 10 s; full suite < 5 min. Slower than that and engineers stop running them.

testcontainers at boundaries

Real DB / Kafka / Redis via Docker. Eliminates mock-vs-real drift; still fast enough at the per-PR level.

Property-based for edges

Hypothesis / fast-check / jqwik generate edge cases. Catches inputs the engineer wouldn't think of.

How to run it

  1. Adopt the rhythm. Sub-minute red; sub-5-min green; sub-3-min refactor. Practice the cycle until it's muscle.
  2. Fast test runner. Watch-mode runner (vitest watch, pytest-watch, gradle continuous). Sub-10-s feedback.
  3. Three Laws default. Strict TDD by default; loosen deliberately for spike work.
  4. Mix schools. Classicist for value objects; mockist for adapters at boundaries.
  5. Pair TDD with refactoring catalogue. Practice 05; named Fowler moves under green.
  6. Add property-based + testcontainers. Hypothesis / fast-check + testcontainers at boundaries.
  7. Code-coverage as a smell, not a target. Sub-80% suspicious; mandated 95% gameable.
  8. Pair / mob TDD sessions. Cycle is teachable; the rhythm spreads by pairing.

Deep dive — what makes TDD actually work

The rhythm is the practice. A 30-minute "TDD cycle" isn't TDD; it's "tests after with a delay". Sub-minute red, sub-5-min green, sub-3-min refactor — measured. Engineers who can't keep that rhythm are usually missing a faster test runner or a clearer next test.

Fail for the right reason. A test that fails because the file doesn't compile teaches nothing. Run the test before writing it; confirm the failure message describes the missing behaviour. Then write production code.

"Fake it" is doctrine, not a shortcut. Beck's "fake it 'til you make it" — return a constant, generalise later — is doctrine. The constant gets the test green; the next test forces generalisation. This is how the public API gets shaped by what tests need, not by what the engineer guessed they'd need.

Refactor under green, never under red. The green bar is the safety net. Refactoring while red is gambling; refactoring while green is mechanical. If a refactor breaks tests, revert (Beck's TCR) — keep the green bar sacred.

Mix London + Detroit. Classicist (real objects, fewer mocks) for value objects + pure functions; mockist (mocks at boundaries) for adapters. Pure mockist tests get coupled to implementation; pure classicist gets slow at boundaries. The mix is the modern default.

Coverage is a smell, not a target. Sub-80% on the new code is suspicious — the engineer wrote production code without a test for it. Mandated 95% is gameable — engineers write tests to hit the number, not to pin behaviour. Use coverage as a smell that surfaces gaps, not a quota.

Spike, then TDD, never spike-only. Beck explicitly allows non-TDD spike work for genuinely unknown territory. The spike output is throwaway; production work that ships goes through TDD afterwards. Don't blur the two.

Lifecycle of a TDD cycle

Pick next behaviour

Smallest meaningful next behaviour to add. List of next tests on a scratchpad.

Red

Write failing test < 60 s. Confirms missing behaviour with the right failure message.

Green

Smallest production code that passes. "Fake it" allowed; generalise via triangulation.

Refactor

Named Fowler moves under green. Tests stay green throughout.

Commit

Optional TCR-style commit; or just continue. Small steady commits beat one big PR.

Loop

Next test from the scratchpad. Dozens of cycles per day.

Worked example — Score value object

Three red-green-refactor cycles to build a Score value object. Each cycle in minutes; production code emerges from what the tests called for.

# cycle 1 — red
def test_score_in_range_is_valid():
    Score(0.75, ceiling=1.0)   # no exception
def test_score_above_ceiling_raises():
    with pytest.raises(ValueError):
        Score(1.5, ceiling=1.0)

# cycle 1 — green (smallest code that passes)
@dataclass(frozen=True)
class Score:
    raw: float
    ceiling: float = 1.0
    def __post_init__(self):
        if not 0 <= self.raw <= self.ceiling:
            raise ValueError("out of bounds")

# cycle 1 — refactor (nothing yet)

# cycle 2 — red
def test_score_ratio_computes_normalised():
    assert Score(0.75, ceiling=1.0).ratio == 0.75
    assert Score(8.0,  ceiling=10.0).ratio == 0.8

# cycle 2 — green
@property
def ratio(self) -> float:
    return self.raw / self.ceiling

# cycle 2 — refactor (extract validator helper? no — overkill yet)

# cycle 3 — red (triangulate ceiling-must-be-positive)
def test_zero_ceiling_raises():
    with pytest.raises(ValueError):
        Score(0.0, ceiling=0.0)

# cycle 3 — green (generalise validator)
def __post_init__(self):
    if self.ceiling <= 0 or not 0 <= self.raw <= self.ceiling:
        raise ValueError("out of bounds")

# Three cycles · ~12 minutes · pinned behaviour + clean code

Three minutes per cycle; production code emerged in step with the tests. The Score type has a small public surface because the tests forced it to. Property-based tests can layer on top to find edge inputs the engineer didn't think of.
