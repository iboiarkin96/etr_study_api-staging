Small PRs — < 400 LOC, < 24 h to review

Every change small enough for a reviewer to hold in working memory; review turnaround under one working day.

TL;DR

Google's internal code-review research (google.github.io/eng-practices) found defect-detection drops sharply past ~200 LOC; the industry rule-of-thumb is < 400 LOC. Reviews finish in under 24 h so the author is never blocked over a working day. Stacked PRs (Graphite, git-spice, Phabricator) let one logical feature flow through review as a sequence of small reviewable units instead of one indigestible dump.

What it is

Small PRs is a working agreement that every change reaches review under a size + time budget:

  • Size: < 400 LOC changed (~200 is the sweet spot; 400 is the upper bound). Excluding generated code and large data fixtures.
  • Review turnaround: < 24 h. Reviewer ack within 1 h during work hours; full review within one working day.
  • Stacked PRs for big features. Graphite, git-spice, Sapling, Phabricator-style diffs — one logical feature flows as a chain of small reviewable units.
  • Scope-limited. One PR = one purpose. Refactor + feature = two PRs (refactor first, see Practice 05).

The discipline exists because reviewer attention is the bottleneck. A 200-LOC PR gets a careful read; a 2 000-LOC PR gets skimmed. Defect detection drops accordingly; review velocity drops because everything queues behind the big one.

Why we need it

Big PRs are the silent killer of code-review quality. A reviewer with 30 min and a 2 000-line diff approves on faith; defects ship; rework is expensive. A reviewer with 30 min and a 200-line diff reads carefully; defects caught; rework happens before merge when it's cheap.

Small PRs also unblock teams. A 300-LOC PR reviewed in 4 hours means the author moves on the same day; a 2 000-LOC PR reviewed over three days means the author is blocked or builds a parallel branch that diverges. The 24-h review target keeps flow high; the size cap keeps quality high.

How others do it

Google — < 200 LOC sweet spot

Google's eng-practices guide (google.github.io/eng-practices) documents the size guidance from internal research: defect-detection drops sharply past ~200 LOC; PRs of 400+ LOC frequently get cursory review. Same guide mandates a 1-hour first-response SLA during work hours.

What we take: Target ~200 LOC; cap at 400. First-response within 1 hour; full review within 24 h.

Source: google.github.io/eng-practices

Graphite / git-spice — stacked PR tooling

Graphite (and OSS git-spice) automate stacked-PR workflows: PR-on-top-of-PR, auto-restack on merge, one-command sync. Used by Vercel, Figma, Notion, Anthropic. Makes the small-PR pattern ergonomic where bare-git makes it tedious.

What we take: Adopt Graphite or git-spice. Bare-git stacked PRs are technically possible and operationally painful; tooling pays for itself in a month.

Source: graphite.dev

Phabricator / Sapling — diff-based review

Phabricator (Facebook, now OSS Phorge) and Sapling (Meta) treat each commit as an independently reviewable unit ("diff"). Stacks are first-class; restacking is automatic. Meta engineers ship hundreds of diffs a week per team.

What we take: Diff-based review fits small PRs naturally. If you can adopt Sapling / Phorge, the workflow is friction-free.

Source: sapling-scm.com

Atlassian / DORA — review-time as a leading indicator

Atlassian's engineering metrics + the DORA research treat code-review turnaround as a leading indicator of delivery health: teams with < 24 h review times consistently outperform on lead time and change-failure rate.

What we take: Track review-turnaround as a team metric. Trend over absolutes; if median climbs above 24 h, fix it before it becomes the default.

Source: dora.dev

Anatomy of a small-PR workflow

Size cap as a CI check

CI flags PRs > 400 LOC (excluding generated). Warning, not block; author justifies in the description.

One purpose per PR

Refactor + feature = two PRs. Renames + behaviour = two PRs. Reviewer doesn't disentangle.

Stacked PR tooling

Graphite / git-spice / Sapling. PR-on-top-of-PR; auto-restack; one-command sync.

First-response SLA

Reviewer acks within 1 h during work hours. Full review < 24 h.

Review-time scorecard

Median time-to-first-review and time-to-merge tracked per team. Trend visible; misses get retro items.

Async-first

Review in PR comments, not in a meeting. Sync only when threads stall after 3 rounds.

How to run it

  1. Adopt stacked-PR tooling. Graphite / git-spice / Sapling. One-week onboarding.
  2. Add size warning to CI. PRs > 400 LOC flagged with link to this page.
  3. Set review-response SLA. 1 h ack, 24 h full review. Calendared review blocks.
  4. Track turnaround. Time-to-first-review + time-to-merge per team, weekly trend.
  5. "Tidy first" split. Refactor lands ahead of feature in a separate PR (Practice 05).
  6. Async by default. Sync only when async stalls after 3 rounds.
  7. Quarterly retro on review hygiene. What's the biggest size offender? Why? Fix the process.

Deep dive — what makes small PRs land

Tooling is the unlock. Stacked PRs in bare git are tedious; in Graphite / git-spice / Sapling they're one command. Without tooling, engineers default to big PRs because the alternative is painful. Budget the tool license; pay back in a month.

Size cap as a warning, not a block. Hard-blocking PRs > 400 LOC penalises edge cases where a big PR is right (generated code, mass renames). Warning with a justification field works better: engineers justify, reviewers calibrate.

One purpose per PR is the dual rule. 200 LOC of "refactor + feature" is harder to review than 400 LOC of "feature". Reviewer cannot tell which line is the new behaviour. Split: refactor first, feature second; each clean; each easy.

First-response SLA is what unblocks flow. A 200-LOC PR sitting 2 days for a first look is worse than a 600-LOC PR reviewed in 4 hours. The author moved on either way; the difference is whether the next 200 LOC is queued or merging. Calendar 30-min review blocks daily.

Async-first beats meetings. Sync review meetings exclude time zones, scale poorly, and lose the written trail. Async PR comments cover 90% of reviews; sync (a 15-min hop on a call) is for stuck threads.

Review-time scorecard. Median time-to-first-review by team, visible. Trend matters; if median climbs over 24 h, retro before it becomes the new default. Pair with on-call rotation so review responsibility is named.

"Tidy first" splits the cognitive load. Kent Beck's "make the change easy, then make the easy change" — refactor PR lands first (zero behaviour change), feature PR lands second (small, on top of clean code). Two small PRs beat one tangled big one.

Lifecycle of a small PR

Drafted

Author writes ≤ 400 LOC, one purpose. Description explains the why; size justified if > 400.

Opened

Stacked on parent if part of a feature chain. CODEOWNERS routes reviewer.

Acknowledged

Reviewer responds within 1 h work-hours. First-pass review < 24 h.

Iterated

Comments addressed; restack handled by tool; second review fast because diff is small.

Merged

Squash with Conventional Commits message; descendants restacked automatically.

Tracked

Time-to-first-review + time-to-merge logged; team scorecard updates.

Worked example — stacked PR chain

Graphite stack for "lesson-player partial-credit scoring": four small PRs, each reviewable in 10 minutes, each landed in under a day.

# gt log (Graphite CLI)
main
└── refactor(player): extract AttemptScorer       PR #4521 · 180 LOC · merged
    └── refactor(player): move difficultyWeight    PR #4522 · 95 LOC  · merged
        └── feat(player): partial-credit scoring   PR #4523 · 240 LOC · in review
            └── docs(player): update SLO + budget  PR #4524 · 60 LOC  · draft

# Each PR independently reviewable; restack auto on merge of parent.
# Total: 575 LOC across 4 PRs vs one 575-LOC PR.
# Review carried 4 × 10 min = 40 min total vs 90+ min skim of a big PR.

Four PRs, total 575 LOC. Each < 250 LOC. Reviewer reads carefully; Graphite handles the restack on every merge. Author isn't blocked while reviewer works; reviewer isn't blocked by a wall of text.

Quality bar

  • Median PR size ≤ 300 LOC. > 400 warns in CI.
  • One purpose per PR. Refactor + feature splits into two.
  • Stacked-PR tool adopted. Graphite / git-spice / Sapling.
  • Review SLA: 1 h ack, 24 h full. Tracked per team.
  • Review-time scorecard visible. Time-to-first-review + time-to-merge median.
  • Async-first. Sync only on stalled threads.
  • Quarterly review-hygiene retro. Biggest size offenders + slowest reviews discussed.
