Docs-as-code — docs live next to the code they describe

Documentation in the repo, reviewed in the same PR as the code, built by CI, deployed on every merge.

TL;DR

Docs-as-code — championed by Anne Gentle and the Write the Docs community — treats documentation with the same workflow as source code: plain-text files (Markdown / reStructuredText / AsciiDoc) in the repo, PR review, CI gates, versioning by git tag. The full GitLab handbook, Stripe's API docs, and Twilio's developer site run this way. When doc and code change in one PR, drift is structurally impossible.

What it is

Docs-as-code is a working agreement that documentation flows through the same machinery as source code:

  • Source in plain text in the repo — Markdown (most common), MDX (Markdown with components), AsciiDoc, or reStructuredText. Versioned with the code; diffable; greppable.
  • Reviewed in PRs alongside the code change. CODEOWNERS routes docs to the right reviewer (often a tech writer or the SME).
  • Built by CI with style and link checks. Vale for prose style, lychee/htmltest for links, build for typos, axe-core for a11y.
  • Deployed on merge via Docusaurus, MkDocs, Hugo, Astro Starlight, or similar. PR previews via Netlify / Cloudflare Pages.

The core invariant: the doc lives next to the code it describes, and the doc PR is the same PR as the code PR. When the code merges, the doc merges; when one is forgotten, CI catches it; drift becomes structurally impossible.

Why we need it

Docs in Confluence / Notion / Google Docs drift the moment the code changes. The engineer ships the code, forgets the doc, and three months later the doc is wrong but still authoritative for new hires. The classic failure: a runbook describes the now-renamed binary; an on-call engineer follows the runbook at 3 am and wastes 20 minutes finding out the binary changed.

Docs-as-code closes the gap by making the doc part of the same PR. CODEOWNERS auto-routes; CI gates fail on broken links and bad style; the merged doc deploys with the code. Reviewers see the doc change next to the code change and reject "code-only" PRs that should have updated the doc. The doc isn't a side project; it's a build artefact.

How others do it

GitLab — public handbook in Git

GitLab's entire 2 000+ page company handbook (about.gitlab.com/handbook) is Markdown in a Git repo. Every change is a PR. Public; editable by anyone (with PR review). GitLab pioneered the "handbook-first" model where the canonical answer to any internal question is "what does the handbook say?".

What we take: If a 2 000-page company handbook can live in Git, your service docs can. Start with READMEs + design docs + ADRs in-repo; the rest follows.

Source: about.gitlab.com/handbook

Stripe — API docs from OpenAPI

Stripe generates stripe.com/docs from the same OpenAPI source that drives the SDKs. Reference docs cannot drift from the API because both are derived from one file. Long-form docs (guides, tutorials) are MDX in-repo with the same PR workflow.

What we take: Generate reference docs from the schema (OpenAPI / GraphQL / Protobuf). Long-form prose lives as MDX in-repo. Both ship from the same pipeline.

Source: stripe.com/docs/api

Write the Docs community

Write the Docs (writethedocs.org), founded ~2013, is the de-facto community for documentation engineering. Their "docs-as-code" guide, Vale style linter, and annual conferences set the modern norms. The Diátaxis framework (Daniele Procida) — tutorials / how-to / reference / explanation — gives docs a structure that survives growth.

What we take: Adopt Diátaxis (tutorials / how-to / reference / explanation) as the navigation. Vale as the style linter. Both are free and battle-tested.

Source: writethedocs.org

Twilio / Vercel / Astro — Docusaurus & Starlight

Twilio runs its developer docs on Docusaurus (Meta open-source). Vercel runs on Nextra. Astro built Starlight as a polished docs-first framework. All three render Markdown / MDX + components, with built-in versioning, search, i18n, dark mode, and PR previews. Plug-and-play; teams ship docs sites in days, not months.

What we take: Don't build your own docs site. Pick Docusaurus / MkDocs Material / Astro Starlight; configure; ship. The five days saved go into actual docs.

Source: docusaurus.io

Anatomy of a docs-as-code pipeline

Markdown / MDX source in the repo

/docs at repo root or co-located with services. Frontmatter for metadata (title, slug, owner, last-reviewed).

Diátaxis structure

Four sections: tutorials (learning), how-to (task-oriented), reference (information), explanation (understanding). Navigation reflects the model.

CI gates: style, links, build, a11y

Vale for prose style; lychee/htmltest for links; site build for typos / broken imports; axe-core for a11y. Required checks; PR blocked on failure.

Generated reference

API reference generated from OpenAPI / GraphQL / Protobuf. Code samples generated from real, tested code (not hand-written). Single source of truth.

PR previews

Each PR builds a preview URL (Netlify / Cloudflare Pages / Vercel). Reviewers click and read instead of imagining Markdown. Quality goes up; review time drops.

CODEOWNERS for docs

/docs/api/** @api-guild; /docs/runbooks/** @sre. Auto-routes review; tech writers / SMEs see the right PRs without manual cc.

How to run it

  1. Move docs into the repo. Markdown files under /docs or co-located. Confluence/Notion content imported (or linked + deprecated).
  2. Pick a site generator. Docusaurus / MkDocs Material / Astro Starlight. Plug-and-play; don't roll your own.
  3. Adopt Diátaxis navigation. Tutorials / how-to / reference / explanation. Maps onto the doc tree.
  4. Wire CI gates. Vale (style), lychee (links), build, axe-core (a11y). All required.
  5. Set up PR previews. Netlify / Cloudflare Pages / Vercel. Preview URL on every PR.
  6. Generate reference docs. OpenAPI → reference pages. Code samples from tested code.
  7. CODEOWNERS for docs. Tech writer / SME auto-routed on doc PRs.
  8. Quarterly doc retro. Top-3 visited pages reviewed; stale pages flagged; navigation audited.

Deep dive — what keeps docs alive

Same PR as the code is the load-bearing rule. A docs PR after the fact is a docs PR that doesn't happen. The CODEOWNERS pattern that routes the docs reviewer in the same PR as the code reviewer is what makes "code change without doc change" visible at review time. Reject "code-only" PRs in areas with user-facing impact.

Generated reference beats hand-written reference. Hand-written API tables drift; generated tables can't. Drive the reference from OpenAPI / GraphQL / Protobuf; the long-form prose stays hand-written but is grounded in generated source. Stripe's docs site is the textbook example.

Diátaxis is the navigation that scales. Without a structure, every page is its own category; readers can't find anything. Tutorials / how-to / reference / explanation map onto how users actually use docs (learn / do / look up / understand). Adopt Diátaxis early; retrofitting it later is painful.

CI gates matter more than the site framework. Vale catches "don't use 'simply' or 'just'"; lychee catches broken links; the build catches typos. The framework choice is largely a styling preference; the CI gates are what protect quality over time.

PR previews change reviewer behaviour. Reading Markdown in a diff is hard; clicking a preview URL is easy. PR previews convert reviews from "I scanned the diff" to "I read the rendered page". Quality improves measurably from this one change.

Tested code samples or no code samples. A code sample that doesn't actually run is worse than no sample. Generate samples from real, CI-tested code; the sample stays correct because the test does.

Quarterly doc retros. Docs rot silently. A 30-minute quarterly retro reviews top-3 visited pages, stale pages, navigation gaps. Without it the corpus drifts; with it the team treats docs as a living artefact.

Lifecycle of a docs change

Drafted

Engineer writes Markdown next to the code change. Frontmatter set (title, slug, owner, last-reviewed).

PR'd alongside code

Same PR as the code change. CODEOWNERS routes docs reviewer + code reviewer. Both required.

CI-gated

Vale / lychee / build / axe-core run. Required checks; PR blocked on failure.

Previewed

Netlify / Cloudflare PR preview URL. Reviewer clicks; reads rendered page; comments.

Merged + deployed

Merge triggers deploy. Production docs reflect the change within minutes. Old version preserved via versioning.

Reviewed quarterly

Top-3 visited / stale pages reviewed. Owner runs the audit; stale pages get a refresh PR or a deprecation note.

Worked example — Study App docs pipeline

Repo layout, CI workflow, and PR-preview wiring for the Study App documentation. Diátaxis sections at the top; CODEOWNERS routes reviewers; CI gates are required checks.

# Repo layout
/docs
  /tutorials/         # learning-oriented
  /how-to/            # task-oriented
  /reference/         # information-oriented (generated from OpenAPI)
  /explanation/       # understanding-oriented
  /handbook/          # team handbook
docusaurus.config.ts  # site framework

# .github/CODEOWNERS (excerpt)
/docs/reference/**    @studyapp/api-guild
/docs/how-to/sre/**   @studyapp/sre
/docs/handbook/**     @studyapp/staff

# .vale.ini (excerpt — prose style)
StylesPath = .github/vale/styles
MinAlertLevel = warning
[*.{md,mdx}]
BasedOnStyles = Vale, Google
Google.We      = NO      # avoid "we"
Google.Will    = warning # discourage "will"
Vale.Spelling  = YES

# .github/workflows/docs.yml (excerpt)
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: errata-ai/vale-action@v2
        with: { fail_on_error: true }
      - uses: lycheeverse/lychee-action@v1
        with: { args: "--no-progress --exclude-mail ./docs" }
      - run: npm ci && npm run build      # docusaurus build
      - run: npx pa11y-ci --config .pa11yci

# netlify.toml (PR previews)
[build]
  command = "npm run build"
  publish = "build"
[context.deploy-preview]
  command = "npm run build -- --baseUrl /preview/"

Diátaxis tree; CODEOWNERS routes the right reviewer; Vale + lychee + build + pa11y as required CI; Netlify renders a preview URL for every PR. Tech writers and SMEs see the right PRs without manual cc.

Quality bar

  • Docs in the repo. Markdown / MDX; PR'd alongside code.
  • Diátaxis navigation. Tutorials / how-to / reference / explanation.
  • Generated reference. From OpenAPI / GraphQL / Protobuf; not hand-rolled.
  • CI gates required. Vale + lychee + build + a11y.
  • PR previews live. Netlify / Cloudflare Pages / Vercel; URL on every PR.
  • CODEOWNERS routes docs. Tech writers / SMEs auto-tagged.
  • Tested code samples. Generated from real tested code.
  • Quarterly doc retro. Top-3 visited / stale pages reviewed.
