API contract-first — schema before code

Write the schema first. Generate the server, the SDKs, the mocks, and the tests from it. Reject any PR that introduces an undocumented endpoint.

TL;DR

Contract-first treats the API schema — OpenAPI for HTTP, GraphQL SDL for GraphQL, Protobuf for gRPC — as the source of truth, written before any handler. CI generates server stubs, client SDKs, mocks, and contract tests from one file. Stripe generates SDKs in seven languages from one OpenAPI source; GitHub's GraphQL schema is the canonical contract for thousands of integrators; Buf has built an entire Protobuf-first toolchain on the premise. Code-first APIs drift; contract-first APIs stay in sync because the schema is the build input, not a side effect.

What it is

Contract-first API development means the API's interface — endpoints, fields, types, error codes — is written in a machine-readable schema language before any handler code, and that schema is the build input that produces everything else. Three dominant schema formats:

  • OpenAPI (formerly Swagger) — for HTTP/JSON APIs. YAML or JSON. Maintained by the OpenAPI Initiative.
  • GraphQL SDL — for GraphQL APIs. Describes types, queries, mutations, subscriptions.
  • Protobuf — for gRPC and other binary RPC protocols. .proto files. Buf provides the de-facto industry toolchain (lint, breaking-change detection, registry).

From the schema, CI generates:

  1. Server stub — routes, request/response models, validation. Engineers fill in business logic only.
  2. Client SDKs — one per supported language, all type-safe, all in sync.
  3. Mocks — runnable servers (Prism, WireMock) so consuming teams can integrate before the real backend exists.
  4. Contract tests — fuzz tests (Schemathesis) that hammer the running service against the schema and fail on drift.

Breaking-change detection runs in CI on every PR (oasdiff for OpenAPI, buf breaking for Protobuf, GraphQL Inspector for GraphQL). A PR that removes a required field or changes a type without bumping the version is blocked at review.

Why we need it

Code-first APIs drift. The handler exists, docs are written by hand in Confluence, the SDK is a different team's reimagination of the server's behaviour, and the mock is whatever the frontend team hand-rolled last quarter. Within six months all four describe slightly different APIs. The bug reports that follow — "the docs say this field is optional but the server rejects null" — are not bugs in the code; they are bugs in the absence of a single source of truth.

Contract-first inverts the dependency: the schema is the input, everything else is the output. A breaking change at the schema layer fails CI before code is even compiled. A new field appears in the SDK the same hour it appears on the server. A consuming team building against the mock cannot diverge from what the real service will eventually return. The compounding return is that integration cost — the largest hidden cost of multi-team API work — collapses.

How others do it

Stripe — one OpenAPI, seven SDKs

Stripe maintains a single OpenAPI specification as the canonical description of its API and generates client libraries in seven languages (Ruby, Python, PHP, Java, Node, Go, .NET) from it. Versioning is date-based: every breaking change introduces a new dated version, and the schema preserves all historical versions so old clients keep working forever. Schema, docs at stripe.com/docs/api, and SDKs all derive from the same source — drift is structurally impossible.

What we take: The schema must be the only source. If docs are written separately or the SDK is hand-rolled, you have already lost. Generate everything from one file.

Source: Stripe — API versioning at Stripe

GitHub — GraphQL evolution model

GitHub's GraphQL API has been the recommended integration surface since 2017. The SDL schema is the contract — public, downloadable, versioned via schema-evolution rules rather than URL versions. Deprecated fields stay in the schema with @deprecated directives; breaking changes go through a public preview period. The Octokit SDKs are generated from the introspection result, not hand-written.

What we take: GraphQL's evolution model — additive changes safe, deprecations stay visible — gives consumers a clear migration path without version sprawl. Use @deprecated; never silent removals.

Source: GitHub — GraphQL API docs

Buf — Protobuf platform

Buf, founded by ex-Uber engineers who built Uber's internal Protobuf platform, ships the de-facto toolchain for contract-first gRPC: buf lint enforces naming and style; buf breaking blocks breaking changes against a registered baseline; the Buf Schema Registry hosts schemas the way npm hosts packages. Companies that ship gRPC at scale (Block/Square, Snowflake, Mux) have moved from hand-curated workflows to Buf.

What we take: Treat breaking-change detection as a non-negotiable CI gate. Without it, "contract-first" decays into "contract-eventually". Buf for Protobuf, oasdiff for OpenAPI, GraphQL Inspector for GraphQL.

Source: buf.build — Protobuf platform

Zalando — Zally + API guidelines

Zalando enforces contract-first across 200+ microservices with the open-source Zally linter and a public set of API guidelines. Every OpenAPI spec must pass 60+ Zally rules (naming, pagination, error contract, cursor format) before deployment. The rules encode Zalando's API principles as fitness functions; the schema is the only thing CI checks at the API layer. Zally is open-sourced at github.com/zalando/zally.

What we take: Pair contract-first with an API-style linter. Lint catches the design issues breaking-change checks miss (consistent naming, pagination shape, error contract). 10 rules is enough at the start; expand as patterns surface.

Source: github.com/zalando/zally

Anatomy of a contract-first pipeline

Schema file in the repo root

One openapi.yaml / schema.graphql / service.proto per service, version-controlled. Linted by spectral / graphql-eslint / buf lint on every PR. No endpoint exists until it is in this file.

Server-stub generator

CI runs openapi-generator / graphql-codegen / protoc on every push and produces typed handler signatures, request/response models, validation middleware. Engineers fill in business logic only — they cannot deviate from the schema's contract.

Client-SDK generator

One generator pass per target language (TypeScript, Go, Python, Java). Outputs published to the internal package registry on every merge. Consumers depend on the generated package — never hand-roll an HTTP client.

Mock server

Prism (OpenAPI), Apollo Server mock mode, or a mock generated from .proto — runs against the same schema and returns example data for every operation. Consumer teams develop against the mock before the real handler exists, eliminating cross-team blocking.

Breaking-change gate in CI

On every PR, run oasdiff breaking / buf breaking / GraphQL Inspector against the main-branch schema. A removed field, tightened type, or changed enum fails the build. Bypassed only with explicit version bump + reviewer sign-off.

Contract-test suite

Schemathesis (OpenAPI) or buf curl (Protobuf) fuzz-tests the running service against the schema in CI: generates random valid requests, asserts the response matches the schema. Catches the case where the handler accidentally diverged from the contract — even though the schema didn't change.

How to run it

  1. Pick the schema language. HTTP/JSON → OpenAPI 3.1. GraphQL → SDL. gRPC → Protobuf. One per service; never mix.
  2. Write the schema first. Endpoints, fields, types, error shapes. Reviewed in a PR before any handler exists. Lint with Spectral / GraphQL-ESLint / buf lint.
  3. Generate the server stub. openapi-generator / graphql-codegen / protoc. Engineers implement business logic in the generated handler signatures only.
  4. Generate client SDKs per language. Published to the internal package registry on every merge. Consumers depend on generated packages.
  5. Stand up a mock server. Prism / Apollo mock / proto mock. Consumer teams develop against it from day one.
  6. Wire the breaking-change gate. oasdiff breaking / buf breaking / GraphQL Inspector on every PR. Required check.
  7. Add the contract-test suite. Schemathesis (or equivalent) in CI against staging. Catches handler-vs-schema drift the schema check can't see.
  8. Pair with a style linter. Zally / Spectral rulesets for naming, pagination, error contract. 10 rules at the start; expand from PR comments that recur.

Deep dive — what makes contract-first land

Schema is the input, not the output. The single biggest mistake teams make is "annotation-driven contract-first" — sprinkling @Operation/@ApiResponse annotations on handlers and letting the framework emit the schema. That isn't contract-first; the schema is downstream of the code, so the contract cannot fail CI before the code is written. Write the schema first by hand (or with a generator from a design doc) and treat the handler as the dependent artefact.

Breaking-change detection is the load-bearing gate. Linting catches style; type-checks catch syntax; only a structural diff against the main-branch schema catches "removed field" / "tightened type" / "dropped enum value". oasdiff, buf breaking, and GraphQL Inspector all encode the breaking-change rules from the respective specs. Run them on every PR as a required check, not an optional warning.

Generated SDKs over hand-rolled clients, always. "The generator output is ugly" is the most common excuse for hand-rolling SDKs. The cost of ugly is small; the cost of drift is enormous. Use the generator output as the SDK. If you want idiomatic ergonomics, wrap the generated client in a thin hand-written facade — but never duplicate the transport layer. Only the generator can stay in sync.

Mocks decouple teams. Prism (OpenAPI) and Apollo's mock mode let a consuming team build the full integration before the producer ships a single handler. The producer's velocity stops being a blocker; the consumer's velocity stops depending on the producer's prioritisation. This is where contract-first earns its keep at the org level — feature lead time across teams drops.

Contract tests catch drift the schema check can't. A schema check verifies the schema didn't change in a breaking way. It does not verify that the running handler still matches the schema. Schemathesis fuzz-tests with random valid requests and asserts the response matches; it catches the handler that accidentally returns null where the schema says required. Pair both checks; they cover different surfaces.

Versioning is a design choice, not an accident. Stripe's date-based versioning, GitHub's GraphQL @deprecated trail, Buf's registry-based breaking detection — each is a deliberate evolution model. Pick one explicitly at adoption time. Without a chosen model, every PR re-litigates "is this breaking?" and the answer drifts with the reviewer.

Pair with API guidelines. Contract-first prevents shape drift; style linters prevent design drift (naming, pagination, error contract). Zally and Spectral encode the team's API principles as fitness functions. Without guidelines, every team's schema looks different and consumer SDKs feel inconsistent — even though each schema is technically correct.

Lifecycle of an API change

Drafted

Schema change written in a PR before handler code. Reviewed for shape, naming, error contract. Style linter passes.

Compat-checked

Breaking-change tool runs against main. Breaking change either rejected or marked with an explicit version bump.

Generated

Server stub + SDKs + mock regenerated. Engineers implement handlers within the generated signatures.

Contract-tested

Schemathesis / buf curl fuzzes the running service against the schema in CI. Drift between handler and schema caught here.

Published

SDKs published to the internal registry. Docs regenerated from the schema. Consumers depend on the new version.

Deprecated / retired

Field or endpoint marked @deprecated with a sunset date in the schema. Removed only after consumers migrate; never silent removal.

Common pitfalls

Pitfall Why it happens What to do instead
Schema generated after the code (annotation-driven) Team adopts springdoc-openapi / drf-spectacular, sprinkles annotations on handlers, calls it "contract-first". In practice the schema is whatever the handler happens to produce, so the contract cannot fail CI before code is written. Schema is the input. Generate handlers from it, not the other way around. If the framework only supports annotation-driven generation, write the schema by hand and pin the handler to it with a CI diff check that fails on drift.
Breaking change shipped — no compat check in CI The schema lives in the repo but no automated check compares the new version to main. An engineer removes a "deprecated" field one customer still uses; integration breaks silently in production. oasdiff breaking / buf breaking on every PR as a required check. Treat "breaking change" as a structural concept (removed field, tightened type, dropped enum), not a judgement call.
Client SDKs hand-written, drift from schema "The generator output is ugly" or "we want a more idiomatic SDK", so the SDK is hand-rolled. Six months later the SDK supports endpoints that no longer exist and is missing fields that have been there for a quarter. Use the generator output as the SDK, even if verbose. For idiomatic ergonomics, wrap with a thin facade — never duplicate the transport. The generator is the only thing that can stay in sync.
No mock — consumers wait for the real backend Producer team is busy; consumer team blocks. Frontend ships hand-mocked data that diverges from the eventual real shape. Integration day surfaces all the drift at once. Stand up a Prism / Apollo mock from the schema on day one. Consumer teams integrate against the mock; switch to real backend with confidence because the shapes match by construction.
Schema check passes but handler returns the wrong shape Schema didn't change in a breaking way, but the handler does. CI passes; production breaks. Schemathesis / contract-test suite in CI alongside the schema check. Fuzz-tests the running service; asserts every response matches the schema. Catches handler drift the schema check can't.
Silent field removals Field marked deprecated; six months later quietly removed. Some consumer was still using it; their integration breaks. Deprecation is a documented sunset, not a silent removal. @deprecated with a date. Remove only after consumer telemetry shows zero usage or sunset date passes — whichever first.

Worked example — lesson-player OpenAPI

A trimmed slice of openapi.yaml for the lesson-player service. The full file lives at services/lesson-player/openapi.yaml; CI runs Spectral lint, oasdiff breaking, server-stub + SDK generation, mock startup, and Schemathesis fuzz.

# services/lesson-player/openapi.yaml
openapi: 3.1.0
info:
  title:   Study App · lesson-player
  version: 2026-05-20  # date-based; bump on breaking
  contact: { name: "Player team", email: "player@studyapp" }

paths:
  /lessons/{lessonId}/state:
    get:
      operationId: getLessonState
      summary: Current state for a lesson + user
      parameters:
        - { in: path,  name: lessonId, required: true, schema: { type: string, format: uuid } }
      responses:
        "200": { $ref: "#/components/responses/LessonState" }
        "404": { $ref: "#/components/responses/NotFound" }
        "429": { $ref: "#/components/responses/RateLimited" }

  /lessons/{lessonId}/attempts:
    post:
      operationId: scoreAttempt
      summary: Score one attempt at a lesson question
      requestBody:
        required: true
        content: { application/json: { schema: { $ref: "#/components/schemas/Attempt" } } }
      responses:
        "200": { $ref: "#/components/responses/AttemptResult" }
        "404": { $ref: "#/components/responses/NotFound" }
        "422": { $ref: "#/components/responses/Validation" }

components:
  schemas:
    Attempt:
      type: object
      required: [questionId, answer]
      properties:
        questionId: { type: string, format: uuid }
        answer:     { type: string, maxLength: 4000 }
        # NOTE: thin shape on purpose; client fetches detail via questionId

# CI pipeline (.github/workflows/api.yaml — excerpt):
#   - spectral lint openapi.yaml --ruleset .spectral.yaml
#   - oasdiff breaking main/openapi.yaml openapi.yaml --fail-on-diff
#   - openapi-generator generate -i openapi.yaml -g go-server   -o ./gen/server
#   - openapi-generator generate -i openapi.yaml -g typescript  -o ./gen/sdk-ts
#   - openapi-generator generate -i openapi.yaml -g python      -o ./gen/sdk-py
#   - prism mock openapi.yaml &  # runs at :4010 for downstream integration
#   - schemathesis run http://localhost:8080/openapi.yaml --checks all

One file drives lint, breaking-change gate, server stub, two SDKs, a mock, and the fuzz suite. The schema is the only thing engineers write; everything else is regenerated on every push.

Quality bar

  • One schema file per service. Repo root, version-controlled, single source of truth.
  • Schema written before any handler. Reviewed in a PR. Handler exists only after the schema is merged.
  • Style linter in CI. Spectral / GraphQL-ESLint / buf lint on every PR.
  • Breaking-change gate in CI. oasdiff breaking / buf breaking / GraphQL Inspector — required check.
  • Generated server stub. Engineers implement business logic only inside generated signatures.
  • Generated SDKs published per merge. No hand-rolled HTTP clients. Generated package in the internal registry.
  • Mock available from day one. Prism / Apollo mock / proto mock running in the integration environment.
  • Contract-test suite in CI. Schemathesis / buf curl against staging. Drift between handler and schema caught.
  • Deprecation has a sunset date. @deprecated with a date; never silent removal.
