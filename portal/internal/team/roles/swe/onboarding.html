SWE onboarding — 60-day journey

Your first 60 days as a developer on Study App, mapped to concrete reads and concrete deliverables. Each step has a time horizon, a short description, the canonical pages to read, and one outcome — the thing you will be able to do when you have finished the step.

Work the steps in order. The first three are compulsory; the rest are strongly encouraged. By the end you will have shipped one feature end-to-end (contract, code, tests, telemetry) independently — that is the bar.

The journey · six milestones

  1. 01
    Day 1 · access

    Get into the project

    Clone the monorepo, open the devcontainer, install the toolchain, get the right accounts (GitLab, registry, staging, observability), and run the test suite green locally. Nothing exciting — but the next five steps depend on this one working.

    • Local development
    • Devcontainers
    • Make commands & workflows

    Outcome. You can run the test suite green locally, start the service in the devcontainer, and hit the staging endpoint.

  2. 02
    Days 2–3 · foundations

    Read the why before the what

    Understand how we organise code, which contracts are source of truth, and where business logic lives. Contract-first APIs are the dev rule; tactical DDD is the modelling style; twelve-factor is the deployment contract.

    • API contract-first
    • DDD (tactical)
    • Twelve-factor app
    • Business logic primer

    Outcome. You can name the service's contract files, find its core domain code, and explain how a request flows from edge to database.

  3. 03
    Week 1 · the way around

    Learn how code flows

    Pair with a senior dev for two sessions — one feature PR, one CI failure debug. Read the trunk-based workflow and the CI pipeline carefully. Open three recent merged PRs and trace each from spec to deploy.

    • Trunk-based development
    • CI green trunk
    • Small PRs
    • Conventional commits

    Outcome. Given any merged PR, you can name its reviewer rules (CODEOWNERS), the CI steps it cleared, and the deploy path it took.

  4. 04
    Week 2 · first ticket

    Run your first PR end-to-end

    Pick one low-risk ticket from the backlog (your buddy will point you at one). Walk it through the zero-to-endpoint tutorial: write the contract, the test, the code, ship the PR green. If TDD is new to you, write the test first — even if it feels slow.

    • Zero-to-endpoint tutorial
    • Test-driven development
    • Pre-commit hooks

    Outcome. You have merged your first PR with passing CI, conventional commit, and a test you wrote first.

  5. 05
    Weeks 3–4 · first deliverable

    Own one full feature

    Take a real upcoming feature and ship it end-to-end: update the contract, build the endpoint behind a feature flag, wire up structured logging and traces, write load + unit tests. Roll it out progressively to staging then prod.

    • Add a POST contract
    • Feature flags
    • Structured logging
    • Distributed tracing

    Outcome. You have shipped one feature behind a flag, with logs and traces, on a real shipping epic.

  6. 06
    Day 60 · sign-off

    Ship one feature independently

    You are now the named dev on a feature with no senior pairing by default. Take it from spec to prod: contract → code → tests → telemetry → progressive rollout → close ticket. The buddy stays available but does not write code. When the feature is live and the SLOs are green — that is the end of onboarding.

    • Progressive delivery
    • SLOs in code
    • API load testing

    Outcome. You have shipped one feature to production independently with green SLOs. You are now a fully ramped developer on Study App.
