Design docs — discuss before you build

A short written proposal that catches wrong paths before a week of coding instead of after.

TL;DR

A design doc is a 2–10 page written proposal — context, problem, two or three options with trade-offs, the chosen path, success metrics, risks, FAQ, open questions — written before implementation begins. The format was made canonical at Google, where any engineer can author one and any engineer can comment. The design doc proposes; the ADR records the result. When designs are discussed in writing first, the team catches the wrong path in a 90-minute review instead of after a week of coding.

What it is

A design doc is a written proposal for a non-trivial technical change. It is the document an engineer writes before implementation begins, not after. The canonical format covers:

  • Context — what currently exists, what changed, what forces are driving this work.
  • Problem statement — the specific problem to solve, with success criteria.
  • Options considered — two or three approaches with pros, cons, trade-offs.
  • Recommended path — the chosen approach and the reasoning.
  • Success metrics — how will we know the change worked?
  • Risks & rollout plan — what could go wrong; how the change is deployed.
  • FAQ — anticipated objections answered in the doc.
  • Open questions — what the author wants feedback on specifically.

Length: 2–10 pages. Less is more. A design doc that exceeds 10 pages typically has scope problems — it is solving multiple problems at once and should be split. Reviewers comment asynchronously over 3–7 days. After resolution, the resulting decision is captured as an ADR; the design doc is archived as the analysis that informed it.

Why we need it

The cheapest moment to discover a wrong design is when it is written down on paper. A 90-minute reading by three senior engineers can catch a flawed assumption that would otherwise take 3 weeks of implementation to surface. The act of writing also forces the author to confront vagueness: ideas that sound coherent in a verbal pitch reveal their gaps when written.

Design docs also create institutional memory. A reader who returns to the doc a year later can see what alternatives were considered, what the team chose, and why. Without a design doc, the same conversation is re-litigated every time a new engineer joins. Google's Design Docs at Google post explicitly identifies this artefact as the most consistently valuable technical-writing investment at scale.

How others do it

Google — design docs everywhere

Google made design docs a universal engineering practice — any engineer can write one, any engineer can comment. Docs follow a standard template (context, scope, requirements, design, alternatives considered) and are stored in a shared Google Docs folder. L6+ engineers routinely review docs across teams as a form of mentoring and cross-pollination. Malte Ubl's 2020 essay Design Docs at Google documented the practice: the doc is the design; the implementation follows from it.

What we take: Design docs are not just for senior engineers writing major proposals — they are for any engineer working on a non-trivial change. The act of writing scales the engineer's thinking before the implementation is committed.

Source: Industrial Empathy — Design Docs at Google

Amazon — 6-page memo culture

Amazon's "6-page memo" culture, established by Jeff Bezos, replaces PowerPoint with narrative documents for major decisions. The format requires complete sentences, full paragraphs, and explicit FAQ sections that anticipate objections. The first 20 minutes of any decision meeting is silent reading. The format is closely related to design docs and has shaped how many ex-Amazon engineers run their teams at startups and other tech companies.

What we take: The FAQ section at the end of the design doc is high-value. Anticipate the questions reviewers will ask and answer them in the doc itself. Reduces review iteration time.

Source: Bezos 2016 letter to shareholders

Stripe — memos + silent reading

Stripe's engineering culture is famously documentation-heavy. Stripe popularised "Memos" — modelled on Amazon's 6-page memo — for major decisions. Each memo follows a structured format and is read silently for the first 20 minutes of any major decision meeting. Stripe also publishes its API design principles as written guidance derived from internal design docs that became canonical.

What we take: The "silent reading at the start of the meeting" practice guarantees everyone has read the doc before discussion. Adopt it for any non-trivial design review.

Source: Stripe — engineering principles

Shopify — design docs in the repo

Shopify keeps design docs in the same repo as the code, alongside ADRs and architecture diagrams (PlantUML). The doc PR is the first PR of the work; implementation PRs cite the doc. Comments live in the PR itself, archived with the commit history. Shopify also runs weekly "design office hours" where engineers bring early-stage proposals before writing a full doc.

What we take: Keep design docs in-repo as Markdown. The PR review surface is excellent for async comments and the doc travels with the code. Run weekly office hours for early-stage proposals — formal docs only after the design has stabilised.

Source: Shopify Engineering

Anatomy of a design doc

Header block

Title, authors, named reviewers, status (draft / in review / accepted / superseded), created date, target decision date. The named reviewer list is the distribution gate — if a stakeholder is not named, they may not see it in time.

Context section

2–4 paragraphs describing what currently exists and what forces drive the proposed change. A reader new to the area should understand the situation without consulting external sources. Cites prior ADRs and design docs where relevant.

Options comparison

2–3 distinct options with pros, cons, effort estimates. One option is the recommended path; the others provide the comparison that makes the recommendation defensible. An option set of one is a proposal disguised as analysis.

Success metrics

Explicit, measurable outcomes. "p99 latency reduced from 800 ms to 200 ms within 30 days" is valid; "performance will improve" is not. Without metrics, the project ships and the team has no way to know if the design was correct.

FAQ & open questions

Two sections at the end. FAQ anticipates objections ("Why not Kafka? Because we already operate SQS at scale…"). Open Questions list what the author explicitly wants feedback on. Reduces review iterations.

Rollout plan

How the change is deployed: feature flag, canary, dual-write, big-bang. Rollback procedure if metrics regress. Without a rollout plan, the design is theoretical; with one, the implementation has a runbook.

How to run it

  1. Decide it needs a design doc. Triggers: cross-service change, new dependency, data-model change, NFR shift, expected ops cost above $1k/month, or anything you cannot finish in a week.
  2. Author the draft (1–2 days). Use the canonical template. Cap at 10 pages. Two real options minimum. Fill the FAQ from the questions you can already predict.
  3. Open a PR (or doc) and name reviewers. 3–5 named reviewers spanning the relevant domains (architect, SRE, security, data, frontend as needed). Tag-based, not "@everyone".
  4. Async review window (3–7 days). Reviewers comment in-thread. Author responds in writing. Open questions get resolved by edits to the doc, not by side conversations.
  5. Sync only when async stalls. After 3 unresolved comment threads or 7 days, schedule a 30-minute review meeting. Start with 10 minutes of silent re-reading.
  6. Capture the decision as an ADR. Once accepted, an ADR is authored citing the design doc. Doc is archived as "accepted". Both link to each other.
  7. Implementation cites the doc. The first implementation PR references the design doc and the ADR. Reviewers checking implementation can trace back to the intended design.

Deep dive — making the doc do its job

Write first, code second. The doc is the cheapest place to discover a wrong path. An hour of writing reveals gaps that a week of coding would only hint at. Teams that write the doc after implementation produce documentation, not design — the artefact is the same shape but does none of the work.

Two real options or none. A doc with one real option and a straw-man "do nothing" is not analysis — it is a proposal in disguise. If only one option exists, write a short ADR and skip the design doc. Reserve the design doc for situations where there is genuine choice to be made; that is where the format earns its cost.

Async-first, sync only on stall. Async commentary lets reviewers process at their own pace and forces the resolution to be written down. Sync meetings are for the rare moments when async has stalled after 3 unresolved threads or 7 days. Default to async; the exception is sync. The written trail is the long-term value.

FAQ is free review iterations. Every objection you can predict and answer in the FAQ is one review iteration you do not pay for. Borrow Amazon's discipline: write the FAQ before opening the doc for review. The first version of the FAQ should answer the five most predictable questions; revisions add the ones reviewers actually raised.

Link to the ADR — both ways. The design doc is the analysis; the ADR is the decision. Without the bidirectional link, future readers find the ADR and have no idea what alternatives were considered, or find the doc and have no idea what was decided. Both must reference each other.

Cap length at 10 pages. Length is a smell. A 30-page doc has scope problems — it is solving multiple problems at once and should be split. Linked appendices for detail are fine; the body itself should fit in one reading.

Lifecycle of a design doc

Draft

Author writes the first version. Not yet circulated. Used for office-hours discussion or early review with the tech lead.

In review

Distributed to named reviewers. Async comments collected for 3–7 days. Author responds in writing; FAQ updated as questions surface.

Sync (rare)

Only if async stalls. 30-min meeting with 10 min of silent re-reading at the start. Resolution written back into the doc.

Accepted

Reviewers signed off. Decision captured as an ADR. Doc archived as "accepted". Implementation can start.

Superseded

A later design replaces this one. Original doc kept for history; status set to "superseded by <new doc>". Both ADRs linked.

Withdrawn

Proposal abandoned before acceptance. Doc kept with status "withdrawn" plus a one-line reason. Preserves the analysis for future readers.

Common pitfalls

Pitfall Why it happens What to do instead
Doc written after implementation is mostly done Coding feels productive; writing feels like overhead. Doc is written retroactively to satisfy process. Reviewers cannot meaningfully change the design — the code already exists. Write the design doc as the first deliverable. The first implementation PR references the design-doc PR. No design doc, no implementation start.
Single-option doc (proposal disguised as analysis) Author already knows what they want. Options section contains one real option and one straw-man. Reviewers cannot push back — there is no real alternative. Require two genuine options. If only one exists, write a short ADR and be honest about the decision being made. The Options section requires real analysis or it should not exist.
Doc length grows to 30+ pages Author adds every detail anticipating questions. Doc is too long to read in one sitting. Reviewers skim and miss critical issues. Cap at 10 pages with linked appendices for detail. If the doc exceeds 10 pages, scope is probably too broad — split into two docs for separable decisions.
No named reviewers — broadcast to "@engineering" Author hopes someone will read it; nobody owns reviewing. Doc sits open for weeks with no decision. Name 3–5 reviewers explicitly, spanning the relevant domains. Named reviewers have an SLA; broadcast has none.
No FAQ — every objection becomes a comment thread Author skips the FAQ to ship faster. Reviewers raise the predictable objections one at a time. Review iterations double. Write the FAQ before opening for review. Answer the five most predictable questions up front. Update the FAQ with every novel question reviewers raise.
Doc accepted but never linked to an ADR Acceptance feels like the end of the work. Nobody writes the ADR. Six months later the team cannot find the decision. Acceptance is not the end — the ADR is. The acceptance criteria include "ADR authored and linked". Both reference each other.

Worked example — lesson-player offline mode

A short design doc for adding offline-mode support to the lesson-player. The header, options table, success metrics, and FAQ shown — the body of the doc lives in-repo at docs/design/2026-05-offline-mode.md.

# DD-014 · Lesson-player offline mode

status:        IN REVIEW
authors:       @oleg, @anna
reviewers:     @ivan (architect) · @maria (security) · @sasha (mobile)
created:       2026-05-19
target decide: 2026-05-26
links:         supersedes none · prior ADR-0087 (PWA shell)

## Context
Mobile users report dropped lessons on metro commutes — Wi-Fi
gaps cause the player to lose state mid-question. iOS App Store
review #4 cited offline as a missing-feature concern.

## Options considered

| Option                          | Pros                       | Cons                          | Effort  |
|---------------------------------|----------------------------|-------------------------------|---------|
| A. Service-worker cache (PWA)   | reuses existing PWA shell  | iOS Safari quirks; 50 MB cap  | 3 wks   |
| B. Native shell + SQLite        | full control; reliable     | new build pipeline; team gap  | 10 wks  |
| C. Hybrid: SW for content,      | balanced; ships in stages  | dual code paths until A2 done | 6 wks   |
|    IndexedDB for state          |                            |                               |         |

Recommendation: Option C. Stages naturally (A first, then B).

## Success metrics
- offline-completion ratio > 95% on metro-route fixture (today: 38%)
- lesson-resume success ratio > 99% (today: 76%)
- zero increase in p99 cold-start latency (today: 1.2 s)
- App-Store rating recovers to 4.6+ within 90 days of ship

## Risks
- iOS Safari storage caps may break large-content lessons → mitigate
  with content-chunking + LRU eviction (design in appendix)
- IndexedDB schema migrations are painful → add migration runner
  before any v2 schema lands

## FAQ
Q: Why not just retry on reconnect?
A: We retry today. The failure mode is loss of in-question state,
   not network loss — the user has to redo the question.

Q: Why not native shell now?
A: Team has no iOS-native depth. Option C buys us 3 months to hire
   before we commit to native.

Q: What about Android?
A: Same SW + IndexedDB path; tested on Chrome / Firefox / Samsung.

## Open questions
1. Should we cache user-generated answers offline? (privacy review needed)
2. What is the eviction policy when the 50 MB cap is hit?
3. Do we expose offline-mode as a setting, or auto-enable on detection?

The doc is 2 pages and fits one screen of attention. Reviewers comment in-thread; the FAQ grows as questions land. Once accepted, an ADR captures the chosen option and links back to this doc.

Quality bar

  • Written before implementation begins. First implementation PR cites the design-doc PR. Retroactive docs do not count.
  • Two real options minimum. Single-option docs are converted to short ADRs. The Options section is real analysis or it is not there.
  • Cap at 10 pages. Body fits in one reading. Detail lives in linked appendices, not in the body.
  • Named reviewers with an SLA. 3–5 reviewers explicitly named; 3–7 day async window; sync only on stall.
  • FAQ before review opens. At least five anticipated questions answered up front; updated with every novel reviewer question.
  • Success metrics measurable. Quantitative targets with baselines, not aspirations.
  • Linked to an ADR on acceptance. Acceptance criteria include "ADR authored and linked both ways".
