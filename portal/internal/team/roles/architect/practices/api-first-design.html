API-first design — contract before code

Write, review, and agree the API contract before any implementation begins — so consumer teams can work in parallel and integration becomes predictable.

TL;DR

API-first means the OpenAPI 3.1 (or AsyncAPI) contract is written, reviewed, and merged as the first deliverable — before code. The spec becomes the source for mock server, SDKs, contract tests, and docs. Drift between code and spec is caught by contract tests in CI. Teams that adopt this reduce integration time by 60–80% because consumers develop against the mock while implementation is in flight.

What it is

API-first is a development practice in which the API contract is the first deliverable. Before any implementation, the team writes an OpenAPI 3.1 spec (for REST) or AsyncAPI 2.x spec (for event-driven). The spec is reviewed, agreed, and merged as a PR. From the spec, the following are generated automatically:

  • Mock server (Prism, WireMock) — consumers develop against the API before implementation exists.
  • Client SDKs — generated via OpenAPI Generator, reducing consumer integration time.
  • Contract tests — Dredd or Schemathesis verify the running implementation matches the spec; CI fails on drift.
  • Documentation — generated from the spec, published on every release.

A spec linter (Spectral) enforces API design guidelines: consistent naming, pagination on collection endpoints, standard error envelope, versioning policy.

Why we need it

Teams that code first and document later accumulate API debt that makes every integration expensive. Without a contract, every consumer reads source code or sends trial HTTP requests. Undocumented breaking changes break consumers silently. Mobile and backend cannot work in parallel — mobile blocks waiting for backend "working enough".

API-first eliminates these failure modes. The mock server enables parallel development from day one. Contract tests make breaking changes visible immediately. Generated docs are always current. At startup speed, the ability for mobile/web/backend to work in parallel is one of the highest-leverage architectural decisions.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Stripe — spec as source of truth

Stripe's entire API is API-first: the OpenAPI spec is authoritative; docs, SDKs (10+ languages), and changelog all derive from it. Their public API design guide defines conventions for naming, pagination, idempotency keys, error envelopes. New endpoints require an API design review before implementation — the spec review is the gate, not the code review.

What we take: The spec review IS the architecture review for API endpoints. Code review validates implementation; spec review validates API design. Separate them.

Source: Stripe — Payment API design

Zalando — published guidelines + linter

Zalando published 64 RESTful API rules enforced by the Zally linter on every spec change. All APIs are designed API-first. New APIs go through a mandatory "API Design Review" with an API guild of 5–10 senior engineers; spec must pass all Zally rules before review proceeds.

What we take: Published guidelines + automated linting reduce API inconsistency by 80–90% across teams. The linter scales; human review handles the judgment calls the linter can't.

Source: Zalando RESTful API Guidelines

Figma — TypeScript types from spec

Figma uses API-first for both their public plugin API and internal service APIs. The public API was designed spec-first, enabling them to publish comprehensive TypeScript types alongside REST docs. Internal APIs follow the same pattern. Their versioning strategy relies on the spec as the source for deprecation timelines.

What we take: Generate TypeScript types (or Python stubs) from the OpenAPI spec and publish them alongside the docs. Consumers with types catch API misuse at compile time rather than runtime.

Source: Figma Engineering — Infrastructure

GitHub — versioned REST + GraphQL coexistence

GitHub's REST API has an OpenAPI spec; the GraphQL API has a schema. Both are first-class. The spec is the contract, the docs are generated, and SDKs ship for multiple languages. GitHub publishes deprecation timelines in the spec itself (machine-readable), enabling client tooling to surface upcoming breakages.

What we take: Machine-readable deprecation in the spec lets SDKs warn consumers before they break. Add a x-deprecated-at extension to endpoints with a date; CI for clients flags upcoming sunsets.

Source: GitHub REST API docs

Across teams: spec is the contract (not the docs), linter + guidelines enforce consistency, mock server unblocks consumers, contract tests gate ship, types/SDKs auto-generated. Without the linter the spec drifts; without the contract test the code drifts.

How it works in this project

Study App keeps OpenAPI 3.1 specs in services/<name>/openapi.yaml. CI runs Spectral on every PR that touches a spec; the lint ruleset (extending Spectral's recommended) is at governance/api/.spectral.yaml. Spec PRs are reviewed by the architect before any implementation work; the PR template includes a checkbox: "Spec change? → no impl PR until spec PR is merged".

Mock server (Prism) auto-deploys from the spec to https://mock.studyapp.dev within 5 min of spec merge. Consumer teams (web, mobile) point at the mock until the real implementation is available. Schemathesis contract tests run on every implementation PR against the running service; failure blocks merge. Generated SDKs (TypeScript, Python, Swift) are published to internal package registries on each release.

A short ADR locks the spec-PR-first rule, the Spectral ruleset, and the rule that any breaking change to a spec requires a new major-version path (not a silent edit).

Deep dive — the theory you need

Spec-first vs code-first — the inversion that matters

Most frameworks (FastAPI, Springfox, NestJS) make code-first easy: annotate the handlers, generate the spec automatically. Looks like API-first but isn't. The problem: API design is driven by implementation convenience — "what's easy to expose in this framework?" — not by consumer needs. The spec describes what the code happened to do, not what consumers needed. The result: APIs that are easy to build and hard to use.

True API-first inverts: write the spec by hand, then implement to it. Use annotation-based generators as validation — generate a spec from the code and diff against the handwritten one. If they don't match, that's a bug to fix.

Contract test depth — beyond 2xx

Most teams' contract tests only cover the happy path. They send valid requests, get 2xx responses, the schema matches. But APIs break consumers on error paths: an undocumented 400 shape, a 500 with HTML in the body, a 404 that returns 200 with an "error" field. Three levels of contract test depth:

  1. L1 — Happy path schema match: response status + body schema matches spec for each endpoint with example.
  2. L2 — Error envelope coverage: at least one test case per documented error code (400/401/403/404/422). Verifies error responses also match schema.
  3. L3 — Property-based fuzzing (Schemathesis): generate random valid + invalid requests from the spec; verify responses are always schema-compliant. Catches edge cases tests didn't think to write.

L1 is table stakes. L2 catches the most common consumer-breaking bugs. L3 finds the bugs you don't know about.

Spectral rules — what's worth enforcing

A starter set of 15–20 high-value rules:

  • Endpoint paths in kebab-case; resource collections plural (/lessons not /lesson).
  • Request/response field names in camelCase consistently.
  • Collection endpoints have pagination (limit+cursor or page/per_page).
  • Error responses use a single envelope shape ({ code, message, details? }).
  • Every response has a documented status code; no implicit 200.
  • All 4xx + 5xx documented per endpoint.
  • Operation IDs unique; descriptive (used by SDK generators as function names).
  • No path parameters in PATCH bodies; no body in GET.
  • Authentication scheme documented at the operation level.
  • Server URL has a major version prefix (/v1); no unversioned endpoints.

Start with these; add rules as patterns emerge from real spec reviews. Cap at ~60 rules — beyond that, engineers stop reading the lint output.

Versioning policy — the load-bearing decision

Two common patterns:

  • URL versioning (/v1/lessons, /v2/lessons) — explicit, cacheable, easy to deprecate. Used by Stripe, GitHub, most public APIs.
  • Header versioning (Accept: application/vnd.studyapp.v2+json) — RESTful-pure, harder to debug, less common.

Pick one; document it; never mix. Add machine-readable deprecation in the spec: x-deprecated-at: 2027-01-01 on operations. SDKs can surface upcoming sunsets to consumers automatically.

Mock fidelity — examples are the contract too

A mock server is only as useful as the examples in the spec. Spec with type: string only → mock returns "string". Spec with example: "Bonjour" → mock returns realistic data. Consumers test against realistic data; the bar for spec quality lifts. Require examples on every schema; lint for it.

Trade-offs — what this practice costs

  • Up-front spec time. 2–8 hours per endpoint vs jumping to code. Pays back many-fold via parallel consumer development, but the cost is visible immediately.
  • Tooling investment. Spectral + Prism + Schemathesis + SDK gen pipeline = 1–2 weeks initial setup. Ongoing maintenance ~1 hr/week per spec.
  • Cultural friction. Engineers used to code-first resent "writing YAML". The fix: show the time saved at integration. The mock server unblocking mobile in week 1 is the converting story.
  • Spec rot. Without contract tests, the spec drifts behind reality. Without lint enforcement, the spec drifts in quality. Both gates are non-negotiable.

Lifecycle

design spec PR → Spectral lint passes → spec review (architect) → spec merge · mock auto-deploys → implementation in parallel → contract test gate → ship + SDK + docs

Spec PR is the gate; implementation PR follows. Mock server auto-deploys on spec merge so consumers can start. Contract tests gate the implementation PR. SDKs and docs auto-publish on release.

Common pitfalls

PitfallWhy it happensWhat to do instead
Spec generated from code Framework auto-doc (FastAPI, Springfox) is easy; team uses it as "the spec". Write spec by hand first; use annotation generator as validation tool (diff against handwritten). Spec drives design; code matches spec.
Contract tests only happy-path Easiest to write; 2xx tests pass. Cover error envelopes (≥ 1 test per documented 4xx/5xx). Add Schemathesis property-based fuzzing for edge cases.
Spec review after implementation "Let's see if it's feasible first." Reviews happen when changes are expensive. Spec review BEFORE implementation. 30-min YAML review prevents days of rework.
No spec examples Spec author skips the example: fields. Spectral rule: every schema has examples. Mock server quality depends on examples; consumer experience does too.
Breaking changes via spec edit Author updates v1 spec in place; consumers' SDKs break silently. Breaking change = new major-version path. Old spec stays; deprecation date marked in x-deprecated-at.
Spec drift — no contract test Spec is "documentation"; not run against code. Contract test in CI on every implementation PR. Failure blocks merge. Without this, spec is decoration.

Worked example

OpenAPI fragment

Study App — POST /v1/lessons/{id}/complete

openapi: 3.1.0
info:
  title: Study App API
  version: 1.4.0
servers:
  - url: https://api.studyapp.com/v1

paths:
  /lessons/{id}/complete:
    post:
      operationId: completeLesson
      summary: Mark a lesson as completed by the current user
      tags: [lessons]
      security: [{ bearerAuth: [] }]
      parameters:
        - { name: id, in: path, required: true, schema: { type: string, format: uuid } }
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [completedAt]
              properties:
                completedAt: { type: string, format: date-time }
                score:       { type: integer, minimum: 0, maximum: 100 }
            example: { completedAt: "2026-05-24T14:30:00Z", score: 92 }
      responses:
        '200':
          description: Lesson completion recorded
          content:
            application/json:
              schema: { $ref: '#/components/schemas/LessonCompletion' }
              example:
                id: "evt_a1b2c3"
                lessonId: "lsn_42"
                userId: "usr_99"
                completedAt: "2026-05-24T14:30:00Z"
                streakDays: 13
        '404': { $ref: '#/components/responses/NotFound' }
        '409':
          description: Lesson already completed today
          content:
            application/json:
              schema: { $ref: '#/components/schemas/ErrorEnvelope' }
              example: { code: "ALREADY_COMPLETED", message: "Lesson already completed today" }

components:
  schemas:
    LessonCompletion:
      type: object
      required: [id, lessonId, userId, completedAt]
      properties:
        id:           { type: string }
        lessonId:     { type: string }
        userId:       { type: string }
        completedAt:  { type: string, format: date-time }
        streakDays:   { type: integer, minimum: 0 }
    ErrorEnvelope:
      type: object
      required: [code, message]
      properties:
        code:    { type: string }
        message: { type: string }
        details: { type: object, additionalProperties: true }

Outcome. The spec was reviewed in 40 minutes by the architect. The mobile team started building the streak-increment UI against the Prism mock at https://mock.studyapp.dev/v1/lessons/.../complete on day 1, returning the realistic streakDays example. The backend implementation took 6 days; when the real endpoint shipped, mobile switched the base URL with no code changes. Schemathesis caught one drift on day 5 — implementation returned completed_at (snake_case) but spec said completedAt (camelCase); the team fixed before merge. Without contract tests, that drift would have shipped and broken mobile.
