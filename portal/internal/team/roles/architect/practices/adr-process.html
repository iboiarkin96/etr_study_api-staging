ADR process — numbered, durable decisions

A one-page record of why each architectural decision was made, stored next to the code it describes — never deleted, only superseded.

TL;DR

An ADR (Michael Nygard, 2011) is a short numbered file with three sections: Context, Decision, Consequences. Permanent integer number (0001, 0042…), status (proposed / accepted / superseded), stored in the repo next to code. Never deleted — only superseded by a later ADR. New hires read ADR 0042 once and stop asking the question in chat.

What it is

An Architecture Decision Record is a one-page document capturing a single architectural decision. The format was popularised by Michael Nygard in his 2011 essay Documenting Architecture Decisions. Three required sections:

  • Context — what's happening, what constraints we face, what forces push the decision. Plain prose. The reader should understand why a decision was needed without consulting external sources.
  • Decision — the chosen path, stated in active voice: "We will use PostgreSQL for the primary datastore." Specific, not aspirational.
  • Consequences — what now changes because of this decision. Both good and bad. An ADR with no downside listed is half-written.

The MADR 4.0 template (Markdown ADR) extends Nygard's original with fields for Status, Date, Deciders, and a Considered Options table — useful at organisational scale. The file has a permanent integer number (ADR 0042) that never changes, even after supersession.

Why we need it

A year from now, the team will not remember why the project chose gRPC over REST for internal services. The decision will become tribal knowledge held by the three engineers who were in the original meeting — and when they leave, that knowledge leaves with them. The next engineer to question the choice re-argues it from scratch.

An ADR fixes this in 30 minutes. The decision is written down with its context. When a new engineer asks "why gRPC?", the answer is "read ADR 0017." The number gives every decision a stable link — across reorgs, repo moves, team changes. Writing the Consequences section also improves the quality of decisions: the team must be honest about trade-offs.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Michael Nygard (originator)

Wrote the original ADR essay in November 2011 on the Cognitect blog. The format was deliberately minimal — three sections, one page, plain prose — because heavier formats (architecture documents, design specs) were never written and never read. The minimalism is the feature: an ADR is fast to write and fast to read, which is the only way it gets done.

What we take: Resist expanding the template. Three sections work because they fit on one screen. Adding "Risks" or "Open Questions" sections reduces adoption faster than it improves quality.

Source: Cognitect — Documenting Architecture Decisions (2011)

Spotify — MADR at scale

Spotify adopted MADR (Markdown ADR) as the canonical template across all squads. Their ADRs include a "Deciders" field (named individuals, not roles) and a "Considered Options" table that captures alternatives explicitly. 200+ squads, every significant choice has an ADR. ADRs are surfaced in Backstage so engineers can find decisions by service, author, or date.

What we take: Named "Deciders" (not roles) are searchable across time; roles change. Add a "Considered Options" table — the trail is the value, not just the final choice.

Source: MADR — Markdown ADR template

AWS Well-Architected — audit asset

AWS Well-Architected Framework recommends ADRs as canonical artefacts for the Operational Excellence pillar. Their guidance: ADRs documenting why a particular AWS service was chosen are gold during compliance audits — "why DynamoDB and not Aurora?" has a documented answer rather than a meeting transcript that no longer exists.

What we take: ADRs are an audit asset. When a cloud provider, regulator, or customer asks "why did you choose X?", an ADR is the answer. Treat them with audit-document rigour.

Source: AWS Prescriptive Guidance — ADRs

ThoughtWorks — ADR Tools

ThoughtWorks built and maintains adr-tools, a CLI for creating, indexing, and superseding ADRs. adr new scaffolds a numbered file; adr supersede 17 99 wires the cross-links automatically. Friction-free creation is what gets ADRs written; the tool eliminates the bookkeeping excuse.

What we take: Tooling matters. If creating an ADR takes 20 minutes of template-copying, engineers don't write them. A 30-second CLI gets ADRs created on the same day as the decision.

Source: adr-tools (Nat Pryce / ThoughtWorks)

Across teams: one page, three sections, permanent number, named deciders, tooling that removes friction, supersession trail not deletion. Add Considered Options table at scale. The file never dies — it just changes status.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Permanent number

Sequential integers (0001, 0042); assigned at PR creation; never reused even after supersession. The number is the stable link across years.

Status field

proposed / accepted / declined / superseded by NNNN. Status changes are themselves PRs. Accepted ADRs are never content-edited (typos OK).

Context (1 paragraph)

What's happening; what forces push the decision. Stand-alone — reader should not need other documents.

Decision (active voice)

"We will use X." Specific, not aspirational. One sentence ideal, one paragraph max.

Consequences (both sides)

Positive and negative outcomes. An ADR with no downside listed is half-written and gets challenged at the first issue.

Considered options (at scale)

Optional MADR table: the alternatives, with one-line pros/cons each. The trail of "we considered X and rejected because Y" prevents re-litigation.

Supersession link

When 0099 supersedes 0017, 0017's status updates with a forward link; 0099 has a backward link explaining what changed.

Repository location

docs/adr/ next to the code it describes. NNNN-kebab-case-title.md. Index page auto-generated in CI.

CLI tooling

adr new / adr supersede commands. 30-second creation kills the "I'll write it later" excuse.

How it works in this project

Study App stores ADRs in docs/adr/ as numbered markdown files. The CLI helper is the scripts/adr.sh wrapper around adr-tools: ./scripts/adr.sh new "Use PostgreSQL for primary store" scaffolds 0042-use-postgresql-for-primary-store.md with the template prefilled.

Every PR that changes architecture (database, queue, framework, deployment topology, security boundary) must add or supersede an ADR. The PR template includes a checkbox: "Architecture change? → link the ADR PR". The architecture review reads the ADR before reviewing the code; if the ADR is missing, the review is rescheduled.

Quarterly retro reviews the ADR catalogue: undocumented major decisions get retroactive ADRs (with explicit "retroactive" tag); ADRs that no longer reflect current practice get superseded or marked deprecated. A short ADR locks the template format, the supersession rule, and the PR-trigger criteria.

Deep dive — the theory you need

Why the file persists — supersession over deletion

The single most-debated rule: never delete an ADR; supersede it. The reason is institutional memory. ADR 0017 chose Library X in 2024; in 2026 ADR 0099 supersedes it with Library Y. A new engineer in 2027 wonders "why didn't they use Library X from the start?" — the answer is in 0017's Context. Deleting 0017 erases the answer; superseding preserves it. The chain 0017 → 0099 is the decision history. Five years of ADRs is a readable archaeology of the architecture.

What counts as an ADR-worthy decision

Not every choice deserves an ADR. The test: would a future engineer ask "why did we choose this?". Heuristics:

  • ✓ ADR-worthy: database choice, message broker, auth mechanism, deployment topology, framework choice, API style (REST/gRPC/GraphQL), language version, key dependency adoption, security model, data partitioning, multi-tenancy approach.
  • ✗ Not ADR-worthy: code style, variable naming, individual function design, library version bumps, single-team conventions documented elsewhere.

Borderline cases lean toward writing the ADR — 30 minutes now saves 3 hours of confusion later.

The Considered Options table at scale

Nygard's original 3-section format works for small teams; Spotify-scale orgs add a MADR Considered Options table. Sample:

OptionProsCons
PostgreSQL ← chosenTeam knows it; ACID; rich types; mature ecosystemSingle-node scaling ceiling at ~5 TB
MySQLTeam knows it; widely deployedType system weaker; ecosystem fragmented
DynamoDBOperational simplicity; AWS-nativeVendor lock-in; not relational; team unfamiliar

The table makes the trail explicit. When someone proposes DynamoDB in 6 months, the ADR shows it was already considered and rejected — re-opens the conversation only with new evidence.

Decision archaeology — the retroactive ADR

Most projects discover, 2 years in, that the biggest architectural choices have no ADRs — they happened organically in week 1 when no process existed. Schedule a quarterly "decision archaeology" review: identify undocumented major decisions; write retroactive ADRs; tag them in the status as retroactive · written N months after decision. Honesty about the origin protects the practice's credibility.

Supersession discipline

A common failure mode: ADRs accumulate but nothing is ever superseded. The list grows; conflicts accumulate; readers can't tell which ADRs are still in effect. Add supersession review to the quarterly architecture retrospective. Any ADR that no longer reflects current practice should either be superseded by a new ADR or have its status changed to "deprecated" with rationale. An active project should have a steady supersession rate — ~5-10% of ADRs superseded per year is healthy.

ADR vs RFC vs design doc — three different artefacts

Three documents often confused:

  • RFC — proposal under discussion. May be rejected. Lives in docs/rfc/. Has authors and reviewers. Outputs an ADR on acceptance.
  • ADR — decision made. Records what was chosen and why. Permanent record.
  • Design doc — detailed engineering design. References the ADR. May be many pages with code, diagrams, test plans.

Healthy flow: RFC proposes → discussion → if accepted, an ADR records the decision → design doc implements. Skipping the ADR makes the decision invisible 6 months later; skipping the design doc makes the implementation invisible.

Trade-offs — what this practice costs

  • Writing time. 20–45 min per ADR with the CLI. Cheap, but visible. Without tooling, can balloon to hours.
  • Process drag. "Wait for the ADR" adds 1–3 days of PR review time. Some teams resent this; the discipline is the value.
  • Bikeshedding risk. ADR reviews can become 50-comment threads. Time-box: ADR review ≤ 3 business days, decisions stand unless serious objections surface.
  • Template drift. Teams add fields ("Risks", "Open Questions"); template grows; people stop writing ADRs. Reset to 3 sections regularly.

Lifecycle

decision needed → draft in PR (proposed) → PR review · time-boxed → accepted (merged) or declined → in effect → superseded by later ADR → quarterly catalogue review

Draft lives in a PR branch as proposed. PR review is time-boxed (≤ 3 business days). On merge, status flips to accepted. The file never changes content again — only its status can change. When superseded, the file's status updates to "superseded by NNNN" with a forward link; the new ADR has a backward link explaining what changed. Quarterly catalogue review identifies stale ADRs needing supersession.

Common pitfalls

PitfallWhy it happensWhat to do instead
ADRs grow into design docs (10+ pages) Authors include implementation detail, code, exhaustive analysis. Hard limit: one page. Implementation goes in a separate design doc that links the ADR.
Major decisions made without an ADR Big choices happen organically without a formal moment. Quarterly "decision archaeology" review: write retroactive ADRs for undocumented choices; tag them clearly.
No supersession discipline List grows; nothing is ever superseded. Quarterly supersession review. Mark deprecated ADRs; supersede outdated ones. ~5-10% turnover per year is healthy.
Template drift — fields multiplying "Let's add a Risks section." Then Open Questions. Then… Reset to 3 sections (Context · Decision · Consequences) at quarterly retro. MADR's Status + Deciders + Considered Options is the cap.
ADR review balloons into bikeshedding Async PR comments accumulate; nobody merges. Time-box review to ≤ 3 business days. Designated decider (architect or named role) calls the merge.
No CLI tooling — every ADR takes an hour Engineers copy a template manually, fight with numbering. Install adr-tools or equivalent. adr new in 5 seconds; adr supersede wires links automatically.

Worked example

ADR · markdown

ADR 0042 — Use PostgreSQL for the primary datastore

# ADR 0042 · Use PostgreSQL for the primary datastore

status:    accepted
date:      2026-05-18
deciders:  @ivan (architect) · @oleg (eng lead) · @anna (data)
supersedes: —
superseded by: —

## Context

Study App needs a primary datastore for user accounts, lesson progress, and
billing. Expected size in 24 months: 5M users, ~50 GB of relational data,
~500 RPS read / 100 RPS write at peak. The team is 6 engineers, 2 of whom
have deep PostgreSQL experience, 0 have DynamoDB. Compliance requires
point-in-time recovery and EU data residency.

## Decision

We will use PostgreSQL 16 as the primary datastore, hosted on managed
AWS RDS in eu-west-1 with cross-AZ replication and 35-day PITR.

## Consequences

Positive:
+ Team already knows the technology; no ramp-up cost
+ ACID transactions match billing's correctness requirements
+ Rich type system (JSONB, arrays) covers structured + semi-structured data
+ Mature ecosystem: migrations (Flyway), backups, monitoring tools
+ Single technology for both transactional and basic analytical queries
  (in the < 50 GB range)

Negative:
- Single-node scaling ceiling at ~5 TB; will need partitioning or
  citus extension if growth exceeds projection
- Operational complexity higher than DynamoDB (we own connections,
  tuning, vacuum)
- Schema migrations require downtime-careful operations at scale

## Considered options

| Option       | Pros                                    | Cons                             |
|--------------|-----------------------------------------|----------------------------------|
| PostgreSQL ← | Team expertise; ACID; rich types        | Single-node ceiling ~5 TB        |
| MySQL        | Team partial expertise; widely deployed | Weaker type system; ecosystem    |
| DynamoDB     | Ops simplicity; AWS-native              | Vendor lock-in; not relational;  |
|              |                                         | team unfamiliar; EU residency    |
|              |                                         | adds complexity                  |
| CockroachDB  | Horizontal scale; PG-compatible         | Higher cost; team unfamiliar;    |
|              |                                         | over-engineered for current size |

Outcome. Written in 35 minutes after the kick-off meeting. The Considered Options table later prevented two re-litigations: a new engineer proposed DynamoDB at month 4 ("AWS-native is better"), and the table's cons answered them in 30 seconds; a vendor pitched CockroachDB at month 8 ("for future scale"), and the table's "over-engineered for current size" gave the architect a reasoned response without re-running the analysis. Without the ADR, both conversations would have been 2-hour meetings.
