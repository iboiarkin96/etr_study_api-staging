NFR definition — make the ilities explicit

Capture quality attributes as measurable acceptance criteria so "fast enough" and "secure enough" mean the same thing to every engineer.

TL;DR

Non-functional requirements (NFRs) are the measurable quality attributes of a system: performance (p99 ≤ 200 ms), availability (99.9%), scalability (10× peak), security (no plaintext secrets), maintainability (MTTR ≤ 1 h). ISO 25010 codifies the categories; arc42 captures them as scenarios with stimulus + response + measure. NFRs live per service, are validated by automation where possible, and re-checked at every architecture review. Without explicit NFRs, "good enough" means whatever the most recent engineer thought it meant.

What it is

Non-functional requirements are the measurable quality attributes a system must satisfy. They are distinct from functional requirements (what the system does) — NFRs describe how well it must do it. ISO/IEC 25010 defines eight characteristics; in practice most engineering teams use a reduced set of five:

  • Performance — latency (p50, p95, p99), throughput (RPM/RPS), and resource utilisation under load.
  • Availability — uptime targets (99.9% / 99.99%), recovery time objectives (RTO), recovery point objectives (RPO).
  • Security — authentication strength, data-at-rest and in-transit encryption, threat-model coverage, OWASP Top 10 compliance.
  • Scalability — ability to handle 10× current load, horizontal-scaling boundaries, statelessness requirements.
  • Maintainability — MTTR (mean time to recovery), test coverage thresholds, runbook coverage, deployment frequency.

Each NFR lives per service in a YAML or Markdown file alongside the code, with a measurable target and a verification mechanism. Where automation is possible (latency, security scans), NFRs run as fitness functions in CI. Where automation is not possible (threat-model coverage), they are reviewed manually during architecture review.

Why we need it

Without explicit NFRs, "fast enough", "secure enough", and "scalable enough" mean different things to every engineer. The team ships a service that works at 100 RPM and discovers at 10× load that it falls over — because no one specified the 10× target. The team builds an authentication system that meets engineer intuition for "secure" and discovers in a SOC 2 audit that the standard required something specific.

NFRs turn implicit expectations into testable acceptance criteria. They make trade-offs explicit at design time rather than discoverable in production. An NFR document also creates a contract with stakeholders — when an executive asks "can we handle Black Friday traffic?", the answer is a referenced NFR with a verification record, not a guess.

How others do it

arc42 — quality requirements framework

arc42 is an open-source architecture-documentation template maintained by Gernot Starke and Peter Hruschka since 2003. Its quality-requirements model uses the "quality tree" — a hierarchical decomposition of quality characteristics into scenarios with stimulus, environment, response, and response measure. The format encourages teams to write quality scenarios rather than vague targets: "If order rate exceeds 1000 RPM, the order service must continue to accept orders with degraded notifications."

What we take: Capture NFRs as scenarios, not numbers. "p99 ≤ 200 ms" becomes "Under 5 000 RPM sustained load, 99% of GET /orders requests return within 200 ms." Scenarios make the test conditions explicit.

Source: arc42 quality requirements

ISO/IEC 25010 — the standard

ISO 25010 is the international standard for software product quality. It defines 8 characteristics: Functional Suitability, Performance Efficiency, Compatibility, Usability, Reliability, Security, Maintainability, Portability. Each is decomposed into sub-characteristics. The standard is the canonical reference for compliance frameworks (SOC 2, ISO 27001) and is cited by many enterprise procurement contracts.

What we take: Use the ISO 25010 vocabulary in NFR docs even when you do not adopt the full standard. Auditors and enterprise customers recognise the terms; they map cleanly to compliance frameworks.

Source: ISO 25010 reference

Google SRE — SLOs operationalise NFRs

Google's SRE practice operationalises performance and availability NFRs through SLOs — measurable targets backed by error budgets. The SRE Book documents the approach: every customer-facing service has an SLO, and engineering work is prioritised based on error-budget consumption. Google trained the industry to think of "availability" as a quantitative target rather than a qualitative aspiration.

What we take: Convert availability and performance NFRs into SLOs from day one. An SLO with an error budget creates accountability that an NFR alone does not. Pair with Practice 17 on SLO/SLA design.

Source: Google SRE — SLOs chapter

SEI / Bass et al. — Software Architecture in Practice

The Software Engineering Institute's Software Architecture in Practice (Bass, Clements, Kazman) formalised the quality-attribute scenario template still used widely today: source · stimulus · artifact · environment · response · response measure. Their Quality Attribute Workshop (QAW) method runs in 1–2 days with stakeholders and produces a prioritised set of scenarios as the input to architecture work.

What we take: Run a one-day QAW-style workshop with stakeholders before the first NFR draft. The scenarios the workshop produces are sharper than anything an architect would write alone — the workshop is where hidden expectations surface.

Source: SEI — Quality Attribute Workshop

Anatomy of an NFR set

Per-service NFR file

YAML or Markdown next to the service code (docs/nfrs.yaml). Captures the agreed NFRs for that service with categories, targets, and verification mechanisms. Updated in the same PR as any architectural change that affects them.

Measurable target

Each NFR has a quantitative target. "Fast" is not measurable; "p99 latency ≤ 200 ms at 5k RPM sustained 5 min" is. Targets without a load context are incomplete — they invite gaming ("we're at 200 ms at 100 RPM").

Scenario (arc42-style)

Optional: write each NFR as a scenario — source · stimulus · environment · response · measure. Forces the trigger condition to be explicit. "Source: external load. Stimulus: 5 000 RPM. Environment: prod, peak hour. Response: 99% requests served. Measure: ≤ 200 ms."

Verification mechanism

For each NFR, the method to verify it. Three categories: automated (load test in CI, SAST scan), operational (production SLO continuously monitored), audited (manual review with named owner). NFRs without verification are aspirations, not requirements.

Owner per NFR

Each NFR names an accountable owner — usually the service tech lead. The owner runs the verification, reports the result at architecture review, and decides whether a violation is acceptable or blocks ship.

Architecture-review gate

NFR set reviewed in every architecture review and quarterly retrospective. Checklist: "NFRs defined?" "Changes re-validated?" "Targets still realistic?" A missing or stale NFR document blocks the review.

How to run it

  1. Run a 1-day QAW (Week 1). Stakeholders, architect, service tech lead. Produce prioritised quality-attribute scenarios in arc42 format.
  2. Pick 5–8 NFRs per service (Week 2). One per category that matters most to this service. Backend transactional service emphasises performance + availability; batch pipeline emphasises throughput + resilience.
  3. Write measurable targets (Week 2). Numbers with load context. Pair availability NFRs with explicit SLOs.
  4. Choose verification per NFR (Week 2–3). Automated where possible (k6, SAST, dependency check); operational SLO for runtime; audited manual review only when no automation exists.
  5. Wire verification into CI (Week 3–4). Latency NFRs become k6 tests on staging deploy; coverage NFRs become CI thresholds; security NFRs become SAST gates. NFRs as fitness functions.
  6. Capture in YAML in-repo. docs/nfrs.yaml alongside the service. Same PR for any change that affects them. Renderable to a service-NFR table.
  7. Review annually and on major change. Architecture retrospective revisits each service's NFRs. Stale NFRs are a higher risk than missing ones — re-validate before drift becomes incident.

Deep dive — making the ilities load-bearing

Numbers with load context, or do not bother. "p99 ≤ 200 ms" is meaningless without "at what load". A service can hit 200 ms at 100 RPM and miss it at 5 000 RPM. Every performance target carries its load context: rate, duration, environment. Same for availability ("99.9% over a 30-day window"), security ("OWASP Top 10 verified by SAST on every PR"), and maintainability ("MTTR < 1 h measured across the last 10 incidents"). Numbers without context invite gaming.

Verification or it is not a requirement. The dividing line between an NFR and an aspiration is verification. Three legitimate verification modes: automated (load test, scan, lint), operational (SLO continuously monitored in prod), audited (named manual review at fixed cadence). An NFR with no verification is wishful thinking. Coach engineers to attach the verification at the same time as the target — never "we'll figure out how to test it later".

Cap at 5–8 NFRs per service. Teams who copy ISO 25010 wholesale end up with 40 NFRs per service; none are verified; the document becomes a checkbox exercise. Pick the 5–8 that matter most for the specific service shape (transactional vs batch vs frontend vs analytical) and verify them properly. Fewer real NFRs beat more nominal ones.

NFRs decompose differently by service shape. A user-facing transactional service prioritises performance + availability + security; a batch pipeline prioritises throughput + correctness + resilience; an analytical store prioritises query latency + data freshness; a public API prioritises stability + auth + rate-limit + abuse. Do not paste the same NFR template everywhere — match the shape.

Pair availability NFRs with SLOs. "99.9% availability" stays nominal unless it is operationalised as an SLO with an error budget and an SLO-burn alert. The SLO is what makes the NFR actionable in week-to-week engineering. See Practice 17 (SLO/SLA design) for the operational machinery.

Re-validate annually. The biggest failure mode is "defined once, never re-checked". Two years later the system has 10× the traffic and the original NFR is either obsolete or unmet — but nobody looks. The annual architecture retrospective is the forcing function: every service's NFR set is re-checked; stale targets get refreshed; unverified targets get verification or get retired.

Trade-offs are first-class. NFRs always trade against each other: higher availability costs latency (more retries); stronger security costs developer velocity (more reviews); higher scalability costs simplicity (more moving parts). Capture the trade-offs explicitly in the NFR doc so future engineers see what was traded away — and can revisit when conditions change.

Lifecycle of an NFR

Elicited

Surfaced in the QAW or a design discussion. Captured as a scenario with stakeholder + stimulus + expected response.

Quantified

Scenario translated into a measurable target with load context. Owner named. Verification mechanism selected.

Active

Verification wired (CI test, SLO, scheduled audit). NFR file updated in-repo. Cited at architecture review and design-doc reviews.

Breached

Verification fails (load test regresses, SLO burns, audit finds a gap). Owner triages: fix, escalate, or accept with a written exception.

Revised

Target or scenario updated as the system evolves. Old version kept in history. Reason for revision noted in the changelog.

Retired

Capability removed or the NFR no longer reflects reality. Retirement note explains what changed; verification removed; ADR updated if applicable.

Common pitfalls

Pitfall Why it happens What to do instead
Aspirational NFRs — no measurable target Team writes "the system should be fast and reliable" because committing to numbers feels risky. Vague NFRs are uncomfortable for no one and useful to no one. Force quantitative targets with load conditions. "p99 ≤ 200 ms at 5 000 RPM sustained" is uncomfortable but useful. Start with conservative targets and refine over time.
NFRs defined once, never re-validated NFR file created at service launch and never updated. Two years later the system has 10× traffic and the original targets are obsolete or unmet — but nobody checks. Re-validate annually and on major architecture changes. Architecture retrospective is the natural forcing function. Stale NFRs are higher risk than missing ones.
Too many NFRs — paralysis Team copies ISO 25010 wholesale and ends up with 40 NFRs per service. None of them are verified. The document becomes a checkbox exercise nobody reads. Cap at 5–8 NFRs per service. Cover the categories that matter most to the specific service shape. Verified small set beats unverified large set.
No verification mechanism NFR has a number but no test, no SLO, no audit. Nobody knows whether it is met. The NFR is theatre. Every NFR ships with a named verification (automated, operational, or audited). No exceptions. The verification is part of the NFR, not a separate concern.
Numbers without load context "p99 ≤ 200 ms" with no rate or duration. Easy to hit at idle; impossible at peak. Target invites gaming. Every performance target carries rate + duration + environment. Same for availability (window), security (scope), maintainability (sample).
NFRs copy-pasted across service shapes Same NFR template applied to a transactional service, a batch pipeline, and an analytical store. None of the targets fit any of them well. Match NFR set to service shape. Transactional services emphasise latency + availability; batch emphasises throughput + correctness; analytical emphasises query latency + freshness.

Worked example — lesson-player NFRs

The starter NFR set for the lesson-player service. Lives in services/lesson-player/docs/nfrs.yaml next to the code; renderable as a table on the service home page.

# services/lesson-player/docs/nfrs.yaml
service: lesson-player
owner:   @oleg
reviewed: 2026-05-19

nfrs:
  - id: NFR-01
    category: performance
    scenario: |
      Source: external user load on metro hours.
      Stimulus: 5 000 RPM sustained for 5 minutes.
      Environment: prod, EU peak hour.
      Response: 99% of GET /lesson/{id}/state served.
      Measure: p99 latency ≤ 200 ms.
    target: "p99 ≤ 200 ms at 5 000 RPM sustained 5 min"
    verify: "k6 on staging deploy; fail if regress > 10%"
    owner:  "@oleg"

  - id: NFR-02
    category: availability
    target: "99.9% over 30-day window"
    verify: "SLO + error-budget alert; burn-rate ≥ 2× pages on-call"
    owner:  "@anna"
    notes:  "trades against latency: retries add p99"

  - id: NFR-03
    category: security
    target: "OWASP Top 10 verified; secrets via AWS KMS; TLS 1.3 only"
    verify: "Semgrep SAST on every PR; nightly dep-CVE scan; annual pentest"
    owner:  "@maria (security)"

  - id: NFR-04
    category: scalability
    target: "handle 10× current peak with horizontal scaling only"
    verify: "monthly k6 step-load test on staging up to 10× peak"
    owner:  "@oleg"

  - id: NFR-05
    category: maintainability
    target: "MTTR ≤ 1 h on last 10 P1 incidents"
    verify: "audit at quarterly retrospective; runbook coverage ≥ 80%"
    owner:  "@anna"

  - id: NFR-06
    category: privacy
    target: "no PII in logs; 30-day attempt-data retention"
    verify: "log scrubber config + retention policy in IaC; quarterly audit"
    owner:  "@maria"

trade-offs:
  - "NFR-02 (availability) ↔ NFR-01 (latency): retries add ~10 ms to p99
     under failure. Accepted; SLO budget covers."
  - "NFR-04 (scalability) ↔ simplicity: horizontal-only constraint
     bans stateful sessions. Documented in ADR-0094."

Six NFRs, each with target + scenario + verification + owner. Trade-offs captured explicitly so future engineers see what was traded away. The file lives in-repo and is updated in the same PR as any architectural change that affects it.

Quality bar

  • Every Tier-1 service has an NFR file in-repo. docs/nfrs.yaml next to the service. Updated in the same PR as architectural changes that affect them.
  • 5–8 NFRs per service. Tailored to service shape. Aspirational checklists rejected; verified small sets accepted.
  • Every NFR has a measurable target with load context. Numbers + rate + duration + environment. No vague language.
  • Every NFR has a named verification. Automated, operational SLO, or audited manual review at fixed cadence. No verification = no NFR.
  • Every NFR has a named owner. Usually the service tech lead. Reports verification result at architecture review.
  • Trade-offs explicit. Where NFRs trade against each other (availability vs latency, security vs velocity), the trade is captured in the NFR file.
  • Annual re-validation. Architecture retrospective revisits every service's NFRs. Stale targets refreshed; unmet ones get an explicit decision.
