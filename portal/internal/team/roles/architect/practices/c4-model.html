C4 model — architecture as code at four zoom levels

Communicate architecture at four levels of zoom so every audience gets the right abstraction — without a whiteboard session.

TL;DR

Simon Brown's C4 model (~2010) defines four zoom levels: Context (system in its environment), Container (apps + data stores), Component (internals), Code (rarely needed). Diagrams as code (PlantUML / Structurizr) in the repo, rendered in CI, stay in sync with reality. A new engineer reads a C4 Context diagram in 5 min and understands the system; without C4 it takes 5 days of conversations.

What it is

The C4 model is a set of four hierarchical diagram types for software architecture, created by Simon Brown. C4 = Context, Container, Component, Code. Each level answers a different question for a different audience:

  • L1 — Context: where does this system fit in the world? Users, external systems, the system. Audience: execs, new joiners, PMs.
  • L2 — Container: what is the system made of? Apps, services, databases, brokers. Audience: developers, architects, ops.
  • L3 — Component: what's inside a container? Major components + responsibilities. Audience: developers in that service.
  • L4 — Code: class-level. Rarely needed — the IDE shows the same thing.

Crucially: diagrams as code. PlantUML or Structurizr DSL in the repo; rendered in CI on every push. The diagrams stay in sync with reality because they're versioned alongside the code.

Why we need it

Every team produces architecture diagrams. Most are inconsistent: different shapes, different vocabulary, different levels of detail. A new engineer faced with five team-specific diagrams learns five visual languages before learning the system. C4 standardises: same notation, same four levels, same vocabulary across teams.

Diagrams-as-code solves the staleness problem. Visio / Lucidchart diagrams drift from reality within months because updating them is a manual chore detached from the code. C4 PlantUML files are PR'd alongside code changes — when the architecture changes, the diagram changes in the same PR.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Simon Brown — originator

Simon Brown developed C4 in 2010 after years of consulting where every team produced incompatible UML diagrams. C4's design principles: hierarchical (zoom in/out), consistent vocabulary (Person / System / Container / Component), notation-agnostic (PlantUML, Mermaid, Structurizr all valid).

What we take: Use the canonical vocabulary (Person, System, Container, Component) consistently. Mixing it with custom terms ("Service", "Application") breaks the cross-team recognisability.

Source: c4model.com

Structurizr — Brown's tool

Structurizr DSL is the canonical "architecture as code" tool from Simon Brown. Single text file defines all 4 levels; renderers produce PNG, SVG, or interactive HTML. Many teams adopt Structurizr DSL + Structurizr Lite (self-hosted) so diagrams render in CI without external services.

What we take: A single DSL file describes the whole architecture; multiple views are derived. Avoids the "out-of-sync L1 vs L2" problem you get with separate diagram files.

Source: Structurizr

ThoughtWorks — C4 in the Tech Radar

ThoughtWorks promoted C4 to "Adopt" status in their Technology Radar in 2018 and has consistently endorsed it since. Their consultants apply C4 across client engagements because the standardised vocabulary makes architecture portable between teams and companies.

What we take: C4 is broadly adopted enough that engineers joining from other companies recognise it immediately. Lower onboarding friction is itself a reason to choose C4 over a bespoke notation.

Source: ThoughtWorks Technology Radar

Backstage / Spotify — IDP integration

Spotify's Backstage internal developer portal renders C4 diagrams as part of each service's catalogue page. The diagram is pulled from the service's own repo (PlantUML files), so the developer-portal view is always current. Engineers find their service's L2 diagram with one click.

What we take: Surface C4 diagrams in the developer portal next to each service. Diagrams that engineers can find with one click get read; diagrams in a wiki get ignored.

Source: Backstage

Across teams: standardised vocabulary, diagrams as code, versioned with the code, surfaced in the developer portal. Without diagrams-as-code the diagrams drift; without portal integration they're not found.

Anatomy

One per system. Shows the system + Persons + external Systems. Required.

L2 Container diagram

One per system. Shows the Containers (apps + data stores + brokers) inside the system. Required.

L3 Component diagram (optional)

One per complex container. Shows major components and their interactions. Useful when L2 hides too much.

Versioned with the code

Architecture change PRs include diagram updates. CI fails if a containers-affecting code change has no diagram update.

Person / System / Container / Component. Don't substitute "Service" / "Application" — recognisability matters.

Portal integration

Rendered diagram surfaced on the service's catalogue page (Backstage or equivalent).

How it works in this project

Study App stores all C4 diagrams as PlantUML in docs/diagrams/c4/. L1 context diagram is at the org level (docs/diagrams/c4/01-context.puml); each service has its L2 container diagram at services/<name>/docs/c4-container.puml. CI renders to SVG on every push; SVGs are committed to docs/diagrams/_rendered/ so the portal can serve them without a build step.

L3 component diagrams exist only for the 2 most complex services (events pipeline, lesson player). L4 is intentionally not used — the IDE shows class structure better than any diagram.

A short ADR locks the vocabulary (Brown's canonical terms), the directory structure, and the rule that any PR adding/removing a Container requires a diagram update in the same PR (enforced by CI label check).

Deep dive — the theory you need

Why diagrams as code

Manually-edited diagrams (Visio, Lucidchart, draw.io exports) drift from reality within months. The author leaves; the diagram becomes a historical artefact nobody trusts. Diagrams-as-code solves this by treating diagrams the same as code: versioned in git, reviewed in PRs, rendered in CI, updated when the system changes. The activation energy to update a PlantUML file in the same PR as the code change is much lower than firing up Lucidchart and exporting a PNG.

L1 + L2 only — most teams overproduce

Many teams generate all four levels for every service. Result: 4× the diagrams to maintain, mostly never read. The right discipline: L1 + L2 are required and read often; L3 only when a Container is complex enough that L2 hides important detail; L4 essentially never (the IDE shows class structure better than any static diagram). Generating L3 + L4 for every service is busy-work that erodes trust in the L1 + L2 (which now look like just-another-diagram).

Canonical vocabulary matters

C4's value depends on cross-team recognisability. When team A uses "Person" and team B uses "Actor" / "User", the visual language drifts and the standardisation benefit dies. Brown's canonical terms — Person / System / Container / Component — are deliberately non-overlapping with common informal usage ("Service" / "App") so they remain unambiguous. Teams that customise the vocabulary lose half the value.

PlantUML vs Structurizr DSL vs Mermaid

Three popular notations:

  • PlantUML (with C4-PlantUML library) — most common, large ecosystem, renders anywhere. Limitation: separate files per view.
  • Structurizr DSL — Brown's own tool. Single DSL file → all 4 views. Best for tight C4 fidelity. Limitation: smaller ecosystem.
  • Mermaid — popular in markdown / docs sites. C4 syntax added in 2022. Limitation: less mature than PlantUML for C4 specifically.

Pick one and stick with it. Mixing creates renderer-incompatibility headaches.

Diagram update enforcement

The diagrams-as-code discipline only works if code and diagram stay in sync. Two enforcement levels:

  1. Soft — PR template asks "did you update the diagram?" with a checkbox. Easy to miss.
  2. Hard — CI check: PRs touching certain directories (e.g., new service folder under services/) require a matching diagram-file change. Fails if missing.

Start with soft, escalate to hard if drift recurs. The hard check has false positives but cures the discipline problem.

Trade-offs — what this practice costs

  • Initial setup. 1–2 days to install C4-PlantUML, write the first L1 + L2, wire CI. One-time.
  • Maintenance. 5–15 min per architecture-touching PR. Real but small.
  • Learning curve. Engineers unfamiliar with C4 need 30 min to learn the four levels + vocabulary. Modest.
  • Tool lock-in. Migrating from PlantUML to Structurizr later is painful (different syntax). Pick once, deliberately.

Lifecycle

pick notation (PlantUML / Structurizr) → draft L1 context → draft L2 container per service → wire CI render → surface in portal → L3 only when L2 hides too much → PR diagram-update check

First setup: pick notation, draft L1 + L2 for current state, wire CI to render. Steady state: every architecture-touching PR updates the relevant diagram. L3 added selectively when the L2 starts to hide important detail.

Common pitfalls

PitfallWhy it happensWhat to do instead
L4 generated for every service "Completeness" instinct. L1 + L2 required; L3 selective; L4 rarely. Over-production erodes trust in the kept diagrams.
Custom vocabulary ("Service", "App") Team's informal terms feel natural. Canonical Person / System / Container / Component. Cross-team recognisability is the whole point.
Diagrams in Lucidchart / Visio Visual editor easier to start with. Diagrams as code from day 1. Editor-only diagrams drift within months.
No PR enforcement "We'll update it later". CI check: architecture-touching PR requires diagram update. Hard rule when soft fails.
Mixed notations across teams One team likes Mermaid; another likes PlantUML. Pick one notation org-wide. Mixed renderers create incompatibility headaches.
Diagrams in a wiki, not the portal Wiki is what's already in use. Surface in the developer portal next to each service. Diagrams that need 3 clicks don't get read.

Worked example

PlantUML · C4 L2

Study App — Lesson player service · L2 container diagram

@startuml lesson-player-container
!include https://raw.githubusercontent.com/plantuml-stdlib/C4-PlantUML/master/C4_Container.puml

title Lesson player · L2 Container diagram

Person(learner, "Learner", "Self-motivated adult learning a new skill")

System_Boundary(c1, "Study App") {
  Container(web, "Web app", "React / Next.js", "Renders the lesson UI in the browser")
  Container(api, "Lesson API", "Go", "Serves lesson content, tracks completion")
  Container(events, "Events service", "Go", "Publishes lesson.completed events")
  ContainerDb(db, "Lesson DB", "PostgreSQL", "Lessons, completions, streaks")
  ContainerQueue(kafka, "Event stream", "Kafka", "Cross-service event bus")
}

System_Ext(cdn, "CDN", "Cloudflare", "Serves lesson video + image assets")
System_Ext(analytics, "Analytics warehouse", "BigQuery", "Receives lesson.completed events")

Rel(learner, web, "Studies lessons", "HTTPS")
Rel(web, api, "Loads lesson + posts completion", "JSON/HTTPS")
Rel(api, db, "Reads + writes", "SQL")
Rel(api, events, "Publishes lesson.completed", "gRPC")
Rel(events, kafka, "Writes events")
Rel(kafka, analytics, "Streams events")
Rel(web, cdn, "Loads assets", "HTTPS")

@enduml

Outcome. The diagram file lives at services/lesson-player/docs/c4-container.puml. CI renders it to SVG on every push. When the team added the events service (Sprint 24), the same PR included the diagram update — adding the Container box and the relationships. Six months later, a new engineer joining the lesson-player squad reads the diagram first; they understand the system in 5 minutes and can immediately ask precise questions (e.g., "why is events a separate Container instead of in the Lesson API?"). Without the diagram, that 5-minute orientation would have been a 2-hour walkthrough with the senior engineer.
