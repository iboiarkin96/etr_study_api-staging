Architecture review — structured change governance

A lightweight, async-first review of significant changes that catches global problems while preserving team autonomy.

TL;DR

Async-first governance with defined scope checklist, reviewer pool, and published SLAs (48 h first feedback, 5 days decision). Triggered by changes that cross service boundaries, add dependencies, or modify shared data. Outcome: Approve / Approve-with-changes / Revise. Catches global issues (security, coupling, data conflicts) at design time when fix cost is minutes; without it, the same issues surface in production at 10–100× cost.

What it is

Architecture review is a structured forum for evaluating significant design changes before they reach production. Three components:

  • Scope checklist — defines when a review is required: new shared dependency, change to a service-to-service contract, data-model change affecting another team, new language / runtime, expected ops cost > $1k/month.
  • Reviewer pool — rotating group of 3–5 senior engineers + architect. For high-impact changes: + security + SRE reps. Pool is published; engineers know who to expect feedback from.
  • Published SLAs — 48 h first feedback, 5 days decision. Without an SLA, reviews become a black hole.

Input: 2-page design doc + checklist (security, observability, privacy, cost, NFRs). Output: Approve / Approve-with-changes / Revise. Reject is rare, reserved for proposals that fail a non-negotiable constraint.

Why we need it

Without review, teams make autonomous decisions that look locally correct but create global problems. Two teams add the same dependency in different versions. Three teams choose different data-partitioning approaches that prevent future migrations. Security gaps obvious to a reviewer ship because no reviewer was involved. Cost to fix surfaces months later at 10–100× design-time cost.

A lightweight process — emphasis on lightweight — catches these at design time. The 48-h async SLA prevents the wait that makes engineers route around the process. The defined scope means most local decisions skip review entirely. Result: faster decisions on local work, better decisions on cross-cutting work.

How others do it

Short tour of how teams known for the practice apply it. Each card is one company, what they do, and what we take from it.

Netflix — async-first reviews

Netflix's architecture reviews are async-first via internal design-doc tooling. Engineers post a 2-page doc; tagged reviewers comment within 24–48 hours; sync meetings only when async fails. The "Netflix culture deck" emphasises high context, low process — the review is lightweight specifically to honour that culture.

What we take: Async-first means most reviews never need a meeting. Reserve sync for stuck threads only — that saves ~80% of meeting time and keeps engineer flow intact.

Source: Netflix Tech Blog

Spotify — squad-of-squads reviews

Spotify's architecture reviews scale via "guilds" — cross-squad groups of senior engineers per domain (data, mobile, web, platform). When a squad's design crosses domains, the relevant guild member is tagged automatically by the design-doc tool. No central architecture board; review is federated.

What we take: Federated review scales better than a central board. Tag-based routing to the relevant senior engineer (by domain) preserves autonomy while ensuring cross-cutting visibility.

Source: Spotify Engineering

Google — readability + design review

Google formalises design review distinct from code review. The design review is conducted by senior staff on a design doc before any code is written; the code review afterward is purely about implementation quality. Google's L6+ engineers are expected to review designs across the company, not just their team.

What we take: Separate design review from code review. Conflating them produces both worse design (rushed) and worse code (reviewed at a level that should have been settled earlier).

Source: Software Engineering at Google

Stripe — published guidelines + design office hours

Stripe publishes engineering principles + API design guidelines that act as the "background standard" for any review. Reviewers cite the guideline rather than personal opinion. Stripe also runs weekly "design office hours" where engineers bring early-stage proposals for informal feedback before submitting a full review.

What we take: Publish principles + guidelines so reviewers cite documents, not opinions. Run weekly office hours so engineers can get early feedback informally — formal review happens only after the design has stabilised.

Source: Stripe — engineering principles

Across teams: async-first, published SLAs, federated routing by domain, design review separate from code review, guidelines cited not opinions. Heavyweight central boards scale poorly; lightweight federated reviews scale linearly.

Anatomy

The parts a complete practice has, with one sentence on each. None of these are optional.

Scope checklist

Trigger criteria documented publicly. Engineer self-checks before opening review. Most changes skip review.

Design doc input

2-page max; Context · Decision · Alternatives · NFR & security checklist. Standardised template.

Reviewer pool

Published list of 3–5 senior engineers + architect. Rotation; nobody is the sole reviewer.

Published SLAs

48 h first feedback, 5 days decision. SLA misses logged at quarterly retro.

Domain-routed reviewers

Tag-based routing to the relevant senior by domain (data, security, SRE, mobile). Federated, not central.

Decision options

Approve / Approve-with-changes / Revise. Reject only for non-negotiable failures. Approve-with-changes is the most common.

Sync meeting as fallback

Only when async stalls (e.g. 5+ comment threads unresolved after 3 days). Time-boxed 30 min.

Linked to ADR

If the review approves a non-trivial architectural choice, an ADR is authored capturing the decision permanently.

How it works in this project

Study App's review trigger: changes that cross service boundaries, introduce new dependencies (libraries, services, SaaS), modify shared data schemas, or expected ops cost > $1k/month. Trigger checklist at governance/architecture-review-scope.md. Engineer opens a PR with a design doc at docs/design/<slug>.md and tags the relevant reviewers.

Reviewer pool published at docs/team/architecture-reviewers.md with rotation. Domain routing: data → @anna, security → @oleg, SRE → external on-call, anything else → architect. SLA: 48 h first feedback, 5 days decision. If async stalls, a 30-min sync is convened.

A short ADR locks the scope-checklist trigger, the SLAs, and the rule that any Approve-with-changes condition becomes a checklist item in the implementation PR.

Deep dive — the theory you need

Why lightweight beats heavyweight

The heavyweight architecture review board pattern (1990s–2000s enterprise: monthly meeting, gatekeeping ARB, mandatory presentation) is dead for a reason. It slows everything down, becomes a status game, and engineers route around it by relabelling changes as "internal refactor". The lightweight pattern works because: (1) most decisions don't need review (defined scope), (2) reviews are async (don't block flow), (3) SLAs prevent black holes, (4) decisions are documented in ADRs (durable). The constraint is not whether to review — it's how to make the review small enough to actually happen.

Decision options — Approve-with-changes is the centre of gravity

Three decisions matter:

  • Approve — design is sound, implement. Rare on first review; most designs improve.
  • Approve-with-changes — design is sound provided X, Y are addressed. Conditions become PR checklist items. Most common outcome.
  • Revise — fundamental issue; back to drawing board. Rare but important to retain as an option; without it Approve-with-changes becomes a rubber stamp.

Reject is reserved for "this violates a non-negotiable constraint" (compliance, security policy). Almost never used; existence of the option keeps the others honest.

Federated routing vs central board

Central ARB: one group reviews everything. Bottlenecks at the ARB's bandwidth. Federated: tag the relevant senior by domain. Scales linearly with team size. The architect is the fallback router when no domain owner is obvious, not the single reviewer. Spotify's guild model is the canonical example; smaller teams use Slack-channel routing (#arch-review with domain tags).

Office hours — the informal pre-review

The formal review is for stable designs. The earlier conversation — "I'm thinking about doing X, do you have 10 minutes?" — is informal but high-value. Weekly office hours (architect available 1–2 hr/week, drop-in) capture this work without bureaucracy. Engineers come with rough sketches; the conversation reshapes the design before the formal doc is written. Catches 70% of fundamental issues before the formal review starts.

The async-first habit

Async-first means: post the design doc; reviewers comment in the doc / PR; engineer responds in-thread; resolution happens in writing. Only when a thread can't resolve in 3 rounds of comments does a sync meeting get scheduled. Async benefits: reviewers process at their own pace; engineer has time to research before responding; the resolution is written down. Sync benefits: faster for genuinely complex / contentious questions. The default is async; the exception is sync.

Trade-offs — what this practice costs

  • Reviewer time. ~2–4 hrs/week per active reviewer. Pool rotation distributes the load.
  • Engineer cycle time. Design-doc-then-review adds 1–2 days vs writing code directly. Pays back at the first cross-team mistake caught at design time.
  • Cultural resistance. Engineers used to "just merge" need to adapt to "design doc first". Show the win at the first averted incident.
  • Process drift. Reviews creep into smaller changes ("just to be safe") and become bureaucracy. Quarterly retro to prune scope-checklist; resist trigger inflation.

Lifecycle

scope checklist self-check → design doc PR → reviewers auto-tagged by domain → async review (48 h) → sync if stuck (30 min) → decision (≤ 5 days) → ADR + impl PR

Engineer self-checks scope; opens design-doc PR; tags reviewers by domain. Async review for 48 h. If stuck after 3 rounds of comments, sync called. Decision within 5 days. Approve-with-changes converts to PR checklist; approved design produces an ADR. Implementation PR cites the ADR.

Common pitfalls

PitfallWhy it happensWhat to do instead
Review becomes a black hole No SLA; reviewers respond when convenient. Publish SLAs (48 h / 5 day). SLA misses logged at quarterly retro. Engineers learn to trust the process only if it has a predictable response time.
Heavyweight central ARB Single group reviews everything; bottlenecks. Federated routing by domain. Senior engineer per area; architect as fallback router, not gatekeeper.
Trigger scope creeps "Better to review than not" — review covers everything. Quarterly retro prunes the trigger list. Most local decisions should NOT require review.
Reviewers cite opinion not policy No published guidelines; reviewer's gut. Publish engineering principles + API guidelines. Reviewers cite documents. Engineers know what to expect.
Design + code review conflated Design issues surface during code review when they're expensive to fix. Separate them. Design review on the doc before code; code review on the implementation. Different scope, different reviewers acceptable.
Sync-first reflex "Let's just hop on a call". Async-first. Sync only when async stalls after 3 rounds. Most reviews resolve without a meeting.

Worked example

Review record

REV-0042 — Adopt Kafka for analytics event stream

# REV-0042 · Adopt Kafka for analytics event stream
status:     APPROVE WITH CHANGES
opened:     2026-05-18  ·  decided: 2026-05-22  ·  decision SLA: 5 days ✓
author:     @oleg
reviewers:  @ivan (architect)  ·  @anna (data)  ·  @maria (security)
scope:      new shared dependency · cross-team data flow · cost > $1k/mo

## Trigger
Analytics needs an event stream to feed both the warehouse and the
recommendation service in real time. Today: a nightly batch via S3.

## Proposal
Adopt Kafka (self-hosted, EU region) as the canonical event stream.

## Reviewer comments

@anna (data) — 2026-05-19:
  + Topic design: align with our domain model (one topic per
    bounded context, not per service)
  ! Schema registry must be in place at launch; missing today
  ? What's the retention policy? 7 days proposed but billing
    events probably need 90

@maria (security) — 2026-05-20:
  ! mTLS for producer/consumer authentication required (not in proposal)
  ! Encryption at rest must use customer-managed KMS keys
  + EU region choice is correct for GDPR

@ivan (architect) — 2026-05-21:
  + Boring-tech: Kafka beats Pulsar / Redpanda for current team
  + Cost analysis is realistic
  ? Have we considered AWS MSK vs self-hosted? Self-hosted is
    fine but write that as the active choice in the ADR

## Decision (2026-05-22)

APPROVE WITH CHANGES. Implementation PR must address:
  C1  Add schema registry from day 1 (Confluent or Karapace)
  C2  mTLS required; document in security review checklist
  C3  Encryption at rest with customer-managed KMS keys
  C4  Retention policy: 7 d default, 90 d for billing topics (justify in ADR)
  C5  Author ADR 0099 documenting "self-hosted Kafka in EU" with
       AWS MSK considered-and-rejected rationale

ADR 0099 to be merged before any production deployment.

## Outcome (post-decision)
Async-only resolution; no sync meeting needed. Total reviewer time:
~3 hours across 3 reviewers. Implementation PR shipped with all
5 checklist items addressed; schema registry caught one breaking
change in the first week of production.

Outcome. The review caught five issues at design time that would each have cost weeks if discovered post-launch: missing schema registry (would have caused a producer/consumer break in week 2), mTLS gap (would have triggered a security audit finding), KMS key issue (would have failed compliance review), retention policy mismatch (would have lost billing audit data), and missing ADR (would have created decision archaeology in 6 months). Total reviewer time: ~3 hours; total cost-of-issues averted: ~6 engineer-weeks. The async format meant no meeting was needed — the resolution lives in writing for future reference.
