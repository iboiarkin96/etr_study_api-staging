Event-driven architecture — decouple through events

Replace synchronous point-to-point calls with immutable domain events to achieve temporal decoupling and resilient integration.

TL;DR

Event-driven architecture (EDA) is a style where services communicate by publishing and consuming immutable domain events rather than calling each other directly. CQRS separates write and read models; Event Sourcing stores the full state history as an ordered log. Together they enable temporal decoupling, free audit trails, and independent service scaling. Netflix, Uber, LinkedIn all run core systems on event-driven backbones. The cost is real (eventual consistency, idempotency, schema discipline), so apply selectively — not as a default for every CRUD endpoint.

What it is

Event-driven architecture is a system-integration style where components communicate by publishing and consuming events. An event is an immutable record of something that happened in the domain: OrderPlaced, PaymentFailed, LessonCompleted. The producer does not know who will consume the event.

Three related patterns that often appear together:

  • Pub-Sub — a broker (Kafka, SQS/SNS, Google Pub/Sub) delivers events from producers to one or more consumer groups. Each consumer group processes every event independently with its own offset.
  • CQRS (Command Query Responsibility Segregation) — separates the write model (commands that change state) from the read model (queries on projections). Greg Young and Martin Fowler popularised it around 2009. Read models are eventually consistent with the write model.
  • Event sourcing — stores every state change as an immutable event in an append-only log. Current state is the result of replaying the log. First described by Fowler in 2005. Enables time travel, audit trails, and replayable migrations.

EDA does not require all three: pub-sub alone is the most common entry point; CQRS adds when read/write patterns diverge; event sourcing only when the audit log or replay genuinely earns its complexity cost.

Why we need it

Synchronous point-to-point calls create two failure modes at startup speed. First, availability coupling: when Service A calls Service B synchronously, A's uptime is bounded by B's uptime. A slow B cascades into a full A outage. Second, team coupling: adding a new consumer of data from Service A requires a contract negotiation with A's team and an API change.

Events eliminate both. A producer publishes to a broker and moves on. Consumers subscribe independently; failure in one consumer does not propagate upstream. The producer team never needs to know about new consumers. The event log also provides a free audit trail — a regulatory requirement that is expensive to retrofit after the fact. The cost is real (eventual consistency, idempotency, schema discipline) so EDA is not the default for every CRUD endpoint — it earns its place where decoupling genuinely matters.

How others do it

LinkedIn — Kafka origin

Apache Kafka was created at LinkedIn around 2010 by Jay Kreps, Neha Narkhede, and Jun Rao to handle LinkedIn's activity feed and operational metrics. The original problem: every team wanted a copy of every event stream and the point-to-point integrations had become unmanageable. Kafka became the central commit log for LinkedIn and was open-sourced through the Apache Foundation in 2011. Jay Kreps's 2013 essay The Log is the canonical articulation of why a distributed commit log is a unifying abstraction.

What we take: A central, durable, replayable log is more useful than its individual consumers realise. Treat the event log as a first-class system, not as a side effect of messaging.

Source: Jay Kreps — The Log

Netflix — Keystone pipeline

Netflix processes over 1 trillion events per day through its Keystone pipeline (Kafka-based). Every user interaction — play, pause, search — is an event. Consumers include the recommendation engine, A/B testing, billing, fraud detection. The entire analytics and personalisation stack is event-driven; no service polls another for state.

What we take: Events as the primary integration primitive from day one, not as a migration from RPC. Designing for replay lets the recommendation engine be retrained on any historical window.

Source: Netflix Tech Blog — Keystone

Uber — Kafka as nervous system

Uber uses Apache Kafka as the central nervous system for its real-time systems. Every trip-state change (requested, matched, started, completed) is published as a domain event. Over 1 million messages per second at peak. The driver-location system, ETA calculation, and surge pricing all consume the same event stream independently. Uber publishes its Kafka patterns via the open-source uReplicator project.

What we take: Make the trip/order lifecycle an event stream from day one. Every business question ("how many active rides at 3 pm?") becomes a stream query rather than an expensive DB aggregation.

Source: Uber Engineering — Kafka use case

Fowler / Young — CQRS & event sourcing

Greg Young formalised CQRS as a pattern in 2009 and popularised event sourcing for financial systems where immutable audit trails are a regulatory requirement. His EventStore is the canonical reference implementation. Martin Fowler codified the patterns on martinfowler.com. The canonical guidance is restraint: "Do not use event sourcing unless you genuinely need the audit log or replay capability — the complexity cost is real."

What we take: Apply event sourcing selectively to core-domain aggregates with regulatory audit requirements. Do not use it as a default persistence mechanism — the query complexity is significant.

Source: Fowler — Event Sourcing

Anatomy of an event-driven system

Domain-event schema

Every event has: eventId (UUID), eventType (namespaced), occurredAt (ISO-8601), correlationId, typed payload. Defined in AsyncAPI 2.x. Breaking changes follow a versioning policy (new topic per major version). Schema registry validates compatibility before deployment.

Event broker

Delivery infrastructure (Kafka, SQS/SNS, Google Pub/Sub). Retention: ≥ 7 days for replay. Consumer groups: each downstream system has its own group so it can replay independently. Partition count sized to expected peak with 2× headroom.

Idempotent consumer

Every consumer must be idempotent — processing the same event twice produces the same result. Implemented via deduplication on eventId with 24-h TTL. Required for at-least-once delivery. Non-idempotent consumers cause duplicate charges, emails, inventory decrements.

Dead-letter queue (DLQ)

Events that fail after 3 retries are routed to a DLQ. DLQ is monitored and alerted. Messages are investigated and replayed manually or by a retry job. Without a DLQ, failed events are silently lost.

Outbox pattern

To guarantee a DB write and an event publish happen atomically, the event is written to an "outbox" table in the same transaction; a relay process publishes it to the broker. Prevents the dual-write problem. Required for any event that drives financial or inventory state.

Schema registry

Central registry (Confluent Schema Registry, AWS Glue) that stores and validates event schemas. CI rejects breaking changes (removed fields, renamed required fields). Backward-compatible additions (new optional fields) are allowed without consumer redeploy.

How to run it

  1. Pick one flow that earns it (Week 1). A flow with ≥ 3 downstream consumers, regulatory audit need, or independent scaling demand. CRUD endpoints with one consumer do not earn EDA — keep them synchronous.
  2. Define event schemas (Week 2). AsyncAPI 2.x. Past-tense names (LessonCompleted, not CompleteLesson). Required fields: eventId, eventType, occurredAt, correlationId, payload. Reviewed by the consumer teams.
  3. Stand up the broker + schema registry (Week 2–3). Kafka / SQS / Pub-Sub with ≥ 7-day retention. Schema registry wired to CI; breaking changes blocked at PR time.
  4. Implement outbox in the producer (Week 3–4). Outbox table + relay. Same transaction as the DB write. No "send event after commit" — that is the dual-write problem dressed up.
  5. Implement consumers idempotently (Week 4–6). Dedup table on eventId. Inject duplicate events in CI to verify idempotency. DLQ wired and alerted.
  6. Run the first replay (Week 6). Replay the topic from the start into a new consumer. Confirm projections match. Replay is the load-bearing capability — verify it works before you need it.
  7. Add consumers without changing the producer (Week 6+). The third and fourth consumer are the proof. If adding them requires producer changes, the design failed somewhere — review.

Deep dive — what makes EDA work

Thin events, not fat events. A common mistake is to put the entire entity state in the event payload so consumers do not have to call back. The result is that every schema change becomes a breaking change and consumers become coupled to the full data model. Publish thin events (IDs and key facts), let consumers fetch detail from the source or maintain their own projections. "Notification + fetch" or "event-carried state transfer" — pick the one that fits the consumer, not the producer.

Idempotency is the consumer's job, always. At-most-once delivery does not exist at scale; at-least-once is the default. Network retries, consumer restarts, broker rebalancing all produce duplicates. Every consumer dedups on eventId with a 24-hour TTL or longer. CI injects duplicate events to verify idempotency. Skip this and you ship duplicate charges to customers.

The outbox pattern solves the dual-write problem. "Write to the DB, then publish the event" is wrong — the second step can fail and the system is now inconsistent. The outbox pattern writes the event to an outbox table in the same transaction as the DB write; a relay publishes the outbox to the broker asynchronously. Producer is now atomically consistent with the broker for the cost of one extra table and one relay process.

Schema discipline is the long-term value. A schema registry that blocks breaking changes at PR time prevents the most common EDA failure: a producer renames a field, three consumers silently break, the team discovers it at month-end when reports are wrong. Backward-compatible changes only — add optional fields, never remove or rename required ones. Major-version breaks live on a new topic; consumers migrate explicitly.

Replay is the test of decoupling. The first replay — pointing a fresh consumer at the beginning of the topic — proves the design works. If the projection matches the live state, the producer is honestly decoupled from the consumer. If not, there is hidden state somewhere the event log does not capture; find it and fix it. Replay capability is also the cheapest disaster-recovery mechanism for projections.

EDA is not free — be selective. The complexity cost (eventual consistency, idempotency, outbox, DLQs, schema registry, operational broker) is real. Apply EDA where decoupling earns it: cross-domain integration, audit-required flows, ≥ 3 consumers, independent scaling. Keep CRUD endpoints synchronous when one team owns both ends and the data does not need to fan out.

Lifecycle of a domain event

Defined

Event named (past tense), schema drafted in AsyncAPI, reviewed by consumer teams, registered in schema registry.

Produced

Producer writes via the outbox pattern. Atomic with the DB write. Relay publishes to the broker; partition key derived from a stable aggregate ID.

Delivered

Broker fans out to each consumer group. Each group has its own offset. At-least-once delivery; ordered within a partition.

Consumed

Consumer dedups on eventId, applies the projection or side effect, commits offset. Failures retried 3× then routed to DLQ.

Replayed

A consumer (new or recovering) is pointed at an earlier offset and reprocesses the topic. Idempotency keeps the projection correct.

Deprecated

Breaking schema change requires a new event type or a new topic version. Old topic kept until all consumers migrate; sunset date published in the schema registry.

Common pitfalls

Pitfall Why it happens What to do instead
Fat events — full entity state in the payload Teams include the entire entity in the event to avoid follow-up calls. Consumers become coupled to the full data model; every schema change is breaking. Publish thin events (IDs + key facts). Consumers fetch detail from the source or maintain their own projections. Notification + fetch beats fat events for evolution.
Non-idempotent consumers — duplicate side effects SWEs assume at-most-once delivery. Network retries and consumer restarts cause duplicate delivery. Duplicate emails, duplicate charges, duplicate inventory decrements follow. Design every consumer for idempotency from day one. Deduplication store keyed on eventId. CI injects duplicate events to verify.
Schema changes break consumers silently Producer adds or renames a field. Consumers in other teams do not know until processing silently fails or produces wrong results. No contract enforcement. Schema registry (Confluent or Glue). Enforce backward-compatible changes only. Separate topics for breaking major versions. CI blocks incompatible PRs.
Dual-write — DB and broker not atomic "Write to DB, then publish event." Second step can fail. System is now inconsistent: a state change exists in the DB but no event was published, or vice versa. Outbox pattern. Event written to outbox table in the same transaction as the DB write. Relay publishes asynchronously. Producer is atomically consistent with the broker.
No DLQ — failures silently lost Consumer fails to process an event. Retries exhausted. Event is dropped. Nobody notices until a customer complains. Every consumer has a DLQ. DLQ monitored and alerted. Messages investigated and replayed by a retry job. DLQ depth is a first-class metric.
EDA applied to every CRUD endpoint as default Team treats EDA as the new RPC. Every endpoint becomes an event. Complexity cost dominates; no decoupling benefit because consumers and producers belong to the same team. Apply selectively. EDA earns its complexity when ≥ 3 consumers, regulatory audit, or independent scaling are real. Keep single-consumer CRUD synchronous.

Worked example — lesson-completed event

An AsyncAPI fragment defining the LessonCompleted event plus the outbox row format. Three consumers subscribe: mastery projector, analytics warehouse, notification service.

# asyncapi/lesson.events.yaml
asyncapi: 2.6.0
info:
  title: Lesson events
  version: 1.0.0
channels:
  lesson.events:
    description: Lesson lifecycle events (past-tense, immutable)
    subscribe:
      message:
        oneOf:
          - $ref: "#/components/messages/LessonCompleted"
          - $ref: "#/components/messages/AttemptScored"

components:
  messages:
    LessonCompleted:
      name: LessonCompleted
      contentType: application/json
      headers:
        type: object
        required: [eventId, eventType, occurredAt, correlationId]
        properties:
          eventId:        { type: string, format: uuid }
          eventType:      { const: "lesson.LessonCompleted.v1" }
          occurredAt:     { type: string, format: date-time }
          correlationId:  { type: string, format: uuid }
      payload:
        type: object
        required: [lessonId, userId, score]
        properties:
          lessonId: { type: string, format: uuid }
          userId:   { type: string, format: uuid }
          score:    { type: number, minimum: 0, maximum: 1 }
          # NOTE: thin event. Full lesson detail fetched by consumers
          # from the lesson API using lessonId. Schema-stable forever.

---
-- producer outbox (same transaction as the lesson-completion write):

INSERT INTO lessons (id, user_id, completed_at, score) VALUES (...);
INSERT INTO outbox (
  event_id, event_type, occurred_at, correlation_id,
  payload_json, status
) VALUES (
  gen_random_uuid(),
  'lesson.LessonCompleted.v1',
  now(),
  :correlation_id,
  jsonb_build_object(
    'lessonId', :lesson_id,
    'userId',   :user_id,
    'score',    :score
  ),
  'PENDING'
);

-- relay process (separate worker):
--   SELECT * FROM outbox WHERE status = 'PENDING' ORDER BY occurred_at;
--   publish to Kafka; UPDATE outbox SET status='PUBLISHED' WHERE id=:id;

The schema is registered before any producer or consumer ships. The outbox row is written in the same transaction as the lesson write — no dual-write window. Three consumer groups subscribe; each dedups on eventId and replays from offset 0 when re-initialised.

Quality bar

  • Every event defined in AsyncAPI and registered. Schema registry blocks breaking changes at PR time. Backward-compatible additions only.
  • Past-tense event names. LessonCompleted, not CompleteLesson. The event records that something happened.
  • Thin events. IDs + key facts only. Consumers fetch detail from the source. Schema stays small.
  • Producer uses the outbox pattern. Event row written in the same transaction as the state change. Relay publishes asynchronously.
  • Every consumer is idempotent. Dedup on eventId with ≥ 24-h TTL. CI verifies via duplicate-event injection.
  • DLQ wired and monitored. 3 retries then DLQ. DLQ depth alerted. Messages investigated and replayed.
  • Replay tested. At least once per topic, a fresh consumer is pointed at offset 0 and the projection is verified to match live state.
  • EDA applied selectively. Synchronous CRUD endpoints remain CRUD. EDA earns its complexity where decoupling is real.
