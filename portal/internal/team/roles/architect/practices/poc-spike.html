Proof of concept & spike — de-risk before committing

Answer architectural unknowns with a focused, time-boxed experiment before committing engineering capacity to the full implementation.

TL;DR

A proof of concept (PoC) answers "can we build it?" — a focused experiment with a clear success criterion, a time box (max 3 days), and an explicit throw-away policy. A spike (Kent Beck, XP 2000) answers "how long will it take?" — a time-boxed exploration with a written output. Both produce a one-page report, not production code. The discipline of "define the question before starting and throw away the code after" prevents PoC code from silently becoming production code — the most common failure mode.

What it is

A proof of concept (PoC) answers one technical feasibility question. Before building a full feature, the architect identifies the riskiest assumption and builds the minimum code needed to validate or invalidate it. The PoC is explicitly not production code: no tests, no error handling, no cleanup. After the question is answered, the code is deleted.

A spike (Kent Beck, Extreme Programming Explained, 2000) answers a different question: "how long will the real implementation take?" It is a time-boxed exploration (max 2 days) of an uncertain area, producing an estimate and a set of findings, not a deliverable. The spike output is a short written document, not code. Code written during a spike is deleted unless it is incidentally production-quality.

Both share the same structure: a question defined before starting, a time box that must not be exceeded, and a written output summarising findings and a recommendation. The difference is the question: a PoC asks can we?, a spike asks how long?.

Why we need it

Architecture mistakes committed to production cost 10–100× more to fix than if they were caught in a 2-day PoC. A team that spends 3 weeks implementing a WebSocket integration before discovering the chosen broker does not support its messaging pattern has wasted 15 engineer-days that a 1-day PoC would have saved.

The throw-away discipline is equally important. PoC code that silently becomes production code is the most common failure mode: the developer finishes the PoC, it works, and the path of least resistance is "this already runs, let's just keep it". Three months later the team is maintaining code with no tests, no error handling, and design choices made purely for speed of exploration. The "delete the code" rule is the most valuable part of the practice.

How others do it

Kent Beck — Spike (XP, 2000)

Kent Beck introduced the Spike in Extreme Programming Explained (2000) as a direct response to uncertain estimates. When the team cannot estimate a story because the technical approach is unknown, a spike is scheduled as a separate story with a fixed time box. The spike output is a revised estimate, not a feature. Beck's rule: "always destroy spike code" — the spike is a learning activity. The spike story is tracked on the board; when the time box expires the spike ends regardless of completeness.

What we take: The spike is an estimation tool, not a "free work" mechanism. Define the question at the start: "What is the estimate for implementing X?" The output is an answer to that specific question.

Source: Agile Alliance — Spikes

Google — spike-then-design-doc

Google uses PoCs extensively as part of its Design Doc process. Before writing a design doc, engineers typically run a 1–2 day technical spike to validate the most uncertain assumption. The spike findings inform the options section of the design doc: "we spiked approach X and found Y, which is why we recommend Z". The spike findings are included as an appendix or inline.

What we take: Link PoC findings to design docs. The PoC provides the experimental evidence that makes the recommendation credible rather than speculative.

Source: Industrial Empathy — Design Docs at Google

Airbnb — honest negative results

Airbnb's infrastructure team uses PoCs to validate migration strategies before committing to full implementation. Their React Native adoption (2016–2018) started with a 2-week PoC before allocating a team to the full migration. The PoC revealed performance characteristics that shaped the migration approach; the migration was ultimately reversed. When the PoC results were later published, they demonstrated that a good PoC produces honest negative results — the PoC should be able to say "this doesn't work" as readily as "this works".

What we take: A PoC must be designed to produce a negative result if the answer is negative. Success criteria defined before the experiment, not after. A PoC that cannot fail is a justification exercise, not a PoC.

Source: Airbnb Engineering — Sunsetting React Native

Amazon — Working Backwards spike

Amazon's Working Backwards process pairs the PR/FAQ document with technical spikes when the feasibility of the proposed customer experience is uncertain. A spike is scheduled to validate the riskiest technical assumption before the PR/FAQ is approved for build. If the spike fails, the PR/FAQ is revised before any team-level capacity is committed. This sequencing — customer-facing narrative first, technical spike second, build third — kept many overconfident projects from absorbing months of capacity.

What we take: Sequence matters. PR/FAQ (or product brief) → spike on the riskiest technical assumption → build. Discovering the technical infeasibility after the team is committed is the expensive mistake.

Source: Working Backwards — Bryar & Carr

Anatomy of a PoC / spike

Success criterion

A specific, binary, measurable condition defined before the PoC starts. "The system can handle 10 k concurrent WebSocket connections on a t3.medium" is valid. "The approach seems feasible" is not. If the criterion cannot be defined before starting, the question is not precise enough — iterate on the question first.

Time box

A hard limit: max 1 day for a spike, max 3 days for a PoC. When the time box expires, the experiment ends regardless of how close it is to a result. Exceeding the box without explicit architect sign-off is not permitted. A PoC that needs more time is either asking the wrong question or under-scoped — both are signals.

One-page report

The required output of every PoC and spike. Sections: (1) question asked, (2) approach taken, (3) findings, (4) recommendation (proceed / abort / re-spike). Max 1 page. Stored in the design-docs folder; referenced by the ADR if the PoC informed an architecture decision.

Throw-away policy

PoC code tagged as throw-away: top-level # POC: DO NOT MERGE comment in the main file, a separate git branch, a ticket to delete the branch after the report is written. No exceptions. Production implementation starts from scratch, informed by the PoC. Policy enforced in code review.

Named owner + reviewer

One owner runs the PoC; one reviewer (usually the architect) signs off on the question, the time box, and the report. Without a named reviewer, success criteria drift and time boxes slip.

Link to design doc / ADR

If the PoC informs an architecture decision, the resulting design doc or ADR cites the report. Cross-links are bidirectional. Future readers can see what experimental evidence backed the decision; the PoC stays useful long after the code is gone.

How to run it

  1. Surface the riskiest unknown (Day 0). Architect + tech lead identify the one assumption that, if wrong, blocks the whole approach. PoC the riskiest unknown first; everything else is a distraction.
  2. Write the question + success criterion. One sentence each. "Can the chosen broker sustain 10 k connections at < 200 ms p99 on a t3.medium?" — yes/no with the measurement defined.
  3. Pick the time box. 1 day for spike, 1–3 days for PoC. Recorded in the ticket. Architect reviews before work begins.
  4. Branch + mark throw-away. Separate poc/<slug> branch with a # POC: DO NOT MERGE header. Ticket has a "delete branch" subtask.
  5. Build the minimum. Skip tests, error handling, polish, configuration. The code must answer the one question — nothing more. Time spent on cleanup is time stolen from learning.
  6. Write the report (last hour). Question, approach, findings, recommendation. One page. Stored next to design docs. Reviewer signs off.
  7. Decide, then delete. Proceed: open a ticket for the production implementation that starts from scratch and links the PoC report. Abort: write what would change the answer in 6 months. Either way, delete the branch.

Deep dive — what makes a PoC actually de-risk

Question first or no PoC. The PoC begins only after the question and the success criterion are written down. A PoC without a question is exploratory tinkering, which has its place but is not a de-risking activity. The single most common failure mode is writing the criterion after the result is in: confirmation bias dressed up as evidence. Architect reviews the criterion before any code is written.

Time box is real. The time box exists to force the team to confront which question is most worth answering. When the box expires, the experiment ends — whether or not the question is answered. A PoC that needs three more days has either asked the wrong question (too big) or revealed something useful about the complexity (also valuable). Exceeding the box without explicit architect sign-off is not allowed.

Throw-away is the load-bearing rule. Without the delete step, "PoC" silently means "untested production code". The branch is named poc/*; the first file carries # POC: DO NOT MERGE; the ticket has a "delete branch" subtask. The production implementation starts fresh, informed by the report. The discipline is unpopular and non-negotiable — the cost of skipping it is exactly the cost the PoC was meant to avoid.

Design for honest negative results. A PoC that cannot fail is theatre. The success criterion is binary and the experiment must be capable of producing "no". Airbnb's React Native PoC famously produced honest negatives that shaped the eventual decision — the PoC earned its keep by being able to say "this doesn't work here". Optimism in PoC framing is the architect's blind spot; force the criterion to be falsifiable.

One PoC, one question. A PoC that tries to answer three questions usually answers none of them well. If the team has three risks, run three small PoCs in sequence — riskiest first. Each one gates the next. Bundling risks into one big PoC turns it into a mini-project, which then resists deletion.

Spikes are estimation, not implementation. Beck's original framing remains right: the spike output is a revised estimate plus what was learned. Spikes that produce code which "just happens to be production-ready" almost always merge that code anyway. Keep the question shape distinct: a spike answers how long?; a PoC answers can we?. Don't conflate them or both lose their meaning.

Link the report to the ADR. The PoC's long-term value is the report, not the code. When the resulting ADR is written, cite the PoC report and link both ways. Six months later a reader sees both the decision and the experimental evidence behind it; the decision is defendable instead of arbitrary.

Lifecycle of a PoC

Proposed

Riskiest unknown identified. Question and success criterion drafted. Time box proposed. Architect reviews before work begins.

Approved

Architect signs off on question, criterion, time box. Branch created with # POC: DO NOT MERGE. Ticket has delete-branch subtask.

Running

Owner builds the minimum code to answer the question. No tests, no polish. Daily check-in with architect if multi-day.

Reported

One-page report written: question · approach · findings · recommendation. Reviewer signs off. Report stored next to design docs.

Decided

Proceed (open production-impl ticket linking the report; write ADR), abort (record what would change the answer), or re-spike (refine the question).

Deleted

Branch deleted. Report archived as the durable artefact. Production implementation starts fresh, citing the report. The throw-away closes the loop.

Common pitfalls

Pitfall Why it happens What to do instead
PoC code shipped to production PoC works. Shipping the code is faster than rewriting it. Deadline pressure. Three months later the team is maintaining untested, unstructured code nobody dares refactor. Enforce throw-away in the PR process. PoC branches never merge to main. If the answer is yes, open a new ticket for production implementation and link the PoC report.
Success criterion defined after the experiment SWE runs the experiment, sees a result they like, then writes the criterion to match. The PoC justifies a decision already made. Write the criterion in the ticket before the PoC starts. Architect reviews. Changing the criterion mid-flight requires explicit justification recorded in the report.
Spike used to avoid estimation instead of enabling it Team adds a spike when uncomfortable with an estimate. Spike becomes a way to defer the estimation conversation indefinitely. Every spike must define its output: "After this spike, we will provide a story-point estimate for X". If the spike cannot produce an estimate, the report explains what specific uncertainty remains.
Time box exceeded silently Owner is "almost done". Day 4 becomes day 7. By the time it ends, the PoC has consumed a week of capacity with no decision. Hard stop at the time box. Architect sign-off required to extend. Exceeding the box without sign-off ends the experiment with a "needs re-scoping" report.
No report written — only code Owner finishes the PoC, demos it in a call, never writes the report. Six months later nobody remembers what was tried or why. Report is a deliverable, not a nice-to-have. PoC is not complete until the report is signed off. Demo + chat is not a substitute for the written page.
One PoC bundling three questions Team thinks one big experiment is more efficient. In practice each question contaminates the others; nothing is conclusively answered. One PoC, one question. Three risks = three small PoCs in sequence, riskiest first. Each gates the next.

Worked example — Kafka vs Redis Streams for lesson events

A representative one-page PoC report for a real architectural unknown: which broker handles our planned event volume cheaply enough. Lives at docs/poc/2026-05-broker-choice.md.

# POC-014 · Kafka vs Redis Streams for lesson.events

owner:      @oleg
reviewer:   @ivan (architect)
time-box:   3 days (2026-05-19 → 2026-05-21)
status:     COMPLETE · proceed with Kafka (managed MSK)
branch:     poc/broker-choice  (deleted 2026-05-22)

## Question
Can Redis Streams sustain 5 000 events/s with p99 publish < 20 ms
and 7-day retention on a single t3.medium-class instance, OR do we
need Kafka (managed MSK) from day 1?

## Success criterion (binary)
- Redis Streams: sustains 5 000 evt/s, p99 ≤ 20 ms, 7-day retention,
  < $200/mo  →  proceed with Redis Streams
- Anything less  →  proceed with managed Kafka (MSK)

## Approach
- single t3.medium Redis 7 instance
- 1 producer (k6 custom script), 3 consumer groups
- ramp 1 k → 5 k → 7 k evt/s; measure p99 + memory + disk
- 24-h soak at 5 k evt/s

## Findings
- p99 publish at 5 000 evt/s: 14 ms (PASS)
- memory at 7-day retention: 18 GB (FAIL — t3.medium has 4 GB;
  needs r5.xlarge at ~ $260/mo, exceeds budget)
- consumer group rebalance under load: 8 s of duplicates per rebalance
  (concerns idempotency budget)
- MSK quote for equivalent: ~ $220/mo + ops support

## Recommendation
PROCEED with managed Kafka (MSK).
Reasons: (1) memory cost on Redis exceeds budget at retention target;
(2) rebalance behaviour creates idempotency stress; (3) MSK price
is comparable and gets us the schema-registry ecosystem.

## Next step
- open ticket: ARCH-241  implement lesson.events on MSK
- write ADR-0098 citing this report
- delete poc/broker-choice branch

One page. Binary criterion declared before the experiment. Honest negative result on Redis. Recommendation links to the next ticket and the ADR-to-be. The branch is deleted; the report is the durable artefact.

Quality bar

  • Question and success criterion written before code. Reviewed by the architect; binary and falsifiable.
  • Time box ≤ 3 days for PoC, ≤ 1 day for spike. Hard stop. Architect sign-off required to extend.
  • One PoC, one question. Multiple risks = multiple small PoCs in sequence; never one bundled experiment.
  • Throw-away policy enforced. poc/* branch, # POC: DO NOT MERGE header, delete-branch subtask on the ticket.
  • One-page report written. Question, approach, findings, recommendation. Signed off by the reviewer; stored next to design docs.
  • Linked to ADR / design doc. When the PoC informs a decision, both link to the report. The report stays useful after the code is gone.
  • Honest negative results encouraged. A PoC that says "no" is as valuable as one that says "yes". Coach for falsifiability in framing.
