RFC process — collaborative design before code

A written proposal open for structured comment from across the organisation before a significant decision is accepted.

TL;DR

An RFC (Request for Comments) is a written proposal for a significant technical decision, opened for structured commentary before acceptance or rejection. The format originates from IETF's 1969 series of internet standards and is now used by Rust, React, Node.js, Kubernetes. The RFC separates "what should we do" (RFC) from "here is what we decided" (ADR). Without an RFC process, significant changes get decided in hallway conversations and critical concerns surface only after implementation begins — when feedback is most expensive.

What it is

An RFC is a written proposal for a significant change, opened for structured commentary before it is accepted or rejected. The format traces back to 1969, when Steve Crocker wrote RFC 1 for ARPANET. The IETF still publishes internet standards as RFCs. In software engineering, the term has been adopted by major projects (Rust, React, Node.js, Kubernetes) as a way to make significant design decisions transparent and collaborative.

A typical RFC contains:

  • Summary — a one-paragraph TL;DR for readers who will not read the full document.
  • Motivation — why are we proposing this? What problem does it solve?
  • Proposed solution — the design in enough detail to evaluate it.
  • Alternatives considered — two or three other approaches with their trade-offs.
  • Open questions — what the author wants feedback on specifically.
  • Risks & unresolved issues — what could go wrong; what is intentionally left for later.

The comment window (typically 1–2 weeks) is async-first — feedback is posted as inline PR comments or in a linked discussion. After the window closes, the author revises and the decision is made. The accepted RFC is archived; the resulting decision is recorded as an ADR.

Why we need it

Without an RFC process, significant decisions get made in hallway conversations or Slack threads. Engineers across timezones miss the discussion. Critical concerns surface only after implementation begins, when feedback is most expensive to incorporate. Decisions feel arbitrary to the engineers who were not in the room.

An RFC fixes both problems. The written proposal forces precision — writing makes vagueness visible. The async commentary window includes engineers across timezones who would otherwise be excluded. By the time implementation starts, the major design questions have been debated and resolved in writing. The RFC trail also creates institutional memory: a year later, the team can understand why a design was chosen and what was considered.

How others do it

Rust — the canonical OSS RFC

The Rust project's RFC process, started in 2014, is the canonical example for open-source. RFCs live in github.com/rust-lang/rfcs as Markdown files; a new RFC is a PR. The comment window is open-ended but typically 7–14 days. A Final Comment Period (FCP) gives one week's warning before merge. The Rust core team makes the final call. Every major Rust feature since 1.0 has an RFC. The process is used by 700+ contributors and has produced 3000+ RFCs.

What we take: The Final Comment Period mechanic is high-value. After active discussion ends, give one week's warning before closing the RFC. The week catches stakeholders who missed the active discussion.

Source: github.com/rust-lang/rfcs

React — community RFC

The React team adopted an RFC process in 2018 after the Hooks proposal was met with both enthusiasm and concern. RFCs are published at github.com/reactjs/rfcs and opened for community comment before features are added to React. Hooks (RFC 0068), Concurrent Mode (RFC 0118), and Server Components (RFC 0188) all went through public RFC commentary. The React team treats RFCs as the primary mechanism for community input on framework direction.

What we take: An RFC process is also a community-management tool. Opening proposals for comment generates engagement and surfaces concerns the core team cannot anticipate alone.

Source: github.com/reactjs/rfcs

Uber — Engineering Standards Council

Uber's internal RFC process, called Engineering Standards Council (ESC), reviews company-wide proposals affecting multiple teams. RFCs are required for: new shared infrastructure, new programming-language adoption, changes to core service contracts, and any decision affecting more than 3 teams. The ESC includes principal engineers from each platform area. Each RFC has a stated "blast radius" describing which teams are affected and which need to vote.

What we take: Define the "blast radius" of each RFC at the start. The reviewer list is derived from the blast radius. Avoids over-soliciting from teams that are not affected and under-soliciting from teams that are.

Source: Uber Engineering — decision-making at scale

Python — PEP process

Python Enhancement Proposals (PEPs), introduced in 2000, are Python's RFC equivalent and the longest-running formal RFC process in software. PEPs have explicit states (Draft / Active / Accepted / Rejected / Superseded / Withdrawn) and a "BDFL-Delegate" or Steering Council acts as the final decision-maker. PEP 1 itself defines the process — meta-RFC. Major language features (typing, async, walrus operator) all went through PEP review.

What we take: Explicit status states (Draft / Active / Accepted / Rejected / Superseded / Withdrawn) make the RFC lifecycle visible. A reader can scan the index and see the state of every proposal without opening it.

Source: PEP 1 — meta-PEP

Anatomy of an RFC

Scope trigger

Defines when an RFC is required vs just an ADR. Typical triggers: affects more than 2 teams, introduces a new programming language or framework, changes a public API contract, or requires more than 4 engineer-weeks. Below the bar, an ADR is sufficient.

Required sections

Summary, Motivation, Proposed Solution, Alternatives Considered, Open Questions, Risks. The Alternatives section is non-negotiable — an RFC that proposes only one solution is a decision in disguise. At least two alternatives must be evaluated with their trade-offs.

Blast radius

Explicit statement of which teams and which surfaces this RFC affects. Derives the reviewer list. Without a stated blast radius, RFCs are over-broadcast or under-broadcast — neither produces good feedback.

Comment window

A defined period (7–14 days) during which comments are collected. Async-first — feedback as PR comments or in a linked discussion. Sync review meetings are supplementary. Window has a published end date so contributors know when feedback must arrive.

Final Comment Period (FCP)

A 1-week warning before close. After active discussion ends, the author labels the RFC "FCP" with the planned closing date. Catches stakeholders who were unavailable during active discussion. Without an FCP, decisions feel rushed to engineers on PTO during the comment window.

Decision + ADR

After FCP, the decision-maker (architect, tech lead, or council) accepts or rejects with a written rationale. Accepted RFC archived to the RFCs repo. The resulting decision recorded as an ADR. Both link to each other — the trail preserves debate + decision.

How to run it

  1. Verify the trigger. Cross-team change, new language/framework, public API shift, > 4 engineer-weeks, or hard-to-reverse decision. Below the bar, write an ADR and skip the RFC.
  2. Draft (1–3 days). Use the canonical template. Summary + Motivation + Proposed Solution + ≥ 2 Alternatives + Open Questions + Risks. Cap body at ~ 8 pages.
  3. State the blast radius. Which teams, which surfaces, which approvals required. Derive the named reviewer list from the blast radius.
  4. Open the RFC PR with a comment-window deadline. Publish dates: open / FCP-target / close. Tag named reviewers; announce in the engineering channel.
  5. Active comment window (7–14 days). Async-first. Author responds in-thread; revisions edited into the doc. Sync only on stuck threads.
  6. Enter Final Comment Period (7 days). Label the RFC "FCP". Re-broadcast in the channel. No major design changes during FCP — only clarifications and resolution of remaining concerns.
  7. Decide. Architect / tech lead / council records the decision in writing. Accept (write the ADR; cite the RFC), reject (write the reasoning), or postpone (state what would need to change).
  8. Archive and link. Accepted RFC stays in the repo with status "Accepted". ADR cites the RFC; RFC cites the ADR. Both surface in the design-docs index.

Deep dive — what makes the RFC actually work

Scope trigger keeps the process credible. The single biggest failure mode is over-application — running an RFC for a decision that could have been an ADR. The overhead becomes a tax on small improvements and engineers stop engaging. Define the trigger explicitly: cross-team, new language, public API, hard-to-reverse. Most decisions should be ADRs, not RFCs. Reserving RFCs for the high-blast-radius minority is what makes the format land when it lands.

Two real alternatives, every time. An RFC with one real option and a straw-man is a decision in disguise. The Alternatives section is the load-bearing artefact: it is where reviewers see what was considered and what was rejected. Without genuine alternatives, the comment window collects compliments instead of analysis. If only one option was seriously considered, write an ADR and be honest about it.

Async is the timezone tax. Sync RFC meetings systematically exclude every contributor not in the meeting timezone. Even one timezone exclusion produces resentment that lasts longer than the decision. Async-first means the resolution lives in writing, all reviewers see the full thread, and contribution does not require being in the right room at 9 am. Sync supplements async only when async threads stall after 3 rounds.

FCP catches the PTO stakeholders. Active discussion runs hot for 5–10 days then slows down. Without an explicit Final Comment Period, the RFC closes while the senior engineer who was on PTO never saw the active phase. The FCP — 7 days, broadcast, no major changes — pulls them back in for a last review. The week feels slow at the time and pays for itself the first time a senior reviewer raises a load-bearing concern in the final days.

The decision-maker is named, not implied. RFCs without a clear decision-maker drift forever. State up front: who accepts or rejects? Architect for technical scope; council for cross-platform; team tech lead for narrower scope. The decision-maker also has the authority to close the RFC after FCP regardless of unresolved concerns — they convert to Open Questions in the ADR.

Bidirectional linking with the ADR. The RFC archives the debate; the ADR archives the decision. Without the link, the ADR reader cannot see the alternatives and the RFC reader cannot see what was finally chosen. Make the link a hard requirement of "accepted" status. A year later, both artefacts are still useful.

Templates do not save bad RFCs. A canonical template helps a good author and does not rescue a bad one. The author's job is to do the actual analysis — alternatives, trade-offs, honest risks. A template can enforce the sections but cannot enforce the thinking. Coach authors; don't rely on the template alone.

Lifecycle of an RFC

Draft

Author writes the first version. Not yet circulated. Used for office-hours discussion or early review with the architect.

Open for comments

Distributed; named reviewers + open channel. 7–14 day async window. Author responds in-thread; revisions edited into the doc.

Final Comment Period

Labelled "FCP" for 7 days. Re-broadcast. No major design changes during FCP — only clarifications and resolution of remaining concerns.

Accepted

Decision-maker signs off. RFC archived as "Accepted". ADR authored citing the RFC. Implementation can start.

Rejected / Postponed

Decision-maker declines with a written reason. RFC kept with status "Rejected" or "Postponed"; postponed RFCs state what would change the answer.

Superseded

A later RFC replaces this one. Original kept with a "Superseded by" link; ADRs of both RFCs cross-reference each other.

Common pitfalls

Pitfall Why it happens What to do instead
RFC process applied to trivial decisions Teams over-apply the process. Every small choice goes through a 2-week RFC. Overhead becomes a tax on small improvements; process loses credibility. Define the scope trigger explicitly. Below the trigger, use an ADR (no comment window). RFCs reserved for cross-team, high-blast-radius, or hard-to-reverse decisions.
Endless RFCs — no convergence No defined window or FCP. Discussion continues indefinitely. Proposal becomes a "perpetual draft". Engineers stop engaging because nothing resolves. Window deadline set at opening. Explicit FCP. Decision-maker closes the RFC after FCP regardless of whether every concern is resolved; remaining ones become Open Questions in the ADR.
Single-option RFC — proposal disguised as RFC Author already knows what they want. Alternatives section is perfunctory. Comment window is theatre — decision was already made. Reject RFCs without genuine alternatives. Alternatives section must include two or more options the author seriously considered. If only one, write an ADR and be honest.
Sync-first reviews exclude timezones Author calls a 60-minute review meeting. Half the named reviewers are asleep or on PTO. Decisions feel arbitrary to those who were not in the room. Async-first. Sync only when threads stall after 3 rounds. Resolution lives in writing; everyone sees the full thread regardless of their timezone.
No named decision-maker RFC drifts because nobody is empowered to close it. Comments accumulate; no decision arrives. State the decision-maker at RFC opening. Architect / tech lead / council depending on blast radius. Decision-maker has authority to close after FCP.
RFC accepted but no ADR written Acceptance feels like the end. ADR never written. Six months later the decision is undocumented and the implementation seems arbitrary. "Accepted" requires the ADR. Bidirectional link mandatory. Both archived together. ADR is the durable artefact; RFC is the analysis behind it.

Worked example — RFC header for "Adopt MSK for lesson events"

A representative RFC opening for the Kafka-vs-Redis decision (sibling to the PoC report). Lives at docs/rfc/0042-msk-lesson-events.md; body cut for brevity.

# RFC-0042 · Adopt managed Kafka (MSK) for lesson events

status:        FCP  (closes 2026-06-02)
authors:       @oleg, @anna
decision-maker: @ivan (architect)
blast-radius:  lesson-player, analytics, mastery-projector,
               notifications team, SRE on-call rotation
reviewers:     @ivan (arch) · @anna (data) · @maria (security)
               · @sasha (mobile) · @sre-lead
opened:        2026-05-18
comment close: 2026-05-26
fcp end:       2026-06-02
links:         supersedes RFC-0017 (Redis Streams trial),
               PoC report POC-014 (broker choice)

## Summary
Adopt managed Kafka (AWS MSK) as the lesson-events backbone for the
next 12 months. Schema registry, 7-day retention, EU region.

## Motivation
Lesson player must fan out events to mastery, analytics, and
notifications. PoC-014 showed Redis Streams memory cost exceeds
budget at our retention target; rebalance behaviour creates
idempotency stress. Need a durable broker with replay.

## Proposed solution
Managed Kafka (MSK) in EU, 3-broker cluster, schema registry on
Confluent. Producers via outbox pattern; consumers per group with
dedup. Detail in body (8 pp).

## Alternatives considered
1. Redis Streams self-hosted  — fails PoC; memory cost & rebalance.
2. AWS Kinesis                — vendor lock-in; replay ergonomics
   poor; no schema registry equivalent.
3. Self-hosted Kafka (no MSK) — operational cost outweighs savings
   at our scale; SRE team capacity insufficient.

## Open questions
- Cost-allocation tags by team — owner: SRE
- Long-term retention > 7 d for billing topics — owner: data
- Disaster-recovery RPO / RTO — owner: SRE

## Risks
- Vendor lock-in to AWS MSK API surface (low — Kafka API is portable)
- Schema-registry availability dependency (mitigate: cached schemas)

## Decision (TBD on 2026-06-02)
... to be filled in by @ivan after FCP closes.
... resulting ADR-0098 will be authored and linked.

RFC header carries blast radius, named decision-maker, comment-window dates, explicit FCP target, and link back to the PoC report. Three real alternatives. The Decision section stays empty until FCP closes — and then the ADR is authored and both link to each other.

Quality bar

  • Scope trigger documented. Cross-team, new language/framework, public API shift, > 4 engineer-weeks, hard-to-reverse. Below threshold = ADR only.
  • At least two genuine alternatives. Reviewed by the architect. Straw-men rejected; single-option RFCs converted to ADRs.
  • Blast radius stated up front. Affected teams + surfaces. Reviewer list derives from blast radius — not "@engineering".
  • Async-first comment window 7–14 days. Sync only when async stalls after 3 rounds.
  • Explicit Final Comment Period (7 days). Re-broadcast at FCP start. No major design changes during FCP.
  • Named decision-maker. Architect / tech lead / council. Authority to close after FCP regardless of unresolved concerns.
  • Accepted RFCs require an ADR. Bidirectional link. RFC archives the debate; ADR archives the decision.
  • Status visible at the index. Draft / Open / FCP / Accepted / Rejected / Postponed / Superseded — scannable without opening the doc.
