Hexagonal architecture — isolate the domain core

Separate the pure domain model from all external dependencies so the core is independently testable and technology-agnostic.

TL;DR

Hexagonal architecture (Ports & Adapters), coined by Alistair Cockburn in 2005, separates the domain core from all external dependencies — databases, HTTP clients, message brokers, frameworks. The domain defines interfaces (ports); technology-specific implementations (adapters) plug in. The result is a domain that can be tested with no infrastructure (sub-500 ms suites), migrated to a new database in hours, and understood without reading framework code.

What it is

Hexagonal architecture (also called Ports & Adapters) is a structural pattern introduced by Alistair Cockburn in 2005. Its central rule: the application core (domain model + use cases) has no dependencies on external infrastructure. All dependencies point inward, toward the domain.

Two sides of the hexagon:

  • Driving side (primary) — the actors that initiate use cases: HTTP controllers, CLI commands, message consumers. They call the domain via use-case interfaces.
  • Driven side (secondary) — the infrastructure the domain needs: databases, external APIs, message brokers. They implement the domain's repository or notification ports.

The domain defines the ports (interfaces). Adapters implement the ports. The domain never imports an adapter. This inversion of control means the domain can be tested with in-memory fake adapters, and migrating from one database to another requires only writing a new adapter — domain code does not change.

Related patterns: Clean architecture (Robert Martin) and Onion architecture (Jeffrey Palermo) are the same idea with slightly different vocabulary. The dependency rule — outer layers depend on inner layers, never the reverse — is shared across all three.

Why we need it

When domain logic is entangled with Spring annotations, SQLAlchemy models, or Kafka consumers, every test requires a running database and a running broker. The test suite takes 5 minutes instead of 5 seconds. Database migrations require updating domain code. Framework upgrades touch business logic. Accidental complexity of the infrastructure invades the domain and makes it fragile.

Hexagonal architecture makes the domain a clean, independently testable unit. A domain test suite with fake adapters runs in under 500 ms with no I/O. The team can swap the database (Postgres → DynamoDB) or the framework (Flask → FastAPI) without rewriting a single use case. At startup speed, this kind of reversibility is worth the initial investment.

How others do it

Alistair Cockburn — originator (2005)

Published the original pattern in 2005 at alistair.cockburn.us. Motivating example: a system that must be driven by both a GUI and a test suite, and that must work with both a real database and an in-memory fake. The hexagonal shape is arbitrary — the pattern has nothing to do with six; the hexagon just communicates "many equal ports" better than a rectangle with two sides.

What we take: The most valuable application is making the domain unit-testable without infrastructure. Write domain tests first; the architecture follows from the testing discipline.

Source: alistair.cockburn.us — Hexagonal Architecture

Robert Martin — Clean architecture

"Uncle Bob" formalised the dependency rule in Clean Architecture (2017): source-code dependencies must point inward, from outer layers (frameworks, drivers) to inner layers (use cases, entities). The famous concentric-circles diagram is functionally equivalent to Cockburn's hexagon. Clean Architecture is the version most engineers encounter first; the vocabulary differs but the rule is the same.

What we take: The dependency rule is the load-bearing idea. Pick one vocabulary (hexagonal, clean, onion) and stick with it — the team needs one shared mental model, not three.

Source: Robert Martin — Clean Architecture

Stripe — pure core, effectful shell

Stripe's payment-processing core is designed with clear separation between domain logic and infrastructure adapters. Charge-creation logic has no direct dependencies on persistence — it operates on domain objects and emits events. Infrastructure adapters handle persistence and external API calls. Stripe tests the entire charge lifecycle in under 100 ms per test with no external calls. Their engineering blog calls this "pure core, effectful shell".

What we take: "Pure core, effectful shell" — all side effects (I/O, external calls) at the boundary; pure domain logic at the centre. The core becomes trivially testable.

Source: Stripe engineering principles

Netflix — DGS module boundaries

Netflix's Domain Graph Service (DGS) framework enforces a ports-and-adapters structure for GraphQL services. The domain layer defines data-fetcher interfaces (ports); infrastructure adapters implement them using databases, REST services, or caches. The framework's module structure prevents adapters from importing domain internals — enforced at the Maven module boundary. DGS is open-source at github.com/Netflix/dgs-framework.

What we take: Enforce the architecture at the module/package level, not just by convention. Boundaries that prevent wrong-direction imports keep the pattern maintainable as the team grows.

Source: Netflix DGS Framework

Anatomy of the hexagon

Domain model

Pure domain classes: aggregates, entities, value objects, domain events. Zero imports from frameworks, ORMs, or HTTP libraries. Enforced by an ArchUnit / Dependency Cruiser fitness function. Domain module has zero transitive deps on spring-web, SQLAlchemy, kafka-python.

Use case (application service)

Orchestrates domain objects for one business scenario (CompleteLesson, ScoreAttempt). Accepts input ports; returns output. No HTTP status codes, no DB queries. Each use case has one public method; ≤ 50 lines is the smell threshold.

Port (interface)

An interface defined in the domain that describes what the domain needs from outside. Repository ports (find, save), notification ports (notify), event ports (publish) are the most common. Named from the domain perspective: LessonRepository, not PostgresLessonDAO.

Adapter (implementation)

Technology-specific implementation of a port. Lives in a separate module outside the domain. Examples: PostgresLessonRepository implementing LessonRepository; KafkaEventPublisher implementing EventPublisher. Adapters import the domain; the domain never imports adapters.

Fake adapter (test double)

In-memory implementation of a port used in domain tests. InMemoryLessonRepository stores lessons in a map. Enables the domain test suite to run in < 500 ms with no I/O. Must implement the full port interface; must not skip validation.

Composition root

One place (main / bootstrap) where concrete adapters are wired to ports. The only file that imports both the domain and every adapter. Configuration changes happen here, not in domain code. Keeps the rest of the codebase ignorant of which adapter is in play.

How to run it

  1. Carve modules (Week 1). Three packages: domain/, adapters/, app/ (composition root). Module boundaries enforced by build config — domain module declares no dependency on adapters.
  2. Move domain into domain/ (Week 1–2). Pure classes only. Strip framework annotations from domain classes; define separate persistence models in the adapter.
  3. Define ports (Week 2). Repository, EventPublisher, Notifier interfaces in domain/ports/. Named from the domain side, not the adapter side.
  4. Implement adapters (Week 2–4). PostgresLessonRepository, KafkaEventPublisher, etc. in adapters/. Adapters depend on the domain; the domain has no idea they exist.
  5. Add fake adapters for tests (Week 3). InMemoryLessonRepository implementing the same port. Domain unit tests use fakes; integration tests use real adapters in a separate test tier.
  6. Wire composition root (Week 4). services/api/app/main.py (or equivalent) is the one place that imports both domain and concrete adapters. Adapter selection by environment variable or config.
  7. Lock with a fitness function (Week 4). ArchUnit / Dependency Cruiser rule blocks any commit that imports an adapter from the domain. The rule is what keeps the architecture from drifting back.

Deep dive — what makes the pattern earn its keep

The dependency rule is the entire pattern. Hexagonal, Clean, and Onion architectures all reduce to one rule: source dependencies point inward, never outward. The domain knows nothing of the adapter; the adapter implements the domain's interface. Get this rule right and the rest follows; violate it and no amount of folder structure will help. Enforce the rule with a fitness function — convention drifts; CI does not.

Ports are domain artefacts, not infrastructure ones. A port's method signature uses only domain types. List[Lesson] is valid; QueryResult or a Pandas DataFrame is not. Naming follows the domain: LessonRepository, not PostgresLessonDAO. When the port leaks adapter vocabulary, the abstraction has failed and the domain is back to being coupled to a specific technology.

One use case, one public method. Use cases that grow 5+ port dependencies and 200+ lines are god services in disguise. Cap each use case at one public method and ≤ 50 lines; split when it gets larger. Smaller use cases compose better, test faster, and read like the business they describe.

Fakes earn the test speed. The headline benefit — sub-500 ms domain test suites — only materialises if fakes implement the full port contract. A fake that skips validation produces tests that pass while the real adapter fails. Treat the fake like a sibling adapter: same contract, same edge cases, same care.

Persistence models are not domain models. The biggest implementation mistake is annotating domain classes with @Entity or @Column. The convenience saves a mapper but couples the domain to the ORM forever. Keep persistence models separate; let the adapter own the mapping. The mapper is cheap; the coupling is not.

Composition root is the seam. One file imports both the domain and every concrete adapter. Adapter selection happens here, via config or DI. Everything else stays ignorant of which adapter is wired. Migration to a new database becomes "write a new adapter, change one line in composition". Without a single composition root, adapter swaps require touching every consumer.

Apply selectively — overkill for CRUD. A 200-line CRUD admin page does not need a hexagon. Hexagonal architecture pays off in the core domain where business logic is non-trivial and replaceability matters. Don't pattern-match: classify the subdomain (core / supporting / generic) first; apply the hexagon to the core only.

Lifecycle of a port

Defined

Interface declared in domain/ports/. Method signatures use domain types only. Documented with a one-line purpose statement.

Faked

In-memory implementation in the test tier. Full contract. Domain tests run against the fake in < 500 ms.

Adapted

Real implementation in adapters/. Persistence model and mapper separate from domain. Integration tests run against the real adapter.

Wired

Composition root selects the concrete adapter at startup. Other layers remain ignorant. Environment variables / config drive selection.

Swapped

A new adapter (DynamoDB, ClickHouse, gRPC client) ships alongside the existing one. Cut-over via config; rollback by reverting one line. Domain unchanged.

Retired

Port no longer needed (capability removed or absorbed). Interface deleted; all adapters removed; composition root cleaned up. Domain code unchanged.

Common pitfalls

Pitfall Why it happens What to do instead
Domain importing ORM models or framework annotations SWEs annotate domain classes with @Entity or @Column for convenience. Domain and persistence become inseparable over time. Define separate persistence models, mapped by the adapter. The domain Lesson class knows nothing of @Table. The adapter owns the mapping; the convenience is not worth the coupling.
God use cases — too many port dependencies A single use case is given every port it might ever need. It becomes a 500-line orchestrator that is untestable and hard to reason about. One use case, one public method, ≤ 3–4 port dependencies. When a use case needs 5+ ports, split into smaller use cases with a coordinator on top.
Leaky abstractions in port interfaces A repository port returns QueryResult or a Pandas DataFrame — a type from the adapter layer. The domain now depends on the adapter's technology vocabulary. All port signatures use only domain types. List[Lesson] is valid; QueryResult is not. If a return type needs naming, define it in the domain package.
Convention-only enforcement Architecture is documented but not enforced. A new developer imports an adapter from the domain by accident; review misses it; six months later the violation is load-bearing. ArchUnit / Dependency Cruiser fitness function blocks any commit that imports an adapter from the domain. Convention drifts; CI does not.
Fakes that diverge from real adapters Fake skips validation or returns shortcuts. Domain tests pass; real adapter fails the same scenario. Confidence is false. Treat the fake like a sibling adapter. Same contract, same edge cases, same care. Optional: a shared contract-test suite both the fake and the real adapter must pass.
Pattern applied uniformly — overkill for CRUD 200-line CRUD admin page gets a full hexagon. Three modules, two adapters, one mapper — for what should have been a 50-line endpoint. Speed loss for no payoff. Classify the subdomain first (core / supporting / generic). Apply hexagonal to the core only. Supporting and generic can stay simpler.

Worked example — Study App lesson core

A trimmed Python sketch of the lesson core. Domain defines the port; adapter implements it; fake powers the domain tests; composition root picks which one runs.

# domain/lessons/ports.py
from typing import Protocol, Optional
from .models import Lesson, LessonId

class LessonRepository(Protocol):
    def get(self, id: LessonId) -> Optional[Lesson]: ...
    def save(self, lesson: Lesson) -> None: ...

class EventPublisher(Protocol):
    def publish(self, event: "DomainEvent") -> None: ...

# domain/lessons/use_cases.py
from dataclasses import dataclass
from .ports import LessonRepository, EventPublisher
from .models import LessonId, Attempt, LessonCompleted

@dataclass(slots=True)
class CompleteLesson:
    repo: LessonRepository
    events: EventPublisher

    def __call__(self, lesson_id: LessonId, attempt: Attempt) -> None:
        lesson = self.repo.get(lesson_id) or _raise_not_found(lesson_id)
        lesson.record(attempt)
        self.repo.save(lesson)
        self.events.publish(LessonCompleted.from_lesson(lesson))

# adapters/postgres_lesson_repo.py
from sqlalchemy.orm import Session
from domain.lessons.ports import LessonRepository
from domain.lessons.models import Lesson, LessonId
from .persistence_models import LessonRow  # ORM model, NOT the domain class

class PostgresLessonRepository(LessonRepository):
    def __init__(self, session: Session) -> None:
        self._session = session

    def get(self, id: LessonId) -> Optional[Lesson]:
        row = self._session.get(LessonRow, id.value)
        return _to_domain(row) if row else None

    def save(self, lesson: Lesson) -> None:
        self._session.merge(_to_row(lesson))

# tests/lessons/test_complete_lesson.py
from domain.lessons.use_cases import CompleteLesson
from tests.fakes import InMemoryLessonRepository, CapturingEventPublisher

def test_completing_lesson_publishes_event():
    repo = InMemoryLessonRepository(seed=[lesson_with(id="L-1")])
    events = CapturingEventPublisher()

    CompleteLesson(repo, events)(LessonId("L-1"), attempt_with(score=0.9))

    assert events.published == ["lesson.LessonCompleted.v1"]
    assert repo.get(LessonId("L-1")).mastery >= 0.9

# app/main.py — the only file that imports both domain and adapters
from domain.lessons.use_cases import CompleteLesson
from adapters.postgres_lesson_repo import PostgresLessonRepository
from adapters.kafka_event_publisher import KafkaEventPublisher

complete_lesson = CompleteLesson(
    repo=PostgresLessonRepository(session),
    events=KafkaEventPublisher(producer),
)

Domain knows nothing of SQLAlchemy or Kafka. The use case takes interfaces; the test wires fakes; the composition root wires the real things. Swapping Postgres for DynamoDB requires a new adapter and one line of services/api/app/main.py — zero changes to the domain.

Quality bar

  • Domain module has zero outgoing infra imports. Verified by ArchUnit / Dependency Cruiser fitness function in CI.
  • Every port named from the domain side. LessonRepository, not PostgresLessonDAO. Signatures use domain types only.
  • Fakes implement the full port contract. Shared contract tests where feasible; no shortcut validation.
  • Domain test suite < 500 ms. No real I/O. Real adapters tested in a separate integration tier.
  • One composition root. One file imports both domain and concrete adapters; adapter selection by config.
  • Use cases capped at one public method, ≤ 50 lines. Larger use cases get split. Port-dependency count ≤ 4.
  • Persistence models separate from domain models. ORM annotations on persistence models only; mapping done in the adapter.
  • Pattern applied selectively. Core domain only; supporting and generic subdomains use simpler shapes.
