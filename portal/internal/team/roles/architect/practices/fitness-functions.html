Fitness functions — guide evolutionary architecture

Automated tests that verify architectural characteristics on every CI run, turning architecture from a slide deck into a failing test.

TL;DR

Fitness functions are automated checks that verify architectural characteristics — latency, coupling limits, security posture, dependency rules — on every CI run. Introduced by Neal Ford, Rebecca Parsons, and Patrick Kua in Building Evolutionary Architectures (2017). They make architectural drift visible in the PR rather than in a quarterly review. The architecture stops being a document and becomes a failing test.

What it is

A fitness function is an automated test that assesses an architectural characteristic of a system. The term comes from evolutionary computing — a fitness function scores how close a candidate solution is to the goal. Applied to software architecture, it measures how close the system is to the desired quality attribute on every CI run.

Three categories:

  • Atomic — run in isolation against the codebase or a running service. Examples: an ArchUnit rule that forbids cyclic package dependencies; a Spectral lint rule that flags API responses without pagination.
  • Holistic — run against the integrated system. Examples: a k6 load test that asserts p99 latency ≤ 200 ms; a chaos experiment that verifies the system degrades gracefully when a dependency is slow.
  • Triggered — run on a schedule rather than per PR. Examples: a weekly dependency audit that fails if a transitive dependency has a critical CVE; a nightly SLO trend check.

The categories are orthogonal: a fitness function can be atomic and triggered (nightly static analysis), or holistic and continuous (every deploy runs a smoke load test).

Why we need it

Architecture decisions have a half-life. The decision to keep module A independent from module B is correct on day 1 and violated by day 180, because developers take the shortest path and the shortest path is often a direct import. Manual architecture reviews catch this six months later, when the coupling is deeply entangled and refactoring costs weeks.

Fitness functions close the feedback loop. The moment a PR introduces a forbidden dependency, the CI job fails. The developer sees the violation before the PR is merged — when it costs 10 minutes to fix instead of 10 days. The architecture stops being aspirational and becomes operational; the architect's leverage shifts from policing the codebase manually to maintaining the test suite that polices it automatically.

How others do it

Ford / Parsons / Kua — originators

Introduced the concept in Building Evolutionary Architectures (2017, O'Reilly). The book provides a taxonomy: atomic vs holistic, triggered vs continuous, static vs dynamic. The authors argue that the most valuable fitness functions are those that enforce characteristics the organisation explicitly chose to optimise for — not generic best practices. A startup that values deployment speed writes different fitness functions than a bank that values auditability.

What we take: Define architectural characteristics before writing fitness functions. The function tests the characteristic; the characteristic reflects a deliberate business priority.

Source: Building Evolutionary Architectures

Netflix — latency & resilience gates

Netflix uses a form of holistic fitness functions through its chaos-engineering practice (Chaos Monkey, Chaos Kong). Each chaos experiment tests a system fitness characteristic — "does the service degrade gracefully when the recommendation engine is unavailable?" — and feeds results back into the architecture. Netflix also runs automated latency-regression tests on every deployment; a deployment that increases p99 latency by > 10% is automatically rolled back.

What we take: Latency and resilience fitness functions are the highest-ROI starting point. A latency-regression test catches more architectural drift than a dependency-rule check.

Source: Netflix Tech Blog — Chaos Engineering

Zalando — Zally API linter

Zalando enforces API fitness functions across 200+ microservices using the open-source Zally API linter. Every API spec change must pass 60+ Zally rules (naming conventions, pagination requirements, error-contract shape) before deployment. The rules encode Zalando's API design principles into automated fitness functions that run in CI. Zally is open-sourced at github.com/zalando/zally.

What we take: API design standards are an ideal fitness-function target — objective, automatable, high-value. Start with 10 rules; expand as violations are discovered.

Source: github.com/zalando/zally

ThoughtWorks — Tech Radar & ArchUnit

ThoughtWorks consultants codify architectural rules with ArchUnit (Java) and similar tools (Dependency Cruiser for JS/TS, Import Linter for Python). Their Tech Radar promotes "Architectural fitness functions" as Adopt-tier guidance. ThoughtWorks's blog posts demonstrate how a small set of well-chosen ArchUnit rules — "no cycles", "ports-and-adapters direction", "domain layer pure" — replaces dozens of recurring code-review comments.

What we take: Start with 5–10 ArchUnit-style rules that encode the team's most-repeated review comments. Each rule retired one recurring comment; the architect's review load drops sharply.

Source: ArchUnit

Anatomy of a fitness function

Architectural characteristic

The quality attribute the function measures. Named, defined, and accepted by the team before writing the test. Example: "Services in the Lessons context must not import packages from Identity." Max 3–5 characteristics per architectural domain to avoid test-suite overload.

Atomic test (static analysis)

Runs against the codebase without a running service. Tools: ArchUnit (Java), Dependency Cruiser (JS/TS), Import Linter (Python), Spectral (OpenAPI/AsyncAPI). Runs in under 30 seconds. Examples: forbidden imports, required interface implementations, banned annotations.

Holistic test (perf / resilience)

Runs against the integrated system. Examples: k6 load test asserting p99 ≤ 200 ms, smoke test verifying graceful degradation when an upstream returns 503. Runs in staging on every deploy. Duration: ≤ 5 min per suite.

Triggered test (schedule)

Runs on a schedule rather than per PR. Examples: weekly transitive-dependency CVE audit, nightly SLO trend, monthly cost-drift check. Catches slow-moving violations that per-PR gates miss.

CI gate

Failures block the PR merge — same as unit-test failures. Exemptions are explicit, signed off by the architect, and expire after 30 days. No silent bypasses. The gate is the mechanism that prevents drift from accumulating.

Expiring exemption

Every exemption carries a sunset date. CI fails again on expiry, forcing a decision: fix the violation or renew the exemption explicitly. Stops "just this once" exemptions from becoming permanent debt.

How to run it

  1. List architectural characteristics (Week 1). Sit with the team. Name 5–8 characteristics that matter (bounded-context isolation, API consistency, p99 latency, dependency hygiene, security posture). Reject vague items.
  2. Mine recurring review comments (Week 1). Look at the last 6 months of code-review threads. Patterns that recur ("don't import X from Y", "use cursor pagination") are fitness-function candidates.
  3. Pick the 5 highest-ROI rules (Week 2). One per characteristic. Each rule must trace back to a named characteristic. Skip easy-but-useless rules (naming conventions) unless naming reflects a real architectural decision.
  4. Implement as atomic tests first (Week 2–3). ArchUnit / Dependency Cruiser / Spectral. Wired into CI. Failures block the PR.
  5. Add 1–2 holistic tests (Week 3–4). A k6 latency test for the core flow; a chaos smoke test for the most critical dependency. Run on post-merge staging deploys to keep PR gate fast.
  6. Set up exemption process (Week 4). Exemption file in-repo with sunset dates. CI parses sunset; expiry triggers a fresh failure. Architect signs off on every new exemption.
  7. Review the suite quarterly. Retire rules that never fire (they encode nothing useful). Add rules from new recurring review comments. Suite stays small and load-bearing.

Deep dive — making the suite load-bearing

Characteristic before code. The temptation is to write rules for whatever is easiest to lint — naming conventions, file structure, formatting. Easy rules add noise without adding architectural signal. Start from the characteristics list: each rule must trace back to a named architectural decision the team agreed to protect. "No circular imports" is valid only if module isolation is an explicit priority. If you cannot name the characteristic, do not write the rule.

Mine review comments for rule candidates. The best fitness functions encode the comments your reviewers leave repeatedly. "Don't import from Y here", "use cursor pagination", "this endpoint needs a 429 response", "no SELECT * in the orders schema". Every recurring comment is a fitness-function candidate; once automated, that review thread never happens again. The architect's leverage shifts from reviewing manually to maintaining the rule suite.

Keep the PR gate fast. A PR gate that takes 30 minutes gets disabled. Atomic tests run in < 30 s; the full PR fitness suite stays under 5 minutes total. Slow holistic tests (multi-minute load tests, full chaos runs) live on the post-merge staging deploy or in a nightly job — not on the PR gate. Speed is the price of credibility.

Exemptions must expire. The single biggest failure mode is "just this once" exemptions that become permanent. Every exemption has a 30-day sunset; CI fails again on expiry. The renewal cost forces a decision: fix the violation or escalate to the architect for an explicit extension with a new reason. No silent perpetual exemptions.

Retire rules that never fire. A rule that never fails in a year is encoding a behaviour the team already does naturally — it adds CI time without adding signal. Quarterly review retires dead rules. The suite stays small; small suites stay credible; credible suites get respected.

Holistic tests beat atomic tests for impact. A latency-regression gate ("p99 must not regress by > 10%") catches more architectural drift than a dozen import rules. Spend the holistic-test budget on what the user actually feels: latency, error rate, graceful degradation. Atomic tests fill in the gaps.

Lifecycle of a fitness function

Proposed

Candidate emerged from a recurring review comment or an explicit characteristic. Drafted with a clear trace to the architectural decision it protects.

Soak (warn-only)

Rule runs in CI but emits warnings instead of failures for the first 1–2 weeks. Author cleans up existing violations; team gets used to the new signal.

Enforced

Rule blocks PR merge. Failures are real. Exemptions go through the architect with a 30-day sunset. CI gate stays under 5 minutes total.

Exempted (temporary)

Specific code paths exempted with a sunset date. CI fails again on expiry. Renewal requires explicit architect sign-off and a written reason.

Reviewed quarterly

Suite reviewed every quarter. Rules that never fire are flagged for retirement. Rules that fire constantly may indicate the rule is too strict or the team needs to refactor.

Retired

Rule no longer adds signal: behaviour is now natural to the team, or the underlying architectural decision shifted. Retired with a short note on what changed.

Common pitfalls

Pitfall Why it happens What to do instead
Functions testing the wrong characteristic Engineers write rules that are easy to automate (naming conventions) rather than rules that reflect important architectural decisions. The suite grows but adds no architectural value. Start from the architectural characteristics list. Each function must trace to a named characteristic the team agreed to protect. "No circular imports" is valid only if isolation is an explicit priority.
Exemptions becoming permanent PR introduces a violation; developer adds an exemption to merge "just this once". Never cleaned up. After six months, 30% of violations are exempted and the suite provides false confidence. All exemptions expire after 30 days. Expiry triggers a fresh CI failure that forces a decision: fix or extend explicitly. Never permanent exemptions.
Holistic tests too slow for the PR gate 30-minute load tests added to the PR gate. Developers disable or skip them. The function is nominally in place but never actually runs. Holistic tests run on the post-merge staging deploy. PR gate stays under 5 minutes total. Long-running tests live in nightly or pre-release runs.
Suite grows without retirement — review noise New rules are added; old rules stay forever. The suite reaches 200 rules; CI takes 15 minutes; developers ignore the output. Quarterly review retires rules that never fire. Cap the suite at a manageable size (~30 rules across all categories). Dead rules add CI time without signal.
No soak period — sudden enforcement New rule lands as a hard gate with 500 existing violations. Whole team has to fix everything on Monday morning. Trust in the suite collapses. Land new rules in warn-only mode for 1–2 weeks. Author cleans up existing violations during the soak. Flip to enforcement only after the codebase is clean.
Rule fires constantly — team adapts by ignoring A rule that fails on every PR signals either a too-strict rule or a real architectural problem. Team adapts by reflexively adding exemptions. Investigate when a rule fires > 5× per week. Either the rule needs softening (false positives), or the underlying drift needs a focused refactor. Don't paper over with exemptions.

Worked example — Study App fitness suite (v1)

The starter set: five atomic rules + two holistic checks, each tracing back to a named architectural characteristic. Lives at fitness/ and runs in CI on every PR.

# fitness/rules.md · Study App architectural fitness suite (v1)

## Characteristics
C1  Bounded-context isolation (DDD)
C2  API consistency (RESTful + cursor pagination)
C3  Latency (p99 ≤ 200 ms on core flows)
C4  Dependency hygiene (no known critical CVEs)
C5  Security (no plaintext secrets, TLS everywhere)

## Atomic rules (ArchUnit + Spectral, < 30 s each)

R-01 [C1] no.imports.from(Lessons → Identity)
     reason: Lessons must not depend on Identity internals.

R-02 [C1] no.imports.from(Identity → Lessons)
     reason: bounded-context isolation is bidirectional.

R-03 [C2] every GET list endpoint must declare a `cursor` query param.
     reason: ban offset/limit pagination at scale.

R-04 [C2] every endpoint must declare 4xx + 5xx responses
     in OpenAPI; ban implicit 500s.

R-05 [C5] no string literal matching /AKIA[0-9A-Z]{16}/ in src/**
     reason: AWS access keys must not ship in code.

## Holistic checks (post-merge staging, < 5 min each)

H-01 [C3] k6 load test on /lessons/{id}/state: p99 ≤ 200 ms over
     5-min run at 200 rps. Regression > 10% blocks deploy.

H-02 [C3] chaos smoke: kill one RDS replica; verify
     /lessons/{id}/state success ratio stays ≥ 99% within 60 s.

## Triggered checks

T-01 [C4] nightly: `npm audit --audit-level=critical` for web/,
     `gradle dependencyCheckAnalyze` for services. CVE found => ticket.

## Exemption format
fitness/exemptions.yml:
  - rule: R-03
    path: src/admin/internal-api
    reason: internal admin endpoints; cursor migration scheduled
    expires: 2026-06-15
    approved-by: ivan

Seven rules, five characteristics. Every rule traces back to a named decision the team explicitly endorsed. Exemptions have a sunset date encoded in YAML so CI fails again automatically when they expire — the suite stays load-bearing.

Quality bar

  • Every rule traces to a named characteristic. Characteristics agreed by the team; rules without a parent characteristic are not added.
  • Suite size capped (~30 across categories). Quarterly review retires dead rules. Adding a new rule requires retiring or merging an old one once cap is reached.
  • PR gate < 5 min total. Atomic + integration tests on PR; holistic tests on post-merge staging; triggered on schedule.
  • Soak period for new rules. 1–2 weeks warn-only before enforcement; existing violations cleaned during soak.
  • Exemptions expire ≤ 30 days. Sunset encoded in repo; CI fails on expiry; renewal requires architect sign-off with reason.
  • Quarterly review. Rules that never fire are retired; rules that fire constantly are investigated for false positives or genuine drift.
  • Holistic tests cover the user-felt characteristics. Latency, error rate, graceful degradation. Atomic tests fill gaps; holistic tests carry the load.
