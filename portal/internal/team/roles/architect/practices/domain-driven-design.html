Domain-driven design — model the business first

Structure software around the business domain so that model boundaries match organisational boundaries.

TL;DR

Domain-driven design (DDD) is a software-design approach introduced by Eric Evans in his 2003 "Blue Book". It centres the software model on the business domain through a shared ubiquitous language and explicit bounded contexts. Each bounded context owns a coherent domain model; a context map describes how contexts integrate (Customer-Supplier, Conformist, ACL, Shared Kernel, Published Language). The outcome is a system whose seams match the business's natural boundaries, enabling team autonomy and lower coordination cost.

What it is

Domain-driven design is a body of patterns and principles for building software whose structure mirrors the business problem. Eric Evans formalised it in 2003 after observing that the most successful projects shared a common trait: the code's vocabulary matched the domain expert's vocabulary. Key building blocks:

  • Ubiquitous language — a shared vocabulary agreed between domain experts and engineers. The same terms appear in code, tests, documentation, and conversations. Removes translation friction.
  • Bounded context — an explicit boundary within which a domain model is valid and consistent. The same word ("product") can mean different things in different contexts; inside a context it has exactly one meaning.
  • Context map — a diagram of all bounded contexts and the typed relationships between them (Customer-Supplier, Conformist, Anti-Corruption Layer, Shared Kernel, Published Language, Separate Ways).
  • Aggregates — clusters of domain objects treated as a single unit for data changes. Each aggregate has a root that enforces invariants and a single transactional boundary.
  • Domain events — immutable past-tense records of something that happened (OrderPlaced, PaymentFailed). The primary integration mechanism between contexts.
  • Strategic distillation — separating the core domain (where the moat lives) from supporting and generic subdomains. Heavy DDD investment goes to the core only.

Why we need it

Without domain boundaries, microservices share database tables, the same concept accumulates three different names across three teams, and every cross-service change triggers a coordination spiral. "User" means the authentication record to Identity, the billing record to Finance, and the analytics profile to Growth — and none of them agree on which one is authoritative.

DDD gives teams autonomy with explicit ownership. A team that owns the Order bounded context can evolve its model independently, because the boundary makes integration points explicit and stable. Startups that invest in DDD early grow their teams without a proportional increase in coordination overhead. Without DDD, scaling the org doubles the meetings.

How others do it

Eric Evans — originator (2003)

Published Domain-Driven Design: Tackling Complexity in the Heart of Software in 2003 after observing patterns in complex enterprise projects. Introduced bounded contexts, ubiquitous language, aggregates, the context map. The "Blue Book" remains the canonical reference. Evans later introduced the "distillation" concept — identifying the core domain (the 10–20% of the model that delivers competitive advantage) vs supporting and generic subdomains.

What we take: Focus DDD investment on the core domain first. Supporting domains can use simpler patterns (Active Record, CRUD) without losing the strategic benefit. Generic domains buy from a vendor behind an ACL.

Source: domainlanguage.com — DDD reference

Vaughn Vernon — Implementing DDD (2013)

Vernon's "Red Book" — Implementing Domain-Driven Design — translated Evans's strategic patterns into concrete tactical guidance: aggregate rules (one transaction per aggregate, reference other aggregates by ID), event-sourcing pairings, hexagonal architecture wiring. Vernon also popularised Event Storming (Alberto Brandolini) as the practical workshop for discovering bounded contexts in a room full of sticky notes.

What we take: Aggregate rules are non-negotiable: one aggregate per transaction; aggregates reference each other by ID, not object graph. Use Event Storming as the discovery workshop — faster than reading code.

Source: vaughnvernon.com — Implementing DDD

Spotify — squads as bounded contexts

Used context mapping to shape the Squad model. Each Squad owns a bounded context (Playlists, Radio, Discovery). Context boundaries align with team boundaries, applying Conway's Law deliberately. The Tribes-and-Squads structure is DDD applied at the organisational level — system architecture and team topology mirror each other by design.

What we take: Draw the context map before the org chart. Team structure should follow domain boundaries, not precede them. When the domain shifts, the org shifts.

Source: Spotify Engineering Culture (2014)

Airbnb — monolith-to-services via DDD

Migrated from a Rails monolith to service-oriented architecture using DDD bounded contexts as the seam finder. The "Service Decomposition" initiative identified 30+ domains (Listings, Bookings, Payments, Identity, Pricing) by running Event Storming with domain experts. Each domain became an independent service owned by one team. The migration took 3 years; without DDD context maps it would have produced ad-hoc service splits that recreated the monolith's coupling across the network.

What we take: Event Storming with domain experts is the fastest way to discover contexts in an existing monolith. Run it before any decomposition plan — service boundaries decided in code review are almost always wrong.

Source: Airbnb Engineering — building services

Anatomy of a DDD implementation

Ubiquitous language glossary

A written dictionary of 20–50 domain terms agreed between engineers and domain experts. Each term has exactly one definition per bounded context. Reviewed quarterly; any new term added to the codebase must appear in the glossary first.

Context map (PlantUML)

A diagram showing all contexts and their typed relationships: Customer-Supplier, Conformist, ACL, Shared Kernel, Published Language, Separate Ways. Stored as a PlantUML file in the repo; reviewed at each architecture retrospective.

Aggregate root

The single entry point to a cluster of domain objects. All state changes go through the root, which enforces invariants. Max recommended size: 3–5 entities. An aggregate that spans more than one transaction is a smell.

Domain events

Immutable records of domain occurrences. Verb past tense (LessonCompleted, AttemptScored). Published as integration events between contexts. Each event carries a correlation ID; retained at least 30 days in the event log.

Anti-Corruption Layer (ACL)

A translation layer that prevents a downstream context from being contaminated by upstream vocabulary. Implemented as a mapper or adapter. Mandatory when integrating with a vendor or legacy system whose model conflicts with the local ubiquitous language.

Strategic distillation

An explicit classification of every context as core, supporting, or generic. Determines investment level: full DDD on core; pragmatic CRUD on supporting; buy + ACL on generic. Without distillation, teams over-engineer admin screens.

How to run it

  1. Run Event Storming (Week 1). Get domain experts and engineers in one room. Use orange sticky notes for domain events. Cluster them into bounded contexts. Identify aggregate roots from the natural pivots.
  2. Draft the context map (Week 1–2). One PlantUML diagram showing all contexts plus typed relationships. Mark which contexts are core, supporting, generic.
  3. Build the glossary (Week 2). 20–50 terms per core context. Agreed with domain experts, not authored by engineers alone. Stored in docs/glossary.md.
  4. Carve aggregates in the core (Week 2–6). Identify aggregate roots; enforce single-transaction boundaries; reference other aggregates by ID. Refactor any cross-aggregate transactions.
  5. Add domain events (Week 4–8). Publish past-tense events at every significant state change in the core. Start with 3–5; expand to cover all aggregate-root mutations.
  6. Add ACLs at generic-domain seams (Week 6–10). Wrap vendor integrations (Stripe, Auth0, Segment) in mapper layers so vendor vocabulary never reaches the core.
  7. Re-run the workshop quarterly. Domain shifts; the map must follow. Quarterly Event Storming catches drift before it becomes architectural debt.

Deep dive — strategic before tactical

Strategic distillation is the load-bearing move. The biggest mistake teams make with DDD is applying tactical patterns (aggregates, domain events, value objects) uniformly across every subdomain. Most of the model is not the moat — it is plumbing. Eric Evans's later work emphasised classifying each subdomain as core, supporting, or generic and matching the investment level to the classification. Heavy DDD on the core; pragmatic CRUD on supporting; buy + ACL on generic. Skip the classification and you over-engineer admin screens while under-investing in the moat.

Bounded context sizes are bounded by cognition. A bounded context should fit in one team's head — 5–8 engineers maximum. When a team grows past 8, the context is probably too large and should split. When two teams own one context, the context is too small or the team boundary is wrong. The size constraint is human, not architectural: a context bigger than one team produces meetings; a context split across teams produces coordination spirals.

Context map relationships are typed for a reason. Customer-Supplier, Conformist, ACL, Shared Kernel, Published Language, Separate Ways — each describes a different integration contract with different change-coordination cost. Untyped relationships ("Service A talks to Service B") hide the cost; teams discover the cost the first time the upstream changes its schema and a dozen downstream teams break.

Aggregates: one transaction, references by ID. Vernon's Red Book makes this rule non-negotiable. One aggregate per transaction; aggregates reference each other by ID, not by object graph. Cross-aggregate consistency is handled by domain events and eventual consistency, not by widening the transaction. Teams that ignore this rule produce monoliths-with-aggregates — all the DDD vocabulary, none of the autonomy.

Event Storming is the discovery workshop, not a deliverable. The sticky notes are not the artefact — the bounded-context map distilled from them is. Run the workshop for 2–4 hours with domain experts present. Reject any context boundary that does not survive the question "would the same change happen on both sides of this line?" — if yes, the line is wrong.

DDD does not require microservices. A modular monolith with strict bounded-context module boundaries delivers most of the benefit at a fraction of the operational cost. Microservices come later — when team-autonomy demands actually exceed module-boundary autonomy. DDD first; microservices on top of DDD only when justified.

Lifecycle of a bounded context

Discovered

Identified in Event Storming as a cluster of related domain events. Tentative name; provisional aggregate roots; tentative integration relationships.

Classified

Tagged core / supporting / generic by the architect + domain expert. Determines investment level for the next quarter's work.

Modelled

Glossary written. Aggregate roots and invariants documented. Module or service carved in the codebase. Domain events published.

Active

Owned by one team. Quarterly review against the map. Glossary kept in sync with codebase. New terms enter glossary before code.

Splitting

Team grew past 8 or domain split surfaced in Event Storming. Plan to split into two contexts with a new typed relationship between them.

Retired

Capability removed or merged into another context. Glossary archived; integration deprecated with a sunset date; domain events stopped.

Common pitfalls

Pitfall Why it happens What to do instead
Applying DDD uniformly across every subdomain Engineers read the Blue Book and apply aggregates + domain events to CRUD admin screens. Overhead is unjustified for supporting and generic subdomains. Classify every context as core / supporting / generic first. Full DDD on core; pragmatic CRUD on supporting; buy + ACL on generic. Investment matches strategic value.
Contexts too large — one giant "platform" Teams draw context boundaries to match existing org boundaries. A 50-engineer "platform team" ends up owning one gigantic context that nobody fully understands. Size contexts by cognitive load: max 5–8 engineers. When a team grows past 8, the context is probably too large — split it. Org follows domain, not the other way around.
Code-centric ubiquitous language Engineers name things after implementation (UserRecord, OrderDTO) rather than domain (Customer, Booking). Domain experts disengage because the vocabulary is foreign. Run Event Storming with domain experts present. Use the sticky-note vocabulary in class names. Reject identifiers with technical suffixes (DTO, Record, Manager) when a domain term is available.
Aggregates that span multiple transactions Team treats "aggregate" as a folder, not a transactional boundary. Cross-aggregate updates happen in one transaction; consistency relies on locks. Enforce Vernon's rule: one transaction per aggregate; aggregates reference each other by ID. Cross-aggregate consistency goes through domain events and eventual consistency.
Untyped context relationships Context map shows arrows without labels. "Service A talks to Service B" hides the integration contract. First upstream-schema change breaks every downstream consumer. Every relationship on the map has a type: Customer-Supplier, Conformist, ACL, Shared Kernel, Published Language, Separate Ways. The type is the contract.
DDD as a microservices justification Team uses DDD vocabulary to motivate splitting a working monolith into 30 services. Operational cost balloons; team autonomy does not actually improve. Start with a modular monolith with strict bounded-context module boundaries. Microservices come later, only when team-autonomy demands exceed what modules can deliver.

Worked example — Study App context map

A trimmed slice of the Study App context map. Three contexts shown — Lessons (core), Identity (supporting), Payments (generic) — with typed relationships and an ACL around the Stripe integration. The full map in docs/architecture/context-map.puml has 11 contexts.

@startuml study-app-context-map
!include <C4/C4_Container>

title Study App · context map (Lessons / Identity / Payments)

System_Boundary(core, "Core domain") {
  Container(lessons, "Lessons", "Kotlin / Spring",
    "aggregates: Lesson, Question, Attempt, Mastery
     events: LessonCompleted, AttemptScored, MasteryReached")
}

System_Boundary(supporting, "Supporting") {
  Container(identity, "Identity", "Kotlin / Spring",
    "User, Session, Permission · mostly CRUD")
}

System_Boundary(generic, "Generic") {
  Container(payments, "Payments", "Kotlin + Stripe SDK",
    "Receipt, Refund · vendor model behind ACL")
}

Rel(identity, lessons, "Customer–Supplier",
    "Identity is upstream; Lessons consumes UserVerified events")
Rel(lessons, payments, "ACL",
    "Lessons emits CourseUnlockRequested; Payments returns
     ReceiptIssued. Stripe vocabulary never crosses the line.")

note bottom of payments
  Generic subdomain — bought via Stripe.
  ACL maps Stripe's `charge.succeeded` →
  domain event `ReceiptIssued`.
  Stripe schema changes do not propagate.
end note

note bottom of lessons
  Core domain — the moat.
  Heavy DDD: aggregates, invariants in code,
  domain events as the integration contract.
  No vendor models allowed inside.
end note
@enduml

The map is a living artefact: regenerated in CI from PlantUML and reviewed at every quarterly architecture retrospective. New contexts must be added to the map before any code lands. The map answers "where does this change belong?" without a meeting.

Quality bar

  • Context map exists in-repo as PlantUML. One context map per product; regenerated in CI; reviewed each quarterly architecture retrospective.
  • Every context tagged core / supporting / generic. Classification visible on the map; drives investment level for the next quarter.
  • Ubiquitous-language glossary ≥ 20 terms per core context. Agreed with domain experts, not written by engineers alone. New terms enter glossary before code.
  • Aggregate roots documented with invariants. One transaction per aggregate; aggregates reference each other by ID. Cross-aggregate updates go through domain events.
  • Domain events published for all significant state changes in the core. Past-tense names, correlation IDs, ≥ 30-day retention in the event log.
  • ACLs wrap every vendor integration. Vendor vocabulary (Stripe, Auth0, Segment) never reaches the core domain. Mapper layer documented per vendor.
  • Event Storming run quarterly. Domain shifts; map follows. Workshop catches drift before it becomes architectural debt.
