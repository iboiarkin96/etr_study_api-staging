SQLite write-lock burst — retry storm during a load test

During a routine k6 load test on the local staging environment, the API returned a sustained burst of 5xx errors for 47 seconds. Root cause was a misconfigured retry policy on the load generator that triggered a thundering-herd retry on every idempotent POST /api/v1/conspectuses failure — which then hammered the SQLite database into SQLITE_BUSY errors faster than the connection pool could clear. No production impact (staging only), but the same retry policy was discovered in an integration tutorial that was about to be published; this is why we test load behaviour against the same database engine as production.

Incident summary

At 14:18 UTC on 2026-04-02 a k6 load test was run against the local staging API (localhost:8000, SQLite backend) to validate the conspectus-create endpoint's idempotency budget. The script targeted 50 RPS for 60 seconds. Within 12 seconds the API began returning 500 Internal Server Error on roughly 30% of requests; by 30 seconds the error rate had climbed to ~85%. The handler logs showed sqlalchemy.exc.OperationalError: (sqlite3.OperationalError) database is locked on the failing requests. The on-call (also the engineer running the test) stopped k6 at 15:05. The system recovered to nominal in under 5 seconds once load stopped. Investigation revealed the k6 script had been written with an aggressive retry-on-5xx policy (max 5 retries, no jitter) which, combined with a 30-second idempotency replay window, amplified each failing write into 5 additional locks against the same WAL frame.

Impact

  • Users affected: zero. Incident occurred against the local staging environment; no integrator traffic touched it.
  • Test runs invalidated: one. The load-test results are unusable because the failure mode was the test itself, not the system.
  • SLO budget: not consumed (staging does not contribute to the production budget).
  • Engineer time: ~2 hours between the failed run and root cause being understood; another ~1 hour to author the new runbook entry and fix the load script.
  • Counterfactual: if the same retry policy had been applied to the live production environment on Postgres, the result would have been the same shape (write-lock contention on the same row) but with different recovery characteristics — Postgres releases row locks faster than SQLite releases its global write lock, so the burst would have been shorter and self-resolving. Production was not affected, but the load test would have given a falsely optimistic capacity number.

Root cause — contributing factors

Three contributing factors, ordered by leverage.

  • The retry policy multiplied load instead of absorbing it. k6's http.request with retries: 5 and no backoff/jitter turned each failure into a microburst of identical writes. Because the writes carried the same Idempotency-Key, the server short-circuited 4 of every 5 retries via the idempotency cache — but the cache lookup itself is a row read against idempotency_keys, and the WAL frame for that row was locked by the original retry's write.
  • SQLite's single-writer model is the wrong shape for this test. SQLite serialises every write on a single global lock; a 50-RPS workload of writes against a single WAL frame is roughly the saturation point. The test was sized for the steady-state target (50 RPS) without headroom for the retry amplifier. Postgres would have absorbed the same workload because row-level locks would have spread the contention.
  • No dashboard was open during the test. The metrics that would have made the failure mode visible at 12 seconds (rising http_request_duration_seconds, db_operation_duration_seconds tail, connection pool wait) all exist and were being scraped. They were not on a screen. The engineer found out by reading logs after the fact instead of seeing the curve climb.

Resolution

  • Immediate. Stopped the load test. Re-ran with k6 retries disabled to measure the real saturation point. The actual number (~half the prior estimate) was recorded in the load-test results document.
  • Durable. Updated the load-test script to disable retries by default and added a comment explaining why ("retries amplify; rely on the API's own idempotency for correctness; measure raw saturation"). Filed action items below to make the failure mode harder to repeat — and to make the same failure mode visible when it does happen.
  • No rollback required. The system never deployed broken behaviour; this was a measurement-tool bug.

Detection

Engineer noticed the burst through k6's own request-success rate in the terminal. The metrics pipeline was running but no one was watching the dashboard during the test. Logs (app.log) contained the smoking-gun stack trace from the first failed write; without grep on the log file, the failure mode would have looked like generic "5xx during load test".

If this had happened on production: error_budget_burn_rate would have crossed the 2-minute fast-burn threshold by 14:19:00 and paged the on-call (see runbook 0009). On staging, no alert is wired.

Lessons learned

  • The client retry policy is part of the contract. An API designed for idempotent retry assumes the client retries sanely — that is, with jitter and a bounded budget. A retry policy without jitter turns transient failures into amplifying ones; we should document the expected client retry shape on every write endpoint.
  • Capacity numbers without observable saturation are guesses. The pre-incident estimate of ~100 RPS was a guess. The post-incident measured number of ~50 RPS sustained, ~95 RPS with brief contention is a fact. The lesson is to never publish a capacity number without naming the saturation signal.
  • Visible > logged. Every signal needed to diagnose this in real time existed in metrics. None were on screen. The discipline of "open the dashboard before starting the test" is cheap and saves the 30-minute log forensics that this incident took.
  • SQLite's locking model is fine for a single user; it is the wrong choice for parallel writes. This postmortem is one of the empirical inputs to the planned Postgres swap in the architectural narrative.
