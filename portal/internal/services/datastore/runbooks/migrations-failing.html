0002 — Migrations failing

Alembic errors, bad migration files, wrong DB path, or schema mismatch after a hand-edit.

Impact

  • Local developer loop blocked until the failing stage clears.
  • No production user impact unless the failure is in shipped artefacts.

Follow-up

  • Capture the root cause in the related postmortem if SEV-2 or higher.
  • Open backlog items for any preventive change discovered during recovery.

Trigger

  • make migrate fails.
  • .venv/bin/python -m alembic revision --autogenerate -m ... generates incorrect diff.
  • App startup fails due to DB schema mismatch.

Fast triage

  1. Confirm .env points at the DB you expect (SQLITE_DB_PATH).
  2. Run migration command explicitly: make migrate.
  3. Inspect current migration heads: .venv/bin/python -m alembic heads.

Most common causes

  • Model changes were made but migration not generated.
  • Migration was built with stale or wrong model imports.
  • Wrong DB file/path in .env.
  • Someone changed the DB by hand; history no longer matches Alembic.

Recovery steps

  1. Confirm model import chain includes new/changed model.
  2. Generate migration from current models: .venv/bin/python -m alembic revision --autogenerate -m describe_change.
  3. Review generated file in services/api/alembic/versions/.
  4. Apply migration: make migrate.
  5. Re-run tests: make test.

Safe rollback strategy (local)

  • If the file is wrong and nobody else has it yet, delete it and generate again with fixed models/imports.
  • Do not rewrite shared migration history without team agreement.

Exit criteria

  • make migrate passes.
  • App starts and uses expected schema.
  • make test passes.

Escalation

If 30 minutes of triage didn't move the needle, or the failure mode isn't on the «Most common causes» list:

  1. Capture state — failing Alembic output (make migrate 2>&1 | tee /tmp/runbook-0002-state.txt), .venv/bin/python -m alembic heads and history, current DB file path from .env, and the recovery steps tried.
  2. Open a SEV-3 incident in the team backlog and link this runbook from it. Schema corruption on a shared DB jumps to SEV-2 immediately.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last-known-good migration revision hash.
  4. If migration history has diverged on a shared DB, do not rewrite history alone — escalate first; coordinated rollback is the only safe path.
  5. If the failure blocks a release tag, escalate to SEV-2 and queue a postmortem (how to write one).

Follow-up

  • If the schema policy is non-obvious, capture it in an ADR.
  • Consider an Incident log: append a one-line entry whenever this runbook fires on a shared DB (date / revision / cause) so recurring patterns surface.
