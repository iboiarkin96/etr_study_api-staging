Why this entry is thin

The UI Kit is build-time only — no runtime, no on-call, no API contract to publish. Keeping the catalog entry to three pages avoids parallel documentation: the showcase is canonical for the component / token surface, and this catalog entry only adds where it lives and who uses it within the monorepo.

What stays here

  • Entity card — owner, lifecycle, source-of-truth path, and the pointer to the live showcase. The page anyone arriving cold should land on first.
  • Architecture — the build pipeline and runtime model (how entry.css + entry.js are assembled per portal).
  • Dependencies — what the kit ships against, who consumes it, and the blast-radius map for breaking changes.

What lives in the showcase

  • Component pages (one per BEM block — entity-card, dep-lanes, spine-grid, …) with anatomy, tones, markup, and content guidelines.
  • Foundations — design tokens, typography ladder, motion / adaptivity contract.
  • Templates — full-page references for the portal layouts.

What is intentionally absent

  • API reference — the kit publishes no HTTP / RPC / message contract. The "API" is the BEM class catalogue, documented in the showcase.
  • Runbooks — no runtime, nothing to recover. CSS regressions are resolved via PR review, not on-call.
  • On-call — no paging surface; ownership is captured on the entity card alone.

The spine never gets to bend, but the irrelevant pages collapse into this single explanation. Other entries (api, monitoring, portal) keep the full six-page spine because they have runtime surface area that the spine is designed to cover.
