Postmortem SEV-3 Resolved Docs pipeline

Portal search returned stale results — search-index drift after a commit-with---no-verify

Portal search stopped surfacing two newly-added how-to pages and returned three results for pages that had already been deleted. Root cause was a commit landed with git commit --no-verify that bypassed the docs-fix step 9 (search-index regeneration). The pre-commit hook was the only place this regeneration happened; --no-verify silently turned the index into a stale artefact. Time from drift to detection: 6 days, via the integrator who tried to search for the new api-load-testing page and got zero results.

Incident summary

On 2026-05-07 at 09:30 UTC a content-only commit landed on main that added two new how-to pages (api-load-testing.html, api-endpoint-testing.html) and deleted three retired runbooks. The author used git commit --no-verify because an unrelated pre-commit hook was failing on a different file. The bypass had a side effect that wasn't obvious: search-index.json is regenerated by the docs-fix Makefile target which the pre-commit hook invokes. With the hook skipped, the index was never re-generated. The two new pages were unreachable via the portal search bar, and three deleted-page entries were returned for matching queries. The drift sat undetected for six days. The integrator who first hit it ("can't find the load-test guide") filed a Slack message; investigation identified the stale index in <5 minutes. Fix was a single make docs-fix run on a clean tree, followed by a commit of the regenerated index. No code or content was lost.

Impact

  • Users affected: every reader of the portal who used the search bar between 2026-05-07 09:30 and 2026-05-13 14:22 UTC. The exact count is unknown — search telemetry was not yet wired (see action items). The integrator who filed the report was probably one of several silently-frustrated users.
  • Pages unreachable via search: 2 new how-tos.
  • Phantom search results: 3 deleted runbook entries; clicking returned 404.
  • SLO budget: not consumed (portal is not an SLO surface today, see SLOs Explanation).
  • Engineer time: ~30 minutes diagnosis and fix; ~1 hour writing the new mandatory rule into CONTRIBUTING.md and the docs-pipeline how-to.
  • Counterfactual: if the integrator hadn't reported it, the drift would have continued. Every subsequent content-only PR that touched the search-index transitively (i.e. almost every PR) would either have re-generated it correctly or would have made the drift worse. We were one well-intentioned --no-verify away from a search index that was structurally divergent from the page set.

Root cause — contributing factors

Three factors, ordered by leverage.

  • Search-index regeneration lived only in the pre-commit hook. make docs-fix is the only command that updates search-index.json; the pre-commit hook is the only place that invokes it implicitly. Bypassing the hook silently bypassed the regeneration. There was no CI check that compared the computed index against what the committed pages would produce — the check used the committed index as ground truth, which made it tautological.
  • --no-verify treats all hook outputs as optional. The git ergonomics give one switch that means "skip everything". Some hooks (lint, format) genuinely should be bypassable when a separate file is failing. Others (regeneration of an artefact other code depends on) absolutely should not. The same flag covers both.
  • The drift was silent. A reader who searches and gets zero results assumes the page doesn't exist, not that the index is wrong. Without telemetry on search queries that returned zero results, the failure mode had no signal. The integrator's report was the first observation; lucky.

Resolution

  • Immediate. Ran make docs-fix on a clean tree, committed the regenerated search-index.json (and the search-index-public.json mirror), pushed. Portal search recovered on the next page load (no cache to bust because the index is fetched by content hash).
  • Durable. The CI check was rewritten to compare the regenerated index against the committed one on the same input page set — if they disagree, CI fails. This catches the exact scenario where the committed index is stale. The rule "--no-verify requires a manual make docs-fix before push" was added to CONTRIBUTING.md and to the agent memory so it survives editor sessions.
  • No content rollback required. The pages were correct; only the search index was wrong.

Detection

An integrator reporting an empty search result. This is the worst kind of detection because it is slow (6 days), depends on a friendly user who files a ticket instead of giving up, and gives no signal of how many users were affected before the report. We are lucky the integrator reported it rather than concluding the docs were incomplete and going elsewhere.

The proper detection — comparing the regenerated index against the committed one — was added as an action item. Telemetry on zero-result searches is the second action item; it would shrink the detection window from days to hours.

Lessons learned

  • A generated artefact that ships in git is a contract. search-index.json is computed from the page set; it sits in the repo as a static asset; consumers read it without re-computation. That makes it a contract just like the OpenAPI baseline — and the same rule applies: CI must compare the computed truth against the committed artefact, not just validate the committed artefact against itself.
  • "Skip the hook" is one of the few git ergonomic affordances that has no fine-grained control. A switch that means "skip everything" will eventually be used for "skip the one thing I know is unrelated to my change". The right response is not to ban --no-verify but to make the consequences of using it surface elsewhere — CI in this case.
  • Silent failures take longer to detect than loud ones, by definition. The lesson is to add the loud signal — here, zero-result-search telemetry — before assuming silent failure modes will stay rare.
  • One well-intentioned engineer is a single point of failure for any rule that lives only in their head. The rule "remember to run docs-fix after --no-verify" is in CONTRIBUTING.md now, and in memory, and enforced by CI. Three durable copies are better than zero.
