0010 — In-page "On this page" sidebar missing or empty

A long page has no right-rail TOC, or the TOC list renders empty. Page shell, heading structure, and runtime checks.

Impact

  • Local developer loop blocked until the failing stage clears.
  • No production user impact unless the failure is in shipped artefacts.

Follow-up

  • Capture the root cause in the related postmortem if SEV-2 or higher.
  • Open backlog items for any preventive change discovered during recovery.

Trigger

  • A long HTML page under services/portal/ (not services/portal/pdoc/) has no "On this page" list, or the list is empty.
  • You expect a main column plus a right-rail <aside class="docs-toc"> filled from the article's h2/h3 headings.

Fast triage

  1. The page loads entry.js from services/frontend/portal/assets_v2/runtime/internal/entry.js (Network tab: no 404).
  2. JavaScript is on. For file:// URLs some browsers block scripts; use a small server, e.g. python -m http.server 8765.
  3. Console: no errors from entry.js before the layout runs.
  4. The page contains <aside class="docs-toc"> with at least one <a class="docs-toc__link" href="#…"> entry, and the article exposes matching h2 id="…" sections.
  5. If the mobile FAB is missing, confirm <button data-component="toc-fab"> is present after the docs-toc aside.

Most common causes

  • No docs-toc aside on the page — without it the right-rail mount has nothing to fill. Add the aside or copy from a sibling page.
  • Headings without id — TOC anchors point at #<id>; if the h2 has no id the link jumps nowhere.
  • Broken markup around a heading — an unclosed tag above the heading hides it from the scroll-spy.
  • Page-type opt-out — landing/cockpit page types may suppress the right-rail TOC by design; how-to pages should always include it.
  • Stale entry bundle — old asset paths pointing at assets/docs-nav.js instead of assets_v2/runtime/internal/entry.js.

Recovery steps

  1. Ensure the page links to services/frontend/portal/assets_v2/runtime/internal/entry.css and entry.js with correct relative depth.
  2. Ensure each section you want in the TOC has an h2 or h3 with a stable id.
  3. Include the right-rail aside near the end of <main>: 
    <aside class="docs-toc" aria-label="On this page">
<p class="docs-toc__title">On this page</p>
<ol class="docs-toc__list">
<li class="docs-toc__item"><a class="docs-toc__link" href="#triage-h">Fast triage</a></li>
</ol>
</aside>
          <button data-component="toc-fab" type="button" aria-label="On this page" aria-expanded="false"></button>
  4. Run make docs-fix so HTML normalization stays consistent, then make docs-check.

Exit criteria

  • "On this page" lists the right sections; the active link updates as you scroll.
  • No console errors from entry.js on that page.
  • Mobile FAB opens a drawer matching the right-rail TOC contents.

Escalation

If 30 minutes of triage didn't restore the TOC, or the failure spans multiple pages (the v2 runtime itself failed to mount):

  1. Capture state — the affected page URL(s), browser console output, entry.js network response (200 + correct MIME type?), git diff against the last known-good page, and the recovery steps tried.
  2. Open a SEV-3 incident in the team backlog and link this runbook from it. Multi-page TOC failure indicating a v2 runtime regression jumps to SEV-2 — the whole portal navigation surface is degraded.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last successful make docs-check commit hash.
  4. If a runtime regression is the cause, revert the offending assets_v2/runtime/internal/entry.js or component commit before debugging further; navigation gaps degrade every page.
