0004 — Pre-commit failing

Hooks fail on commit: staging conflicts, ruff-format, mypy, hygiene checks, or the no-artifact-files guard.

Impact

  • Local developer loop blocked until the failing stage clears.
  • No production user impact unless the failure is in shipped artefacts.

Follow-up

  • Capture the root cause in the related postmortem if SEV-2 or higher.
  • Open backlog items for any preventive change discovered during recovery.

Trigger

  • make pre-commit-check fails.
  • git commit fails because a hook failed.

Fast triage

  1. Run hooks: make pre-commit-check.
  2. Check which hook failed (ruff, format, mypy, hygiene hooks).
  3. Fix issues, then run again; hooks may edit files for you.

Most common causes

  • ruff-format changed files but they were not staged before commit.
  • mypy reports type errors after schema/service edits.
  • Hygiene hooks found trailing spaces/EOF/line endings.
  • Merge conflict markers accidentally committed.
  • no-artifact-files refused a staged local artifact (.coverage, *.db, PR_BODY.md, changelog-llm-draft.md).

Special case: no-artifact-files refused a staged file

Output looks like:

✗ Refusing to commit local artifact files (these belong in .gitignore):
.coverage
PR_BODY.md
      Run 'git restore --staged <file>' to unstage.

The hook (scripts/check_no_artifact_files.sh) blocks commits that contain local CI/runtime artifacts which should remain gitignored. The patterns it rejects are .coverage, *.db, PR_BODY.md, and changelog-llm-draft.md. Recovery:

  1. Unstage each listed file: git restore --staged <path>.
  2. Confirm the file is in .gitignore — if not, add it; if it is, you accidentally added it with git add -f or via a path glob (git add -A on a worktree that already had it tracked).
  3. Commit again.

Do not bypass the hook with --no-verify. The patterns are chosen because tracking these files leaks per-machine state into the repo (the 2026-05-07 portal bug audit closed exactly this failure mode).

Recovery steps

  1. Apply auto-fixes: make lint-fix, make format-fix.
  2. Stage updated files: git add -A.
  3. Re-run hooks: make pre-commit-check.
  4. If mypy still fails, run make type-check and fix typing issues.

Special case: unstaged changes conflict

If commit output shows stash/rollback conflict:

  • Stage all intended files first: git add -A.
  • Run make pre-commit-check.
  • Commit again.

Exit criteria

  • make pre-commit-check passes.
  • No auto-modified files remain unstaged.
  • Commit succeeds without hook rollback.

Escalation

If 30 minutes of triage didn't clear the hook, or the failure mode isn't on the «Most common causes» list:

  1. Capture state — make pre-commit-check 2>&1 | tee /tmp/runbook-0004-state.txt, git status, the offending file paths, and the recovery steps tried.
  2. Open a SEV-3 incident in the team backlog and link this runbook from it.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last successful pre-commit run's commit hash.
  4. Do not bypass with --no-verify — that escalates the incident from «one failing hook» to «whole repository in unknown state», and the 2026-05-07 bug audit closed exactly this failure mode.

Follow-up

  • To change which hooks run or how strict they are, edit .pre-commit-config.yaml.
  • If the same failure keeps happening, add a line to this runbook.
