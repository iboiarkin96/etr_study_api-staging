portal · Runbooks

Operational guidance for the portal service. Because the portal has no runtime application logic, almost every "incident" is a build-time issue — reproducible by running make docs-fix locally. Authoritative procedures live in internal/sre/runbooks/ and are indexed below by symptom.

Kroki unreachable — partial-render strategy

The render step uses Kroki at https://kroki.io/plantuml/svg. When the host is offline or the network blocks egress (corporate proxy, sandboxed CI), the build fails on first changed PUML.

  1. Confirm: curl -sI https://kroki.io. If TLS fails, the issue is the network, not Kroki.
  2. Workaround: commit the PUML sources without the SVGs. Re-run make docs-fix when the network is restored. The fingerprint cache (input-hashes.json) detects changes and regenerates only what needs regeneration.
  3. Long-term mitigation: bundle a local plantuml.jar + Java runtime in the dev container, and let tools/docs/regenerate_docs.py fall back to local rendering. Tracked in backlog.

Rebuild the portal locally

# Full rebuild (UML + pdoc + search index)
make docs-fix

# Verify only — no writes (used by pre-commit / CI)
make docs-check

# Specific pieces
.venv/bin/python tools/docs/regenerate_docs.py --force      # ignore fingerprint cache
      .venv/bin/python tools/docs/build_pagefind_index.py     # Pagefind bundle only
