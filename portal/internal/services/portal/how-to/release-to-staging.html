Release to staging

Promote a feature branch through the staging branch so CI publishes it to the staging Pages site (<owner>.github.io/<app>-staging/). Use this any time you want to validate a change on a real Pages domain before merging to main — the only environment that catches base-path, Pagefind-fetch and cross-page navigation bugs. The deployment topology is fixed by ADR 0034.

Before you start

  • Your change lives on a feature/* branch in <app> and is pushed to the remote. Branch naming per ADR 0017.
  • make docs-check is green locally — autogen drift will fail the deploy pipeline the same way it fails the prod one. See Run the docs pipeline.
  • You have Write access to <app> and merge rights on the staging branch (branch protection lets approved PRs land but blocks direct pushes).
  • The cross-repo deploy secret is configured. You do not need to touch it for a normal release — it is already wired in Actions. If you suspect it is missing or expired, jump to Troubleshooting.
  • You are not editing <app>-staging directly. That repo is CI-owned. The only path that legitimately writes to it is deploy-staging.yml.

Steps

  1. Open a PR from your feature branch to staging. 
    gh pr create --base staging --head feature/<your-slug> \
  --title "<short summary>" \
          --body "Promotes <feature> to the staging site for validation."
    Note the base is staging, not main. A PR straight to main bypasses the staging URL entirely — the gate this whole flow exists to provide.
  2. Wait for CI on the PR to go green. Same docs-check gate that protects prod. Address failures on the feature branch — do not merge red.
  3. Merge the PR into staging. Squash or merge-commit; both are accepted. Branch protection requires at least one approving review and all status checks green.
  4. Watch deploy-staging.yml run. 
    gh run watch --workflow=deploy-staging.yml
    Pipeline: checkout → build Pagefind index + stage _site (identical bytes to deploy-prod.yml; project base resolved at runtime by the portal — see ADR-0034 amendment 2026-06-12) → peaceiris/actions-gh-pages@v3 pushes the build to <app>-staging's main with force_orphan: true.
  5. Validate on the staging URL. Open https://<owner>.github.io/<app>-staging/ within ~60 s of the workflow completing. Use a private window — GitHub Pages caches aggressively and a stale tab will lie to you. Walk the change end-to-end on the real domain.
  6. Open the promotion PR staging → main. 
    gh pr create --base main --head staging \
  --title "Release <date or slug>" \
          --body "Promotes the validated staging set to prod."
    This is typically a batch promotion (one or more validated feature merges accumulated on staging). Merge after review; the production environment Required reviewer gate fires on the deploy job — approve the deployment explicitly in the Actions tab.
  7. Confirm prod. Watch deploy-prod.yml, then load the prod URL in a private window. If prod renders correctly, the release is complete.
  8. Sync the staging branch back to main. The squash-merge in step 6 advanced main with a new SHA while staging still holds the pre-squash history. Run make sync-staging so the next feature branches off a clean common ancestor — full procedure in Sync staging after a prod merge.

Verify

  • The staging URL serves your change within ~60 s of the merge to staging (Pages CDN propagation window).
  • Network panel on the staging URL shows zero 404s — asset paths resolved correctly under the /<app>-staging/ base.
  • Pagefind search loads and returns results on the staging URL (validates the bundle fetched from the right base path).
  • deploy-staging.yml reports a successful push to <owner>/<app>-staging with one commit added to its main branch (force_orphan: true keeps the history at depth 1).
  • After the promotion PR merges, the prod URL updates without you touching <app>-staging — the two pipelines are independent.
  • The prod deploy waited for your approval click in the production environment (Actions → Deploy prod → Review deployments).

Troubleshooting

SymptomLikely causeFix
deploy-staging.yml fails at the push step with 403 Resource not accessible by integration. GitHub App installation token expired, App uninstalled, or the App secret was rotated. Reissue the App private key, update STAGING_DEPLOY_APP_KEY in <app> → Settings → Secrets → Actions. Re-run the failed workflow.
Staging URL returns 404 on every page even after a successful deploy. Pages source not set on <app>-staging, or pointing at the wrong branch. <app>-staging → Settings → Pages → Source = Deploy from a branch, branch = main, folder = / (root). Wait one more minute and reload.
Page loads but assets (CSS, JS, Pagefind bundle) 404. Runtime project-base resolver in portal-href.js / search.js failed to derive the staging prefix — either the URL doesn't contain /portal/ (root index landed somewhere else) or those scripts didn't load. Open DevTools on the staging URL: confirm portal-href.js + search.js loaded; confirm window.location.pathname contains /portal/. Verify the artifact has the root redirect index.html → portal/. Re-run deploy-staging.yml after fixing the underlying script load.
Staging looks fine in your browser but your reviewer sees old content. CDN/browser cache — Pages serves with long cache headers. Reviewer opens the URL in a private window or appends ?v=<timestamp>. Not a deploy problem.
Two staging deploys ran at the same time and one failed with non-fast-forward. Concurrent merges to staging raced past the workflow's concurrency group. Re-run the failed workflow — the second push will fast-forward over the first. If it recurs, confirm concurrency: { group: pages-deploy-staging, cancel-in-progress: false } is set on the workflow.
Prod deploy started without your approval. The production environment is missing the Required reviewer protection rule. <app> → Settings → Environments → production → Required reviewers → add yourself. Re-test on the next release.
You urgently need to take staging offline. Demo gone wrong; pre-release content leaked. Push an empty placeholder build to staging, or rename the <app>-staging repo (Pages URL changes; old URL 404s within minutes). Prod is unaffected.
