Docs How-to

Run the docs pipeline

Regenerate machine-owned docs artefacts (UML, service catalog HTML, pdoc, search index) and verify there is no autogen drift before commit. Use this after any change under services/portal/ or any source the pipeline syncs from (Makefile targets, FastAPI error catalog, Python docstrings, service catalog YAML).

Before you start

  • .venv exists — make venv && make install ran successfully once on this clone.
  • Working tree clean (or you accept that the regenerated artefacts will mix with your in-flight edits in the same diff).
  • You're not editing files the pipeline owns by hand — see Managed paths below.

Steps

  1. Regenerate artefacts. 
    make docs-fix
    Each of the 9 stages is idempotent — running the command twice produces the same result. Last log line should read == DOCS-FIX: SUCCESS ==.
  2. Inspect what changed. 
    git status
    Expect modifications only under the paths listed in Managed paths. Anything else means the pipeline touched something it shouldn't — re-check your edits.
  3. Run the gate. 
    make docs-check
    This runs the four validation checks (HTML, design, feedback, spec), re-runs docs-fix, then compares git diff HEAD before and after. If the diffs match, the working tree is converged. Last log line should read Docs check passed (no drift).
  4. If drift was detected, stage the regenerated files, commit them with your edits, and re-run step 3 to confirm the gate is now green.

Verify

  • make docs-fix exits 0 with == DOCS-FIX: SUCCESS == as the final line.
  • make docs-check exits 0 with Docs check passed (no drift) as the final line.
  • A second make docs-fix immediately after the first produces an identical diff — idempotency confirms the pipeline converged.
  • CI's docs-check job is green on the PR (same command, same scripts).

What make docs-fix does (10 stages)

Each stage runs in order; later stages may depend on earlier ones (e.g. step 10 indexes HTML normalized by step 5 and the catalog rebuilt at step 9).

  1. UML rendering — tools/docs/regenerate_docs.py sends services/portal/internal/handbook/uml/**/*.puml to Kroki and writes SVGs under services/portal/internal/handbook/uml/rendered/. Requires network. Skipped per-file when the hash hasn't changed.
  2. Marker sync — tools/docs/sync_docs.py overwrites marker blocks in README.md and services/portal/internal/services/api/reference/errors.html from the live FastAPI error catalog. Prose outside markers is never touched.
  3. Markdown → HTML companions — tools/docs/render_docs_html.py rewrites the .html file next to any .md companion in governed trees. Only runs when companion .md files exist.
  4. HTML structure repair — tools/docs/repair_docs_html.py parses every services/portal/**/*.html through html5lib and rewrites it back, fixing unclosed tags, attribute quoting, and similar low-level drift.
  5. HTML formatting — tools/docs/format_docs_html.py normalizes the v3 stylesheet/script imports, indentation (two-space, <pre> interiors preserved), and the <main class="container"> shell on every v3 page.
  6. Service catalog — tools/docs/render_service_descriptors.py reads services/portal/internal/services/*/catalog-info.yaml and rewrites the per-service hub HTML, the catalog hub, and the services subtree in nav-tree-internal.json.
  7. Maintainers attribute — tools/docs/ensure_docs_maintainers.py injects data-maintainer-ids on every <body> that's missing one, so the v2 author-chip can resolve the «Edited by» block.
  8. Python API reference (pdoc) — pdoc regenerates HTML for the app package into services/portal/internal/services/api/code-reference/; tools/docs/normalize_pdoc_output.py then strips unstable reprs.
  9. Internal catalog rebuild — tools/docs/build_catalog.py walks every tagged page under services/portal/internal/ and rewrites catalog/by-quadrant/*.html, catalog/by-service/*.html, catalog/by-topic/*.html, catalog/recent.html, and catalog/index.html from data-page-type / data-service / data-role / data-updated attributes.
  10. Pagefind bundle — tools/docs/build_pagefind_index.py mirrors every HTML page through a visibility-tagging pre-pass (data-pagefind-filter="visibility:internal|public", data-pagefind-meta="kind:<taxonomy>"), runs the pinned Pagefind CLI, and writes the chunked bundle to services/frontend/portal/pagefind/. tools/docs/check_pagefind_visibility.py immediately asserts the internal/public split. The bundle is gitignored — rebuilt on every CI Pages deploy. Authority: ADR 0033.

Managed paths

Three ownership boundaries — answer the question «if I hand-edit this file, will docs-fix overwrite me?»

🤖 Generated — don't hand-edit

Edit the source instead; docs-fix regenerates the output on every run.

  • internal/handbook/uml/rendered/*.svg · stage 1, from .puml sources
  • internal/services/<name>/index.html + catalog hub + services subtree of nav-tree-internal.json · stage 6, from catalog-info.yaml
  • internal/services/api/code-reference/ · stage 8, from Python docstrings (ADR 0016)
  • internal/catalog/{by-quadrant,by-service,by-topic,recent,index}.html · stage 9, from data-* attributes on every tagged page
  • services/frontend/portal/pagefind/ · stage 10, gitignored Pagefind bundle (ADR 0033)

🪧 Partially managed — markers only

Inside the marker comments: auto-overwritten. Outside: hand-edited prose.

  • README.md · stage 2
  • internal/services/api/reference/errors.html · stage 2, from the live FastAPI error catalog

✍️ Hand-edited — yours

Your normal prose territory. Stages 4–5 only repair structure and re-indent on every run.

  • services/portal/**/*.html (except backlog/ and the generated trees above)
  • env/example · not touched by docs-fix; copied to .env by make env-init
  • public/reference/api/openapi-baseline.json · managed by make openapi-accept-changes
  • public/reference/api/index.html · static Swagger UI

Troubleshooting

The five failure modes you'll hit most often. Catch them at make docs-check, not on the CI run.

  • «Docs check failed» on first run after a docs edit. Expected — docs-fix regenerated artefacts to match your edits. Re-stage the regenerated files, commit alongside your edit, re-run make docs-check.
  • UML stage (1/10) network error. Kroki call failed. Retry. If persistent, check connectivity to the Kroki endpoint configured in tools/docs/regenerate_docs.py; PUML files unchanged since the last successful render are skipped by content hash.
  • Marker sync (2/10) reports «out of sync» under --check. Means README.md or errors.html markers diverged from the live source. Run make docs-fix to refresh, then commit.
  • Catalog stage (6/10) rewrote a service hub you weren't expecting. The YAML at services/portal/internal/services/<name>/catalog-info.yaml is authoritative — your manual edits to the rendered index.html get overwritten. Edit the YAML instead.
  • Search index (9/9) shows huge diff after a tiny edit. The index is rebuilt unconditionally — JSON formatting + ordering may shift even on tiny content changes. Commit the diff as-is; the diff is deterministic across machines.
