portal · Dependencies

What the portal service depends on (build-time and runtime) and who depends on it. The portal has no runtime upstream service — every dependency is consumed at build time, and the runtime is just static-file serving.

Upstream — build-time dependencies

ComponentUsed for
Python (≥ 3.12)Runs tools/docs/regenerate_docs.py, tools/docs/build_pagefind_index.py, pdoc.
PlantUML / Kroki (https://kroki.io/plantuml/svg)Renders all .puml sources to SVG. Authority: ADR 0020. Fallback when Kroki is unreachable: portal · Runbooks.
pdocGenerates the api code reference under catalog/api/code-reference/. Authority: ADR 0016.
OpenAPI document (from api)Embedded into services/portal/public/reference/api/. Build-time consumer of the api service.
pre-commitRuns tools/docs/regenerate_docs.py --check on every commit; blocks divergence between PUML sources and committed SVGs. Authority: ADR 0001.

In-repo frontend assets — the ui-kit

The CSS / JS layer lives in the same monorepo and is served by nginx alongside the HTML pages. No install step — pages reference assets via relative paths. The ui-kit service is the source of truth for all visual and interactive behaviour of both portals; the portal service only references its tokens, components, and runtime bundles.

Path Contents Authority
services/frontend/portal/assets_v2/ Design tokens, component CSS/JS, per-portal runtime bundles, nav-tree mocks. ui-kit
services/portal/ui-kit/ Live showcase pages (third "portal") covering tokens, components, templates. /portal/ui-kit/
services/portal/internal/handbook/uml/ PlantUML sources (architecture, sequences, make-flow diagrams). Rendered SVGs live alongside the sources. ADR 0020

Changes to services/frontend/portal/assets_v2/ affect both the public and internal portals. Component documentation lives at the live showcase (/portal/ui-kit/) — do not copy components locally per RFC 0004.

Runtime dependency

ComponentRole
nginx-alpine (or equivalent static server) Serves services/portal/{public,internal}/ over HTTPS. No application logic, no auth, no state. Per ADR 0028 phase 1.

The portal does not call the api at runtime. The OpenAPI document is built into the static tree at build time.

Downstream — who consumes the portal

ConsumerWhat they read
IntegratorsPublic landing, getting-started guides, OpenAPI sandbox under /reference/api/.
Maintainers (this audience)Internal portal: ADRs, RFCs, runbooks, audits, handbook, this catalog, system design.
Reviewers / interviewersSame as maintainers — the portal is the artefact reviewers see when assessing the project.
