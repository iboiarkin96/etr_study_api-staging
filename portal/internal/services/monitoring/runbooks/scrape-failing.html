0008 — Observability scrape failing

Prometheus target DOWN, empty /metrics, or Grafana panels showing No data.

Observability
Severity

Owner
Ivan Boyarkin
Last reviewed

Impact

  • Local developer loop blocked until the failing stage clears.
  • No production user impact unless the failure is in shipped artefacts.

Follow-up

  • Capture the root cause in the related postmortem if SEV-2 or higher.
  • Open backlog items for any preventive change discovered during recovery.

Trigger

  • Prometheus target is DOWN.
  • /metrics endpoint returns non-200 or empty payload.
  • Grafana dashboard panels show No data.

Fast triage

  1. Ensure API is running: make run.
  2. Ensure observability stack is running: docker compose -f services/monitoring/docker-compose.observability.yml up -d.
  3. Run smoke-check: python scripts/check_observability_links.py.
  4. Verify endpoint directly: curl http://127.0.0.1:8000/metrics.
  5. Open Prometheus targets page and inspect scrape error details.

Useful links:

  • Prometheus targets: http://127.0.0.1:9090/targets
  • Grafana dashboard: Study App Observability
  • Quick Prometheus queries: RPS · Error rate · API p95 · DB p95

Most common causes

  • API process is down or started on a different port.
  • METRICS_ENABLED=false in runtime environment.
  • Wrong scrape target in services/monitoring/prometheus/prometheus.tpl.yml (rendered to services/monitoring/prometheus/prometheus.yml).
  • Docker cannot reach the host (host.docker.internal missing or broken).

Recovery steps

Endpoint-level checks

  • Run curl -i http://127.0.0.1:8000/metrics and confirm status 200.
  • Confirm payload contains metric names such as http_requests_total.

Prometheus target checks

  • Open http://127.0.0.1:9090/targets.
  • If target is down, inspect PROMETHEUS_SCRAPE_TARGET, then re-run python tools/monitoring/render_prometheus_config.py and docker compose -f services/monitoring/docker-compose.observability.yml up -d to apply config.
  • Restart stack: docker compose -f services/monitoring/docker-compose.observability.yml down && docker compose -f services/monitoring/docker-compose.observability.yml up -d.

Grafana checks

  • Open Grafana (http://127.0.0.1:3001) and verify Prometheus datasource is healthy.
  • Generate test traffic to create fresh metrics:
for i in {1..20}; do curl -s http://127.0.0.1:8000/live  > /dev/null; done
      for i in {1..20}; do curl -s http://127.0.0.1:8000/ready > /dev/null; done

Exit criteria

  • Prometheus target is UP.
  • /metrics returns expected metrics.
  • Grafana dashboard panels show non-empty time-series data.

Escalation

If 30 minutes of triage didn't restore the scrape, or the failure mode isn't on the «Most common causes» list:

  1. Capture state — curl -i http://127.0.0.1:8000/metrics output, Prometheus targets page screenshot (http://127.0.0.1:9090/targets), the rendered services/monitoring/prometheus/prometheus.yml, relevant env vars (METRICS_ENABLED, PROMETHEUS_SCRAPE_TARGET), docker compose ps, and the recovery steps tried.
  2. Open a SEV-3 incident in the team backlog and link this runbook from it. Loss of scrape on a production-facing build jumps to SEV-2 — SLO alerts go silent the moment metrics stop flowing, so every minute extends the blind window.
  3. Page on-call (or, for solo work, the project incident commander) with the captured state and the last known-good ops config commit hash.
  4. If a rollback is config-only (env var, Prometheus target, Docker network), revert the offending commit before debugging further; observability gaps compound incident cost.

Follow-up

  • If scrape targets or metrics policy changed, update ops docs or an ADR.
  • If the same scrape failure repeats, add a note to this runbook.
