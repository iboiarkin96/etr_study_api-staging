The six-page spine

A spine is a fixed, ordered set of six pages that every service in this catalog is required to expose. The shape is borrowed from Backstage's service-catalog model and locked in by RFC 0004. The promise is simple: whichever service you land on — known or unknown to you — the same six headings live at the same six URLs, so navigation, on-call response, and review are predictable instead of a treasure hunt.

Why a fixed spine

  • Navigability. If you've read one entity card, you can read all of them. No prose-shaped docs that hide answers in unique places.
  • On-call ergonomics. An incident at 3 a.m. needs the runbook at a known URL — not "wherever the author felt like putting it".
  • Reviewability. Maintainers and reviewers can ask "is the Dependencies page up to date?" without first reverse-engineering the doc layout.
  • Completeness signal. A missing spine page is visible as a missing page — not as a service that simply chose to omit a topic.

The six pages

  1. 01

    Entity card

    The "what is this" page: owner, lifecycle (stable / experimental / deprecated), stack, source-of-truth repo, runtime surface (ports, endpoints, schedules), and links to the rest of the spine. The starting point for anyone arriving cold.

  2. 02

    Architecture

    How the service is built internally. C4 L2/L3 diagrams (PlantUML / Kroki, per ADR 0020), key containers and components, the principal request and data flows, and the non-obvious design choices with their rationale.

  3. 03

    API reference

    The contract the service publishes — HTTP endpoints, scrape / probe surface, recording rules, message-bus topics, CLI flags, or whatever shape "API" takes for this artefact. Auto-generated where possible, hand-curated where context matters.

  4. 04

    Runbooks

    Symptom → diagnosis → fix, written for the on-call who has never seen this service before. Each runbook starts with the alert text or user-visible symptom and ends with a verified recovery procedure. No prose-only explanations.

  5. 05

    Dependencies

    Upstream services and data this artefact reads from, downstream consumers that read from it, and the libraries / infra it ships against. The blast-radius map for changes and incidents.

  6. 06

    On-call

    Ownership and escalation: who pages first, who pages second, what hours, what SLOs back it, which alert routes flow here, and how to hand off. The single page that gets opened during an incident before anything else.

Conventions

  • Spine pages live under /services/<name>/ at fixed filenames (index.html, architecture.html, api-reference.html, runbooks.html, dependencies.html, on-call.html).
  • If a page does not yet exist for a given service, its chip in the catalog tile is rendered as a dashed placeholder with a tooltip — not silently omitted.
  • Pages that are not relevant for a given service (e.g. on-call for a build-only artefact) still exist as one-paragraph "not applicable, here's why" stubs. The shape never gets to bend.
