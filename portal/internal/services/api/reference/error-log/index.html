Error log resource

Learning-error resource hub: what it stores, the data contract, and the two operations. Resource: /api/v1/errors. Owner: Ivan Boyarkin. Last reviewed 2026-04-29.

Overview

Single entry point for the learning errors resource: what it stores, how it ties to the methodology, the data contract for clients, and the two HTTP operations that read and write it. Per-operation analyst specs are linked in the operations table below.

Business context

Error log captures one short learner-typed note per mistake or weak cue. It is the bridge between a failed retrieve and the next Transform pass: the learner sees the queue when planning their next session, and the in-conspectus error sidebar shows only entries tied to that specific note.

Methodology anchor: methodology · retrieve → error log. System-design anchor: persistence model — learning_errors and decision D5 (errors stay distinct from review-tag outcomes).

Domain model

Storage table: learning_errors. One row per error. Append-only — rows are never updated, only inserted. Optional foreign keys connect each error to the conspectus and the review log it was captured against.

FieldTypeDB columnRole
error_uuid string · uuid error_uuid (PK) Stable identity; UUIDv4 stamped server-side at insert.
owner_client_uuid string · uuid owner_client_uuid (FK → users) Resolved from the request's (system_user_id, system_uuid); never client-supplied.
message string message Free-text description (1–2000 chars). PII high.
conspectus_uuid string · uuid · nullable conspectus_uuid (FK, ON DELETE SET NULL) Optional link; conspectus must be owned by the same learner.
review_log_id integer · int64 · nullable review_log_id (FK) Optional link to the specific review row that surfaced the error.
created_at string · date-time (UTC) created_at Server stamp at insert.

Lifecycle

Error log is append-only: once written, an error row is read-only for the rest of its life.

  • Insert. POST /api/v1/errors writes one row in a single transaction.
  • Read. GET /api/v1/errors lists the learner's errors, optionally filtered by conspectus.
  • Edit / delete. Out of scope in v1 — see the per-operation Out of scope sections.
  • Cascade behaviour. Deleting a conspectus uses ON DELETE SET NULL on learning_errors.conspectus_uuid — the error survives, the link clears.

Technical overview

Planned implementation follows the internal API layering: middleware in services/api/app/main.py (auth, rate limit, observability), resource router, service validation, and repository persistence. Writes require idempotency (POST /api/v1/errors); the list endpoint is read-only.
