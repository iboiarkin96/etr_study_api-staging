Overview

Full replace of a user's editable fields. Unlike PATCH, every editable field must be supplied; missing fields reset to their defaults (e.g. username=null, is_row_invalid=0).

§1Business context

Useful when the integrator wants idempotent UPSERT semantics: send the desired full state every time, the server stores exactly that. Easier for sync clients that re-emit the same shape repeatedly.

§2User stories

  • US-USER-PUT-1. As an integrator, I want to push the authoritative profile state every time without diffing client-side.
  • US-USER-PUT-2. As an integrator, I want PUT to clear optional fields when omitted, so my CRM-style sync is symmetric.

§10Side effects

  • On success. One transaction: UPDATE users SET all editable fields to body values (defaults for missing optional fields), updated_at = now(); INSERT INTO idempotency_keys; emit update_user_succeeded (info).
  • On idempotent replay. No DB writes; emit update_user_idempotent_replay (info).
  • On error. Rollback.
  • Tables touched: users (write); reads FK references in systems, invalidation_reasons, timezones.

§11Idempotency & concurrency

  • Namespace: PUT /api/v1/user/{system_uuid}/{system_user_id}.
  • Key composition: Idempotency-Key + SHA-256 of canonical body.
  • Replay status: 200 with stored body.
  • TTL: 24 h.
  • Concurrency: v1 — last writer wins. Two concurrent PUTs from different keys both commit; updated_at reflects the last.

§13Edge cases checklist

  • Repeated PUT with identical body. Updates updated_at each time but row content unchanged. Idempotency layer prevents duplicates within TTL.
  • PUT then PATCH. Both succeed sequentially.
  • Body sets only required field. All optional fields reset to null/default.
  • PUT does not create. If the user does not exist, 404 — no upsert in v1.
  • Concurrent PUT vs PATCH. Last writer wins; readers may see intermediate states briefly.
