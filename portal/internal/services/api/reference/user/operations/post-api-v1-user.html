Overview

Create a new User row for the authenticated tenant. Identifies a learner inside the integrator's tenant via the composite natural key (system_user_id, system_uuid); the server generates a stable client_uuid used as the owner key on every downstream resource.

§1Business context

Every other endpoint resolves the learner via the composite key. Without this endpoint there is no surface to mint that key. The server-issued client_uuid isolates internal storage from integrator-supplied identifiers — they can rotate their identifiers later without touching our row graph.

§2User stories

  • US-USER-CREATE-1. As an integrator, I want to enrol a learner once and reuse the same composite key on every other call.
  • US-USER-CREATE-2. As an integrator, I want create to be idempotent so retries never produce duplicates.

§10Side effects

  • On success. INSERT INTO users; INSERT INTO idempotency_keys; emit create_user_succeeded (info).
  • On duplicate. Warn-level create_user_duplicate; no DB write.
  • On idempotent replay. Emit create_user_idempotent_replay (info); no DB writes.
  • Tables touched: users (write); reads FK references in systems, invalidation_reasons, timezones.

§11Idempotency & concurrency

  • Namespace: POST /api/v1/user.
  • Key composition: Idempotency-Key + SHA-256 of canonical body.
  • Replay status: 201 with stored body.
  • TTL: 24 h.
  • Concurrency: not applicable — natural-key uniqueness on the composite handles double-creates.

§13Edge cases checklist

  • Concurrent enrol with same composite key. First wins; second receives USER_101.
  • Enrol then immediately PATCH. Both succeed sequentially; updated_at reflects the PATCH.
  • Long unicode full_name. Validate by char count and byte size.
  • Timezone case sensitivity. IANA names are case-sensitive; europe/berlin → 422.
  • Idempotency replay across DB schema migration. Replay still returns the originally stored body.
