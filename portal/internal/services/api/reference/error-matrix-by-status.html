Reference API Conventions

Error matrix by status

How to mix shared COMMON_* errors with entity-prefixed codes (for example USER_*), and how to review routers by HTTP status.

Rule of thumb

  • Use COMMON_* when the same error applies everywhere.
  • Use entity prefixes (for example, USER_* for user) for domain-specific cases.
  • Never change the meaning of a published code + key pair.
  • Mirror declared responses in OpenAPI examples and tests.

Where to define reusable responses

  • services/api/app/errors/common.py and services/api/app/errors/user.py — one place for stable code, key, and default message values, plus the validation maps. Import from here; do not copy the strings.
  • services/api/app/openapi/responses.py — shared response blocks for routers.
  • services/api/app/openapi/examples/errors.py — OpenAPI example payloads built from the catalog.
  • services/api/app/api/v1/*.py — endpoint-level assembly of common and entity responses.

Wire-format examples per status class

What the integrator actually sees on the wire. Envelope shape is defined once in _shared/error-envelope.html; each example below conforms to it. Fields used: code (numeric machine ID), key (stable string), message (developer-facing prose), request_id (correlation), optional details[] for field-level errors.

400 — bad request (idempotency precondition)

HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": {
    "code": "COMMON_400",
    "key": "IDEMPOTENCY_KEY_REQUIRED",
    "message": "Missing required `Idempotency-Key` header for write operation."
  },
  "request_id": "req_01HZ3PXQ2K7B9TJYC6V0M4F8WN"
      }

401 — unauthenticated

HTTP/1.1 401 Unauthorized
Content-Type: application/json

{
  "error": {
    "code": "COMMON_401",
    "key": "SECURITY_AUTH_REQUIRED",
    "message": "Missing or invalid API key in header `X-API-Key`."
  },
  "request_id": "req_01HZ3PXQ2K7B9TJYC6V0M4F8WP"
      }

404 — entity not found / cross-tenant collapsed

Cross-tenant access deliberately returns the same envelope as a true miss — never 403 — per _shared/auth.html.

HTTP/1.1 404 Not Found
Content-Type: application/json

{
  "error": {
    "code": "CONS_404",
    "key": "CONSPECTUS_NOT_FOUND",
    "message": "Conspectus not found."
  },
  "request_id": "req_01HZ3PXQ2K7B9TJYC6V0M4F8WQ"
      }

409 — idempotency conflict

HTTP/1.1 409 Conflict
Content-Type: application/json

{
  "error": {
    "code": "COMMON_409",
    "key": "IDEMPOTENCY_KEY_REUSED_WITH_DIFFERENT_PAYLOAD",
    "message": "Idempotency key was already used with another payload."
  },
  "request_id": "req_01HZ3PXQ2K7B9TJYC6V0M4F8WR"
      }

422 — field-level validation (details array)

The envelope's top-level code/key is COMMON_000 / COMMON_VALIDATION_ERROR; per-field issues are enumerated in details[] using entity-prefixed keys so clients can render field-targeted error messages.

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/json

{
  "error": {
    "code": "COMMON_000",
    "key": "COMMON_VALIDATION_ERROR",
    "message": "Request validation failed.",
    "details": [
      {
        "field": "timezone",
        "code": "USER_007",
        "key": "USER_CREATE_TIMEZONE_INVALID",
        "message": "Timezone must be a valid IANA zone."
      },
      {
        "field": "full_name",
        "code": "USER_004",
        "key": "USER_CREATE_FULL_NAME_TOO_LONG",
        "message": "full_name must be ≤ 256 characters."
      }
    ]
  },
  "request_id": "req_01HZ3PXQ2K7B9TJYC6V0M4F8WS"
      }

500 — unexpected server error

The code/key pair is intentionally omitted on 5xx — clients should not key UI off it. Only request_id is stable, for the integrator to quote when filing a ticket.

HTTP/1.1 500 Internal Server Error
Content-Type: application/json

{
  "error": {
    "message": "Internal server error."
  },
  "request_id": "req_01HZ3PXQ2K7B9TJYC6V0M4F8WT"
      }

Router review checklist (status-first)

  1. List all response statuses declared in router responses={...}.
  2. Mark each status as COMMON or ENTITY.
  3. Ensure COMMON statuses come from services/api/app/openapi/responses.py.
  4. Ensure ENTITY statuses use entity-prefixed code space and mapped examples.
  5. Validate with tests and make verify.
