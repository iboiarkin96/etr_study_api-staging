Pagination

Convention for list endpoints under /api/v1/**. Per-operation specs describe their own filters and ordering; the pagination shape itself is shared and lives here.

v1 default — bounded list with a hard cap. Endpoints return at most 100 items per call, ordered deterministically, with no cursor. Cursor pagination is added per-endpoint when usage data shows the cap is hit; until then, listing endpoints rely on the cap + filters being sufficient.

Default — bounded list

  • Cap: 100 items per response. The integrator cannot raise it.
  • Order: deterministic, stated by the endpoint (e.g. next_review_at ascending, then conspectus_uuid as a stable tiebreaker for total order).
  • Filters: per-endpoint query parameters (e.g. slot, due_before).
  • Response shape: bare JSON array; no envelope, no meta, no total. Adding a wrapper would be a breaking change.

Why no wrapper in v1. Wrapping costs every client a layer of unwrapping; v1 lists are bounded by the cap and not cursorable. When a list grows beyond the cap and needs cursors, that endpoint gains a wrapper as an additive change behind a query flag, or as a v2 path.

Cursor pagination — added per endpoint

When an endpoint genuinely needs more than 100 items, it gains cursor pagination. The shape:

  • Query parameters: cursor (opaque string, optional) and limit (1–100, default 50).
  • Response: object { items: [...], next_cursor: string | null }. next_cursor is non-null when more items exist.
  • Cursors are opaque base64 of an internal struct (last seen sort key + tiebreaker). Clients must not parse them.
  • Cursors are not stable across schema migrations of the sort key; integrators are warned in the endpoint description.

Adding cursor pagination to an existing array-returning endpoint is a breaking change unless gated behind a new query flag. See versioning for the rules.

Stable ordering rules

  1. Pick a domain-meaningful primary sort (next_review_at, created_at, …).
  2. Add a tiebreaker on a unique column (UUID lexicographic) so the order is a total order. Without this, cursors are unstable and integration tests are flaky.
  3. Document the order on the operation page (Section 7 query params or Section 8 response).
  4. Never rely on insertion order or DB physical order; both can change.

Errors

All standard middleware errors apply (401, 429, …). Pagination-specific errors are minimal in v1:

  • 422 COMMON_000 COMMON_VALIDATION_ERROR — invalid limit (out of range) or invalid filter value.
  • 422 on cursor parse failure — the client should treat any cursor it did not just receive from this endpoint as invalid.

Anti-patterns

  • Returning a total count on every page. Forces a full-table scan; we don't do it. Clients that need totals build them out-of-band.
  • Offset/limit for cursor-style endpoints. Offset shifts when data mutates between pages; cursor is the right primitive. Offset is fine only on tiny, immutable lists.
  • Client-decoded cursors. Forbidden. Cursors are opaque and may change shape without notice.
  • Mixing wrapper and bare array in one endpoint. Choose one and stick with it for that endpoint's lifetime.

Traceability
