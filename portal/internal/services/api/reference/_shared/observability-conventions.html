Observability conventions

Naming and shape rules for log events, metrics, traces, and alerts emitted by API handlers. Per-operation specs only declare their events — the names, fields, labels, and severity follow the conventions on this page. Governing ADRs: 0023 (logging), 0009, 0011.

Log events

Naming

  • Lifecycle of a request emits one request_done info line from middleware. Always.
  • Operation events follow {resource}_{verb_past_tense}[_{qualifier}]: conspectus_reviewed, user_created, conspectus_review_idempotent_replay.
  • Failures use a verb that names the failure mode, not the HTTP status: conspectus_review_revision_conflict (good) vs conspectus_review_409 (bad).

Severity

LevelUse forExamples
infoSuccessful business outcomes; expected client errors.conspectus_reviewed, conspectus_review_not_owned (4xx).
warningMisuses of the API by a client that we want to notice but not page on.idempotency_payload_mismatch, conspectus_review_revision_conflict, validation_error, request_body_limit_exceeded.
errorInvariant violations; bugs we want to fix.conspectus_review_schedule_missing.
exceptionUncaught — global handler; always paired with a stack trace and request_id.request_failed.

Required structured fields

  • request_id — propagated from X-Request-Id header (server-generates if absent). Carried in every log line in a request scope.
  • method, path_template, status, elapsed_ms — on the middleware request_done line.
  • For business events: the canonical resource ID (conspectus_uuid, client_uuid, …) and the relevant state fields named by the operation spec.

Forbidden contents

  • API keys in any field — mask to last 4 chars before logging.
  • Idempotency keys verbatim — log the SHA-256 prefix (idempotency_key_hash), not the raw value.
  • PII beyond the canonical IDs — no email, no learner-typed prose from dense_paragraph / bullets.
  • Stack frames in non-exception levels — only the global handler attaches them.

Metrics

Naming

  • HTTP-layer metrics from middleware: http_requests_total (counter), http_request_duration_seconds (histogram).
  • Business metrics: {resource}_{verb}_{kind} — kind ∈ {total, duration_seconds, in_progress}. Examples: conspectus_review_total, conspectus_review_duration_seconds.
  • Database timing inherits the shared db_operation_duration_seconds histogram with labels {operation, table}.

Standard labels

  • method, path_template, status_code — middleware metrics.
  • Business metrics: high-cardinality fields go to logs, not metric labels. Allowed labels are bounded enums (tag, slot_from, slot_to, status_code). Never label by user UUID or free-text.

Histogram buckets

Default duration buckets: 0.005, 0.01, 0.025, 0.05, 0.1, 0.2, 0.5, 1, 2, 5 (seconds). Per-operation specs may override; document the override in the spec.

Traces (OpenTelemetry)

  • One root span per HTTP request, named after the route ({method} {path_template}).
  • Service boundaries get child spans named {resource}.{verb} (e.g. conspectus.review.apply).
  • DB calls get a span named db.{operation} with attribute db.statement (parameterised, never inlined values).
  • Span attributes: the same canonical IDs as logs; never log full payloads.
  • trace_id and span_id are added to every log line automatically.

SLO targets and error-budget policy live in ADR 0011. Per-operation specs declare their alert triggers; this page only lists the global ladder.

Per-operation checklist

For Section 14 of an operation spec to pass the lint, the following must be filled:

  • Log events: name, level, when, required structured fields.
  • Metrics: name, kind, labels (bounded), buckets if non-default.
  • Traces: at least the apply-span and the DB child span.
  • Alerts: severity (page / ticket / info), trigger, threshold, sustain window.
