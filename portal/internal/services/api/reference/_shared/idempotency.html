Idempotency — shared rules

Single source of truth for how the internal API enforces idempotent writes. Per-operation specs link here for the global rules and only describe their own deltas (namespace value, key composition, replay status). Governing ADR: ADR 0006.

When idempotency is required

Required for every state-mutating HTTP method (POST, PATCH, PUT, DELETE) under /api/v1/**, unless the operation is documented as read-only.

Not used for GET and HEAD. The header is silently ignored on read endpoints — no error, no validation.

The Idempotency-Key header

HeaderTypeRequiredNotes
Idempotency-Key string Yes (for writes) Client-generated token that uniquely identifies one logical write attempt. Printable ASCII (no whitespace-only). Length 1–128. Recommended: UUID v4, or {operation_id}-{stable_business_key}-{rfc3339_timestamp}. Keep the same key for retries; rotate when the intent changes.

How the server keys deduplication

The composite storage key in idempotency_keys is:

  • owner_client_uuid — derived from (system_user_id, system_uuid) in the body, OR from the API-key principal for endpoints that do not carry a user composite key.
  • endpoint_path — operation-specific value, declared on the per-operation spec page. Always includes the HTTP method; for parameterised paths, the concrete resource UUID is substituted into the namespace so that each resource has its own bucket.
  • idempotency_key — the verbatim Idempotency-Key header value.
  • payload_hash — SHA-256 of the canonicalised JSON body (see Canonical body).
  • response_status, response_body, response_headers — stored on first success for replay.
  • created_at — used to enforce the 24-hour TTL.

Scope. Uniqueness is on (owner_client_uuid, endpoint_path, idempotency_key). Two different users can use the same key without colliding (per system-design · idempotency_keys scope).

Canonical body for hashing

The payload_hash is SHA-256 of the request body normalised to a canonical JSON form so semantically equal payloads collide deterministically:

  • Object keys sorted ascending by code point.
  • Insignificant whitespace stripped (no spaces between tokens).
  • Strings preserved verbatim including unicode escapes.
  • Numbers serialised by the standard library JSON encoder (no trailing zeros).
  • Arrays preserve order — order is part of the payload identity.

Implementations should reuse a single canonicalize(body) -> bytes helper across all writers to avoid drift. Drift between clients is acceptable — the integrator is responsible for sending the same bytes on retries.

TTL and clean-up

Default TTL: 24 hours from created_at. Rows older than TTL are eligible for deletion by the cleanup job — the running handler ignores them. Per-operation specs may shorten or extend the TTL via NFR; document the deviation explicitly on the operation page.

Idempotency vs optimistic concurrency

These are distinct concerns and a spec must answer both:

  • Idempotency protects against the same logical intent being retried by the same caller (network retry, client crash). Replay returns the original outcome.
  • Optimistic concurrency protects against two distinct intents racing on the same resource. Replay is irrelevant — the second intent is rejected with a state-version conflict (e.g. schedule_revision guard on review).

An operation that needs both must implement both: the idempotency key handles retries, a row-version column handles racing intents. They never substitute for each other.

Client recipe

  1. Generate one Idempotency-Key per logical write intent (e.g. before the user clicks "Save").
  2. Persist the key locally until the request succeeds or you decide to abandon it.
  3. On any retry — network error, 5xx, timeout — resend the same key with the same body.
  4. If you change the body, generate a new key. Sending the old key with a new body returns 409.
  5. On 409 COMMON_409, treat as a programmer mistake (you reused a key with different bytes). Do not silently retry.
