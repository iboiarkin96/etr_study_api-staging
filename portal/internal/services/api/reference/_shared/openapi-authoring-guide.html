OpenAPI authoring guide

How to translate an internal analyst-spec page into the public OpenAPI document at services/portal/public/reference/api/openapi-baseline.json. Per-operation specs include a Section 16 — OpenAPI authoring hints; this page covers the cross-cutting rules. Governing ADRs: 0007, 0025, 0026.

Audience. The developer who will write or update the OpenAPI JSON after the spec is approved. The systems analyst writes the internal page; the developer translates it. Drift between the two is a defect.

Direction of authority

Internal spec is the source of truth for requirements. The OpenAPI file is the contract for external integrators — it must reflect what the internal spec says, narrowed for an external audience. Ambiguity is resolved by consulting the internal page; if the OpenAPI is unclear, fix it from the spec, not the other way around (per ADR 0026).

Naming & conventions

  • operationId: {verb}{Resource} in camelCase (createUser, reviewConspectus, listDueConspectuses). Stable forever — clients use it as a method name in generated SDKs.
  • Tags: PascalCase resource (Conspectus, User, ErrorLog). One tag per operation.
  • Path templates: lower-snake-case identifiers in braces ({conspectus_uuid}) — match the internal field names.
  • Component schemas (under components.schemas): PascalCase (ConspectusResponse, ReviewTag, ScheduleRevision, Uuid, Timestamp, ErrorEnvelope).
  • Field names in JSON: snake_case verbatim from the internal field tables (and field conventions).

Component reuse — the $ref policy

Always prefer $ref over inline schemas. Define each schema once under components.schemas, then reference everywhere.

  • Required components for every spec: ErrorEnvelope, Uuid, Timestamp.
  • Resource components: define the "view" objects (ConspectusResponse, UserResponse) once and reuse for 200 / 201 bodies across endpoints.
  • Body components: define one per write endpoint (ReviewConspectusBody, CreateUserBody); do not inline.
  • Enum components: define once (ReviewTag, Slot) and reference in both request and response.

Examples — what to attach where

Every 200 / 201 response gets at least one examples.success.value. Every declared error code gets its own example under responses.{status}.content.application/json.examples.{key}.value. Example bodies are copied verbatim from the internal spec (Section 7 / 8 / 12) — minor edits like ellipses only.

Why verbatim: the example becomes the contract on Swagger UI; it also doubles as the fixture for contract tests. Copy-paste, do not paraphrase.

What to hide from external integrators

  • Internal log-event names (conspectus_reviewed, …).
  • Internal metric / trace span names.
  • ADR references in description.
  • Internal IDs (owner_client_uuid) — clients only see system_user_id / system_uuid.
  • Schedule-policy internal identifiers (etr_methodology_four_slot) — these may leak as opaque strings in responses, but their semantics are not part of the public contract.
  • Test-plan and observability sections (sections 14, 15) — internal-only.

Versioning

URL-based versioning at /api/v1/**. Breaking changes require a new major version path. Additive changes go into the current version. See versioning and ADR 0004.

Lint & CI gates

  • spectral lint on the OpenAPI document — strictness configured per ADR 0007.
  • schemathesis contract tests run against the live API for every operationId: no 5xx on schema-conforming input; response examples match live shape.
  • Internal cross-check (scripts/spec_consistency.py) — every internal spec maps to one OpenAPI operationId and vice versa; every error code/key in section 12 exists in error catalog.
