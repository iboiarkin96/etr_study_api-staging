Error envelope — shared contract

Every non-2xx HTTP response from /api/v1/** carries a uniform JSON envelope. The envelope shape is governed by ADR 0003; this page is the operational summary linked from per-operation Error catalog tables.

Schema

FieldTypeRequiredNotes
codestringYes Stable, machine-readable error code grouped by domain: COMMON_…, USER_…, CONS_…, ERR_…. Pattern ^[A-Z]+_(?:[0-9]{3}|[0-9]{1,4})$. Append-only; renames forbidden.
keystringYes Symbolic key suitable for i18n / log search: SCREAMING_SNAKE_CASE. Pairs 1:1 with code. Append-only.
messagestringYes Human-readable English message. Not for end-user display — clients translate key. 1–256 chars. Wording may change without ADR; the key/code stays.
sourcestringYes Logical originator: {resource}.{action} or middleware identifier. Pattern ^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)+$.
detailsobject | arrayOptional Operation-specific structured payload. Validation: array of {loc, msg, type}. Business conflicts: object with relevant identifiers. No PII, no secrets, no stack traces.

Examples

Simple — missing idempotency header

HTTP/1.1 400 Bad Request
Content-Type: application/json

{
"code": "COMMON_400",
"key": "IDEMPOTENCY_KEY_REQUIRED",
"message": "Missing required `Idempotency-Key` header for write operation.",
"source": "middleware.idempotency"
}

With details — validation error

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/json

{
"code": "COMMON_000",
"key": "COMMON_VALIDATION_ERROR",
"message": "Request validation failed.",
"source": "conspectus.review",
"details": [
{
"loc": ["body", "tag"],
"msg": "value is not a valid enumeration member; permitted: 'easy', 'hard', 'forgot'",
"type": "type_error.enum"
}
]
}

With details — business conflict

HTTP/1.1 409 Conflict
Content-Type: application/json

{
"code": "CONS_409",
"key": "CONSPECTUS_REVIEW_REVISION_CONFLICT",
"message": "Schedule revision changed since you read it; re-fetch and retry.",
"source": "conspectus.review",
"details": {
"expected_schedule_revision": 7,
"actual_schedule_revision": 9
}
}

Authoring rules for new errors

  1. Pick the right domain prefix (COMMON_ / USER_ / CONS_ / ERR_).
  2. Allocate the next free numeric suffix per error catalog. Never reuse a retired number.
  3. Define code, key, English message in services/api/app/errors/<domain>.py.
  4. Pick a source that already exists, or extend the convention only by ADR.
  5. Document the new error on the operation spec page (Section 12 — Error catalog) and run the consistency lint.
  6. details shape is operation-local — describe it in the spec, attach an OpenAPI example.

Anti-patterns

  • Branching on HTTP status alone. Always read code / key; HTTP status loses information (e.g. four different 404s).
  • PII or secrets in message or details. The error envelope is logged and emailed to integrators — keep it sterile.
  • Stack traces or stack frames. Forbidden in production responses. Include the X-Request-Id instead so support can find the trace server-side.
  • Localised message. Always English; localisation is the client's job using key.
  • Renaming an existing code or key. Forbidden — clients build dashboards and alerting on these strings.
  • Free-form details per call. Define the shape per error key and stick to it; otherwise dashboards break.
