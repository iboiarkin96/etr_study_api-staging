Non-functional requirements — SLO, observability, packaging

IDRequirementDescription
NFR-1 Performance Typical requests complete under 300 ms p95 with a local DB and light load. Capacity figures are illustrative — see system design — load and capacity.
NFR-2 Reliability Writes are transactional; failures roll back and return stable errors per ADR 0003.
NFR-3 Maintainability Strict layering (routers → services → repositories); contract and endpoint tests are mandatory per ADR 0002.
NFR-4 API governance OpenAPI and error contracts evolve in an additive, backward-compatible manner per ADR 0004.
NFR-5 Observability Structured logs (ADR 0023); Prometheus on /metrics (config-gated); /health and /ready; optional local stack per ADR 0009; SLO guidance in ADR 0011.
NFR-6 Security by default API-key auth, per-route rate limits, body-size limit, CORS allowlist, security headers per ADR 0005. Field-level routing surface in api-reference · security defaults.
NFR-7 Idempotency Critical writes support safe retries via Idempotency-Key and persisted deduplication (ADR 0006).
NFR-8 Packaging Production Dockerfile and OCI workflow per ADR 0015; CD pipeline per ADR 0021.

Hybrid SQL + object storage — failure modes

When conspectus bodies are externalised (cue sheet stored in object storage rather than inline JSON) there is no distributed transaction between the RDBMS and the external store. Recovery relies on status fields (sync_status, content_sha256) and compensating actions.

ScenarioDetectionMitigation
SQL committed; blob write failed sync_status in pending / failed; missing hash or ETag Retry upload with the same Idempotency-Key; do not delete the SQL row from GC.
Blob written; SQL rolled back Orphan object key Key prefixing, TTL GC, no user-visible attachment until SQL commits.
Drift between SQL and blob content_sha256 mismatch Fail read or serve stale per policy; repair job re-fetches or re-uploads the canonical body.

Architectural rationale (why hybrid, what SQL must anchor) lives in api · Architecture — content storage; this page covers the operational side only.

Diagram regeneration

All architecture diagrams (C4 L1 / L2 / L3, ER, sequences) are PlantUML sources under services/portal/internal/uml/. After editing a .puml file, regenerate the SVG: 

make docs-fix
# or, equivalently
      .venv/bin/python tools/docs/regenerate_docs.py

The script uses Kroki (https://kroki.io/plantuml/svg) — network is required only when sources actually change (the input-hashes.json fingerprint cache skips unchanged files). Conventions: ADR 0020.
