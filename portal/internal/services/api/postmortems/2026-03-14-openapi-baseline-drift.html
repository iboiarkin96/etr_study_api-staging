OpenAPI baseline drift — silent field rename slipped to PR review

A field on the UserResponse schema was quietly renamed in a feature branch. Local make verify never ran the OpenAPI diff because the developer ran make test only. CI caught it 4 minutes into the PR build — but the integrator-facing baseline would have shipped with a breaking change had the contract test not existed. No production impact; this is the canonical example of why ADR 0007 exists.

Incident summary

On 2026-03-14, during work on the User profile feature branch (feat/user-profile-display-name), the UserResponse Pydantic model field full_name was renamed to display_name in services/api/app/schemas/user.py. The change was semantically reasonable inside the codebase. It was a breaking change on the wire: every integrator's deserialiser would have failed on the next response. The author ran make test locally — all tests green, including the user-resource integration tests, because the tests were updated in the same commit. The CI pipeline ran make verify on the PR, which includes the OpenAPI baseline diff (scripts/openapi_check.py). The diff failed; the PR build went red; the change was reverted to additive (added display_name alongside full_name, with full_name kept and deprecated). Total time from breakage commit to detection: ~14 minutes. No code reached main.

Impact

  • Users affected: zero. The break was caught pre-merge.
  • Integrators affected: zero. No artefact shipped.
  • SLO budget: not consumed.
  • Engineer time: ~45 min between the failed CI run and the revert PR landing. Cost: one half-day's planned work pushed by one day.
  • Counterfactual: without the CI contract diff, the rename would have shipped on the next release. Every integrator deserialiser branching on data.full_name would have raised a missing-field error on the first request after deploy. Detection would have come from the integrator, not us.

Root cause — contributing factors

Real incidents almost always have several contributing factors, not one cause. Listed below in order of leverage (closing the first would have prevented the others).

  • Local-loop didn't include the contract diff. make test runs unit + integration; it does not run make openapi-check. The developer's mental model was "tests pass = safe to push". The contract test only ran in CI.
  • Rename felt cosmetic. full_name → display_name was discussed as a naming improvement in a code review the previous week. The change reads as a rename inside the codebase, not as a contract break. Pydantic field renames carry no semantic warning.
  • The tests were updated in the same commit. Updating tests to match the new field name made the local test suite green, which removed the last signal the developer had. The tests asserted the new shape, not the contract.
  • OpenAPI baseline diff output is dense. The failure message lists every changed line in the JSON baseline; for a one-field rename that is ~30 lines of diff. The "what should I do?" wasn't immediately clear from the output; the developer had to read runbook 0007 to understand the fix path.

Resolution

  • Immediate. The rename was converted to an additive change: both full_name and display_name are now returned. full_name is documented as deprecated; removal will follow the standard deprecation flow (see deprecation policy).
  • Durable. A pre-push git hook was added that runs make openapi-check when services/api/app/schemas/**/*.py changes (see action items). This moves the signal from CI back to the developer's terminal, where it can be acted on without the round-trip.
  • No code rollback required. The break never reached main.

Detection

The OpenAPI baseline diff in make verify (CI). Specifically: scripts/openapi_check.py compares the runtime app.openapi() output against the committed services/portal/public/reference/api/openapi-baseline.json and exits non-zero on any difference. This is the script that enforces ADR 0007.

The contract test caught the change. Nothing else would have. The unit and integration tests were green; the type-check (mypy) was green; the lint (ruff) was green. Without the baseline diff, this lands.

Lessons learned

  • "Tests pass" ≠ "contract safe." A test suite that is updated alongside the production code asserts the new shape, not the contract. The contract diff is the only check that asserts continuity with the previously shipped artefact.
  • The cheapest place to catch a contract break is the developer's terminal. CI catches it in 4 minutes; a pre-push hook catches it in 4 seconds. We should not need to wait for CI for a check that runs locally.
  • Renames look cosmetic; on a wire schema they are breaking. This is obvious in retrospect and was the failure mode here. Worth saying out loud in the docs so the next engineer doesn't have to learn it the same way.
  • Dense diff output is a usability bug. The error message is a tool surface; making it actionable (lead with the recovery path, not the raw diff) is part of what makes the check effective.
