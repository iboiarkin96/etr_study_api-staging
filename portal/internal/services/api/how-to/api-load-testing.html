API How-to

Run an API load test

How to drive a configurable mix of HTTP requests against the API to surface latency, error rates, and SLO signals in metrics dashboards.

Why run load-style traffic

  • Unit tests (make test) verify the contract. A load run shows the 2xx / 4xx / 5xx mix, the latency distribution, and the error shape in Grafana or Prometheus — signals you cannot get from pytest alone.
  • Load runs are not a substitute for unit tests, and they mutate state: user-create scenarios write to the database, so run them only against local or non-production environments.

Environment setup

  1. Virtualenv: make venv && make install (or equivalent).
  2. .env from env/example (make env-init), with valid APP_HOST and APP_PORT.
  3. Migrations: make migrate.
  4. For GET-by-id scenarios (optional scenarios/user/get.py), set ROTATE_SYSTEM_USER_IDS to real system_user_id values and adjust SHARE_OF_GROUP between user/create.py and user/get.py so they sum to 1.0 per user group.

Simulating HTTP 500 (local and non-prod only)

Normal routes rarely return a steady stream of 500s. To emit 5xx from the same API process, so http_requests_total counts them, set the following in .env:

LOADTEST_HTTP_500=true

After restarting the app, only the internal route GET /__loadtest/http500 is available (hidden from OpenAPI). Do not enable this flag in production.

Starting the API

From the repository root (adjust the path):

cd /path/to/study_app
source .venv/bin/activate
      make run

If you also want Grafana and Prometheus locally, run docker compose -f services/monitoring/docker-compose.observability.yml up -d in another terminal while make run keeps the API in the first. Details: Local development.

In a second terminal, run the traffic generator (see below).

Generator configuration (Python)

Scenarios live under tools/load_testing/scenarios/. See tools/load_testing/README.html for the full contract (with diagrams). Summary:

  • scenarios/weights.py — GROUP_WEIGHTS (e.g. user, observability_5xx).
  • Each scenario module exports GROUP, SHARE_OF_GROUP, MIX (sums to 1.0 within the file), and SCENARIOS (scenario key → callable building a request).
  • Validation failures for user create are typically 422 — see user_payload.py and user/create.py.
  • HTTP 500 without touching the DB: small share via scenarios/observability/http500.py and LOADTEST_HTTP_500=true.

Commands

Plan only (no network):

python -m tools.load_testing.runner --dry-run

Full run (default 100 requests; set LOAD_TEST_BASE_URL / LOAD_TEST_API_KEY if needed):

python -m tools.load_testing.runner --total-requests 200

Optional delay between requests:

python -m tools.load_testing.runner --total-requests 200 --delay-ms 50

Observability

Start the stack before or alongside the API (docker compose -f services/monitoring/docker-compose.observability.yml up -d). After the traffic run, open Grafana (port from .env, default 3001) and the Study App Observability dashboard; in Prometheus, /alerts when SLO rules are configured.

Verify

  • Generator run completes without uncaught exception; the runner's summary line shows the total request count and the per-scenario breakdown.
  • Prometheus shows scrape target UP for the API for the entire run window; no gap in the latency / RPS / 5xx panels on Grafana.
  • Latency and 5xx panels stayed within SLO targets (or the breach was the explicit point of the test, with the breach window recorded).
  • If simulate_500 scenario was enabled: LOADTEST_HTTP_500=true on the API side AND the generator drove non-zero traffic to /__loadtest/http500; observed 5xx rate matches the configured ratio.
  • No rate-limit-induced 429 mixed in with the expected error classes (would skew the run); if so, increase API_RATE_LIMIT_REQUESTS_LOADTEST or lower --delay-ms.
  • Report or session note attached to the team backlog with traffic shape, observed SLIs, and any defects opened.

Troubleshooting

  • Connection refused. The API isn't running or the base_url is wrong. Start the API with make run-loadtest-api (which sets the high-limit env vars) and double-check the generator's base_url.
  • Script errors on «#». A comment was passed as an argument. Re-run without a trailing # …; the runner doesn't strip shell comments.
  • Expected 200, got 404 on the user GET. Fix ROTATE_SYSTEM_USER_IDS in user/get.py — the generator is asking for ids that don't exist in the seeded set.
  • 429 Too Many Requests. Lower the intensity through --delay-ms, or raise the dev rate limit (API_RATE_LIMIT_REQUESTS_LOADTEST) and restart the API.
