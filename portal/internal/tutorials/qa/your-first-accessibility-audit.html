Your first accessibility audit

Run an automated scan on one portal page, then walk through it with the keyboard. The point is to feel the two halves of an a11y check — what a scanner catches and what only a human notices — not to certify the page. Real audits run the same two passes on a wider surface.

⚙️ 3 phases ⏱ 25–35 min 📚 First-time 💻 Local stack

– Before you start

  • Portal serves locally on http://127.0.0.1:3000.
  • Node 20+ with npx on PATH (you'll run axe via npx, no global install).
  • Chrome or Firefox with the DevTools you normally use.
  • You've skimmed the accessibility checklist — accessibility once so the categories are familiar.

0 Goal

  • Pick one page — default is the internal hub.
  • Run axe-core against it and capture the JSON output.
  • Walk the page with the keyboard only (no mouse, no trackpad).
  • Write a one-page report at services/portal/internal/scratch/a11y-run-<date>.md with axe findings + keyboard findings + verdict.

1 Automated scan with axe

Run axe-core against the page (URL escaping matters — keep the quotes):

npx @axe-core/cli "http://127.0.0.1:3000/services/portal/internal/index.html" \
        --save services/portal/internal/scratch/axe-raw.json

First run downloads the CLI. When it finishes, the terminal prints a summary by impact level (critical / serious / moderate / minor) and the JSON is on disk.

Pull the violations list out for the report:

jq '[.[0].violations[] | {id, impact, help, nodes: (.nodes | length)}]' \
        services/portal/internal/scratch/axe-raw.json

✓ Check your progress

The JSON file exists and the jq command prints an array of objects (possibly empty). Copy that array into your report under a heading "Automated findings". If empty, write "axe: 0 violations" — that is also a result.

2 Keyboard-only walk

Close DevTools, put your mouse away. Open the page and use only Tab, Shift+Tab, Enter, Space, and arrow keys. From the accessibility checklist §3 (keyboard):

  • Focus visible: every focusable element shows a clear outline; no :focus { outline: none } leftovers.
  • Tab order is logical: top-to-bottom, left-to-right, matches visual order — no jumps into the footer mid-content.
  • No focus traps: you can leave every interactive widget (modal, dropdown, drawer) with Esc or Tab.
  • Skip link: the first Tab from the URL bar reveals a "Skip to main content" link.
  • All interactive elements reachable: every chip, button, link, toggle — including the theme toggle — can be activated from the keyboard.

✓ Check your progress

Five bullets above each marked PASS or FAIL in your report under "Keyboard findings". For each FAIL, write "at element X, saw Y, expected Z" — same shape as a defect.

3 Verdict

Add a verdict block at the bottom of your report:

## Verdict
- Page: <url>
- axe: N critical, N serious, N moderate, N minor
- Keyboard: N findings
- **Overall**: accept / conditional / block
- Top three things to fix:
1. ...
2. ...
3. ...

Treat any critical or serious axe finding as a block-level verdict. The first time through, this calibrates your sense of what severity actually means.

✓ You're done

  1. Report file exists under scratch/.
  2. It has two findings sections (automated, keyboard) and a verdict with a concrete top-3.
  3. Top three to-fix items name the element and expected behaviour, not just the rule id.

That is the a11y audit loop — axe scan → keyboard walk → verdict + top-3. Real audits widen the surface and add colour-contrast, zoom, and screen-reader passes; the shape is the same.

✚ Want more? Add a screen-reader sanity pass

Optional and skippable on a first run — many testers don't have a screen reader installed yet. Open the audited page with the OS screen reader: macOS Cmd+F5 (VoiceOver), Windows Ctrl+Alt+N (NVDA if installed). Listen to the first 60 seconds of reading and confirm:

  • The page announces a single <h1>.
  • Headings descend without skipping levels (no h2 → h4).
  • Landmarks are present (banner, navigation, main, contentinfo).
  • Images are either described or marked decorative — no raw filenames read aloud.

If you ran this pass, add a "Screen-reader findings" section to your report with four PASS/FAIL bullets and a "Screen reader: N findings" line above the Overall in the verdict block.
