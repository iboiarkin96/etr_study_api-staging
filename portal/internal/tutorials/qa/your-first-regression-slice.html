Your first regression slice

Pick one feature, pull its critical-path tests, run them against a candidate build, write a slice report with pass-rate and blockers. The point is to feel the regression loop — scope → pick → run → verdict — not to certify a release. Real regression runs widen the surface and add automation; the shape is the same.

⚙️ 4 phases ⏱ 30–45 min 📚 First-time 💻 Local stack

– Before you start

  • make venv && make install succeeded and make test passes on main.
  • You've finished QA · Tester onboarding Phase 1–2 (you can run pytest, you can read a defect report).
  • You know what a critical path is — the steps a user must complete to get value, not every feature in the app.
  • You've skimmed the API endpoint acceptance checklist once so you recognise the test categories.

0 Goal

  • Pick one feature with a clear user-visible promise — default is user create + read (the v1 user CRUD slice).
  • Identify 3–5 critical-path tests for that feature and pin them as the regression slice.
  • Run the slice against the current build, record pass/fail per test, time the run.
  • Write a slice report at services/portal/internal/scratch/regression-slice-<date>.md with scope, results, blockers, and a single-line verdict (green / amber / red).

1 Scope and pick

Open the test tree and find tests that touch the chosen feature. For the default user create + read slice:

find tests -name "test_user_*.py" -type f
pytest --collect-only tests/api/v1/test_user_create.py -q
pytest --collect-only tests/api/v1/test_user_read.py -q

From the collected list, pick 3–5 tests that match the critical path. The filter rule is sharp:

  • Happy path first. If the test exercises the success case end-to-end (request → response → DB state), it belongs in the slice.
  • One representative error per failure mode. One 4xx that proves validation works; one 4xx that proves authz works. Not every variant.
  • No edge cases. Empty strings, max-length payloads, Unicode boundaries — those are unit-test concerns. The regression slice asks "does the user-visible promise still hold?".

Write the picks in the report under "Scope":

# Regression slice: user create + read

**Build under test**: <commit-sha> on branch <name>
**Run by**: <handle> · <date>

## Scope (3–5 tests)
1. `tests/api/v1/test_user_create.py::test_create_user_ok`
2. `tests/api/v1/test_user_create.py::test_create_user_validation_error`
3. `tests/api/v1/test_user_read.py::test_get_user_ok`
4. `tests/api/v1/test_user_read.py::test_get_user_not_found`

✓ Check your progress

The Scope section has between 3 and 5 named tests, each pointing at a real nodeid that pytest --collect-only emitted. If you picked 8, the slice is too wide — cut it down; you can always run a wider sweep next time.

2 Run the slice

Run only the picked tests, with timing, and capture the output:

make test-one path="tests/api/v1/test_user_create.py::test_create_user_ok \
  tests/api/v1/test_user_create.py::test_create_user_validation_error \
  tests/api/v1/test_user_read.py::test_get_user_ok \
  tests/api/v1/test_user_read.py::test_get_user_not_found" 2>&1 \
  | tee services/portal/internal/scratch/regression-slice-raw.txt

If make test-one isn't defined on your local checkout (older clones, or a fork without the helper), drive pytest directly — same selection, same capture: 

pytest --durations=0 \
  tests/api/v1/test_user_create.py::test_create_user_ok \
  tests/api/v1/test_user_create.py::test_create_user_validation_error \
  tests/api/v1/test_user_read.py::test_get_user_ok \
  tests/api/v1/test_user_read.py::test_get_user_not_found 2>&1 \
  | tee services/portal/internal/scratch/regression-slice-raw.txt

Paste the per-test result into the report under "Results":

## Results

| #  | Test                                          | Result | Time   | Notes               |
|----|-----------------------------------------------|--------|--------|---------------------|
| 1  | test_create_user_ok                           | PASS   | 0.12s  | —                   |
| 2  | test_create_user_validation_error             | PASS   | 0.04s  | —                   |
| 3  | test_get_user_ok                              | PASS   | 0.08s  | —                   |
| 4  | test_get_user_not_found                       | FAIL   | 0.06s  | 500 instead of 404  |

**Pass-rate**: 3 / 4 (75%) · **Wall time**: 0.30s

✓ Check your progress

The table has one row per scoped test; result column is PASS or FAIL (no «partial», no «flaky» on a first run); pass-rate and wall-time are computed. If a test errored before assertion (import error, fixture failure), mark it BLOCKED and note the cause — that's a different signal than a real fail.

3 Triage blockers

For every FAIL or BLOCKED row, write one blocker entry in the report. Same shape as a defect — the slice report feeds the backlog, not the inbox.

## Blockers

### B1 · test_get_user_not_found — 500 instead of 404
- **Reproduce**: `GET /api/v1/users/00000000-0000-0000-0000-000000000000`
- **Expected**: HTTP 404 with `{"detail":"User not found"}`
- **Actual**: HTTP 500, traceback in server log
- **Severity**: high — known endpoint, wrong status code, user-facing
- **Suspected cause**: missing `HTTPException` raise in the not-found branch
- **Owner**: <name> · **Target fix**: before next slice

Don't speculate beyond what the test output and one quick log glance support — guesses get hard to retract. If the cause isn't obvious in 5 minutes, write "cause unknown — needs dev triage" and move on.

✓ Check your progress

Every non-PASS row has a numbered blocker entry. Each entry names the reproduce step, expected, actual, severity, and an owner. If a single blocker explains multiple failed tests, group them under one entry — slice reports prefer one cause over duplicate noise.

4 Verdict

Close the report with a single-line verdict the next reader can act on without re-reading the whole file:

## Verdict

- **Slice**: user create + read · 4 tests · 0.30s wall time
- **Pass-rate**: 75% (3 / 4)
- **Verdict**: 🔴 RED — B1 is a user-visible regression in a known endpoint.
- **Recommendation**: hold the candidate build; re-run slice after B1 lands.
- **Top three to fix**:
  1. B1 · GET /users/{id} 500 path → 404
  2. (none)
  3. (none)

The traffic-light is a calibration, not a vote: any high-severity blocker on a user-facing path is RED regardless of pass-rate; a single low-severity blocker on a corner case is AMBER; all-PASS is GREEN.

✓ You're done

  1. Report file exists under scratch/ with Scope, Results, Blockers, Verdict.
  2. Verdict is one of GREEN / AMBER / RED with one explicit reason.
  3. Every FAIL is traceable to a numbered blocker entry.

That is the regression loop — scope → run → triage → verdict. Real slices widen scope, automate the run, and feed verdicts into release gates; the shape is the same.

↺ Want to start over?

rm services/portal/internal/scratch/regression-slice-*.md services/portal/internal/scratch/regression-slice-raw.txt

The slice report lives only in scratch/ — no tests were modified, no defects were filed. The repo stays as-is.
