Your first API checklist run

Pick one running endpoint, walk it through the API acceptance checklist, and record the result. The point is to feel the checklist as a working instrument — request shape, error shape, observability, contract — not to accept a real release. Real acceptance reviews use the same checklist on a larger surface.

⚙️ 4 phases ⏱ 30–40 min 📚 First-time 💻 Local stack

– Before you start

  • make run is up on http://127.0.0.1:8000.
  • curl and jq on PATH.
  • You've skimmed the API endpoint acceptance checklist — API endpoint acceptance once so the section names are familiar.

0 Goal

  • Pick one read endpoint — default is GET /api/v1/health/live.
  • Run all four phases of the API endpoint acceptance checklist against it: request, response, errors, observability.
  • Produce a one-page run report at services/portal/internal/scratch/qa-run-<date>.md with each item ticked or flagged.
  • Decide a verdict: accept / conditional / reject — and write down why.

1 Request shape

Hit the endpoint with the simplest possible request:

curl -i http://127.0.0.1:8000/api/v1/health/live

From the API endpoint acceptance checklist §1, verify:

  • Method is GET (idempotent — safe to retry).
  • Path is plural-noun-style and versioned (/api/v1/...).
  • No required query params for a baseline call.
  • Content-Type on the response is application/json.
  • 200 OK with a body — not 204 No Content for a status probe.

✓ Check your progress

Five bullets above each marked PASS or FAIL in your report. If anything FAILs, write the actual value next to it — that's what a defect would cite.

2 Response shape

Inspect the body:

curl -s http://127.0.0.1:8000/api/v1/health/live | jq .

From the API endpoint acceptance checklist §2:

  • Top-level is a JSON object (not a bare array, not a string).
  • Field names are snake_case.
  • Timestamps (if any) are ISO 8601 with timezone.
  • Status field, if present, uses a closed vocabulary ("ok" | "degraded" | "down") — no free-form strings.
  • Schema matches the OpenAPI definition. Verify:
curl -s http://127.0.0.1:8000/openapi.json \
              | jq '.paths["/api/v1/health/live"].get.responses["200"]'

✓ Check your progress

All bullets PASS, and the OpenAPI snippet's $ref resolves to a schema whose required fields match what you saw in the body. Note any extra or missing fields in the report.

3 Error paths

Provoke at least three error classes:

# 1. Wrong method
curl -i -X POST http://127.0.0.1:8000/api/v1/health/live

# 2. Wrong path (typo)
curl -i http://127.0.0.1:8000/api/v1/health/liv

# 3. Bad accept header
curl -i -H "Accept: text/xml" http://127.0.0.1:8000/api/v1/health/live

From the API endpoint acceptance checklist §3:

  • Method-not-allowed returns 405 (not 404 or 500).
  • Typo path returns 404 with a JSON body, not an HTML error.
  • Unsupported accept returns 406 or falls back to JSON — never an unhandled stack trace.
  • Every error body is the same canonical shape (e.g. { "detail": "..." }).
  • No internal paths, stack frames, or secrets leak in error responses.

✓ Check your progress

Five bullets PASS in your report. For each failure, copy the actual response body (truncated to 200 chars) so the defect would be reproducible from the report alone.

4 Observability + verdict

Tail the app log while you re-issue the happy-path request:

# in a second terminal
tail -f logs/app.log &

              curl -s -o /dev/null -w '%{http_code}\n' http://127.0.0.1:8000/api/v1/health/live

From the API endpoint acceptance checklist §4:

  • One structured log line per request (not zero, not many).
  • Line has method, path, status, duration_ms, request_id.
  • Repeat the request with a header -H "X-Request-Id: tutorial-abc123" — the log line should echo it.

Write the verdict at the bottom of your report:

## Verdict
- **Accept** / Conditional accept / Reject
- Failing items:
- (none) | §X bullet Y: actual was <...>
- Notes:
- one or two sentences on observed risk

✓ You're done

  1. The report file exists under scratch/.
  2. Every checklist bullet has PASS / FAIL written next to it.
  3. The verdict line is filled in.

That is the API-acceptance loop — pick endpoint → walk request, response, errors, observability → verdict. Real release acceptance runs the same loop on every changed endpoint; only the surface widens.
