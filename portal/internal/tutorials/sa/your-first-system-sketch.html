Your first system sketch — C4 in PlantUML

Draw a C4 Context and Container diagram for one slice of the study app, render them in the portal, and publish a one-page note linking both. The point is to feel the sketch loop — pick scope → write PlantUML → render → publish — not to architect anything. Real reviews use the same loop with deeper diagrams.

⚙️ 4 phases ⏱ 30–45 min 📚 First-time 💻 Local stack

– Before you start

  • Portal serves locally on http://127.0.0.1:3000.
  • make docs-fix has run before (you've used the docs pipeline).
  • You've skimmed SA practice — C4 in PlantUML once; you know what Context vs Container means.
  • Diagrams: PlantUML only (per ADR 0020 — C4 / PlantUML conventions) — never Mermaid.

0 Goal

  • Two PlantUML files: services/portal/internal/scratch/sketch-context.puml and scratch/sketch-container.puml.
  • Both render in the portal's PlantUML pipeline (no syntax errors).
  • One markdown note scratch/sketch-note.md that embeds both diagrams and answers two questions: what's in scope and what's the riskiest dependency.

Scope for the sketch: "a learner finishes a quiz and sees their weekly streak update". Small, end-to-end, no infra surprises.

1 C4 Context — who, outside the system, talks to it?

Create scratch/sketch-context.puml:

@startuml sketch-context
!include <C4/C4_Context>
title Context — quiz finish updates weekly streak

Person(learner, "Learner", "Returning user, finishing a quiz")
System(study_app, "Study App", "Quizzes, streaks, analytics")
System_Ext(analytics, "Analytics warehouse", "Event store for product metrics")

Rel(learner, study_app, "Finishes a quiz, sees streak")
Rel(study_app, analytics, "Emits quiz_finished + streak_updated events", "HTTPS / batch")

@enduml

One Person, one System, one System_Ext. Three relationships max. If you need more, you're sketching a Container diagram, not Context.

✓ Check your progress

File saves cleanly. Open it in the PlantUML preview (your editor's extension, or npx -y plantuml-server): the diagram renders with one person on the left, one system in the middle, one external on the right.

2 C4 Container — what's inside the box?

Create scratch/sketch-container.puml:

@startuml sketch-container
!include <C4/C4_Container>
title Container — inside Study App

Person(learner, "Learner")

System_Boundary(study_app, "Study App") {
Container(web, "Web client", "Browser SPA", "Renders quiz + streak UI")
Container(api, "API", "FastAPI", "Quiz finish + streak read endpoints")
ContainerDb(db, "Primary DB", "SQLite/Postgres", "Quizzes, attempts, streaks")
Container(events, "Event emitter", "Background worker", "Publishes to analytics")
}

System_Ext(analytics, "Analytics warehouse")

Rel(learner, web, "Finishes quiz")
Rel(web, api, "POST /quiz/finish", "HTTPS/JSON")
Rel(api, db, "Writes attempt, recalculates streak")
Rel(api, events, "Enqueues quiz_finished")
Rel(events, analytics, "Publishes events", "HTTPS/batch")
Rel(web, api, "GET /streak/weekly", "HTTPS/JSON")
@enduml

Four containers inside the boundary, one external. Each relationship names a verb and a protocol. The Container diagram zooms one level into the System node from Phase 1.

✓ Check your progress

File renders without syntax errors. Every container is connected to at least one other container — none are floating. The external analytics warehouse sits outside the System_Boundary box.

3 Publish the note that ties them together

Create scratch/sketch-note.md:

# Sketch — quiz finish updates weekly streak

## Scope
- In: learner finishes a quiz, attempt persists, streak recalculates, UI shows new streak, event flows to analytics.
- Out: streak across weeks, badges, leaderboards, multi-device sync.

## Context
![Context](sketch-context.puml)

## Container
![Container](sketch-container.puml)

## Riskiest dependency
The event emitter is the riskiest link: it's async, it has no caller waiting
for an answer, and a failure there silently breaks analytics attribution
without breaking the learner UX. **Mitigation to consider next**: retries with
DLQ, plus a smoke metric on `quiz_finished` event rate vs `quiz/finish` 2xx
rate.

## Decisions deferred
- Sync vs async streak update.
- Storage shape for streak (denormalized counter vs derived view).

Scope is two lines. Riskiest dependency is one paragraph with one mitigation. Deferred decisions are bullets, not paragraphs. This shape is what a senior SA would skim in 60 seconds.

✓ Check your progress

File has all five sections (Scope / Context / Container / Riskiest / Deferred). Both ![...](...) references match the .puml filenames you created. Open the markdown in your editor's preview — the diagrams should appear if your editor supports PlantUML rendering.

4 Render and verify in the portal 

make docs-fix

The docs pipeline picks up new PlantUML files under scratch/ and produces rendered SVGs alongside them (check for sketch-context.svg and sketch-container.svg in the same folder after the run). Last log line: docs-fix: OK.

✓ You're done

  1. Two .puml files, both render to SVG.
  2. One .md note with all five sections, embedding both diagrams.
  3. You can answer in one sentence: "the riskiest part of this slice is X because Y".

That is the sketch loop — scope → Context → Container → note with risk + deferred. Real reviews extend by one or two C4 levels (Component, Code) and add ADRs for the deferred decisions.
