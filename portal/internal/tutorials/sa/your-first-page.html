Your first portal page

Clone a minimal template, register it in the sidebar, regenerate, and watch it appear in portal search. The publishing loop is identical for every page in this portal — once you've done it for a throwaway page, you can do it for any reference, how-to, or explanation page. Conventions per section are covered separately in the layout how-to.

⚙️ 5 phases ⏱ 15–20 min 📚 First-time 💻 Local stack

– Before you start

  • Portal serves locally — opening http://127.0.0.1:3000/services/portal/internal/index.html shows the Internal hub.
  • GNU make and Python 3.11+ on PATH (needed by make docs-fix).
  • Clean working copy on a branch you don't mind dirtying.

0 Goal

By the end of this tutorial you will have:

  • One new page at services/portal/internal/scratch/my-first-page.html.
  • One sibling entry in nav-tree-internal.json placing it in the sidebar.
  • The page appearing in portal search when you type its title.

The page lives under scratch/ on purpose — production pages live under reference/, how-to/, or explanation/ and follow per-section conventions documented in the layout how-to. This tutorial only teaches the file → nav → regen loop.

1 Copy the minimal template

Create the directory and the page in one go:

mkdir -p services/portal/internal/scratch

Create services/portal/internal/scratch/my-first-page.html:

<!doctype html>
<html lang="en" data-portal="internal" data-theme="light">
<head>
<meta charset="utf-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title>My first page · Scratch · Study App Internal</title>
<link rel="stylesheet" href="../../../frontend/portal/assets_v2/runtime/internal/entry.css">
</head>
<body data-page-type="reference" class="docs-shell">
<header class="topbar" role="banner">
<a class="topbar__brand" href="../index.html">Study App · Internal</a>
<nav class="topbar__breadcrumbs" aria-label="Breadcrumb">
<ol class="docs-breadcrumbs__list">
<li class="docs-breadcrumbs__item"><a class="docs-breadcrumbs__link" href="../index.html">Internal</a></li>
<li class="docs-breadcrumbs__item" aria-current="page">My first page</li>
</ol>
</nav>
</header>
<aside data-component="sidebar"
data-nav-src="../../../frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json"
aria-label="Section navigation"></aside>
<main>
<article class="docs-prose">
<h1>My first portal page</h1>
<p class="lede">Throwaway page created while doing the portal-page tutorial.</p>
</article>
</main>
<script type="module" src="../../../frontend/portal/assets_v2/runtime/internal/entry.js"></script>
</body>
</html>

Note the relative paths: ../../../frontend/... — three levels up because the page sits at depth 3 (services/portal/internal/scratch/). Every portal page uses this same shell; only the depth changes.

✓ Check your progress

Open http://127.0.0.1:3000/services/portal/internal/scratch/my-first-page.html. The page renders with the topbar and your h1. The left sidebar is empty — that's expected, we haven't registered the page yet.

2 Make it yours

Change the <title>, the <h1>, the lede, and (optionally) add one extra section so the page has more than one heading. Tiny example: 

<h2>What I learned</h2>
<ul>
<li>Where portal pages live.</li>
<li>How they wire into the sidebar.</li>
</ul>

✓ Check your progress

Refresh the browser. Your edits show up. Right-side TOC picks up the new <h2> — that confirms the prose styles are applied.

3 Register the page in the sidebar

Open services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json. Find the very end of the root children array (the line that reads ] right above the final closing } of the file) and add a comma after the last element, then paste this group: 

,
{
"id": "scratch",
"label": "Scratch",
"icon": "🧪",
"children": [
{
"id": "scratch-my-first-page",
"label": "My first page",
"href": "/services/portal/internal/scratch/my-first-page.html"
}
]
}

✓ Check your progress

Reload any internal portal page (e.g. the Internal hub). The sidebar now ends with a Scratch group containing My first page. Click it — your page opens with the sidebar populated.

Nothing in the sidebar? The two likeliest causes: a missing comma between entries, or a stray trailing comma. Both break JSON. Run python -m json.tool < services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json > /dev/null — silence means valid; any error names the line.

4 Regenerate artifacts

The sidebar reads nav-tree-internal.json directly, but search is pre-built. Regenerate it:

make docs-fix

This rebuilds the Pagefind bundle at services/frontend/portal/pagefind/ (gitignored — built locally for browser preview; CI rebuilds it before every Pages deploy), the internal catalog under services/portal/internal/catalog/, and fixes any redirects that fell out of sync. Last log line should read DOCS-FIX: SUCCESS.

✓ Check your progress

make docs-fix exited with DOCS-FIX: SUCCESS. git status shows your new HTML page and the nav-tree edit (the Pagefind bundle does not appear — it is gitignored by design).

5 Verify in portal search

Reload the portal in your browser. Open the Internal hub and type my first page into the search box in the topbar. Your page should appear in the results within one or two keystrokes; clicking it lands on the page with the sidebar populated and the breadcrumb ending at My first page.

Empty result set? The Pagefind bundle didn't rebuild — re-run make docs-fix and confirm services/frontend/portal/pagefind/ was touched after your last edit (the directory itself is gitignored, but you can inspect ls -l modification times). If the bundle is fresh and search still misses, the page may not match any rule in tools/docs/build_pagefind_index.py's kind_for() — add a rule there.

✓ You're done

  1. Page renders at scratch/my-first-page.html.
  2. Sidebar shows the new Scratch → My first page entry.
  3. Portal search returns the page on its title.

That is the full publishing loop — every new portal page, regardless of mode, follows file → nav-tree entry → make docs-fix → verify in search.

↺ Want to start over?

The scratch directory is throwaway by design. To clean up:

rm -rf services/portal/internal/scratch
git checkout -- services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json
make docs-fix

git checkout restores the JSON to its committed state, so the Scratch group disappears from the sidebar after the next regen.
