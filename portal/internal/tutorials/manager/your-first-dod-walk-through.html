Your first DoD walk-through

Take one almost-done story, walk every Definition of Done item, mark pass/fail/N-A, write a one-line verdict and the next blocker. The point is to feel the DoD loop — read each item → check against artefacts → mark → summarise — not to certify a release. Real walk-throughs cover bigger slices; the shape is the same.

⚙️ 4 phases ⏱ 25–35 min 📚 First-time 📝 Paper-only

– Before you start

  • A pen and paper, or any plain-text editor.
  • One story that feels done — code merged, tests green, but not yet shipped or accepted. (Default candidate below if you don't have a real one.)
  • You've finished Manager · Your first sprint slice so the DoR / DoD vocabulary is familiar.
  • You've read the team's DoD list once — or are using the default below.

0 Goal

  • Pick one story — default is a fictional "Add user-profile avatar upload".
  • Walk a 6-item Definition of Done — the default below, or your team's actual DoD if you have one in hand. Mark each item PASS / FAIL / N-A with evidence.
  • Identify the first blocker — the item that, if cleared, lets the next two items be re-checked cheaply.
  • Write a one-page report at services/portal/internal/scratch/dod-walk-<date>.md with the table, verdict, and the single next blocker.

1 Frame the story and the DoD

At the top of the report, name the story and pin the DoD list you're walking:

# DoD walk-through: <story title>

**Story ID**: <ticket id>
**Outcome**: <one-sentence user-visible promise>
**Author**: <dev handle>
**Walk by**: <your handle> · <date>

## DoD list (6 items)

1. Code merged to `main`
2. Unit tests added/updated, all green
3. API spec page updated (if API touched)
4. Observability: one log line + one metric per new code path
5. Documentation page updated (Tutorial / How-to / Reference / Explanation)
6. Reviewed by ≥ 1 person outside the author's pair

If your team already has a written DoD, swap it in here — the walk-through shape stays the same. The 6 above is a generic stand-in; most real DoDs run 6–12 items and add things like migration up/down, ADR/RFC linkage, or demo capture.

✓ Check your progress

The story header is filled in and the DoD list is exactly what your team uses (or the default 10). Each item is checkable — passes a "could a second person verify this in 30 seconds?" test. If an item reads "quality is high", rewrite it before walking; vague items produce vague walk-throughs.

2 Walk every item

For each item, write one line: result, evidence, and (for FAIL) the gap. No item is skipped — N-A is a valid result, but it requires a one-sentence justification.

## Walk

| # | Item                         | Result | Evidence                            | Gap (if FAIL)                   |
|---|------------------------------|--------|-------------------------------------|---------------------------------|
| 1 | Code merged                  | PASS   | PR #1234 merged 2026-05-18          | —                               |
| 2 | Unit tests                   | PASS   | 12 tests in test_avatar.py          | —                               |
| 3 | API spec page                | FAIL   | reference/api/post-avatar.html      | response 413 not documented     |
| 4 | Observability                | FAIL   | one log line only, no metric        | metric `avatar_upload_total`    |
| 5 | Documentation                | FAIL   | how-to/users/upload-avatar missing  | new how-to page needed          |
| 6 | Outside reviewer             | PASS   | @reviewer approved                  | —                               |

Two rules keep the walk honest:

  • Evidence is a pointer, not an opinion. A file path, a PR number, a screenshot URL — not "looks good to me".
  • FAIL beats partial. If an item is half-done, write FAIL and put the missing half in the Gap column. "Partial" hides work.

✓ Check your progress

Every row has a result and either evidence or (for FAIL) a one-line gap. No ? in the result column. If you can't verify an item in 30 seconds, mark it FAIL — needs verification and continue; the walk-through surfaces gaps, it doesn't fill them.

3 Pick the single next blocker

Multi-blocker reports get ignored. The discipline is to name one item to fix next — the one that unblocks the most other items, not the most painful one.

## Next blocker

**Item**: #5 — Documentation page missing
**Why this one**: writing the new how-to forces #3 (API spec) to be updated
(the how-to links to it) and surfaces what #4 (observability) needs to expose,
so clearing #5 makes the next walk-through faster on three items.
**Owner**: <author handle>
**Target**: next sprint review

If two items genuinely tie for "unblocks the most", pick the cheaper one. The point is to start, not to optimise.

✓ Check your progress

The Next blocker section names one item, one owner, one target. The "Why this one" line points at the items the blocker unblocks. If you wrote "all the FAILs are important" — that's the failure mode this phase prevents; pick one and defend it.

4 Verdict

Close the report with a single-line verdict the next reader can act on without re-reading the table:

## Verdict

- **Story**: <title>
- **DoD score**: 3 PASS / 3 FAIL (out of 6 walked)
- **Verdict**: 🟠 AMBER — story is not Done; three doc/observability gaps.
- **Next blocker**: #5 — Documentation page missing (clears #3 + #4 cheaply)
- **Re-walk by**: <date>

The traffic-light maps to action: 🟢 GREEN — accept and ship; 🟠 AMBER — accept conditionally with a named follow-up; 🔴 RED — story is not Done, return it. A single FAIL on a user-visible promise (a real bug, a security gap) is RED regardless of the score.

✓ You're done

  1. Report file exists under scratch/ with story header, DoD table, next blocker, verdict.
  2. Every DoD item has a result; no ? rows.
  3. The next-blocker line names one item, owner, and date.

That is the DoD loop — frame story → walk items → pick blocker → verdict. Real walks cover wider slices and feed sprint review; the shape is the same.
