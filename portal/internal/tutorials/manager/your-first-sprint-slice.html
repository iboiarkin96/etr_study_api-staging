Your first sprint slice

Take one trivial goal, map it from outcome to two stories, and write each one against the Definition of Ready. The point is to feel the shape-the-work loop — outcome → impact → story → DoR — not to commit to delivery. Real planning uses the same loop on bigger goals.

⚙️ 4 phases ⏱ 30–45 min 📚 First-time 📝 Paper-only

– Before you start

  • You can edit Markdown — no app, no DB, no auth required.
  • You've skimmed impact mapping and Definition of Ready / Done once each.

0 Goal

  • One outcome statement — measurable, one sentence.
  • One impact map: 1 goal → 2 actors → 3 impacts → 3 deliverables (the example below splits one actor across two impacts; that's the healthy minimum).
  • Two user stories, each passing Definition of Ready.
  • One sprint-slice markdown file at services/portal/internal/scratch/sprint-slice-<date>.md holding all four sections above.

The goal is intentionally trivial — "let a returning user see how many quizzes they've completed this week" — so the loop is in focus, not the domain content.

1 Outcome statement

Create the slice file and write one outcome line at the top:

# Sprint slice — week of <YYYY-MM-DD>

## Outcome
By end of sprint, returning users will see their weekly quiz count on the home screen,
and at least 30% of returning sessions will open the home screen before any other page.

From the impact-mapping practice, a usable outcome has three traits:

  • Who changes behaviour (returning users).
  • What they will see or do (weekly quiz count on home).
  • How we'll know (≥30% of returning sessions open home first).

✓ Check your progress

Your outcome line names all three traits. If you find yourself writing "improve engagement" or "delight users", trim — those are vibes, not outcomes.

2 Impact map (one goal, two actors, two impacts, two deliverables)

Below the outcome, add the map:

## Impact map
### Goal
Returning users open the app more often.

### Actor 1 — returning learner (last seen 1–14 days)
- Impact: feels rewarded for showing up
- Deliverable A1: weekly streak count on home (read-only)
- Impact: knows what to do next
- Deliverable A2: "continue last topic" CTA on home

### Actor 2 — analytics team
- Impact: can attribute home-screen reopens to the streak surface
- Deliverable B1: emit `home_home_screen_viewed` event with `returning_user=true`

Two actors, three deliverables total — that's a healthy first map. If actor 2 has no deliverable, the impact has no evidence and the outcome can't be measured.

✓ Check your progress

Every deliverable maps back to exactly one impact, and every impact to exactly one actor. No deliverable is orphaned. No actor is impact-less.

3 Two stories that pass Definition of Ready

Pick two deliverables from the map and write them as user stories:

## Story 1 — Weekly streak on home
**As a** returning learner
**I want** to see how many quizzes I completed this week
**so that** I get a sense of momentum before I open anything else.

### Acceptance criteria
- Home screen shows "X quizzes this week" (0 to 99+).
- Counter is read-only; tapping it navigates to the quiz history page.
- Count resets to 0 every Monday 00:00 in user timezone.

### Out of scope
- Streak across weeks, badges, sharing.

### Dependencies
- None — counter can be derived from existing `quiz_attempts` table.

### Definition of Ready
- [x] Outcome link clear (impact map A1)
- [x] Acceptance criteria written
- [x] Out-of-scope listed
- [x] Dependencies named
- [x] Smallest useful slice (no streak history, no badges)

Repeat for story 2 (analytics event, deliverable B1). Both should be in the same file so DoR check can be done in one scan.

✓ Check your progress

Both stories have the five DoR checkboxes ticked. Re-read each story cold: could a developer start tomorrow without asking you a clarifying question? If no, the failing item is your DoR gap.

4 Sanity check — RACI in one line per story

At the bottom of the file, add a RACI table:

## RACI
| Story | R (does) | A (accountable) | C (consulted) | I (informed) |
|-------|----------|-----------------|---------------|--------------|
| 1     | dev      | manager (you)   | sa, qa        | analytics    |
| 2     | dev      | manager (you)   | analytics     | sre          |

One R per story. One A per story (almost always you). Multiple C/I are fine. If you have two R's or two A's, the story is too big — split it (see RACI practice).

✓ You're done

  1. Slice file exists under scratch/ with all four sections.
  2. Every story passes DoR (5/5 checkboxes).
  3. RACI rows have one R and one A each.

That is the planning loop — outcome → impact map → DoR-ready stories → RACI. Real sprint slices follow the same shape; the difference is bigger outcomes and more actors, not new ceremony.
