Your first migration — Alembic up and down

Generate a trivial migration, apply it, verify it in SQLite, then roll it back to a clean state. The point is the upgrade/downgrade loop — not the column itself. Real schema work uses the same shape with thicker upgrade() / downgrade() bodies.

⚙️ 4 phases ⏱ 20–30 min 📚 First-time 💻 Local stack

– Before you start

  • make venv && make install ran successfully.
  • SQLite CLI (sqlite3) on PATH.
  • .env present (so make migrate can resolve SQLITE_DB_PATH).
  • You've applied the baseline at least once: make migrate has been run before.

0 Goal

  • A new Alembic revision file under services/api/alembic/versions/ adding one nullable column tutorial_note to timezones.
  • alembic upgrade head applies it cleanly.
  • sqlite3 .schema timezones shows the column.
  • alembic downgrade -1 removes it and the DB matches baseline again.

We pick timezones because the baseline already created and seeded it — touching one column on a seeded table is the smallest realistic migration.

1 Generate the revision file

Create a hand-written revision (faster than autogenerate for a one-column change):

.venv/bin/alembic revision -m "tutorial_add_note_to_timezones"

Alembic prints the path it created, e.g. services/api/alembic/versions/<hash>_tutorial_add_note_to_timezones.py. Note the file name — you will edit it in the next step.

✓ Check your progress

git status shows one new untracked Python file in services/api/alembic/versions/. The file's revision string matches the hash in its filename; down_revision points to the previous head (e.g. "20260411_0001").

2 Fill in upgrade and downgrade

Replace the empty stubs Alembic generated with these two bodies:

from alembic import op
import sqlalchemy as sa

def upgrade() -> None:
    op.add_column(
        "timezones",
        sa.Column("tutorial_note", sa.String(length=120), nullable=True),
    )

def downgrade() -> None:
    with op.batch_alter_table("timezones") as batch:
        batch.drop_column("tutorial_note")

The batch_alter_table wrapper is the SQLite-safe way to drop a column (SQLite cannot drop columns in-place; Alembic rewrites the table for you). On PostgreSQL op.drop_column alone would work — the batch form is portable.

✓ Check your progress

git diff services/api/alembic/versions/ shows only the two function bodies (plus the auto-generated header Alembic wrote). Python syntax is valid: .venv/bin/python -c "import ast; ast.parse(open('<file>').read())" prints nothing.

3 Apply and verify 

make migrate

Last log line reads ✓ Migrations applied. Now verify the column exists on the live SQLite DB (path comes from .env's SQLITE_DB_PATH — replace <db> with that path): 

sqlite3 <db> ".schema timezones"

Output lists the table definition and includes a tutorial_note VARCHAR(120) line. Alembic's revision pointer should also have advanced: 

.venv/bin/alembic current

Prints your new revision id with (head) after it.

✓ Check your progress

All three commands succeed in order: make migrate ends with ✓, .schema shows the new column, alembic current reports your revision as head.

4 Roll back to baseline

Tutorial done — now undo the migration so the repo is clean:

.venv/bin/alembic downgrade -1

Alembic re-runs downgrade(). Confirm the column is gone:

sqlite3 <db> ".schema timezones"

The tutorial_note line should be absent. .venv/bin/alembic current now reports 20260411_0001 (head) — the baseline.

✓ You're done

  1. alembic current shows the baseline as head.
  2. .schema timezones has no tutorial_note.
  3. The revision file you created is the only untracked change.

That is the migration loop — revision → upgrade/downgrade bodies → make migrate → verify → downgrade. Real schema work follows the same shape; the difference is what goes into the two function bodies and how carefully you preserve data.

↺ Want to start over?

.venv/bin/alembic downgrade -1   # back to the baseline, if you haven't already
rm services/api/alembic/versions/*tutorial_add_note_to_timezones.py

If something went sideways and you can't downgrade cleanly, delete the SQLite file at SQLITE_DB_PATH and run make migrate again — the baseline rebuilds the schema and re-seeds timezones.
