Your first background task

Wire one trivial route that schedules a write-to-file after the response, call it, and watch the side effect land. The point is to feel the background-task loop — endpoint returns first, side effect happens after — not to build a real worker. Real workers use the same shape with retries, queues, and persistence.

⚙️ 4 phases ⏱ 25–35 min 📚 First-time 💻 Local stack

– Before you start

  • make venv && make install succeeded.
  • The dev server starts with make run (http://127.0.0.1:8000 by default).
  • Repo on a clean branch — you'll revert at the end.
  • You've finished SWE · Your first endpoint or feel comfortable adding a route under services/api/app/api/v1/.

0 Goal

  • Add one new route — POST /api/v1/ping-bg — that schedules a background write.
  • The background function writes a one-line entry to scratch/bg-ping.log with a timestamp.
  • Hit the endpoint with curl and confirm: response returns first, log line appears within a second.
  • Revert the change with git checkout . so the repo is clean again.

1 Wire the route

Create a new file at services/api/app/api/v1/ping_bg.py with the route and the background function in one place — it's a tutorial; in real code the worker function would live in services/api/app/services/.

from datetime import datetime, UTC
from pathlib import Path

from fastapi import APIRouter, BackgroundTasks

router = APIRouter()

SCRATCH_LOG = Path("services/portal/internal/scratch/bg-ping.log")

def _write_ping(note: str) -> None:
    """Background worker — append one timestamped line to the log file."""
    SCRATCH_LOG.parent.mkdir(parents=True, exist_ok=True)
    line = f"{datetime.now(UTC).isoformat()} bg-ping note={note}\n"
    SCRATCH_LOG.write_text(SCRATCH_LOG.read_text() + line if SCRATCH_LOG.exists() else line)

@router.post("/ping-bg")
def ping_bg(background_tasks: BackgroundTasks, note: str = "hello") -> dict[str, str]:
    """Schedule a background write, return immediately."""
    background_tasks.add_task(_write_ping, note)
    return {"status": "scheduled", "note": note}

Register the router in services/api/app/api/v1/__init__.py (or wherever the v1 router is composed — match the pattern already used for hello):

from app.api.v1.ping_bg import router as ping_bg_router

api_v1_router.include_router(ping_bg_router, tags=["dev-tutorial"])

✓ Check your progress

make run starts without import errors. Visit http://127.0.0.1:8000/docs — POST /api/v1/ping-bg is listed under the dev-tutorial tag. If you see an import error, check the router registration matches the existing hello pattern.

2 Call the endpoint

From a second terminal, call the new route:

curl -X POST "http://127.0.0.1:8000/api/v1/ping-bg?note=first" -w "\n%{time_total}s\n"

You should see two things, in this order:

  1. Response body: {"status":"scheduled","note":"first"} followed by a tiny time_total (typically < 50 ms).
  2. A new line appended to services/portal/internal/scratch/bg-ping.log within a second of the response.

Call it a second time with a different note to confirm appending works:

curl -X POST "http://127.0.0.1:8000/api/v1/ping-bg?note=second"
cat services/portal/internal/scratch/bg-ping.log

✓ Check your progress

The log file has two timestamped lines, in call order, each tagged with the matching note= value. The HTTP responses came back before the file was written — that's the whole point of the background-task pattern. If the log file is empty, the most common cause is a permissions issue on scratch/ — create the directory by hand and retry.

3 Read the trade-offs

FastAPI's BackgroundTasks is the lightest possible scheduler. Before you reach for it in real code, register the limits:

  • Same process. The worker runs in the same uvicorn process. A crash mid-task loses the work; an OOM takes the API with it.
  • No retries. If the worker raises, FastAPI logs and moves on. You write retry logic yourself, or you don't have it.
  • No durability. Tasks scheduled before a deploy are gone after the deploy.
  • No observability hooks. No queue depth, no per-task timing, no alerts — only what you log yourself.

That maps to a clear rule: BackgroundTasks is for fire-and-forget side effects that are safe to lose — emit-a-webhook, write-an-audit-line, refresh-an-in-memory-cache. The moment "what if it fails" matters, the answer is a real queue (Celery, RQ, Arq, SQS) and a separate worker process — not this.

✓ Check your progress

You can name, in one sentence each, the four limits and the rule that distinguishes BackgroundTasks from a real queue. Write the rule on an index card if it helps — you'll re-read it the next time someone proposes BackgroundTasks for a job that can't be lost.

4 Revert

Tutorial done — clean the repo so nothing leaks into a real change.

git checkout -- app/api/v1/__init__.py
rm -f app/api/v1/ping_bg.py
rm -f services/portal/internal/scratch/bg-ping.log

Re-run make verify from a clean tree to confirm nothing else drifted.

✓ You're done

  1. git status is clean.
  2. make verify passes.
  3. You can answer "when would I use BackgroundTasks vs a real queue?" in one sentence.

That is the background-task loop — route schedules → handler returns → worker runs → side effect lands. Real workers add durability, retries, and isolation; the loop shape is the same.
