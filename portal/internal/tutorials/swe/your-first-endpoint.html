Tutorial SWE · Foundations

Your first endpoint, end-to-end

Boot the project, write a tiny read endpoint, document it, and ship a green make verify. The point is to feel the full dev loop on the actual codebase — Make targets, local DB, sidebar, search — not to learn how to ship production code.

⚙️ 4 phases ⏱ 30–45 min 📚 First-time onboarding

– Before you start

  • Python 3.11+, GNU make, git, and a Chromium-based browser.
  • Repo cloned to a fresh working copy (no in-progress branch).
  • ~500 MB free disk space for the venv and local DB.
  • Port 8000 free on 127.0.0.1 (default API bind).
  • Docker is optional — only required if you want the dockerised DB instead of SQLite.

0 Goal

By the end of this tutorial you will have:

  • The API running locally and answering /live.
  • A new GET /api/v1/hello endpoint you wrote yourself, wired through schema, service, and router.
  • One test covering the new endpoint.
  • An internal docs page for the endpoint, reachable from the sidebar.
  • A green make verify on the resulting branch.

1 Local setup and first successful call

  1. Create the virtual environment and install dependencies: 
    make venv && make install
    The command ends with Dependencies installed.
  2. Initialize local config: 
    make env-init
    You see .env created from env/example. If .env already exists, keep the current file.
  3. Apply database migrations: 
    make migrate
    Output ends with Migrations applied.
  4. Start the API: 
    make run
    Uvicorn binds to 127.0.0.1:8000 by default, or to the host and port set in .env.
  5. From a second terminal, hit the health endpoint: 
    curl -i http://127.0.0.1:8000/live
    Expected response: HTTP/1.1 200 OK and a JSON status body.

✓ Check your progress

You should be able to start the API, hit /live, and see a 200 OK response. If anything fails, stop here and fix the local setup before moving on — every following phase assumes the app runs cleanly.

2 Write GET /api/v1/hello

You will add four small files and register the router. The endpoint takes one optional name query parameter and returns a greeting. No database, no migrations, no error-code conventions — that is on purpose.

2.1 Add the response schema

Create services/api/app/schemas/hello.py:

from pydantic import BaseModel, Field

class HelloResponse(BaseModel):
    message: str = Field(..., description="Greeting message.")

The schema is the public contract of the endpoint — every additive change to it ripples to clients.

2.2 Add the service

Create services/api/app/services/hello_service.py:

def build_greeting(name: str) -> str:
    return f"Hello, {name}!"

2.3 Add the router

Create services/api/app/api/v1/hello.py:

from fastapi import APIRouter, Query

from app.schemas.hello import HelloResponse
from app.services.hello_service import build_greeting

router = APIRouter(prefix="/hello", tags=["Hello"])

@router.get("", response_model=HelloResponse)
def hello(name: str = Query("world", min_length=1, max_length=64)) -> HelloResponse:
    return HelloResponse(message=build_greeting(name))

2.4 Register the router

In services/api/app/main.py, mount the new router under /api/v1:

from app.api.v1 import hello as hello_router

app.include_router(hello_router.router, prefix="/api/v1")

2.5 Try it

Restart the API (make run) and call the endpoint:

curl -s "http://127.0.0.1:8000/api/v1/hello?name=Ada"
        # {"message":"Hello, Ada!"}

Open http://127.0.0.1:8000/docs — the new Hello tag and operation should appear in Swagger.

✓ Check your progress

The endpoint returns a 200 with the expected JSON, and Swagger shows the new operation. If Swagger does not list it, the router is not registered — re-check step 2.4.

3 Test and document

3.1 Cover the endpoint with a test

Create tests/api/v1/test_hello.py:

def test_hello_default(client):
    response = client.get("/api/v1/hello")
    assert response.status_code == 200
    assert response.json() == {"message": "Hello, world!"}

def test_hello_named(client):
    response = client.get("/api/v1/hello?name=Ada")
    assert response.json() == {"message": "Hello, Ada!"}

Run just this file to confirm it passes:

.venv/bin/python -m pytest tests/api/v1/test_hello.py

Open services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json, find the existing services-api-reference-user entry, and add this sibling right after its closing brace (inside the same parent children array): 

{
"id": "services-api-reference-hello",
"label": "Hello",
"href": "/services/portal/internal/services/api/reference/hello/index.html"
}

Regenerate every doc artifact (search index, sidebar, redirects):

make docs-fix

✓ Check your progress

  1. Open http://127.0.0.1:3000/services/portal/internal/services/api/reference/hello/index.html — the page renders with the breadcrumb Internal / Services / API / Reference / Hello.
  2. The left sidebar shows Hello under Catalog → API → Reference.
  3. The portal search box finds the page when you type hello. No hit? The JSON entry is the most likely cause — re-check the comma placement and re-run make docs-fix.

4 Ready for review

  1. Run the full local gate. This is the target CI runs: 
    make verify
    It covers drift detection, baseline accessibility, the OpenAPI contract, and the test suite. The last log line should read verify: OK.
  2. Confirm the diff covers both code and docs in one branch: 
    git diff --stat main
    You should see files under services/api/app/ (schema, service, router, test), under services/portal/internal/services/api/reference/hello/, and one line in services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json.

✓ You're done

You ran the project, added one endpoint, tested it, documented it, and passed make verify. That is the full loop — every change in this repo follows the same shape.

↺ Want to start over?

The tutorial is throwaway by design. To reset:

git checkout -- .
git clean -fd app/api/v1/hello.py app/schemas/hello.py app/services/hello_service.py \
tests/api/v1/test_hello.py services/portal/internal/services/api/reference/hello/
rm -rf .venv

Re-run from Phase 1. make migrate is idempotent on a fresh DB, so nothing else needs unwinding.
