Your first capacity drill

Pick one endpoint, load it with a tiny generator, watch latency and saturation degrade, name the bottleneck, file the follow-up. The point is to feel the capacity loop — load → observe → name limit → next step — not to certify the system. Real capacity work widens the surface and adds isolated environments; the shape is the same.

⚙️ 4 phases ⏱ 30–45 min 📚 First-time 💻 Local stack

– Before you start

  • make run starts the API at http://127.0.0.1:8000.
  • hey or ab on PATH (brew install hey on macOS); both produce equivalent output for this drill.
  • Repo on a clean branch — you'll only write a report, not modify the app.
  • You've skimmed runbook 0009 — error-budget exhaustion once so the SRE Trigger / Triage / Recovery shape is familiar.

0 Goal

  • Pick one read endpoint — default is GET /api/v1/hello.
  • Generate three load levels (baseline, shoulder, knee) and record latency p50/p95/p99 + error rate at each.
  • Identify which resource saturates first (CPU, memory, sockets, DB).
  • Write a drill report at services/portal/internal/scratch/capacity-drill-<date>.md with the three-level table, the named bottleneck, and one follow-up action.

1 Baseline at one user

Start the server in one terminal:

make run

In a second terminal, hit the endpoint with one virtual user for 10 seconds and capture the run:

hey -z 10s -c 1 "http://127.0.0.1:8000/api/v1/hello" \
  | tee services/portal/internal/scratch/capacity-drill-l1.txt

From the output, write the first row of the report table:

# Capacity drill: GET /api/v1/hello

**Build under test**: <commit-sha> on <branch>
**Run by**: <handle> · <date>
**Generator**: `hey -z 10s` against `http://127.0.0.1:8000/api/v1/hello`

## Results

| Level     | Concurrency | RPS    | p50    | p95    | p99    | Errors |
|-----------|:-----------:|:------:|:------:|:------:|:------:|:------:|
| Baseline  | 1           | …      | …      | …      | …      | 0      |
| Shoulder  | 10          |        |        |        |        |        |
| Knee      | 100         |        |        |        |        |        |

✓ Check your progress

The Baseline row is filled in with numbers from your hey output. If p99 is already > 10× p50 at one user, the system is sick before load — stop and file that as the finding; no point pushing it harder.

2 Step up to shoulder and knee

Run the same generator at 10 and 100 concurrent users:

hey -z 30s -c 10 "http://127.0.0.1:8000/api/v1/hello" \
  | tee services/portal/internal/scratch/capacity-drill-l2.txt

hey -z 30s -c 100 "http://127.0.0.1:8000/api/v1/hello" \
  | tee services/portal/internal/scratch/capacity-drill-l3.txt

While each run executes, watch the server in a third terminal:

top -pid $(pgrep -f "uvicorn app.main")

Fill in the Shoulder and Knee rows. The signal is the shape, not the absolute numbers:

  • Linear: RPS scales with concurrency, p95 stays close to baseline. System has headroom.
  • Plateau: RPS stops growing but errors stay near zero. Something is rate-limiting cleanly (CPU saturation, single-threaded handler).
  • Cliff: errors spike, p99 detaches from p95. Something is collapsing (pool exhaustion, OOM, timeouts).

✓ Check your progress

All three rows of the table are filled. You can name the shape between shoulder and knee (linear / plateau / cliff) in one word and point at the column that confirms it. If two columns disagree (e.g. RPS plateaus but p99 spikes), pick the more user-visible signal — that's the one that matters in the report.

3 Name the bottleneck

The table tells you that the system slowed; the next step is why. Write one sentence per candidate, then mark the most likely one:

## Bottleneck candidates

- **CPU**: top showed uvicorn at <X>% during the knee run;
  <single-core saturated | distributed across cores | idle>.
- **Memory**: RSS held at <X> MB; no swap activity.
- **Sockets / file descriptors**: connection-refused errors at knee → likely.
- **Database**: SQLite is single-writer; `GET /hello` does not touch it,
  so DB is not in path here.
- **Network**: localhost — irrelevant for this drill.

**Most likely**: <chosen candidate> — <one-sentence why>.

If two candidates look equally likely, pick the one that's cheapest to verify and write the verification step as the follow-up. The drill is not a debugging session — it's a sighting; production-grade triage uses a runbook.

✓ Check your progress

Every candidate is either marked irrelevant (with a reason) or scored with a one-sentence observation. The "Most likely" line names one candidate and one reason. If the most likely cause is "unknown — needs isolation", that is a valid finding; record it and continue.

4 Follow-up

Close the report with a single follow-up the team can actually pick up:

## Follow-up

- **Knee at**: <concurrency> vusers / <RPS> rps
- **First saturating resource**: <chosen candidate>
- **Next step (single action)**: <e.g. "raise uvicorn workers to 4 and
  re-run knee level", or "isolate to a Linux box without macOS network
  quirks", or "instrument the handler with `time.perf_counter` and re-run
  shoulder">.
- **Owner**: <name> · **Target**: <date>

One action, one owner, one date. Capacity findings sprawl into "we should rebuild on Rust" — don't let yours do that on the first drill.

✓ You're done

  1. Report file exists under scratch/ with the three-level table, bottleneck candidates, follow-up.
  2. The follow-up line names one action, one owner, one date.
  3. You can answer "where is the knee on GET /hello?" with a number, not a vibe.

That is the capacity loop — load → observe → name limit → single follow-up. Real drills run on isolated hardware, sweep wider, and feed SLO budgets; the shape is the same.

↺ Want to start over?

rm services/portal/internal/scratch/capacity-drill-*.{md,txt}
