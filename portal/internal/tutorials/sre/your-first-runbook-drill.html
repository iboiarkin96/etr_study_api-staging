Your first runbook drill

Manufacture a tiny failure, page yourself, and walk the runbook end-to-end until exit criteria are met. The point is to feel the drill loop — trigger → triage → recovery → postmortem stub — not to handle a real incident. Real on-call walks the same loop with thicker steps and more witnesses.

⚙️ 4 phases ⏱ 30–45 min 📚 First-time 💻 Local stack

– Before you start

  • make venv && make install ran successfully.
  • Repo on a clean branch — drills modify a test file, you'll revert at the end.
  • You've skimmed runbook 0001 — tests failing once; sections are Trigger / Fast triage / Causes / Recovery / Exit / Follow-up.

0 Goal

  • One intentionally broken assertion, committed to a local drill/ branch.
  • make test red — the trigger condition runbook 0001 names.
  • You walk all six sections of runbook 0001 against this failure.
  • A postmortem stub at services/portal/internal/scratch/drill-<date>.md capturing what you saw and how long each section took.
  • Branch deleted, repo back on the original commit.

1 Manufacture the failure

Cut a drill branch and break one assertion:

git switch -c drill/runbook-0001-$(whoami)

Open any small unit test (e.g. a health-check or trivial schema test) and change one expected value so the assertion fails: 

# Example diff — pick any test in tests/ with a hard-coded expected value
- assert response.status_code == 200
+ assert response.status_code == 201

Confirm the failure surfaces:

make test

Expected: red. Note the start time — this is your "page received" timestamp for the postmortem.

✓ Check your progress

make test exits non-zero. The failing assertion in the output matches the one you edited (no other tests should be red — if they are, your change was too broad; revert and pick a smaller test).

2 Walk Trigger + Triage

Open runbook 0001. Out loud (yes, out loud — it forces the loop), answer:

  1. Trigger — does your failure match the runbook's trigger condition? (red make test) ✓
  2. Fast triage — execute each command the runbook names. For each one, note in your stub:
    • command run
    • output (one-line summary)
    • hypothesis after seeing it

✓ Check your progress

Your stub has a Triage section with one entry per runbook command. The last line is a single hypothesis (most likely: "assertion mismatch in test X, value drift").

3 Walk Causes + Recovery

From runbook 0001:

  1. Causes — your hypothesis from Phase 2 should map to one of the runbook's listed causes. Write the match into the stub ("matches cause #N: stale assertion"). If nothing matches, that is itself a finding — the runbook needs an extra cause.
  2. Recovery — execute the recovery the runbook names for that cause. In your case: revert the test change. 
git checkout -- tests/<file>
make test

✓ Check your progress

make test exits 0 (green). Note the time — this is your recovery timestamp. Time-to-recovery (TTR) = recovery − trigger.

4 Exit criteria + postmortem stub

From runbook 0001:

  1. Exit — verify every exit-criterion bullet. For this drill: make test green, no lingering branch changes, no uncommitted edits outside the drill scope.
  2. Follow-up — fill out the postmortem stub:
# Drill postmortem — runbook 0001
- Drill date: <date>
- Trigger time: <t0>
- Recovery time: <t1>
- TTR: <t1 − t0> minutes

## Triage transcript
- (commands + outputs + hypotheses from Phase 2)

## Hypothesis → cause mapping
- (Phase 3 match)

## Runbook friction
- Steps that were unclear: ...
- Steps that were missing: ...
- Commands that didn't work as written: ...

## Follow-up
- [ ] File issue: ...
- [ ] Update runbook 0001: ...

The Runbook friction section is the real value of the drill — it turns a synthetic failure into a feedback loop on the runbook itself.

✓ You're done

  1. Stub file exists under scratch/, all sections filled in (Runbook friction may be empty — that's fine).
  2. Repo back on main (or your original branch), drill branch deleted.
  3. You can name TTR in minutes and one thing the runbook could improve.

That is the drill loop — manufacture failure → walk runbook → time it → write friction. Game-day exercises run the same loop on larger failures with witnesses (see SRE practice game-days).

↺ Want to start over (or clean up)?

git switch main
git branch -D drill/runbook-0001-$(whoami) 2>/dev/null || true
rm services/portal/internal/scratch/drill-*.md
