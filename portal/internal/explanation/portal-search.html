Portal Explanation

How portal search works

You type into the topbar, the right page appears, you click — that is the whole user-visible contract. Underneath sits a static-site search engine that is small enough to ship in the same git tree as the docs, fast enough to answer in one keystroke, and strict enough to keep internal-only pages out of the public portal. This page explains the parts and why they fit together the way they do. The decision record is ADR-0033; it supersedes ADR-0027.

The picture in one paragraph

At build time, every HTML page under services/portal/ is fed through a thin Python pre-pass that tags it with visibility (internal or public) and a kind (ADR, runbook, tutorial, practice, …) — then handed to the Pagefind indexer. Pagefind emits a sharded, lazily-loadable search bundle. At read time, the topbar input dynamically imports pagefind.js on first focus, the user's keystroke fires a search, the engine downloads only the index shards it needs, and the topbar renders the top 12 results with our badges and excerpts.

Why Pagefind, not a hand-written index

The portal shipped its own BM25 implementation for the better part of a year (ADR-0027): a Python builder that walked the HTML tree, a JSON file with inverted postings, and a ranker in search.js. It worked. The reason it had to go was not relevance — it was the maintenance cost of every adjacent concern. Every new content type asked for a new badge rule. Every corpus growth pushed the committed JSON closer to three megabytes, all of which arrived on first paint. Every browser upgrade was a risk surface no one was monitoring.

Pagefind solves the same problem the way the rest of the industry has settled on it for static sites: a Rust indexer that reads the rendered HTML directly, a sharded bundle the runtime fetches in pieces, and a BM25-with-stemming ranker we do not own. Cold payload drops from ~2.3 MB to ~92 kB gzipped; per-query cost is around one 30 kB shard plus the small excerpt fragments for the results actually shown. The ranking layer becomes someone else's responsibility, and the responsibility that stays on us — what to index, how to carve internal from public, what to call each result — is the responsibility we are actually good at.

What the build does

The work happens in tools/docs/build_pagefind_index.py as step [10/10] of make docs-fix. It has two phases.

First, every page under services/portal/ is mirrored into var/tmp/portal-pagefind-staged/. The mirror is necessary because the indexer wants attributes on the page that the page itself does not have to carry — we annotate <main> with two extra attributes:

  • data-pagefind-filter="visibility:internal" or visibility:public, derived from the page's path prefix under services/portal/. This is the load-bearing invariant that keeps the public portal from leaking internal-only URLs through search.
  • data-pagefind-meta="kind:<taxonomy>", derived from the same path. ADRs, runbooks, tutorials, templates, practices, explanations, how-tos, references, services, handbook entries, and catalog pages all get a stable label, which the runtime turns into the badge you see on the right side of each result.

Second, the pinned Pagefind binary (pagefind==1.5.2 + pagefind_bin_extended==1.5.2 via PyPI, so there is no Node toolchain to manage) is run against the staged tree with SOURCE_DATE_EPOCH=0. The result lands in services/frontend/portal/pagefind/. Immediately after, a sibling script tools/docs/check_pagefind_visibility.py walks the emitted fragments and asserts that no public/-prefixed URL slipped through with visibility:internal and vice versa.

What the bundle looks like

The output is not one file but a small directory of small files, content-hashed by Pagefind itself:

FileRole
pagefind.jsThe runtime module the topbar dynamically imports.
pagefind-entry.jsonTiny manifest telling the runtime which language-specific shards exist.
wasm.en.pagefindThe English stemmer / matcher, compiled to WebAssembly.
*.pf_metaPer-language metadata block — small, always loaded.
index/*.pf_indexSharded inverted index. Only the shard a query needs is fetched.
fragment/*.pf_fragmentOne file per indexed page, gzipped JSON with title, excerpt, URL, filters, and metadata. Fetched lazily, only for the results actually displayed.
filter/*.pf_filterCompact lookup of filter buckets (e.g. visibility:public) used by the engine to narrow before scoring.

The bundle is committed to git the same way the pdoc API reference is committed: Pagefind embeds per-build hashes in its meta and filter blobs, so two runs of the indexer on the same input do not produce byte-identical output. The committed snapshot is the source of truth, and the docs-check drift gate restores it from HEAD before diffing, exactly as it already does for pdoc. The mechanical reason to commit is not aesthetic preference — it is GitHub Pages, which deploys whatever is in the git tree at deploy time and has no build step of its own.

What the runtime does on a keystroke

The topbar input is mounted by services/frontend/portal/assets_v2/ui-kit/components/search.js. Six things happen between the first keystroke and the rendered dropdown:

  1. Auto-inject. On page load, mountSearch() looks for every .topbar__actions in the DOM and slips a data-component="search" placeholder into each, so authors do not have to add the markup per page.
  2. Lazy import. On first focus of the input, pagefind.js is dynamically imported. Nothing about search costs anything for readers who never click into it.
  3. baseUrl resolution. Pagefind defaults to using its own bundle path as the URL prefix for results. We override it: the current page's pathname is matched for /portal/, and everything up to and including that segment becomes the baseUrl. The same code therefore produces /services/portal/internal/… on the dev server and /<repo>/portal/internal/… on a GitHub Pages project site, without a build-time switch.
  4. Visibility filter. On a page where <html data-portal="public">, the call to pagefind.search() includes { filters: { visibility: "public" } }. Internal-portal pages pass no filter and therefore see everything.
  5. Top-12. A 120 ms debounce, top-12 truncation, and a parallel call to result.data() for each fetches the per-result fragment. Fragments include the title, the excerpt with <mark> wrappers around matched terms, and the kind metadata.
  6. Render. Each hit becomes an <a> in the dropdown with the title, the kind badge, the path, and the excerpt. Keyboard navigation (↑, ↓, Enter, Esc) is wired directly on the input; the global hotkeys / and ⌘K are handled by hotkeys.js and focus the same input.

The internal/public split

Two facets are doing the load-bearing work here: the build-time data-pagefind-filter attribute and the runtime filters: { visibility: ... } argument. Together they replace the old setup where the build emitted two separate JSON files — an internal index and a strict subset for public — and a pre-commit gate diffed them. The new design is the same invariant expressed in fewer artefacts: one bundle, one indexer pass, one runtime filter depending on which portal asked. The strict-subset invariant is preserved by tools/docs/check_pagefind_visibility.py, which fails the build if a public/ URL ever ends up tagged as internal.

GitHub Pages

GitHub Pages deploys static files from the git tree, with no build step of its own. Our bundle ships with the rest of the portal, so search works on a fresh clone published straight to Pages — provided two small things are also in place:

  • A .nojekyll at the publish root, so Jekyll does not eat files that start with an underscore. Pagefind itself does not use that pattern, but the rule is the same one its docs recommend, and it is cheap.
  • The baseUrl derivation above, which is what makes the same JavaScript work on https://username.github.io/repo/portal/… without per-environment configuration.

The internal-vs-public split is a search-time concern, not a deployment-time one. A public Pages site still ships the full bundle, including internal-page fragments — and that means it leaks every internal page's title and first excerpt to anyone who reads the raw bundle. If we ever publish the public portal to a truly public GitHub Pages site, the build will need a second mode that emits a public-only bundle. Today the deployment target is private; the split exists so that the public-portal frontend behaves correctly when readers reach it through the internal mirror.

Adding a new page

The flow for an author adding a new page is unchanged from the legacy setup:

  1. Write the page under services/portal/internal/<quadrant>/<slug>.html (or public/ for the public portal).
  2. Run make docs-fix. The pre-pass picks the page up, infers its visibility from the path, infers its kind from the path's first non-portal segment, and Pagefind re-indexes.
  3. Commit the page. The Pagefind bundle at services/frontend/portal/pagefind/ is gitignored — local make docs-fix rebuilds it for browser preview, and the Pages workflow rebuilds it on every CI run before deploy (per the ADR 0033 amendment 2026-06-08). No bundle artefacts go into your commit.

Search picks the page up automatically. If the badge is wrong, the path probably does not match any rule in tools/docs/build_pagefind_index.py's kind_for() — add a rule there.
