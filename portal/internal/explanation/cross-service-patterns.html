Cross-service patterns — the namespace for when there are two

Study App ships as one deployable today. Distributed tracing, service-to-service auth, and saga-style transactions are not patterns we run — they are patterns we will need to decide on the day a second service ships. This page is the namespace for those decisions and the early-warning list of design forces that will trigger each section being filled. It is deliberately empty in places; the alternative is to design distributed systems we don't yet need.

Why a placeholder, not nothing

Three forces make the empty-stub approach cheaper than starting from a blank page on day two-service-zero. (1) Cross-service patterns interlock — distributed tracing and service-to-service auth are not separable decisions; a stub forces an early read on the interlocks. (2) The architectural posture of «single service today» is itself a decision worth recording, with the alternatives we rejected at every monolith-vs-microservice fork. (3) The day a second service is proposed (RabbitMQ worker in BL-049, or any future split), this page is where the design notes land. The audit trail starts here, not in a Slack thread.

Today's posture — one service, three sub-components

The topology says four «services» live in the repo (api, portal, monitoring, ui-kit), but only api is a runtime service; the rest are static assets, build artefacts, or the portal itself. The domain model describes six bounded contexts, all enforced inside one process. This page therefore documents three explicit non-decisions — patterns we deliberately have not paid for — and the signal that will end each non-decision.

The three explicit non-decisions

  • No distributed tracing. Spans live within one process; observability is structured logs + request IDs, not OpenTelemetry spans crossing a wire. Reason: one process means causality is reconstructable from the log stream alone; OTLP would add a collector dependency for zero present-day benefit.
  • No service-to-service auth. There is no «service B calls service A» surface. Internal calls are function calls. Reason: introducing mTLS or service-account JWTs before they have a caller is the canonical premature-design failure.
  • No distributed transactions. Writes are single-database; SQLite's single-writer model is the consistency boundary. Reason: sagas, two-phase commits, and outbox patterns each add operational complexity that we don't yet pay rent for.

What ends each non-decision

Each of the three sub-sections below names the explicit trigger — the design force that, once it fires, turns the placeholder into a real Explanation page. Until the trigger fires, the section stays a stub and the architectural posture is to not pre-emptively design.

PatternTrigger to fill this sectionDefault landing decision when the trigger fires
Distributed tracing ≥ 2 processes participate in any user-visible request, or any async worker emits events that downstream services subscribe to (BL-049 RabbitMQ is the first candidate). OpenTelemetry SDK with OTLP export to a local collector; traceparent propagated in every cross-process call. ADR drafted before the second process ships, not after.
Service-to-service auth Any internal API surface that a second service calls (not the public API used by external integrators, which already has the API-key contract from ADR 0005). Short-lived service-account JWTs signed by a shared secret; mTLS rejected unless we are also paying for service-mesh tooling. The trade is in ADR 0005 § Alternatives already.
Distributed transactions / sagas Any write that has to be atomic across two data stores (e.g. PostgreSQL primary plus a second store like Redis or a search index). Transactional outbox + idempotent consumer, not 2PC. The idempotency-key contract from ADR 0006 is the foundation; the consumer side fills in here.

Distributed tracing — stub

Not yet relevantSingle-process spans live in structured logs; correlation is by request ID, not by trace ID. Re-evaluate the day a second process participates in a request.

When filled, this section will document the choice of OpenTelemetry SDK + OTLP collector vs alternatives (Honeycomb agent / Datadog APM / homegrown trace IDs in logs), the W3C traceparent propagation contract on every internal hop, sampling strategy (default head-based vs adaptive vs tail-based once we have the data), and the cost of each rejected alternative. The single load-bearing question to answer first: «can our existing log aggregator continue to be the primary observability surface, or does the trace become it?» — the answer determines whether OTLP is additive or a replacement.

Service-to-service auth — stub

Not yet relevantThere is no internal API surface. The day a second service exists, this stub becomes a real ADR — likely short-lived JWTs over mTLS.

When filled, this section will document the rejected alternatives explicitly: (a) mTLS-only with no application-layer identity — rejected because operations like rate-limit and authz cannot read «which service is calling» from a cert; (b) long-lived API keys reused across internal callers — rejected because rotation becomes a fleet operation; (c) SPIFFE / SPIRE workload identity — rejected unless we already run a service mesh that issues the SVIDs for free. The default landing decision is short-lived JWTs signed by a shared secret, with mTLS as transport hardening when the call crosses a network boundary outside our trust zone.

Distributed transactions / sagas — stub

Not yet relevantAll writes go to one SQLite file. The day a second store enters the write path, the outbox-pattern decision is forced.

When filled, this section will document the contract for atomic-across-stores writes: a transactional outbox table local to the primary store, an idempotent consumer reading the outbox and applying side effects (search-index update, cache invalidation, downstream event publish). Rejected alternatives: (a) two-phase commit across stores — rejected because most modern stores don't expose the XA protocol; (b) compensating sagas without idempotency keys — rejected because compensation is unsound without exactly-once application; (c) «eventually consistent and hope» — rejected by name, because that is not a pattern, it is a missing design.

Tradeoffs of keeping this page as a stub

Stub pages are themselves a tradeoff. We pay maintenance cost on a page that is not yet load-bearing; the alternative is paying design cost during an outage on day-two-service-zero. The pattern below pins the trade.

Win

Architecture decisions land in a known location

The day a second service is proposed, this page is where the «what about tracing / auth / sagas?» questions get answered — not in a Slack thread, not in a one-off ADR drafted under deadline pressure.

Cost

A stub page is a maintenance item

Every audit pass has to re-check whether the page is still aspirational or has become real. The mitigation is the explicit trigger table above — when the trigger fires, the section gets filled or the page gets retired.

Win

The «single service» posture is recorded as a decision, not an accident

Stating «we deliberately do not run distributed tracing because we have one process» is stronger than silence. Silence reads as «they didn't think about it»; this page proves we did.

Cost

Pre-emptive design is a known anti-pattern

If we fill the stubs before the trigger fires, we are designing for hypothetical futures — exactly what the principles warn against. The discipline is to keep the stubs empty until the trigger lands, even when filling them would feel satisfying.
