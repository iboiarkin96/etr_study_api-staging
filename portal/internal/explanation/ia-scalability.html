IA scalability — why the structure holds at 10×, and when it stops

The portal carries ~570 pages today. The information architecture was sized for that number to triple before any structural change is needed. This page explains the load-bearing rules that make that true, the four IA-debt patterns to watch for, and the explicit signals that mean the structure has stopped holding and needs a re-architecture pass.

What's actually scaling

"Information architecture" in this repo means three things, and only one of them dominates the cost: (1) the folder layout under services/portal/internal/, (2) the sidebar nav tree in nav-tree-internal.json, and (3) the cross-link graph between pages. Of these, the sidebar nav tree is the bottleneck — every new page has to be placed in it, every renamed page has to be tracked, and visual scannability degrades as the tree grows. A folder layout can stay flat for a long time; a 2000-entry sidebar is unusable at any depth.

The four rules that make 10× possible

These four rules are not aesthetic — each one prevents a specific failure mode that would otherwise force a restructure long before page count justified it.

  • Diátaxis-first folder layout. Every top-level folder maps to a quadrant (tutorials/, how-to/, reference/, explanation/) or a clearly named non-quadrant artefact (governance/, team/, services/). When the next page arrives, "where does it live?" has one answer because the question reduces to "which quadrant?". The audit trail that proves this works is in diataxis.html § 03 (Layer 2).
  • Sidebar mirrors the filesystem, not the social structure. The sidebar tree is generated from filesystem locations, not from "who owns it" or "which team cares". This means that doubling the team does not double the sidebar — a 10-person team and a 100-person team navigate the same tree.
  • Cross-cutting concerns live once. Auth, idempotency, error envelope, observability conventions all live under a single _shared/ directory; per-operation specs link to them instead of restating them. Quadrupling the number of operation specs adds 4× the links but not 4× the words — the _shared/ directory grows by zero.
  • Acronym casing is a hard rule, not a preference. The CI check on nav-tree-internal.json rejects "Api" / "Adr" / "Sre". Drift here is the #1 navigation bug at any scale because acronyms are visually load-bearing; enforcing the rule mechanically removes the most common source of late-stage IA rework.

The four IA-debt patterns to watch for

IA debt is invisible until the structure already needs work. Each of these four patterns is an early warning that one of the four load-bearing rules above is being violated; catching them before the page count hits the ceiling is what keeps the next restructure cheap.

  • Folder-vs-quadrant drift. A folder named reference/ starts accumulating Explanation-shaped pages. The audit in 2026-05 found this happening in handbook/ (11 pages declared explanation, 8 declared how-to, despite the folder name); the fix is to either move pages to quadrant-aligned siblings or to explicitly document the folder as "mixed quadrants, judged by data-page-type". Doing nothing means the rule "folder = quadrant" silently stops being true.
  • Per-page CSS instead of kit components. A page-local <style> block for a rail-list, a gradient card, or a tradeoff grid. The first such block looks harmless; the third is a kit fork. The rule is in feedback_no_per_page_css — promote repeated styles to the kit, never to the page.
  • Sidebar entries without a parent group. A page that appears at the top level of the sidebar because "it doesn't fit anywhere". The right response is to create a new group when ≥ 3 pages share the same parent concept; the wrong response is to keep adding orphans until the top level has 30 entries and visual scanning fails.
  • See-also footer blocks. The pattern says "I have related content but no Diátaxis-clean way to surface it". Every standalone "See also" block on a page is a missed cross-quadrant hinge; the kit pattern (gradient "What to read next" card with 3–5 Reference/How-to/Explanation links) does the same job and lets the IA absorb growth.

Where the ceiling actually sits

The current structure was sized with three explicit ceilings in mind. Crossing any of them is the signal that an IA pass is overdue.

SignalCeilingWhat changes when crossed
Total page count~1500Sidebar requires search-first navigation; flat scanning stops being viable. Add a sidebar-internal search input bound to the existing search-index.
Top-level sidebar entries~12The top level needs grouping or collapsing. Introduce a meta-level above the current quadrants (e.g. "Build" / "Run" / "Govern").
Pages per leaf-level group~25The group needs a hub page with internal filtering (chip-based) so the reader doesn't scroll past a 25-row sidebar list. The existing pattern under tutorials/ and how-to/incidents/runbooks/ is the model.
Per-service folders under services/3Today: one (services/api/). When a second service ships, the cross-service patterns namespace (see below) is no longer speculative.
Distinct cross-cutting concerns in _shared/~15The current 10 are organised flat. At 15+ the directory needs a sub-grouping or a category index page.

What the next restructure looks like

When the page count crosses ~1500 — and not before — the structure that follows is partially specified. The architecture decisions to lock in early are: keep Diátaxis as the primary axis (it has survived 5× growth so far without ambiguity), keep _shared/ as the canonical cross-cutting location, keep acronym-casing CI. The structure to add: a meta-grouping above the quadrants ("Build" / "Run" / "Govern") so the sidebar collapses to ~5 top-level entries; a cross-service patterns directory at services/_shared/ when service #2 ships; a sidebar-internal search input bound to the existing client-side search index.

What is explicitly out of scope for the next pass: replacing Diátaxis with a different model (no evidence it has hit its ceiling); auto-generating the sidebar from the filesystem (the hand-maintained tree is the rule of last resort, but the social cost of reorganising it is what keeps the IA stable).

Common failure modes

  • "We'll restructure when we get to 10× the pages." The four IA-debt patterns above land long before 10× and are the actual rate-limiting step. Watching them is cheap; the restructure they save is not.
  • Restructuring without a stated ceiling. Moving folders because the new layout "feels better" is rework. The five ceilings above are the bar for action; without them, restructures arrive on taste and disappear on the next person's taste.
  • Per-team folders. A folder named after a team is a folder that has to be renamed when the team changes. The Diátaxis quadrants and the deployable-services list both survive reorgs.
  • Treating the sidebar as documentation. The sidebar is a navigation aid; the documentation is the pages. A sidebar that tries to summarise each page in its label is a sidebar that breaks on every page rename.

Tradeoffs

The rules above are not free. They impose discipline that the team pays for every PR; the trade is that the team does not pay for a quarterly restructure.

Win

A new page lands in the right place by default

The Diátaxis quadrant tells you the folder; the folder tells you the sidebar group; the kit components tell you the markup. No design call required per page.

Cost

Page authors have to learn the model

A new author can't skip the principles page. The cost is a 60-minute onboarding read; the alternative is per-author IA drift that compounds.

Win

CI catches the four most common drifts

JSON validity, acronym casing, asset references, h1 uniqueness — all enforced in pre-commit. The structure is mechanically defended.

Cost

Mechanical checks miss semantic drift

A page declaring data-page-type="explanation" while reading as a how-to passes every CI gate. Only review catches this. The check is human attention, and it is the most expensive check in the system.
