Why a Definition of Done

Every artefact a team commits to — an API spec, an ADR, a runbook, a postmortem — has a moment where the author hands off responsibility to readers. The Definition of Done makes that moment unambiguous: a fixed list of questions the artefact must answer before handover, so ambiguity surfaces as a lint failure today rather than as a bug six weeks later.

Where this applies

This page is the SA-level Explanation; the principles hold for any artefact whose value depends on a clean handover from author to readers. Concrete DoDs live in the template alongside each artefact's structural list:

  • API operation specs — checklist in the API spec template. The running example below comes from here because it's the project's most exercised DoD.
  • ADRs — checklist in the ADR template. Status badge + ratification milestones; same lint-vs-review split.
  • Runbooks — checklist in the runbook template. Required sections (Trigger / Triage / Causes / Recovery / Exit / Escalation) are the artefact's DoD.
  • Postmortems — checklist in the postmortem template. Timeline-UTC + SMART action items are the artefact's DoD.

What follows talks about specs because that's the canonical worked example; substitute «artefact» wherever the argument turns on the artefact's shape rather than on API-specific surface.

The problem a DoD solves

Artefacts drift. A first draft answers the obvious questions — request shape, response shape, happy path — and is silent about the awkward ones: what happens on retry, who is allowed to call this, what error code the client sees when a precondition fails, what gets logged. Silence reads as “use your judgement” to the reader, and judgement varies by person and by week.

The result is a class of bugs that no test ever catches because no test was written for them: an idempotency key that two endpoints interpret differently, an error envelope whose code field changes shape between operations, an alert that fires on a metric nobody emits, an ADR ratified before its alternatives were spelt out. None of these are implementation mistakes. Each is a question the artefact did not ask.

A Definition of Done is the answer: a fixed list of questions every artefact must answer before it can be handed over. The cost of answering them is borne once, by the author, instead of repeatedly by the readers.

The contract of handover

The DoD describes what the reader is owed when picking up the artefact — and, by symmetry, what the author owes the rest of the project.

  • Implementability without re-asking. A developer (for a spec), an on-call (for a runbook), or a reviewer (for an ADR) should be able to act on the artefact without coming back to the author for clarification on the load-bearing details.
  • Testability without invention. Acceptance criteria, edge cases, exit criteria, action items — concrete enough that another role can write tests, scripts, or follow-up tickets directly against them, not against an imagined reading.
  • Traceability without archaeology. A reviewer should be able to follow a requirement back to the methodology that motivated it and forward to the ADR that constrains it, without grepping the repo.

This is why the checklist behind the DoD is mostly about presence, not quality: presence is what a linter can verify. Quality is what peer review is for. The two checks are deliberately separated.

Why a checklist — and why a long one

Checklists are out of fashion in some quarters: they feel bureaucratic, they slow you down, you can game them. Each criticism is true and each is beside the point. The DoD checklist exists for the same reason a pre-flight checklist exists: not because pilots forget how to fly, but because the cost of forgetting one item is high enough that systematic recall is cheaper than relying on attention.

The list is long — for API specs it's twenty items plus quality gates — because that is the surface of the artefact. Trimming it would not make the surface smaller, only invisible. Each item exists because, in some previous artefact of the same kind, its absence caused a real problem: an error code missing from the catalog, an idempotency rule discovered in production, a metric that did not exist when the alert fired, an ADR alternative spelt out only after the decision was already merged.

Why cross-cutting topics live in shared pages

For API specs: idempotency, authentication, pagination, the error envelope, versioning, observability conventions — every operation touches some of them, but the answer is the same for every operation. The same pattern recurs in other artefact families: ADR statuses live in one ratification ADR, runbook triage shapes live in the runbook template, postmortem timeline conventions live in the postmortem template. Inlining the rules in each artefact would drift: one would say one thing, another would say another, and there would be no canonical reference to break the tie.

So the DoD requires that each artefact link to the shared page rather than restate it. The structural lint enforces the link; the content is maintained in one place. This is the same principle that puts shared types in a library rather than copy-pasted across services: one definition, many uses.

Lint vs review — two layers of done

Two distinct checks gate every artefact, and they answer different questions.

  • The linter asks: is every required section present, is every cross-link resolvable, is the identifier unique (operationId for a spec, data-adr-weight for an ADR, runbook id for a runbook), is the status badge valid? It runs in CI and is cheap to repeat. It cannot judge whether an acceptance criterion is meaningful.
  • The reviewer asks: does this artefact, taken as a whole, describe something a downstream role could act on? They read the prose, follow the user story (or the alternatives, or the recovery), sanity-check the edge cases, and form a judgement.

Conflating the two is a common failure mode. A linter-only DoD certifies that the artefact is well-formed gibberish; a review-only DoD relies on the reviewer to remember twenty things at once. The DoD is split because the checks are different in kind.

Status as a contract about change

The status badge — draft, in review, approved, implemented, deprecated — is not decorative. It tells a reader what is safe to assume.

An approved artefact means downstream work can start without expecting the contract to shift under it. An implemented spec means the OpenAPI document and the live service match; subsequent changes must respect versioning rules. ADRs use the same scale (decision → documentation → implementation) via data-adr-weight. A deprecated artefact means new consumers should not adopt it; the lifecycle is winding down on a known schedule.

Statuses are coarse on purpose. Finer-grained states would invite arguments — “mostly approved”, “approved-pending-X” — and rob the badge of its load-bearing role: a single word that tells you whether to build on this artefact or wait.

Common failure modes

  • “Done is when the lint passes.” The linter checks shape, not substance. A spec full of placeholders can be green. Review is non-optional.
  • “We'll fill in observability later.” Observability deferred is observability never. By the time the alert is needed, the metric does not exist, and the spec has moved to a different author.
  • “Out of scope: none.” Writing None. is allowed; leaving the section empty is not. The presence of an explicit “None.” is the signal that the author considered the question.
  • “This operation is too simple for a full spec.” Simple operations still touch auth, errors, versioning, and observability. The cost of a short spec is small. The cost of a missing one is paid by every future reader.
