Embedded Swagger UI — interactive API sandbox

A dedicated sandbox deployment with CORS allow-listing for the public docs origin, so integrators can use Swagger UI's Try it out against a real (non-production) backend instead of a browser-only validator.

Supersededhistorical

The original "browser-only validation" approach (Ajv + Swagger UI requestInterceptor, no real cross-origin traffic) was placed on hold on 2026-04-12 and replaced on 2026-04-24 by a dedicated sandbox backend with CORS allow-listed for https://iboiarkin96.github.io. This page records the final decision; the historical "no cooperating server" path is described in Why the first approach did not work.

Context

The HTTP contract for integrators is the committed OpenAPI snapshot and the runtime spec (ADR 0025, ADR 0007). Browsing the spec in a static site is not enough: partners and internal providers need round trips against a real process and persistent storage to build confidence before production.

The first approach tried to deliver validation without a cooperating server (Ajv, interceptors, no real HTTP). It proved brittle. What was missing sat on the server side: a reachable sandbox backend, correct CORS for the public docs origin, and sandbox-scoped credentials — not more client-side plumbing.

Decision

  1. Dedicated sandbox deployment. Run a separate app instance at a stable HTTPS base URL (e.g. a PaaS with a fixed hostname, per ADR 0021 for the image). It uses its own persistent database — not production, not shared with ad-hoc local runs. The sandbox is long-lived enough for integration work (not a throwaway pytest process).
  2. CORS allow-list for the public docs site. The sandbox's CORS configuration includes https://iboiarkin96.github.io (and only origins we explicitly approve). This makes browser fetch from GitHub Pages first-class, without a public CORS proxy.
  3. Swagger UI: Try it out against the sandbox only. Update services/portal/public/reference/api/index.html to tryItOutEnabled: true, with a servers override (or clear default) that points to the sandbox base URL — never production. Use persistAuthorization: true so the Authorize step is not lost on every refresh.
  4. Pin Swagger UI assets. Vendor the Swagger UI CSS/JS under services/portal/ or load a version-pinned CDN build with integrity (SRI). The version is chosen and bumped in-repo, not by whatever the CDN returns today.
  5. No Ajv / interceptor validation layer in the static site. Reject resurrecting openapi-sandbox.js and request interceptors. The service validates; errors follow ADR 0003 and the OpenAPI.
  6. Credential model: scoped API keys for the sandbox only. Integrators use sandbox-issued, scoped API keys (or equivalent headers) in Swagger Authorize — not production keys. Keys are revocable, least-privilege for the sandbox, and must not be committed to the repo. Production credentials must never be pasted into a public static page; environment separation and policy are part of the security story (ADR 0005).
  7. Sandbox is explicitly non-production. UI copy and runbooks state that data is test data and may be reset. Contract parity with production is a goal; data and blast radius are not.

Scope

  • In scope: services/portal/public/reference/api/index.html and Swagger options; sandbox deployment URL; CORS on the sandbox; key issuance/rotation process for sandbox keys at a high level; SRI or vendored assets.
  • Out of scope: production deployment and production CORS policy; full partner billing or a separate "developer platform" product; per-key rate limits, quotas, and 24/7 SLOs for the sandbox (best-effort and operational follow-up are acceptable). Those may be added later in runbooks or a follow-up ADR without changing this direction.

Alternatives considered

Five alternatives evaluated before adopting the sandbox-deployment + CORS-allow-list approach (later superseded). Each rejected with a rooted reason. Pattern from Tradeoff disclosure.

Postman collection / «Run in Postman»

rejected

Why considered: no extra server, familiar to integrators, Postman-native sharing.

Why rejected: requires Postman install + account; not a one-click flow from the public docs URL. The single-URL-in-browser property was the load-bearing UX gate.

Mock server (Prism, WireMock, MSW)

rejected

Why considered: cheap, example-shaped responses, no real DB.

Why rejected: mocks cannot prove persistence, idempotency, or real error paths — the exact properties integrators need to validate. Returns false-positive «works» for cases that would fail against the real service.

Public CORS proxy

rejected

Why considered: avoids server-side CORS changes; quick to wire up.

Why rejected: sandbox credentials and request payloads would cross a third party. Unacceptable trust boundary for any secret material, even scoped sandbox keys.

Browser-only validation (Ajv + interceptors, first attempt)

rejected

Why considered: no server changes at all; the schema validates client-side.

Why rejected: no authoritative round trip — only JSON-shape correctness, never business-rule confirmation. Failed the purpose of Try it out, which is to prove the service actually accepts and returns the expected payload.

Point Try it out at production

rejected

Why considered: zero infra — production already exists.

Why rejected: test traffic mutates or reads real customer data, every integrator click creates a real row. Strictly unacceptable for any non-trivial service.

Trade we accepted (later superseded). Long-lived sandbox deployment with sandbox-scoped keys and CORS allow-listing for the public docs origin. Superseded when the integrator base did not materialise (see § Why this ADR was superseded below).

Consequences

Positive

wins
  • Integrators validate payloads and read real responses and persistence from a stable public URL.
  • Same industry pattern as Stripe, PayPal, Amazon SP-API at a small-team scale: isolate environment, separate credentials, explicit CORS.
  • No Ajv/interceptor stack to maintain on the static site; fewer moving parts than the browser-only path.
  • Readers still get a quick visual view of the OpenAPI snapshot even before the sandbox is up.

Trade-offs

costs

Two named tradeoffs accepted on purpose (decision later superseded). Shape from Tradeoff disclosure.

  • Long-lived sandbox deployment (not throwaway). We gain stable URL + persistent state integrators can reason about across sessions. We lose «zero infra» — there is now a server to keep alive, a DB to clean. We accepted it because integration validation needs persistence; ephemeral mocks failed the use case this ADR exists to serve.
  • Sandbox API keys live in the browser. We gain Try it out as a one-click flow from the public docs site. We lose the «no credentials in browser» property a hosted proxy would give. We accepted it by treating sandbox keys as scoped + rotatable (never production material); the alternative (public CORS proxy) crossed a third party with our secrets — strictly worse.

Revisit — would we do this again?

The interactive-sandbox path was deprioritised when the integrator base shrank to a single internal consumer. Investment in a long-lived sandbox + CORS allow-list pays back only when there is a partner programme; with one internal team the static OpenAPI baseline + curl recipes carries the load. The artefacts (vendored Swagger UI, sandbox CORS pattern) remain documented here as the right answer the day partner integration scales.

Compatibility and migration

  • Browsing the spec without executing remains possible; Try it out becomes available once index.html is updated and the sandbox is up.
  • Rollback: set tryItOutEnabled: false and remove the sandbox server from defaults; pause the sandbox app independently.

Implementation plan

  1. Deploy the sandbox: image from ADR 0021, own DB, record the public base URL (placeholder example: https://etr-study-api-sandbox.example).
  2. Configure CORS on the sandbox (e.g. CORS_ORIGINS per ADR 0005) to include https://iboiarkin96.github.io. Verify with OPTIONS and a browser from that origin.
  3. Define how sandbox API keys are created, scoped, and rotated; document for integrators (not in OpenAPI text alone).
  4. Update services/portal/public/reference/api/index.html: tryItOutEnabled: true, persistAuthorization: true, servers → sandbox only, pinned Swagger UI + SRI or vendored assets. Prose: "Try it out targets the sandbox — not production."
  5. Smoke: from the GitHub Pages URL, Authorize, execute a write, then read back to confirm persistence.

Validation

  • Preflight: curl -s -X OPTIONS <sandbox>/<path> -H "Origin: https://iboiarkin96.github.io" -H "Access-Control-Request-Method: POST" -i — must reflect an allowed origin for the docs site.
  • Browser E2E: POST then GET (or equivalent) with sandbox credentials; assert non-production markers if you expose them (e.g. environment header).
  • SRI: if using CDN, script/link integrity matches the pinned file.
  • Contract: CI still runs make contract-test on the OpenAPI baseline; the sandbox does not replace that gate.

Goal

Offer a single public URL — for example https://iboiarkin96.github.io/etr_study_api/openapi/index.html — where a developer (internal provider or external partner) can open the API reference, use Try it out, and execute real HTTP requests against a sandbox deployment: real validation, real status codes, real side effects in a non-production database. No local clone, no Postman install, no waiting on someone else to run a server.

The page is an integration validation surface, not a substitute for make contract-test / CI: the OpenAPI file remains the normative contract; the sandbox exercises it against live code and storage.

Why the first approach did not work

The original approach tried to deliver validation without a cooperating server: Ajv in the browser, requestInterceptor / responseInterceptor, and tryItOutEnabled: false on real fetches. Three issues dominated:

  1. CORS. The GitHub Pages origin (iboiarkin96.github.io) was not an allowed browser origin for the API. Preflight failed; any true cross-origin call was blocked. Interceptors avoided real calls, so no useful signal reached the network.
  2. Fragile client coupling. Swagger UI requestInterceptor / responseInterceptor are tied to a specific bundle; CDN version drift can break the wiring independently of this repo.
  3. Wrong layer. JSON shape checks answer "valid JSON schema?"; integrators need "does the service accept, persist, and return what we expect?" That requires server-side validation and a real store — a sandbox, not a static linter.

Conclusion: fix the environment (sandbox + CORS + auth), not the interceptors.

How large API programs handle the same problem

Public docs often stay static (HTML + OpenAPI) while Try it out or console tools send traffic to a separate test or sandbox stack. Isolation, credentials, and CORS for the docs origin are first-class product concerns, not an afterthought. Representative public models:

Provider (example) What they document Link
Stripe Isolated test environments ("sandboxes"), test API keys; live vs test data separation docs.stripe.com/sandboxes, testing use cases
PayPal Virtual sandbox, sandbox endpoints, sandbox accounts and credentials separate from live PayPal sandbox testing guide
Amazon SP-API Dedicated sandbox hosts; static and dynamic mock behaviors; explicit sandbox operations where needed SP-API sandbox

Common structure: a dedicated or logically isolated test environment; credentials that are only valid there; the docs origin allowed by CORS (or a hosted proxy owned by the product); the browser as a thin client; the server as validator. This ADR follows that pattern: GitHub Pages for the spec + Swagger UI, sandbox API for execution.
