Structured Logging, Request Correlation, and Local Elasticsearch

NDJSON logs with stable fields, an X-Request-Id header on every response, and an optional Compose stack (Elasticsearch + Kibana + Filebeat) for local search.

Context

Why this matters: Plain text logs are fine for a single terminal but do not scale to centralized search, dashboards, or joining logs with future distributed traces. Large teams standardize on structured logs (JSON), request correlation (X-Request-Id), and optional trace/span ids once OpenTelemetry is adopted.

Elasticsearch and Kibana are widely used for log search and analytics. Elastic publishes Docker images that are free to run locally for development and learning; production licensing and support are separate commercial decisions. This ADR covers local optional Compose only—not a commitment to Elastic Cloud or a paid tier.

Decision

  • Support LOG_FORMAT=text and LOG_FORMAT=json (NDJSON: one JSON object per line). When LOG_FORMAT is unset, the process default is json so request_id is a first-class field. Stable JSON fields include @timestamp, level, message, service (LOG_SERVICE_NAME), app_env, request_id, trace_id, span_id (latter two null until tracing is wired).
  • Add HTTP middleware (outermost) that sets X-Request-Id on every response: reuse a valid incoming header or generate a UUID.
  • CORS expose_headers (CORS_EXPOSE_HEADERS) lists X-Request-Id (and rate-limit headers) so browser fetch on another origin can read them; OpenAPI is patched so Swagger UI shows an optional Parameters → X-Request-Id field and response headers for each operation.
  • Optional Docker Compose stack: Elasticsearch + Kibana + Filebeat (services/monitoring/docker-compose.logging.yml), mounting ./logs read-only into Filebeat. Ship NDJSON only: use LOG_FORMAT=json (or rely on the default json when LOG_FORMAT is unset).
  • Full distributed tracing in Elasticsearch (Elastic APM or OTLP → Elastic) remains out of scope here; see backlog item 17 (OpenTelemetry).

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives evaluated before adopting NDJSON + X-Request-Id + optional Compose ES. Each rejected with a rooted reason.

Plain-text logs with logger.info() + tail

rejected

Why considered: the Python default; zero setup; readable in a terminal.

Why rejected: text logs cannot be queried by field. The «find every log line for this request_id» operation degrades to grep-with-regex, which fails as soon as fields contain spaces, multi-line stack traces, or non-ASCII data. Structured logs make the query exact.

Hosted log aggregator (Datadog / Honeycomb / Loki Cloud)

rejected

Why considered: turnkey ingest + UI, no local stack to operate.

Why rejected: vendor lock at the ingest path; we ship NDJSON with field names tuned to a vendor's schema. The Elasticsearch + Kibana local stack uses ECS shapes (open standard); migrating to a different consumer later is a config-change, not a rewrite. Defer hosted aggregation until we ship for money.

OpenTelemetry collector + OTLP from day one

rejected

Why considered: the «one signal pipeline» pattern — logs, metrics, traces all flow through OTel.

Why rejected: OTel collector is a separate process with its own config + version skew. We are not yet emitting traces; bringing the collector in for logs alone overshoots the cost. Revisit when distributed tracing lands (backlog BL-017).

Trade we accept by choosing NDJSON + X-Request-Id + optional ES Compose. Stdlib-only JSON formatter, no external dependency, ECS-aligned. We accept it because each rejected alternative either forfeits the field-query property (text), locks us to a vendor (hosted), or adds a process before the benefit lands (OTel-from-day-one).

Consequences

Positive

wins
  • Logs can be indexed and searched in Kibana during local debugging.
  • Correlation id is visible to API clients and appears in log lines.
  • No new Python dependency for JSON logging (stdlib only).

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • NDJSON with stable schema (vs free-form text). We gain a log that Kibana, jq, and any future log shipper can index without bespoke parsers. We lose readability when tailing without a pretty-printer — a raw NDJSON line is unfriendly to the human eye. We accept it because jq or lnav is one pipe away, and the indexable property is the load-bearing «can we grep request_id in production-style storage?» test.
  • Elasticsearch + Kibana stack is heavy. We gain a real log-search UX that mirrors what production logging would look like. We lose ~2 GiB of memory when the stack is up, and accept that the local stack is optional (not a default). We accept it because the alternative — toy log viewers — fail to surface the indexing semantics (mapping conflicts, data streams) that real production debugging hits. Better to learn those on a local stack than during the first SEV-2.

Revisit — would we do this again?

Yes — the «request_id on every line» bet has paid back on every debug session. One refinement we would commit to on day one if starting today:

  • Document the «data-stream vs index pattern» Kibana gotcha as a runbook entry, not just in the ADR. The «study-app-logs-* vs *study-app-logs*» trap (Kibana shows zero docs while files grow on disk) cost ~30 min on first encounter and is recurring. A dedicated runbook step would let on-call resolve it in 60 s.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Kibana index pattern (important)

Create the data view with index pattern *study-app-logs*, not only study-app-logs-*. Otherwise Discover may target no indices: Elasticsearch 8 often writes Filebeat output to data stream backing indices named .ds-study-app-logs-…, which do not match a prefix of study-app-logs-. A symptom is "No results" for valid request_id queries even though documents exist in logs/app.log. Filebeat is configured with setup.template.type: legacy so new daily indices align with study-app-logs-*; the broad pattern still matches old and new names.

JSON lines must follow ECS shapes where the index template expects objects: use service.name (nested under service), not a string at service. Otherwise bulk indexing can fail with mapper_parsing_exception and indices show docs.count: 0 while logs/app.log still grows on disk.
