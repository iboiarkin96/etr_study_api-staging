External and internal API documentation

OpenAPI is the sole normative HTTP contract for external integrators; services/portal/internal/ is the engineering narrative for behavior and rationale.

Context

Why this matters: One group needs a stable contract and release notes; the other needs business rules, how things behave in production, and why we built it that way — hard to put fully in OpenAPI. If we mix both in random prose, external readers see internal detail they do not need, and engineers search code when text is missing.

We already govern HTTP quality via ADR 0007 (OpenAPI snapshot, examples, CI) and public change history via ADR 0013. This ADR does not change those rules. It only names which artifact is official for which audience and where internal narrative lives (services/portal/internal/; see ADR 0026).

Decision

  1. External contract (integrators). The OpenAPI document committed under services/portal/public/reference/api/ (baseline snapshot and the same contract exposed at runtime when the app serves /openapi.json) is the sole normative description of the public HTTP API for external development: paths, methods, schemas, declared errors, examples, and documented headers. Partners and third-party clients should treat it as immutable truth for what we publish at a given release; changes flow through versioning and changelog policy (ADR 0004, ADR 0013).
  2. External discovery aids. Swagger UI at /docs (when the app runs) and the static services/portal/public/reference/api/index.html page are views on that contract; they are not independent sources of truth. If explorer text disagrees with the committed baseline after a merge, the fix is to align code and baseline (ADR 0007), not to treat the browser copy as authoritative.
  3. Internal service documentation. Narrative that explains why the API behaves as it does, business invariants, cross-cutting processes (idempotency, error monitoring expectations, messaging topics), and traceability of decisions over time lives under services/portal/internal/ as HTML, governed by ADR 0026. Layout is multi-page: a landing page, service overview, an HTTP API hub under services/portal/internal/api/, and per-resource folders with a single hub (index.html) plus optional per-endpoint pages — see ADR 0026 and services/portal/internal/handbook/howto/0002-internal-service-docs-layout.html (contributor layout; STRUCTURE.md links there). It does not replace OpenAPI for external consumers.
  4. Python API reference. Generated HTML under services/portal/pdoc/ remains the reference for in-repo Python modules (per ADR 0016), distinct from HTTP OpenAPI and from services/portal/internal/.

Scope

  • In scope: Audience split, location of internal HTML docs, primacy of OpenAPI for external HTTP contract.
  • Out of scope: A separate public developer portal product, monetization, or org-wide API catalogue outside this repository.

Alternatives considered

Three alternatives evaluated before adopting the OpenAPI-external + internal/-narrative split. Each rejected with a rooted reason.

Single prose manual that covers everything

rejected

Why considered: one place to open, no audience confusion, classic «developer guide» pattern.

Why rejected: duplicates OpenAPI for the contract surface — and the duplicate drifts. Within a year the prose manual would describe a different shape than what the spec test enforces, and integrators would split into «which one do I trust?» factions. OpenAPI's CI-checked guarantee is the load-bearing property; any artefact that competes with it forfeits its own correctness.

OpenAPI-only, no internal/ narrative tree

rejected

Why considered: minimal files, one source of truth, no audience split to maintain.

Why rejected: OpenAPI cannot carry «why this 422 instead of 409», «what business invariant this enforces», or «why the read-after-write is eventually consistent here». Forcing those into the spec either bloats it (and breaks the «integrator-skimmable» promise) or pushes the team to keep the knowledge in chat — where it dies on rotation.

External developer portal product (e.g. Backstage, Mintlify, Redocly)

rejected

Why considered: nicer UX, multi-language SDK tabs, hosted analytics, organisation-wide discovery.

Why rejected: not justified at our scale (single internal integrator). The hosted portal would be empty 80% of the time and adds infrastructure the team would have to operate. We adopt the «one in-repo internal portal + raw OpenAPI for external» pattern; revisit the day there is a partner programme to discover.

Trade we accept by choosing OpenAPI-external + internal/-narrative. Two surfaces to maintain — but for two different audiences, with no overlap in authority. We accept it because each rejected alternative either lets prose drift from the contract test (single manual), starves the engineering audience of «why» (OpenAPI-only), or pre-commits to infrastructure we have not built (external portal).

Consequences

Positive

wins
  • Clear expectation: integrators trust OpenAPI + changelog; engineers extend internal docs when behavior is non-obvious.
  • Aligns with existing governance (ADR 0007, 0013) without changing contract-test meaning.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • Two surfaces (OpenAPI + internal/). We gain audience-fit: integrators read what they need, engineers read what they need, neither sees the other's noise. We lose the «one URL to send» convenience and accept that authors must keep the internal pages in sync when behaviour changes. We accept it because the alternative — one shared surface — produces both audiences seeing each other's noise; clarity for both is the bigger win.
  • OpenAPI is the contract; internal/ is rationale. We gain a uniform answer to «who wins on conflict?» — OpenAPI does. We lose the option of contradicting it on purpose (e.g. «the real behaviour is different, OpenAPI is a stub»). We accept it because the contract-test pipeline (ADR 0007) enforces OpenAPI matches reality; any «stub OpenAPI» pattern fights that gate, which is the wrong fight.

Revisit — would we do this again?

Yes. One refinement we would commit to on day one if starting today:

  • Add a cross-link convention in every internal spec back to the OpenAPI operation. Today many internal pages reference behaviour without naming the operation; a small data-openapi-operation-id="postUser" attribute on every spec page would let a CI step verify the cross-reference is current. The internal-and-OpenAPI alignment would become machine-checkable instead of human-reviewed.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation plan

  1. Adopt the conventions in the Decision section across the repository.
  2. Update related templates, runbooks, and CI gates to reflect the rule.
  3. Cross-link from CHANGELOG and onboarding docs so future contributors discover the decision.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.
  • Documentation updates: README, CHANGELOG, and related ADRs cross-reference this decision.
