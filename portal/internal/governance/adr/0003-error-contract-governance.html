Error Contract Governance

One versioned error contract — stable codes and keys, additive evolution, shared transport pool plus per-entity matrices.

Context

Why this matters: Clients cannot rely on free-text messages; they change with wording and locale. Stable codes and keys let apps and partners branch on fixed values, log clearly, and show their own user-facing text. One-off errors also make incidents harder to fix: “something failed” in logs is not enough to act on.

As we add endpoints, we need one shared list of transport errors (auth, rate limits, idempotency) and domain-specific tables per entity, next to OpenAPI and tests.

Decision

We use one versioned error contract in code and OpenAPI:

  1. Error payloads are part of public API contract.
  2. Use stable numeric code and symbolic key.
  3. code + key are immutable once published.
  4. Contract evolves additively; existing semantics cannot be silently changed.
  5. Validation and business errors follow explicit, documented schemas.
  6. Error catalog is split into:
    • COMMON pool for service-wide reusable errors (security/rate-limit/idempotency/core transport).
    • Entity matrices for domain-specific errors (for example USER_*).

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were evaluated before committing to the (code, key) + COMMON-vs-entity model. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

HTTP status codes only (no error envelope)

rejected

Why considered: the most RESTful answer — every error is a status code, the body carries a human-readable message, no custom catalog to maintain. Used by many small public APIs.

Why rejected: integrators need to branch on specific business errors (USER_001 «email already exists» vs USER_013 «email format invalid»), both of which are 422. Without a stable machine-readable discriminator, the integrator's only option is to regex the message — which breaks on i18n and on every wording change. Status codes are the transport layer of the contract; the semantic layer is the missing half.

RFC 7807 Problem Details (application/problem+json)

rejected

Why considered: the IETF standard for HTTP error payloads (RFC 7807, March 2016). Has a type URI that doubles as a documentation link, plus title, detail, status, instance. Recognised by FastAPI, Spring, and Microsoft's API guidelines.

Why rejected: RFC 7807's type is a URI, which is opaque to consumers that want a typed switch statement; «branch on https://errors.example.com/user/email-taken» reads as ceremony, not as a stable enum value. We adopt parts of the spirit — the type-as-docs-link idea reappears in § Would we do this again? below — but the wire format with (code, key) ergonomics beats URI-as-discriminator for typed-client integration.

gRPC-style status enum + Status / Details proto

rejected

Why considered: Google's gRPC defines a fixed Code enum (16 canonical codes) plus a details array of typed messages — strongly-typed, well-tooled, widely adopted in service meshes.

Why rejected: we ship HTTP/JSON over OpenAPI, not gRPC. Forcing 16-code-enum semantics into JSON loses the OpenAPI responses{} generation pipeline we rely on (ADR 0007); and the canonical-code set was designed for service-to-service RPC, not for partner integrations that need rich validation matrices like USER_001…USER_013.

Trade we accept by choosing (code, key) + COMMON pool. Each error needs an explicit numeric code and a symbolic key that we cannot reuse once published — that is the curation tax. We accept it because the cost of getting any of the rejected alternatives wrong is paid at the integrator's side (regex on messages breaks in production; URI-as-discriminator forces switch-on-string everywhere; gRPC-enum-on-HTTP loses OpenAPI tooling). Our cost is paid once per error code at authoring time and never again.

Consequences

Positive

wins
  • Predictable client integration behavior.
  • Better incident diagnostics and observability.
  • Easier compatibility management across API versions.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure: «we gain X, we lose Y, we accept it because Z».

  • Catalog as a curated artefact. We gain stable, immutable (code, key) pairs that clients branch on without fear of silent semantic drift. We lose the freedom to refactor error semantics once a code is published (a typo in USER_007 stays forever) and pay a curation tax every time a new business rule lands. We accept it because the integrator's branching code is the load-bearing surface — every silent semantic change there manifests as an outage on the integrator's side, not ours, where we cannot detect or fix it.
  • Two-layer split (COMMON pool + per-entity matrix). We gain a single canonical place for cross-cutting transport errors (auth, rate limit, idempotency) that every endpoint reuses verbatim. We lose the ability to vary transport behaviour per endpoint without a deliberate exception (and we accumulate boilerplate when an endpoint needs even a tiny per-route tweak). We accept it because the alternative — inline error declarations per router — drifted so reliably in past projects that COMMON-vs-USER_* is the cheapest deduplication boundary we know.

Revisit — would we do this again?

Yes, with one borrowing from RFC 7807. The (code, key) pair has aged well — every spec page now links to its error matrix and integrators have a stable switch surface. One addition we would commit to on day one if starting today:

  • Add a type URI field that points at the error's documentation page. Borrow this verbatim from RFC 7807. The integrator's runtime would surface a clickable link straight to the error matrix (e.g. /services/api/reference/_shared/error-catalog.html#USER_001) on every error response — turning every 4xx into a self-documenting incident. Cost is one extra string per error; benefit is that on-call triage stops at the response payload instead of needing to grep the catalog. We retrofit this; would have been free at day one.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Contract model

Two wire shapes — one for single-fact business errors, one for multi-field validation errors. Both are emitted as JSON; both are declared in services/api/app/schemas/errors.py and surfaced in OpenAPI via services/api/app/openapi/examples/errors.py.

Business error shape

Returned for any 4xx that names a single business or transport fault.

{
  "code": "...",
  "key": "...",
  "message": "...",
  "source": "business"
          }

Validation error shape

Returned for 422 when one or more request fields fail validation. The top-level envelope carries the endpoint and an array of per-field facts; each errors[] item is itself a complete error record with field-level pointers.

{
  "error_type": "validation_error",
  "endpoint": "...",
  "errors": [
    {
      "code": "...",
      "key": "...",
      "message": "...",
      "field": "...",
      "source": "...",
      "details": "..."
    }
  ]
          }
Field Type Purpose
code string stable numeric code (e.g. USER_001); the integrator's switch surface
key string symbolic identifier paired with code; both are immutable once published
message string human-readable text; locale-dependent, not for branching
field string dotted-path pointer to the offending request field (e.g. user.email)
source string where the error was raised — validation / business
details object extra context (constraint name, attempted value, range); optional, free-shape

Error catalog structure

Common reusable pool (service-wide)

Use COMMON_* for errors that should be reused consistently across all endpoints.

  • Examples: auth required, rate-limit exceeded, body too large, idempotency key conflicts.
  • Implementation: reusable OpenAPI response blocks in services/api/app/openapi/responses.py.
  • Rule: do not duplicate these responses inline in routers unless behavior differs.

Entity matrices (domain-specific)

Use entity-prefixed codes for business/validation rules specific to one domain.

  • Example: USER_001...USER_013 validation matrix and USER_101, USER_404.
  • Rule: entity code meaning is immutable once released.
  • Rule: each entity matrix must be reflected in OpenAPI examples and tests.

Operating rules

  1. Add new codes; do not repurpose old ones.
  2. Keep fallback validation codes for unmapped errors.
  3. Prefer shared COMMON response blocks for cross-cutting errors.
  4. Entity routers must keep their own matrix for domain-only errors.
  5. Update OpenAPI examples and docs in the same change.
  6. Ensure make docs-fix and make release-check pass before release.
