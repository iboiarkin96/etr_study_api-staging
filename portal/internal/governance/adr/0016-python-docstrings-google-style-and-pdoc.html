Python Docstrings (Google Style) and pdoc

Standardise on Google-style docstrings in app/, scripts/, tests/, and tools/ so reviewers, pdoc, and assistants follow the same checklist.

Context

Why this matters: OpenAPI describes HTTP; it does not replace module-level intent for internal services, repositories, and edge cases. Consistent docstrings make code review faster and give pdoc something stable to render for internal API reference.

One agreed style (Google-style) avoids bikeshedding and helps humans and assistants follow the same checklist when touching Python under services/api/app/, tests/, and tools/.

Decision

  1. Style: Use Google-style docstrings for all new and updated Python in services/api/app/, tests/, and tools/. Public functions and classes MUST have a docstring unless the name is completely self-explanatory and private (leading _); even then, prefer a one-line summary for anything non-trivial.
  2. Language: Docstring text is in English (same as code identifiers and API error keys). Comments for local teams may use other languages in separate docs, not in docstrings.
  3. First line: Imperative or descriptive one-line summary (subject: verb), ending without a period if it is a phrase fragment, or with a period if it is a full sentence—either is acceptable; stay consistent within a file.
  4. Sections (use when applicable): Args, Returns, Raises, Yields, Note / Notes, Attributes (classes and dataclasses).
  5. pdoc: Generated HTML under services/portal/internal/services/api/code-reference/ is the browseable view; keeping docstrings accurate is part of maintainability. See Documentation pipeline, §6.

Scope

  • In scope: docstrings, module-level """...""" for packages, ADR and editor rules.
  • Out of scope: replacing OpenAPI descriptions; changing Ruff/mypy rules solely for docstrings.

Alternatives considered

Three alternatives evaluated before adopting Google-style + pdoc. Each rejected with a rooted reason.

NumPy-style docstrings

rejected

Why considered: dominant in scientific Python stacks; rich underline-style sections.

Why rejected: visually heavier (every parameter section gets its own underline + indent), and the repo already had Google-style scattered in scripts/sync_docs.py when this ADR landed. Picking NumPy would have forced a one-shot rewrite of every existing docstring; picking Google made the existing files conformant for free.

Sphinx (reStructuredText) + autodoc

rejected

Why considered: maximum features — cross-references, custom directives, the Read-the-Docs ecosystem, used by most large Python projects.

Why rejected: Sphinx is the heaviest tool in Python docs land. conf.py, intersphinx mapping, theme customisation, RST syntax all add setup cost. pdoc gives a «cargo doc»-style zero-config experience that matches our «keep the toolchain small» bet — see ADR 0001's «no extra non-Python dependencies» reasoning.

No docstrings — type hints + module-level comments only

rejected

Why considered: «the code is the docs» — type hints carry the shape, comments carry the intent, no separate docstring discipline.

Why rejected: type hints describe shape but not intent — they cannot say «raises HTTPException on cross-tenant access» or «idempotency replay returns the stored response, not a fresh one». pdoc has no signal to render without docstrings. The «no docstrings» world is the world where every new reader has to read the implementation; we already pay the docstring tax once per function and the savings compound across every reader.

Trade we accept by choosing Google-style + pdoc. Every public function pays the docstring-authoring tax and contributors must learn the Google section names. We accept it because each rejected alternative either rewrites the world (NumPy), brings a heavier toolchain than our «keep it small» bet allows (Sphinx), or sacrifices intent for shape (no docstrings).

Consequences

Positive

wins
  • Single convention for reviews and for generated API HTML.
  • Agents can follow the ADR and the Cursor rule without guessing.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • Opportunistic (not retroactive) migration. We gain a zero-cost adoption — files keep working unchanged until touched. We lose temporary inconsistency: a reader of the codebase sees a mix of old phrasing and Google-style for a few months. We accept it because forcing a one-shot rewrite would mean a single PR that touches every Python file (a merge-conflict magnet); opportunistic updates land naturally.
  • English-only docstrings (no Russian or other local-language tags). We gain alignment with code identifiers, API error keys, and the rest of the public surface — everything is in one language. We lose the «more accessible to native speakers» property a multilingual policy would carry. We accept it because the codebase has one writing voice and that voice is English; bilingual docstrings would compound the maintenance surface without a matching audience.

Revisit — would we do this again?

Yes. One refinement we would add on day one:

  • Wire a docstring-coverage gate (interrogate) into make verify. Today docstring presence relies on review discipline; interrogate would surface gaps quantitatively (e.g. «services/api/app/services/conspectus.py at 60% — needs 4 more functions documented»). Cheap to add, hard to skip.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation plan

  1. Adopt the conventions in the Decision section across the repository.
  2. Update related templates, runbooks, and CI gates to reflect the rule.
  3. Cross-link from CHANGELOG and onboarding docs so future contributors discover the decision.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.
  • Documentation updates: README, CHANGELOG, and related ADRs cross-reference this decision.

Module

Optional longer context: what callers use this for, main exports.
        """

Function

"""Do one specific thing (summary line).

Args:
name: What this parameter represents.
other: Second parameter.

Returns:
What the caller receives.

Raises:
ValueError: When the input is invalid.
HTTPException: When returning a business-layer error (document status if non-obvious).
        """

Class

"""Short description of the class role.

Attributes:
field: Meaning of this attribute.
        """

Method / property

Same as function; use Args including self only when non-obvious.

Checklist for humans and LLM agents

  • After changing behavior, update the docstring in the same PR.
  • Do not restate type information that is already in type hints unless it adds semantics.
  • For FastAPI route handlers, document idempotency, security assumptions, and replay behavior where not obvious from OpenAPI.
  • For service/repository methods, document Raises for HTTPException and business rules.
  • Prefer linking related concepts with :class:`... / :func:`... (Sphinx-style cross-refs) where pdoc can resolve them; otherwise use plain backticks for code names.
  • Run make docs-fix locally when changing public app APIs to verify pdoc rendering.
