Portal staging — separate GitHub Pages repository per environment

Stand up a pre-release stage for the portal on GitHub Pages without breaking the prod publish pipeline. A single repo cannot host two Pages sites, so we keep <app> as the source of truth and add a thin mirror repo <app>-staging that CI publishes to on every push to the staging branch of <app>. Prod stays on main, staging gets its own URL, own cache, own 404, and rollbacks are a single revert.

Context

The portal is published on GitHub Pages directly from the repository's main branch — ADR-0033 made the build static enough for this to work without server-side infrastructure. The publish pipeline now has one gap: there is no intermediate environment between «works on my laptop with python -m http.server» and «live on the prod URL».

That gap matters because the portal has client-side behaviour that a local file:// or single-port dev server does not exercise:

  • Pagefind bundle loading uses fetch relative to the deployed base path; a wrong BASE_URL only manifests under a real Pages domain.
  • The UI Kit v2 web components rely on absolute asset paths declared in nav-tree-internal.json.
  • Theme persistence and breadcrumbs cross-link across many pages — broken relative refs become obvious only on a real navigation graph.

GitHub Pages itself imposes the hard constraint: one repository publishes exactly one site, sourced from one branch. There is no built-in second environment, no per-PR preview, and no first-class staging URL. Any solution has to work around that limit.

Decision

We adopt the dual-repository pattern: the source repo <app> publishes prod from main; a second public repo <app>-staging, which never holds application source, publishes the staging build from its own main branch, populated only by CI.

  1. Branch model in <app>: feature/* → PR → staging → PR → main. Hotfixes may go straight to main; they are forward-ported to staging in the next merge. Branch naming follows ADR-0017.
  2. Push to main triggers .github/workflows/deploy-prod.yml — published via the official actions/upload-pages-artifact + actions/deploy-pages path so nothing extra lands in git. Project-base resolution is runtime, not build-time — see amendment 2026-06-12.
  3. Push to staging triggers .github/workflows/deploy-staging.yml — pushed via peaceiris/actions-gh-pages@v3 with external_repository: <owner>/<app>-staging, publish_branch: main, and force_orphan: true so the staging history stays a single commit and the repo does not bloat over time.
  4. Cross-repo write authority is held by a fine-grained PAT with contents:write scoped to <app>-staging, stored as the STAGING_DEPLOY_TOKEN secret on <app> (W0, 2026-06-11). A GitHub App with the same scope, fetched via actions/create-github-app-token, remains the documented future option when audit-trail or non-user-bound authority becomes a requirement.
  5. Both workflows declare concurrency: { group: pages-deploy-<env>, cancel-in-progress: false } so concurrent pushes serialise per-environment without dropping a build silently.
  6. Environment-specific configuration (API URL, feature flags, robots-meta) is supplied through GitHub Environments (production, staging) declared on <app>. The prod environment carries a Required reviewer for the deploy job.

Scope

  • In scope: one prod environment and one staging environment for the portal published from <app>.
  • In scope: the branch model, CI workflows, cross-repo authentication, and the project-base contract the static portal honours at runtime (resolved from window.location.pathname; see amendment 2026-06-12).
  • Out of scope: per-PR preview deployments (none of the available Pages mechanisms support it cleanly).
  • Out of scope: password-protected staging — GitHub Pages on a public repo does not authenticate. A noindex meta tag is the only mitigation we ship.
  • Out of scope: multi-environment layering (qa / uat / canary). If a third environment becomes necessary, a successor ADR replaces this one.

Alternatives considered

Four alternatives evaluated before adopting the dual-repository pattern. Each rejected with a rooted reason.

No staging — push straight to prod

rejected

Why considered: zero infrastructure, zero CI changes, no second token to rotate. The portal is static — «just open the HTML locally» catches most regressions.

Why rejected: Pagefind fetches the bundle relative to the deployed base path; nav-tree links are absolute; theme persistence is cross-page. A class of bugs only appears under a real Pages domain at the right base path, and the prod URL is the wrong place to discover them. Cost of one demo-time regression is larger than the cost of standing up a second Pages site.

Single repo, staging in a /staging/ subfolder

rejected

Why considered: one repo, one Pages site, no cross-repo token. CI pushes prod build to / and staging build to /staging/ using keep_files: true.

Why rejected: shared 404 (prod's 404.html serves staging too), shared edge cache (a prod deploy can warm the CDN with assets staging then has to live with), shared CNAME, and a non-trivial race when prod and staging push concurrently to the same publish branch. Cheap to set up, leaky in operation — and the BASE_URL juggling needed to keep two sub-builds asset-correct doubles the surface that can break.

Cloudflare Pages / Netlify / Vercel as the staging host

rejected

Why considered: per-branch preview and per-PR preview deployments out of the box, native Production vs Preview env vars, Access-based password protection on free tiers.

Why rejected: introduces a vendor outside the GitHub surface where everything else already lives — CI, registry, governance, audit trail. We would be running two unrelated deploy runtimes (Actions for prod, the vendor for staging) with two cache models and two outage stories. The win on PR previews does not pay for the split until the team is larger than it is. Reconsidered when team or volume grows; a successor ADR is the path.

Per-PR preview via GitHub Pages alone

rejected

Why considered: theoretically achievable by publishing each PR into /pr-<n>/ inside a single gh-pages branch, with a cleanup hook on PR close.

Why rejected: inherits every problem of the subfolder option (shared 404, cache, CNAME) and adds a cleanup workflow whose failure modes — orphaned previews, accidental deletion of prod, branch-protection conflicts — are harder to recover from than the original problem. GitHub Pages has no first-class preview primitive; faking one inside it is a maintenance trap.

Trade we accept by choosing dual repos. We carry one extra repository and one cross-repo write token. In return we get isolated domains, isolated caches, isolated 404 pages, independent rollback, and a deploy flow that uses no infrastructure outside GitHub.

Consequences

Positive

wins
  • Staging gets its own URL, own edge cache, own 404.html, own CNAME — every Pages-specific class of bug surfaces in isolation, not on the prod domain.
  • Promotion is a git operation, not a deploy script: staging → main via PR. Reverting a prod deploy is a single revert on main, then the next CI run heals Pages.
  • Prod publish remains on the official actions/deploy-pages path — no committed build artefact in <app>, no churn in the prod repo's history.
  • No new vendor surface. CI, secrets, audit log, governance all stay inside GitHub.
  • Project-base contract becomes explicit and tested — serving the same _site bytes under a second base path (/<app>-staging/ vs /<app>/) exercises the runtime resolver in portal-href.js + search.js, catching base-path regressions before prod sees them.

Trade-offs

costs

Three named tradeoffs. Shape from Tradeoff disclosure.

  • Cross-repo write authority lives in CI. We gain a publish path that does not require humans to touch <app>-staging. We lose the «one repo, one trust boundary» story — a leaked CI secret can overwrite the staging site. We accept it because the staging site has no user data, no production credentials, and the secret is scoped to contents:write on one repo only (no organisation-wide write, no workflow rights).
  • No per-PR preview. We gain a single, predictable staging URL the team can rely on. We lose the ability to compare two unmerged feature branches side-by-side in deployed form. We accept it because team size is small enough that staging serialisation is acceptable; the moment that stops being true, a successor ADR moves staging to a host that has native preview deploys.
  • Public-by-default staging. We gain free hosting on GitHub Pages. We lose the option to hide pre-release content behind auth — the staging site is reachable by anyone with the URL, and a noindex meta tag is a request, not a control. We accept it because the staging build will not contain secrets and the threat model is «accidental discovery», not «motivated adversary».

Revisit — what triggers a successor?

This decision should be re-opened when any of the following becomes true: (a) the team grows past the point where one contended staging branch is the bottleneck and per-PR previews start to matter, (b) the portal begins to render content that should not be world-readable in pre-release form, or (c) we want a unified preview/staging story for more than just the portal (e.g. the future public API explorer). All three roads lead away from raw GitHub Pages — Cloudflare Pages and Netlify are the most likely successors.

Compatibility and migration

Roll-out runs in three waves (W0–W2). Each wave is independently reversible.

  • W0 — Provisioning. Create the public repo <owner>/<app>-staging (no source, no LICENSE-only is fine). Enable Pages → Source = Deploy from a branch, branch = main, folder = / (root). Create a GitHub App with contents:write scoped to <app>-staging; install it on the owner; store the app ID and private key as STAGING_DEPLOY_APP_ID and STAGING_DEPLOY_APP_KEY in <app>'s Actions secrets. PAT fallback: a fine-grained PAT with the same scope, stored as STAGING_DEPLOY_TOKEN.
  • W1 — Pipelines. Land .github/workflows/deploy-prod.yml and .github/workflows/deploy-staging.yml. Verify locally that the same _site bytes resolve correctly under both /<app>/ and /<app>-staging/ base paths — runtime resolver carries the project-base contract, no build-time parameterisation needed (amendment 2026-06-12). First push to staging publishes the staging site within ~60 s.
  • W2 — Hardening. Add branch-protection rules to main and staging in <app> (require PR, at least one approving review, status checks green). Declare GitHub Environments production and staging with environment-scoped secrets; add a Required reviewer to the production environment so prod deploys wait for an explicit click. Inject <meta name="robots" content="noindex"> into the staging build at compile time.

Rollback strategy. Disabling staging is removing deploy-staging.yml and the secrets — the prod path is untouched. Disabling prod-staging coupling is reverting the merge that created the staging branch protections. The <app>-staging repo can be archived without touching <app>.

Implementation mapping

  • Prod workflow: .github/workflows/deploy-prod.yml — trigger push: branches: [main]; uses actions/upload-pages-artifact + actions/deploy-pages; environment production.
  • Staging workflow: .github/workflows/deploy-staging.yml — trigger push: branches: [staging]; uses peaceiris/actions-gh-pages@v3 with external_repository and force_orphan: true; environment staging.
  • Cross-repo token: fine-grained PAT (contents:write on <app>-staging), secret STAGING_DEPLOY_TOKEN on <app>. GitHub App installation tokens via actions/create-github-app-token (secrets STAGING_DEPLOY_APP_ID, STAGING_DEPLOY_APP_KEY) remain a documented future option.
  • Project-base contract: resolved at runtime by services/frontend/portal/assets_v2/ui-kit/components/portal-href.js (HTML/nav hrefs) and services/frontend/portal/assets_v2/ui-kit/components/search.js (Pagefind baseUrl), both reading from window.location.pathname. The static _site artifact is identical for prod and staging.
  • Branch protections: main and staging require PR + 1 review + status checks; force-push disallowed.
  • Operational procedure: How-to · Release to staging.

Validation

  • After W0: visiting https://<owner>.github.io/<app>-staging/ serves a placeholder; Pages settings page reports «Your site is live».
  • After W1: a commit on staging updates the staging URL within ~60 s; the prod URL is unchanged; deploy-staging.yml reports a successful push to <app>-staging.
  • After W1: a commit on main updates the prod URL via actions/deploy-pages; the staging URL is unchanged.
  • After W1: two concurrent pushes (one to main, one to staging) both succeed — neither cancels the other.
  • After W2: a force-push to main is rejected by branch protection; a deploy to the production environment pauses until the Required reviewer approves.
  • Asset paths: open the staging URL with the network panel filtered to 404s — zero failed asset loads. Same check on the prod URL.
