Idempotency For Write Operations

Critical writes require an Idempotency-Key; the server deduplicates by key + payload hash so retries are safe.

Context

Why this matters: Networks drop packets; clients time out and retry; mobile apps tap “submit” twice. If the server treats every retry as a new action, you get duplicate users, double charges, or conflicting updates. Idempotency means: “send the same logical request again with the same key, and the server does not apply the side effect twice—it returns the same outcome as the first success.”

This is usual for payment and write APIs; we use it for critical writes so integrations stay safe when clients retry.

Decision

  1. All critical write operations must support idempotent retry behavior.
  2. Client sends Idempotency-Key for write requests.
  3. Server deduplicates by (endpoint, idempotency_key, payload_hash).
  4. Same key + same payload replays original success response.
  5. Same key + different payload returns conflict error.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were evaluated before adopting Idempotency-Key + body-hash dedup. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

No idempotency — client must not retry

rejected

Why considered: the simplest possible policy. No header, no dedup table, no body hashing. Used by many internal-only APIs where the caller is fully trusted to handle retries.

Why rejected: our integrator is a mobile app. Mobile networks drop packets; users double-tap submit; OS-level retries happen below the app's awareness. «Client must not retry» is not enforceable for any of these. The failure mode is duplicate user rows or double-applied PATCHes, both of which require a manual data fix — the worst incident class for a solo on-call.

Natural idempotency via client-generated resource IDs

rejected

Why considered: let the client choose the resource UUID; the server treats PUT /users/<client-uuid> as idempotent by definition. Clean REST, no extra header. Used by some Google Cloud APIs (predictable resource names).

Why rejected: the client-chosen UUID then becomes part of the permanent public surface. A client that re-uses the same UUID across user sessions (accidentally or via a buggy client lib) would silently collide with another user's resource. Idempotency keys are scoped to «one logical request» and expire (24 h); resource IDs cannot expire without breaking every URL ever issued.

At-least-once + conditional PUT (CAS / ETag)

rejected

Why considered: the S3 / DynamoDB pattern — every PUT carries an If-Match: <etag> header; conflicting writes are rejected with 412. Strong consistency, no dedup table needed.

Why rejected: CAS only works when the resource already exists (you need an ETag to compare). For POST (creation) it provides no help at all — and the duplicate-create case is exactly where idempotency matters most for a solo learner-app where every duplicate user adds noise to the corpus. CAS is a complement to idempotency-keys, not a replacement.

Trade we accept by choosing Idempotency-Key + body-hash dedup. Every critical-write endpoint carries one extra header and one extra DB row per request. We accept it because each rejected alternative either fails open under the exact failure mode we are designing for (no idempotency on a flaky-network mobile client), accepts a permanent surface change that cannot be undone (client-generated resource IDs), or solves the wrong half of the problem (CAS for create). The header + table is the smallest mechanism that covers both POST and PUT/PATCH uniformly.

Consequences

Positive

wins
  • Safe retries for transient failures and network timeouts.
  • Reduced risk of duplicate writes in real integrations.
  • Explicit, testable retry contract for client teams.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure: «we gain X, we lose Y, we accept it because Z».

  • Mandatory Idempotency-Key on every critical write. We gain a uniform contract that the client cannot accidentally lose (no «I forgot the header on retry»). We lose the «one-liner curl example» — every write needs the header attached, even in dev. We accept it because the curl-friction is paid once per developer when they read the spec, while the duplicate-write incident in production is paid by the integrator at 3 a.m. and is unrecoverable without a manual data fix.
  • Persistent dedup table (extra DB row per write). We gain replay safety across server restarts — a 30-second retry after a pod restart still hits the same idempotency record. We lose one extra DB write per critical-write endpoint and accumulate rows in idempotency_keys. We accept it because the 24 h TTL bounds growth (a cron prunes), and the DB-write cost is dwarfed by the actual business write that follows; the alternative is a memory cache that vanishes on restart exactly when retries matter most.

Revisit — would we do this again?

Yes, with one client-side helper we did not ship. The header + body-hash + 24 h TTL has aged well — zero on-call incidents tied to duplicate writes, and the integrator's retry code is a one-line «attach UUID4 as the header» recipe. One thing we would add on day one if starting today:

  • Publish a tiny client-side helper (Python + JS) that generates the key and retries on transient failures. Stripe ships stripe.idempotency_key() in their SDKs; we left this as «attach a UUID4 yourself» in the spec. Two integrators have re-written the same retry-with-backoff wrapper from scratch — a 30-line helper in app/integrations/ would have saved both of those, and would let the spec point to the canonical implementation instead of describing it.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Initial Implementation

Missing key 400 IDEMPOTENCY_KEY_REQUIRED —

  • Tests must include replay scenario and key-conflict scenario.
  • Runbooks should cover idempotency key misuse and retry errors.
