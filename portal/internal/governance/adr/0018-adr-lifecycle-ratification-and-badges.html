ADR Lifecycle, Status Model, and Ratification

Define one ordered weight scale and ratification pattern (Issue + PR + accepted date) so every ADR shows a current milestone and an auditable trail without a separate tracker.

Context

Why this matters: Readers must see whether an ADR is still a draft, accepted, or done. One small machine-readable field on each page keeps the site accurate without a separate "ADR status" spreadsheet.

ADRs are static HTML under services/portal/internal/governance/adr/. We use one lifecycle: what we decided, how we ratified it (Issue + PR), and how far documentation and code have progressed — without a new ticket system only for status.

Status model (normative). One ordered scale (-1 … 7): Decision (0–2), Documentation (3–4), Implementation (5–7). Authors set one attribute on <main>: data-adr-weight. That number is the current milestone. The UI shows Current status and the Status log (green = done; yellow = current step except Implementation: Done, which stays green when it is current; gray = not reached). Code: services/frontend/portal/assets/docs-nav.js (renderLifecycleStatusBlocks). RFCs under services/portal/internal/governance/rfc/ use the same scale with data-rfc-weight on <main> (see RFC index).

The ADR HTML template is the starting shell: data-adr-weight, optional value table, and sections (Ratification, Context, Decision, …). New ADRs copy that file; they use this model, they do not replace it.

Decision

  1. Status model (normative here). Fixed milestones: Decision — Proposed (0), Accepted (1), Superseded (2); Documentation — Draft (3), Ready (4); Implementation — Not started (5), In progress (6), Done (7). Bad data-adr-weight values are clamped in script. This ADR introduced the model; older ADRs may not use it until someone updates them.
  2. Ratification. For work after this ADR, put the discussion Issue (ADR template, [ADR] title), merge PR, and acceptance date in the ADR Ratification section (see template). ADRs 0017 and below used older rules; from 0018 on, use this pattern.
  3. Documentation changelog. Important changes under services/portal/ go in services/portal/CHANGELOG.md. The root CHANGELOG.md is for product/API (see ADR 0013).
  4. Docs-only PRs. Pull requests that only change docs (including ADRs) can merge to main before code; code can follow in later PRs.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives evaluated before adopting the data-adr-weight + 8-state model. Each rejected with a rooted reason.

Free-text status field (e.g. «Status: Accepted» in prose)

rejected

Why considered: the MADR / Nygard default — a one-line status in the document body. No JS, no attribute, no parser.

Why rejected: free-text status cannot drive a renderer (Status pill, lifecycle popover, status-timeline). It also drifts: «Accepted» vs «accepted» vs «active» vs «in use» all show up in the same corpus over time. A machine-readable enum (the weight) collapses the variation and lets the renderer normalise the visible label.

External ADR tracker (Linear / Notion / Confluence)

rejected

Why considered: rich UI for browsing decisions by status, filtering, querying. Used by some enterprise teams.

Why rejected: separates the status from the artefact. Same failure mode as wikis (ADR 0001's rejected wiki alternative): the day someone updates the Notion status but not the file, the source of truth becomes ambiguous. The renderer also needs the artefact's status, so the external tracker becomes a parallel system to keep in sync, not a replacement.

Five-state model (Proposed / Accepted / Deprecated / Superseded / Rejected) — Nygard original

rejected

Why considered: the canonical Nygard taxonomy from «Documenting Architecture Decisions» (cognitect.com 2011). Well-known, less ceremony.

Why rejected: conflates the decision lifecycle («Accepted») with the implementation lifecycle («has the code shipped yet?»). We routinely have ADRs that are decision-Accepted but implementation-in-progress — the five-state model has no place to show that. The 8-state expansion (Decision · Documentation · Implementation phases) lets the renderer show exactly where a decision is along its arc.

Trade we accept by choosing data-adr-weight + 8-state model. Every author updates one HTML attribute when the milestone changes. We accept it because each rejected alternative either decouples renderer from artefact (free-text), introduces a second system to drift against (external tracker), or fails to separate decision-vs-implementation (Nygard five-state).

Consequences

Positive

wins
  • One agreed status semantics for all ADRs; the template and JS stay aligned.
  • Readers see current milestone and full log without leaving the page.
  • Discussion and ratification remain auditable via Issues and PRs.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • One data-adr-weight attribute as source of truth. We gain a single machine-readable field that the renderer, the audit, and the search index can all key off. We lose the «just update Notion» convenience and accept that authors must touch the file when the milestone changes. We accept it because docs-as-code (ADR 0001) means the artefact is the truth; a separate status tracker would re-introduce the drift this ADR exists to prevent.
  • Legacy ADRs not retroactively conformant. We gain a zero-cost adoption — ADRs ≤ 0017 keep working unchanged. We lose visual uniformity across the ADR set until each old file is touched. We accept it because opportunistic updates land naturally during the 2026-06 audit-pass; a one-shot rewrite would have produced a single PR touching every ADR (impossible to review).

Revisit — would we do this again?

Yes — the model has held up across 30 ADRs. One refinement we would commit to on day one:

  • Wire data-adr-weight drift into the CI gate. Today the weight is honour-system; an ADR can claim «Implementation: Done» while its referenced runbook is missing. A small CI check that follows the cross-links and flags «claims Done but the artefact does not exist» would close the «aspirational status» loophole. We retrofit this; would have been free at day one.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation plan

  1. Adopt the conventions in the Decision section across the repository.
  2. Update related templates, runbooks, and CI gates to reflect the rule.
  3. Cross-link from CHANGELOG and onboarding docs so future contributors discover the decision.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.
  • Documentation updates: README, CHANGELOG, and related ADRs cross-reference this decision.

How we mark ADR work

Three pieces — keep them distinct:

  1. GitHub Issue (discussion). Use .github/ISSUE_TEMPLATE/adr_discussion.md. Start the title with [ADR] so people can find the thread. Discussion stays in the Issue. Labels are optional; the template and title prefix are enough.
  2. ADR page (lifecycle on the site). Set data-adr-weight on <main> so the docs UI shows progress. That is only the published milestone; it does not replace Issue and PR history.
  3. Ratification. In the merged ADR, link the Issue, merge PR, and acceptance date so the HTML matches the audit trail.

Roadmap rows in services/portal/internal/governance/backlog/ use their own status pills. Branch names follow ADR 0017. Those are unrelated to data-adr-weight.
