Testing Strategy — Automated, Contract, and Load

Combine fast automated API tests, OpenAPI contract checks, and optional load testing so each layer answers the question only it can answer — correctness, drift, and behaviour under volume.

Context

Why this matters: Different tests answer different questions. Fast API tests prove correctness of behavior and errors. Contract tests prove the published OpenAPI snapshot did not change by accident. Load tests prove behavior under concurrency and rate limits—something unit tests cannot do—and should not run on every commit if they are slow or environment-specific.

ADR 0002 mandates API tests; ADR 0007 adds OpenAPI checks. This ADR ties those layers together and places optional load testing beside observability (ADR 0009, ADR 0011).

Decision

Layers

  1. Automated API / integration tests (required) — pytest against the FastAPI app (in-process TestClient or equivalent). Covers happy paths, validation, auth, and business errors. This is the default regression net before merge.
  2. Contract / OpenAPI governance (required) — make openapi-check and make contract-test guard schema drift and breaking changes relative to the checked-in baseline.
  3. Manual exploratory testing — allowed for UX and edge cases not worth automating; findings should become tests or ADRs when behavior must stay fixed.
  4. Load and traffic-shape testing (optional, not a default CI gate) — used to validate latency/error mix, rate limits, and dashboards under synthetic traffic. Implemented via the repository tool python -m tools.load_testing.runner with Python scenarios under tools/load_testing/scenarios/. Run against local or staging instances, not as a substitute for unit/API tests.

Load testing and safety

  • Simulating HTTP 5xx against the same process for metrics requires an explicit opt-in endpoint enabled only when LOADTEST_HTTP_500=true (see application configuration). Never enable in production.
  • Load scripts must not embed secrets; use the same .env / key conventions as local development.
  • Interpret results together with Prometheus/Grafana (SLO panels per ADR 0011), not raw request counts alone.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives evaluated before adopting the four-layer strategy. Each rejected with a rooted reason.

Unit-tests-only (no integration / contract / load)

rejected

Why considered: the fastest possible feedback loop. Each module tested in isolation, no fixtures.

Why rejected: unit tests cannot prove the contract holds end-to-end. The 2026-03-14 OpenAPI baseline-drift postmortem is exactly what happens without contract tests; unit tests passed throughout the drift and the integration test in tests/api/v1/ was the one that finally caught it.

External load tool (k6 / Locust / JMeter)

rejected

Why considered: mature load-testing ecosystems with rich reporting, distributed workers, and JS / Python scenario DSLs.

Why rejected: running k6 against our app requires installing k6 on the contributor's machine plus reproducing the auth/idempotency header dance in the k6 DSL. The in-tree Python runner reuses the same models the API uses (pydantic), so scenarios stay in sync automatically. Switching to k6 is the right move the day load testing becomes a CI gate; not before.

Property-based testing (Hypothesis) as the only API-level layer

rejected

Why considered: Hypothesis-style property tests find unknown-unknowns at the schema boundary that example-based tests miss. Faster path to high coverage.

Why rejected: property tests need an oracle (a separate model to check against), which doubles the maintenance surface. Example-based tests are the documentation surface integrators read (per ADR 0002's «success+failure» minimum). The right answer is example floor + property layer for invariants (see ADR 0002 retrospective), not property tests alone.

Trade we accept by choosing four-layer + in-tree load runner. Each layer adds ~10-20 min of authoring time per endpoint. We accept it because each rejected alternative either misses the drift class this strategy exists to catch (unit-only), adds tooling overhead disproportionate to the gain at our scale (k6), or removes the documentation surface integrators rely on (property-only).

Consequences

Positive

wins
  • Clear separation: CI proves correctness; load tools prove behavior under volume and error mix.
  • Repeatable traffic recipes checked into git (YAML).

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • Load testing optional, not a CI gate. We gain a fast pre-merge pipeline (no minutes spent on slow synthetic load). We lose «every PR proves it doesn't regress under volume». We accept it because regressions under volume in this project come from query shape (not concurrency); pre-merge load testing would burn CI budget without catching the failure mode we actually see.
  • In-process TestClient for API tests (not real HTTP). We gain millisecond-fast test feedback (~5 ms per request vs ~20 ms HTTP). We lose coverage of the ASGI server's behaviour (uvicorn signal handling, header rewriting). We accept it because the ASGI layer is third-party code we do not modify; the test pyramid's «integration» level is for our code paths, not framework verification.

Revisit — would we do this again?

Yes, with one missing layer. The example-based floor + contract gate + optional load runner has caught the regressions we have seen. The missing piece (called out in ADR 0002 retrospective too):

  • Add a Hypothesis property layer for invariants. Specifically for the conspectus model (created_at ≤ updated_at, four-slot scheduling monotonicity, ownership comparison). Backlog item BL-006. The property layer is the cheapest way to catch unknown-unknowns that example tests will not produce.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.
