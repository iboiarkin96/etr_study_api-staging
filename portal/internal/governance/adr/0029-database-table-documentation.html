Database table documentation conventions and template

Normative naming and column conventions for every table in services/api/app/models/, plus a fixed-shape per-table documentation template under services/portal/internal/catalog/api/data/.

Context

The application persists state in a relational store (SQLite today per ADR 0015; the schema is portable to PostgreSQL). Twenty-plus columns across the existing ORM models (services/api/app/models/core/, services/api/app/models/reference/) already follow consistent — but unwritten — conventions: UUID primary keys, timezone-aware audit timestamps, soft-delete flags, indexed foreign keys, snake_case plural table names. Until now, that convention lived only in code review and in maintainers' heads.

New collaborators have no normative reference for "what does a well-formed table look like in this repo?" — and there is no canonical place where each table's business meaning, lifecycle, and the HTTP endpoints that write or read it are documented. The OpenAPI baseline describes the wire shape; the ORM source describes the column layout; nothing in services/portal/ joins those two views into a single authoritative table-level narrative.

We need (a) a normative naming and structural convention for tables and columns, and (b) a fixed-shape documentation template for each table, with the table specs reachable from both the internal API hub and the catalog entity for the api service.

Decision

1. Naming conventions

  • Table names are snake_case, plural (e.g. users, systems, invalidation_reasons, timezones, idempotency_keys). The plural form makes the cardinality unambiguous in joins.
  • Column names are snake_case, singular, and describe the value, not its representation (e.g. created_at, not created_dt; full_name, not name_str).
  • Primary keys for core business tables are UUID strings stored as String(36), named <entity_singular>_uuid (e.g. client_uuid for users, system_uuid for systems). Stable across systems; never expose autoincrement integers in API surfaces.
  • Primary keys for reference tables may be a short stable code string (e.g. timezones.code = "Europe/Moscow") when the value is itself the canonical identifier in an external dictionary (IANA, ISO).
  • Foreign-key columns mirror the referenced PK and end with _uuid (or the referenced PK name for code-keyed tables). Example: users.system_uuid → systems.system_uuid; users.timezone → timezones.code.
  • Index names follow SQLAlchemy defaults; explicit unique-constraint names use the prefix uq_<table>_<cols> (e.g. uq_users_system_user_id_system_uuid).

2. Mandatory column set per table category

Every model in services/api/app/models/ belongs to one of three categories. Each category has a mandatory column set:

Category Examples Mandatory columns
Core business users, future conspectuses, review_slots <entity>_uuid PK · created_at (TZ-aware UTC, DateTime(timezone=True)) · updated_at (TZ-aware UTC, with onupdate) · is_row_invalid (Integer 0/1, default 0) · nullable invalidation_reason_uuid FK to invalidation_reasons.
Reference / lookup systems, invalidation_reasons, timezones A stable identifier PK (<entity>_uuid or code) · a human label column (name / description) · no audit columns required (rows are seeded and rarely change).
Operational / infrastructure idempotency_keys, future request_audit_log A stable identifier PK · TTL or expiry column when applicable · indexed columns matching the lookup pattern (e.g. by route + key hash).

Soft-delete contract. Core tables MUST use the is_row_invalid + invalidation_reason_uuid pair instead of DELETE FROM. The flag is 0 = valid, 1 = invalid. invalidation_reason_uuid is non-null exactly when is_row_invalid = 1; the reason FK points at invalidation_reasons for an explainable audit trail.

Timezone-aware timestamps. All created_at / updated_at columns use DateTime(timezone=True) with a Python default of datetime.now(UTC) (a tzinfo-aware value) — never naive datetimes, never local time.

Foreign keys are indexed. Any column that is a FK MUST carry index=True in its mapped_column(...) declaration so that joins and lookups remain cheap as the table grows. (SQLAlchemy does not auto-index FKs.)

3. Per-table documentation requirements

Every table has a documentation page under services/portal/internal/handbook/data/ (amended 2026-05-17; see §4 for rationale). The page is reachable from:

  • The data hub at services/portal/internal/handbook/data/index.html.
  • The api service catalog entity card at services/portal/internal/services/api/index.html (Links section) — cross-links to the data hub.
  • Every internal API operation page (services/portal/internal/services/api/reference/<resource>/operations/<method>-…html) that reads or writes the table — via a "Tables touched" cross-reference.
  • The internal sidebar (services/frontend/portal/assets_v2/ui-kit/mocks/nav-tree-internal.json) under Reference → Data.

The table-spec page MUST contain the following sections, in this order:

  1. Metadata block — name, category (core / reference / operational), source ORM module path under services/api/app/models/, owner, last reviewed.
  2. Summary — one short paragraph describing the table's business purpose. This is the answer to "why does this table exist?".
  3. Schema — a column-by-column table with: column name · type · nullable · default · FK target · index · description.
  4. Constraints — primary key, unique constraints, check constraints, indexes (other than FK indexes).
  5. Relationships — 1-N / N-1 / N-N relationships to other tables, with bullet points stating direction, FK column, and the business meaning of the relation.
  6. Lifecycle — who creates rows (which HTTP endpoint or seed script), who updates, who reads, retention / soft-delete policy.
  7. Cross-references — a list of: ORM source path, related HTTP endpoints (per-method internal spec pages), related ADRs (e.g. ADR 0006 for idempotency_keys), and seed migrations if any.
  8. History — the standard table per RFC 0003.

The copy-paste shell lives at services/portal/internal/handbook/sa/templates/data-table.html.

4. Placement

Original decision (2026-05-08): per RFC 0004, the api service catalog entity was at services/portal/internal/catalog/api/ and table specs lived one level deeper at services/portal/internal/catalog/api/data/, to avoid inventing a new top-level branch and to keep the per-service spine intact.

Amendment (2026-05-17): after the Diataxis reorganisation, the Reference layer became the canonical home for "what is" content. DB-table specs are exactly that — schema descriptions, not service operations — so they were relocated to services/portal/internal/handbook/data/ (sibling to reference/api/). The api service entity card keeps a cross-link to the new location; the per-service spine is unchanged.

Template moved from services/api/data/_template.html to services/portal/internal/reference/templates/data-table.html (joins ADR / RFC / API-spec / audit templates under reference/templates/).

Scope

  • In scope: naming and column conventions for every table in services/api/app/models/, the per-table documentation template, the placement of table specs under services/portal/internal/handbook/data/ (amended 2026-05-17), and the cross-reference contract between table specs and HTTP-operation spec pages.
  • Out of scope: choosing the storage engine (ADR 0015), migration tooling, query-performance tuning, replication topology, multi-tenant data partitioning, full-text-search indexes (these need their own ADRs).

Alternatives considered

Document tables only in ORM docstrings (pdoc)

Pros: zero duplication, source-of-truth is code.

Cons: pdoc shows class shape and column types but cannot describe business lifecycle, related HTTP endpoints, or cross-table invariants in a structured, indexable form. Reviewers and analysts would have to read Python source to answer "which endpoint writes this column?".

Embed table descriptions inline in each HTTP-operation spec page

Pros: single page tells the whole story per endpoint.

Cons: every endpoint that touches a shared table re-states the schema; updates drift; the "what is the canonical description of users?" question has no answer.

Top-level services/portal/internal/data/ branch

Pros: easy to find.

Cons: violates the per-service spine RFC 0004 just established; if a future service ships its own database, two parallel locations would compete. Keeping data under the owning service's catalog node scales better.

Auto-generate table docs from SQLAlchemy metadata

Pros: zero hand-authored drift on column types.

Cons: business meaning, lifecycle, and HTTP cross-refs are not derivable from metadata; the auto-generated output would still need a hand-authored prose layer. For a one-engineer scope, the cost of a generator outweighs the cost of typing a column table by hand.

Consequences

Positive

wins
  • Single authoritative description per table. Reviewers, analysts, and future engineers have one page to read for any table.
  • Forced cross-references. Operation pages that touch a table cite the table page; the table page cites the operations. The corpus stays internally consistent.
  • Defensible at Senior-SA bar. Naming and column conventions match the patterns reviewers expect (UUID PKs, audit timestamps, soft-delete pair, indexed FKs); the doc template matches the Backstage IDP discipline of always documenting an owned resource.
  • Forward-compatible with Postgres migration. The conventions hold regardless of storage engine.

Trade-offs

costs

Two canonical tradeoffs. Shape from Tradeoff disclosure.

  • Hand-authored column tables (vs auto-generated). We gain prose layer — business meaning, lifecycle stages, cross-HTTP references — that metadata cannot carry. We lose the «column-type changes propagate for free» property and accept drift risk if the ORM moves without the doc moving. We accept it because each spec lists the ORM source path and «tables-touched» is a required field on operation specs; PR review catches drift at the moment it would land, not after.
  • Per-table pages, not catalog dump. We gain one canonical URL per table that operations and runbooks can link to. We lose ~5 HTML files of upfront authoring and accept that on Postgres migration these double if read-replica views are documented separately. We accept it because the «one URL per table» property is the load-bearing «where is the authoritative description?» test; without it, every operation spec re-states the schema and the corpus drifts.

Revisit — would we do this again?

This ADR is Proposed; the «would we do this again?» question is forward-looking until the first table pages ship. The clearest day-one investment we would commit to alongside the first set:

  • Wire a CI check that compares the column list in each table doc against SQLAlchemy metadata. The doc-drift risk in tradeoff #1 above is exactly the kind of thing CI can catch deterministically. A 30-line script that grep'd the column-table cells, loaded the ORM model, and diffed the names + types would turn the «honour-system» discipline into a mechanical gate.

Compatibility and migration

  • Backward-compatibility impact: none. Documentation-only ADR; no runtime, schema, or wire-format change.
  • Migration plan:
    1. This ADR ratifies the conventions.
    2. Initial spec pages land for the four user-touching tables (users, systems, invalidation_reasons, timezones) plus idempotency_keys.
    3. Subsequent backlog items add specs for any future tables (conspectuses, review_slots, etc.) before merging the ORM model.
    4. Operation-spec pages under services/portal/internal/api/ gain a "Tables touched" cross-reference list.
  • Rollback: revert the offending PR; conventions and template can be removed without runtime impact.

Implementation plan

  1. Land this ADR.
  2. Create the data hub at services/portal/internal/catalog/api/data/index.html and the template at _template.html.
  3. Author the four user-related table specs: users.html, systems.html, invalidation_reasons.html, timezones.html.
  4. Cross-link from services/portal/internal/api/user/index.html and per-method pages under services/portal/internal/api/user/operations/ ("Tables touched" cross-reference).
  5. Add a "Database tables" group to the internal sidebar (services/frontend/portal/assets/internal-sidebar.js INTERNAL_SIDEBAR_NAV).
  6. Update the api catalog entity card (services/portal/internal/catalog/api/index.html) Links list to include the data hub.

Validation

  • Technical checks: make verify green; scripts/check_adr_rfc_links.py resolves every cross-reference; make docs-check drift gate green.
  • Operational checks: every table page renders with the standard portal chrome; the data hub is reachable in two clicks from services/portal/internal/index.html → Reference → Data.
  • Documentation updates: this ADR is linked from services/portal/internal/governance/adr/index.html; the data hub is linked from the api entity card; services/portal/CHANGELOG.md records the addition.
