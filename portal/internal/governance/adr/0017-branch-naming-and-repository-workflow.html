Branch Naming and Repository Workflow

Use trunk-based main with prefixed branches, semver tags, and a documented hotfix path so dashboards and release tooling stay scannable without GitFlow ceremony.

Context

Why this matters: Branch names are metadata. Prefixes like feature/ and fix/ make dashboards and release tooling scannable. A single integration branch (main) that stays green keeps trunk-based habits simple for small teams—without heavy GitFlow ceremony.

Documenting tags, semver, and rare hotfix forward-ports avoids "we only did that once in Slack" knowledge loss.

Decision

Integration branch

  • main is the default integration branch: it should remain releasable (quality gates green on the latest commit). Feature work lands via pull request into main unless a rare exception is agreed (hotfix branch policy below).

Branch naming

Use lowercase ASCII, hyphens as separators, and a short type prefix:

Prefix Use for Example
feature/ New behavior, API additions, user-visible changes. feature/user-export-csv
fix/ Bug fixes and regressions. fix/idempotency-409-body
services/portal/ Documentation-only changes under services/portal/, README, ADRs. services/portal/adr-0017-branches
chore/ Tooling, CI, formatting sweeps, dependencies without product behavior change. chore/bump-ruff
refactor/ Internal restructuring without intended behavior change (still covered by tests). refactor/user-service-split

Optional: suffix or segment with an issue tracker id if the team uses one, e.g. feature/42-export-csv or fix/gh-12-rate-limit. Avoid spaces, underscores except when matching an external id, and very long names—keep the slug readable in Git UI and terminal tabs.

Lifespan and updates

  • Prefer one logical change set per branch per PR when practical; split unrelated edits.
  • Keep the branch reasonably current with main using rebase or merge from main before merge; choose one style per PR and avoid mixing noisy merge commits unless the team standardizes on merge commits for traceability.
  • After the PR merges, delete the remote feature branch when Git hosting offers it, to reduce clutter (local branches may be deleted by the author).

Releases and tags

  • Release versions follow Semantic Versioning as already stated in CHANGELOG.md (see ADR 0013).
  • Git tags for releases use the form vMAJOR.MINOR.PATCH (e.g. v1.2.0), created on the commit that corresponds to the released changelog section.

Hotfixes

If a production fix must ship without taking everything on main, branch from the tagged release (e.g. fix/prod-1-1-2-null-deref), fix, tag a patch, and forward-port the same commit to main (or cherry-pick) so drift does not accumulate. This is exceptional; default flow remains PR → main → release.

Scope

  • In scope: naming, integration model, tags, contributor expectations.
  • Out of scope: Git hosting UI settings (branch protection rules), required reviewers—configure per org policy.

Alternatives considered

Three alternatives evaluated before adopting trunk-based + prefixed branches. Each rejected with a rooted reason.

Free-form branch names

rejected

Why considered: zero rules, zero enforcement, every contributor names their branch however.

Why rejected: the changelog path-filter gate (ADR 0013) and future automation (per-prefix CI fan-out, auto-labelling of PRs) all depend on being able to read branch metadata. Free-form names forfeit that lever permanently. The cost of prefixing is paid once per branch creation; the cost of free-form is paid every time someone tries to filter PR lists.

GitFlow with long-lived develop

rejected

Why considered: Vincent Driessen's classic — develop integrates, main tracks releases, dedicated release/* branches stabilise. Standard for products with parallel release trains.

Why rejected: GitFlow's mental overhead (5 branch types, explicit merge directions) is calibrated for teams shipping multiple release trains in parallel. We ship one trunk; the «which branch to merge into» question is trivial. GitFlow would add ceremony without the matching benefit.

Trunk-based without any branches (push to main directly)

rejected

Why considered: some high-velocity teams (e.g. Google's monorepo) commit straight to trunk behind feature flags. Maximum simplicity.

Why rejected: we have no feature-flag infrastructure (backlog BL-013/BL-038) to hide work-in-progress behind. Without flags, direct-to-trunk means every commit must be releasable independently — a constraint our docs-and-spec-heavy PRs cannot meet today. The PR-into-main pattern is the smallest mechanism that gives us reviewable units without forcing every commit to be production-ready.

Trade we accept by choosing trunk-based + prefixed branches. Contributors learn five prefixes (feature/, fix/, services/portal/, chore/, refactor/). We accept it because each rejected alternative either forfeits branch-name automation (free-form), adds ceremony without matching velocity gain (GitFlow), or assumes infrastructure we have not built (direct-to-trunk).

Consequences

Positive

wins
  • Consistent names in PR lists and local git branch output.
  • Clear place to document hotfix and tagging expectations next to changelog policy.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • Prefix-naming is policy, not enforcement. We gain a low-friction convention contributors learn in a day. We lose the «branch with no prefix lands on main» refusal that a server-side hook would give. We accept it because the failure mode of a wrong prefix is «reviewer asks for a rename», which is reversible; the failure mode of a hook is «contributor stuck at push» which corrodes the contributor's confidence in the tooling.
  • Single integration branch (no develop). We gain the trunk-based simplicity «one branch tells you what is in production». We lose the GitFlow staging buffer where a half-baked feature can sit on develop without leaking to main. We accept it because the buffer's benefit only materialises when there are parallel release trains; with one trunk it is staging-for-its-own-sake.

Revisit — would we do this again?

Yes. One refinement we would add on day one:

  • Add a GitHub Actions check that fails the PR if the source branch does not match a prefix from the table. Today the prefix policy is honour-system; a small actions/github-script check would catch the «I forgot to prefix» case at PR open time and rename automation would be one click away.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation plan

  1. Adopt the conventions in the Decision section across the repository.
  2. Update related templates, runbooks, and CI gates to reflect the rule.
  3. Cross-link from CHANGELOG and onboarding docs so future contributors discover the decision.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.
  • Documentation updates: README, CHANGELOG, and related ADRs cross-reference this decision.
