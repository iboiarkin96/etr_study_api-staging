Make Command Taxonomy and Workflow Entrypoints

Two-level Make targets: atomic commands as source-of-truth building blocks, plus memorable scenario entrypoints (fix → verify → release).

Context

Why this matters: A long Makefile lists many targets; without naming rules, every developer invents a personal script. That fragments onboarding (“how do I run what CI runs?”) and makes documentation brittle. Clear atomic commands (one job each) plus a few scenario entrypoints (fix → verify → release) match how people actually work.

We standardize on memorable verbs for whole workflows so new contributors do not hunt through the Makefile during every fix or release.

Decision

We use two levels of Make targets:

  1. Atomic commands remain source-of-truth building blocks (format/lint/test/docs/etc.).
  2. Scenario entrypoints provide memorable, high-level workflows.

Scenario entrypoints introduced:

  • make fix - canonical fix workflow.
  • make verify - canonical local quality gate.
  • make release-check - canonical pre-release gate.
  • make release DEPLOY_CMD='...' - canonical release execution flow.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were evaluated before adopting the two-level taxonomy. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints. This ADR is cited as the canonical exemplar in Tradeoff disclosure § 07.

Keep ad-hoc scripts/*.sh wrappers (what we had)

rejected

Why considered: the original setup. Each developer maintained their own shortcuts in scripts/; new contributors copied whatever they found.

Why rejected: drift was the #1 onboarding complaint. PRs kept adding redundant scripts that did almost-the-same thing under different names (scripts/test.sh vs scripts/run-tests.sh vs scripts/check.sh), and CI ran a fourth variation. The day a new joiner could not find the canonical «how do I run what CI runs?» answer was the day to fix this.

Switch to just / task / nox

rejected

Why considered: modern Make-alternatives with cleaner syntax, named arguments, and proper variables. just (Casey Rodarmor) is the most popular successor; task is the Go-ecosystem version. Both are explicitly designed to be «Make but nicer».

Why rejected: Make is already installed on every dev machine and CI runner without configuration. Adding a tool-install gate for «slightly nicer syntax» would burn onboarding budget we'd rather spend on docs-check, type-check, and contract-test. The footgun (tab-vs-space) is paid once per contributor when they read a target; the install-tax of just would be paid by every new contributor and every CI runner forever.

Single monolithic Makefile with no taxonomy

rejected

Why considered: the default «just dump every target in» pattern that most Makefiles end up with. Cheapest authoring cost.

Why rejected: make help returned 80+ lines with no grouping; new joiners could not predict the target name for a workflow they'd never run. The lookup cost on make help is the load-bearing newcomer interaction in this project — pessimising it would have undone the value of having a help target at all.

Trade we accept by choosing the two-level taxonomy. Every new atomic target costs ~30 s of help-text curation, and a new scenario target costs the composite recipe + a help entry. We accept it because the lookup cost on make help is the load-bearing newcomer interaction and it pays back within the first day; the rejected alternatives either invite the drift this ADR exists to prevent (ad-hoc scripts), add an install-tax we cannot justify (modern alternatives), or pessimise the very interaction we are optimising (flat list).

Consequences

Positive

wins
  • Lower onboarding and day-to-day cognitive load.
  • Clear separation between reusable primitives and operational workflows.
  • More consistent local and CI usage patterns.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure.

  • Two-level structure (atomic + scenario). We gain readable make help output where the workflows a newcomer needs (make verify, make release) sit at the top, and the building blocks they may want to compose sit underneath. We lose the freedom of a flat list — every new atomic target needs to be filed under the right category in the help text, and a new scenario needs an explicit composite recipe. We accept it because flat 80-target output is the failure mode this ADR was built to prevent; help-text upkeep is a 30-second cost per new target.
  • Make's tab-sensitive syntax stays as the load-bearing build system. We gain a build system installed on every dev machine and CI runner since 1977 — no version pinning, no toolchain install. We lose the slightly cleaner syntax of just / task and accept that contributors will occasionally trip on tab-vs-space (the canonical Make footgun). We accept it because the syntax-tax is paid once per contributor when they read a target; the install-tax of a modern alternative would be paid by every new contributor and every CI runner forever.

Revisit — would we do this again?

Yes, with one curation aid we did not build. The make verify / make fix / make release trio has become the canonical onboarding answer to «how do I run X?» and the two-level structure has held without drift across ~30 atomic targets. One thing we would add on day one if starting today:

  • Make help text generated from per-target docstring comments (## …), not hand-curated. Today the help text is a separate help target that grep-formats the file; a contributor who adds a new target without updating the help block can land it silently. The community-standard ## description pattern (parsed by a 5-line awk recipe) keeps the help in the same place as the target, so they cannot drift. The curation rule in § Implementation rules would then be «add the ## line» instead of «update make help manually».

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation rules

  1. Keep one canonical command per workflow; avoid duplicate aliases with identical behavior.
  2. When adding a new atomic target, update make help under Atomic commands.
  3. When adding a new multi-step process, expose it as scenario entrypoint in make help.
  4. Keep release flow explicit and parameterized via DEPLOY_CMD.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.
  • Documentation updates: README, CHANGELOG, and related ADRs cross-reference this decision.

Command taxonomy

Atomic commands

  • Environment: venv, install, requirements, env-check.
  • Code quality: format-fix, format-check, lint-fix, lint-check, dead-code-check, type-check.
  • Contracts: openapi-check, contract-test, openapi-accept-changes.
  • Tests: test, test-one, test-warnings.
  • Docs: docs-fix, docs-check.
  • Local Docker stacks (optional): metrics — observability-up, observability-down, observability-smoke; structured logs — logging-up, logging-down, logging-reset, logging-smoke, logging-es-query (ADR 0023).
  • Deploy primitive: external deployment command passed via DEPLOY_CMD.

Scenario entrypoints

  • Fix loop: make fix.
  • Local verification: make verify.
  • Pre-push alignment with CI (dependency audit + strict docs sync): make verify.
  • Release gate: make release-check.
  • Release execution: make release DEPLOY_CMD='...'.
