API Versioning Policy

Path-based majors (/api/v1); only additive changes within a major; breaking changes require a new major and a 90-day deprecation window.

Context

Why this matters: A public API is a promise. If you rename a field or change an error meaning without warning, clients break — often only in production. The major version in the URL (/api/v1) is a clear signal: everything under that prefix follows one set of compatibility rules.

Without written rules, teams ship “small” changes that are actually breaking. Clear rules for additive vs breaking changes and a deprecation period keep trust with integrators.

Decision

We use path-based major versions. Within one major, we prefer additive, backward-compatible changes:

  1. Major API version is encoded in URL prefix (current: /api/v1).
  2. v1 accepts only backward-compatible changes.
  3. Breaking changes require a new major version (for example /api/v2).
  4. Error contract semantics for existing code/key pairs are immutable.
  5. Deprecated contract elements must keep a migration window of at least 90 days (or two release cycles, whichever is longer).

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were evaluated before committing to path-based majors + 90-day deprecation. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

Header-based versioning (Accept: application/vnd.studyapp.v1+json)

rejected

Why considered: RESTful purist choice — used by GitHub's REST API, recommended by many API design guides. Keeps URLs clean («the URL is the resource, not the representation») and allows content negotiation across versions without path changes.

Why rejected: hides the version from every developer-experience surface that matters in a solo project — log lines, browser dev-tools, copy-pasted curl commands, dashboards. The integrator's first debug step is «what was the URL?»; with path versioning the answer is in the URL, with header versioning it's in a header that may or may not have been logged. The clean-URL benefit is invisible to humans; the debug-cost is paid daily.

Date-pinned versioning (Stripe-style Stripe-Version: 2023-10-16)

rejected

Why considered: Stripe's industry-leading pattern — every change ships under a new dated version, the integrator pins explicitly, and the backend maintains N parallel response shapes via compatibility shims. Maximum stability for the integrator.

Why rejected: date-pinning works because Stripe has the infrastructure to run N response transformers in parallel and the operational maturity to deprecate dates safely over years. We have neither — no version table, no shim layer, no operational dashboards to watch a deprecation. The cost overshoots the benefit at our scale by ~10×; the day we have multiple integrators with conflicting upgrade timelines is the day to revisit, not before.

No versioning — evolution-only (Facebook Graph API early days)

rejected

Why considered: the simplest possible policy — never bump major, ship breaking changes only via deprecation cycles. Forces every breaking idea to be re-expressed as a new field / new endpoint, which often produces cleaner designs.

Why rejected: the policy provides no signal at all when a genuine breaking change is needed (a wrong abstraction, a security flaw in the public shape). Without a major-bump escape hatch, breaking-change requests pile up as «small» additions that bend the model. The path-major + 90-day window adds a one-time cost when a v2 lands but prevents the slow-rot failure mode that Graph API itself documented as the reason for the eventual v2.0/v2.1/v2.2 deprecation chain.

Trade we accept by choosing path-based majors + 90-day window. We pay ~30 min of governance overhead per breaking-change proposal (write up the migration, set the sunset date, update headers) and accept the «URL change on v2 cut-over» one-time pain. We accept it because each rejected alternative either hides the version from daily debugging (header-based), demands infrastructure we have not built (date-pinned), or offers no escape hatch when the public shape is wrong (no-versioning). The path version is the smallest mechanism that is both visible and final.

Consequences

Positive

wins
  • Predictable API evolution for clients.
  • Lower risk of accidental contract breaks.
  • Clear migration expectations.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure: «we gain X, we lose Y, we accept it because Z».

  • Path-in-URL major version. We gain a version that every HTTP intermediary (cache, log, proxy, curl) sees at zero parse cost — log lines carry the version, dashboards filter on it, and a partner reproducing a bug in the browser dev-tools sees the version in the address bar. We lose the «evergreen URL» promise (a v2 cut-over forces every partner to update their base URL string). We accept it because curl-debuggability + log-grep are the load-bearing developer-experience surfaces; the day a v2 lands, the path change is itself the cleanest signal that something materially changed.
  • 90-day minimum migration window. We gain a predictable runway every integrator can budget against (one full quarter, enough to land the change inside their own release process). We lose the option to drop a known-bad behaviour quickly — a typo'd field name has to stay live for 90 days even when nobody is using it. We accept it because trust-with-integrators is unrecoverable once breached; the 90-day floor is the smallest window that fits a single quarterly release cycle, and the cost of an over-long deprecation is paid only by us, not by them.

Revisit — would we do this again?

Yes — but the policy has not yet been stress-tested. v1 is still the only major; the path-versioning choice has paid back daily on curl-debug and log-grep, the 90-day window has not yet been exercised because no breaking change has landed. Two refinements we would add on day one if starting today:

  • Automate a «breaking-change detector» as a PR gate. ADR 0007 already diffs the OpenAPI baseline; extend it to classify diffs as additive (allowed) vs breaking (requires major-bump) per the rules in § Compatibility Rules above. Today the classifier lives in the reviewer's head, which is the wrong place for a policy this load-bearing.
  • Ship a «what counts as a breaking change?» decision tree as a sibling reference page. The rules in § Compatibility Rules are dense prose; new contributors re-derive the answer from first principles every time. A 1-page yes/no tree (with «add optional field» → yes, «narrow an enum» → no, etc.) would cut review time by half.

Compatibility Rules

Breaking changes (forbidden inside same major)

  • Endpoint removal or rename.
  • Field removal, rename, type change, or semantic change.
  • Existing optional field becoming required.
  • Enum value removal or semantic repurposing.
  • Repurposing existing business/validation error code and key.

Non-breaking changes (allowed inside same major)

  • Adding new endpoints.
  • Adding optional fields.
  • Adding new error codes with new keys.
  • Documentation/example improvements without runtime behavior change.

Implementation Guidance

For contract-related changes:

  1. Update request/response schemas.
  2. Update router responses and OpenAPI examples.
  3. Add compatibility and regression tests.
  4. Run make release-check and make docs-fix.
  5. Record policy-level decisions in ADR.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.
  • Documentation updates: README, CHANGELOG, and related ADRs cross-reference this decision.

Deprecation Policy

When deprecating behavior, document migration path and timeline in:

  • README.md
  • services/portal/internal/analysis/system-design.html
  • ADR notes (if policy-level)

If deprecation headers are enabled, use:

  • Deprecation: true
  • Sunset: <date>
  • Link: <migration-guide>; rel="deprecation"
