Service-catalog-first IA — services as backbone, Diátaxis as metadata lens

The internal portal adopts a service-catalog-first information architecture modelled on Spotify Backstage. Every deployable artefact (services/<svc>/) owns its runbooks, postmortems, on-call rotation, API spec, and architecture page. Cross-cutting craft (style, principles, templates, authoring how-tos, SRE / QA / SWE practice essays) lives under a single root, handbook/. Diátaxis becomes a metadata attribute (data-page-type), surfaced through generated lens-pages under catalog/by-quadrant/ — not a folder structure. Roles are aggregator hubs (team/roles/<role>/) that link in, never duplicate. This ADR ratifies the convention; the migration ships in eight waves over ~6–8 working days.

Context

The internal portal carries 468 HTML pages today. A 2026-06-06 IA audit (companion artefact ~/Desktop/portal-ia-target-2026-06-06.html) identified six duplication hotspots where a new page has two-or-three plausible homes:

  1. Diátaxis quadrants exist three times — top-level tutorials/ how-to/ reference/ explanation/, inside foundations/{explanation,reference}/, and inside each services/<svc>/{how-to,reference,explanation,code-reference}/. A new service-tied how-to can plausibly live in three folders.
  2. Authoring how-to inside reference/sa/ — eight pages with data-page-type="how-to" sit under foundations/reference/sa/authoring/, contradicting the folder name.
  3. Runbooks split — ten project-wide runbooks live at how-to/incidents/runbooks/; per-service runbook index pages live at services/<svc>/runbooks.html and re-link into the central pool. Two-hop navigation for the most stress-sensitive content on the site.
  4. Roles appear in four places with asymmetric coverage — tutorials/<role>/, foundations/reference/<role>/, explanation/<role>/, team/roles/<role>/. Adding a new role (SWE-mobile, data-eng) requires touching four trees.
  5. Data — two sources of truth — handbook/data/{invalidation_reasons, invariants, systems, timezones, users}.html holds table-level facts; services/datastore/index.html is the catalog entry for the persistence artefact and someone reading the latter expects table docs there.
  6. «Foundations» is two ideas — foundations/reference/sa/ is the SA-handbook canon (ratified by ADR 0031); foundations/reference/{dev,qa,sre}/ are thin role-reference indices; foundations/explanation/reader-personas.html is cross-cutting personas reference. The single folder name means «rulebook for authors» and «cross-cutting role reference» simultaneously.

Two target IAs were drafted (Variant A — role-at-the-spine; Variant B — Diátaxis-at-the-spine with services as flat catalog). A third — Variant C — service-catalog-first — was added after surveying what tier-1 IT companies actually run today: Spotify Backstage (service catalog + TechDocs + Software Templates, adopted at Netflix, Lyft, American Airlines) and the convergent pattern in GitLab handbook, Microsoft Learn (Diátaxis as a metadata lens, not folders), and Stripe / Cloudflare docs. The portal already carries Backstage-spec catalog-info.yaml on every service (apiVersion: backstage.io/v1alpha1, kind: Component); Variant C is the formalisation of a model the portal has been growing into intuitively.

Decision

The conventions below are normative for the internal portal from ratification onwards:

D1 · Service is the first-class entity

Anything tied to one deployable artefact lives under services/<svc>/: runbooks, postmortems, API specs, architecture diagrams, on-call rotations, service-tied how-tos, service ADR projections. Rule of thumb: if the artefact would lose its home when the service is retired, it belongs here. Each service carries a Backstage-spec catalog-info.yaml as the canonical descriptor (apiVersion: backstage.io/v1alpha1); the renderer reads it to produce the service tile on services/index.html and the entity card on services/<svc>/index.html.

D2 · Cross-cutting craft goes to handbook/

Anything that applies across services — style guide, principles, templates, authoring how-tos, SRE / QA / SWE practice essays, threat model, test pyramid, data-table reference — lives under handbook/. The rule of thumb: if every service would copy this page, it belongs in handbook/ once. The SA handbook canon (~24 pages) ratified by ADR 0031 moves to handbook/ root unchanged in content; the lens-chip vocabulary from ADR 0031 D2 continues to apply.

D3 · Diátaxis is metadata, not a folder

Every page declares data-page-type on <body> (already true across the portal). Quadrant navigation is rendered as a generated lens-page under catalog/by-quadrant/{tutorial,how-to,reference,explanation}.html from that metadata. No folder asserts a quadrant by name — folders are organised by subject (service, handbook topic, role hub), quadrant is a view. This eliminates the «three competing quadrant trees» from Context point 1.

D4 · Role hubs aggregate, never duplicate

team/roles/<role>/ is the single canonical home for role-keyed content (radar, 25 practices, role onboarding). Cross-cutting role-relevant pages (e.g. handbook/sre/observability.html) live under handbook/ and surface in the SRE role hub by tag-driven aggregation, not by copy. Adding a new role (SWE-mobile, data-eng) requires creating exactly one folder under team/roles/, not four.

D5 · One canonical URL — everything else is a view

Every page exists in exactly one place (its canonical URL). Aggregator pages (catalog/by-*/, role hubs, service overview cards) link in; they never copy prose. A new lens (e.g. by-lifecycle, by-team) is added by extending the generator, never by duplicating content.

D6 · Role token normalised to swe

The role token dev is renamed to swe everywhere (folder paths, body data-role values, nav-tree slugs, sidebar labels — the last already says «SWE»). The rename ships in W3 with redirects from old URLs.

D7 · New page-level metadata schema

Every published page declares the following attributes on <body>; the lint adds them to the required set in W6:

Scope

  • In scope:
    • Top-level IA rename — foundations/ → handbook/.
    • Per-service folders services/<svc>/ absorb runbooks, postmortems, service-tied how-tos.
    • Project-wide how-to/incidents/runbooks/ and postmortems/ trees disperse.
    • Top-level reference/{data,qa,security,manager,service,uml}/ migrates to handbook/.
    • Top-level explanation/{dev,qa,sre}/ migrates to handbook/{swe,qa,sre}/.
    • New catalog/ tree carrying autogenerated lens-pages.
    • Role-token rename dev → swe.
    • New body-level metadata attributes (D7).
  • Out of scope:
    • governance/ — ADRs, RFCs, audits, backlog, traceability. Untouched.
    • onboarding/ — cold-start, glossary, 1-day plan. Untouched; only outbound cross-links refreshed.
    • services/<svc>/code-reference/ — pdoc autogen; ADR 0016.
    • services/api/reference/ — OpenAPI conspectus autogen; ADR 0025.
    • assets_v2/ui-kit/ — UI Kit v2 components and tokens; no new components proposed.
    • Public portal under services/portal/public/ — separate IA per RFC 0004.

Alternatives considered MANDATORY · MIN 2

  1. Variant A — role-portals at the spine, Diátaxis inside each role.
    • Rejected because: role-keying repeats the four Diátaxis quadrants six times under roles/<role>/, producing 24+ L2 folders. The sidebar grows linearly with role count, hitting the IA-scalability ceiling that RFC 0006 explicitly warns against. Adding the SWE-mobile / data-eng / ML roles in the next quarters would make this worse, not better.
    • Pros considered: most audience-pure; a new contributor sees «everything about my role» in one tree.
    • Cons that decided it: cross-role pages (services, data, governance) still live outside role trees, so the audience-purity is partial. The sidebar cost is real and recurring.
  2. Variant B — Diátaxis at the spine, services as a flat catalog.
    • Rejected because: framework-textbook (Django, Cloudflare, Microsoft Learn cite Diátaxis), but not what Spotify / Netflix / Lyft run on internal portals. The «services as flat catalog» part is half of Backstage; the «cross-cutting how-to/ as parallel corner» is what large orgs explicitly move away from as their service count grows. The portal would need to be re-IA'd again at ~1 000 pages to add service-owned runbooks / postmortems / specs — exactly what Variant C ships now.
    • Pros considered: smaller blast radius (~15 % pages move vs ~28 % for Variant C), one-time cost ~5 working days (vs 6–8 for C). Strong for a portal that stays at ~500 pages indefinitely.
    • Cons that decided it: the wins are one-time; the loss («service has nothing under it» feel) compounds with every new service. The 2026-05-30 IA audit and this portal's growth direction both point past B.
  3. Keep current structure, add a tag-based search layer only.
    • Rejected because: tags do not fix the «where do I put this new page?» author problem — they fix only the «where do I find an existing page?» reader problem. The author still faces three competing quadrants and four role folders on every commit. Tag retrofits over an inconsistent tree are a known anti-pattern in large portals.
    • Pros considered: zero migration cost.
    • Cons that decided it: defers the structural problem; the duplication hotspots from Context grow with the portal.

Consequences

Positive

  • Author decision time per new page drops to a single top-down 6-question check — see the choice rule in handbook/sa/authoring/internal-service-docs-layout.html (lands in W5).
  • Adding a new service requires one folder + one catalog-info.yaml; the existing renderer picks it up. Adding a new role requires one folder under team/roles/.
  • Service-tied content (runbooks, postmortems, on-call) is one click from the service catalog tile — the highest-stress access path on the portal.
  • The lens-page generator (catalog/by-*/) makes Diátaxis, role, and topic views first-class without folder duplication.
  • Backstage-shape is universally recognised by tier-1 reviewers and external integrators.

Trade-offs

  • ~28 % of pages move across W1–W5. Mitigated by HTML-meta-refresh redirect stubs for every URL; the stubs stay live indefinitely for external inbound links.
  • W1 (foundations/ → handbook/) has ~109 inbound files. Mitigated by an automated link-rewrite pass + manual smoke of five high-traffic pages.
  • W6 metadata sweep touches every page. Tool-pass adds data-service / data-role / data-topic / data-lifecycle / data-updated with defaults inferred from path; author opt-in only for data-role on handbook pages.
  • Writing the lens generator (W7) is net-new code — ~200 lines of Python extending scripts/build_search_index.py. One-time cost.
  • The «services as backbone» framing means cross-service essays no longer fit physically under a service. They move to handbook/sre/ /qa/ /swe/. Acceptable: the existing role-hub pages already work this way.

Compatibility and migration

  • Backward-compatibility impact: moderate. Every moved URL ships a redirect stub (<meta http-equiv="refresh">) at the old path. The stubs are gradient-removed in W8 only for paths with zero inbound links after the search-index regeneration.
  • Migration plan: eight waves, ~6–8 working days. Full per-page table in the companion ~/Desktop/portal-ia-variant-c-2026-06-06.html. Wave outline:
    1. W0 — this ADR + ADR 0031 amendment + role-token decision + metadata schema land.
    2. W1 — foundations/ → handbook/ split (largest blast radius).
    3. W2 — runbooks & postmortems disperse to services/<svc>/ and handbook/sre/runbooks/.
    4. W3 — explanation/{dev→swe,qa,sre}/ → handbook/{swe,qa,sre}/; tutorials fold into team/roles/<role>/onboarding.html.
    5. W4 — top-level reference/ migrates to handbook/.
    6. W5 — top-level how-to/ remaining trees split between services/portal/how-to/ and handbook/sa/authoring/.
    7. W6 — metadata sweep across all pages + lint rule for required attributes.
    8. W7 — catalog/ lens-page generator + top-level index.html Backstage-style landing.
    9. W8 — nav-tree normalise, dead-redirect cleanup, ADR 0031 supersession chain finalised.
  • Rollback strategy: mechanical at the wave level — every wave is one commit (or one PR), every move is a git mv + redirect stub. git revert <wave-sha> undoes the wave; the redirect stubs from later waves keep older inbound links working until rolled back themselves. Full rollback to pre-W0 is a single revert chain.

Implementation plan

  1. Ratify this ADR (Proposed → Accepted) — same-day if no objections by reviewers.
  2. Amend ADR 0031 with a path-rename note: every occurrence of foundations/reference/<discipline>/ reads as handbook/<discipline>/ after W1; the canon (D1–D5 of ADR 0031) is unchanged.
  3. Add the D7 metadata schema to handbook/sa/style-guide.html (one section); the lint rule lands in W6.
  4. Open the wave tracker — backlog tasks W1–W8 in governance/backlog/ with one PR per wave, each PR closed by make docs-check.
  5. Ship waves in order; do not interleave. Each wave references «ADR 0032 · W<n>» in commit message.

Validation

  • Manual review (per wave): each wave PR runs make docs-check and a manual smoke of the five highest-traffic moved pages before merge. Reviewers cite «ADR 0032 · D<n>» on author choices.
  • CI checks (W6 onwards): tools/docs/validate_docs_html.py asserts the D7 metadata schema on every page under services/portal/internal/ except governance/, onboarding/, autogen reference, and code-reference/:
    1. Body declares data-page-type, data-service, data-lifecycle.
    2. data-page-type ∈ {tutorial, how-to, reference, explanation, landing}.
    3. data-service ∈ {api, portal, datastore, monitoring, ui-kit, none}.
    4. data-lifecycle ∈ {draft, review, published, deprecated}.
  • Lens-page health (W7 onwards): the generator fails the build if any service has no catalog-info.yaml, if any page has unresolved tags, or if catalog/by-quadrant/<q>.html would render zero entries.
  • Documentation updates: handbook/sa/style-guide.html gains the D7 schema; handbook/sa/authoring/internal-service-docs-layout.html (moved from how-to/docs/ in W5) carries the choice rule from this ADR's Decision section; the SA-handbook hub at handbook/index.html links to this ADR.
