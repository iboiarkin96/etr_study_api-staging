Mandatory API Testing Policy

Every API change ships with at least one success test and one failure test; without them the change is incomplete.

Context

Why this matters: Automated tests are the cheapest place to find bugs. A bug in CI costs minutes; in production it costs trust, on-call time, and data fixes. Manual “click through” does not scale as the API grows or more people touch the code.

API tests are part of the contract: if there is not at least one success test and one failure test, the change is not complete.

Decision

Endpoint-level automated tests are mandatory before release:

  1. Any API change must include tests.
  2. Endpoint coverage must include at least:
    • one success scenario (happy path),
    • one failure scenario (validation or business error).
  3. Changes without tests are considered incomplete.
  4. Quality gates must enforce test execution before deployment.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were considered before adopting per-endpoint tests as a release gate. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

Manual QA pass before release

rejected

Why considered: the cheapest possible starting point for a solo project — no fixtures to maintain, no CI overhead, the author runs through a click-through checklist before each release.

Why rejected: manual QA does not generate the executable behaviour examples that the portal relies on. Each spec page references tests/api/v1/test_*.py as the canonical answer to «what does this endpoint promise?»; without those tests the spec has nothing to point at. Single tester also does not scale past one or two endpoints per session, and the system already has ~9 operation specs.

Contract tests only (schemathesis / OpenAPI fuzzing)

rejected

Why considered: the OpenAPI baseline (ADR 0007) is already authoritative. A tool like schemathesis derives tests from the schema automatically, eliminating the per-endpoint authoring tax entirely.

Why rejected: OpenAPI alone cannot verify business invariants — that conspectus.created_at ≤ conspectus.updated_at, that ownership comparison returns 404 not 403, that idempotency keys are honoured on replay. Schema-driven tests would have missed every one of these properties at the surface where they matter. We use schema validation as a defence layer, not as the primary signal.

Trunk-based without pre-merge test gates (fix-forward)

rejected

Why considered: large teams (Facebook, Google) ship without strict pre-merge testing for the trunk and rely on production telemetry + automated rollback to catch regressions. Faster iteration, less ceremony per PR.

Why rejected: fix-forward needs signal — dashboards, traces, alert routing — to detect regressions in production. We explicitly do not yet have observability beyond logs (narrative § 09 — observability is gap #2), and the 99.5 % SLO does not absorb regressions found post-merge. Pre-merge gate is the cheaper insurance until the production signal matures.

Trade we accept by choosing per-endpoint success+failure tests. Every API change pays a fixed ~10–30 min testing tax and ~1–2 min of CI per run. We accept that cost because each of the rejected alternatives requires infrastructure we do not yet have (a QA team for manual, mature observability for fix-forward) or accepts a class of bugs we cannot afford to ship (business-invariant violations for contract-tests-only). The tax is paid once per endpoint and stays paid; the rejected alternatives' costs are paid every release.

Consequences

Positive

wins
  • Early regression detection.
  • Higher confidence in API contract stability.
  • Faster onboarding with executable behavior examples.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure: «we gain X, we lose Y, we accept it because Z».

  • Slower iteration on every API change. We gain executable examples that double as documentation (a failing test points at the broken behaviour better than any prose). We lose ~10–30 min per endpoint adding the success+failure pair, plus the maintenance tax when the test fixtures rot. We accept it because the SLO of 99.5 % (ADR 0011) leaves no budget for trivial regressions; a bug caught in CI costs minutes, a bug caught after merge burns error budget.
  • Slower pre-release pipeline. We gain a CI gate that refuses changes without proof of behaviour. We lose ~1–2 min per pipeline run and rule out the «push first, fix forward» workflow used by larger teams with mature observability. We accept it because we explicitly do not yet have observability beyond logs (narrative § 09), and without dashboards a fix-forward strategy has no signal to act on.

Revisit — would we do this again?

Yes, with one calibration. The «one success + one failure» minimum has aged well — it sets the floor low enough that contributors do not skip tests but high enough to catch the most common regression class (validation drift). One thing we would adjust on day one if starting today:

  • Pair this with a property-based test layer (Hypothesis) earlier. The success+failure example pair catches known regressions but cannot find the unknown-unknowns at the schema boundary; we are now retrofitting Hypothesis for the conspectus model (backlog item BL-006) and the asymmetry between «examples for known cases, property tests for invariants» reads as the right shape we should have committed to from the beginning. The example-based floor stays; the property layer would have caught two of the three on-file postmortems' surface bugs at PR time.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.
