Changelog and Release Notes

Keep a Keep-a-Changelog file in the repository root and enforce updates when user-facing paths change, so consumers and contributors share one obvious place for "what changed".

Context

Why this matters. When something breaks in production, the first question is "what changed?" A changelog in the repo answers that without combing through dozens of commits. It is also the surface integrators read: they should not need access to internal tickets to learn about API or security changes.

Ad-hoc notes in chat do not scale. We keep Keep a Changelog at the repository root, aligned with Docs as Code, and enforce updates when user-facing paths change.

Decision

Where and how

  • The repository maintains a root CHANGELOG.md following Keep a Changelog, with an [Unreleased] section for ongoing work and dated sections when versions are released.
  • User-facing changes (behavior, API contract, security, notable fixes) belong under ### Added, ### Changed, ### Fixed, ### Deprecated, ### Removed, ### Security as appropriate. Purely internal refactors, test-only edits, or mechanical CI updates that do not affect consumers may be omitted when they do not match the trigger paths in Validation (below).

Merge and PR conventions

  • Prefer squash merge into main (or master) so one commit per change carries a clear summary; that message is a good place for a changelog-related pointer when useful.
  • When a pull request touches user-facing areas (see Validation), it must also update [Unreleased] in CHANGELOG.md in the same PR, unless an escape hatch applies.
  • Escape hatches (documented for automation and humans):
    • Include [skip changelog] or skip-changelog in the PR title (for pull-request checks) or in commit messages in the pushed range (for post-merge push checks) when the change is mechanical and needs no release note.
    • Optional GitHub label skip changelog may be used where enabled; teams should mirror the same intent as the text tokens above.

Automation (LLM)

An optional script may call an HTTP API (OpenAI-compatible or similar) using keys from the developer or CI environment to draft bullet points from git log. The output is assistive only: humans review and edit before merge. Nothing automatically overwrites CHANGELOG.md on main without a normal PR.

Local workflow: scripts/changelog_draft.py

The script never writes CHANGELOG.md. It prints a draft to stdout; you copy or adapt the result into [Unreleased] in a normal commit/PR.

How the git range works

The script reads commits in the range --since..--head: commits reachable from --head that are not in --since. Defaults are --since main --head HEAD, i.e. the new commits on your current branch. Point --head at a branch name to draft for that branch from elsewhere.

Staged changes are not in the git history. If you have only run git add, the range may be empty and the script will print _No changes in this range._ Either commit first, or pass --include-working-tree (see below).

Common runs

  1. Before any commits exist on the branch. python scripts/changelog_draft.py --since main --head HEAD --include-working-tree. This adds git diff --cached --stat (staged) and git diff --stat (unstaged) to the prompt, so the model can summarise uncommitted work.
  2. After one or more commits. python scripts/changelog_draft.py --since main --head feature/my-feature. Add --include-working-tree if local edits should also be considered.
  3. Inspect the prompt without calling the API. python scripts/changelog_draft.py --print-log (add the same flags you would use for a real run).
  4. Save the draft to a file. python scripts/changelog_draft.py ... -o /tmp/changelog-llm-draft.md writes UTF-8 alongside stdout. Still merge into CHANGELOG.md by hand under the right ### headings — never raw-append stdout to the changelog file.
  5. One-step convenience. python scripts/changelog_draft.py with no flags uses main..HEAD, includes working tree stats, and writes changelog-llm-draft.md. Override with CHANGELOG_HEAD=feature/x, CHANGELOG_SINCE=main, or CHANGELOG_DRAFT_FLAGS= for commits-only.

Secrets and smoke test

Set OPENROUTER_API_KEY or OPENAI_API_KEY in the repository root .env file (see env/example). Optional variables: OPENAI_BASE_URL, OPENAI_MODEL, and the OpenRouter attribution headers described there. Smoke-test the key with python scripts/llm_ping.py.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives evaluated before committing to root CHANGELOG.md + path-filter gate. Each rejected with a rooted reason.

Conventional Commits + autogenerated changelog

rejected

Why considered: the «commit message is the changelog» pattern — feat:, fix:, BREAKING CHANGE: prefixes auto-generate release notes via standard-version or similar.

Why rejected: autogeneration produces a flat «here is every commit» list that integrators do not actually read. The Keep-a-Changelog pattern of human-curated, categorised entries (Added / Changed / Fixed / Security) carries enough editorial judgment that the artefact is worth reading. Conventional Commits forces commit-message discipline that does not yield equivalent reader value.

GitHub Releases as the source of truth (no in-repo file)

rejected

Why considered: use the GitHub Releases UI to author notes per tag. Less file churn, native UI for browsing.

Why rejected: couples the changelog to GitHub — readers without web access cannot see it, and a repo mirror anywhere else loses the history. The whole Docs-as-Code argument (ADR 0001) is that the source of truth lives in git; release notes are a derivative view, not the canonical artefact.

No changelog — rely on PR titles and release tags

rejected

Why considered: minimal overhead — readers can search PRs by tag range.

Why rejected: «search the PR list» is exactly what the changelog is built to replace. The first question after a production incident is «what changed?»; if the answer is «open ten PRs and read them», the team has already lost the on-call minute.

Trade we accept by choosing root CHANGELOG.md + path filter. Every user-facing PR pays a 1-minute changelog-entry tax. We accept it because each rejected alternative either auto-generates noise (Conventional Commits), couples docs to a vendor (GitHub Releases), or fails the very lookup this ADR exists to optimise (no changelog).

Consequences

Positive

wins
  • Consumers and contributors share one obvious place for "what changed."
  • CI enforces the habit without parsing every intermediate commit in a multi-commit branch.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • Path-filter CI gate (vs. parse every commit). We gain a deterministic, fast PR check — «touched app/? then CHANGELOG.md must change too». We lose perfect coverage: a behaviour change purely in a config file outside the watchlist will not trigger the gate. We accept it because false negatives are caught at review time, while false positives (mechanical PRs forced to touch the file) are corrosive to the contributor's trust in the gate.
  • Squash-merge preference (vs. preserve commits). We gain one summary commit per change that maps cleanly to one changelog line. We lose the per-WIP commit history of long branches. We accept it because the changelog audience is the integrator (who reads release notes), not the archaeologist (who reads git blame); preserving WIP-commit messages on main would obscure the «what changed» answer this ADR is built to surface.

Revisit — would we do this again?

Yes, with one polish. The path-filter gate + squash-merge + LLM draft script has held up across ~6 months of PRs. One refinement:

  • Auto-rollup [Unreleased] on every history-table edit, not just on tag. Today the [Unreleased] section accumulates until release; auto-running scripts/changelog_draft.py as a daily cron with the previous HEAD~24h..HEAD range would keep the file warm and let on-call hot-grep against it during incidents instead of waiting for the next tag.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation plan

  1. Adopt the conventions in the Decision section across the repository.
  2. Update related templates, runbooks, and CI gates to reflect the rule.
  3. Cross-link from CHANGELOG and onboarding docs so future contributors discover the decision.

Validation

  • Pull requests targeting main or master: if the diff changes any path under services/api/app/, services/api/alembic/, services/portal/public/reference/api/, or the root README.md, then CHANGELOG.md must appear in the same diff unless a skip hatch applies.
  • Pushes to main or master: the same path rule applies to the range before..after from GitHub Actions. Commit messages in that range may satisfy the skip hatch. If before is all zeros (e.g. new branch pointer), the gate does not apply to that push.
