Environment Profiles and Config Governance

Three explicit profiles (dev, qa, prod), one committed template, layered loading, and strict validation that blocks unsafe settings outside dev.

Context

Why this matters: Twelve-factor style apps read config from the environment so the same build can run in dev, staging, and prod. If profiles are fuzzy, teams accidentally ship debug logging, mock auth, or open CORS to production—often discovered by incident, not review.

We need explicit profiles (dev, qa, prod), one documented template, and validation that rejects unsafe combinations in non-dev environments.

Decision

  • Adopt explicit environment profiles: dev, qa, prod only.
  • Single committed template: env/example (copy to .env via make env-init).
  • Load configuration in layers: .env -> env/<APP_ENV> -> optional ENV_FILE.
  • Automated tests use APP_ENV=qa (not a separate profile). Legacy APP_ENV=test maps to qa.
  • Normalize legacy aliases (for example local -> dev).
  • Expose effective profile in GET /live as app_env.
  • Enforce strict runtime validation for unsafe settings in qa/prod.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were evaluated before adopting three explicit profiles + fail-fast validation. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

No profiles — every setting is an independent env var

rejected

Why considered: the strict twelve-factor reading. No profile name, no validation rules tied to it — every behaviour is one explicit env var that the operator sets. Maximum composability.

Why rejected: with N independent env vars, every unsafe combination is reachable by accident. The matrix of «which combinations are valid?» grows exponentially and lives in nobody's head. A profile name collapses the matrix to «which row of the validation table am I on?», which is a tractable question. Composability we do not need is paid in clarity we do.

External secret/config service (Vault, AWS Parameter Store, GCP Secret Manager)

rejected

Why considered: the «secrets and config out of the env entirely» pattern — the app reads from a vault at boot. Standard for enterprise deployments with rotation requirements.

Why rejected: a vault adds a separate failure mode (vault down ⇒ app cannot boot), a separate auth model (vault credentials are themselves a secret), and a separate local-dev story (vault stub or shared dev secrets). For a solo project with no secret-rotation requirement yet, the env-file + git-ignored .env pattern is strictly cheaper. The day we have multi-tenant secret rotation is the day to revisit.

More profiles: dev, test, staging, prod, preview, …

rejected

Why considered: match a Heroku-style review-app workflow with one profile per deployment kind. More granular guarantees per environment.

Why rejected: three profiles cover «running locally», «running with prod-like constraints» (qa/test), and «running in production». Adding more profiles invites «which one applies?» ambiguity (is the load-test environment qa or its own load?) and grows the validation matrix N×. Three is the smallest set that lets us write meaningful per-profile rules without combinatorial blowup.

Trade we accept by choosing three explicit profiles + fail-fast validation. Every new validation rule is paid once at authoring time and stays paid. We accept it because each rejected alternative invites a failure mode this ADR exists to prevent: no-profiles ships unsafe combinations silently, external vaults add an outage axis we have not justified, and more profiles inflate the matrix faster than the value grows. Three profiles + a committed template is the smallest mechanism that is both auditable and operable.

Consequences

Positive

wins
  • Clear branch logic for environment behavior with safer defaults.
  • Reduced risk of accidental insecure production configuration.
  • Better developer onboarding for environment-specific setup.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure.

  • Fail-fast on unsafe config (startup refuses to boot). We gain a process that cannot serve traffic with mock auth in qa / prod — the validation rule is the gate, not a runtime check that can be silently disabled. We lose the option of «boot in degraded mode and warn» (some teams prefer this for partial-outage scenarios). We accept it because partial-boot with mock auth in production is the exact failure mode this ADR exists to prevent; the «soft warn» pattern always converges on «the warning was ignored, now we have the incident».
  • Profile name baked into the runtime (APP_ENV). We gain a single switch that the validation rules and the /live endpoint can read, so the effective profile is visible end-to-end. We lose the «one binary, infinitely composable env vars» purity that some twelve-factor purists prefer. We accept it because the profile is the load-bearing branch in our validation rules; making it implicit (derive from other vars) is the recipe for «which profile am I actually in?» confusion.

Revisit — would we do this again?

Yes, with one piece we did not commit to early enough. The three-profile + fail-fast model has caught two unsafe pre-deploys (a leftover LOG_LEVEL=DEBUG in a prod dry-run, a default mock key in a qa smoke). One addition we would commit to on day one if starting today:

  • Validate the env file before docker build, not at runtime. Today the validation runs at app startup; a misconfigured deployment produces a crash-loop pod that the operator has to inspect to learn what went wrong. A CI step that exercises services/api/app/core/config.py's validator with the production env file (with secrets stubbed) would fail the deploy at the pipeline step, not at the pod. The validator already exists; wiring it into pre-deploy CI is a 30-min job that closes the «caught at boot, not at build» gap.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.
