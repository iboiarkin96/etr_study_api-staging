Container Image

Provide a single Dockerfile aligned with make install so the bits CI tests can run on a laptop, a VM, or a cloud host without depending on a specific account.

Context

Why this matters: Local Python (make run) is the fastest feedback loop for feature work. Containers add a packaging boundary: the same bits you test in CI can run on a laptop, in a VM, or on a cloud host behind a registry.

Docker stays optional for daily coding; it complements the venv workflow. We ship a Dockerfile so integration tests and release flows do not depend on a specific cloud account.

Decision

  • Provide a Dockerfile at services/api/Dockerfile that installs dependencies from the same requirements.txt as make install, copies application and Alembic assets, runs as non-root user uid 1000, and sets ENTRYPOINT to the in-container /app/scripts/container_entrypoint.sh (copied from the host at services/api/scripts/container_entrypoint.sh): alembic upgrade head then Uvicorn. The image does not run make (no venv layout in-container).
  • Default container paths: SQLite file under /data (configurable via SQLITE_DB_PATH), logs under /tmp/logs when not overridden, to avoid permission issues on read-only layers.
  • Keep SQLite + single replica for this ADR's scope; document that scaling out requires a shared database.
  • Add Makefile helper docker-build and developer guide services/portal/internal/handbook/sa/authoring/docker-image-and-container.html.

Scope

  • In scope: image layout, entrypoint behavior, docker run documentation.
  • Out of scope: hosted registry promotion (see ADR 0021), production ingress/TLS, replacing SQLite with PostgreSQL.

Alternatives considered

Three alternatives evaluated before adopting the single-Dockerfile + entrypoint-migration pattern. Each rejected with a rooted reason.

docker-compose for the API container

rejected

Why considered: simple multi-container dev — bring up the API alongside any sidecars (DB, cache) with one command.

Why rejected: the observability compose stack (services/monitoring/docker-compose.observability.yml) already exists separately and is the right place for multi-container dev (Prometheus + Grafana + Blackbox). A second compose.yml for «just the API» would compete with it and force every contributor to learn which compose file to bring up for which task. A bare docker run study-app-api:local is the smaller mechanism.

Init container for migrations + app container without Alembic

rejected

Why considered: Kubernetes-native pattern — clean separation of concerns, the app container has zero migration code.

Why rejected: the pattern only pays off in K8s, where init containers are first-class. Standalone docker run for a contributor on a laptop has no init-container concept; supporting both modes means duplicating image config or shipping two images. The entrypoint-driven «migrate, then serve» path works identically in dev, CI, and registry pulls.

Distroless / scratch base image

rejected

Why considered: smaller attack surface, smaller image — distroless Python images strip out the package manager and shell, leaving only the runtime.

Why rejected: our entrypoint is a shell script (services/api/scripts/container_entrypoint.sh) that runs alembic upgrade head before uvicorn — distroless has no shell. We could rewrite the entrypoint as Python, but the shell version is the same path used in dev (make run), which is the load-bearing «one path for local and container» property. Distroless is the right move the day attack-surface review becomes a constraint; not before.

Trade we accept by choosing single-Dockerfile + entrypoint-driven migration. One image, one entrypoint, one shell path. We accept the larger base image and SQLite/single-replica coupling because each rejected alternative either splinters the «one mode for dev / CI / container» promise (compose, init container) or breaks the «shell entrypoint same as host run» property (distroless).

Consequences

Positive

wins
  • Single reproducible artifact (image) aligned with CI and release practices.
  • Readiness/liveness endpoints exercise the same checks used behind load balancers and in health probes.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • SQLite + single-replica scope. We gain a self-contained image that needs no external infrastructure to run — one docker run and the API is up. We lose multi-replica HA (SQLite cannot be the shared store across replicas). We accept it because the scope of this ADR is the packaging artefact, not the deployment topology; multi-replica is a Postgres-swap concern (narrative § 09), gated by trigger, not date.
  • Entrypoint runs Alembic before Uvicorn. We gain a single path that works identically in dev / CI / registry pulls — boot the container and the schema is up-to-date. We lose the K8s-pattern separation of init-vs-app concerns. We accept it because the boundary the init pattern enforces is not load-bearing in our deploy (we are not on K8s); the path-uniformity property is.

Revisit — would we do this again?

Yes. One refinement we would add on day one:

  • Build the image in CI on every PR, not just on tag. Today docker-build is a manual make target; a broken Dockerfile lands silently because nothing tests it pre-merge. A 90-second build job in .github/workflows/test.yml would catch dependency drift between requirements.txt and what the image actually installs.

Compatibility and migration

  • Backward-compatibility impact: none for Python APIs; additive files only.
  • Migration plan: N/A.
  • Rollback strategy: continue with make run without using the image.

Implementation plan

  1. Add services/api/Dockerfile, services/api/scripts/container_entrypoint.sh, services/api/.dockerignore.
  2. Document in developer guide and this ADR; link from ADR index.

Validation

  • Technical: docker build -t study-app-api:local . succeeds; docker run with appropriate env returns HTTP 200 on /live.
