OpenAPI Governance and Usability Standard

Treat OpenAPI as the contract: strict operation metadata, complete examples, response links, snapshot tests, and CI checks.

Context

Why this matters: OpenAPI is not only a human-readable page — it is the contract that clients and tests use. Thin examples, missing error shapes, and missing headers push integrators to read code or guess. That does not scale and costs support time.

We already had docs; the gap was quality and consistency: full validation examples, links between related operations, snapshot tests so the contract cannot drift quietly, and CI that treats the spec like application code.

Decision

We use an OpenAPI governance standard: strict operation metadata, complete examples for errors and writes, response links where workflows exist, and automated contract checks in the pipeline.

Scope

  • In scope: OpenAPI schema quality, endpoint docs usability, examples, links, response contracts, CI checks.
  • Out of scope: Public API monetization policies, external developer portal platform migration.

Alternatives considered

Three alternatives were evaluated before adopting strict OpenAPI governance. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints. Pattern from Tradeoff disclosure.

Keep ad-hoc docs and improve them organically

rejected

Why considered: the cheapest possible policy. Fix examples when an integrator asks, no governance overhead. The pattern most small APIs ship with.

Why rejected: ad-hoc means «improvements happen when someone complains». Our integrator is a single mobile team — they don't have the bandwidth to file granular feedback on every gap, so silent gaps accumulate. The eventual incident is «integrator misread the contract» which is unrecoverable from our side; the cost of a clear baseline is paid once per endpoint at authoring time, while ad-hoc drift is paid every time a new integrator joins.

RAML / API Blueprint as the primary spec format

rejected

Why considered: competing API description formats — RAML (MuleSoft) and API Blueprint (Apiary) have richer examples and traits/templates, and were popular for a stretch in 2015-2018.

Why rejected: FastAPI generates OpenAPI natively from Pydantic models; RAML / Blueprint would mean maintaining the spec separately from the code, which is the exact drift pattern this ADR is built to prevent. OpenAPI also has the biggest tooling ecosystem (Swagger UI, schemathesis, Spectral, openapi-diff) — every Phase 1-4 control here has off-the-shelf tooling. RAML's «traits» nicety is not worth the loss.

gRPC + Protobuf as the primary contract

rejected

Why considered: typed binary contract, strict by design, generated client SDKs in many languages. Used by Google internally and many service-mesh deployments.

Why rejected: our integrator is a mobile / browser client that wants HTTP/JSON over the public internet. gRPC over HTTP/2 with a translation gateway (grpc-gateway) is possible but adds an entire transcoding layer to debug, and breaks curl-debuggability — which is one of the load-bearing developer-experience surfaces (see ADR 0004). HTTP/JSON + OpenAPI is the smallest mechanism that gives the integrator a typed contract and the team a curl-able endpoint.

Trade we accept by choosing OpenAPI + strict governance. Every endpoint pays an up-front cost of ~20-40 min for complete examples, operation metadata, response links, and snapshot tests, plus an ongoing CI tax for linting (Spectral) and contract diffing. We accept it because the rejected alternatives either accept silent drift (ad-hoc), demand a parallel spec we'd have to keep in sync (RAML), or break the curl-debug primitive (gRPC). OpenAPI is the only spec format where the contract is the FastAPI app, not a second source.

Consequences

Positive

wins
  • Swagger becomes actionable: clear workflows, realistic examples, and stable response semantics.
  • Higher confidence for integrators and lower support load.
  • Breaking changes become detectable early via contract checks.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure.

  • Up-front authoring cost per endpoint. We gain a spec that integrators can read without bouncing to the code (complete 422 examples, response links, operation IDs, code samples). We lose ~20-40 min per endpoint and pay a recurring tax whenever a response shape changes. We accept it because the integrator's read of the spec is the load-bearing handover, and a spec with thin examples is operationally worse than no spec at all — readers stop trusting it after the first wrong example.
  • CI investment in linting + snapshot tests + contract diffing. We gain a build that fails when the spec drifts (Spectral lint, openapi-diff against the committed baseline). We lose CI minutes and accept that breaking-change PRs need explicit baseline updates. We accept it because the alternative — silent drift between code and baseline — is exactly what produced the 2026-03-14 OpenAPI baseline-drift postmortem; the CI gate is the cheapest insurance against that class of incident.

Revisit — would we do this again?

Yes, the highest-leverage decision in the project so far. The OpenAPI baseline-diff gate has caught two would-be breaking changes pre-merge and remains the one CI step nobody asks to skip. Two refinements we would commit to on day one if starting today:

  • Block PRs on baseline diff, not just warn. The 2026-03-14 postmortem named this exactly — for the first three months the CI step was non-blocking, and a long branch silently drifted. We now block; would have been free at day one.
  • Wire Spectral lint into the same PR gate as the type checker. Today Spectral runs in CI but its failures are reported separately; folding them into make verify with the same red-block treatment as mypy would close the «I'll fix it in a follow-up» loop that lets weak spec PRs land.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation plan

  1. Phase 1 (implemented): improve user endpoints with complete 422 examples, links, operation IDs, code samples, and read-after-write flow.
  2. Phase 2: add shared components.responses and reusable error definitions across routers.
  3. Phase 3: enforce OpenAPI linting (Spectral) and contract diff checks in CI/pre-deploy gates.
  4. Phase 3.1 (implemented): add strict snapshot contract test against accepted OpenAPI baseline.
  5. Phase 4: expand code samples and external docs links for operational recovery/runbooks.

Validation

  • Technical checks: make type-check, make test, make contract-test, make docs-fix.
  • Operational checks: manual Swagger walkthrough for create/get workflow, 4xx/422/429 examples and links.
  • Documentation updates: ADR index and generated handbook table via docs sync pipeline.

Standard requirements

  • Every operation has stable operationId, clear summary, and response model.
  • Validation errors use stable code/key envelopes with complete examples for all mapped rules.
  • Write operations document idempotency behavior and conflict semantics.
  • Where workflow exists, response links guide client to the next operation.
  • Key operations include code samples (curl minimum; optional additional languages).
  • Error responses include documented headers when relevant (for example rate-limit headers).
  • Deprecation must use explicit OpenAPI markers and migration guidance.
  • Snapshot contract tests are required to detect accidental changes in fields, error codes, and schema types.
