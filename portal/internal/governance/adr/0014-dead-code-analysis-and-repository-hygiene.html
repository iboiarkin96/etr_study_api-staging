Dead Code Analysis and Repository Hygiene

Pair a strict per-PR import lint with a softer scheduled dead-code scan so routine work stays fast while deeper cleanup still happens on a cadence.

Context

Why this matters: Dead code is not harmless—it confuses readers, hides the real execution path, and makes refactors riskier. Unused imports are easy wins: the linter catches them cheaply. Whole unused functions are harder: static "dead code" detectors use heuristics and can false-positive on dynamic registration or tests-only symbols.

We combine a strict gate (Ruff on every PR) with a softer periodic scan (Vulture) so routine work stays fast while deeper cleanup still happens on a schedule.

Decision

Adopt a two-layer approach:

  1. Ruff (required on every PR): keep the existing E/F rules so F401 (unused imports) is enforced in make lint-check / CI. Extend with RUF100 to remove obsolete noqa directives that no longer match an enabled rule.
  2. Vulture (advisory): configure [tool.vulture] in pyproject.toml for paths app, tests, and scripts. Run locally via make dead-code-check. Do not add Vulture to the default PR quality gate; run it on a weekly schedule (and manual dispatch) in GitHub Actions so maintainers triage findings without blocking every change.

Removals of allegedly dead code require human confirmation and tests (or equivalent evidence) before merge; static analysis alone is not sufficient for deletion.

Scope

  • In scope: configuration, Make target, scheduled workflow, contributor checklist.
  • Out of scope: automatic deletion, mandatory Vulture on every PR, full codebase purge in one change.

Alternatives considered

Three alternatives evaluated before adopting Ruff-strict + Vulture-scheduled. Each rejected with a rooted reason.

Vulture in CI for every PR

rejected

Why considered: fastest possible feedback on dead code — every PR catches the dynamic-registration cases that Ruff misses.

Why rejected: Vulture's heuristics false-positive on FastAPI dependency-injection helpers (e.g. Depends(get_db)) and pytest fixtures. Each false positive forces a whitelist entry or noqa comment; the whitelist itself becomes the next thing to maintain. The signal-to-noise ratio on PR-level Vulture is too low to be a green/red gate.

Ruff-only (no Vulture at all)

rejected

Why considered: zero extra dependency, very stable signal on the import layer.

Why rejected: imports are the easy case. Unused helper functions, dead conditional branches, and superseded handlers all pass Ruff cleanly because the import line still exists. The weekly Vulture run is the only mechanism that surfaces those — without it the codebase accumulates execution-path noise that compounds every quarter.

Coverage-driven dead-code detection (coverage.py + diff against test runs)

rejected

Why considered: «if the line is never covered by any test, flag it». Stronger signal than static heuristics — actual execution data.

Why rejected: coverage-driven detection conflates «no test covers this» with «code is dead». Plenty of live code paths (error handlers, rate-limit overflow branches) have no direct test by design. The false-positive class is worse than Vulture's because the failure mode is «delete a real error handler because no test triggers it», which produces production incidents instead of just whitelist churn.

Trade we accept by choosing Ruff-strict + Vulture-scheduled. Ruff catches the easy class on every PR; Vulture catches the hard class on a weekly cron without blocking routine work. We accept the human-triage step on the weekly run because static analysis alone is not sufficient evidence for deletion; the cost of a wrong delete is paid in production, the cost of human triage is paid in maintainer time.

Consequences

Positive

wins
  • Unused imports and stale noqa markers stay caught by the normal gate.
  • Optional deeper scans run on a predictable cadence without blocking routine work.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • Scheduled (vs. PR-gated) Vulture. We gain a fast PR pipeline that never blocks on a false positive. We lose the ability to catch a freshly-introduced dead function until next Monday's run. We accept it because the dead function is harmless until merged into main; the staleness window is bounded by the cron cadence and never accumulates beyond one week.
  • Maintainer triage on every weekly failure. We gain a guaranteed «someone looked at this» step before any deletion. We lose ~10 min of maintainer time on weeks with findings. We accept it because static-analysis-driven auto-delete is the failure mode this ADR exists to prevent; the alternative is a bot that deletes load-bearing FastAPI dependency-injection helpers.

Revisit — would we do this again?

Yes. The two-layer split (PR-fast + weekly-thorough) has produced ~2 maintainer-triaged removals per quarter without any false-delete incidents. One refinement we would add:

  • File the weekly Vulture findings as a GitHub Issue automatically. Today the workflow's red badge is the only signal; an auto-filed issue with the findings as a checklist would let triage happen async without anyone having to remember to check the workflow tab.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.
