High-relevance client-side docs search (inverted index, IDF, and ranking boosts)

Replace the simple field-presence scorer with an inverted-index lexical model — IDF, length-normalized TF, and precision boosts — plus append-only telemetry for query and click events.

Context

Docs are hosted as static files. We need fast, useful search across ADRs, runbooks, internal guides, developer pages, and API docs.

A simple "word present or not" scorer is easy to maintain but weak on large sites: common words win too often, long pages rank too high, and prefix search while typing needs better behavior.

Decision

We replace the old scorer with an inverted-index lexical model that combines:

  1. Per-field term frequencies at build time (title, URL, section, content).
  2. IDF-based weighting at query time.
  3. Log-scaled TF and document length normalization.
  4. High-precision boosts for exact phrase, all-token coverage, and title prefix matches.
  5. Prefix expansion only for the last query token for good type-ahead UX.

We also add search telemetry (append-only) in the app SQLite database so local runs stay simple and we can verify behavior.

Scope

  • In scope: all HTML pages under services/portal/, including services/portal/pdoc/ and services/portal/public/reference/api/.
  • Out of scope: neural/vector search, typo-edit-distance search, multilingual morphology.

Alternatives considered

Three alternatives evaluated before adopting the in-house inverted-index lexical model. Each rejected with a rooted reason.

Keep simple field-presence scoring

rejected

Why considered: tiny implementation, no IDF math, fewer parameters to maintain.

Why rejected: as the corpus crossed ~150 pages, common-word queries («docs», «adr») surfaced random pages because every page mentioned the word once. The precision-recall tradeoff was strictly the wrong way; users started copying URLs out of the sidebar instead of using search. The IDF + length-norm model fixes both failure modes at the cost of ~50 lines of code.

Hosted search providers (Algolia DocSearch, Pinecone)

rejected

Why considered: Algolia DocSearch ships free for open-source docs, rich relevance, analytics dashboard.

Why rejected: requires a crawler that the provider maintains on a schedule we do not control, plus per-page Algolia attributes scattered across the corpus. Internal pages also need access control we cannot delegate to a third party. The client-side index ships in the same artefact as the rest of the docs and never crosses a vendor boundary.

Client-side third-party libraries (lunr.js, MiniSearch, FlexSearch)

rejected

Why considered: mature ranking options, well-tested, large communities.

Why rejected: bundle sizes range 25-80 KB gzipped — larger than our entire docs-nav.js. The custom inverted index is ~3 KB of code that does exactly what we need (per-field weighting, prefix-of-last-token, exact-phrase boost) with no surplus. We re-evaluate when query volume justifies the library's ergonomics.

Trade we accept by choosing the in-house inverted-index model. Authors maintain ~50 lines of ranking code and one index-build script. We accept it because each rejected alternative either keeps the broken precision baseline (field-presence), introduces vendor lock and access-control gaps (hosted), or ships 10× the bundle for features we do not use (third-party libs).

Consequences

Positive

wins
  • Higher relevance for exact and navigational queries.
  • Predictable and debuggable ranking math.
  • Fast query path via postings-based candidate generation.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • Larger client-side index (KB-scale vs minimal). We gain typed-while-typing prefix search with field-weighted ranking. We lose ~100-300 KB of JSON shipped to every reader before they can search. We accept it because the loading is one-shot per session and cached; the precision improvement compounds across every search forever after.
  • Lexical-only — no semantic similarity. We gain deterministic ranking that is debuggable, reproducible, and explainable. We lose the «search for «idempotency» finds pages about «replay safety»» property a vector model would give. We accept it because corpus-vocabulary is small (~200 pages), and the «similar concept under a different name» case is rare; URL conventions and acronym casing carry most of the lookup work.

Revisit — would we do this again?

Yes — the precision improvement is the single most-cited UX win since the portal redesign. One refinement we would commit to on day one if starting today:

  • Wire the search-telemetry events into a per-query «zero-results» dashboard. Today the events stream into SQLite but nobody graphs them; queries that returned zero results would be the highest-signal input for «what content is missing?» — they are the user telling us, in their own words, what they expected to find.

Compatibility and migration

  • Backward-compatibility impact: low (search implementation change only).
  • Migration plan: generate new index version in existing docs pipeline.
  • Rollback strategy: revert to previous scorer and simple index schema.

Theoretical basis

Representation

Each document is represented as:

d = (title, url, section, preview, content_len, tf_title, tf_url, tf_section, tf_content)

where tf_field(term, d) is term frequency in that field.

Normalization

Input text is lowercased, punctuation-normalized, and whitespace-collapsed:

N(x) = trim(collapseSpaces(lowercase(x)))

Tokens are alphanumeric terms extracted from normalized text.

Core ranking formula

For query tokens T, base score is:

score_base(d, T) = Σ over t in T:
idf(t) * (
w_title   * log(1 + tf_title(t, d))
+ w_url     * log(1 + tf_url(t, d))
+ w_section * log(1 + tf_section(t, d))
+ w_content * log(1 + tf_content(t, d))
        )

Weights are tuned for precision-first behavior:

w_title = 8.0
w_url = 4.0
w_section = 2.0
        w_content = 1.4

IDF and length normalization

idf(t) = log(1 + (N + 1) / (df(t) + 0.5))

len_ratio(d) = content_len(d) / avg_content_len
norm(d) = 1 / (1 + 0.08 * max(0, len_ratio(d) - 1))

        score_norm(d, T) = score_base(d, T) * norm(d)

This downweights very common tokens and prevents long pages from winning by volume alone.

Precision boosts

Final score adds deterministic bonuses:

score_final = score_norm
+ B_all_tokens_in_title
+ B_all_tokens_in_url
+ B_exact_phrase_in_title
+ B_exact_phrase_in_url
+ B_title_prefix
        + B_exact_section

These bonuses enforce intuitive ranking for navigational queries and short phrase queries.

Complexity model

With an inverted index, query complexity becomes proportional to postings, not all documents:

Build: O(total_tokens)
Query: O(sum_postings_for_query_terms + rerank_candidates)
        Space: O(vocabulary + postings + doc_metadata)

This is substantially faster than scanning every document on each query.

Telemetry events and storage

Client events

  • search_query: one emitted query execution with query length, token count, result count, latency, and top-N impressions.
  • search_result_click: click-through on a result (mouse/keyboard), including rank and URL.
  • search_success: first successful click in a search session; includes time-to-success and time-to-click.
  • search_query_error: client-side index-load failure for observability and diagnostics.

Persistence model

  • All events are written append-only to docs_search_events in the current application SQLite DB.
  • Telemetry data and business data share one DB file to simplify local setup.
  • SQLite runs in WAL mode to reduce writer contention for frequent event inserts.

Metric definitions (canonical formulas)

Let Q be all search_query events in a time window, and S be first search_success per session in the same window.

Zero-result rate

zero_result_rate = count(q in Q where q.results_count = 0) / count(Q)

Query CTR

query_ctr = count(distinct query_id in Q with at least one search_result_click) / count(Q)

Time-to-first-success

TTFS = distribution of s.time_to_success_ms for s in S
        reported as p50 and p75

For this ADR, dashboard defaults are: p50 and p75 over the selected rolling window.
