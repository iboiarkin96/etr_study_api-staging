Portal search engine — migrate to Pagefind (English-only, GitHub-Pages-compatible)

Replace the in-house BM25 indexer and ranker (ADR-0027) with Pagefind — a maintained static-site search engine. The portal ships a lazily-loaded ~92 kB cold-payload index instead of a 2.3 MB monolithic JSON, keeps the existing UI shell, hotkeys, badges, and internal/public split, and becomes deployable to GitHub Pages without architectural change.

Context

ADR-0027 introduced a hand-written inverted index (scripts/build_docs_search_index.py, ~340 lines) plus a hand-written BM25 ranker in search.js (~270 lines of ranking logic). It works, but it costs ongoing maintenance — every change to the corpus shape, every new content type, and every browser API drift lands on us. The committed monolithic JSON index has also grown to 2.3 MB served on every first paint.

Three external constraints shape this revisit:

  • The portal is moving to English-only content (separate decision, 2026-06-06). Cyrillic stemming is no longer a constraint on engine choice.
  • We want the portal to be deployable to GitHub Pages without server-side search infrastructure.
  • The remaining ranking work that ADR-0027 deferred (telemetry-fed zero-results dashboard, query analytics) is dwarfed by what an off-the-shelf engine already gives us — shipped, tested, and someone else's responsibility.

Decision

We migrate portal search to Pagefind — a Rust-built static-site search engine maintained by CloudCannon and the Astro team.

  1. Build-time: tools/docs/build_pagefind_index.py mirrors services/portal/**/*.html into a staged tree, injects data-pagefind-filter="visibility:internal|public" and data-pagefind-meta="kind:..." onto each <main>, then runs the pinned pagefind CLI against the staged tree with SOURCE_DATE_EPOCH=0 for reproducibility.
  2. Runtime: the existing topbar search component swaps its BM25 inner loop for a thin adapter over pagefind.search(query). UI shell, hotkeys (/, ⌘K, Esc), debounce, badge taxonomy, and <mark> highlighting remain unchanged.
  3. Internal/public split moves from two JSON files to a single bundle filtered at query time via Pagefind facets.
  4. The built bundle services/frontend/portal/assets/pagefind/ is committed so GitHub Pages can deploy the portal directly from the git tree. Pagefind embeds per-build hashes in pf_meta / pf_filter, so the bundle is not byte-stable between runs; docs-check handles this the same way it already handles pdoc — restore the committed bundle from HEAD after docs-fix, then diff (the committed snapshot is the source of truth for drift).
  5. Visibility invariant is enforced by tools/docs/check_pagefind_visibility.py, invoked from make docs-fix right after the build — it scans the bundle and asserts every public URL carries visibility:public and every internal URL carries visibility:internal.

Pagefind is pinned via PyPI (pagefind==1.5.2 + pagefind_bin_extended==1.5.2) so the bundled binary is reproducible on every Python-aware CI runner without a separate Node/npm step.

Scope

  • In scope: all HTML pages under services/portal/ (internal + public).
  • In scope: the in-house BM25 builder, ranker, ranking telemetry, and the two-file public/internal split — deprecated and removed by waves W4/W5 (see Migration plan).
  • Out of scope: neural/vector search, server-side analytics dashboards, multilingual morphology (portal is English-only).

Alternatives considered

Four alternatives evaluated before adopting Pagefind. Each rejected with a rooted reason.

Keep ADR-0027 (in-house BM25)

rejected

Why considered: zero migration cost, well-understood code, already tuned for our corpus.

Why rejected: the 2.3 MB monolithic JSON is served to every reader on first paint and grows linearly with the corpus. Maintenance falls entirely on us — tokenizer, ranker, prefix expansion, length norm, telemetry. Pagefind ships the same precision behaviour, lazy-loaded in ~92 kB gz cold-payload, and the ranking layer becomes someone else's responsibility.

MiniSearch / Lunr / FlexSearch (client-side JS libraries)

rejected

Why considered: mature client-side ranking, no Rust binary in the build pipeline.

Why rejected: they consume a JSON index we still have to author and ship in full to the browser — payload stays in the megabytes, lazy-loading is not part of the model. Lunr is maintenance-mode upstream. Picking any of them keeps us as the owners of build + tokenizer + corpus-specific tuning code.

Algolia DocSearch (SaaS)

rejected

Why considered: free tier for open-source docs, mature relevance, hosted analytics.

Why rejected: internal pages cannot be crawled by a third party without access-control delegation we do not want to grant. The crawl schedule and ranking are controlled by the vendor. Pagefind keeps the index in the same artefact as the docs and never crosses a vendor boundary.

Orama / Stork

rejected

Why considered: Orama is a young JS engine with growing community; Stork is/was Rust-based with similar static-site posture.

Why rejected: Orama still takes a JSON-shaped input — we would author and ship our own builder, so the payload-reduction advantage disappears. Stork has been unmaintained since 2023, which contradicts the entire premise of BL-064.

Trade we accept by choosing Pagefind. We add a pinned Rust binary to the CI dependency surface (delivered via PyPI wheel, so CI install is a single pip install). In return we shed ~600 lines of in-house indexer and ranker code, a 2.3 MB first-paint payload, and the entire maintenance backlog for the search subsystem.

Consequences

Positive

wins
  • Cold-payload drops from 2.3 MB to ~92 kB gz (~25× reduction). Per-query overhead drops to ~30 kB.
  • ~600 lines of in-house BM25 / tokenizer code deleted from the long-term maintenance surface.
  • Internal/public split moves from two committed JSON files to a single facet — half the build artefacts, no subset-invariant check needed.
  • The portal becomes statically deployable to GitHub Pages (with .nojekyll) — no architectural change required.
  • Per-result excerpts (data()) lazy-loaded — large pages no longer cost payload until clicked.

Trade-offs

costs

Two named tradeoffs. Shape from Tradeoff disclosure.

  • External engine — ranking is no longer ours to tune. We gain a maintained BM25 + stemming pipeline shipped, tested, and updated by someone else. We lose the ability to add a custom precision boost the way ADR-0027 did (all-tokens-in-title, exact-phrase, title-prefix). We accept it because the reference-query fixture (≥18/20 top-1, ≥17/20 top-3 superset) shows Pagefind reaches parity, and the engine exposes facets for the few cases where we want categorical narrowing.
  • Pinned Rust binary in the CI dependency surface. We gain a 0.8 s full-corpus build (vs ~3 s for the legacy Python builder) and lazy-loaded chunks. We lose pure-Python install simplicity — CI now downloads a ~52 MB platform-specific wheel (pagefind_bin_extended). We accept it because the wheel is installed through pip install -r requirements.txt — the same flow we already use for every other binary dependency (mypy, ruff, pdoc).

Revisit — would we do this again?

Yes — the payload and maintenance numbers alone justify it. The one refinement we would commit to on day one if starting today: ship the Pagefind facet UI from the first cutover instead of treating it as optional W6 polish. The kind/visibility filters land for free once the bundle is in place, and the UX win of «search inside ADRs only» is large enough that we should not gate it behind a second PR cycle.

Compatibility and migration

Migration runs in six waves (W0–W5) plus optional polish W6, tracked under BL-064:

  • W0 — Spike on the spike branch spike/bl-064-pagefind ✅ (2026-06-06: 0.82 s build, 92 kB cold-payload, 18/20 reference queries).
  • W1 — This PR. New builder + visibility check + Makefile step + pre-commit hook. Legacy index still generated in parallel. Client unchanged.
  • W2 — Frontend dual-mode behind a feature flag (?engine=pagefind).
  • W3 — UI parity (badges from data-pagefind-meta, snippet highlight mapping, public-portal filter); Playwright eval harness in CI.
  • W4 — Cutover. DEFAULT_ENGINE = "pagefind". This ADR flips to Accepted. ADR-0027 flips to Superseded by ADR-0033 with a link in its header.
  • W5 — Delete the legacy builder, ranker, mocks, and the two JSON indices.
  • W6 (optional) — Facet chips in the search panel.

Rollback strategy: legacy code remains generated and shipped through W4. A single-line revert of DEFAULT_ENGINE returns the portal to the ADR-0027 ranker without a re-deploy of the bundle.

Amendment · 2026-06-08 — index built on CI, not committed

Two weeks of running with the Pagefind bundle committed to services/frontend/portal/assets/pagefind/ surfaced a class of friction the original decision did not anticipate:

  • Hash drift on every rebuild. The CLI is not bit-stable across runs even with SOURCE_DATE_EPOCH=0: pagefind-entry.json embeds a per-build hash (e.g. en_853a55789e → en_2a82752140 → en_49f59d561d) that changes for an unchanged page set; .pf_meta filename is hash-derived, so a no-op rebuild creates orphan blobs that need explicit removal.
  • Pre-commit normalisers fight the CLI. end-of-file-fixer and trailing-whitespace rewrite pagefind.js / pagefind-worker.js / pagefind-component-ui.js / pagefind-modular-ui.css on commit; the next make docs-fix run undoes the normalisation; cycle repeats indefinitely.
  • Reviewer noise. Routine PRs carry 100–700 blob additions/deletions that crowd out the actual content diff.
  • Workaround tax. make docs-check previously restored the Pagefind tree from HEAD before the drift diff (same pattern as pdoc) — a symptom-level patch that did not stop the noise on every commit cycle.

The industry pattern for static-site search artefacts (Astro / Hugo / Docusaurus / Eleventy / Backstage docs) is build on CI, do not commit: the index is a build artefact, not source. We adopt that pattern.

Changes landing with this amendment

  • .gitignore ignores the entire Pagefind tree.
  • services/frontend/portal/assets/pagefind/ is dropped from git tracking via git rm --cached (files remain on disk for local dev preview).
  • .github/workflows/deploy-prod.yml installs pagefind==1.5.2 + pagefind_bin_extended==1.5.2, runs tools/docs/build_pagefind_index.py + tools/docs/check_pagefind_visibility.py, then stages the just-built tree into the Pages artifact. Local make docs-fix still builds it for browser smoke.
  • Makefile docs-check drops the git checkout HEAD -- services/frontend/portal/assets/pagefind restore step — no longer needed once the tree is untracked.

Status pill remains Accepted: the engine choice and architecture do not change — only the publication boundary moves from the source tree to the build pipeline. ADR-0027 stays Superseded by this ADR.

Validation

  • Run make docs-fix and verify both legacy [9/10] and pagefind [10/10] steps succeed.
  • Run make docs-check on a clean checkout — the pagefind bundle must be drift-free (SOURCE_DATE_EPOCH=0).
  • Run tools/docs/check_pagefind_visibility.py — every public URL must carry visibility:public, every internal URL visibility:internal.
  • Run tests/portal_search/spike_eval.py --bundle services/frontend/portal/pagefind --queries tests/portal_search/reference_queries.yaml — fragment-level smoke must pass.
  • Browser-level evaluation (W3): Playwright-driven scripts/search_eval.py against the 20-query fixture; gate top_1 ≥ 18/20, top_3 superset ≥ 17/20.
  • Manual smoke: /, ⌘K, Esc, ArrowUp/Down, Enter on the topbar input.
