Health, Readiness, and Prometheus Observability

Separate liveness and readiness probes, Prometheus metrics on /metrics, and OpenAPI snapshot contract tests in the quality gate.

Context

Why this matters: Runtimes and load balancers must know whether to send traffic to an instance: liveness (“is the process alive?”) and readiness (“can this instance serve requests right now?”—including DB). Without separate probes, restarts and routing decisions are guesses.

Metrics (latency, status codes, DB time) turn “it feels slow” into graphs and alerts. We had a basic health path but needed explicit readiness and a Prometheus baseline aligned with staging/production.

Decision

  • Introduce /live for process liveness checks.
  • Introduce /ready for dependency readiness checks (including DB probe).
  • Expose Prometheus metrics via /metrics.
  • Collect API latency, HTTP status counters, and DB operation latency with low-cardinality labels.
  • Adopt OpenAPI snapshot contract tests (make contract-test) in the quality gate.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were evaluated before adopting /live + /ready + pull-based Prometheus. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

One unified /health endpoint (no liveness / readiness split)

rejected

Why considered: simpler — one URL, one shape, one piece of doc. The default pattern in many small services.

Why rejected: a unified endpoint conflates two questions the runtime needs to answer separately. «DB is briefly unreachable» should pull the instance out of the load-balancer pool but not restart it; «process is stuck in a deadlock» should restart it. With one endpoint, every probe failure looks the same and the orchestrator's policy is forced to be «restart on any failure» — which causes more outages than it prevents during transient DB hiccups.

Push-based metrics to a hosted vendor (Datadog / New Relic / Honeycomb)

rejected

Why considered: «buy not build» for observability — vendor SDK in the app, hosted dashboards, alerting and traces out of the box. Saves the dockerised Prometheus + Grafana stack.

Why rejected: vendor-lock at the wire format. Every metric name, label, tag, and aggregation policy ends up vendor-specific; migrating to a second vendor (or self-host) is a full rewrite of the metric layer. Pull-based /metrics keeps the application's observability contract neutral — the consumer is swappable. For a solo project that has not yet picked a long-term observability stack, optionality is worth more than turnkey ergonomics.

OpenTelemetry collector + OTLP from day one

rejected

Why considered: the modern «one signal pipeline for metrics, logs, traces» pattern. OTLP is becoming the de-facto wire format; adopting it now means traces «just work» when we add them later.

Why rejected: OpenTelemetry collector is a separate process to operate, with its own config, version skew, and failure modes. We do not yet emit traces (Vol. 2 audit gap #2 — observability beyond logs) and our metrics needs are served by the Prometheus client library in-process. Adding the OTel collector before we have traces or distributed services is paying the operational tax before the benefit lands. We re-evaluate this the day we add the second service.

Trade we accept by choosing /live + /ready + pull Prometheus. We pay for two endpoints (vs one) and operate a dockerised Prometheus stack locally (vs paying a vendor). We accept it because each rejected alternative forces a worse outcome: unified /health conflates restart-vs-route-out, vendor-push locks the wire format, and OTel-from-day-one taxes us before the benefit. The split-endpoint + pull-metrics pair is the smallest mechanism that is both vendor-neutral and restart-policy-friendly.

Consequences

Positive

wins
  • Faster diagnosis of incidents via explicit readiness and standardized metrics.
  • Reduced risk of accidental contract drift through snapshot tests.
  • Consistent observability baseline from local to staging/production.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure.

  • Two endpoints (/live and /ready) instead of one. We gain the ability for a load balancer or orchestrator to distinguish «process up but DB not reachable yet» (don't route, don't restart) from «process dead» (restart). We lose the simpler one-endpoint mental model. We accept it because the day a deploy is rolling and the new instance has not yet completed its DB connection-pool warmup is the day this distinction prevents an incident; the cost of the extra endpoint is paid once, the benefit pays every deploy.
  • Pull-based Prometheus on /metrics. We gain the standard ecosystem — every off-the-shelf dashboard, alert template, and exporter assumes pull. We lose the option of «push to a hosted observability vendor» without a translator (statsd-exporter, OpenTelemetry collector). We accept it because the vendor-push pattern locks us into one vendor's wire format; pull-based /metrics keeps the contract neutral and lets us swap the consumer (Prometheus / Mimir / Cortex / Grafana Agent) without touching the app.

Revisit — would we do this again?

Yes, with one piece of homework that is overdue. The /live + /ready + /metrics trio has aged well — the contract is stable, the dockerised stack runs in dev and staging, and the OpenAPI snapshot test has caught two would-be contract drifts. One thing we are explicitly missing today (Vol. 2 audit gap #2):

  • Wire /metrics to a live dashboard from day one, not just expose the endpoint. Section 14 of every operation spec promises metric names that no one is graphing yet. Today the metrics are emitted; tomorrow nobody knows whether the burn-rate calculation is correct because the input is unmeasured. Standing up the dockerised Grafana with one canonical dashboard (HTTP RPS, p50/p95/p99, error rate, DB query latency) would close the «we have metrics» / «we have observability» gap that the architecture narrative names as the next architectural move.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.

Operational Model

  • Prometheus uses pull-based scraping from /metrics.
  • Dockerized Prometheus + Grafana is the default local/staging bootstrap path.
  • Production deployment may run the same stack outside Docker (for example on a VM) with the same endpoint contract.
