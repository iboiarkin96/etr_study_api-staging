Monorepo with service boundaries

One repository, three service trees under services/{api,portal,monitoring} — independent deploy units (per-service Dockerfile, per-service CI job) without the multi-repo coordination tax.

Context

The repository currently mixes three runtime concerns under a single Python application tree:

  • The HTTP API (FastAPI in app/): public /api/v1 surface, governed by ADR 0004 and ADR 0007.
  • The documentation portal (static HTML/CSS/JS under services/portal/): ADRs, RFCs, internal docs, OpenAPI explorer, search index — Docs as Code per ADR 0001.
  • The monitoring stack (Prometheus / Grafana / optional local Elasticsearch + Kibana): observability wiring per ADR 0009 and ADR 0023.

These concerns share one image, one CI pipeline, and one deploy unit. That coupling is fine while the project is small, but it blocks several Senior-SA-level evolutions: the portal cannot be served by a static-asset host independently of the API, the monitoring stack cannot be promoted or rolled back without the API, and the API image carries documentation bytes it never needs at runtime.

A textbook answer would be to split into three GitHub repositories (one per service) and glue them together via a GitHub Organization or a meta-repo with submodules. For a solo project under the 2026-05-30 deadline, that path multiplies CI configuration, version-skew risk, contract-sync ceremony, and cross-repo refactor cost — all without changing what actually runs in containers. We need a decision that captures the architectural separation without the multi-repo tax.

Decision

  • Keep a single GitHub repository. Move from the flat app/ + services/portal/ layout to a service-rooted layout under services/:
    • services/api/ — FastAPI runtime: routes, models, migrations, the existing container entrypoint (services/api/scripts/container_entrypoint.sh), and the API Dockerfile.
    • services/portal/ — static documentation site (today's services/portal/ tree), served by a thin static-asset image (e.g. nginx:alpine) with its own Dockerfile.
    • services/monitoring/ — Prometheus / Grafana / Elasticsearch / Kibana / Filebeat configuration, dashboards, and any custom exporters; its own Dockerfile only when an exporter or sidecar is built locally.
  • Each service is an independent deploy unit: its own Dockerfile, its own dependency surface (requirements.txt for the API, package.json only if a build step is added later, declarative configs for monitoring), and its own image tag in GHCR per ADR 0021.
  • A top-level docker-compose.yml orchestrates the three services for local development. The existing observability compose file is consolidated into services/monitoring/ and referenced from the root compose via extends or include.
  • CI runs as a matrix job per service: each service builds, tests, and lints independently. Cross-cutting checks (link checker, OpenAPI contract tests, ADR/RFC link checker) stay at repo root and run once.
  • Governance artifacts — ADRs, RFCs, runbooks, audits, CHANGELOG, OpenAPI baseline — remain outside services/. They are repo-wide concerns, not per-service concerns. Their physical home is services/portal/internal/governance/{adr,rfc,audit,backlog}/ and services/portal/internal/sre/runbooks/ per RFC 0004; the OpenAPI baseline lives at services/portal/public/reference/api/ per ADR 0007; services/portal/CHANGELOG.md stays at services/portal/ root as the release journal. (Amendment 2026-05-08 — replaces the original specific path list with the underlying invariant "outside services/".)
  • This decision does not introduce a separate database per service, a service mesh, or inter-service RPC. The portal remains a pure static site; monitoring scrapes the API over HTTP. Splits beyond that are out of scope and need their own ADRs.

Scope

  • In scope: repository folder layout, per-service Dockerfile rule, root docker-compose.yml orchestration, CI matrix shape, governance-artifact placement.
  • Out of scope: splitting the database, introducing inter-service RPC or a message bus, replacing SQLite with PostgreSQL (see ADR 0015 trade-offs), choosing a static-host runtime image for the portal, multi-cluster or multi-region deployment topology.

Alternatives considered

Status quo: single flat layout, one Dockerfile, one CI pipeline

Pros: zero migration cost; familiar layout for the current solo workflow.

Cons: no enforced boundary between API, portal, and monitoring — any of them can pull the others into their dependency closure. Cannot release the portal without rebuilding the API image. Forces a future "true microservices" migration to do the folder split and the repo split at the same time, doubling risk.

Multi-repo + GitHub Organization

Pros: textbook microservices answer; per-repo permissions, issues, and release cycles; each repo looks "small".

Cons: triples the CI configuration; ADR/OpenAPI/CHANGELOG must either duplicate across repos or live in a fourth "governance" repo with its own sync problem; cross-cutting refactors (e.g. a new error-code in ADR 0003) become 3-PR coordination exercises; contract drift between API and portal becomes a real risk because they are no longer edited in the same diff. Net cost is high for one engineer with a 2026-05-30 deadline.

Multi-repo + meta-repo with git submodules

Pros: keeps a single "front door" repo for orchestration and governance; per-service repos can evolve independently.

Cons: submodules are notoriously hostile to day-to-day editing — every cross-cutting change is at least four commits in two repos with a pinned-SHA dance; CI for the platform repo must clone submodules and reproduce per-service tests; contributors need to understand git submodule update --init --recursive and submodule branch tracking. Power tool, wrong size for this project.

Single repo, single Dockerfile, multiple CMD targets

Pros: minimal change; one image to push to GHCR.

Cons: portal and monitoring drag the API runtime closure into their image; image size and security surface stay coupled; no independent deploy unit, just an illusion of one. Defeats the architectural goal.

Consequences

Positive

wins
  • Independent deploy units without multi-repo overhead. Each service has its own image tag and can be promoted, rolled back, or scaled without touching the others.
  • Smaller, cleaner images. The API image stops shipping services/portal/; the portal image stops shipping Python; monitoring is configuration, not a Python runtime.
  • Cross-cutting changes stay one PR. An error-code rename, an ADR update, or an OpenAPI contract bump still lands in a single commit, reviewed against all affected services.
  • Defensible at the Senior SA bar. The decision matches the layout used by Google, Meta, Uber, and other monorepo-first organizations; the trade-off table above is the answer to the interview question "why not microservices in separate repos?"
  • Migration path is incremental. The folder split is a no-behavior-change refactor; per-service Dockerfiles, compose, and CI matrix can land in subsequent PRs.

Trade-offs

costs

Two canonical tradeoffs (the four-bullet list is preserved as the «what changes» surface; these two name the accepted-cost-vs-gain shape). Shape from Tradeoff disclosure.

  • Folder boundary, not repo boundary. We gain the architectural separation (independent deploy units, per-service image, per-service CI matrix) without the multi-repo coordination tax (3× CI config, 3-PR refactors, contract-drift risk). We lose the explicit «only this team can touch this service» guarantee that separate repos enforce via permissions. We accept it because the per-team isolation is not the constraint for a solo project; folder boundaries + CODEOWNERS is the lighter mechanism that scales when team grows.
  • Path depth + tooling-config churn. We gain «refactor without renaming images» — the API stays at services/api/ through every internal restructure. We lose flat app/main.py imports and accept that Ruff, pytest, pip-audit, and the link checker need path updates. We accept it because the depth cost is paid once at the rename, while the «one repo, three deploy units» benefit pays every release.

The implementation-level costs (CI runtime, coarse permissions, partial-microservices answer) are documented in the original four bullets above — kept verbatim as the operational checklist contributors will hit during the four-phase rollout.

Revisit — would we do this again?

This ADR is Proposed; the «would we do this again?» question is forward-looking until Phase 1 ships. The clearest day-one investment we would commit to alongside Phase 1:

  • Add a CODEOWNERS file at the same commit as Phase 1's folder split. Today the codebase has no CODEOWNERS; the day a second contributor lands is the day folder-as-boundary needs reviewer enforcement. Adding it during the folder split is free; retrofitting it later is friction.

Compatibility and migration

  • Backward-compatibility impact: medium. Public HTTP contract is unchanged; image names and tooling paths change. Consumers of the GHCR image (per ADR 0021) must switch from the single study-app tag to study-app-api, study-app-portal, study-app-monitoring.
  • Migration plan (phased, each phase shippable on its own):
    1. Phase 1 — folder split, no behavior change. Move app/ to services/api/app/, leave services/portal/ in place but add a placeholder services/portal/ that points at ../../docs/; update Ruff, pytest, pip-audit, and the link checker for the new paths. CI stays green with the existing single image.
    2. Phase 2 — per-service Dockerfiles. Move the API Dockerfile into services/api/, add a static-asset Dockerfile in services/portal/, add a thin services/monitoring/Dockerfile only if a sidecar is built locally.
    3. Phase 3 — root docker-compose.yml. Replace ad-hoc local orchestration with a single root compose file that brings up all three services. Consolidate the existing observability compose into services/monitoring/.
    4. Phase 4 — CI matrix per service. Split the CI workflow so each service builds and tests independently; cross-cutting checks (link checker, OpenAPI tests) stay at root.
    Each phase is a separate backlog item; this ADR ratifies the destination, not the schedule. Status (2026-06-09): all four phases landed in the BL-065 PR (Phase 1 — folder split; Phase 2 — per-service Makefiles + per-service scripts; Phase 3 — root docker-compose.yml; Phase 4 — CI matrix + per-service GHCR tag scheme). Boundary enforcement: _shared/scripts/check_service_imports.py pre-commit hook (master C11).
  • Rollback strategy: revert the offending phase. Phase 1 is a pure rename, fully reversible by git revert; later phases are additive (new Dockerfiles, new CI jobs) and can be reverted without touching the API runtime.

Implementation plan

  1. Create the four backlog items (one per migration phase) under services/portal/internal/governance/backlog/ with explicit acceptance criteria; do not bundle them into one ticket.
  2. Land Phase 1 first — folder split with no behavior change — and verify make verify stays green per ADR 0008.
  3. Land Phase 2 (per-service Dockerfiles) only after the API image still passes ADR 0015 validation (docker build, /live 200).
  4. Land Phase 3 (root compose) and update developer onboarding under services/portal/internal/handbook/developer/ in the same PR.
  5. Land Phase 4 (CI matrix) only after Phase 2 + 3 are stable for at least one release per ADR 0017.

Validation

  • Technical checks: make verify remains green after each phase; per-service docker build succeeds in Phase 2; root docker compose up brings up the API, the portal, and Prometheus + Grafana in Phase 3; CI matrix shows three independent green jobs in Phase 4.
  • Operational checks: /live and /ready on the API service still match ADR 0009; structured-logging contract holds per ADR 0023; SLOs from ADR 0011 are unaffected.
  • Documentation updates: this ADR linked from services/portal/internal/governance/adr/index.html; services/portal/CHANGELOG.md records the proposal; per-phase backlog items in services/portal/internal/governance/backlog/; developer onboarding under services/portal/internal/handbook/developer/ updated when Phase 3 lands.
