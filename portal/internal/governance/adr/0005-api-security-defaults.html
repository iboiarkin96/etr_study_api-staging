API Security Defaults

One security baseline for every /api/v1/* route: rate limit, CORS allowlist, headers, body cap, and a default auth strategy.

Context

Why this matters: Security is not one switch — it is a set of defaults: who may call the API, how much traffic they can send, which browser origins we trust, and how large bodies may be. If those defaults differ between a laptop and staging/production, teams often find gaps only after abuse or incidents.

Set a baseline early (even with mock auth) so integrators learn real headers and keys. Moving later to real identity is then mostly configuration, not a full redesign.

Decision

Protected routes under /api/v1/* share one security baseline:

  1. Per-client rate limiting is enabled by default.
  2. CORS is allowlist-based and configured from environment.
  3. Security headers are attached to every response.
  4. Request body size is capped via configurable global limit.
  5. Basic auth strategy is required in all environments; current default is mock API-key auth.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

Three alternatives were evaluated before committing to the single-baseline middleware. Each is real (another serious team in the industry uses it) and explicitly rejected with a reason rooted in our constraints.

Per-route ad-hoc security (no middleware baseline)

rejected

Why considered: the most flexible approach — each router declares its own auth dependency, its own rate limit, its own body size cap. Common in early-stage Express / FastAPI codebases that have not yet centralised middleware.

Why rejected: drift. The day someone forgets the auth dependency on a new endpoint is the day the endpoint ships unprotected. With a centralised middleware the contract is «every route under /api/v1/* is protected by default»; opt-out is the explicit deviation, not the default. The failure mode this ADR exists to prevent is the per-route variant's normal operating state.

API gateway in front (Kong / AWS API Gateway / Cloudflare Workers)

rejected

Why considered: standard mid-size pattern — terminate rate limiting, auth, CORS, and body caps at an edge gateway, leaving the app service to handle only business logic. Lets the gateway scale independently and centralises observability of edge failures.

Why rejected: infra dependency. Adding a gateway means local dev needs to emulate it (Kong-in-Docker, AWS SAM local, or a Workers stub), and every contributor has to know two failure modes — app and gateway. We run one FastAPI service in production; adding a gateway overshoots the operating complexity by ~5× without a matching benefit. The day we have multiple backend services is the day to revisit, not before.

OAuth 2.0 / OIDC provider from day one (Auth0 / Keycloak / Cognito)

rejected

Why considered: the «do auth right from the start» argument — bring in a real identity provider, real JWTs, real token rotation, so migration later is a no-op. Used by enterprise teams who already have an IdP.

Why rejected: we are single-tenant, single-integrator, and the API key is the tenant (the composite (system_user_id, system_uuid) is the learner inside the tenant). OIDC machinery — discovery, JWKS rotation, refresh-token flow, identity-claim mapping — is empty ceremony until there is more than one integrator and more than one identity provider. The current mock API-key strategy + the API_AUTH_STRATEGY switch makes the future swap a configuration change, not a redesign.

Trade we accept by choosing one middleware baseline + mock-auth default. We pay ~30 min of local-setup overhead per contributor (configuring env vars to match the baseline) and live with single-process rate-limit accuracy until we have a second replica. We accept it because each rejected alternative either invites the exact drift this ADR exists to prevent (per-route ad-hoc), demands infra we have not justified (API gateway), or accepts machinery whose value is zero at our scale (OIDC). The baseline is the smallest mechanism that is both uniform and visible.

Consequences

Positive

wins
  • Security constraints are explicit and testable from day one.
  • Client teams know non-functional contract limits early.
  • Future migration to production auth can replace mock strategy without policy rewrite.

Trade-offs

costs

Two named tradeoffs accepted on purpose. Shape from Tradeoff disclosure: «we gain X, we lose Y, we accept it because Z».

  • One baseline across every environment. We gain dev ≡ staging ≡ prod — no «works on my laptop, fails when the rate limiter is on» class of incidents. We lose the faster dev loop you get when CORS, rate-limit, and body cap are disabled locally. We accept it because dev/prod drift is the exact incident class this ADR exists to prevent; the dev-loop delta is ~50 ms per request and never compounds, while drift fails open at 2 a.m. and burns SLO.
  • In-memory, process-local rate limit. We gain zero-infra enforcement at sub-millisecond cost (services/api/app/core/security.py sliding window). We lose multi-instance accuracy — each replica enforces its own window, so an attacker hitting two replicas gets 2× the allowance. We accept it because we run single-instance today; the trigger to swap to Redis is «two replicas in production», not «hit some throughput threshold».

Revisit — would we do this again?

Yes, with one piece of homework we should have done in week 1. The baseline-middleware bet has aged well — there is still no daily «forgot to attach auth» incident, and the dev ≡ prod equivalence has paid back on every contributor onboarding. One thing we would commit to on day one if starting today:

  • Write a threat model first (STRIDE + adversary tiers), then derive these defaults from it. Today the rate-limit, CORS allowlist, and body cap are defaults we believe are right; they would be controls that defend against a named tier of attacker if a threat model existed. Vol. 2 audit High gap #4 names this — the threat-model page is the missing «receipts» for this ADR. Each control here would cite the tier it mitigates (T2 motivated attacker, T3 script kiddies), and the «we accept X out of scope» line would be in the threat model instead of implicit. We retrofit this; would have been free at day one.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Configuration

Concern Environment variables Note
Rate limiting API_RATE_LIMIT_REQUESTS, API_RATE_LIMIT_WINDOW_SECONDS per-client sliding window
Body cap API_BODY_MAX_BYTES global limit; over-cap returns 413
CORS CORS_ALLOW_ORIGINS, CORS_ALLOW_METHODS, CORS_ALLOW_HEADERS, CORS_ALLOW_CREDENTIALS allowlist-based; never * in production
Auth API_AUTH_STRATEGY, API_MOCK_API_KEY, API_AUTH_HEADER, API_PROTECTED_PREFIX current default: mock API-key strategy; production swap is configuration only

Contract impact

  • Unauthorized requests return 401 with stable security error payload.
  • Body overflow returns 413.
  • Rate-limit overflow returns 429 and Retry-After header.
  • Protected endpoint docs must include these responses in OpenAPI.
