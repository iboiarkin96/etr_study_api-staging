Adopt Docs as Code

Treat documentation with the same git workflow, review, and CI gates as application code, so it stays aligned with the runtime behaviour it describes.

Last reviewed

Context

Why this matters: Docs that live only in wikis or chat are easy to lose when people leave. They also drift from the code if nothing forces updates together. Docs as Code means we use the same tools as for the app (git, review, CI) for narrative content too: diagrams, API text, and guides move with commits and releases.

As the API grows, partners need one place to learn behavior, operators need runbooks that match production, and the team needs onboarding that does not rely on word of mouth. Updates without checks fall behind the real service.

Decision

We use Docs as Code as a standard practice:

  1. Keep documentation in repository and version it with code.
  2. Treat OpenAPI contracts, error schemas, and examples as part of public API contract.
  3. Use command-driven generation/synchronization for docs artifacts.
  4. Enforce documentation checks in quality gates before deployment.

Scope

  • In scope: the conventions, rules, and artefacts named in the Decision section above.
  • Out of scope: any system, process, or surface not explicitly listed here.

Alternatives considered

The four alternatives below were considered before adopting Docs as Code. Each is named, evaluated, and explicitly rejected with the reason that drove the decision.

Wiki (Confluence / Notion / similar)

rejected

Why considered: familiar tooling, rich-text editing, no markup syntax to learn, search and access control built in.

Why rejected: documentation lives outside the repo, drift between code and docs is the default state, every page edit lacks the review discipline of a PR, and history of "why this was changed" is owned by the wiki's audit log instead of git. Two teams the author worked with previously ended up with stale wikis that nobody trusted within 18 months. The drift problem is the entire motivation for this ADR.

Static site generator (Hugo / Jekyll / Docusaurus)

rejected

Why considered: Markdown-first authoring, built-in theming, large ecosystem, common choice for developer docs.

Why rejected: adds a build step before pages render, requires a Node or Hugo toolchain on every contributor's machine, and ties the portal's visual identity to a third-party template that we'd then customise heavily. A raw HTML + light Make pipeline produces the same artefact with zero non-Python dependencies and one less layer to debug when something goes wrong. Considered Hugo + a custom theme; the customisation cost exceeded the build-step saving.

Markdown-only (no rendered HTML companions)

rejected

Why considered: Markdown is the lingua franca for developer docs, renders natively on GitHub, no rendering step needed.

Why rejected: Markdown cannot express the visual structures we rely on (chip rows, paired-card consequences, gradient "What to read next" cards, complex tables with sticky heads). Forcing every page into Markdown either loses these visual structures or routes around Markdown with inline HTML, defeating the simplicity argument. The compromise — author HTML, accept Markdown companions where convenient — preserves the rich kit-component vocabulary without forbidding Markdown.

Generate everything from code (pure pdoc / OpenAPI-only)

rejected

Why considered: "the code is the documentation" — generate the API reference from docstrings and the OpenAPI baseline, generate everything else from code comments. Zero hand-authored prose.

Why rejected: code-generated docs answer "what does this function do?" but never "why does the system exist?" or "how do I respond to this on-call page?". Tutorials, how-tos, and explanations cannot be generated from code — they describe intent, not behaviour. We use code-generation for the parts that can be (API reference under services/api/code-reference/, error catalogues under errors.html) and hand-author the rest.

Trade we accept by choosing Docs as Code. The hand-authored HTML + make docs-fix pipeline ties documentation to the contributor's git workflow — there is no «edit docs without opening a PR» path. We accept the higher up-front contributor cost (HTML authoring, mandatory make docs-check before commit, slower CI) because the cost of the rejected alternatives is paid every day they're in use, not just at adoption time: wikis drift silently, SSGs add a toolchain, Markdown-only loses kit components, code-generation cannot carry intent. Docs as Code makes the cost visible and lets the pipeline enforce it.

Consequences

Positive

wins
  • Documentation remains aligned with runtime behavior.
  • Changes are auditable in PR history.
  • Release process blocks stale docs and missing generated artifacts.

Trade-offs

costs

Three named tradeoffs accepted on purpose. Shape from Tradeoff disclosure: «we gain X, we lose Y, we accept it because Z».

  • Slower pre-release pipeline. We gain auditable proof that committed docs match generated artefacts. We lose ~30 s per make verify run for docs-fix + diff comparison. We accept it because the alternative — stale generated files merged into main — turns the search index into a liability within a sprint.
  • Authoring tax on every SWE. We gain documentation that ships in the same PR as the code it describes (no «I'll update the wiki later»). We lose the option of «docs are someone else's problem» as a contributor mindset. We accept it because the deliverable is the portal, not only the binary (ADR 0026); spec velocity is the load-bearing metric, not feature velocity.
  • HTML-first authoring vs Markdown ergonomics. We gain access to kit components (chip rows, paired-card consequences, tables with sticky heads) without inline-HTML workarounds. We lose Markdown's native rendering on GitHub for those pages and accept a slightly higher contributor learning curve. We accept it because the rich-visual vocabulary is what lets the portal carry decision documents at scale; Markdown companions are still supported where they fit.

Revisit — would we do this again?

Yes, verbatim — with two day-one additions. The four rejected alternatives have aged well: the «wiki drift» failure mode is now documented in three real postmortems (postmortem corpus) where contributors did try to short-circuit the gate, and the cost of fixing those drifts confirmed the pipeline-as-policy bet. Two things we would add on day one if starting today:

  • Make docs-check a pre-commit hook from day one, not just a pre-PR gate. We learned this in the 2026-05-07 search-index drift postmortem — pre-commit hooks alone do not catch autogen drift, and developers were committing without running make docs-check first. The hook would have saved the incident.
  • Commit to the substance contract from day one, not retrofit it. The first ~25 ADRs (this one included) had thin alternatives blocks for the first year; the five-gate substance check only landed in 2026-05. Starting with it as a template would have saved the audit-pass that produced this very revision.

Compatibility and migration

  • Backward-compatibility impact: none — pure governance / convention change with no wire-level effect for existing consumers.
  • Migration plan: apply the rules to new work; bring existing artefacts into compliance during normal touch-ups.
  • Rollback strategy: revert the merge PR; no data or schema changes to undo.

Implementation in this repository

Section entry points: index.html

Each documentation section under services/portal/ uses a single HTML entry file named index.html, so paths are consistent and predictable when browsing the tree in the editor or on Git hosting. Link to the section as …/section/index.html.

Authoring model

Narrative documentation under services/portal/internal/governance/adr/, services/portal/internal/team/roles/swe/, services/portal/internal/how-to/incidents/runbooks/, and related trees is authored as HTML and normalized by tools/docs/format_docs_html.py. Optional Markdown files (*.md) in those folders are supported: when present, tools/docs/render_docs_html.py writes companion *.html next to them. Markdown is not required for those sections.

Quality gate: docs-check

make docs-check records git diff HEAD, runs docs-fix, and compares the diff again. If anything changed, committed generated output was stale — run make docs-fix and commit. See Documentation pipeline.

Validation

  • Technical checks: CI lints, type-checks, and contract tests stay green on every PR.
  • Operational checks: the relevant runbook reflects this rule.
  • Documentation updates: README, CHANGELOG, and related ADRs cross-reference this decision.

Usage

Typical workflow:

make test
make docs-fix
          make release-check
