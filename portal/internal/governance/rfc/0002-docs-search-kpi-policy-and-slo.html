Goals and non-goals

This policy sets targets for docs search quality and what to do when numbers move out of range.

Goals

  • Clear KPIs for docs search on services/portal/**/*.html.
  • Warning and breach limits so on-call knows when to act.
  • Same review rhythm and responses across releases.

Non-goals

  • Semantic search, typo fixes, or multilingual ranking.
  • Replacing the technical detail in ADR 0033.

Background

Context and motivation are summarised in the Goals section above; deeper rationale lives in the related ADR(s) cross-linked from Metadata.

Design

The full technical specification is captured in the Implementation section below — split into sub-sections as needed when the spec grows.

Implementation

  1. Apply the Design as a single delivery slice; track in the backlog.
  2. Update related templates, runbooks, and CI gates to reflect the rule.

Validation

  • Local checks: the relevant make target stays green.
  • CI / automated checks: contract / lint / docs-check gates pass.
  • Operational checks: the matching dashboard or runbook reflects the new behaviour.

Rollout plan

  • Rollout phases: single PR — no staged rollout required.
  • Feature flags: none.
  • Rollback strategy: revert the merge PR.
  • Backward compatibility: no wire-level impact for existing consumers.

Canonical KPI definitions

  • Zero-result rate = count(search_query where results_count = 0) / count(search_query).
  • Query CTR = count(distinct query_id with at least one search_result_click) / count(search_query).
  • TTFS (time-to-first-success) = distribution of first search_success.time_to_success_ms per session (p50/p75).

Data source: GET /internal/telemetry/docs-search/metrics?window_minutes=<N>.

Targets (rolling 7-day window)

KPI Target Warning threshold Breach threshold
Zero-result rate ≤ 12% > 12% and ≤ 18% > 18%
Query CTR ≥ 45% < 45% and ≥ 35% < 35%
TTFS p50 ≤ 25s > 25s and ≤ 40s > 40s
TTFS p75 ≤ 60s > 60s and ≤ 90s > 90s

KPI definitions are in § Canonical KPI definitions. Warning threshold = investigate if the KPI stays here for 2+ consecutive days. Breach threshold = escalate; open backlog work if any KPI is here for 24h (see § Response playbook).

Operating cadence

  • Weekly: review KPIs for the last 7 and 30 days.
  • Before big doc releases: confirm no KPI is in breach.
  • Monthly: add a short trend note to services/portal/CHANGELOG.md or an audit page when numbers move a lot.

Response playbook

Warning (one KPI in warning for 2+ days)

  1. Look at top zero-result queries and low click-through queries.
  2. Rebuild the Pagefind bundle: make docs-fix (or python3 tools/docs/build_pagefind_index.py for the bundle only).
  3. Check recent content or nav changes that could hurt ranking.

Breach (any KPI in breach for 24h)

  1. Open a backlog item and assign an owner to fix within 48h.
  2. Run the checks in ADR 0033 (local validation + telemetry).
  3. Fix quickly: rebuild index, roll back ranking changes, or fix nav/content.
  4. If users see clearly worse search, add a short runbook or changelog note.

Runbook links

  • ADR 0033 — Pagefind troubleshooting
  • Runbook: Logging failing (telemetry/log signal checks)
  • Runbook: Quality-check failing (CI and docs pipeline gates)

Change management

  • Change KPI thresholds only after at least 30 days of data and a short written reason.
  • Update this RFC and mention the change in services/portal/CHANGELOG.md.
  • Update an ADR only if the architecture changes — not for small threshold tweaks.
