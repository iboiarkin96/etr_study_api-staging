Public vs internal documentation portal IA

The shipped information architecture for the whole services/portal/ portal — public Diátaxis layer for external integrators, internal Diátaxis layer wrapping a Backstage-shaped service catalog for the team — grounded in four industry frameworks and delivered across six phases between 2026-05-08 and 2026-06-06.

Goals and non-goals

This RFC defines a single, defensible information architecture for the whole services/portal/ portal — splitting it cleanly into a public layer for external API consumers and an internal layer for the team — and grounds the choice in the four industry frameworks that have proven the pattern at scale: Diátaxis (public), the Backstage Internal Developer Platform (internal), the Google SRE Book chapter spine (operational hub), and the GitLab handbook single-portal-with-IA-boundary pattern (audience policy). It supersedes the audience-mixed status quo and amends one bullet in ADR 0028 so its "governance at repo root" rule reads as the more general "outside services/".

Goals

  • Adopt Diátaxis as the public-layer organising principle (tutorials / how-to / reference / explanation).
  • Adopt the Backstage Software Catalog + TechDocs per-service spine as the internal-layer organising principle.
  • Adopt the Google SRE Book chapter spine for the internal operational hub (SLOs → monitoring → on-call → incident response → postmortems).
  • Adopt the GitLab handbook single-portal pattern: audience boundary is an IA boundary, not an auth boundary.
  • Consolidate every team-facing page under services/portal/internal/; leave only the public router, release CHANGELOG, generated diagrams, and CSS/JS at services/portal/ root.
  • Introduce two HTML-attribute conventions on every page: data-portal (on <html>) and data-page-type (on <body>).
  • Provide a phased migration (A→F) where each phase ships independently and leaves make verify green.

Non-goals

  • Physical move to services/portal/. That is ADR 0028 Phase 1 — separate backlog item. The IA chosen here is forward-compatible.
  • Filling new pages with full prose. Phases A–F deliver skeletons, indexes, and 1-paragraph stubs with cross-links. Body content for each new hub follows in subsequent backlog items.
  • Auth gating. The public/internal split is structural and labelled, not access-controlled. For a solo monorepo, this is the GitLab-handbook posture: write nothing secret in any layer.
  • Backstage runtime install. We borrow the Software Catalog and TechDocs naming and structuring discipline; we do not run the Backstage application.
  • Visual styling. Existing portal hero / internal-layout CSS is reused. Public pages get no new CSS class; only the existing class="internal-layout" continues to gate the internal sidebar.

Background

Today's services/portal/ tree is audience-mixed and framework-less. Concrete failures:

  • The Swagger sandbox at services/portal/public/reference/api/index.html — the only artifact an external integrator needs — is mislabeled Internal · API Reference in its hero. The page that should be the front door for integrators is hidden behind an internal label.
  • The portal landing at services/portal/index.html mixes three audiences in one hero (external integrators, internal team, "people on this project"). A first-time visitor cannot tell what is for them.
  • There is no data-audience attribute, no public/internal CSS split, no nav branching. class="internal-layout" exists but is a layout flag (gates the sidebar), not an audience flag, and only services/portal/internal/** uses it.
  • ADR 0025 + ADR 0026 already carved a partial split for the API surface (OpenAPI external, services/portal/internal/api/ internal). The whole portal needs the same discipline, grounded in industry frameworks rather than invented locally.
  • Internal operational docs (logging, metrics, observability, debugging) have ADR-level rationale (ADR 0009, ADR 0011, ADR 0023) but no narrative hubs — readers cannot find the "how do we observe production?" page because it does not exist as a coherent surface.
  • Engineering practice docs are scattered: services/portal/internal/handbook/howto/, services/portal/internal/handbook/developer/, services/portal/internal/handbook/qa/ sit at repo-services/portal/ root next to services/portal/internal/, even though they are all team-facing. The split is "by historical accident", not by audience or function.

ADR 0028 (Proposed 2026-05-07) ratifies a destination of services/{api,portal,monitoring} but stages implementation across four phases. Before the folder split, per-service Dockerfiles, root compose, and CI matrix phases ship, the documentation portal itself needs a coherent IA — otherwise services/portal/ inherits today's mixed-audience layout and the architectural separation is cosmetic. This RFC fills that gap.

Design

The design hinges on three audience layers (Public Diátaxis, Internal handbook, Internal service catalog) each owning a distinct URL prefix, with Backstage descriptors driving entity cards and a single nav-tree per portal. Full design narrative — folder layout, URL contract, shared shell tokens — is captured across the six phases below in Implementation.

Implementation — all six phases shipped

Six phases, all shipped between 2026-05-08 (Phase A) and 2026-06-06 (acceptance). Phases A–B touched tooling and added new content; Phases C–F moved content. The folder-move phases (D, E, F) were paired with one-shot path-rewrite scripts that were deleted after use. Two phases diverged from the proposed shape — see §6 «As-built variances» below.

Validation

Local checks

  • make verify after every phase — green.
  • make docs-fix rebuilds the Pagefind bundle at services/frontend/portal/pagefind/; after Phase B confirm tools/docs/check_pagefind_visibility.py reports zero leaks between visibility:internal and visibility:public, and a manual search for a public-only and internal-only term in the topbar returns the expected scoped results.
  • tools/docs/check_adr_rfc_links.py passes after Phase F (script paths updated).
  • Manual smoke: python -m http.server from repo root; visit http://localhost:8000/docs/ — router renders, both CTAs work.

CI / automated checks

  • make verify in CI — same gates as local.
  • Pre-commit hooks updated for new file patterns (Phase B for openapi; Phase D for runbooks; Phase E for handbook; Phase F for governance).

Operational checks

  • Search UI — audience filter pill returns disjoint results for Public vs Internal after Phase B.
  • End-to-end after Phase F: integrator at services/portal/index.html reaches Swagger in ≤ 3 clicks, never touches internal; teammate reaches the SRE logging hub in ≤ 3 clicks.

Rollout plan

  • Rollout phases: A → B → C → D → E → F, one PR per phase. Phase A is text-only; Phases B–F each combine a content scaffold with a path migration scoped to one subtree.
  • Feature flags: none. The IA is structural; runtime behaviour does not change.
  • Rollback strategy: each phase is reversible by git revert. Phases C–F use one-shot migration scripts that are deleted after the migration; the underlying file moves are tracked by git.
  • Backward compatibility: the only consumer-facing path change is services/portal/public/reference/api/openapi-baseline.json → services/portal/public/reference/api/openapi-baseline.json in Phase B. ADR 0007 governance applies; downstream consumers of GHCR images per ADR 0021 are unaffected because the baseline is an artifact path, not a runtime endpoint.

Change management

Update this RFC when the audience boundary, framework choices, or per-service spine changes. Open a new RFC when a fundamentally different IA is needed (e.g., multi-portal split with auth gating, switching from Diátaxis to a product-shaped public layer). Owner after merge: docs maintainer (currently Ivan Boyarkin).

  • Update this RFC when: a new audience appears (e.g., partner integrators with different visibility); a new internal layer is added (e.g., data/ for analytics docs); per-service spine adds or drops a child.
  • Open a new RFC when: physical move to services/portal/ (ADR 0028 Phase 1) makes the path layout obsolete; auth gating becomes a requirement; a separate consumer help-centre is split off from this portal.
