RFC · 0006 Proposed Program: Multi-service IA

Portal IA scalability — three layers, one URL per page

The portal already feels crowded at ~250 pages and will not survive 1000 in its current flat shape. This RFC lays two new layers above the entity catalog from RFC 0005 — audience portals (who the reader is) and domain hubs (what concern the work touches) — so every page still lives at one URL but can be reached from three different doors. Closes gap §9 from the 2026-05-30 Architect's-Bar audit.

Why this RFC exists

The audit's verdict was specific: «работает для ~250 страниц; что if 1000+? Нужна policy: какие dimension we slice on, какие top-level buckets разрешены, как handle cross-cutting concerns». The three failure modes below are already visible at today's scale — they will not get better on their own, and the cost of fixing them rises super-linearly with page count.

  1. Flat quadrant
    tutorials/

    A quadrant becomes a list nobody can scan.

    Today
    10 sub-folders by role, ~25 tutorials in one tree.
    At 1000
    No one browses 100 tutorials in a sidebar — they search. Search has no facets either.
  2. Stranded
    team/roles/sa/

    Role guides live where a reader doesn't look.

    Today
    150 SA practice pages under «team» — a folder that sounds like a people directory.
    At 1000
    Readers looking for «SA practices» land on a member's profile first.
  3. Duplicated
    auth · obs · versioning

    Cross-cutting concerns scatter across folders.

    Today
    The same concern lives in ADRs, Foundations, per-service how-tos, and Practices.
    At 1000
    No domain owner; no single hub a reader can land on; drift accelerates.

The target picture — three orthogonal layers

Each page belongs to exactly one address and is reachable from three different entry points. The layers do not contain each other in a folder sense — they contain each other in a reading sense: a reader arriving from the audience layer narrows by domain, narrows again by entity, and lands on the page. Pages don't move when audiences or domains are added or removed; only the cross-link tables regenerate.

L1

Audience portals — who the reader is

filter by data-audience

8 portals at launch — sa, architect, dev, sre, qa, pm, manager, security. Each one a landing + onboarding journey (Day-1 / Week-1 / Month-1) + curated reading list + audience-scoped sidebar slice. A reader picks a hat and sees a curated portal instead of the firehose.

L2

Domain hubs — what concern the work touches

filter by data-domain

7 hubs at launch — api, data, observability, security, governance, delivery, platform. Each one owns an ADR-index slice, a Foundations-page slice, a Practices slice, and has a single owner team. Cross-cutting work that today scatters across folders gets one home per concern.

L3

Entity catalog — which thing the page describes

per RFC 0005 — unchanged

Five entity types — System / Component / Library / API / Resource — each with its own spine matrix. This is the layer where pages physically live. The catalog grows from 4 to 50+ entities without structural rework; this RFC does not touch the catalog model.

One page · three doors

Concrete example. Take a real runbook: p95-latency-spike.html. It lives at one URL inside the api Component's catalog spine. After this RFC, the same page surfaces on three landings — and no one has to maintain three copies.

  1. L1 audience
    /sre/

    SRE on-call lands on their portal.

    Door
    Day-1 checklist → «runbooks you'll touch».
    Why
    Page carries data-audience="sre".
  2. L2 domain
    /domains/observability/

    Domain owner browses the hub.

    Door
    «Runbooks» slice on the hub landing.
    Why
    Page carries data-domain="observability".
  3. L3 entity
    /catalog/components/api/

    Service owner opens its spine.

    Door
    Runbooks quadrant of the api Component.
    Why
    This is the canonical home — page lives here.

The canonical URL is /catalog/components/api/runbooks/p95-latency-spike.html. The other two landings render cards that link to it. The nav-tree generator and the curated-reading-list generator consume the page's facets to decide where the card appears.

A reader's first 30 seconds

One scenario, two timelines. New systems analyst, Day 1, looking for «how do I write a domain model».

Today

6 clicks

Portal home has no SA-specific door. Sidebar shows 8 top-level folders. Guess «tutorials» — open tutorials/sa/ — domain-model isn't there. Try «how-to» — nothing for SA. Try team/roles/sa — 150 practice pages, hard to scan. Eventually land on handbook/sa/domain-model.html.

Target

2 clicks

Portal home → pick «I'm an SA» → open /sa/. The SA landing has a «Day-1 checklist» with «Write your first domain model» as a top item. One more click and the reader is on the page. For the reader who already knows the term: search «domain model» with the audience facet «SA» — one click.

Migration — three phases

Each phase is independently shippable. Each step is preceded by a one-line «what I will do» note and followed by a one-line «what was done» note in the PR description. No URL ever 404s — the redirect contract (next section) is enforced by CI.

  1. P1 6 weeks
    additive

    Nothing moves. Pure additive.

    Ships
    IA-scalability policy page, IA-vocabulary page, facets hook (warn), redirect hook, SA + Architect pilot portals.
    Risk
    Zero. Revertable by git revert.
    404s
    0.
  2. P2 3 months
    moves + redirects

    Move ~250 pages. Stubs everywhere.

    Ships
    Folder moves along L1/L2 split; facets hook promoted to error; search facets; audience-scoped sidebar.
    Risk
    href churn — mitigated by check_redirect_consistency.py.
    404s
    0. Stubs live ≥ 12 months.
  3. P3 6 months
    federation

    A hub forks into a sub-portal.

    Ships
    Hub scorecards, stale-detection digests, first federation candidate when a hub crosses ~150 pages.
    Risk
    One-way door — reversal needs a follow-up RFC.
    404s
    0. Sub-portal shares URL space with the shell.

Redirect contract — zero broken links

Every move in Phase 2 follows the same five-step contract. CI rejects any move that skips a step. External readers, bookmarks, Slack pins, and ADR cross-references continue to work without modification. 

1.  Create new page at target path           (git mv when content is unchanged)
2.  Replace old path with a redirect stub    (0-second meta-refresh + canonical link)
3.  Append row to redirects.html             (old → new · date · RFC)
4.  Rewrite in-repo links to old path        (caught by check_redirect_consistency.py)
5.  Stub stays ≥ 12 months                   (removal recorded in redirects.html)

The move-log at handbook/redirects.html becomes the single audit trail for «where did that page go».

How we'll know it's working

Four numbers, surfaced on every hub's hero scorecard at Phase 3 Step 3.2. None of them is new instrumentation — they all derive from frontmatter facets, git mtime, and the existing search-telemetry endpoint. The targets are the falsifiable claims of this RFC: if any of them drifts past its threshold for two consecutive months, the RFC is failing and the response is a scope adjustment, not a tighter target.

SignalTargetHow we measureOut-of-bounds action
Discoverability
reader		 Median ≤ 3 clicks from portal home to the page the reader needed (today: ~5–6 for audience-scoped queries). Existing search-telemetry endpoint + landing-page click logs grouped by data-audience. Audit the audience portal's curated reading list; promote the missed page into Day-1 / Week-1.
Freshness
content		 Median last-reviewed age per hub ≤ 180 days. Git mtime per file, joined with frontmatter data-domain; rolled up per hub. Weekly digest issue auto-filed for the hub owner with the stalest five pages.
Concentration
structure		 ≤ 150 pages per hub before federation triggers. Page count grouped by data-domain + data-audience. Fork the hub into a sub-portal per the federation contract — this is a planned event, not a regression.
Integrity
links		 Zero broken links across the portal. Existing check-asset-refs + new check_redirect_consistency.py in make docs-check. Pre-commit gate blocks the PR; no merge until 0.

A.5 — Open questions to resolve before Phase 2

  1. Audience-portal naming: sa/ (proposed) vs audience/sa/ — URL length wins; sa/.
  2. Domain-hub naming: domains/api/ prefix is mandatory — collision with the api Component.
  3. QA placement: audience portal only; testing patterns stay in practices/ per RFC 0005.
  4. SA vs Architect overlap: two portals; SA owns requirements/domain/process/NFR, Architect owns C4/ADR-curation.
  5. Pilot audience: SA first (richest source), SRE second.
  6. Federation threshold: 150 pages (lower triggers premature federation).
  7. Search facets UI: filter-chips (reuse RFC-index toolbar component).
  8. Redirect stub lifetime: 12 months; older becomes noise.
