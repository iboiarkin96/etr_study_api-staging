Glossary — acronyms & domain terms

One canonical index for every acronym and domain term used across the portal. Link here from your pages instead of expanding inline — keeps prose tight and the vocabulary auditable.

ABCD EFGH IKLM NOPQ RSTU VWY
A 4 terms
ADR
Architecture Decision Record — a single-page record of one accepted technical decision. Captures Context · Decision · Consequences · Alternatives. See: ADR index
API
Application Programming Interface — the HTTP / gRPC / AsyncAPI surface a service exposes. Always written in upper case. See: API service
AsyncAPI
Asynchronous API spec — the OpenAPI equivalent for event-driven contracts. Camel-cased; never asyncapi.
a11y
Accessibility — numeronym (a + 11 letters + y). Refers to the practice and the audit checks (axe, Lighthouse, manual).
B2 terms
BPMN
Business Process Model and Notation — diagram standard for cross-functional business processes. Used in SA spine for process flow.
BRD
Business Requirements Document — pre-implementation artefact capturing what is being built and why, before technical design.
C3 terms
C4
Context · Containers · Components · Code — Simon Brown's 4-level architecture diagram model. We render L1–L3 in PlantUML.
CI/CD
Continuous Integration / Continuous Delivery — automated build, test, lint, and deploy pipeline. Docs CI runs link / a11y / h1 / acronym gates.
CRUD
Create · Read · Update · Delete — the four canonical persistence operations exposed by most REST resources.
D4 terms
DDD
Domain-Driven Design — strategic (bounded contexts, ubiquitous language) and tactical (aggregates, entities, value objects) modelling.
Diátaxis
Documentation framework: Tutorial · How-to · Explanation · Reference. Every page belongs to exactly one quadrant. Principles.
DoD
Definition of Done — checklist that a slice / endpoint / page must satisfy before it counts as complete.
DX
SWE Experience — how good or bad it feels to author and use the system. Subject of regular audits under governance/audits/.
E2 terms
EM
Engineering Manager — one of the six canonical roles. Owns team topology, roadmap, RACI.
Error budget
The amount of unreliability an SLO permits per window. When exhausted, feature work pauses for reliability work.
F1 term
Fitness function
An automated test that enforces an architectural quality (e.g. layering, dependency direction, latency budget). Failing fitness fails CI.
G2 terms
gRPC
RPC framework built on HTTP/2 + Protocol Buffers. Note the casing — lowercase g, uppercase RPC.
GraphQL
Query language for APIs with a typed schema. Not used here yet; mentioned only in alternatives sections of ADRs.
H1 term
Hex architecture
Hexagonal / ports-and-adapters architecture. Application core depends on abstractions; adapters depend on the core.
I2 terms
IA
Information Architecture — how content is organised, named, and linked. Subject of the current restructure.
IaC
Infrastructure as Code — declarative configuration (Terraform / Pulumi / OpenTofu) stored alongside application code.
K1 term
KPI
Key Performance Indicator — a metric that maps to a business outcome. Distinct from an SLI (technical) or OKR (goal).
L1 term
Lede
The opening paragraph of a page that promises what the page delivers. Required by the style guide on every authored page.
M1 term
MTTR
Mean Time To Recovery — average time from incident detection to service restoration. One of the DORA metrics.
N1 term
NFR
Non-Functional Requirement — qualities the system must have (latency, availability, security, accessibility). Distinct from functional requirements.
O2 terms
OpenAPI
Specification for describing HTTP APIs. Camel-cased: OpenAPI, never openapi. We use 3.1.
OKR
Objectives and Key Results — goal-setting framework. Objectives are qualitative; key results are measurable.
P3 terms
PlantUML
The only diagram syntax used here. Never Mermaid. ADR 0020.
PoC
Proof of Concept — throwaway implementation built to de-risk a decision. Lives under governance/rfc/ or a spike branch.
Postmortem
Blameless write-up of an incident. Timeline · Root cause · Contributing factors · Action items.
Q1 term
QA
Quality Assurance — the role, the practices, and the artefacts (checklists, test plans, audits). Always upper case.
R3 terms
RACI
Responsible · Accountable · Consulted · Informed — responsibility-assignment matrix. We publish one per SDLC stage.
REST
Representational State Transfer — the resource-oriented HTTP API style we use. Always upper case.
RFC
Request for Comments — proposal that needs review before becoming an ADR. Lives under governance/rfc/.
S7 terms
SA
Systems Analyst — one of the six roles. Owns the spine matrix, BRD, NFRs, BPMN, glossary, and the authoring contract.
SDLC
Software Development Life Cycle — the discovery → design → build → ship → operate → retire loop.
SLA
Service-Level Agreement — externally promised SLO with a penalty attached. SLA = SLO + money.
SLI
Service-Level Indicator — a raw measurement (e.g. p95 latency, success rate).
SLO
Service-Level Objective — target value for an SLI over a window. SLO = SLI ≥ target.
SPOF
Single Point Of Failure — a component whose failure brings down the whole system. ADRs and architecture reviews call out SPOFs explicitly.
SRE
Site Reliability Engineering — one of the six roles. Always upper case.
T2 terms
TDD
Test-Driven Development — red / green / refactor loop. Tests written before production code.
TF-IDF
Term Frequency · Inverse Document Frequency — ranking formula used by the portal search index.
U2 terms
UI Kit
The portal's component & token library. Lives under services/frontend/portal/assets_v2/ui-kit/. Currently v3.
UML
Unified Modeling Language — modelling syntax. We use the PlantUML dialect; always upper case.
V1 term
VPC
Virtual Private Cloud — isolated cloud network segment. Mentioned in deployment docs, not yet provisioned here.
W1 term
WCAG
Web Content Accessibility Guidelines — the canonical a11y spec. We target WCAG 2.2 AA.
Y1 term
YAGNI
You Aren't Gonna Need It — XP heuristic. Don't build for hypothetical future requirements.
