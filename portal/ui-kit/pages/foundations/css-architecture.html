Foundations

CSS architecture

BEM naming, a four-layer token cascade, one shell. No magic numbers, no portal forks, no per-page CSS.

The four-layer cascade

Every rule in the kit reaches the page through one of four layers, in this order. Later layers may only set tokens, never override component selectors.

LayerOwnsPath
1 · Tokens · baseradius, space, shadow, ease, layout dimsui-kit/tokens/_base.css
2 · Tokens · typographyfont-family, scale (--fs-100..--fs-900), weights, line-heightstokens/_typography.css
3 · Tokens · themecolour for light + darktokens/_theme-light.css, _theme-dark.css
4 · Tokens · portalaccent only — not componentstokens/_portal-internal.css, _portal-public.css

Anything below the four layers is a component. Components reference tokens, never raw values.

BEM naming

Every class follows .docs-block / .docs-block__element / .docs-block--modifier. The docs- prefix is the kit's namespace — anything without it is portal-level (.topbar, .docs-shell) or a token utility.

Example

.docs-card                      /* block */
.docs-card__title               /* element */
.docs-card__body                /* element */
.docs-card--elevated            /* modifier — shadow up */
.docs-card--ghost               /* modifier — transparent */
        .docs-card--interactive:hover   /* state — pseudo */

Three rules, no exceptions

  1. Tokens before values. If you typed 16px, ask which space token it is. If you typed #2563eb, ask which accent.
  2. One file per component. card.css owns .docs-card* and nothing else. Cross-component glue lives in layout.css.
  3. Portal layer only flips tokens. If you find yourself writing html[data-portal="public"] .docs-card { … }, you are forking the component. Stop, extract a token.

Entry bundles

Each portal has one CSS entry that imports the kit in canonical order:

/* runtime/internal/entry.css */
@import "../../ui-kit/tokens/_base.css";
@import "../../ui-kit/tokens/_typography.css";
@import "../../ui-kit/tokens/_theme-light.css";
@import "../../ui-kit/tokens/_theme-dark.css";
@import "../../ui-kit/tokens/_portal-internal.css";

@import "../../ui-kit/components/layout.css";
@import "../../ui-kit/components/prose.css";
/* …component imports… */

      @import "./internal-overrides.css";   /* last — portal-only tweaks */

Order matters: tokens first, layout second, components after, portal-overrides last. The overrides file should be a few lines, not a dumping ground.

Anti-patterns

  • Per-page stylesheets — if a page needs CSS that no other page needs, the page is wrong, not the kit.
  • Inline style="…" beyond demos — exception only inside .docs-example wrappers where the showcase explicitly demonstrates raw markup.
  • !important — never in component CSS. Acceptable only in prefers-reduced-motion guards.
  • Hard-coded breakpoints — the three tiers (≥1025 / 761–1024 / ≤760) are the contract. No 768, no 992, no 1280.
