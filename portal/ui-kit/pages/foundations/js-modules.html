Foundations

JS modules

Small, framework-free ES modules. One file per component, one mount function, idempotent. The runtime is a six-line boot() — no router, no virtual DOM, no build step.

The shape of a component module

Every JS component exposes a single mount<Name>(root = document). It finds its slot via [data-component="<name>"], attaches behaviour, and exits.

/* ui-kit/components/theme-toggle.js */
export function mountThemeToggle(root = document) {
const buttons = root.querySelectorAll('[data-component="theme-toggle"], .docs-theme-toggle');
buttons.forEach((btn) => {
if (btn.dataset.themeToggleBound === "true") return;   // idempotent
btn.dataset.themeToggleBound = "true";
btn.addEventListener("click", () => toggle());
});
      }

Discovery via data-component

Components find their hosts by attribute, not class. Class is for styling, attribute is for behaviour. This separation means CSS can render a static visual without booting JS, and JS can find its slot regardless of the visual.

[data-component="toc-fab"]toc-fab.js · mountTocFab()no

[data-component="drawer"]drawer.js · mountDrawer()yes — auto-created on docs-shell pages if absent, plus a topbar hamburger

The runtime entry

A portal has exactly one JS entry. It imports component modules and calls them in order. Order matters when components inject markup that later components find.

/* runtime/internal/entry.js */
import { mountThemeToggle } from "../../ui-kit/components/theme-toggle.js";
import { mountBreadcrumbs } from "../../ui-kit/components/breadcrumbs.js";
import { mountDrawer }      from "../../ui-kit/components/drawer.js";
import { mountSidebar }     from "../../ui-kit/components/sidebar.js";
import { mountTocFab }      from "../../ui-kit/components/toc-fab.js";
import { mountToc }         from "../../ui-kit/components/toc.js";
import { mountRocket }      from "../../ui-kit/components/rocket.js";
/* …others… */

function boot() {
mountThemeToggle();
mountBreadcrumbs();
mountDrawer();   // may auto-create a sidebar slot
mountSidebar();  // renders both the main aside and the drawer's copy
mountTocFab();
mountToc();
mountRocket();
}

if (document.readyState === "loading") {
document.addEventListener("DOMContentLoaded", boot);
} else {
boot();
      }

Module contract

  1. Idempotent. Calling mountX(root) twice on the same root is a no-op. Use a data-…-bound guard on bound elements.
  2. Root-scoped. Always accept root = document; never query document.querySelector directly. This is how the drawer copy of the sidebar gets mounted.
  3. Silent on missing slot. If querySelectorAll returns empty, return cleanly — never throw.
  4. No globals. All exports are named. No window.MyComponent stashes.
  5. No build step. Modules are loaded as <script type="module">. No bundler, no tree-shake, no transform.

Legacy bootstraps (syntax highlighting)

Two utilities still live under assets/ rather than the v3 tree: the zero-dep token highlighter (docs-syntax.js) and its theme (docs-syntax-theme.css). They predate the v3 entry; rewriting them in v3 is a future task. Until then, v3 wires them in two places:

  • ui-kit/components/syntax-highlight.js exports mountSyntaxHighlight(), which injects assets/docs-syntax.js once as a <script defer>. It is called from entry.js right after mountCode.
  • runtime/internal/entry.css @imports assets/docs-syntax-theme.css so the st-* token classes pick up their colours from the same light/dark variables that the rest of the kit uses.

Mount-once is guarded by a data-docs-syntax-bootstrap marker on the injected script tag. Don't add a second copy of the script in any page; the entry handles it.

Anti-patterns

  • Reaching into another component's DOM. If A needs B, B should expose an export, not a CSS selector.
  • setTimeout waiting for the DOM. Use DOMContentLoaded or MutationObserver.
  • Re-querying the same selector inside an event handler. Cache it on mount.
  • Async without await. If a mount returns a promise, the caller should be ready.
