Syntax highlight

Bootstrap shim. The v3 entry imports syntax-highlight.js, which injects the legacy zero-dependency tokenizer (assets/docs-syntax.js) once. The tokenizer auto-binds on DOMContentLoaded and emits st-* span classes; theme colours come from docs-syntax-theme.css (imported by entry.css). Inline <code> is left alone — only <pre><code> is highlighted.

Why a shim

The tokenizer is an IIFE that self-binds on load. v3's entry.js used to pull it in by accident via legacy docs-nav.js. When we dropped that legacy import, v3 pages lost their syntax colours overnight. syntax-highlight.js restores them without re-implementing the tokenizer inside the v3 tree — it simply document.head.appendChild()s the legacy script (once; guarded by data-docs-syntax-bootstrap="1").

Usage

Mark a code block with class="language-<lang>". Auto-detection works for most cases, but explicit is better for shared snippets.

<pre class="docs-code"><code class="language-python">
def greet(name: str) -> str:
    return f"Hello, {name}!"
      </code></pre>

After bootstrap the <pre> picks up data-lang="python" and inner tokens are wrapped in <span class="st-*">.

Supported languages

  • python
  • bash · sh · shell
  • json
  • yaml · yml
  • http — request/response lines, headers, status codes.
  • javascript · js

Detection: explicit class="language-…" / class="lang-…" wins. Otherwise the tokenizer sniffs the content (e.g. HTTP/ opener, --- YAML head, JSON brace, shebang).

Token classes

ClassTokenColor var
st-kw keyword (def, if, return…)--st-kw
st-str string literal--st-str
st-cmt comment--st-cmt
st-num number--st-num
st-key object / map key--st-key
st-bi builtin / type--st-bi
st-var variable ($VAR)--st-var
st-flagCLI flag (--flag)--st-flag
st-dec decorator--st-dec
st-op operator / punctuation--st-op

All token colours are CSS custom properties that track data-theme automatically — light + dark palettes live side-by-side in docs-syntax-theme.css.

Safety

  • HTML-escape first, tokenize after. The tokenizer escapes </>/& before any regex runs, so user-provided code blocks can't inject markup.
  • Never re-highlights. A bootstrapped block carries a sentinel and is skipped on subsequent passes.
  • Graceful failure. Parse errors are swallowed silently — the block stays as plain text, the page never crashes.
  • Inline code untouched. Only <pre><code> is processed; inline <code> keeps its monospace styling without token wrapping.

Files involved

  • ui-kit/components/syntax-highlight.js — the v3 bootstrap shim (23 LOC).
  • assets/docs-syntax.js — the actual tokenizer (legacy IIFE, ~250 LOC).
  • assets/docs-syntax-theme.css — colour variables for light/dark; imported by entry.css directly so themes work even before the JS bootstraps.
  • code.html — the .docs-code wrapper showcase (chrome, copy button).
