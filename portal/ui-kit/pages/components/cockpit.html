Cockpit

Page-template component. Operations cockpit: a KPI strip, a filter+view toolbar, and a swappable view container (table · timeline · by-profession · board). Used by the backlog cockpit (internal/governance/backlog/) and the ops-cockpit template. CSS lives in two files — cockpit.css (generic primitives) + backlog-cockpit.css (backlog extensions); the controller mounts on <section data-component="backlog-cockpit"> and orchestrates 3 view renderers + the multi-filter-chips state hash.

Live exemplars

  • internal/governance/backlog/index.html — the production cockpit (real tasks.json + sprints.json, all 3 views, sprint switcher).
  • ui-kit/pages/templates/ops-cockpit.html — the kit template (static markup; canonical anatomy for new cockpit pages).

Anatomy

  1. Cockpit header — title + lede (.backlog-cockpit__header / .backlog-cockpit__lede) and a deadline status pill.
  2. KPI strip — .cockpit-kpi-grid of .cockpit-kpi-card tiles (each sits on a .docs-card). Each tile: __hint (mono uppercase) · __value (tabular-nums) · __footer (delta + sparkline).
  3. Sprint strip — .backlog-sprint-strip with .backlog-sprint-chips rendered from sprints.json. Each chip carries aria-pressed for the active sprint and dates underneath.
  4. Filter bar — .backlog-filters with one .backlog-filters__row per facet (profession · status · priority). Each row uses <div data-component="multi-filter-chips">.
  5. View switcher — .docs-view-switcher with data-target="#backlog-views" and data-storage-key; one .docs-view-switcher__btn per view.
  6. Views container — .cockpit-views with data-view="…" driving which child .cockpit-view is visible. Children: .cockpit-view--table, .cockpit-view--timeline, .cockpit-view--by-profession, .cockpit-view--board.

CSS surface

cockpit.css — generic primitives

  • .cockpit-kpi-card + __header / __hint / __value / __footer — KPI tile inner layout (surface comes from .docs-card).
  • .cockpit-kpi-grid — responsive grid wrapper for KPI tiles.
  • .cockpit-toolbar, .cockpit-filter-row — page-level toolbar primitives.
  • .cockpit-views + .cockpit-view + .cockpit-view--{board,table,timeline} — view-switcher container; [data-view="…"] picks the active child.
  • .cockpit-card__header / __footer — sub-rows applied inside a .docs-card on the board view (status pill + tags).
  • .cockpit-timeline + __lane / __head / __body / __item — multi-lane sprint timeline. Current-sprint lane gets --current.

backlog-cockpit.css — backlog extensions

  • .backlog-cockpit__header / __lede — page header overrides (the deadline pill sits next to the lede).
  • .backlog-sprint-strip + __label · .backlog-sprint-chips · .backlog-sprint-chip + __row / __dates / .backlog-sprint-dot — sprint switcher.
  • .backlog-filters + __row / __label — per-facet filter row.
  • .cockpit-views[data-view="by-profession"] .cockpit-view--by-profession — extra view modifier added by the backlog (not in cockpit.css).
  • .backlog-board + __column / __column-head / __column-icon / __column-title / __column-count — by-profession board.

JS modules

  • backlog-cockpit.js — page controller. Mounts on [data-component="backlog-cockpit"]; fetches tasks.json + sprints.json; renders the sprint switcher and KPI strip; delegates view rendering to the three renderers below; listens for multifilterchange / viewchange window events from multi-filter-chips.js + view-switcher.js.
  • backlog-table.js — pure function renderBacklogTable(container, tasks, sprintsIndex); renders a sticky-head zebra table with status / priority pills.
  • backlog-timeline.js — pure function renderBacklogTimeline(container, tasks, sprintsIndex); renders the multi-lane sprint timeline.
  • backlog-profession-board.js — pure function renderBacklogProfessionBoard(container, tasks); renders the by-profession kanban board.

The three view renderers are pure — they take a container and the already-filtered data and rewrite the container's innerHTML. All filter / sort / fetch logic lives in backlog-cockpit.js.

Data shape

The controller expects two JSON sources referenced via data-tasks-src and data-sprints-src on the mount node. See tasks.json and sprints.json for the canonical shape (id, title, profession, status, priority, sprint, etc.). The deadline constant lives at the top of backlog-cockpit.js (DEADLINE_ISO).

Adding a new cockpit

  1. Copy the markup from ui-kit/pages/templates/ops-cockpit.html into the new page.
  2. Provide tasks.json + sprints.json next to the page and point data-tasks-src / data-sprints-src at them.
  3. Register any new view by adding a .docs-view-switcher__btn[data-view="…"] and a matching .cockpit-view--… child + a CSS rule in cockpit.css turning it on under .cockpit-views[data-view="…"].
  4. Write the renderer next to the others (backlog-{name}.js) and dispatch to it from backlog-cockpit.js.

Naming note

Class names use a split prefix on purpose: .cockpit-* for primitives that any cockpit reuses, .backlog-* for backlog-specific extensions. The template-annotator registry consumes cockpit-kpi-card and cockpit-timeline — keep those names stable. Page-template, not decomposed: the primitives only make sense composed together inside a cockpit.
