Multi filter chips

JS-only behaviour. Multiple filter-chip groups that cooperate over a shared URL hash — each group owns one key, all keys live in one query-string. The backlog cockpit uses three groups (profession, status, priority) plus a sprint switcher to drive table / timeline / by-profession views.

Mount

<div data-component="multi-filter-chips" data-state-key="profession" data-default="all" aria-label="Profession">
  <button class="docs-chip docs-chip--filter" data-value="all"        aria-pressed="true">All</button>
  <button class="docs-chip docs-chip--filter" data-value="qa"         aria-pressed="false">QA</button>
  <button class="docs-chip docs-chip--filter" data-value="dev"        aria-pressed="false">Dev</button>
</div>

<div data-component="multi-filter-chips" data-state-key="status" data-default="all" aria-label="Status">
  <button class="docs-chip docs-chip--filter" data-value="all"  aria-pressed="true">All</button>
  <button class="docs-chip docs-chip--filter" data-value="todo" aria-pressed="false">To do</button>
  <button class="docs-chip docs-chip--filter" data-value="done" aria-pressed="false">Done</button>
      </div>

Two groups, two keys. Picking «QA» + «To do» yields #profession=qa&status=todo. Defaults are written as absence — #profession=qa (status is at its default «all» so it's omitted).

Attributes

  • [data-component="multi-filter-chips"] — mount marker. Removed after mount; the group then carries role="group".
  • [data-state-key] — the hash key this group owns. Required, must be unique on the page.
  • [data-default] — the «absence» value (typically "all"). When the active chip equals this value, the key is deleted from the hash.
  • [data-value] on each chip — the value written to the hash for that chip.
  • [aria-label] on the group — falls back to data-state-key if absent.

Events

Two channels keep cooperating modules in sync:

  • Outgoing — multifilterchange on window with detail: { key, value } on every chip click. Listeners (e.g. backlog-cockpit.js) re-render their view.
  • Incoming — hashchange on window resets every group to the value in the hash. Lets browser back/forward work without a page reload.
window.addEventListener("multifilterchange", (e) => {
  const { key, value } = e.detail;

Keyboard

  • ← / → — move focus to the previous / next chip within the group and activate it.
  • Tab — moves between groups (each chip is a real <button>).
  • Enter / Space — activate the focused chip (native button behaviour).

State semantics

  • The URL hash is the single source of truth across all groups.
  • On load, each group reads its key from the hash; unknown values fall back to data-default.
  • Default values are written as absence — URLs without filters stay clean.
  • When the user picks a chip, the controller updates only that key in the hash (other keys preserved).
  • Initial activation is silent (persist: false, emit: false) so listeners don't fire on first paint.

When to use which

  • filter-chips — one axis, one container. The chips themselves toggle [data-filter-hidden] on items in data-target. No events.
  • multi-filter-chips — many cooperating axes; the chip groups are state-only (no built-in DOM filtering). The consumer listens for multifilterchange and renders accordingly.

Live exemplar
