Info button

Small circular i glyph that sits next to a heading. Pairs a short data-tooltip preview on hover/focus with a deeper explanation behind a modal on click — so the page itself stays uncluttered while the “why this exists” context is one tap away. The button itself in this lede opens its own explainer; try it.

Anatomy

  • Element — <button type="button"> with class docs-info-button. Always a real button (not an anchor) because the primary action is a JS modal open, not navigation.
  • Glyph — 14–16 px stroked SVG (viewBox="0 0 24 24", stroke-width="2", stroke-linecap="round"): circle + vertical line + dot. Identical across all instances.
  • Tooltip preview — data-tooltip="…" renders a one-sentence question or hint on hover and keyboard focus via the kit's tooltip directive.
  • Modal trigger — onclick="docsModal.open('<modal-id>')" opens a paired .docs-modal elsewhere in the document.
  • Label — aria-label matches the tooltip text so screen-reader users get the same hint.
  • Tone — single accent-tinted appearance (no data-tone variants). The component is informational; it doesn't carry severity.

In context

Two canonical placements ship today:

  • Hero H1 — service catalog landing (/services/portal/internal/services/index.html): opens a modal that explains the six-page spine.
  • Section H2 — UI Kit catalog entry (/services/portal/internal/services/ui-kit/index.html): opens a modal that explains why the entry is thin.

Both placements use the same class. The button hovers -0.18em above the heading baseline so it sits visually centred against the cap-height of the title type.

States

Three interaction states share the same accent palette via color-mix:

  • Idle — soft accent-tinted background, muted accent-tinted border, accent-tinted glyph.
  • Hover / focus-visible — background and border deepen, glyph snaps to var(--accent), button lifts a further -0.04em.
  • Focus-visible — additional 2 px accent outline at 2 px offset for keyboard users (independent of the hover lift).

Cursor is help (not pointer) to signal that the button discloses information rather than navigates.

Accessibility

  • Real button. Always render <button type="button"> — never a styled span — so keyboard activation (Enter/Space) and screen-reader semantics work out of the box.
  • Match label and tooltip. Keep aria-label and data-tooltip in sync; assistive tech and visual users should hear the same hint.
  • Phrase as a question. “What is the six-page spine?” reads better than “more info” and primes the modal contents.
  • Modal must focus its panel on open and trap focus until dismissed — handled by the kit's docsModal behaviour, but verify when adding new modals.
  • Reduced motion. The hover/focus translate is disabled under prefers-reduced-motion: reduce — the button still visually responds via colour, just without the lift.
  • Decorative SVG. Mark the glyph aria-hidden="true"; the accessible name comes from aria-label on the button.

Markup

Drop the button immediately inside the heading element. The class itself handles vertical alignment against the title type. 

<h2 id="catalog-spine">
Catalog spine
<button type="button"
class="docs-info-button"
aria-label="Why is this entry thin?"
data-tooltip="Why is this entry thin?"
onclick="docsModal.open('why-thin-modal')">
<svg width="14" height="14" viewBox="0 0 24 24" fill="none"
stroke="currentColor" stroke-width="2"
stroke-linecap="round" stroke-linejoin="round" aria-hidden="true">
<circle cx="12" cy="12" r="10"/>
<line x1="12" y1="16" x2="12" y2="12"/>
<line x1="12" y1="8" x2="12.01" y2="8"/>
</svg>
</button>
</h2>

<!-- Elsewhere in the document (typically just before </main>): -->
<section class="docs-modal" id="why-thin-modal" hidden>
…
      </section>

Pick width="14" for H2 placements and width="16" for H1 placements so the glyph scales with the title.

Content guidelines

  • Reserve for non-trivial “why this exists”. If the answer fits in a sentence right next to the heading, just write the sentence. Reach for the info button when the explanation is a paragraph or three.
  • Tooltip is a teaser, not the answer. One short interrogative line: “What is the six-page spine?” The full answer lives in the modal.
  • One info button per heading. If a section needs two explainers, that's two separate sections.
  • Modal IDs end in -modal. Convention used by every existing pairing (spine-reference-modal, why-thin-modal); makes them grep-able.
  • Don't use it as a help icon for forms. The component is tuned for heading context. For inline field hints, use the tooltip directive directly on the label.

What is the info button?

The info button is a deliberately small affordance with a deliberately dual surface area: a quick tooltip preview, then a deep modal answer.

  • Hover or focus reveals a one-line preview via the kit's tooltip directive — enough to decide whether to commit to the full explanation.
  • Click or activate opens a modal carrying the deep answer — a paragraph, a list, sometimes a diagram.

The pattern keeps catalog and landing headings clean while preserving a single discoverable entry-point for “why does this entry look like this?” questions. It replaces inline H2 + paragraph blocks that interrupt the prose flow.

A note on the class name

The CSS class is docs-info-button — namespaced under home-hero because that's where the component first shipped (the “What is the six-page spine?” trigger in the internal services catalog hero).

The styling is fully generic: no parent-selector dependencies, no assumptions about surrounding markup. You can drop the button next to any <h1>, <h2>, or <h3> and it will sit correctly against the baseline.

A future rename to a non-namespaced class (docs-info-btn?) is on the table — until then, treat the existing class as the canonical hook and follow the placements in this showcase.
