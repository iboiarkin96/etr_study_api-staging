Design canvas card

Hero "design stage" visual — a warm aurora canvas with drifting orb mesh, big stencil UI KIT type, a 3D-tilted frosted-glass specimen card with holographic conic shimmer and cursor-lit spotlight, plus four floating token chips. Lives in the right slot of landing-portal as the visual counterpart to the terminal-card on the public landing — same slot, different story (kit specimen vs. API call).

Mouse parallax writes --dcc-mx / --dcc-my on the host; CSS turns those into rotateX/Y for the stage, plus z-translation for the glass card and individual translation/rotation for every floating chip. The stage is intentionally always-light (regardless of data-theme) so the aurora reads consistently.

Example

Hover and move the cursor across the stage — every layer reacts to pointer position with a separate parallax amount.

Markup

One element, no inner markup. The component builds the eight-layer stack in JS, wires pointer parallax, and reveals on intersection.

<div data-component="design-canvas-card"></div>

No attributes — the card is a fixed hero composition. To customise palette, type sample, chip labels, or stage dimensions, override the --dcc-* custom properties (see Tokens) on the host.

Anatomy

Eight stacked layers from back to front. Every layer below the glass card is aria-hidden — only the card chrome is meaningful to assistive tech.

Pointer parallax

One module-level pointermove listener writes normalised [0..1] pointer coords to two custom properties on the host, batched through requestAnimationFrame:

host.style.setProperty("--dcc-mx", pendingX);
      host.style.setProperty("--dcc-my", pendingY);

The JS does nothing else with motion. All depth and tilt is CSS:

  • Stage — rotateX(calc((0.5 - var(--dcc-my)) * 14deg)) + rotateY(calc((var(--dcc-mx) - 0.5) * 16deg))
  • Card — sits at translateZ(40px) with a small x/y nudge for parallax-against-stage
  • Tokens — each chip uses a different z-offset (30–70 px) and its own sign on the x/y nudge, so corners react asymmetrically
  • Card highlight — radial spotlight at calc(var(--dcc-mx) * 100%) / calc(var(--dcc-my) * 100%), faded when the host isn't hovered

The listener resets to centre (0.5/0.5) on pointerleave and pointercancel, so the stage glides back to neutral when focus moves away.

States

  • Pre-intersection — .dcc__stage, .dcc__token, .dcc__bigtype, .dcc__hud, .dcc__foot all start at opacity: 0.
  • Entered viewport — IntersectionObserver (threshold 0.3) adds .is-visible; layers fade in on a staggered ladder (HUD/foot 60 ms → bigtype 120 ms → stage 260 ms → tokens 480/540/600/660 ms).
  • Hover — palette swatch --3 grows to flex: 1.35; CTA arrow nudges translateX(3px); cursor highlight fades to opacity: 0.85.
  • Reduced motion — all keyframes and transitions are killed; the stage renders flat (no parallax tilt), tokens / bigtype / chrome become fully visible immediately, observer is skipped.

Uses

Currently mounted in two places — both as the right-hand visual of a portal hero:

  • UI Kit home (services/portal/ui-kit/index.html) — hero visual slot.
  • Landing · Portal template — canonical reference implementation for any future portal hero.

The mount is automatic — mountDesignCanvasCard() is wired into runtime/internal/entry.js. No per-page boilerplate.

When to use this vs. terminal-card

Both fit the same hero slot but carry different narratives. Pick by audience:

  • design-canvas-card — for landing surfaces that sell the system: design tokens, components, the UI Kit itself. Reads as "specimen on a workbench".
  • terminal-card — for landings that sell the API: live curl call, response, "Try in OpenAPI Explorer" CTA. Reads as "make a request".

Don't render both in the same hero. They occupy the same visual slot and compete for the same attention.

A11y

  • The card is decorative — all inner layers are aria-hidden. The glass card itself carries aria-label="Component preview" so AT users get one descriptive landmark instead of a dozen meaningless chips.
  • No focusable controls; nothing in the hero is keyboard-navigable. The "Inspect" CTA is a <span>, not a link.
  • prefers-reduced-motion: reduce kills every animation, transition, and 3D tilt; the card renders flat with all layers fully visible.
  • user-select: none + cursor: default on the host — the aurora isn't selectable; nothing inside is meant to be copied.
  • Backdrop-filter fallback: browsers without backdrop-filter support get a solid warm-white card fill so the frosted look degrades gracefully.
