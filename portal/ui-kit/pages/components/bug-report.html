Bug report

Single-purpose component that lets readers open a prefilled GitHub issue for the page they are reading. The trigger lives as a 36 × 36 topbar action next to the theme toggle; pages can drop additional inline triggers anywhere in prose. Every trigger opens the same shared modal at document.body.

  • Trigger — auto-injected icon button before .docs-theme-toggle in every .topbar__actions. Antennae idle-wiggle on their own; hover / focus snaps them outward.
  • Modal — single shared instance. Feedback-type select, free-text area (10+ chars), submit opens GitHub in a new tab with the page URL, type, and description prefilled.
  • Clipboard — the description is copied to the clipboard as a backup before the new tab opens, so a navigation glitch never loses what the reader typed.

Live button

The button in the topbar is the live component. The demos below are wired to the same modal — click any trigger. Two modes are shown: the auto-injected icon button (left), identical to the one in the topbar, and an opt-in inline text trigger (right) you can drop inside prose.

Tip: the bug icon in the topbar also has a tooltip — hover or Tab-focus it to see the new tooltip component in action.

Hover or focus the icon button to see the antennae snap outwards; otherwise they idle-wiggle on their own (disabled under prefers-reduced-motion).

Anatomy

The icon button is a 36 × 36 px rounded square (10 px radius) that matches the geometry of .docs-theme-toggle so the two sit shoulder-to-shoulder. Inside it lives a 22 × 22 px SVG bug glyph with three semantic groups: the two antennae and the body — each animated separately.

Why two coprime-ish loop durations. Real insects don't flick both antennae at once. The left antenna runs on a 7.3 s loop and the right on 9.1 s; their lowest common multiple is ~66 s, so the two animations rarely return to the same combined phase during a normal page visit. Each antenna spends roughly 95 % of its loop at 0° with two short flicks per cycle, so the bug looks alert without ever appearing busy or twitchy.

Generated markup

The runtime injects the button into every .topbar__actions automatically — you don't need to write the markup. If you want a second trigger somewhere (e.g. an inline link in long docs), drop any element with data-component="bug-report" or data-bug-report-open; the runtime upgrades it on mount. Empty triggers become the icon button; triggers with text become the inline opt-in style and keep their content.

<!-- Auto-injected into the topbar -->
<button class="docs-bug-report-btn" type="button"
        aria-label="Report a bug on this page">
  <svg class="docs-bug-report-btn__icon" …>…</svg>
</button>

<!-- Opt-in inline trigger (kept as a text button) -->
      <button data-component="bug-report" type="button">Report an issue with this page</button>

Modal contract

One modal instance is created lazily on the first open and reused for every subsequent open. There is no second copy; opening while already open is a no-op.

  1. Shape. .docs-bug-report-modal covers the viewport with position: fixed; inset: 0; z-index: 95. The panel (.docs-bug-report-modal__panel) is centred, min(480px, 100%) wide, max-height: 86vh. Backdrop tints to rgba(0,0,0,0.5) with a 4 px backdrop blur.
  2. Entrance. translateY(12px) scale(0.98) → 0 / 1, 220 ms var(--ease-spring). Disabled under prefers-reduced-motion.
  3. Header. Heading "Report a bug" and a close button (.docs-bug-report-modal__close, tooltip "Close this form (Esc)") aligned to the right.
  4. Fields.
    • <select id="docs-bug-report-type"> — six categories, default Other: Incorrect or outdated content, Missing explanation, Broken link or navigation, Visual / layout bug, Accessibility issue, Other.
    • <textarea id="docs-bug-report-text"> — required, minLength=10, 4 rows, placeholder example. Live status text below the form explains current state via aria-live="polite".
  5. Submission. On submit:
    1. Trim message; if length < 10 → status flips to --error, focus returns to the textarea, no navigation happens.
    2. Build the GitHub URL via buildIssueUrl(message, feedbackType) — see GitHub issue payload.
    3. Best-effort navigator.clipboard.writeText(message) as a backup; failure is silent (insecure contexts and file:// previews lack clipboard).
    4. Status flips to --success "Opening GitHub in a new tab…"; the URL opens via window.open(url, "_blank", "noopener,noreferrer").
    5. After 600 ms, status resets to "Thanks!", textarea clears, the modal closes.
  6. Close affordances. Close button, backdrop click, Esc (document-level keydown listener). Each restores focus to the trigger that opened the modal.
  7. Body lock. While the modal is open the runtime sets document.body.style.overflow = "hidden" to prevent background scroll bleed; it's cleared on close.

GitHub issue payload

buildIssueUrl() returns a deterministic https://github.com/iboiarkin96/study_bot/issues/new?… URL with these query parameters:

ParamValue
template docs_feedback.md — selects the matching GitHub Issue Form.
title [Docs feedback] <page path> (pathname, leading slashes stripped, or (home)).
labels docs-feedback (the constant LABELS, joined by comma).
body Markdown with five sections: ## Page, ## URL, ## Feedback type, ## What should be improved, ## Additional context. The first three are filled automatically; the fourth carries the user's textarea contents (or a placeholder if blank); the fifth is always a placeholder for screenshots / refs.

States

State Trigger Visual Motion
Idle Default once mountBugReport() has run. --card background, 1 px --line border, --muted glyph. Antennae loop independently — left 7.3 s, right 9.1 s; each ~95 % at rest with two short flicks per cycle.
Hover Pointer over the button. Border + glyph tint to --accent. Idle loops pause; antennae lock to ±12°; body translateY(-0.5px) scale(1.04).
Focus-visible Keyboard focus. 2 px accent outline at 2 px offset. Same as hover (antennae lock + body scale).
Active Pointer or key down. Whole button transform: scale(0.96). 140 ms var(--ease-out).
Reduced motion @media (prefers-reduced-motion: reduce). Static; hover still tints colours. Antennae loops + body transitions disabled. Modal entrance disabled.
Modal open Click or Enter/Space on any trigger. Body gains overflow:hidden; backdrop blurs; centred panel slides up + fades in. Spring entrance only; the topbar button keeps its idle wiggle underneath.

A11y

  • The icon button uses aria-label="Report a bug on this page"; the SVG is aria-hidden and focusable="false".
  • The modal is role="dialog" aria-modal="true", labelled by the heading (aria-labelledby="docs-bug-report-title").
  • Esc closes the modal (document-level keydown listener); backdrop click also closes.
  • Focus moves to the textarea on open (via requestAnimationFrame) and returns to the trigger on close.
  • Submission status uses aria-live="polite" so screen readers hear the prep / success / error transition.
  • Submitting a textarea shorter than 10 characters surfaces an error in the live region instead of navigating — the page never leaves on an invalid submit.
  • Antennae and body animations are disabled under prefers-reduced-motion: reduce; the modal entrance animation is also suppressed.

Do / Don't

DoDon't
Add extra triggers anywhere in prose via data-component="bug-report" or data-bug-report-open — the runtime wires them to the same shared modal. Mount a second modal instance manually. There is exactly one modal at document.body; duplicates re-bind listeners.
Verify the prefers-reduced-motion: reduce path on every visual change. Antennae loops, body scale, and the modal spring entrance must all stop under reduce. Add motion-only feedback that a click was registered. The modal opening is the confirmation; the antennae wiggle is ambient, not informational.
Keep the SVG strictly currentColor + stroke-width: 1.85 px. The glyph inherits the topbar palette so theme switches just work. Hardcode hex colours into the SVG or rely on fill. Stroke-only currentColor is the contract.
Let the runtime own the button end-to-end: mountBugReport() finds every .topbar__actions, injects a button before the theme toggle, and binds clicks. Page authors should never hand-author <button class="docs-bug-report-btn">. Pre-render the icon button or hand-author the SVG glyph. The static path is removed; baking the markup into pages duplicates a contract that must be kept in lock-step.
Update REPO, TEMPLATE, and LABELS at the top of bug-report.js in a single edit when retargeting the GitHub repository or template. Branch the runtime to support multiple repositories or templates simultaneously. The contract is one repo + one template per build.

Tokens used

  • --accent, --accent-contrast, --card, --bg, --bg-mute, --line, --line-soft
  • --text, --text-strong, --muted, --danger, --success
  • --radius-sm, --radius-lg
  • --shadow-lg, --ease-out, --ease-spring, --ease-in-out
  • --fs-200, --fs-300, --fs-500, --fw-semibold, --lh-snug
  • --space-2, --space-3, --space-4, --space-5

Configuration

The GitHub repository, issue template, labels, and feedback categories live as module-level constants at the top of bug-report.js — change them in one place to retarget every portal build.

const REPO     = "iboiarkin96/study_bot";
const TEMPLATE = "docs_feedback.md";
const LABELS   = ["docs-feedback"];

const FEEDBACK_TYPES = [
  "Incorrect or outdated content",
  "Missing explanation",
  "Broken link or navigation",
  "Visual / layout bug",
  "Accessibility issue",
  "Other",
      ];

The exported entry point is mountBugReport(root = document). It upgrades any author-placed [data-component="bug-report"] / [data-bug-report-open] triggers, then injects a fresh icon button before each .docs-theme-toggle (or appends to the end of .topbar__actions if no toggle is present).
