Compare grid

Side-by-side bad/good (or before/after) panels. Two-column on desktop, single-column on mobile. Each cell gets a tone-tinted background + matching label chip — danger on the «bad» side, success on the «good» side.

Example

Bad

«ETR storage rewrite for Q3»

  • No problem stated — what was wrong with the old storage?
  • No alternatives considered. Why this approach?
  • No consequence list. What breaks downstream?
Good

«0042 · Move card content to row-level storage»

  • Context — JSON blob limits queries, 30% of cards exceed 4 KB.
  • Alternatives — keep blob (rejected: query cost), per-field columns (rejected: schema churn).
  • Consequences — read path: +1 join; write path: 3 statements instead of 1; migration N PRs.

Anatomy

  • .pa-compare — grid wrapper, two columns at ≥760 px.
  • .pa-compare__cell — bordered panel; add --bad or --good modifier for the tone.
  • .pa-compare__label — small uppercase chip («Bad», «Anti-pattern», «Before», «Cost», …).
  • Inside each cell — any prose, list, code block. The cell's p/ul/li get tightened margins.

Markup

<div class="pa-compare">
  <div class="pa-compare__cell pa-compare__cell--bad">
    <span class="pa-compare__label">Bad</span>
    <p>…</p>
  </div>
