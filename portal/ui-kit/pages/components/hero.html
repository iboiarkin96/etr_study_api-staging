      </section>

States

  • Default — no interaction.
  • With scene-bg — wrap a .docs-scene-bg sibling for the radial-gradient backdrop.
  • Reduced motion — rocket slot animation suppressed via prefers-reduced-motion.

A11y

  • The hero must contain exactly one <h1> per page.
  • Eyebrow is decorative text — keep it short and avoid making it the only heading source.
  • If a visual slot is decorative (rocket), mark aria-hidden="true"; if it carries meaning, give it an accessible name.

Tokens used

  • --fs-900, --fs-700, --fs-500
  • --accent (eyebrow tint), --text-strong (title)
  • --space-5…--space-7 (vertical rhythm)
  • --grad-mesh-1…--grad-mesh-3 when combined with scene-bg

Hero vs landing-portal

The portal ships two hero surfaces on purpose — they solve different problems:

  • hero.css (this component, .home-hero*) — the multi-purpose hero used by section indexes, role hubs, deep-dives, and the catalog hub. Variants cover eyebrow / title / lede / pillars / portrait. ~491 matches across the portal.
  • landing-portal.css (.lp-*) — the bespoke landing-page hero used only by the portal entry page. WebGL backdrop, animated pillars grid, tickers, decrypt-on-load. ~30 matches, single consumer.

Rule for new landing-style pages: extend one of these two — don't fork a third. If the page needs a calm, content-first hero, copy the home-hero variant that fits. If the page needs the «showcase the brand» treatment, copy the landing-portal template. The two are deliberately split because their dependencies diverge (multi-purpose hero ships in entry.css; landing-portal lazy-loads WebGL); merging them would force the WebGL bundle onto every section index.
