Interactive checklist

Checklist of large tone-railed cards with real checkboxes whose state persists in localStorage. Pairs a progress meter on top (count · bar · reset) with a strikethrough-on-check item style. Use for «walk this list before opening a PR» gates and similar walk-throughs the reader returns to over multiple sessions.

Example

0 / 3 gates passed

State is persisted under interactive-checklist:ui-kit-demo-v1; refresh and the boxes stay ticked.

Mount

<div data-component="interactive-checklist" data-storage-key="sa-principles-v1">
  <div class="checklist-meta">
    <span class="checklist-meta__progress">
      <span class="checklist-meta__num">0</span><span> / 5 gates passed</span>
    </span>
    <span class="checklist-meta__bar"><span class="checklist-meta__bar-fill"></span></span>
    <button type="button" class="checklist-meta__reset">Reset</button>
  </div>

  <ol class="checklist-list">
    <li class="checklist-item" data-tone="accent">
      <label class="checklist-item__label">
        <input type="checkbox" class="checklist-item__input" data-check-id="…">
        <span class="checklist-item__box" aria-hidden="true"></span>
        <div class="checklist-item__body">
          <p class="checklist-item__num"></p>
          <p class="checklist-item__q">Question?</p>
          <p class="checklist-item__yes"><strong>Yes if</strong> …</p>
          <p class="checklist-item__see">Maps to: <a href="…">Deep-dive →</a></p>
        </div>
      </label>
    </li>
    …
  </ol>

  <p class="checklist-footnote"><strong>Stop rule.</strong> If any gate isn't ticked …</p>
      </div>

Attributes

  • [data-component="interactive-checklist"] on the wrapper — mount marker.
  • [data-storage-key] on the wrapper — unique per checklist across the site. Stored under interactive-checklist:<key>.
  • [data-check-id] on each input — unique within the checklist. The key under which the tick is persisted.
  • [data-tone] on each item — one of accent · info · warn · success · rose. Drives --card-tone for the left rail, the checkbox, and the «See: …» link.

Tones

  • data-tone="accent" → --accent
  • data-tone="info" → --info
  • data-tone="warn" → --warn
  • data-tone="success"→ --success
  • data-tone="rose" → --rose

JS controller

interactive-checklist.js exports mountInteractiveChecklist(), which the runtime entry calls during boot. Per mount:

  • Reads state from localStorage[interactive-checklist:<key>] and ticks matching inputs.
  • Updates the count, the bar width, and the checklist-meta[data-complete] attribute on every change.
  • Wires .checklist-meta__reset to clear all ticks and re-render.
  • Storage errors (private mode, quota) are swallowed — the checkbox still works, state just isn't persisted.

A11y

  • Each item is a real <label> wrapping a real <input type="checkbox">. The whole card is the click target; the visual checkbox is decorative.
  • The hidden input is keyboard-focusable; :focus-visible draws the focus ring on .checklist-item__box so it tracks the input.
  • The visual box uses aria-hidden="true" because its content («✓») is purely decorative — the input is the accessible source of truth.
  • The progress bar is aria-hidden; the count next to it carries the same information textually.

When to use which

  • Quality checklist — static «pass/fail» grid; no state, no persistence. Use when the goal is to display a rubric, not walk through it.
  • Interactive checklist — multi-session walk-throughs the user re-opens (PR gates, release-readiness, on-call ramp-up).
  • Prose-only check items — use the simple <ul class="docs-checklist"> primitive from prose.
