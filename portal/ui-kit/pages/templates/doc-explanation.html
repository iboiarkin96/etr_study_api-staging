ADR 0030 Accepted

Why two portals share a single UI-kit

The public and internal portals look different on purpose. They behave the same on purpose. This ADR records how those two facts coexist in one component library.

Context

The public docs serve customers and prospects: marketing-grade typography, restrained palette, no operational widgets. The internal docs serve engineering and product: dense layouts, status pills, runbooks, dashboards. For two years we maintained two CSS forests — drift was constant, and the same bug typically had to be fixed twice.

Decision

One component library (assets_v2/ui-kit/) ships every visible widget. The portals only override tokens — color, accent, density, and a small set of layout knobs scoped to html[data-portal="public"] / html[data-portal="internal"].

Consequences

Good: a fix in button.css ships to both portals on the same commit. The portals can diverge stylistically — accent color, surface tone, typographic hierarchy — without forking behavior.

Bad: any cross-portal divergence must now be expressed as a token, not a CSS rule. Drive-by class overrides are tempting and they break the contract; reviewers should bounce them.

Alternatives considered

Keep the fork

Simpler to start, but the drift cost compounds. Two years of evidence in the previous repo.

Add a build step

Tooling overhead we don't want. The repo is intentionally framework-free.
